## 7.1 分布式基础理论

### 7.1.1 CAP 定理与 BASE 理论

#### CAP 定理

**CAP 定理**是分布式系统设计中最基础的理论，由 Eric Brewer 提出。它指出一个分布式系统最多只能同时满足以下三个特性中的两个：

| 特性 | 全称 | 含义 |
|------|------|------|
| **C** | Consistency（一致性） | 所有节点在同一时刻看到的数据完全一致 |
| **A** | Availability（可用性） | 每个请求都能在合理时间内收到非错误的响应 |
| **P** | Partition Tolerance（分区容错性） | 系统在网络分区发生时仍能继续工作 |

> **核心结论：在分布式环境中，网络分区（P）是必然发生的，因此系统只能在 CP 和 AP 之间做选择。**

**CP 系统**（强一致性）：ZooKeeper、HBase、Redis Cluster
- 牺牲可用性，保证数据一致
- 适用场景：金融交易、库存扣减

**AP 系统**（高可用性）：Eureka（⚠️ Eureka 2.x 已废弃，推荐使用 Nacos 或 Consul）、Cassandra、DynamoDB
- 牺牲强一致性，保证服务可用
- 适用场景：社交点赞、缓存系统

```
         C（一致性）
        / \
       /   \
      /     \
     /  CP   \    AP
    /   区域   \  区域
   /     |      \
  /      |       \
P（分区容错）---- A（可用性）
```

#### BASE 理论

**BASE 理论**是对 CAP 中 AP 方案的延伸，是 Basically Available（基本可用）、Soft State（软状态）、Eventually Consistent（最终一致性）的缩写。

| 概念 | 含义 | 示例 |
|------|------|------|
| **基本可用** | 允许损失部分可用性 | 响应时间增加、降级服务 |
| **软状态** | 允许系统存在中间状态 | 订单"支付中"状态 |
| **最终一致性** | 数据最终达到一致状态 | 异步复制、消息队列 |

```java
/**
 * BASE 理论实践示例：订单创建采用异步扣减库存（最终一致性）
 */
@Service
public class OrderService {

    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    /**
     * 创建订单 - 同步写订单 + 异步扣库存（AP 方案）
     */
    @Transactional
    public Order createOrder(OrderRequest request) {
        // 1. 同步：创建订单（状态为"待支付"）
        Order order = new Order();
        order.setProductId(request.getProductId());
        order.setQuantity(request.getQuantity());
        order.setStatus(OrderStatus.PENDING);
        orderMapper.insert(order);

        // 2. 异步：发送消息扣减库存（最终一致性）
        DeductStockMessage message = new DeductStockMessage();
        message.setOrderId(order.getId());
        message.setProductId(request.getProductId());
        message.setQuantity(request.getQuantity());
        rocketMQTemplate.convertAndSend("stock-deduct-topic", message);

        return order;
    }
}
```

#### 常见面试题

**Q: 为什么分布式系统中 P 是必须满足的？**

A: 分布式系统由多个节点通过网络通信组成，网络本身是不可靠的，光纤被挖断、交换机故障、DNS 解析异常等都会导致网络分区。既然 P 是不可避免的，系统设计只能在 CP 和 AP 之间权衡。

**Q: 一致性有哪些级别？**

A:
- **强一致性**：写入后立即对所有后续读取可见（如 ZooKeeper 的同步写）
- **顺序一致性**：所有进程看到操作的全局顺序一致
- **因果一致性**：有因果关系的操作按序可见
- **最终一致性**：在没有新更新的情况下，最终所有副本数据一致
- **弱一致性**：不保证任何顺序

### 7.1.2 分布式事务

#### 2PC（两阶段提交）

**2PC（Two-Phase Commit）** 是最经典的分布式事务协议，引入**协调者（Coordinator）**角色。

```
阶段一：Prepare（准备阶段）
┌──────────┐     prepare      ┌──────────┐
│ 协调者    │ ──────────────→  │ 参与者 A  │
│          │ ←──────────────  │          │  Yes/No
│          │ ──────────────→  │ 参与者 B  │
│          │ ←──────────────  │          │  Yes/No
└──────────┘                   └──────────┘

阶段二：Commit/Rollback（提交/回滚阶段）
┌──────────┐     commit       ┌──────────┐
│ 协调者    │ ──────────────→  │ 参与者 A  │
│          │ ──────────────→  │ 参与者 B  │
└──────────┘                   └──────────┘
```

**优点**：原理简单，实现方便
**缺点**：
- **同步阻塞**：参与者锁定资源直到事务结束
- **单点故障**：协调者宕机导致参与者一直阻塞
- **数据不一致**：协调者发送 commit 时部分参与者收不到

#### 3PC（三阶段提交）

3PC 在 2PC 基础上增加了 **CanCommit** 阶段和**超时机制**：

1. **CanCommit**：协调者询问参与者是否可以执行（不锁资源）
2. **PreCommit**：参与者预执行并锁定资源
3. **DoCommit**：真正提交或回滚

**改进**：引入超时机制，参与者等待超时后自动中断事务
**缺点**：仍无法完全解决网络分区导致的数据不一致问题，且多一次通信开销

#### TCC（Try-Confirm-Cancel）

**TCC** 是一种**业务层面的补偿型事务**方案，每个操作分为三个步骤：

```java
/**
 * TCC 分布式事务示例 - 跨账户转账
 */
public interface TransferService {

    /**
     * Try 阶段：冻结资源（预留金额）
     */
    @TwoPhaseBusinessAction(
        name = "transfer",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    boolean tryTransfer(
        @BusinessActionContextParameter(paramName = "fromAccountId") Long fromAccountId,
        @BusinessActionContextParameter(paramName = "toAccountId") Long toAccountId,
        @BusinessActionContextParameter(paramName = "amount") BigDecimal amount
    );

    /**
     * Confirm 阶段：确认执行（实际扣减/增加）
     */
    boolean confirm(BusinessActionContext context);

    /**
     * Cancel 阶段：取消执行（释放冻结金额）
     */
    boolean cancel(BusinessActionContext context);
}

@Service
public class TransferServiceImpl implements TransferService {

    @Autowired
    private AccountMapper accountMapper;

    @Override
    @Transactional
    public boolean tryTransfer(Long fromAccountId, Long toAccountId, BigDecimal amount) {
        // 冻结转出账户金额
        accountMapper.freezeAmount(fromAccountId, amount);
        // 记录转入账户待入账
        accountMapper.addPendingAmount(toAccountId, amount);
        return true;
    }

    @Override
    @Transactional
    public boolean confirm(BusinessActionContext context) {
        Long fromAccountId = (Long) context.getActionContext("fromAccountId");
        Long toAccountId = (Long) context.getActionContext("toAccountId");
        BigDecimal amount = (BigDecimal) context.getActionContext("amount");

        // 实际扣减冻结金额
        accountMapper.deductFrozen(fromAccountId, amount);
        // 待入账转为实际余额
        accountMapper.confirmPending(toAccountId, amount);
        return true;
    }

    @Override
    @Transactional
    public boolean cancel(BusinessActionContext context) {
        Long fromAccountId = (Long) context.getActionContext("fromAccountId");
        Long toAccountId = (Long) context.getActionContext("toAccountId");
        BigDecimal amount = (BigDecimal) context.getActionContext("amount");

        // 释放冻结金额
        accountMapper.unfreezeAmount(fromAccountId, amount);
        // 取消待入账
        accountMapper.cancelPending(toAccountId, amount);
        return true;
    }
}
```

#### Saga 模式

**Saga** 将长事务拆分为多个本地短事务，每个本地事务有对应的**补偿操作**：

```
T1 → T2 → T3 → T4（正常流程）
      ↓
   T2补偿 → T1补偿（T3失败时回滚）
```

```java
/**
 * Saga 模式示例 - 电商下单
 */
@Service
public class OrderSaga {

    @Autowired
    private OrderService orderService;
    @Autowired
    private InventoryService inventoryService;
    @Autowired
    private PaymentService paymentService;

    /**
     * 正向操作链
     */
    public void createOrder(Order order) {
        try {
            // Step 1: 创建订单
            orderService.create(order);
            // Step 2: 扣减库存
            inventoryService.deduct(order.getProductId(), order.getQuantity());
            // Step 3: 扣减余额
            paymentService.pay(order.getUserId(), order.getAmount());
        } catch (Exception e) {
            // 反向补偿链（按相反顺序执行补偿）
            compensateOrder(order);
            throw e;
        }
    }

    /**
     * 补偿操作链
     */
    private void compensateOrder(Order order) {
        try {
            paymentService.refund(order.getUserId(), order.getAmount());
        } catch (Exception ignored) {
            // 记录补偿失败，需要人工介入
        }
        try {
            inventoryService.restore(order.getProductId(), order.getQuantity());
        } catch (Exception ignored) {
        }
        try {
            orderService.cancel(order.getId());
        } catch (Exception ignored) {
        }
    }
}
```

#### Seata 分布式事务框架

**Seata** 是阿里开源的分布式事务解决方案，支持四种模式：

| 模式 | 原理 | 适用场景 |
|------|------|----------|
| **AT 模式** | 自动补偿，拦截 SQL 生成回滚日志 | 通用场景（推荐） |
| **TCC 模式** | 手动编写 Try/Confirm/Cancel | 高性能、非关系型数据库 |
| **Saga 模式** | 基于状态机的长事务 | 业务流程长 |
| **XA 模式** | 基于数据库 XA 协议 | 需要强一致性 |

```java
/**
 * Seata AT 模式示例 - 只需一个 @GlobalTransactional 注解
 */
@Service
public class BusinessService {

    @Autowired
    private OrderClient orderClient;
    @Autowired
    private StorageClient storageClient;
    @Autowired
    private AccountClient accountClient;

    /**
     * 全局事务：下订单 + 扣库存 + 扣余额
     * Seata 自动生成回滚日志，无需手动编写补偿逻辑
     */
    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void purchase(PurchaseRequest request) {
        // 1. 创建订单
        orderClient.create(request.getUserId(), request.getProductId(),
                           request.getOrderCount(), request.getAmount());
        // 2. 扣减库存
        storageClient.deduct(request.getProductId(), request.getOrderCount());
        // 3. 扣减账户余额
        accountClient.debit(request.getUserId(), request.getAmount());
    }
}
```

**Seata AT 模式核心原理**：
1. **一阶段**：业务 SQL 执行前，解析 SQL 生成**回滚日志（undo_log）**，保存到数据库；执行业务 SQL；删除回滚日志（异步）
2. **二阶段提交**：异步清理回滚日志
3. **二阶段回滚**：根据 undo_log 还原数据

#### 常见面试题

**Q: 2PC 和 TCC 的区别是什么？**

A: 2PC 是数据库层面的协议，依赖数据库的锁机制，性能较差，存在阻塞问题；TCC 是业务层面的补偿机制，不依赖数据库锁，灵活性高，但需要开发者手动编写 Try/Confirm/Cancel 三个方法，开发成本较高。

**Q: Seata AT 模式如何保证事务的隔离性？**

A: Seata AT 模式默认使用**全局读已提交**隔离级别。在写操作时，Seata 会获取**全局锁**（通过数据库的 SELECT ... FOR UPDATE 实现），确保在全局事务提交前，其他事务不能修改同一行数据。

### 7.1.3 分布式 ID 生成（雪花算法）

#### 为什么需要分布式 ID

在分布式系统中，数据库分库分表后无法使用自增 ID，需要全局唯一的 ID 生成方案。

#### 常见方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **UUID** | 简单、无中心化 | 无序、长度长、无法趋势递增 |
| **数据库自增** | 有序、简单 | 单点瓶颈、扩展困难 |
| **Redis 自增** | 性能好 | 依赖 Redis 可用性 |
| **雪花算法** | 有序、高性能、不依赖中心 | 时钟回拨问题 |

#### 雪花算法（Snowflake）

雪花算法由 Twitter 开源，生成一个 **64 bit 的 long 型 ID**：

```
0 | 00000000 00000000 00000000 00000000 00000000 0 | 00000 | 00000 | 000000000000
  |-------------------41 bit-----------------------|--5 bit--|--5 bit--|----12 bit----|
  |                    时间戳                        | 数据中心ID| 机器ID  |   序列号    |
```

- **1 bit**：符号位，始终为 0
- **41 bit**：时间戳（毫秒级），可用约 69 年
- **5 bit**：数据中心 ID（0~31）
- **5 bit**：机器 ID（0~31）
- **12 bit**：序列号（同一毫秒内 0~4095）

```java
/**
 * 雪花算法实现
 */
public class SnowflakeIdGenerator {

    // 起始时间戳（2024-01-01）
    private static final long EPOCH = 1704067200000L;

    // 各部分占用的位数
    private static final long SEQUENCE_BITS = 12L;
    private static final long WORKER_ID_BITS = 5L;
    private static final long DATACENTER_ID_BITS = 5L;

    // 最大值
    private static final long MAX_SEQUENCE = ~(-1L << SEQUENCE_BITS);        // 4095
    private static final long MAX_WORKER_ID = ~(-1L << WORKER_ID_BITS);      // 31
    private static final long MAX_DATACENTER_ID = ~(-1L << DATACENTER_ID_BITS); // 31

    // 位移
    private static final long WORKER_ID_SHIFT = SEQUENCE_BITS;
    private static final long DATACENTER_ID_SHIFT = SEQUENCE_BITS + WORKER_ID_BITS;
    private static final long TIMESTAMP_SHIFT = SEQUENCE_BITS + WORKER_ID_BITS + DATACENTER_ID_BITS;

    private final long workerId;
    private final long datacenterId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public SnowflakeIdGenerator(long workerId, long datacenterId) {
        if (workerId > MAX_WORKER_ID || workerId < 0) {
            throw new IllegalArgumentException("workerId 超出范围");
        }
        if (datacenterId > MAX_DATACENTER_ID || datacenterId < 0) {
            throw new IllegalArgumentException("datacenterId 超出范围");
        }
        this.workerId = workerId;
        this.datacenterId = datacenterId;
    }

    public synchronized long nextId() {
        long currentTimestamp = System.currentTimeMillis();

        // 时钟回拨检测
        if (currentTimestamp < lastTimestamp) {
            throw new RuntimeException("时钟回拨，拒绝生成 ID");
        }

        // 同一毫秒内，序列号递增
        if (currentTimestamp == lastTimestamp) {
            sequence = (sequence + 1) & MAX_SEQUENCE;
            if (sequence == 0) {
                // 序列号溢出，等待下一毫秒
                currentTimestamp = waitNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }

        lastTimestamp = currentTimestamp;

        return ((currentTimestamp - EPOCH) << TIMESTAMP_SHIFT)
                | (datacenterId << DATACENTER_ID_SHIFT)
                | (workerId << WORKER_ID_SHIFT)
                | sequence;
    }

    private long waitNextMillis(long lastTimestamp) {
        long timestamp = System.currentTimeMillis();
        while (timestamp <= lastTimestamp) {
            timestamp = System.currentTimeMillis();
        }
        return timestamp;
    }
}
```

**时钟回拨解决方案**：
1. 回拨时间短（<5ms）：等待追上
2. 回拨时间长：抛出异常或使用备用 workerId
3. 使用**百度 UidGenerator** 或 **美团 Leaf** 等增强方案

### 7.1.4 一致性哈希

#### 核心原理

**一致性哈希（Consistent Hashing）** 解决了传统哈希在节点增减时大量数据迁移的问题。

```
传统哈希：hash(key) % N（N 为节点数）
问题：N 变化时，几乎所有 key 的映射都会改变

一致性哈希：
1. 将整个哈希空间组织成一个虚拟的圆环（0 ~ 2^32 - 1）
2. 将节点（服务器）映射到环上
3. 将数据 key 映射到环上，顺时针找到第一个节点即为存储节点
```

```
              Node C
            /
    key4   /   key1
      \   /
       \ /
  Node A -------- Node B
       / \
     /     \
   key5   key2
            \
            key3
```

**虚拟节点**：为每个物理节点创建多个虚拟节点，使数据分布更均匀。

```java
/**
 * 一致性哈希实现（带虚拟节点）
 */
public class ConsistentHash<T> {

    // 虚拟节点数
    private static final int VIRTUAL_NODES = 150;

    // 哈希环：排序的 TreeMap，支持快速查找
    private final TreeMap<Long, T> hashRing = new TreeMap<>();

    // 真实节点集合
    private final List<T> realNodes = new ArrayList<>();

    public ConsistentHash(List<T> nodes) {
        this.realNodes.addAll(nodes);
        for (T node : realNodes) {
            addNode(node);
        }
    }

    /**
     * 将节点加入哈希环
     */
    public void addNode(T node) {
        for (int i = 0; i < VIRTUAL_NODES; i++) {
            String virtualNodeName = node.toString() + "##VN" + i;
            long hash = hash(virtualNodeName);
            hashRing.put(hash, node);
        }
    }

    /**
     * 移除节点
     */
    public void removeNode(T node) {
        for (int i = 0; i < VIRTUAL_NODES; i++) {
            String virtualNodeName = node.toString() + "##VN" + i;
            long hash = hash(virtualNodeName);
            hashRing.remove(hash);
        }
    }

    /**
     * 获取 key 应该存储的节点
     */
    public T getNode(String key) {
        if (hashRing.isEmpty()) {
            return null;
        }
        long hash = hash(key);
        // 顺时针找到第一个节点
        Map.Entry<Long, T> entry = hashRing.ceilingEntry(hash);
        if (entry == null) {
            // 环绕到第一个节点
            entry = hashRing.firstEntry();
        }
        return entry.getValue();
    }

    /**
     * FNV1_32_HASH 算法
     */
    private long hash(String str) {
        final int p = 16777619;
        long hash = 2166136261L;
        for (int i = 0; i < str.length(); i++) {
            hash = (hash ^ str.charAt(i)) * p;
        }
        hash += hash << 13;
        hash ^= hash >> 7;
        hash += hash << 3;
        hash ^= hash >> 17;
        hash += hash << 5;
        // 取正数
        return Math.abs(hash);
    }
}
```

#### 常见面试题

**Q: 一致性哈希在节点增减时，数据迁移量是多少？**

A: 只有**环形上相邻节点之间的数据**需要迁移。假设有 N 个节点，增加一个节点时，平均只有 `1/N` 的数据需要迁移。虚拟节点数量越多，数据分布越均匀，迁移比例越接近理论值。

## 7.2 消息队列（MQ）

### 7.2.1 RabbitMQ / Kafka / RocketMQ 对比

| 特性 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| **开发语言** | Erlang | Scala/Java | Java |
| **单机吞吐量** | 万级 | 百万级 | 十万级 |
| **延迟** | 微秒级 | 毫秒级 | 毫秒级 |
| **消息可靠性** | 高（confirm + persistent） | 高（副本机制） | 高（同步/异步刷盘） |
| **消息丢失** | 低 | 低（配置得当） | 低 |
| **消费模型** | 推模式 | 拉模式 | 推/拉模式 |
| **适用场景** | 业务消息、即时通信 | 日志、大数据 | 电商、金融交易 |
| **Topic 数量** | 万级 | 百级（过多影响性能） | 万级 |
| **事务消息** | 不支持分布式事务消息（可通过 Publisher Confirm + 本地消息表实现可靠投递） | 支持（仅半事务） | 支持 |
| **消息回溯** | 不支持 | 支持（offset） | 支持（时间戳） |
| **社区活跃度** | 高 | 非常高 | 高（阿里主导） |

```
选型建议：
├── 中小规模业务系统、需要复杂路由 → RabbitMQ
├── 大数据场景、日志收集、高吞吐 → Kafka
└── 电商/金融、需要事务消息、顺序消息 → RocketMQ
```

### 7.2.2 消息可靠性投递

#### RabbitMQ 可靠性投递

```java
/**
 * RabbitMQ 生产者确认机制
 */
@Configuration
public class RabbitMQConfig {

    @Bean
    public RabbitTemplate rabbitTemplate(ConnectionFactory connectionFactory) {
        RabbitTemplate template = new RabbitTemplate(connectionFactory);

        // 1. 开启消息确认模式（消息到达 Exchange 后回调）
        template.setConfirmCallback((correlationData, ack, cause) -> {
            if (ack) {
                log.info("消息成功到达 Exchange");
            } else {
                log.error("消息未到达 Exchange，原因：{}", cause);
                // 重发或记录到本地消息表

        // 2. 开启消息返回模式（消息从 Exchange 无法路由到 Queue 时回调）
        template.setReturnsCallback(returned -> {
            log.error("消息无法路由到 Queue，exchange={}, routingKey={}",
                    returned.getExchange(), returned.getRoutingKey());
        });

        return template;
    }

    @Bean
    public DirectExchange orderExchange() {
```

### 7.2.3 消息重复消费与幂等性

#### 为什么会重复消费

```
生产者发送 → [MQ] → 消费者消费
                ↑         ↓
           消费成功      网络超时
           ACK 丢失      生产者重发
```

#### 幂等性解决方案

```java
/**
 * 基于 Redis 的幂等性方案
 */
@Component
public class IdempotentConsumer {

    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    @Autowired
    private OrderMapper orderMapper;

    /**
     * 消费消息时先检查是否已处理过
     */
    @RabbitListener(queues = "order.queue")
    public void handleMessage(String messageBody,
                              @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag,
                              Channel channel) throws IOException {
        String messageId = getMessageId(messageBody); // 消息唯一ID

        // 1. Redis SETNX 判断是否已消费
        Boolean isFirst = redisTemplate.opsForValue()
                .setIfAbsent("msg:consumed:" + messageId, "1", 24, TimeUnit.HOURS);

        if (Boolean.FALSE.equals(isFirst)) {
            // 重复消息，直接 ACK
            channel.basicAck(deliveryTag, false);
            log.warn("重复消息，已跳过，messageId={}", messageId);
            return;
        }

        try {
            // 2. 执行业务逻辑
            Order order = JSON.parseObject(messageBody, Order.class);
            orderMapper.insert(order);

            // 3. 手动 ACK
            channel.basicAck(deliveryTag, false);
        } catch (Exception e) {
            // 4. 处理失败，NACK 并重新入队
            channel.basicNack(deliveryTag, false, true);
            // 删除幂等标记，允许重试
            redisTemplate.delete("msg:consumed:" + messageId);
        }
    }
}
```

**其他幂等方案**：

- **乐观锁（版本号）**：`UPDATE table SET value=value+1, version=version+1 WHERE id=? AND version=?`
- **状态机**：订单状态只能从"待支付"变为"已支付"，不能重复变更

### 7.2.5 消息积压处理

#### 积压原因

- 消费者处理速度 < 生产者发送速度
- 消费者宕机或处理异常
- 突发流量洪峰

#### 解决方案

```
方案一：紧急扩容消费者
1. 临时增加消费者实例数量
2. 提高消费者并行度（增加消费线程数）

方案二：临时队列转发
1. 创建一个临时 topic/queue
2. 将积压消息快速转发到临时队列
3. 用更多消费者消费临时队列

方案三：消息丢弃（非核心业务）
1. 只保留最近 N 小时的消息
2. 丢弃过期的积压消息
```

#### 常见面试题

**Q: 如何保证消息不丢失？**

A: 三个环节都要保证：
1. **生产者端**：开启 confirm 确认机制 + 本地消息表
2. **MQ 端**：Exchange 和 Queue 持久化 + 消息持久化 + 集群部署

### 7.3.2 负载均衡策略

#### 常见策略

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **轮询（Round Robin）** | 按顺序依次分配 | 简单、均匀 | 不考虑服务器性能差异 |
| **随机（Random）** | 随机选择 | 简单 | 可能分布不均 |
| **加权轮询** | 按权重比例分配 | 考虑性能差异 | 权重配置需人工调整 |
| **最少连接** | 分配给当前连接数最少的服务 | 动态适应 | 需要维护连接计数 |
| **一致性哈希** | 相同请求到相同节点 | 适合有状态服务 | 节点变化时有部分迁移 |
| **区域感知** | 优先同区域 | 降低延迟 | 实现复杂 |

```java
/**
 * 自定义负载均衡策略（Spring Cloud LoadBalancer）
 */
@Component
public class CustomLoadBalancer implements ReactorServiceInstanceLoadBalancer {

    private final ServiceInstanceListSupplier supplier;

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        return supplier.get().next().map(instances -> {
            // 加权最少连接策略示例
            ServiceInstance best = null;
            int minConnections = Integer.MAX_VALUE;

            for (ServiceInstance instance : instances) {
                int connections = getActiveConnections(instance);
                int weight = getInstanceWeight(instance);
                int score = connections * 100 / weight;

                if (score < minConnections) {
                    minConnections = score;
                    best = instance;
                }
            }
            return new DefaultResponse(best);
        });
    }
}
```

### 7.3.3 服务熔断与降级

#### 核心概念

- **熔断（Circuit Breaker）**：当下游服务异常率超过阈值时，**自动切断请求**，避免级联故障
- **降级（Fallback）**：熔断或超时后，返回**兜底数据**或执行兜底逻辑

#### 熔断器三种状态

```
        失败率超过阈值
  ┌──────────────────────┐
  ↓                      │
关闭 ──────────────→ 打开
  ↑                      │
  │    等待超时时间后      │
  └──── 半开 ←────────────┘
            │
       探测请求成功 → 关闭
       探测请求失败 → 打开
```

#### Sentinel 熔断降级示例

### 7.3.4 限流算法

#### 令牌桶算法（Token Bucket）

```
        固定速率放入令牌
    ┌───────────────────┐
    │   令牌桶（容量 N）  │ ←── 每秒添加 R 个令牌
    └───────────────────┘
              │
         请求到达时取令牌
         有令牌 → 通过
         无令牌 → 拒绝/排队
```

**特点**：允许一定程度的**突发流量**（桶内有累积令牌时）

#### 漏桶算法（Leaky Bucket）

```
    请求 → ┌─────────────┐ → 固定速率流出
           │   漏桶       │
           │  （队列缓冲）  │
           └─────────────┘
```

**特点**：流出速率**恒定**，适合平滑流量

#### 滑动窗口算法

```
固定窗口（有临界问题）：
|--- 1s ---|--- 1s ---|--- 1s ---|
           ↑
     0.5s: 90个请求    1.5s: 90个请求 → 1s内180个请求（超限但未被检测）

滑动窗口（更精确）：
|--- 1s ---|
      |--- 1s ---|
           |--- 1s ---|
    窗口随时间滑动，统计窗口内的请求数
```

```java
/**
 * 滑动窗口限流器实现
 */
public class SlidingWindowRateLimiter {

    private final int maxRequests;     // 窗口内最大请求数
    private final long windowSizeMs;   // 窗口大小（毫秒）
    private final ConcurrentLinkedDeque<Long> window = new ConcurrentLinkedDeque<>();

    public SlidingWindowRateLimiter(int maxRequests, long windowSizeMs) {
        this.maxRequests = maxRequests;
        this.windowSizeMs = windowSizeMs;
    }

    /**
     * 尝试获取许可
     */
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        long windowStart = now - windowSizeMs;

        // 移除窗口外的请求记录
        while (!window.isEmpty() && window.peekFirst() < windowStart) {
            window.pollFirst();
        }

        // 判断窗口内请求数是否超限
        if (window.size() < maxRequests) {
            window.addLast(now);
            return true;
        }
        return false;
    }
}
```

#### 常见面试题

**Q: 熔断和降级的区别是什么？**

A: 熔断是一种**保护机制**，当下游服务异常率过高时自动切断请求，防止故障扩散；降级是一种**兜底策略**，在服务不可用时返回预设的默认值或执行简化逻辑。熔断是降级的触发条件之一，降级是熔断后的处理方式。

**Q: 令牌桶和漏桶的区别？**

A: 令牌桶允许突发流量（桶中有累积令牌时可以一次性处理多个请求），漏桶强制以恒定速率处理请求。令牌桶更适合允许短时间突发的场景，漏桶更适合需要严格平滑流量的场景。

```java
 * Redis 分布式锁基础实现
 */
@Component
public class RedisDistributedLock {

    @Autowired
    private StringRedisTemplate redisTemplate;

    /**
     * 尝试获取锁
     * @param lockKey   锁的 key
     * @param requestId 请求唯一标识（防止误删别人的锁）
     * @param expireTime 过期时间（秒）
     */
    public boolean tryLock(String lockKey, String requestId, long expireTime) {
        // SET lock_key request_id NX EX expire_time
        Boolean result = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, requestId, expireTime, TimeUnit.SECONDS);
        return Boolean.TRUE.equals(result);
    }

    /**
     * 释放锁（Lua 脚本保证原子性）
     */
    public boolean releaseLock(String lockKey, String requestId) {
        String script =
                "if redis.call('get', KEYS[1]) == ARGV[1] then " +
                "   return redis.call('del', KEYS[1]) " +
                "else " +
                "   return 0 " +
                "end";
        Long result = redisTemplate.execute(
                new DefaultRedisScript<>(script, Long.class),
                Collections.singletonList(lockKey),
                requestId
        );
        return Long.valueOf(1L).equals(result);
    }
}
```

```java
/**
 * Redisson 分布式锁使用
 */
@Service
public class InventoryService {

    @Autowired
    private RedissonClient redissonClient;

    /**
     * 扣减库存 - 使用 Redisson 分布式锁
     */
    public boolean deductStock(Long productId, int quantity) {
        String lockKey = "stock:lock:" + productId;
        RLock lock = redissonClient.getLock(lockKey);

        try {
            // 尝试加锁：等待 5 秒，锁自动释放时间 30 秒
            boolean acquired = lock.tryLock(5, 30, TimeUnit.SECONDS);
            if (!acquired) {
                throw new BusinessException("系统繁忙，请稍后重试");
            }

            // 执行业务逻辑
            int stock = getStock(productId);
            if (stock < quantity) {
                return false;
            }
            updateStock(productId, stock - quantity);
            return true;

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BusinessException("获取锁被中断");
        } finally {
            // 释放锁（仅释放自己持有的锁）
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

#### Redisson 看门狗机制

```
Redisson 看门狗（Watchdog）原理：
1. 默认 lockWatchdogTimeout = 30 秒
2. 加锁时不指定 leaseTime，看门狗自动续期
3. 看门狗每隔 10 秒（30/3）检查锁是否还被持有
4. 如果被持有，自动将锁的过期时间重置为 30 秒
5. 客户端宕机后，看门狗停止续期，锁自动过期释放

时间线：
加锁 → [10s续期] → [10s续期] → [10s续期] → 客户端宕机 → 锁30s后自动释放
```

### 7.4.2 ZooKeeper 分布式锁

#### 原理

```
ZooKeeper 分布式锁基于临时顺序节点：
1. 客户端在 /locks 路径下创建临时顺序节点 /locks/lock-0000000001
2. 获取 /locks 下所有子节点并排序
3. 判断自己创建的节点是否为最小序号
4. 如果是最小 → 获取锁成功
5. 如果不是最小 → 监听（Watch）前一个节点的删除事件
6. 前一个节点删除（释放锁）→ 收到通知 → 重新判断

/locks
  ├── lock-0000000001  ← 最小，获取锁
  ├── lock-0000000002  ← 监听 lock-0000000001
  └── lock-0000000003  ← 监听 lock-0000000002
```

```java
/**
 * ZooKeeper 分布式锁实现
 */
public class ZkDistributedLock implements Watcher {

    private final ZooKeeper zk;
    private final String lockPath;
    private String currentPath;
    private String watchPath;
    private CountDownLatch latch;

    public ZkDistributedLock(String connectString, String lockName) throws Exception {
        this.zk = new ZooKeeper(connectString, 30000, this);
        this.lockPath = "/locks/" + lockName;
        // 确保父节点存在
        ensurePath(lockPath);
    }

    /**
     * 获取锁
     */
    public void lock() throws Exception {
        // 1. 创建临时顺序节点
        currentPath = zk.create(lockPath + "/lock-", new byte[0],
                ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.EPHEMERAL_SEQUENTIAL);

        // 2. 尝试获取锁
        if (!tryLock()) {
            // 3. 未获取到锁，等待前一个节点释放
            waitForLock();
        }
    }

    private boolean tryLock() throws Exception {
        // 获取所有子节点并排序
        List<String> children = zk.getChildren(lockPath, false);
        Collections.sort(children);

        // 当前节点的序号
        String currentNode = currentPath.substring(lockPath.length() + 1);
        int currentIndex = children.indexOf(currentNode);

        if (currentIndex == 0) {
            // 是最小节点，获取锁成功
            return true;
        }

        // 监听前一个节点
        String prevNode = children.get(currentIndex - 1);
        watchPath = lockPath + "/" + prevNode;
        return false;
    }

    private void waitForLock() throws Exception {
        latch = new CountDownLatch(1);
        // 监听前一个节点的删除事件
        zk.exists(watchPath, this);
        latch.await();
        // 被唤醒后重新尝试获取锁
        if (!tryLock()) {
            waitForLock();
        }
    }

    @Override
    public void process(WatchedEvent event) {
        if (event.getType() == Event.EventType.NodeDeleted) {
            latch.countDown();
        }
    }

    /**
     * 释放锁（删除节点）
     */
    public void unlock() throws Exception {
        zk.delete(currentPath, -1);
    }
}
```

```java
public class DatabaseDistributedLock {

     * 获取锁（基于唯一索引 + 乐观锁）

            DistributedLock lock = new DistributedLock();
            lock.setLockKey(lockKey);
            lock.setOwner(owner);
            lock.setExpireTime(LocalDateTime.now().plusSeconds(expireSeconds));
            lockMapper.insert(lock);

    public boolean releaseLock(String lockKey, String owner) {
        // 使用 owner 条件防止误删
        int rows = lockMapper.deleteByOwner(lockKey, owner);
        return rows > 0;
    }
}
```

#### 常见面试题

**Q: Redis 分布式锁如何防止死锁？**

A: (1) 设置过期时间（`SET key value NX EX`），即使客户端宕机锁也会自动释放；(2) 使用 Redisson 的看门狗机制自动续期；(3) 释放锁时使用 Lua 脚本验证身份，防止误删别人的锁。

**Q: Redis 主从切换时锁会丢失吗？如何解决？**

A: 会丢失。客户端在 Master 上加锁后，Master 还未同步到 Slave 就宕机了，Slave 升级为 Master 后锁信息丢失。解决方案是使用 **RedLock 算法**：向 N 个独立的 Redis 节点同时加锁，超过半数成功才算加锁成功。

```java
/**
 * JWT 认证全局过滤器
 */
@Component
@Slf4j
public class JwtAuthGlobalFilter implements GlobalFilter, Ordered {

    @Autowired
    private JwtTokenProvider jwtTokenProvider;

    // 白名单路径（不需要认证）
    private static final List<String> WHITE_LIST = Arrays.asList(
            "/api/auth/login",
            "/api/auth/register",
            "/api/auth/refresh"
    );

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        // 白名单放行
        if (isWhiteListed(path)) {
            return chain.filter(exchange);
        }

        // 获取 Token
        String token = extractToken(exchange.getRequest());
        if (token == null) {
            return unauthorizedResponse(exchange, "未登录或 Token 已过期");
        }

        // 验证 Token
        try {
            Claims claims = jwtTokenProvider.parseToken(token);
            String userId = claims.getSubject();
            List<String> roles = claims.get("roles", List.class);

            // 将用户信息传递给下游服务
            ServerHttpRequest request = exchange.getRequest().mutate()
                    .header("X-User-Id", userId)
                    .header("X-User-Roles", String.join(",", roles))
                    .build();

            return chain.filter(exchange.mutate().request(request).build());

        } catch (ExpiredJwtException e) {
            return unauthorizedResponse(exchange, "Token 已过期");
        } catch (JwtException e) {
            return unauthorizedResponse(exchange, "无效的 Token");
        }
    }

    private String extractToken(ServerHttpRequest request) {
        String bearerToken = request.getHeaders().getFirst("Authorization");
        if (bearerToken != null && bearerToken.startsWith("Bearer ")) {
            return bearerToken.substring(7);
        }
        return null;
    }

    private Mono<Void> unauthorizedResponse(ServerWebExchange exchange, String message) {
        ServerHttpResponse response = exchange.getResponse();
        response.setStatusCode(HttpStatus.UNAUTHORIZED);
        response.getHeaders().setContentType(MediaType.APPLICATION_JSON);
        String body = "{\"code\":401,\"message\":\"" + message + "\"}";
        DataBuffer buffer = response.bufferFactory().wrap(body.getBytes(StandardCharsets.UTF_8));
        return response.writeWith(Mono.just(buffer));
    }

    @Override
    public int getOrder() {
        return -100; // 最高优先级
    }
}

/**
 * JWT 工具类
 */
@Component
public class JwtTokenProvider {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.expiration}")
    private long expiration;

    private SecretKey getSigningKey() {
        return Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }

    /**
     * 生成 Token
     */
    public String generateToken(Long userId, List<String> roles) {
        return Jwts.builder()
                .subject(String.valueOf(userId))
                .claim("roles", roles)
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + expiration))
                .signWith(getSigningKey())
                .compact();
    }

    /**
     * 解析 Token
     */
    public Claims parseToken(String token) {
        return Jwts.parser()
                .verifyWith(getSigningKey())
                .build()
                .parseSignedClaims(token)
                .getPayload();
    }
}
```

#### 常见面试题

**Q: Nacos 如何实现服务发现？**

A: Nacos 支持两种模式：(1) **Push 模式**：服务端主动推送变更给客户端（UDP）；(2) **Pull 模式**：客户端定时拉取服务列表并缓存到本地。Nacos 同时使用两种模式，Push 保证实时性，Pull 作为兜底。客户端本地有缓存，即使 Nacos 宕机也能使用缓存的服务列表进行调用。

**Q: Eureka 的自我保护模式是什么？**

A: 当 Eureka Server 在短时间内丢失过多客户端心跳（超过 85%），会进入自我保护模式。此时不再剔除任何服务实例，保护注册表完整性。这适用于网络分区场景，避免因网络问题导致大量健康服务被误剔除。生产环境建议开启自我保护模式。

#### 热更新原理

```
Nacos Config 热更新流程：
1. 客户端启动时拉取配置并缓存到本地
2. 客户端注册监听器到 Nacos Server（长轮询）
3. 配置变更时，Nacos Server 推送变更通知
4. 客户端收到通知后重新拉取最新配置
5. Spring Context 刷新 Bean（@RefreshScope）
6. @ConfigurationProperties 标注的 Bean 自动更新属性值

长轮询机制：
客户端 → POST /configs/listener（携带配置 MD5）
Nacos Server → 30s 内配置无变化返回空
Nacos Server → 配置变化立即返回变更的 dataId
客户端 → 重新拉取变更的配置
```

#### 常见面试题

**Q: SkyWalking 的链路追踪原理是什么？**

A: SkyWalking Agent 通过 Java Agent 技术（字节码增强）在应用启动时自动注入追踪代码。当请求进入时，Agent 自动创建 Span（跨度），记录操作名称、开始时间、标签等信息。Span 之间通过 Trace ID 关联形成完整的调用链。Trace ID 通过 HTTP Header（SW8）在服务间透传，实现跨服务的链路串联。

**Q: Prometheus 和 ELK 的区别？**

A: Prometheus 专注于**指标监控**（Metrics），适合监控 CPU、内存、QPS、延迟等数值型数据，使用 Pull 模式采集数据。ELK 专注于**日志分析**（Logs），适合存储和检索文本型日志数据，使用 Push 模式采集数据。两者互补，通常一起使用：Prometheus 做指标监控和告警，ELK 做日志查询和问题排查。

## 7.9 常见面试题汇总

### Q1: 什么是 CAP 定理？你在项目中如何选择 CP 还是 AP？

A: CAP 定理指出分布式系统无法同时满足一致性（C）、可用性（A）和分区容错性（P）。由于网络分区不可避免，实际只能在 CP 和 AP 之间选择。选择依据是业务场景：金融交易、库存扣减等需要强一致性的场景选择 CP（如 ZooKeeper）；社交点赞、评论等允许短暂不一致的场景选择 AP（如 Eureka、Nacos 临时实例）。在实际项目中，大部分互联网应用采用 AP + 最终一致性方案，通过消息队列、TCC 等手段保证数据最终一致。

### Q2: 分布式事务有哪些解决方案？你用过哪种？

A: 常见方案有 2PC、3PC、TCC、Saga、本地消息表和 Seata。我在项目中使用过 Seata AT 模式，只需在方法上加 `@GlobalTransactional` 注解，Seata 自动拦截 SQL 生成 undo_log 实现自动回滚，对业务代码侵入小。对于非关系型数据库或高性能场景，会使用 TCC 模式手动编写三个阶段。本地消息表适合对实时性要求不高的异步场景。

### Q3: 如何保证消息不丢失？

A: 从三个环节保证：(1) **生产者端**：开启 RabbitMQ 的 confirm 确认机制，确保消息到达 Exchange；使用本地消息表保证消息一定能发出。(2) **MQ 端**：Exchange、Queue 和消息都设置持久化（durable=true）；MQ 集群部署，消息多副本存储。(3) **消费者端**：关闭自动 ACK，在业务逻辑执行成功后手动 ACK；消费失败时进行重试并记录日志。

### Q4: 如何处理消息重复消费问题？

A: 消息重复消费的根本原因是网络抖动导致 ACK 丢失。解决方案是保证消费端的**幂等性**。常用方案有：(1) **Redis SETNX**：消费前以消息唯一 ID 为 key 写入 Redis，已存在则跳过；(2) **数据库唯一索引**：利用唯一约束防止重复插入；(3) **乐观锁**：通过版本号机制防止重复更新；(4) **状态机**：订单状态只能单向流转，重复消费不会改变状态。

### Q5: Redis 分布式锁和 ZooKeeper 分布式锁的区别？

A: (1) **性能**：Redis 基于内存操作，性能更高；ZooKeeper 需要磁盘持久化，性能稍低。(2) **可靠性**：ZooKeeper 基于 ZAB 协议保证强一致性，更可靠；Redis 主从切换时可能丢失锁。(3) **锁释放**：Redis 依赖过期时间 + 看门狗续期；ZooKeeper 通过临时节点，session 断开自动释放。(4) **公平性**：ZooKeeper 通过顺序节点实现公平锁；Redis 默认非公平。高并发场景优先选 Redis，强一致性场景优先选 ZooKeeper。

### Q6: 什么是熔断降级？Sentinel 和 Hystrix 的区别？

A: 熔断是当下游服务异常率超过阈值时自动切断请求的保护机制，降级是熔断后返回兜底数据或执行兜底逻辑。Sentinel 和 Hystrix 的区别：(1) Sentinel 支持更丰富的限流规则（QPS、线程数、关联限流）；(2) Sentinel 支持热点参数限流、系统保护；(3) Hystrix 使用线程池隔离，Sentinel 使用信号量隔离（更轻量）；(4) Sentinel 提供可视化控制台，支持动态规则推送；(5) Hystrix 已停止维护，Sentinel 持续更新。

### Q7: 限流算法有哪些？令牌桶和漏桶的区别？

A: 常见限流算法有固定窗口、滑动窗口、令牌桶和漏桶。令牌桶以固定速率向桶中放入令牌，请求需要获取令牌才能通过，桶中有累积令牌时允许突发流量。漏桶以固定速率从桶中流出请求，流入速率可以任意，但流出速率恒定，适合平滑流量。令牌桶更适合允许短时间突发的场景（如秒杀），漏桶更适合需要严格匀速流出的场景。

### Q8: Spring Cloud Gateway 的核心组件有哪些？

A: Gateway 有三个核心组件：(1) **Route（路由）**：由 ID、目标 URI、Predicate 和 Filter 组成，是网关的基本构建块。(2) **Predicate（断言）**：匹配 HTTP 请求的条件，支持 Path、Method、Header、Query、Cookie 等多种匹配方式。(3) **Filter（过滤器）**：分为 GlobalFilter（全局）和 GatewayFilter（路由级），可以在请求转发前后对请求和响应进行修改，如添加 Header、限流、熔断等。

### Q9: Nacos 和 Eureka 的区别？

A: (1) Nacos 支持 AP 和 CP 两种模式切换，Eureka 只有 AP 模式；(2) Nacos 内置配置中心，Eureka 需配合 Spring Cloud Config；(3) Nacos 支持服务分组和命名空间，方便多环境管理；(4) Nacos 支持权重路由和流量管理；(5) Eureka 已停止维护（2.x），Nacos 持续更新且国内社区活跃；(6) Nacos 支持推送和拉取两种服务发现方式，Eureka 只有拉取。

### Q10: 什么是服务雪崩？如何防止？

A: 服务雪崩是指一个服务的不可用导致级联故障，最终使整个微服务系统不可用。例如服务 A 调用服务 B，服务 B 响应变慢导致线程池耗尽，进而影响服务 A，最终整个系统崩溃。防止措施：(1) **熔断降级**：使用 Sentinel/Hystrix 在异常率过高时快速失败；(2) **限流**：控制请求量，防止系统过载；(3) **超时设置**：合理设置调用超时时间，避免长时间等待；(4) **线程隔离**：不同服务使用独立线程池；(5) **缓存**：对热点数据增加缓存，减少下游调用。

### Q11: 一致性哈希解决了什么问题？如何优化？

A: 一致性哈希解决了传统哈希（hash % N）在节点增减时大量数据迁移的问题。它将节点和数据都映射到一个哈希环上，数据顺时针找到最近的节点存储。节点增减时只影响环上相邻节点间的数据，迁移量约为 1/N。优化方案是引入**虚拟节点**：为每个物理节点在环上创建多个虚拟节点，使数据分布更均匀，避免数据倾斜问题。通常每个物理节点对应 150~200 个虚拟节点。

### Q12: 雪花算法的原理是什么？如何解决时钟回拨问题？

A: 雪花算法生成 64 位 long 型 ID，由 1 位符号位 + 41 位时间戳 + 5 位数据中心 ID + 5 位机器 ID + 12 位序列号组成。同一毫秒内通过序列号区分，不同毫秒通过时间戳区分。时钟回拨的解决方案：(1) 回拨时间短（<5ms）时等待追上；(2) 回拨时间长时抛出异常或使用备用 workerId；(3) 使用增强方案如美团 Leaf（基于数据库号段）或百度 UidGenerator（使用环形缓冲区）。

### Q13: 如何设计一个高可用的微服务架构？

A: 高可用微服务架构的关键设计：(1) **服务注册发现**：Nacos 集群部署，支持健康检查和自动摘除；(2) **负载均衡**：客户端负载均衡 + 同集群优先策略；(3) **熔断降级**：Sentinel 实现服务熔断、限流和降级；(4) **网关**：Spring Cloud Gateway 统一入口，实现认证、限流、路由；(5) **配置中心**：Nacos Config 实现配置集中管理和热更新；(6) **链路追踪**：SkyWalking 实现全链路追踪和性能监控；(7) **日志**：ELK 统一日志收集和查询；(8) **监控告警**：Prometheus + Grafana 实现指标监控和告警；(9) **容灾**：多机房部署、数据备份、故障自动转移。

### Q14: 消息积压了怎么处理？

A: 首先排查积压原因（消费者异常、消费速度慢、突发流量）。处理方案：(1) **消费者扩容**：临时增加消费者实例和消费线程数（注意 Kafka 需确保分区数足够）；(2) **临时队列转发**：如果积压量极大，编写临时程序快速将积压消息转发到扩容后的临时 topic，用更多消费者并行消费；(3) **排查消费逻辑**：检查是否有慢 SQL、外部调用超时等性能瓶颈；(4) **丢弃过期消息**：非核心业务可以丢弃过期的积压消息；(5) **预防措施**：合理设置消费并发度、增加监控告警、做好容量规划。

### Q15: Seata 的 AT 模式和 TCC 模式的区别？

A: AT 模式是**自动补偿**模式，Seata 拦截业务 SQL 自动生成回滚日志（undo_log），对业务代码无侵入，只需加 `@GlobalTransactional` 注解。但 AT 模式依赖关系型数据库，有全局锁开销。TCC 模式是**手动补偿**模式，需要开发者手动编写 Try（预留资源）、Confirm（确认执行）、Cancel（取消执行）三个方法，开发成本高但灵活性更强，适用于非关系型数据库或需要细粒度资源控制的场景。

### Q16: 什么是幂等性？在分布式系统中如何实现？

A: 幂等性是指对同一操作执行一次和执行多次的效果相同。在分布式系统中，由于网络重试、消息重复等原因，接口必须保证幂等性。实现方式：(1) **唯一索引**：数据库层面防止重复插入；(2) **乐观锁**：通过版本号机制防止重复更新；(3) **状态机**：订单状态只能单向流转；(4) **Token 机制**：请求前先获取 Token，提交时验证并删除 Token；(5) **Redis 去重**：以请求唯一标识为 key，SETNX 判断是否已处理。实际项目中通常组合使用多种方案。

### Q17: 微服务之间如何通信？

A: 微服务间通信主要有两种方式：(1) **同步调用**：使用 HTTP/REST（Feign）或 RPC（Dubbo、gRPC），适合需要实时响应的场景，但存在级联故障风险；(2) **异步消息**：使用消息队列（RabbitMQ、Kafka、RocketMQ），适合解耦和削峰填谷，但增加了系统复杂度。选择依据是业务场景：查询类操作适合同步调用，命令类操作（如订单创建后的通知、积分发放）适合异步消息。实际项目中通常是同步 + 异步混合使用。

### Q18: 如何实现接口的灰度发布？

A: 灰度发布（金丝雀发布）的实现方式：(1) **网关层路由**：在 Spring Cloud Gateway 中根据请求 Header、Cookie 或用户标签将部分流量路由到新版本服务；(2) **负载均衡层**：自定义 LoadBalancer 规则，按权重将一定比例的流量分发到新版本实例；(3) **Nacos 元数据**：在新版本实例的 metadata 中标记版本号，网关或负载均衡根据版本号进行路由；(4) **全链路灰度**：通过请求 Header 透传灰度标识，确保灰度请求在整个调用链中始终路由到灰度实例。
