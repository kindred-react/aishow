## 一、设计原则（SOLID + 其他）

设计原则是设计模式的基础，理解原则才能灵活运用模式。它们不是死板的规则，而是指导我们写出可维护、可扩展代码的**经验总结**。

### 1.1 单一职责原则（SRP - Single Responsibility Principle）

**定义：** 一个类应该只有一个引起它变化的原因。换句话说，一个类只负责一项职责。

**核心思想：** 如果一个类承担了过多的职责，那么当其中一个职责需要变化时，可能会影响到其他职责的正常运作。这就是"耦合"的根源。

```java
// ❌ 违反SRP的示例：一个类既负责用户数据存储，又负责邮件通知
public class UserService {
    public void saveUser(User user) {
        // 保存到数据库
        userDao.insert(user);
        // 发送欢迎邮件
        emailService.sendWelcomeEmail(user);
    }
}

// ✅ 遵循SRP的示例：职责分离
public class UserService {
    private final UserDao userDao;
    private final NotificationService notificationService;

    public UserService(UserDao userDao, NotificationService notificationService) {
        this.userDao = userDao;
        this.notificationService = notificationService;
    }

    public void register(User user) {
        userDao.insert(user);
        notificationService.sendWelcome(user);
    }
}

// UserDao 只负责数据持久化
@Repository
public class UserDao {
    public void insert(User user) {
        // 数据库操作
    }
}

// NotificationService 只负责通知
@Service
public class NotificationService {
    public void sendWelcome(User user) {
        // 发送邮件/短信
    }
}
```

**UML 类图（文字描述）：**
```
UserService
  ├── UserDao（数据层）
  └── NotificationService（通知层）
```

**React/前端对比：**
在React中，SRP体现为组件拆分和自定义Hook的分离。比如一个用户列表组件，不应该同时包含数据获取逻辑和UI渲染逻辑：
```jsx
// ❌ 违反SRP
function UserList() {
  const [users, setUsers] = useState([]);
  useEffect(() => { fetchUsers().then(setUsers); }, []);
  return <div>{users.map(u => <div key={u.id}>{u.name}</div>)}</div>;
}

// ✅ 遵循SRP：自定义Hook + 展示组件
function useUsers() {
  const [users, setUsers] = useState([]);
  useEffect(() => { fetchUsers().then(setUsers); }, []);
  return users;
}
function UserList() {
  const users = useUsers();
  return <div>{users.map(u => <div key={u.id}>{u.name}</div>)}</div>;
}
```

> ⭐ **面试高频问题：**
> - Q: 单一职责原则中的"职责"如何界定？
> - A: 职责可以理解为"变化的原因"。如果一个类有多个变化的原因，就需要拆分。实际判断时，可以问自己：这个类修改的原因有几个？如果多于一个，就可能违反了SRP。但也要注意，过度拆分会导致类的数量爆炸，需要根据项目规模和团队情况权衡。

### 1.2 开闭原则（OCP - Open/Closed Principle）

**定义：** 软件实体（类、模块、函数）应该对扩展开放，对修改关闭。即在不修改已有代码的前提下，通过扩展来添加新功能。

**核心思想：** 当需求变化时，我们通过添加新代码来实现，而不是修改已经测试通过的代码。这要求我们做好**抽象**和**多态**的设计。

```java
// ❌ 违反OCP的示例：每次新增支付方式都要修改原有代码
public class PaymentService {
    public void pay(String type, double amount) {
        if ("ALIPAY".equals(type)) {
            System.out.println("支付宝支付: " + amount);
        } else if ("WECHAT".equals(type)) {
            System.out.println("微信支付: " + amount);
        } else if ("CREDIT_CARD".equals(type)) {
            // 新增支付方式需要修改这里 —— 违反OCP！
            System.out.println("信用卡支付: " + amount);
        }
    }
}

// ✅ 遵循OCP的示例：通过策略模式 + 工厂模式实现扩展开放
// 定义支付策略接口
public interface PaymentStrategy {
    void pay(double amount);
}

// 各支付方式实现接口
public class AlipayStrategy implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("支付宝支付: " + amount);
    }
}

public class WechatPayStrategy implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("微信支付: " + amount);
    }
}

// 新增支付方式只需新增类，无需修改已有代码
public class CreditCardStrategy implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("信用卡支付: " + amount);
    }
}

// 支付服务依赖抽象，不依赖具体实现
public class PaymentService {
    private final PaymentStrategy paymentStrategy;

    public PaymentService(PaymentStrategy paymentStrategy) {
        this.paymentStrategy = paymentStrategy;
    }

    public void pay(double amount) {
        paymentStrategy.pay(amount);
    }
}
```

**UML 类图（文字描述）：**
```
<<interface>> PaymentStrategy
  ├── AlipayStrategy
  ├── WechatPayStrategy
  └── CreditCardStrategy  （新增，无需修改已有代码）

PaymentService --> PaymentStrategy（依赖抽象）
```

**React/前端对比：**
React中的开闭原则体现在组件的可组合性上。通过`children` prop或render props，我们可以在不修改组件源码的情况下扩展功能：
```jsx
// ✅ 对扩展开放：通过children或render props
function Card({ title, children }) {
  return <div className="card"><h2>{title}</h2>{children}</div>;
}
// 使用时可以放入任意内容，无需修改Card组件
<Card title="用户信息"><UserProfile /><UserOrders /></Card>
```

> ⭐ **面试高频问题：**
> - Q: 开闭原则和策略模式有什么关系？
> - A: 策略模式是开闭原则的经典实现方式。通过定义策略接口，新增策略时只需新增实现类，调用方代码无需修改。开闭原则是目标，策略模式是手段之一。
> - Q: 如何在项目中落地开闭原则？
> - A: 关键在于识别出"变化点"，对变化点进行抽象。常用的手段包括：策略模式、模板方法模式、观察者模式等。同时，合理使用接口和抽象类来定义扩展点。

### 1.3 里氏替换原则（LSP - Liskov Substitution Principle）

**定义：** 所有引用基类的地方必须能透明地使用其子类的对象。子类可以扩展基类的功能，但不能改变基类原有的功能。

**核心思想：** 继承是一种"IS-A"关系。子类必须完全兼容基类的行为契约。如果子类不能完全替代基类，那么继承关系就是不合理的。

```java
// ❌ 违反LSP的经典示例：正方形不是长方形
public class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) { this.width = width; }
    public void setHeight(int height) { this.height = height; }
    public int getArea() { return width * height; }
}

// 正方形继承长方形 —— 但行为不一致！
public class Square extends Rectangle {
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width; // 正方形宽高相等
    }

    @Override
    public void setHeight(int height) {
        this.width = height;
        this.height = height; // 破坏了基类的行为契约
    }
}

// 使用方代码会出问题
public class Demo {
    public static void resize(Rectangle r) {
        r.setWidth(5);
        r.setHeight(10);
        // 期望面积是50，但如果传入Square，面积是100！
        assert r.getArea() == 50; // ❌ 失败！
    }
}

// ✅ 遵循LSP的修复方案：提取公共抽象
public interface Shape {
    int getArea();
}

public class Rectangle implements Shape {
    private final int width;
    private final int height;

    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }

    @Override
    public int getArea() { return width * height; }
}

public class Square implements Shape {
    private final int side;

    public Square(int side) { this.side = side; }

    @Override
    public int getArea() { return side * side; }
}
```

**LSP 的四个约束条件：**
1. 子类必须实现父类的所有抽象方法
2. 子类的输入参数类型必须与父类相同或更宽松（前置条件不能加强）
3. 子类的返回值类型必须与父类相同或更严格（后置条件不能削弱）
4. 子类不能抛出父类未声明的异常（除非是父类异常的子类）

> ⭐ **面试高频问题：**
> - Q: 里氏替换原则的核心是什么？举一个违反的例子？
> - A: 核心是子类必须能完全替代父类使用而不改变程序的正确性。经典反例是"正方形继承长方形"——正方形重写了setWidth/setHeight使得宽高始终相等，破坏了长方形"宽高可以独立设置"的行为契约。正确做法是提取Shape接口，让Rectangle和Square各自实现。
> - Q: LSP和面向对象继承有什么关系？
> - A: LSP是继承关系的约束条件。不是所有的"IS-A"语义都适合用继承实现。如果子类需要重写父类的非抽象方法来改变行为，很可能违反了LSP，此时应该考虑用组合替代继承。

### 1.4 接口隔离原则（ISP - Interface Segregation Principle）

**定义：** 客户端不应该被迫依赖它不使用的方法。应该将臃肿的接口拆分为更小、更具体的接口。

**核心思想：** 接口应该"小而专"，而不是"大而全"。一个类对另一个类的依赖应该建立在最小的接口上。

```java
// ❌ 违反ISP的示例：臃肿的接口
public interface Animal {
    void eat();
    void fly();
    void swim();
    void run();
}

// 狗不会飞，但被迫实现fly方法
public class Dog implements Animal {
    @Override
    public void eat() { System.out.println("狗吃东西"); }

    @Override
    public void fly() {
        throw new UnsupportedOperationException("狗不会飞！"); // 被迫实现不需要的方法
    }

    @Override
    public void swim() { System.out.println("狗游泳"); }

    @Override
    public void run() { System.out.println("狗跑步"); }
}

// ✅ 遵循ISP的示例：接口拆分
public interface Eatable { void eat(); }
public interface Flyable { void fly(); }
public interface Swimmable { void swim(); }
public interface Runnable { void run(); }

// 狗只需要实现它支持的行为
public class Dog implements Eatable, Swimmable, Runnable {
    @Override
    public void eat() { System.out.println("狗吃东西"); }

    @Override
    public void swim() { System.out.println("狗游泳"); }

    @Override
    public void run() { System.out.println("狗跑步"); }
}

public class Bird implements Eatable, Flyable {
    @Override
    public void eat() { System.out.println("鸟吃东西"); }

    @Override
    public void fly() { System.out.println("鸟飞翔"); }
}
```

**UML 类图（文字描述）：**
```
Animal（臃肿接口）
  ├── Dog（被迫实现fly，不合理）
  └── Bird

拆分后：
Eatable    Flyable    Swimmable    Runnable
  │          │          │            │
  ├── Dog ───┤── Dog ───┤── Dog ─────┤
  ├── Bird ──┤
  └── Bird ──┘
```

**React/前端对比：**
ISP在React中体现为Props的精简设计。不要把所有可能的props都塞进一个组件，而是让每个组件只接收它需要的props：
```jsx
// ❌ 违反ISP：传入大量不需要的props
<UserCard user={user} onEdit={onEdit} onDelete={onDelete} onBan={onBan} onExport={onExport} />

// ✅ 遵循ISP：拆分为更具体的组件
<UserCard user={user} />
<UserActions onEdit={onEdit} onDelete={onDelete} />
<AdminActions onBan={onBan} onExport={onExport} />
```

> ⭐ **面试高频问题：**
> - Q: 接口隔离原则和单一职责原则有什么区别？
> - A: SRP关注的是类的职责划分（一个类只做一件事），ISP关注的是接口的设计（接口要小而专）。SRP是从实现者的角度出发，ISP是从调用者的角度出发。一个类可能只负责一件事（符合SRP），但它对外暴露的接口可能仍然太大（违反ISP）。

### 1.5 依赖倒置原则（DIP - Dependency Inversion Principle）

**定义：**
1. 高层模块不应该依赖低层模块，二者都应该依赖其抽象。
2. 抽象不应该依赖细节，细节应该依赖抽象。

**核心思想：** 面向接口编程，而不是面向实现编程。这是Spring框架IoC（控制反转）和DI（依赖注入）的理论基础。

```java
// ❌ 违反DIP的示例：高层直接依赖低层实现
public class UserService {
    // 直接依赖具体实现 —— 耦合！
    private MySQLUserDao userDao = new MySQLUserDao();

    public User getUser(Long id) {
        return userDao.findById(id);
    }
}

// 如果要切换到Oracle，必须修改UserService —— 违反OCP

// ✅ 遵循DIP的示例：依赖抽象
// 抽象层
public interface UserDao {
    User findById(Long id);
}

// 低层模块
@Repository
public class MySQLUserDao implements UserDao {
    @Override
    public User findById(Long id) {
        // MySQL实现
        return jdbcTemplate.queryForObject("SELECT * FROM users WHERE id = ?", ...);
    }
}

// 高层模块依赖抽象
@Service
public class UserService {
    private final UserDao userDao;

    // 通过构造器注入依赖（Spring自动注入）
    public UserService(UserDao userDao) {
        this.userDao = userDao;
    }

    public User getUser(Long id) {
        return userDao.findById(id);
    }
}
```

**UML 类图（文字描述）：**
```
违反DIP：                     遵循DIP：
UserService                   UserService
    │                              │
    └── MySQLUserDao              └──<<interface>> UserDao
                                       ├── MySQLUserDao
                                       └── OracleUserDao
```

**React/前端对比：**
DIP在前端体现为依赖注入和抽象层。比如数据获取层不应该直接依赖具体的API实现，而是通过抽象的服务层：
```jsx
// ❌ 直接依赖具体实现
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  useEffect(() => {
    fetch(`/api/users/${userId}`).then(r => r.json()).then(setUser);
  }, [userId]);
}

// ✅ 依赖抽象（自定义Hook作为抽象层）
function useUserService() {
  const getUser = async (id) => {
    const response = await fetch(`/api/users/${id}`);
    return response.json();
  };
  return { getUser };
}
```

> ⭐ **面试高频问题：**
> - Q: 依赖倒置原则和Spring的IoC有什么关系？
> - A: DIP是IoC的理论基础。Spring通过IoC容器管理Bean的生命周期和依赖关系，实现了"高层模块不直接创建低层模块的对象，而是由容器注入"。这样高层模块只需要声明依赖的接口，具体的实现由容器在运行时决定。
> - Q: 依赖倒置和依赖注入是一回事吗？
> - A: 不是。依赖倒置是一种设计原则（依赖抽象而非具体），依赖注入是实现依赖倒置的一种手段。除了DI，还可以通过Service Locator、工厂模式等方式实现依赖倒置。

### 1.6 迪米特法则（LoD - Law of Demeter）

**定义：** 一个对象应该对其他对象保持最少的了解。又叫"最少知道原则"（Principle of Least Knowledge）。

**核心思想：** 只与直接的朋友通信，不与陌生人说话。降低类之间的耦合度。

```java
// ❌ 违反LoD的示例：链式调用跨越了多个对象
public class School {
    private List<Classroom> classrooms;

    public List<Classroom> getClassrooms() { return classrooms; }
}

public class Classroom {
    private Teacher teacher;
    public Teacher getTeacher() { return teacher; }
}

public class Teacher {
    private String name;
    public String getName() { return name; }
}

// 调用方需要了解School、Classroom、Teacher的内部结构
public class Demo {
    public void printTeacherName(School school) {
        // 链式调用 —— 违反LoD！
        String name = school.getClassrooms().get(0).getTeacher().getName();
        System.out.println(name);
    }
}

// ✅ 遵循LoD的示例：封装内部细节
public class School {
    private List<Classroom> classrooms;

    // 提供封装好的方法，隐藏内部结构
    public String getFirstTeacherName() {
        if (classrooms == null || classrooms.isEmpty()) {
            return null;
        }
        Classroom firstClass = classrooms.get(0);
        return firstClass.getTeacherName();
    }
}

// 调用方只需要和School交互
public class Demo {
    public void printTeacherName(School school) {
        String name = school.getFirstTeacherName(); // ✅ 只与直接朋友通信
        System.out.println(name);
    }
}
```

> ⭐ **面试高频问题：**
> - Q: 迪米特法则在实际开发中如何把握度？
> - A: LoD的核心是减少不必要的耦合。但过度应用会导致大量的"转发方法"（wrapper），增加代码量。实际开发中，对于稳定的、变化不频繁的内部结构，适度的链式调用是可以接受的。关键是不要让外部代码依赖你内部的实现细节。

### 1.7 合成复用原则（CRP - Composite Reuse Principle）

**定义：** 优先使用对象组合（has-a），而不是类继承（is-a）。

**核心思想：** 继承是一种强耦合关系，子类与父类紧密绑定。组合是一种松耦合关系，可以在运行时动态改变。GoF设计模式中也提到："优先使用对象组合，而非类继承"。

```java
// ❌ 过度使用继承
public class Car extends Engine { } // 逻辑上不合理：Car不是Engine

// ✅ 使用组合
public class Car {
    private Engine engine; // Car has-a Engine

    public Car(Engine engine) {
        this.engine = engine;
    }

    public void start() {
        engine.start();
    }
}
```

**React/前端对比：**
React官方文档明确推荐组合优于继承。React中几乎没有使用继承的场景，全部通过组合、render props、自定义Hook来实现代码复用：
```jsx
// ❌ React不推荐继承
class EnhancedButton extends Button { }

// ✅ 推荐组合 + HOC/render props/custom hooks
function withAuth(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const isAuthenticated = useAuth();
    return isAuthenticated ? <WrappedComponent {...props} /> : <LoginPrompt />;
  };
}
```

### 1.8 KISS 原则（Keep It Simple, Stupid）

**定义：** 保持简单直白，不要过度设计。

**核心思想：** 大多数系统不需要复杂的设计。简单的代码更容易理解、测试和维护。

```java
// ❌ 过度设计
public interface UserOperationStrategyFactory {
    UserOperationStrategy createStrategy(OperationType type);
}
// 对于简单的CRUD操作，引入策略工厂可能是过度设计

// ✅ 简单直接
public class UserService {
    private final UserDao userDao;
    public UserService(UserDao userDao) { this.userDao = userDao; }
    public User findById(Long id) { return userDao.findById(id); }
    public void save(User user) { userDao.insert(user); }
}
```

### 1.9 DRY 原则（Don't Repeat Yourself）

**定义：** 每一项知识在系统中必须有单一、明确、权威的表示。

**核心思想：** 避免重复代码。重复代码意味着当需要修改时，必须在多个地方同步修改，容易遗漏。

```java
// ❌ 重复代码
public class OrderService {
    public double calculateTotal(List<OrderItem> items) {
        double total = 0;
        for (OrderItem item : items) {
            total += item.getPrice() * item.getQuantity();
        }
        return total;
    }
}

public class InvoiceService {
    public double calculateTotal(List<OrderItem> items) {
        double total = 0;
        for (OrderItem item : items) {
            total += item.getPrice() * item.getQuantity(); // 重复！
        }
        return total;
    }
}

// ✅ 提取公共方法
public class OrderUtils {
    public static double calculateTotal(List<OrderItem> items) {
        return items.stream()
            .mapToDouble(item -> item.getPrice() * item.getQuantity())
            .sum();
    }
}
```

### 1.10 YAGNI 原则（You Aren't Gonna Need It）

**定义：** 不要为将来可能的需求提前设计，只实现当前需要的功能。

**核心思想：** 过度设计是浪费。当需求真的出现时再重构也不迟。


## 二、创建型模式

创建型模式关注**对象的创建方式**，将对象的创建与使用分离，降低系统的耦合度。

### 2.1 单例模式（重点！）

**定义：** 确保一个类只有一个实例，并提供一个全局访问点。

**适用场景：**
- 全局配置管理器
- 数据库连接池
- 线程池
- 日志记录器
- Spring中的Bean（默认单例）

#### 2.1.1 饿汉式（线程安全）

```java
/**
 * 饿汉式单例 —— 类加载时就创建实例
 * 优点：线程安全（由ClassLoader保证）
 * 缺点：不管用不用都会创建，可能浪费内存
 */
public class HungrySingleton {
    // 1. 私有构造方法，防止外部new
    private HungrySingleton() {}

    // 2. 类加载时就创建实例（static final保证线程安全）
    private static final HungrySingleton INSTANCE = new HungrySingleton();

    // 3. 提供全局访问点
    public static HungrySingleton getInstance() {
        return INSTANCE;
    }
}
```

#### 2.1.2 懒汉式（线程不安全 → synchronized → DCL双重检查锁）

**版本1：线程不安全（不要使用！）**
```java
public class LazySingletonUnsafe {
    private static LazySingletonUnsafe instance;

    private LazySingletonUnsafe() {}

    public static LazySingletonUnsafe getInstance() {
        if (instance == null) { // 多线程可能同时进入
            instance = new LazySingletonUnsafe(); // 导致创建多个实例
        }
        return instance;
    }
}
```

**版本2：synchronized 方法（线程安全但性能差）**
```java
public class LazySingletonSync {
    private static LazySingletonSync instance;

    private LazySingletonSync() {}

    // 整个方法加锁 —— 每次调用都要获取锁，性能差
    public synchronized static LazySingletonSync getInstance() {
        if (instance == null) {
            instance = new LazySingletonSync();
        }
        return instance;
    }
}
```

**版本3：DCL 双重检查锁（推荐！）**
```java
/**
 * DCL双重检查锁单例
 * 关键点：
 * 1. volatile 防止指令重排序
 * 2. 两次null检查减少锁竞争
 * 3. synchronized 只保护实例化过程
 */
public class DCLSingleton {
    /**
     * volatile 关键字非常重要！
     * new DCLSingleton() 实际分为3步：
     *   1. 分配内存空间
     *   2. 初始化对象
     *   3. 将引用指向内存地址
     * JVM可能将步骤2和3重排序为 1→3→2
     * 如果没有volatile，线程A执行到3但未执行2时，线程B判断instance不为null直接返回，使用了一个未初始化的对象！
     */
    private static volatile DCLSingleton instance;

    private DCLSingleton() {
        // 防止反射攻击
        if (instance != null) {
            throw new RuntimeException("请使用getInstance()获取实例");
        }
    }

    public static DCLSingleton getInstance() {
        // 第一次检查：避免不必要的加锁（性能优化）
        if (instance == null) {
            synchronized (DCLSingleton.class) {
                // 第二次检查：防止多个线程同时通过第一次检查
                if (instance == null) {
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}
```

**DCL 执行流程图（文字描述）：**
```
线程A                              线程B
  │                                  │
  ├─ 第一次检查 instance==null? ──→ ├─ 第一次检查 instance==null?
  │  YES                             │  YES
  ├─ 获取锁                          │  等待锁...
  ├─ 第二次检查 instance==null?      │
  │  YES                             │
  ├─ new DCLSingleton()              │
  ├─ 释放锁 ──────────────────────→ ├─ 获取锁
  │                                  ├─ 第二次检查 instance!=null
  │                                  ├─ 释放锁
  │                                  ├─ 返回已创建的实例
```

#### 2.1.3 静态内部类实现

```java
/**
 * 静态内部类单例 —— 利用类加载机制保证线程安全
 * 优点：延迟加载 + 线程安全 + 无锁，性能优于DCL
 * 原理：外部类加载时不会立即加载内部类，只有调用getInstance()时才会加载
 */
public class InnerClassSingleton {
    private InnerClassSingleton() {}

    // 静态内部类在外部类加载时不会被加载
    // 只有调用 getInstance() 时才会触发内部类的加载和初始化
    private static class Holder {
        // JVM保证类加载过程的线程安全性
        private static final InnerClassSingleton INSTANCE = new InnerClassSingleton();
    }

    public static InnerClassSingleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

**类加载时序（文字描述）：**
```
1. Class.forName("InnerClassSingleton") → 只加载外部类，不加载Holder
2. InnerClassSingleton.getInstance() → 触发Holder类的加载
3. Holder类加载 → 初始化INSTANCE（JVM保证线程安全）
4. 返回INSTANCE
```

#### 2.1.4 枚举实现（最佳实践）

```java
/**
 * 枚举单例 —— 《Effective Java》作者Joshua Bloch推荐
 * 优点：
 *   1. 天然线程安全（枚举实例创建由JVM保证）
 *   2. 天然防止反射攻击（反射创建枚举会抛IllegalArgumentException）
 *   3. 天然防止序列化破坏（枚举的readResolve()由JVM实现）
 */
public enum EnumSingleton {
    INSTANCE;

    private AtomicLong counter = new AtomicLong(0);

    public long increment() {
        return counter.incrementAndGet();
    }
}

// 使用
public class Demo {
    public static void main(String[] args) {
        long count = EnumSingleton.INSTANCE.increment();
        System.out.println(count); // 1
    }
}
```

#### 2.1.5 单例模式的破坏与防御

**破坏方式1：反射**
```java
// 反射可以调用私有构造方法，破坏单例
public class ReflectionAttack {
    public static void main(String[] args) throws Exception {
        Constructor<DCLSingleton> constructor =
            DCLSingleton.class.getDeclaredConstructor();
        constructor.setAccessible(true); // 破坏私有访问限制

        DCLSingleton instance1 = DCLSingleton.getInstance();
        DCLSingleton instance2 = constructor.newInstance(); // 创建了新实例！

        System.out.println(instance1 == instance2); // false！单例被破坏
    }
}

// 防御：在私有构造方法中检查
private DCLSingleton() {
    if (instance != null) {
        throw new RuntimeException("不允许通过反射创建实例");
    }
}
// 注意：这种防御对饿汉式有效（因为instance在构造前已初始化），对懒汉式无效
// 枚举单例天然免疫反射攻击
```

**破坏方式2：序列化**
```java
// 序列化再反序列化会创建新对象
public class SerializationAttack {
    public static void main(String[] args) throws Exception {
        DCLSingleton instance1 = DCLSingleton.getInstance();

        // 序列化
        ByteArrayOutputStream bos = new ByteArrayOutputStream();
        ObjectOutputStream oos = new ObjectOutputStream(bos);
        oos.writeObject(instance1);

        // 反序列化
        ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
        ObjectInputStream ois = new ObjectInputStream(bis);
        DCLSingleton instance2 = (DCLSingleton) ois.readObject();

        System.out.println(instance1 == instance2); // false！单例被破坏
    }
}

// 防御：实现readResolve方法
public class DCLSingleton implements Serializable {
    private static volatile DCLSingleton instance;

    private DCLSingleton() {}

    public static DCLSingleton getInstance() {
        if (instance == null) {
            synchronized (DCLSingleton.class) {
                if (instance == null) {
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }

    // 反序列化时JVM会调用此方法，返回已有实例而非创建新对象
    private Object readResolve() {
        return instance;
    }
}
// 枚举单例天然免疫序列化攻击
```

#### 2.1.6 Spring 中的单例

> ⭐ **面试高频问题：**
> - Q: 单例模式有几种实现方式？推荐哪种？
> - A: 常见的有饿汉式、懒汉式（synchronized）、DCL双重检查锁、静态内部类、枚举。推荐枚举方式（Effective Java推荐），如果需要延迟加载且不能用枚举，推荐静态内部类方式。DCL在实际项目中也很常用，但要注意volatile关键字。
> - Q: 为什么DCL需要volatile？
> - A: `new Object()` 不是原子操作，分为：分配内存→初始化对象→引用赋值。JVM可能将步骤2和3重排序。如果没有volatile，线程A执行到步骤3但未执行步骤2时，线程B可能看到instance不为null，直接返回一个未初始化的对象。volatile通过内存屏障禁止指令重排序。
> - Q: Spring的单例和Java单例有什么区别？
> - A: Java单例是ClassLoader级别的，一个JVM中只有一个实例。Spring单例是IoC容器级别的，一个ApplicationContext中只有一个实例。如果有多个ApplicationContext，同一个Bean会有多个实例。此外，Spring单例不保证在反序列化或反射攻击下的安全性。

> ⭐ **面试高频问题：**
> - Q: 简单工厂、工厂方法、抽象工厂的区别？
> - A: 简单工厂由一个工厂类根据参数决定创建哪种产品（违反OCP）；工厂方法定义工厂接口，每个产品对应一个工厂类（符合OCP，但工厂类数量多）；抽象工厂用于创建产品族（一组相关产品），每个工厂可以创建多种产品。简单工厂→工厂方法→抽象工厂，复杂度递增，适用场景不同。
> - Q: Spring的BeanFactory和FactoryBean有什么区别？
> - A: BeanFactory是Spring IoC容器的基础接口，负责管理和获取Bean实例。FactoryBean是一个特殊的Bean接口，用于自定义Bean的创建逻辑。当调用getBean("myFactoryBean")时，如果myFactoryBean实现了FactoryBean接口，返回的是getObject()的结果，而不是FactoryBean本身。要获取FactoryBean本身，需要在名称前加"&"：getBean("&myFactoryBean")。

### 2.3 建造者模式

**定义：** 将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。

**适用场景：**
- 对象有很多可选参数（替代多参数构造器的telescoping constructor问题）
- 需要创建不可变对象
- 构建过程需要多个步骤

```java
/**
 * 建造者模式 —— 手动实现
 */
public class HttpRequest {
    private final String url;
    private final String method;
    private final Map<String, String> headers;
    private final String body;
    private final int timeout;

    // 私有构造器，只能通过Builder创建
    private HttpRequest(Builder builder) {
        this.url = builder.url;
        this.method = builder.method;
        this.headers = Collections.unmodifiableMap(builder.headers);
        this.body = builder.body;
        this.timeout = builder.timeout;
    }

    // Getter方法...

    public static class Builder {
        // 必填参数
        private final String url;
        // 可选参数（有默认值）
        private String method = "GET";
        private Map<String, String> headers = new HashMap<>();
        private String body = null;
        private int timeout = 30000;

        public Builder(String url) {
            this.url = Objects.requireNonNull(url, "URL不能为空");
        }

        public Builder method(String method) {
            this.method = method;
            return this;
        }

        public Builder addHeader(String key, String value) {
            this.headers.put(key, value);
            return this;
        }

        public Builder body(String body) {
            this.body = body;
            return this;
        }

        public Builder timeout(int timeout) {
            this.timeout = timeout;
            return this;
        }

        public HttpRequest build() {
            return new HttpRequest(this);
        }
    }
}

// 使用
public class Demo {
    public static void main(String[] args) {
        HttpRequest request = new HttpRequest.Builder("https://api.example.com/users")
            .method("POST")
            .addHeader("Content-Type", "application/json")
            .addHeader("Authorization", "Bearer token123")
            .body("{\"name\":\"张三\"}")
            .timeout(5000)
            .build();
    }
}
```

```java
    // append方法返回this，支持链式调用
    @Override
    public StringBuilder append(String str) {
        super.append(str);
        return this; // 返回this，实现链式调用
    }

    @Override
    public StringBuilder append(char c) {
        super.append(c);
        return this;
    }

    // 最终构建出String
    @Override
    public String toString() {
        return new String(value, 0, count);
    }
}

// 使用
String result = new StringBuilder()
    .append("Hello")
    .append(", ")
    .append("World")
    .append("!")
    .toString(); // "Hello, World!"
```

> ⭐ **面试高频问题：**
> - Q: 建造者模式和工厂模式有什么区别？
> - A: 工厂模式关注"创建什么类型的对象"，建造者模式关注"如何一步步构建一个复杂对象"。工厂模式通常一步就能创建对象，建造者模式需要多步构建。工厂模式返回不同类型的产品，建造者模式返回同一类型但配置不同的产品。
> - Q: 什么时候用建造者模式？
> - A: 当一个对象有很多参数（特别是可选参数）时，使用建造者模式比telescoping constructor更清晰。Effective Java推荐当构造器参数超过4个时考虑使用建造者模式。

### 2.4 原型模式

**定义：** 用原型实例指定创建对象的种类，并且通过拷贝这些原型创建新的对象。

**核心思想：** 通过克隆已有对象来创建新对象，而不是通过new。适用于创建对象成本较高的场景。

```java
/**
 * 浅拷贝 vs 深拷贝
 */
public class ShallowCopyExample implements Cloneable {
    private String name;
    private int[] scores; // 引用类型

    @Override
    protected ShallowCopyExample clone() throws CloneNotSupportedException {
        return (ShallowCopyExample) super.clone(); // 浅拷贝
    }

    // 浅拷贝问题：scores数组是引用拷贝，修改克隆对象的scores会影响原对象
}

public class DeepCopyExample implements Cloneable {
    private String name;
    private int[] scores;

    @Override
    protected DeepCopyExample clone() throws CloneNotSupportedException {
        DeepCopyExample cloned = (DeepCopyExample) super.clone();
        // 深拷贝：手动克隆引用类型字段
        cloned.scores = this.scores.clone();
        return cloned;
    }
}

// 更推荐的深拷贝方式：序列化
public class DeepCopyViaSerialization {
    @SuppressWarnings("unchecked")
    public static <T extends Serializable> T deepCopy(T object) {
        try {
            ByteArrayOutputStream bos = new ByteArrayOutputStream();
            ObjectOutputStream oos = new ObjectOutputStream(bos);
            oos.writeObject(object);

            ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
            ObjectInputStream ois = new ObjectInputStream(bis);
            return (T) ois.readObject();
        } catch (Exception e) {
            throw new RuntimeException("深拷贝失败", e);
        }
    }
}
```

**浅拷贝 vs 深拷贝对比：**
```
浅拷贝：
  原对象: { name: "张三", scores: [90, 85, 92] }
  克隆对象: { name: "张三", scores: [90, 85, 92] }  ← scores指向同一个数组！
  修改克隆对象.scores[0] = 100 → 原对象.scores[0] 也变成100！

深拷贝：
  原对象: { name: "张三", scores: [90, 85, 92] }
  克隆对象: { name: "张三", scores: [90, 85, 92] }  ← scores是独立的数组
  修改克隆对象.scores[0] = 100 → 原对象不受影响
```

> ⭐ **面试高频问题：**
> - Q: 浅拷贝和深拷贝的区别？
> - A: 浅拷贝只复制基本类型字段和引用类型的引用地址，引用类型指向的对象仍然是共享的。深拷贝会递归复制所有引用类型的对象，创建完全独立的副本。Object.clone()默认是浅拷贝，要实现深拷贝需要手动处理每个引用类型字段，或使用序列化方式。

## 三、结构型模式

结构型模式关注**类和对象的组合方式**，通过继承和组合来获得新的功能。

### 3.1 代理模式（重点！）

**定义：** 为其他对象提供一种代理以控制对这个对象的访问。

#### 3.1.1 静态代理

```java
/**
 * 静态代理：代理类和目标类实现相同的接口，代理类内部持有目标类的引用
 * 缺点：一个接口需要写一个代理类，接口方法增加时代理类也要修改
 */
// 公共接口
public interface UserDao {
    void save(User user);
    void delete(Long id);
}

// 目标对象
@Repository
public class UserDaoImpl implements UserDao {
    @Override
    public void save(User user) {
        System.out.println("保存用户: " + user.getName());
    }

    @Override
    public void delete(Long id) {
        System.out.println("删除用户: " + id);
    }
}

// 静态代理对象
public class UserDaoProxy implements UserDao {
    private final UserDao target; // 持有目标对象的引用

    public UserDaoProxy(UserDao target) {
        this.target = target;
    }

    @Override
    public void save(User user) {
        System.out.println("[代理] 开始事务");
        try {
            target.save(user);
            System.out.println("[代理] 提交事务");
        } catch (Exception e) {
            System.out.println("[代理] 回滚事务");
            throw e;
        }
    }

    @Override
    public void delete(Long id) {
        System.out.println("[代理] 开始事务");
        try {
            target.delete(id);
            System.out.println("[代理] 提交事务");
        } catch (Exception e) {
            System.out.println("[代理] 回滚事务");
            throw e;
        }
    }
}
```

#### 3.1.2 JDK 动态代理（Proxy + InvocationHandler）

```java
/**
 * JDK动态代理：基于接口的代理，通过反射在运行时动态生成代理类
 * 核心类：Proxy、InvocationHandler
 */
import java.lang.reflect.InvocationHandler;
import java.lang.reflect.Method;
import java.lang.reflect.Proxy;

// 1. 定义接口
public interface UserService {
    void save(User user);
    void delete(Long id);
    String findById(Long id);
}

// 2. 目标实现
@Service
public class UserServiceImpl implements UserService {
    @Override
    public void save(User user) {
        System.out.println("保存用户: " + user.getName());
    }

    @Override
    public void delete(Long id) {
        System.out.println("删除用户: " + id);
    }

    @Override
    public String findById(Long id) {
        System.out.println("查询用户: " + id);
        return "User_" + id;
    }
}

// 3. 通用 InvocationHandler
public class TransactionInvocationHandler implements InvocationHandler {
    private final Object target; // 目标对象

    public TransactionInvocationHandler(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("[JDK动态代理] 开始事务");

        try {
            // 调用目标方法
            Object result = method.invoke(target, args);
            System.out.println("[JDK动态代理] 提交事务");
            return result;
        } catch (Exception e) {
            System.out.println("[JDK动态代理] 回滚事务");
            throw e;
        }
    }
}

// 4. 创建代理对象
public class JdkProxyDemo {
    public static void main(String[] args) {
        UserService target = new UserServiceImpl();

        // 创建代理对象
        // 参数1：类加载器  参数2：目标实现的接口数组  参数3：InvocationHandler
        UserService proxy = (UserService) Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            target.getClass().getInterfaces(),
            new TransactionInvocationHandler(target)
        );

        // 通过代理调用方法
        proxy.save(new User("张三"));
        // 输出：
        // [JDK动态代理] 开始事务
        // 保存用户: 张三
        // [JDK动态代理] 提交事务
    }
}
```

**JDK动态代理原理（文字描述）：**
```
Proxy.newProxyInstance() 做了什么？
1. 生成代理类的字节码（$Proxy0.class）
2. 加载生成的字节码（defineClass）
3. 创建代理类实例（传入InvocationHandler）

生成的代理类大致结构：
public class $Proxy0 extends Proxy implements UserService {
    public $Proxy0(InvocationHandler h) {
        super(h);
    }

    public void save(User user) {
        // 构造Method对象
        Method m = UserService.class.getMethod("save", User.class);
        // 调用InvocationHandler的invoke方法
        super.h.invoke(this, m, new Object[]{user});
    }

    public void delete(Long id) {
        Method m = UserService.class.getMethod("delete", Long.class);
        super.h.invoke(this, m, new Object[]{id});
    }
}
```

#### 3.1.3 CGLIB 动态代理（Enhancer + MethodInterceptor）

```java
/**
 * CGLIB动态代理：基于继承的代理，通过生成目标类的子类来实现代理
 * 不要求目标类实现接口
 * 核心类：Enhancer、MethodInterceptor
 */
import net.sf.cglib.proxy.Enhancer;
import net.sf.cglib.proxy.MethodInterceptor;
import net.sf.cglib.proxy.MethodProxy;

// 目标类（不需要实现接口）
@Service
public class OrderService {
    public void createOrder(String orderId) {
        System.out.println("创建订单: " + orderId);
    }

    public void cancelOrder(String orderId) {
        System.out.println("取消订单: " + orderId);
    }
}

// MethodInterceptor
public class LogMethodInterceptor implements MethodInterceptor {
    @Override
    public Object intercept(Object obj, java.lang.reflect.Method method,
                            Object[] args, MethodProxy proxy) throws Throwable {
        System.out.println("[CGLIB代理] 方法 " + method.getName() + " 开始执行");

        long startTime = System.currentTimeMillis();

        // 调用父类（目标类）的方法
        // 注意：使用proxy.invokeSuper而不是method.invoke
        Object result = proxy.invokeSuper(obj, args);

        long endTime = System.currentTimeMillis();
        System.out.println("[CGLIB代理] 方法 " + method.getName()
            + " 执行完毕，耗时: " + (endTime - startTime) + "ms");

        return result;
    }
}

// 创建CGLIB代理
public class CglibProxyDemo {
    public static void main(String[] args) {
        Enhancer enhancer = new Enhancer();
        // 设置目标类为父类（CGLIB通过继承实现代理）
        enhancer.setSuperclass(OrderService.class);
        // 设置回调
        enhancer.setCallback(new LogMethodInterceptor());

        // 创建代理对象
        OrderService proxy = (OrderService) enhancer.create();

        proxy.createOrder("ORD-001");
        // 输出：
        // [CGLIB代理] 方法 createOrder 开始执行
        // 创建订单: ORD-001
        // [CGLIB代理] 方法 createOrder 执行完毕，耗时: 5ms
    }
}
```

#### 3.1.4 JDK 代理 vs CGLIB 代理对比

| 对比维度 | JDK 动态代理 | CGLIB 动态代理 |
|---------|-------------|---------------|
| **实现方式** | 基于接口（实现InvocationHandler） | 基于继承（生成子类） |
| **要求** | 目标类必须实现接口 | 目标类不能是final类 |
| **性能** | JDK 8后性能已大幅提升 | 生成子类有一定开销，但调用速度快 |
| **原理** | 反射调用 | FastClass机制（非反射） |
| **依赖** | JDK内置，无需额外依赖 | 需要cglib库 |
| **Spring AOP** | 目标有接口时默认使用 | 目标无接口时使用 |

#### 3.1.5 Spring AOP 中的代理选择

```java
/**
 * Spring AOP 代理选择策略：
 * 1. 如果目标类实现了接口 → 默认使用JDK动态代理
 * 2. 如果目标类没有实现接口 → 使用CGLIB代理
 * 3. 可以通过配置强制使用CGLIB
 */
@Configuration
@EnableAspectJAutoProxy(proxyTargetClass = true) // 强制使用CGLIB
public class AopConfig {
}

// Spring Boot 2.x 之后默认使用CGLIB代理
// application.properties:
// spring.aop.proxy-target-class=true（默认值）

// Spring AOP 代理的底层实现
// org.springframework.aop.framework.ProxyFactory
public class ProxyFactory {
    // 决定使用哪种代理
    public AopProxy createAopProxy(AdvisedSupport config) {
        if (config.isOptimize() || config.isProxyTargetClass()
            || hasNoUserSuppliedProxyInterfaces(config)) {
            return new CglibAopProxy(config); // CGLIB
        } else {
            return new JdkDynamicAopProxy(config); // JDK
        }
    }
}
```

```
JDK动态代理：
  <<interface>> UserService
    ├── UserServiceImpl（目标类）
    └── $Proxy0（动态生成的代理类，实现UserService接口）
          └── 持有 InvocationHandler → 调用 UserServiceImpl

CGLIB动态代理：
```

> ⭐ **面试高频问题：**
> - Q: JDK动态代理和CGLIB动态代理的区别？
> - A: JDK动态代理基于接口，通过Proxy.newProxyInstance()生成实现相同接口的代理类，通过InvocationHandler.invoke()调用目标方法。CGLIB基于继承，通过Enhancer生成目标类的子类，通过MethodInterceptor.intercept()调用父类方法。JDK代理要求目标类实现接口，CGLIB不能代理final类和final方法。Spring AOP中默认优先使用JDK代理，Spring Boot 2.x后默认使用CGLIB。
> - Q: Spring AOP的代理机制是什么？
> - A: Spring AOP基于动态代理实现。如果目标类实现了接口，使用JDK动态代理；如果没有实现接口，使用CGLIB。Spring Boot 2.x之后默认使用CGLIB。代理对象在Bean初始化后阶段创建（AbstractAutoProxyCreator.postProcessAfterInitialization）。

### 3.2 适配器模式

**定义：** 将一个类的接口转换成客户端期望的另一个接口。适配器模式使得原本由于接口不兼容而不能一起工作的类可以一起工作。

```java
/**
 * 类适配器：通过继承来适配
 */
// 目标接口（客户端期望的接口）
public interface Target {
    void request();
}

// 被适配者（已有的接口，不兼容）
public class Adaptee {
    public void specificRequest() {
        System.out.println("被适配者的特殊请求");
    }
}

// 类适配器：继承被适配者，实现目标接口
public class ClassAdapter extends Adaptee implements Target {
    @Override
    public void request() {
        super.specificRequest(); // 调用被适配者的方法
    }
}

/**
 * 对象适配器：通过组合来适配（推荐）
 */
public class ObjectAdapter implements Target {
    private final Adaptee adaptee; // 持有被适配者的引用

    public ObjectAdapter(Adaptee adaptee) {
        this.adaptee = adaptee;
    }

    @Override
    public void request() {
        adaptee.specificRequest(); // 委托给被适配者
    }
}
```

**Spring MVC 中的 HandlerAdapter**
```java
/**
 * Spring MVC 中的 HandlerAdapter 是适配器模式的经典应用
 * 不同的Controller类型（Controller、HttpRequestHandler、@RequestMapping等）
 * 有不同的处理方法签名，HandlerAdapter将它们适配为统一的处理流程
 */
// HandlerAdapter 接口
public interface HandlerAdapter {
    boolean supports(Object handler);
    ModelAndView handle(HttpServletRequest request,
                        HttpServletResponse response,
                        Object handler) throws Exception;
}

// 适配不同类型的Handler
public class RequestMappingHandlerAdapter implements HandlerAdapter {
    @Override
    public boolean supports(Object handler) {
        return handler instanceof HandlerMethod; // @RequestMapping标注的方法
    }

    @Override
    public ModelAndView handle(HttpServletRequest request,
                                HttpServletResponse response,
                                Object handler) throws Exception {
        // 处理 @RequestMapping 标注的方法
        return ((HandlerMethod) handler).invokeAndHandle(request, response);
    }
}

// DispatcherServlet 通过 HandlerAdapter 调用各种类型的Handler
public class DispatcherServlet {
    private List<HandlerAdapter> handlerAdapters;

    protected ModelAndView handle(HttpServletRequest request, ...) {
        Object handler = getHandler(request);
        // 遍历找到支持该handler的适配器
        for (HandlerAdapter adapter : handlerAdapters) {
            if (adapter.supports(handler)) {
                return adapter.handle(request, response, handler);
            }
        }
        throw new ServletException("No adapter for handler");
    }
}
```

```
Spring MVC:
```

> - Q: 类适配器和对象适配器有什么区别？
> - A: 类适配器通过继承被适配者来实现适配，Java单继承的限制导致只能适配一个类。对象适配器通过组合（持有被适配者的引用）来实现适配，更灵活，可以适配被适配者的子类。实际开发中推荐使用对象适配器。

### 3.3 装饰器模式

**定义：** 动态地给一个对象添加一些额外的职责。就增加功能来说，装饰器模式相比生成子类更为灵活。

```java
/**
 * 装饰器模式的核心：装饰器和被装饰对象实现相同接口
 * 装饰器内部持有被装饰对象的引用，在调用前后添加额外行为
 */
// 组件接口
public interface DataSource {
    Connection getConnection();
}

// 具体组件
public class BasicDataSource implements DataSource {
    @Override
    public Connection getConnection() {
        System.out.println("获取基本数据库连接");
        return new Connection();
    }
}

// 装饰器基类
public class DataSourceDecorator implements DataSource {
    protected final DataSource wrapped;

    public DataSourceDecorator(DataSource wrapped) {
        this.wrapped = wrapped;
    }

    @Override
    public Connection getConnection() {
        return wrapped.getConnection();
    }
}

// 具体装饰器1：连接池
public class PooledDataSource extends DataSourceDecorator {
    public PooledDataSource(DataSource wrapped) {
        super(wrapped);
    }

    @Override
    public Connection getConnection() {
        System.out.println("[连接池] 从池中获取连接");
        return super.getConnection();
    }
}

// 具体装饰器2：日志
public class LoggingDataSource extends DataSourceDecorator {
    public LoggingDataSource(DataSource wrapped) {
        super(wrapped);
    }

    @Override
    public Connection getConnection() {
        System.out.println("[日志] 获取连接前");
        Connection conn = super.getConnection();
        System.out.println("[日志] 获取连接后");
        return conn;
    }
}

// 使用：可以自由组合装饰器
public class Demo {
    public static void main(String[] args) {
        DataSource ds = new BasicDataSource();
        ds = new PooledDataSource(ds);       // 添加连接池功能
        ds = new LoggingDataSource(ds);       // 添加日志功能
        ds.getConnection();
        // 输出：
        // [日志] 获取连接前
        // [连接池] 从池中获取连接
        // 获取基本数据库连接
        // [日志] 获取连接后
    }
}
```

**Java IO 中的装饰器模式**
```java
/**
 * Java IO 是装饰器模式的经典应用
 * InputStream 是抽象组件
 * FileInputStream 是具体组件
 * BufferedInputStream、DataInputStream 等是装饰器
 */
// 装饰器链
InputStream is = new BufferedInputStream(       // 添加缓冲功能
    new DataInputStream(                         // 添加数据类型读取功能
        new FileInputStream("data.txt")          // 基本文件读取
    )
);

// 类结构：
// InputStream（抽象组件）
//   ├── FileInputStream（具体组件）
//   ├── FilterInputStream（装饰器基类）
//       ├── BufferedInputStream（缓冲装饰器）
//       ├── DataInputStream（数据类型装饰器）
//       └── PushbackInputStream（回推装饰器）
```

**装饰器模式 vs 代理模式**
```
相同点：都包装一个对象，都实现相同的接口
不同点：
  代理模式：控制对对象的访问，代理和被代理对象的关系在编译时确定
  装饰器模式：增强对象的功能，装饰器的组合在运行时动态决定
  代理模式通常只有一个代理，装饰器模式可以多层嵌套
```

**React HOC 对比**

> ⭐ **面试高频问题：**
> - Q: 装饰器模式和代理模式的区别？
> - A: 装饰器模式关注"增强功能"，代理模式关注"控制访问"。装饰器可以多层嵌套，每层添加不同功能；代理通常只有一层。装饰器的客户端知道自己在使用装饰器，代理的客户端可能不知道自己在使用代理。装饰器强调"增加"职责，代理强调"代替"访问。
> - Q: Java IO中哪些类使用了装饰器模式？
> - A: InputStream/OutputStream/Reader/Writer体系大量使用装饰器模式。FilterInputStream/FilterOutputStream/FilterReader/FilterWriter是装饰器基类，BufferedInputStream、DataInputStream、PushbackInputStream等是具体装饰器。使用时通过层层包装来组合功能。

### 3.4 桥接模式

**定义：** 将抽象部分与它的实现部分分离，使它们都可以独立地变化。

**适用场景：** 当一个类有两个独立变化的维度时，使用桥接模式将它们分离。

```java
/**
 * 桥接模式示例：形状 + 颜色
 * 形状（抽象）和颜色（实现）是两个独立变化的维度
 */
// 实现接口（颜色）
public interface Color {
    void applyColor();
}

public class RedColor implements Color {
    @Override
    public void applyColor() { System.out.print("红色"); }
}

public class BlueColor implements Color {
    @Override
    public void applyColor() { System.out.print("蓝色"); }
}

// 抽象类（形状）
public abstract class Shape {
    protected Color color; // 桥接：持有实现接口的引用

    public Shape(Color color) {
        this.color = color;
    }

    public abstract void draw();
}

// 具体抽象
public class Circle extends Shape {
    public Circle(Color color) { super(color); }

    @Override
    public void draw() {
        System.out.print("圆形 - ");
        color.applyColor();
    }
}

public class Square extends Shape {
    public Square(Color color) { super(color); }

    @Override
    public void draw() {
        System.out.print("方形 - ");
        color.applyColor();
    }
}

// 使用：两个维度可以独立扩展和组合
public class Demo {
    public static void main(String[] args) {
        Shape redCircle = new Circle(new RedColor());
        Shape blueSquare = new Square(new BlueColor());

        redCircle.draw();  // 圆形 - 红色
        blueSquare.draw(); // 方形 - 蓝色
    }
}
```

### 3.6 享元模式

**定义：** 运用共享技术有效地支持大量细粒度的对象。减少内存使用，提高性能。

**核心思想：** 将对象的状态分为"内部状态"（可共享，不随环境变化）和"外部状态"（不可共享，随环境变化）。共享内部状态，外部状态由客户端传入。

```java
/**
 * 享元模式示例：围棋棋子
 * 内部状态：颜色（黑/白）—— 可共享
 * 外部状态：位置 —— 不可共享，由客户端管理
 */
// 享元接口
public interface ChessPiece {
    void display(Position position);
}

// 具体享元
public class GoPiece implements ChessPiece {
    private final String color; // 内部状态（可共享）

    public GoPiece(String color) {
        this.color = color;
    }

    @Override
    public void display(Position position) {
        System.out.println(color + "棋子，位置: (" + position.getX() + ", " + position.getY() + ")");
    }
}

// 享元工厂
public class ChessPieceFactory {
    private static final Map<String, ChessPiece> pieces = new HashMap<>();

    public static ChessPiece getPiece(String color) {
        return pieces.computeIfAbsent(color, GoPiece::new);
    }

    public static int getTotalPieces() {
        return pieces.size();
    }
}

// 外部状态
public class Position {
    private final int x;
    private final int y;
    public Position(int x, int y) { this.x = x; this.y = y; }
    public int getX() { return x; }
    public int getY() { return y; }
}

// 使用
public class Demo {
    public static void main(String[] args) {
        ChessPiece black1 = ChessPieceFactory.getPiece("黑色");
        ChessPiece black2 = ChessPieceFactory.getPiece("黑色");
        ChessPiece white1 = ChessPieceFactory.getPiece("白色");

        System.out.println(black1 == black2); // true！同一个对象

        black1.display(new Position(1, 2));
        black2.display(new Position(3, 4));
        white1.display(new Position(2, 3));

        System.out.println("总棋子数: " + ChessPieceFactory.getTotalPieces()); // 2
    }
}
```

```java
    static final int low = -128;
    static final int high;
    static final Integer[] cache;

    static {

        cache = new Integer[(high - low) + 1];

    }
```

### 3.7 组合模式

**定义：** 将对象组合成树形结构以表示"部分-整体"的层次结构。组合模式使得用户对单个对象和组合对象的使用具有一致性。

```java
/**
 * 组合模式示例：文件系统
 */
// 抽象组件
public abstract class FileSystemNode {
    protected String name;

    public FileSystemNode(String name) {
        this.name = name;
    }

    public abstract void display();
    public abstract long getSize();
}

// 叶子节点：文件
public class File extends FileSystemNode {
    private final long size;

    public File(String name, long size) {
        super(name);
        this.size = size;
    }

    @Override
    public void display() {
        System.out.println("文件: " + name + " (" + size + " bytes)");
    }

    @Override
    public long getSize() {
        return size;
    }
}

// 组合节点：目录
public class Directory extends FileSystemNode {
    private final List<FileSystemNode> children = new ArrayList<>();

    public Directory(String name) {
        super(name);
    }

    public void add(FileSystemNode node) {
        children.add(node);
    }

    public void remove(FileSystemNode node) {
        children.remove(node);
    }

    @Override
    public void display() {
        System.out.println("目录: " + name + "/");
        for (FileSystemNode child : children) {
            child.display(); // 递归显示
        }
    }

    @Override
    public long getSize() {
        return children.stream().mapToLong(FileSystemNode::getSize).sum();
    }
}

// 使用
public class Demo {
    public static void main(String[] args) {
        Directory root = new Directory("根目录");
        Directory docs = new Directory("文档");
        Directory pics = new Directory("图片");

        docs.add(new File("简历.docx", 1024));
        docs.add(new File("报告.pdf", 2048));
        pics.add(new File("风景.jpg", 4096));

        root.add(docs);
        root.add(pics);

        root.display();
        System.out.println("总大小: " + root.getSize() + " bytes");
    }
}
```

## 四、行为型模式

行为型模式关注**对象之间的通信和职责分配**。

### 4.1 策略模式（重点！）

**定义：** 定义一系列算法，将每一个算法封装起来，并使它们可以互相替换。策略模式让算法的变化独立于使用算法的客户端。

**核心思想：** 消除大量的if-else/switch-case，将不同策略封装为独立的类，通过上下文来选择使用哪个策略。

```java
/**
 * 策略模式示例：支付方式选择
 */
// 策略接口
public interface PaymentStrategy {
    void pay(PaymentContext context);
}

// 上下文（包含支付信息）
@Data
@Builder
public class PaymentContext {
    private String orderId;
    private double amount;
    private String userId;
    private Map<String, String> extra; // 额外参数
}

// 具体策略：支付宝
@Component("ALIPAY")
public class AlipayStrategy implements PaymentStrategy {
    @Override
    public void pay(PaymentContext context) {
        System.out.println("支付宝支付: 订单=" + context.getOrderId()
            + ", 金额=" + context.getAmount()
            + ", 用户=" + context.getUserId());
        // 调用支付宝SDK...
    }
}

// 具体策略：微信支付
@Component("WECHAT")
public class WechatPayStrategy implements PaymentStrategy {
    @Override
    public void pay(PaymentContext context) {
        System.out.println("微信支付: 订单=" + context.getOrderId()
            + ", 金额=" + context.getAmount()
            + ", 用户=" + context.getUserId());
        // 调用微信支付SDK...
    }
}

// 具体策略：信用卡
@Component("CREDIT_CARD")
public class CreditCardStrategy implements PaymentStrategy {
    @Override
    public void pay(PaymentContext context) {
        String cardNo = context.getExtra().get("cardNo");
        System.out.println("信用卡支付: 订单=" + context.getOrderId()
            + ", 金额=" + context.getAmount()
            + ", 卡号=" + cardNo);
        // 调用支付网关...
    }
}

// 策略工厂（结合Spring）
@Component
public class PaymentStrategyFactory {
    private final Map<String, PaymentStrategy> strategyMap;

    // Spring自动注入所有PaymentStrategy的Bean
    @Autowired
    public PaymentStrategyFactory(Map<String, PaymentStrategy> strategyMap) {
        this.strategyMap = strategyMap;
    }

    public PaymentStrategy getStrategy(String type) {
        PaymentStrategy strategy = strategyMap.get(type);
        if (strategy == null) {
            throw new IllegalArgumentException("不支持的支付方式: " + type);
        }
        return strategy;
    }
}

// 支付服务（上下文）
@Service
public class PaymentService {
    private final PaymentStrategyFactory strategyFactory;

    @Autowired
    public PaymentService(PaymentStrategyFactory strategyFactory) {
        this.strategyFactory = strategyFactory;
    }

    // 没有if-else！通过策略工厂获取对应策略
    public void pay(String payType, PaymentContext context) {
        PaymentStrategy strategy = strategyFactory.getStrategy(payType);
        strategy.pay(context);
    }
}

// Controller
@RestController
@RequestMapping("/api/payment")
public class PaymentController {
    @Autowired
    private PaymentService paymentService;

    @PostMapping("/pay")
    public Result pay(@RequestParam String payType, @RequestBody PaymentContext context) {
        paymentService.pay(payType, context);
        return Result.success();
    }
}
```

> - Q: 策略模式如何消除if-else？

### 4.2 模板方法模式

**定义：** 定义一个操作中的算法的骨架，而将一些步骤延迟到子类中。模板方法使得子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤。

```java
/**
 * 模板方法模式示例：数据处理流程
 */
public abstract class DataProcessor {
    // 模板方法（final，不允许子类修改算法骨架）
    public final void process() {
        readData();
        validateData();
        transformData();
        saveData();
        notifyCompletion();
    }

    // 抽象方法：子类必须实现
    protected abstract void readData();
    protected abstract void transformData();

    // 具体方法：子类可以直接继承使用
    protected void validateData() {
        System.out.println("[通用] 数据校验");
    }

    protected void saveData() {
        System.out.println("[通用] 保存数据到数据库");
    }

    // 钩子方法：子类可以选择性覆盖
    protected void notifyCompletion() {
        System.out.println("[通用] 处理完成通知");
    }
}

// 具体实现1：CSV数据处理
```

**Spring 中的模板方法**
```java
/**
 * Spring JdbcTemplate 是模板方法模式的经典应用
 * 它将数据库操作的固定流程（获取连接→创建Statement→执行→处理结果→关闭资源）
 * 封装为模板，变化的部分通过回调接口让用户自定义
 */
// JdbcTemplate 的使用
@Repository
public class UserDao {
    @Autowired
    private JdbcTemplate jdbcTemplate;

    public User findById(Long id) {
        return jdbcTemplate.queryForObject(
            "SELECT id, name, email FROM users WHERE id = ?",
            (rs, rowNum) -> new User(
                rs.getLong("id"),
                rs.getString("name"),
                rs.getString("email")
            ),
            id
        );
    }

    public List<User> findAll() {
        return jdbcTemplate.query(
            "SELECT id, name, email FROM users",
            (rs, rowNum) -> new User(
                rs.getLong("id"),
                rs.getString("name"),
                rs.getString("email")
            )
        );
    }
}

// JdbcTemplate 内部的模板方法（简化）
public class JdbcTemplate {
    public <T> T query(String sql, RowMapper<T> rowMapper, Object... args) {
        Connection conn = null;
        PreparedStatement ps = null;
        ResultSet rs = null;
        try {
            conn = DataSourceUtils.getConnection(dataSource);
            ps = conn.prepareStatement(sql);
            // 设置参数...
            rs = ps.executeQuery();
            // 处理结果集 —— 这是变化的部分，通过RowMapper回调
            return rowMapper.mapRow(rs, 0);
        } finally {
            // 关闭资源 —— 固定的流程
            JdbcUtils.closeResultSet(rs);
            JdbcUtils.closeStatement(ps);
            DataSourceUtils.releaseConnection(conn, dataSource);
```

> - Q: 模板方法模式中，模板方法为什么通常声明为final？
> - A: 模板方法定义了算法的骨架（执行步骤和顺序），不允许子类修改。如果子类可以覆盖模板方法，可能会破坏算法的整体结构。子类只能通过覆盖具体的步骤方法来定制行为。

> - A: 模板方法基于继承，通过子类覆盖方法来改变行为；策略模式基于组合，通过注入不同的策略对象来改变行为。模板方法控制算法的大流程，策略模式关注某个具体算法的替换。模板方法是"自上而下"的控制，策略模式是"委托"式的控制。

### 4.3 观察者模式

**定义：** 定义对象间的一种一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都得到通知并被自动更新。

```java
/**
 * 观察者模式：推模式 vs 拉模式
 */
// 观察者接口
public interface Observer {
    // 推模式：被观察者主动推送数据
    void update(String message);
}

// 被观察者（主题）
public class Subject {
    private final List<Observer> observers = new ArrayList<>();
    private String state;

    public void attach(Observer observer) {
        observers.add(observer);
    }

    public void detach(Observer observer) {
        observers.remove(observer);
    }

    public void notifyObservers() {
        for (Observer observer : observers) {
            // 推模式：主动推送state
            observer.update(state);
        }
    }

    public void setState(String state) {
        this.state = state;
        notifyObservers(); // 状态变化时通知所有观察者
    }
}

// 具体观察者
public class EmailObserver implements Observer {
    @Override
    public void update(String message) {
        System.out.println("[邮件通知] 收到消息: " + message);
    }
}

public class SmsObserver implements Observer {
    @Override
    public void update(String message) {
        System.out.println("[短信通知] 收到消息: " + message);
    }
}

// 使用
public class Demo {
    public static void main(String[] args) {
        Subject subject = new Subject();
        subject.attach(new EmailObserver());
        subject.attach(new SmsObserver());

        subject.setState("订单已发货");
        // 输出：
        // [邮件通知] 收到消息: 订单已发货
        // [短信通知] 收到消息: 订单已发货
    }
}
```

### 4.4 责任链模式

**定义：** 将请求的发送者和接收者解耦，使多个对象都有机会处理这个请求。将这些对象连成一条链，并沿着这条链传递请求，直到有一个对象处理它为止。

```java
/**
 * 责任链模式示例：审批流程
 */
// 抽象处理器
public abstract class ApprovalHandler {
    protected ApprovalHandler next; // 下一个处理器

    public ApprovalHandler setNext(ApprovalHandler next) {
        this.next = next;
        return next; // 支持链式调用
    }

    public abstract void handle(ApprovalRequest request);
}

// 审批请求
@Data
@AllArgsConstructor
public class ApprovalRequest {
    private String applicant;
    private BigDecimal amount;
    private String description;
}

// 具体处理器：主管（审批5000以下）
public class ManagerHandler extends ApprovalHandler {
    @Override

            System.out.println("主管审批通过: " + request.getDescription());
        } else if (next != null) {
            next.handle(request); // 传递给下一个处理器
        }
    }
}

// 具体处理器：总监（审批10000以下）
public class DirectorHandler extends ApprovalHandler {
    @Override

        } else if (next != null) {
            next.handle(request);
        }
    }
}

// 具体处理器：CEO（审批10000以上）
```

### 4.5 状态模式

**定义：** 允许一个对象在其内部状态改变时改变它的行为。对象看起来似乎修改了它的类。

**状态模式 vs 策略模式**
```
相同点：都通过接口封装不同的行为，通过委托来实现多态
不同点：
  状态模式：状态之间有流转关系，状态切换由内部驱动
  策略模式：策略之间是平行的，策略选择由客户端决定
  状态模式：强调"状态变化导致行为变化"
  策略模式：强调"算法可以互相替换"
```

> ⭐ **面试高频问题：**
> - Q: 状态模式和策略模式的区别？
> - A: 结构上相似，但意图不同。状态模式中，状态之间有流转关系（待支付→已支付→已发货），状态切换由对象内部决定。策略模式中，策略之间是平行关系，由客户端选择使用哪个策略。状态模式强调"对象的行为随状态变化"，策略模式强调"算法的可替换性"。如果用一句话概括：状态模式是"自己选策略"，策略模式是"别人帮你选策略"。

### 4.7 迭代器模式

**定义：** 提供一种方法顺序访问一个聚合对象中各个元素，而又不暴露该对象的内部表示。

```java
/**
 * Java Iterator 是迭代器模式的内置实现
 */
// 自定义聚合类
public class MyList<T> implements Iterable<T> {
    private final Object[] elements;
    private int size;

    public MyList(int capacity) {
        elements = new Object[capacity];
        size = 0;
    }

    public void add(T element) {
        elements[size++] = element;
    }

    @Override
    public Iterator<T> iterator() {
        return new MyIterator();
    }

    private class MyIterator implements Iterator<T> {
        private int cursor = 0;

        @Override
        public boolean hasNext() {
            return cursor < size;
        }

        @Override
        @SuppressWarnings("unchecked")
        public T next() {
            if (!hasNext()) throw new NoSuchElementException();
            return (T) elements[cursor++];
        }
    }
}

// 使用

list.add("Hello");
list.add("World");

// for-each语法糖（编译后使用Iterator）

    System.out.println(s);
}

// 等价于
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    System.out.println(s);
}
```

### 4.8 中介者模式

**定义：** 用一个中介对象来封装一系列的对象交互。中介者使各对象不需要显式地相互引用，从而使其耦合松散，而且可以独立地改变它们之间的交互。

```java
/**
 * 中介者模式示例：聊天室
 */
// 中介者接口
public interface ChatMediator {
    void sendMessage(String message, User user);
    void addUser(User user);
}

// 具体中介者
public class ChatRoom implements ChatMediator {
    private final List<User> users = new ArrayList<>();

    @Override
    public void addUser(User user) {
        users.add(user);
    }

    @Override
    public void sendMessage(String message, User user) {
        for (User u : users) {
            if (u != user) { // 不发送给自己
                u.receive(message);
            }
        }
    }
}

// 同事类
public abstract class User {
    protected ChatMediator mediator;
    protected String name;

    public User(ChatMediator mediator, String name) {
        this.mediator = mediator;
        this.name = name;
    }

    public abstract void send(String message);
    public abstract void receive(String message);
}

public class ChatUser extends User {
    public ChatUser(ChatMediator mediator, String name) {
        super(mediator, name);
    }

    @Override
    public void send(String message) {
```

### 4.9 备忘录模式

**定义：** 在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，以便以后恢复。

```java
/**
 * 备忘录模式示例：文本编辑器撤销/重做
 */
// 备忘录
public class TextMemento {
    private final String text;
    private final int cursorPosition;

    public TextMemento(String text, int cursorPosition) {
        this.text = text;
        this.cursorPosition = cursorPosition;
    }

    public String getText() { return text; }
    public int getCursorPosition() { return cursorPosition; }
}

// 发起人（需要保存状态的对象）
public class TextEditor {
    private String text = "";
    private int cursorPosition = 0;

    public void type(String newText) {
        text += newText;
        cursorPosition = text.length();
    }

    public void delete(int count) {
        if (count <= text.length()) {
            text = text.substring(0, text.length() - count);
            cursorPosition = text.length();
        }
    }

    // 创建备忘录
    public TextMemento save() {
        return new TextMemento(text, cursorPosition);
    }

    // 从备忘录恢复
    public void restore(TextMemento memento) {
        this.text = memento.getText();
        this.cursorPosition = memento.getCursorPosition();
    }

    public String getText() { return text; }
}

// 管理者（负责保存备忘录）
public class History {
    private final Stack<TextMemento> undoStack = new Stack<>();
    private final Stack<TextMemento> redoStack = new Stack<>();

    public void push(TextMemento memento) {
        undoStack.push(memento);
        redoStack.clear(); // 新操作清空重做栈
    }

    public TextMemento undo() {
        if (undoStack.isEmpty()) return null;
        TextMemento memento = undoStack.pop();
        redoStack.push(memento);
        return memento;
    }

    public TextMemento redo() {
        if (redoStack.isEmpty()) return null;
        TextMemento memento = redoStack.pop();
        undoStack.push(memento);
        return memento;
    }
}

// 使用
public class Demo {
    public static void main(String[] args) {
        TextEditor editor = new TextEditor();
        History history = new History();

        editor.type("Hello");
        history.push(editor.save()); // 保存状态

        editor.type(" World");
        history.push(editor.save()); // 保存状态

        System.out.println(editor.getText()); // "Hello World"

        editor.restore(history.undo()); // 撤销
        System.out.println(editor.getText()); // "Hello"

        editor.restore(history.redo()); // 重做
        System.out.println(editor.getText()); // "Hello World"
    }
}
```

### 4.10 访问者模式

**定义：** 表示一个作用于某对象结构中的各元素的操作。它使你可以在不改变各元素的类的前提下定义作用于这些元素的新操作。

```java
/**
 * 访问者模式示例：编译器AST遍历
 */
// 访问者接口
public interface Visitor {
    void visit(NumberNode node);
    void visit(AddNode node);
    void visit(MultiplyNode node);
}

// 元素接口
public interface ASTNode {
    void accept(Visitor visitor);
}

// 具体元素
public class NumberNode implements ASTNode {
    private final int value;
    public NumberNode(int value) { this.value = value; }
    public int getValue() { return value; }

    @Override
    public void accept(Visitor visitor) {
        visitor.visit(this); // 双重分派
    }
}

public class AddNode implements ASTNode {
    private final ASTNode left;
    private final ASTNode right;

    public AddNode(ASTNode left, ASTNode right) {
        this.left = left;
        this.right = right;
    }

    public ASTNode getLeft() { return left; }
    public ASTNode getRight() { return right; }

    @Override
    public void accept(Visitor visitor) {
        visitor.visit(this);
    }
}

// 具体访问者：计算求值
public class EvaluateVisitor implements Visitor {
    @Override
    public void visit(NumberNode node) {
        // 叶子节点，值就是自身
    }

    @Override
    public void visit(AddNode node) {
        // 递归访问子节点
        node.getLeft().accept(this);
        node.getRight().accept(this);
    }

    @Override
    public void visit(MultiplyNode node) {
        node.getLeft().accept(this);
        node.getRight().accept(this);
    }
}

// 具体访问者：打印表达式
public class PrintVisitor implements Visitor {
    @Override
    public void visit(NumberNode node) {
        System.out.print(node.getValue());
    }

    @Override
    public void visit(AddNode node) {
        System.out.print("(");
        node.getLeft().accept(this);
        System.out.print(" + ");
        node.getRight().accept(this);
        System.out.print(")");
    }

    @Override
    public void visit(MultiplyNode node) {
        System.out.print("(");
        node.getLeft().accept(this);
        System.out.print(" * ");
        node.getRight().accept(this);
        System.out.print(")");
    }
}
```

> ⭐ **面试高频问题：**
> - Q: 访问者模式的使用场景？
> - A: 适用于对象结构比较稳定，但需要在此结构上定义很多新操作的场景。如编译器AST遍历、XML文档处理、代码分析工具等。访问者模式的核心是"双重分派"，通过accept方法调用visitor的visit方法，visitor的visit方法再调用元素的accept方法，从而在运行时根据元素的实际类型选择正确的visit方法。

### 4.11 解释器模式

**定义：** 给定一个语言，定义它的文法的一种表示，并定义一个解释器，这个解释器使用该表示来解释语言中的句子。

```java
/**
 * 解释器模式示例：简单的数学表达式解释器
 */
// 抽象表达式
public interface Expression {
    int interpret();
}

// 终结符表达式：数字
public class NumberExpression implements Expression {
    private final int number;

    public NumberExpression(int number) {
        this.number = number;
    }

    @Override
    public int interpret() {
        return number;
    }
}

// 非终结符表达式：加法
public class AddExpression implements Expression {
    private final Expression left;
    private final Expression right;

    public AddExpression(Expression left, Expression right) {
        this.left = left;
        this.right = right;
    }

    @Override
    public int interpret() {
        return left.interpret() + right.interpret();
    }
}

// 非终结符表达式：减法
public class SubtractExpression implements Expression {
    private final Expression left;
    private final Expression right;

    public SubtractExpression(Expression left, Expression right) {
        this.left = left;
        this.right = right;
    }

    @Override
    public int interpret() {
        return left.interpret() - right.interpret();
    }
}

// 使用
public class Demo {
    public static void main(String[] args) {
        // 解释表达式: (10 + 5) - 3
        Expression expression = new SubtractExpression(
            new AddExpression(new NumberExpression(10), new NumberExpression(5)),
            new NumberExpression(3)
        );

        System.out.println(expression.interpret()); // 12
    }
}
```

**实际应用：**
- 正则表达式引擎（java.util.regex）
- SQL解析器
- 表达式计算引擎（Spring EL、SpEL）
- 模板引擎（Thymeleaf、FreeMarker）

> ⭐ **面试高频问题：**
> - Q: 解释器模式在实际开发中常用吗？
> - A: 不太常用。解释器模式适合简单的文法，对于复杂的语言，推荐使用解析器生成工具（如ANTLR）或直接使用现成的引擎（如SpEL）。解释器模式的缺点是类膨胀（每条规则对应一个类）和性能问题（递归调用）。但在面试中需要了解其原理和应用场景。

### 5.2 单例模式（Bean 默认作用域）

```java
/**
 * Spring Bean默认是singleton作用域
 * Spring通过三级缓存解决循环依赖问题
 */
// Bean的作用域
@Scope("singleton")  // 默认，每个IoC容器一个实例
@Scope("prototype")  // 每次请求创建新实例
@Scope("request")    // 每个HTTP请求一个实例
@Scope("session")    // 每个HTTP Session一个实例

// Spring单例注册表（简化）
public class DefaultSingletonBeanRegistry {
    // 一级缓存：存放完全初始化好的Bean
    private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);
    // 二级缓存：存放提前暴露的Bean（半成品）
    private final Map<String, Object> earlySingletonObjects = new HashMap<>(16);
    // 三级缓存：存放Bean工厂对象
    private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);

    public Object getSingleton(String beanName) {
        // 1. 从一级缓存获取
        Object singletonObject = singletonObjects.get(beanName);
        if (singletonObject == null) {
            // 2. 从二级缓存获取
            singletonObject = earlySingletonObjects.get(beanName);
            if (singletonObject == null) {
                // 3. 从三级缓存获取工厂对象
                ObjectFactory<?> singletonFactory = singletonFactories.get(beanName);
                if (singletonFactory != null) {
                    singletonObject = singletonFactory.getObject();
                    // 放入二级缓存
                    earlySingletonObjects.put(beanName, singletonObject);
                    // 从三级缓存移除
                    singletonFactories.remove(beanName);
                }
            }
        }
        return singletonObject;
    }
}
```

```java
// 2. Controller 接口 → SimpleControllerHandlerAdapter

        // 处理返回值

    }
}
```

### 6.1 手写单例模式（DCL）

```java
/**
 * 面试必考：手写DCL双重检查锁单例

public class Singleton implements Serializable {
    // volatile防止指令重排序
    private static volatile Singleton instance;

    // 私有构造方法
    private Singleton() {
        // 防止反射攻击
        if (instance != null) {
            throw new RuntimeException("Use getInstance()");
        }
    }

    // DCL双重检查锁
    public static Singleton getInstance() {
        if (instance == null) {                    // 第一次检查（无锁）
            synchronized (Singleton.class) {       // 加锁
                if (instance == null) {            // 第二次检查（有锁）
                    instance = new Singleton();     // 创建实例
                }
            }
        }
        return instance;
    }

    // 防止序列化破坏
    private Object readResolve() {
        return instance;
    }
}

// 面试加分项：同时写出枚举单例
```

- 如何防止反射破坏？→ 构造方法中检查instance是否为null

**对比总结：**

| 维度 | JDK动态代理 | CGLIB代理 |
|------|-----------|----------|
| 实现原理 | 基于接口，Proxy生成代理类 | 基于继承，Enhancer生成子类 |
| 要求 | 目标类必须实现接口 | 目标类不能是final |
| 性能 | JDK8+已优化，接近CGLIB | 生成子类有开销，调用快 |
| 调用方式 | 反射调用（Method.invoke） | FastClass机制（非反射） |
| 依赖 | JDK内置 | 需要cglib库 |
| Spring默认 | 有接口时使用 | 无接口或强制配置时使用 |

### 6.5 观察者模式实现

```java
/**
 * 面试要求：手写观察者模式
 */
// 方式一：传统实现
public interface Observer {
    void update(String event, Object data);
}

public class EventManager {
    private final Map<String, List<Observer>> listeners = new HashMap<>();

    public void subscribe(String eventType, Observer observer) {
        listeners.computeIfAbsent(eventType, k -> new ArrayList<>()).add(observer);
    }

    public void unsubscribe(String eventType, Observer observer) {
        listeners.getOrDefault(eventType, Collections.emptyList()).remove(observer);
    }

    public void notify(String eventType, Object data) {
        listeners.getOrDefault(eventType, Collections.emptyList())
            .forEach(observer -> observer.update(eventType, data));
    }
}

// 方式二：Java内置实现
public class MyEvent extends EventObject {
    public MyEvent(Object source, String message) {
        super(source);
        this.message = message;
    }
    private final String message;
    public String getMessage() { return message; }
}

public class MyListener implements EventListener {
    public void handleEvent(MyEvent event) {
        System.out.println("收到事件: " + event.getMessage());
    }
}
```

### 6.7 Spring 用了哪些设计模式

> ⭐ **面试标准答案：**

Spring框架中使用了以下设计模式：

1. **工厂模式**：BeanFactory、ApplicationContext管理Bean的创建
2. **单例模式**：Bean默认scope为singleton
3. **代理模式**：AOP通过JDK动态代理或CGLIB实现
4. **模板方法**：JdbcTemplate、RestTemplate、RedisTemplate
5. **观察者模式**：ApplicationEvent事件机制
6. **适配器模式**：HandlerAdapter适配不同类型的Handler
7. **责任链模式**：Filter Chain、Spring Security过滤器链
8. **装饰器模式**：BeanWrapper增强Bean的属性访问能力
9. **策略模式**：Resource资源加载、HttpMessageConverter
10. **建造者模式**：BeanDefinitionBuilder构建BeanDefinition
11. **前端控制器模式**：DispatcherServlet统一处理请求
12. **外观模式**：JdbcTemplate简化JDBC操作

**加分回答：** Spring的循环依赖解决使用了三级缓存，其中涉及工厂模式（ObjectFactory）和代理模式（提前暴露代理对象）。Spring Boot的自动配置使用了条件注解（@ConditionalOnXxx），可以看作是策略模式的变体。

### 6.8 如何选择设计模式

> ⭐ **面试标准答案：**

选择设计模式的决策树：

```
需要创建对象？
  ├── 只需要一个实例 → 单例模式
  ├── 创建过程复杂（多参数） → 建造者模式
  ├── 根据条件创建不同类型 → 工厂模式/简单工厂
  ├── 创建产品族 → 抽象工厂
  └── 通过克隆创建 → 原型模式

需要改变对象的行为？
  ├── 运行时切换算法 → 策略模式
  ├── 状态变化导致行为变化 → 状态模式
  ├── 不修改代码扩展功能 → 装饰器模式
  └── 控制对象访问 → 代理模式

需要对象之间的通信？
  ├── 一对多通知 → 观察者模式
  ├── 多个对象依次处理 → 责任链模式
  ├── 对象间解耦通信 → 中介者模式
  └── 封装请求为对象 → 命令模式

需要组织对象结构？
  ├── 树形结构 → 组合模式
  ├── 接口不兼容 → 适配器模式
  ├── 两个维度独立变化 → 桥接模式
  ├── 简化子系统接口 → 外观模式
  └── 共享细粒度对象 → 享元模式
```

**实际开发中的建议：**
1. **不要为了用模式而用模式**：先解决问题，模式是手段不是目的
2. **从简单开始**：KISS原则，能用简单方案就不要引入复杂模式
3. **识别变化点**：对可能变化的部分使用模式，稳定的部分保持简单
4. **结合框架**：Spring已经内置了很多模式，优先使用框架提供的机制
5. **重构时引入**：代码坏味道出现时，通过重构引入设计模式

**React工程师学习建议：**
- 策略模式 ≈ React中的render props / 自定义Hook
- 观察者模式 ≈ Redux的subscribe/dispatch
- 装饰器模式 ≈ React HOC
- 组合模式 ≈ React组件树
- 外观模式 ≈ React的Context API
- 模板方法 ≈ React生命周期方法
- 代理模式 ≈ React的forwardRef / useMemo

---

> **总结：** 设计模式不是银弹，而是解决特定问题的经验总结。面试中不仅要记住每种模式的定义和代码，更要理解其**适用场景**、**优缺点**和**与其他模式的区别**。在实际项目中，合理使用设计模式可以提高代码的可维护性和可扩展性，但过度使用反而会增加复杂度。掌握SOLID原则是理解设计模式的基础，原则指导模式的选择，模式是原则的具体实践。

#### 1. 策略接口 + 多个策略实现


#### 2. 策略工厂 + Spring 自动注入策略 Map

```java
// 3. 策略工厂（Spring 自动注入所有策略实现到 Map）
@Service
public class PaymentService {

    // Spring 会自动将所有 PaymentStrategy 实现注入到这个 Map
    // key = @Component 的 bean name（如 "alipayStrategy"）
    // value = 策略实例
    private final Map<String, PaymentStrategy> strategyMap;
    private final PaymentStrategy defaultStrategy;

    @Autowired
    public PaymentService(Map<String, PaymentStrategy> strategyMap) {
        this.strategyMap = strategyMap;
        this.defaultStrategy = strategyMap.values().stream()
            .findFirst()
            .orElseThrow(() -> new IllegalStateException("没有可用的支付策略"));
    }

    /**
     * 根据支付方式执行支付（消除 if-else）
     */
    public PayResult pay(String payType, BigDecimal amount, String orderNo) {
        PaymentStrategy strategy = strategyMap.values().stream()
            .filter(s -> s.getType().equals(payType))
            .findFirst()
            .orElseThrow(() -> new IllegalArgumentException("不支持的支付方式: " + payType));

        return strategy.pay(amount, orderNo);
    }
}
```

#### 3. 完整代码示例

```java
// 使用方式
@RestController
@RequestMapping("/order")
public class OrderController {

    @Autowired
    private PaymentService paymentService;

    @PostMapping("/pay")
    public Result pay(@RequestParam String payType,
                      @RequestParam BigDecimal amount,
                      @RequestParam String orderNo) {
        // 没有 if-else，通过策略模式自动路由
        PayResult result = paymentService.pay(payType, amount, orderNo);
        return Result.success(result);
    }
}

// 传统 if-else 写法（对比）
public PayResult payOldWay(String payType, BigDecimal amount, String orderNo) {
    if ("ALIPAY".equals(payType)) {
        // 支付宝支付逻辑...
    } else if ("WECHAT".equals(payType)) {
        // 微信支付逻辑...
    } else if ("BANK_CARD".equals(payType)) {
        // 银行卡支付逻辑...
    } else {
        throw new IllegalArgumentException("不支持的支付方式");
    }
    // 每新增一种支付方式都要修改这里，违反开闭原则
}
```

> ⭐ **面试问答：策略模式如何消除 if-else？**
>
> 答：策略模式将每种条件分支封装为独立的策略类，通过策略工厂（或 Spring 自动注入的 Map）根据类型查找对应策略。新增策略只需添加新的实现类，无需修改工厂代码，符合开闭原则。Spring 中可以利用依赖注入自动收集所有策略实现到 Map 中，key 为 bean name，value 为策略实例，非常优雅。

### 补充三、建造者模式与 Lombok @Builder（⭐⭐中频）

#### 1. @Builder 原理

```java
// 使用 @Builder 注解
@Builder
public class User {
    private Long id;
    private String name;
    private Integer age;
    private String email;
}

// Lombok @Builder 生成的代码（等价于）
public class User {
    private Long id;
    private String name;
    private Integer age;
    private String email;

    // 私有全参构造器
    private User(Long id, String name, Integer age, String email) {
        this.id = id;
        this.name = name;
        this.age = age;
        this.email = email;
    }

    // Builder 类
    public static class UserBuilder {
        private Long id;
        private String name;
        private Integer age;
        private String email;

        UserBuilder() {}

        public UserBuilder id(Long id) {
            this.id = id;
            return this;  // 链式调用
        }

        public UserBuilder name(String name) {
            this.name = name;
            return this;
        }

        public UserBuilder age(Integer age) {
            this.age = age;
            return this;
        }

        public UserBuilder email(String email) {
            this.email = email;
            return this;
        }

        public User build() {
            return new User(this.id, this.name, this.age, this.email);
        }
    }

    public static UserBuilder builder() {
        return new UserBuilder();
    }
}

// 使用方式
User user = User.builder()
    .id(1L)
    .name("张三")
    .age(25)
    .email("zhangsan@example.com")
    .build();
```

#### 2. 与构造器的区别

```
建造者模式 vs 构造器 vs Setter：

1. 构造器（Telescoping Constructor）
   new User(1L, "张三", 25, "zhangsan@example.com");
   缺点：参数多时难以阅读（哪个参数是什么？），参数顺序容易搞错

2. Setter（JavaBean 模式）
   User user = new User();
   user.setId(1L);
   user.setName("张三");
   user.setAge(25);
   user.setEmail("zhangsan@example.com");
   缺点：对象可能处于不一致状态（set 到一半就被使用），线程不安全

3. 建造者模式（@Builder）
   User user = User.builder()
       .id(1L)
       .name("张三")
       .age(25)
       .email("zhangsan@example.com")
       .build();
   优点：
   - 可读性好（参数名清晰）
   - 灵活（可选参数，不需要全部设置）
   - 不可变对象（build() 返回的对象可以设为不可变）
   - 线程安全（build 后对象不可变）

适用场景：
  - 类有 4 个以上参数
  - 参数大部分是可选的
  - 需要创建不可变对象
  - 需要多种配置组合
```

```java
// @Builder 与其他 Lombok 注解组合使用

// @Builder + @AllArgsConstructor
@Builder
@AllArgsConstructor
public class User {
    private Long id;
    private String name;
    private Integer age;
}

// @Builder.Default：设置默认值
@Builder
public class User {
    @Builder.Default
    private Integer age = 18;
    @Builder.Default
    private Boolean active = true;
}

// 自定义 Builder（继承场景）
public class BaseEntity {
    private Long id;
    private LocalDateTime createTime;
}

@Builder
public class User extends BaseEntity {
    private String name;
    private Integer age;
}

// 使用 @SuperBuilder 处理继承
@SuperBuilder
public class BaseEntity {
    private Long id;
    private LocalDateTime createTime;
}

@SuperBuilder
public class User extends BaseEntity {
    private String name;
    private Integer age;
}

// 使用
User user = User.builder()
    .id(1L)
    .createTime(LocalDateTime.now())
    .name("张三")
    .age(25)
    .build();
```

> ⭐ **面试问答：建造者模式和构造器模式的区别？**
>
> 答：构造器模式（Telescoping Constructor）通过多个重载构造器处理可选参数，参数多时可读性差，且参数顺序容易混淆。建造者模式通过链式调用设置属性，可读性好，参数名清晰，且支持可选参数。建造者模式适合参数较多（4个以上）、大部分参数可选、需要创建不可变对象的场景。Lombok 的 @Builder 注解自动生成 Builder 类，简化了建造者模式的实现。
