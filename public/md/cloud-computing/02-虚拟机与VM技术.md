# 02 虚拟机与VM技术

## 2.1 虚拟机基础

### 2.1.1 什么是虚拟机

虚拟机（Virtual Machine，VM）是通过软件模拟的完整计算机系统，拥有自己的虚拟 CPU（vCPU）、虚拟内存（vMemory）、虚拟磁盘（vDisk）和虚拟网卡（vNIC），并运行独立的 Guest OS。虚拟机与物理机一样可以执行完整的操作系统和应用程序，但从用户角度看，其行为与一台真实的物理计算机几乎无异。

虚拟机的核心价值在于**资源隔离**和**资源复用**：一台物理服务器可以通过虚拟化技术运行多个虚拟机，每个虚拟机相互隔离、互不影响，从而大幅提升硬件资源的利用率。虚拟机也是云计算 IaaS 层的基础技术单元，AWS EC2、阿里云 ECS 等云服务器本质上都是虚拟机。

### 2.1.2 虚拟机的组成
- **虚拟硬件**：vCPU、vMemory、vDisk、vNIC
- **Guest OS**：运行在虚拟机中的操作系统
- **虚拟磁盘文件**：.vmdk（VMware）、.qcow2（KVM）、.vdi（VirtualBox）

## 2.2 Hypervisor（虚拟机监控器）

### 2.2.1 Type 1 裸金属 Hypervisor

Type 1 Hypervisor（也称裸金属型 Hypervisor）直接运行在物理硬件之上，无需宿主操作系统，直接管理和调度硬件资源。由于没有中间层，性能接近裸机，是生产环境和云厂商的首选。

| 产品 | 厂商 | 特点 |
|------|------|------|
| ESXi | VMware（现属 Broadcom） | 企业级，功能最全，vSphere 生态成熟 |
| Hyper-V | Microsoft | 与 Windows Server 深度集成，Azure 底层基于此 |
| Xen | 开源（Xen Project） | AWS 早期使用，现已逐步迁移至 Nitro 架构 |
| KVM | 开源（Linux 内核模块） | Linux 内置，性能优异，当前云厂商主流选择 |

### 2.2.2 Type 2 宿主型 Hypervisor

Type 2 Hypervisor（也称宿主型 Hypervisor）运行在宿主操作系统之上，作为普通应用程序安装。虚拟机的 I/O 请求需要经过宿主 OS，因此性能有一定损耗，但安装和使用简单，适合开发和测试环境。

| 产品 | 特点 |
|------|------|
| VirtualBox | 免费开源，支持 Windows/Linux/macOS，适合开发测试 |
| VMware Workstation / Fusion | 功能强大，Workstation 用于 Windows/Linux，Fusion 用于 macOS |
| Parallels Desktop | macOS 专用，与 Apple Silicon 深度集成，体验流畅 |

### 2.2.3 Type 1 vs Type 2 对比
| 维度 | Type 1 | Type 2 |
|------|--------|--------|
| 性能 | 接近裸机 | 有宿主OS开销 |
| 安全性 | 更高 | 较低 |
| 适用场景 | 生产环境 | 开发/测试 |
| 管理复杂度 | 较高 | 较低 |

## 2.3 KVM 深入（云厂商主流）

### 2.3.1 KVM 架构

KVM（Kernel-based Virtual Machine）是基于 Linux 内核的虚拟化模块，于 2007 年合入 Linux 内核主线（2.6.20）。KVM 将 Linux 内核转变为一个 Hypervisor，其架构分为以下几层：

```
+----------------------------------+
|        用户空间管理工具            |
|   (libvirt / virsh / virt-manager)|
+----------------------------------+
|        QEMU（设备模拟）            |
+----------------------------------+
|        KVM 内核模块               |
|  (kvm.ko + kvm-intel.ko/amd.ko)  |
+----------------------------------+
|        Linux 内核                 |
+----------------------------------+
|        物理硬件                   |
+----------------------------------+
```

**核心组件**：
- **kvm.ko**：KVM 核心内核模块，提供虚拟化基础设施
- **kvm-intel.ko / kvm-amd.ko**：CPU 厂商特定的硬件虚拟化扩展模块，依赖 Intel VT-x 或 AMD-V
- **QEMU**：运行在用户空间，负责设备模拟（磁盘、网卡、显卡等）
- **libvirt**：虚拟化管理 API 和工具集，提供统一的虚拟机管理接口

**工作原理**：KVM 利用 CPU 的硬件虚拟化扩展（Intel VT-x / AMD-V），将 Guest OS 的特权指令 trap 到内核态处理。Guest OS 执行非特权指令时直接在物理 CPU 上运行（零开销），执行特权指令时由 KVM 模块拦截并模拟，从而实现接近原生的性能。

### 2.3.2 QEMU 与 KVM 的关系

QEMU 和 KVM 是互补关系，两者协同工作：

- **KVM**：负责 CPU 和内存的虚拟化，利用硬件虚拟化扩展提供高性能的指令执行。KVM 本身只处理 CPU 虚拟化，不提供设备模拟。
- **QEMU**：负责 I/O 设备的模拟，包括虚拟磁盘（VirtIO Block）、虚拟网卡（VirtIO Net）、显示设备等。QEMU 可以独立运行（纯软件模拟，速度慢），也可以与 KVM 配合使用（KVM 加速，性能接近原生）。

**VirtIO**：KVM/QEMU 生态中的半虚拟化 I/O 框架。Guest OS 安装 VirtIO 驱动后，可以直接与 QEMU 的后端设备通信，避免全设备模拟的性能开销。使用 VirtIO 驱动可使磁盘 I/O 和网络性能提升数倍。

**简单类比**：KVM 是"发动机"（提供计算加速），QEMU 是"车身"（提供完整的外设模拟），libvirt 是"方向盘"（提供管理接口）。

### 2.3.3 libvirt 虚拟化管理

libvirt 是一个开源的 API 和管理工具集，提供统一的虚拟化管理接口，支持 KVM、QEMU、Xen、LXC 等多种虚拟化技术。

**核心组件**：
- **libvirtd**：守护进程，接收管理请求并执行虚拟机操作
- **virsh**：命令行管理工具，最常用的交互方式
- **virt-manager**：图形化管理工具，适合桌面环境
- **virt-install**：命令行虚拟机创建工具
- **XML 配置文件**：虚拟机的完整定义，包括 CPU、内存、磁盘、网络等所有配置

**虚拟机 XML 配置示例**（`virsh dumpxml vm-name` 输出）：
```xml
<domain type='kvm'>
  <name>my-vm</name>
  <memory unit='KiB'>4194304</memory>          <!-- 4GB 内存 -->
  <vcpu placement='static'>2</vcpu>             <!-- 2 个 vCPU -->
  <os>
    <type arch='x86_64'>hvm</type>              <!-- 硬件辅助虚拟化 -->
  </os>
  <devices>
    <disk type='file' device='disk'>
      <driver name='qemu' type='qcow2'/>        <!-- qcow2 磁盘格式 -->
      <source file='/var/lib/libvirt/images/my-vm.qcow2'/>
      <target dev='vda' bus='virtio'/>          <!-- VirtIO 磁盘 -->
    </disk>
    <interface type='bridge'>
      <source bridge='virbr0'/>
      <model type='virtio'/>                    <!-- VirtIO 网卡 -->
    </interface>
  </devices>
</domain>
```

### 2.3.4 常用 KVM 操作命令

```bash
# ===== 虚拟机生命周期管理 =====

# 查看所有虚拟机（包括已关机的）
virsh list --all

# 启动虚拟机
virsh start my-vm

# 关闭虚拟机（优雅关机，等同执行 shutdown）
virsh shutdown my-vm

# 强制关机（等同拔电源）
virsh destroy my-vm

# 重启虚拟机
virsh reboot my-vm

# 挂起（暂停到内存，恢复后继续运行）
virsh suspend my-vm

# 恢复挂起的虚拟机
virsh resume my-vm

# ===== 虚拟机配置管理 =====

# 导出虚拟机 XML 配置
virsh dumpxml my-vm > my-vm-backup.xml

# 编辑虚拟机配置（会打开编辑器）
virsh edit my-vm

# 基于修改后的 XML 重新定义虚拟机
virsh define my-vm-backup.xml

# 取消定义（删除虚拟机配置，不删除磁盘文件）
virsh undefine my-vm

# ===== 虚拟机创建 =====

# 使用 virt-install 创建新虚拟机
virt-install \
  --name my-vm \
  --memory 4096 \
  --vcpus 2 \
  --disk path=/var/lib/libvirt/images/my-vm.qcow2,size=50,format=qcow2,bus=virtio \
  --network bridge=virbr0,model=virtio \
  --cdrom /path/to/ubuntu-22.04.iso \
  --os-variant ubuntu22.04 \
  --graphics vnc,listen=0.0.0.0 \
  --noautoconsole

# 通过克隆创建虚拟机
virt-clone \
  --original my-vm \
  --name my-vm-clone \
  --file /var/lib/libvirt/images/my-vm-clone.qcow2

# ===== 快照管理 =====

# 创建快照（磁盘和内存）
virsh snapshot-create-as my-vm --name "snapshot-1" --description "安装完成后的快照"

# 查看快照列表
virsh snapshot-list my-vm

# 恢复到指定快照
virsh snapshot-revert my-vm --snapshotname "snapshot-1"

# 删除快照
virsh snapshot-delete my-vm --snapshotname "snapshot-1"

# ===== 热迁移 =====

# 将虚拟机从当前主机实时迁移到目标主机
virsh migrate --live my-vm qemu+ssh://dest-host/system

# ===== 监控与调试 =====

# 查看虚拟机实时资源使用
virsh dominfo my-vm
virsh dommemstat my-vm
virsh vcpuinfo my-vm

# 连接到虚拟机控制台
virsh console my-vm

# 查看虚拟机块设备信息
virsh domblklist my-vm

# 查看虚拟机网络接口信息
virsh domiflist my-vm
```

## 2.4 虚拟机 vs 容器

### 2.4.1 架构对比
```
虚拟机：
  App → Bins/Libs → Guest OS → Hypervisor → Host OS → 硬件

容器：
  App → Bins/Libs → Docker Engine → Host OS → 硬件
```

### 2.4.2 详细对比

| 维度 | 虚拟机 (VM) | 容器 (Container) |
|------|------------|-----------------|
| 启动时间 | 分钟级 | 秒级 |
| 镜像大小 | GB 级 | MB 级 |
| 性能开销 | 5-15% | 接近原生 |
| 隔离级别 | 硬件级（强隔离） | 进程级（弱隔离） |
| OS 支持 | 任意 OS | 共享 Host OS 内核 |
| 密度 | 每台宿主几个~几十个 | 每台宿主几百~几千个 |
| 适用场景 | 需要不同OS、强隔离 | 微服务、快速部署 |

### 2.4.3 何时选择 VM vs 容器

**选择虚拟机（VM）的场景**：
- **需要不同的操作系统内核**：如同时运行 Windows Server 和 Linux，或需要特定内核版本
- **强安全隔离需求**：金融、政务等场景，需要硬件级隔离确保安全边界
- **遗留应用迁移**：传统单体应用直接迁移到云端，无需重构
- **内核模块依赖**：应用依赖特定的内核模块（如 DPDK、特殊驱动）
- **多租户环境**：为不同客户提供完全隔离的运行环境

**选择容器的场景**：
- **微服务架构**：将应用拆分为多个独立服务，容器天然适合
- **CI/CD 流水线**：构建、测试、部署环境一致性，容器镜像保证"构建一次，到处运行"
- **快速弹性伸缩**：容器秒级启动，适合 Kubernetes 的 HPA（水平 Pod 自动伸缩）
- **开发环境一致性**：消除"在我机器上能跑"的问题
- **高密度部署**：在有限的硬件资源上运行大量服务实例

**实际趋势**：现代云架构中，容器和虚拟机并非互斥。许多企业采用"VM 中运行容器"的混合模式，如 AWS 的 EC2 上运行 EKS（Kubernetes），兼顾了 VM 的隔离性和容器的敏捷性。

## 2.5 虚拟机网络

### 2.5.1 虚拟网络模式
- **桥接模式（Bridge）**：VM 直接接入物理网络，获得与物理机同网段的 IP 地址，可被外部直接访问
- **NAT 模式**：VM 通过宿主机 NAT 转换访问外网，外部无法直接访问 VM（需端口转发）
- **仅主机模式（Host-Only）**：VM 只能与宿主机通信，无法访问外网，适合测试环境

### 2.5.2 虚拟交换机（vSwitch）

虚拟交换机（Virtual Switch）是在软件层面实现的二层网络交换设备，负责虚拟机之间、虚拟机与物理网络之间的数据包转发。

**Open vSwitch（OVS）**：
- 开源的虚拟交换机，功能丰富，是 OpenStack Neutron 和许多 SDN 方案的标准组件
- 支持标准 802.1Q VLAN、VXLAN、GRE、STP、LACP 等协议
- 支持 OpenFlow 协议，可实现 SDN 可编程网络控制
- 支持流表（Flow Table）和隧道（Tunnel），适合大规模云环境
- 支持 NetFlow、sFlow、镜像端口等网络监控功能

**OVS 常用操作**：
```bash
# 创建虚拟网桥
ovs-vsctl add-br br-int

# 将物理网卡接入网桥
ovs-vsctl add-port br-int eth0

# 将虚拟机的 tap 设备接入网桥
ovs-vsctl add-port br-int tap-xxx

# 创建内部端口（用于路由/DHCP）
ovs-vsctl add-port br-int veth-dhcp -- set interface veth-dhcp type=internal

# 查看网桥配置
ovs-vsctl show

# 查看流表
ovs-ofctl dump-flows br-int
```

**Linux Bridge**：Linux 内核内置的网桥，功能相对简单，适合小规模环境。KVM 默认使用 Linux Bridge（virbr0）。

## 2.6 虚拟机存储

### 2.6.1 存储类型
- **本地存储**：直接使用宿主机磁盘，性能好但不支持跨主机迁移
- **共享存储**：SAN/NAS，支持 VM 迁移，是生产环境的标准配置
- **分布式存储**：Ceph、GlusterFS，通过软件定义存储，可横向扩展

### 2.6.2 存储格式对比
| 格式 | 特点 | 使用场景 |
|------|------|---------|
| RAW | 性能最好，不支持快照 | 高性能数据库 |
| qcow2 | 支持快照、压缩、加密 | KVM 默认 |
| VMDK | VMware 原生格式 | VMware 环境 |

## 2.7 虚拟机迁移

### 2.7.1 冷迁移

冷迁移（Cold Migration）是指在虚拟机关机状态下将其从一个物理主机迁移到另一个物理主机。

**迁移过程**：
1. 关闭源主机上的虚拟机
2. 将虚拟机的磁盘文件复制到目标主机（通过 scp、rsync 或共享存储）
3. 将虚拟机的 XML 配置文件迁移到目标主机
4. 在目标主机上重新定义并启动虚拟机

**特点**：
- 实现简单，可靠性高
- 迁移过程中业务中断（停机时间取决于磁盘文件大小和网络带宽）
- 不要求源主机和目标主机共享存储
- 适合计划内的维护窗口操作

### 2.7.2 热迁移（Live Migration）

热迁移（Live Migration）是指在虚拟机持续运行、业务不中断的情况下将其从一个物理主机迁移到另一个物理主机。

**迁移原理（预拷贝机制，Pre-Copy）**：

```
阶段1: 迭代预拷贝（Iterative Pre-Copy）
  ┌─────────────────────────────────────────────┐
  │  源主机                      目标主机        │
  │  ┌──────────┐   全量内存拷贝   ┌──────────┐  │
  │  │ VM 内存  │ ──────────────→ │ VM 内存  │  │
  │  └──────────┘                └──────────┘  │
  │                                             │
  │  VM 继续运行，内存被修改的页面（脏页）        │
  │  需要重新拷贝，反复迭代直到脏页率足够低       │
  └─────────────────────────────────────────────┘

阶段2: 停机切换（Downtime）
  ┌─────────────────────────────────────────────┐
  │  暂停 VM（极短时间，通常 < 1 秒）             │
  │  拷贝剩余脏页                                │
  │  在目标主机上恢复 VM 运行                     │
  │  更新网络路由（ARP/NDP 更新）                 │
  └─────────────────────────────────────────────┘

阶段3: 清理
  ┌─────────────────────────────────────────────┐
  │  源主机释放 VM 资源                          │
  │  迁移完成                                    │
  └─────────────────────────────────────────────┘
```

**前提条件**：
- 源主机和目标主机必须共享存储（如 NFS、Ceph RBD、SAN），或使用块复制模式
- 目标主机 CPU 指令集必须兼容源主机（或启用 CPU 特性屏蔽）
- 网络带宽足够大，确保内存拷贝能在合理时间内完成
- 目标主机有足够的可用资源（CPU、内存）

**KVM 热迁移命令**：
```bash
# 实时迁移到目标主机
virsh migrate --live --persistent --undefinesource my-vm \
  qemu+ssh://dest-host/system

# 带压缩的迁移（适用于网络带宽有限的情况）
virsh migrate --live --compressive my-vm \
  qemu+ssh://dest-host/system

# 设置最大停机时间（毫秒）
virsh migrate --live --maxdowntime 500 my-vm \
  qemu+ssh://dest-host/system
```

## 2.8 面试常见问题

- Q1: Type 1 和 Type 2 Hypervisor 的区别？
  - **参考答案要点**：Type 1（裸金属型）直接运行在物理硬件上，无宿主 OS，性能接近裸机，安全性高，适合生产环境（如 KVM、ESXi、Hyper-V）。Type 2（宿主型）运行在宿主 OS 之上，作为应用程序安装，有宿主 OS 的性能开销，但安装简单，适合开发测试（如 VirtualBox、VMware Workstation）。云厂商全部使用 Type 1。

- Q2: KVM 和 Docker 的本质区别是什么？
  - **参考答案要点**：KVM 是硬件级虚拟化，通过 Hypervisor 模拟完整硬件，每个 VM 运行独立的 Guest OS，隔离性强（硬件级），但资源开销大。Docker 是操作系统级虚拟化（容器），共享宿主机内核，通过 Namespace 实现隔离、Cgroup 实现资源限制，轻量快速。KVM 适合需要不同 OS 或强隔离的场景，Docker 适合微服务和快速部署。

- Q3: 什么是热迁移？如何实现？
  - **参考答案要点**：热迁移（Live Migration）是在 VM 运行不中断的情况下迁移到另一台物理主机。KVM 采用预拷贝（Pre-Copy）机制：(1) 全量拷贝内存到目标主机；(2) 迭代拷贝期间产生的脏页；(3) 当脏页量足够小时短暂停机（通常 < 1 秒），拷贝最后的数据并在目标主机恢复运行。前提条件包括共享存储、CPU 兼容、足够的网络带宽。

- Q4: 为什么云厂商（AWS/阿里云）底层大多用 KVM？
  - **参考答案要点**：(1) KVM 是 Linux 内核原生模块，开源免费，无许可费用；(2) 性能优异，利用硬件虚拟化扩展接近原生性能；(3) 生态丰富，配合 QEMU、libvirt、VirtIO 形成完整方案；(4) 社区活跃，持续获得 Linux 社区的优化和功能更新；(5) 灵活性高，可以深度定制和优化。AWS 早期用 Xen，后迁移至基于 KVM 的 Nitro 架构。

- Q5: 虚拟机的隔离性比容器好在哪里？有什么代价？
  - **参考答案要点**：VM 通过 Hypervisor 实现硬件级隔离，每个 VM 有独立的内核和硬件资源，攻击面小，一个 VM 被攻破不影响其他 VM。容器共享宿主机内核，隔离基于 Linux Namespace 和 Cgroup，属于进程级隔离，存在内核漏洞导致容器逃逸的风险。代价方面：VM 资源开销大（每个 VM 需要独立的 Guest OS，占用 GB 级内存），启动慢（分钟级），密度低。容器开销小（MB 级），启动快（秒级），密度高。
