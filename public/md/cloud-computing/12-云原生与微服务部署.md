# 12 云原生与微服务部署

## 12.1 云原生概述

### 12.1.1 什么是云原生

云原生（Cloud Native）是 CNCF（Cloud Native Computing Foundation，云原生计算基金会）定义的一套构建和运行现代化应用程序的方法论和技术体系。

**CNCF 官方定义（v1.0，2024 年更新）：**

> 云原生技术使组织能够在现代动态环境（如公有云、私有云和混合云）中构建和运行可扩展的应用程序。容器、服务网格、微服务、不可变基础设施和声明式 API 是这种方法的典范。
>
> 这些技术使松散耦合的系统具有弹性、可管理和可观测性。结合强大的自动化，它们允许工程师以最小的工作量频繁且可预测地进行影响重大的变更。

**云原生的六大核心技术特征：**

| 特征 | 说明 | 代表技术 |
|------|------|---------|
| **容器化** | 应用打包为容器镜像，实现环境一致性 | Docker、containerd、CRI-O |
| **微服务** | 应用拆分为独立部署的小型服务 | Spring Boot、Go Micro、gRPC |
| **Service Mesh** | 服务间通信的基础设施层 | Istio、Linkerd、Consul Connect |
| **不可变基础设施** | 服务器一旦部署不再修改，变更通过替换实现 | 容器镜像、AMI |
| **声明式 API** | 通过声明期望状态，由系统自动收敛 | Kubernetes CRD、Terraform |
| **DevOps / GitOps** | 开发运维一体化，Git 作为唯一事实来源 | ArgoCD、Flux、GitHub Actions |

**云原生的演进历程：**

```
传统架构 → 虚拟化 → 容器化 → 微服务 → Service Mesh → Serverless → 云原生
                                                        │
                                                        ▼
                                              CNCF 云原生全景图
```

## 12.2 12-Factor App（十二要素应用）

### 12.2.1 十二要素清单
| # | 要素 | 核心原则 |
|---|------|---------|
| 1 | 基准代码 | 一份代码库，多份部署 |
| 2 | 依赖 | 显式声明依赖 |
| 3 | 配置 | 环境变量存储配置 |
| 4 | 后端服务 | 把后端服务当作附加资源 |
| 5 | 构建、发布、运行 | 严格分离三个阶段 |
| 6 | 进程 | 以一个或多个无状态进程运行 |
| 7 | 端口绑定 | 通过端口绑定提供服务 |
| 8 | 并发 | 通过进程模型进行扩展 |
| 9 | 易处置 | 快速启动和优雅终止 |
| 10 | 开发/生产等价 | 保持环境一致性 |
| 11 | 日志 | 把日志当作事件流 |
| 12 | 管理进程 | 后台管理任务一次性运行 |

### 12.2.2 实践示例

**Factor 1: 基准代码（Codebase）**

一份代码库通过 Git 管理，通过分支实现多份部署：

```
myapp/                    # 单一 Git 仓库
├── app/                  # 应用代码
├── config/               # 配置模板
├── deploy/               # 部署配置（K8s manifests）
├── tests/                # 测试代码
└── scripts/              # 构建/部署脚本

部署映射：
  main 分支     → 生产环境（prod）
  staging 分支  → 预发环境（staging）
  dev 分支      → 开发环境（dev）
```

显式声明依赖，不依赖系统级包：


```dockerfile
FROM python:3.12-slim
COPY requirements.txt .
```

配置通过环境变量注入，不硬编码到代码中：

**Factor 4: 后端服务（Backing Services）**

后端服务（数据库、缓存、消息队列等）通过 URL 配置，可随时替换：

```yaml
# 开发环境使用本地 SQLite
env:
  - name: DATABASE_URL
    value: "sqlite:///./dev.db"

# 生产环境使用 PostgreSQL
env:
  - name: DATABASE_URL
    valueFrom:
      secretKeyRef:
        name: myapp-secrets
        key: DATABASE_URL
```

**Factor 5: 构建、发布、运行（Build, Release, Run）**

三个阶段严格分离：

```
构建（Build）：
  代码 + 依赖 → 可执行包（Docker 镜像）
  产物：myapp:sha-abc123（不可变）

发布（Release）：
  构建产物 + 配置 → 可运行单元
  产物：myapp:v2.5.0-prod（包含特定配置的镜像版本）

运行（Run）：
  在执行环境中启动应用
  操作：kubectl apply、docker run
```

```bash
# 构建阶段
docker build -t myapp:sha-$(git rev-parse --short HEAD) .
docker push myapp:sha-abc123

# 发布阶段（不可变标签）
docker tag myapp:sha-abc123 myapp:v2.5.0
docker push myapp:v2.5.0

# 运行阶段
kubectl set image deployment/myapp myapp=myapp:v2.5.0
```

**Factor 6: 进程（Processes）**

应用以无状态进程运行，状态外置到后端服务：

```python
# 错误做法：将状态存储在本地文件/内存
sessions = {}  # 内存中的 session 存储

# 正确做法：使用 Redis 存储状态
import redis
r = redis.from_url(os.environ["REDIS_URL"])

def set_session(session_id, data):
    r.setex(f"session:{session_id}", 3600, json.dumps(data))

def get_session(session_id):
    data = r.get(f"session:{session_id}")
    return json.loads(data) if data else None
```

```yaml
        - name: myapp
          image: myapp:v2.5.0
```

```yaml
  name: myapp-hpa

    name: myapp
```

**Factor 9: 易处置（Disposability）**

快速启动和优雅终止：

```python
# 优雅终止处理
import signal
import sys

shutdown = False

def handle_shutdown(signum, frame):
    global shutdown
    shutdown = True
    logger.info("收到终止信号，开始优雅关闭...")

signal.signal(signal.SIGTERM, handle_shutdown)
signal.signal(signal.SIGINT, handle_shutdown)

# 主循环中检查 shutdown 标志
while not shutdown:
    process_request()
```

```yaml
# Kubernetes 优雅终止配置
spec:
  terminationGracePeriodSeconds: 30
  containers:
    - name: myapp
      lifecycle:
        preStop:
          exec:
            command: ["/bin/sh", "-c", "sleep 5"]  # 等待 5 秒再终止
      # 就绪探针 - 终止前从 LB 摘除
      readinessProbe:
        httpGet:
          path: /ready
          port: 8080
        periodSeconds: 3
```

## 12.3 Service Mesh（服务网格）

### 12.3.1 什么是 Service Mesh

Service Mesh（服务网格）是一个专门处理服务间通信（东西向流量）的基础设施层，通常以 Sidecar 代理的形式部署在每个服务实例旁边，透明地拦截和治理所有服务间网络流量。

**核心能力：**

| 能力 | 说明 | 典型功能 |
|------|------|---------|
| **流量管理** | 控制服务间请求的路由和分发 | 负载均衡、流量分割、重试、超时、熔断、故障注入 |
| **安全** | 服务间认证和授权 | mTLS（双向 TLS）、服务身份、RBAC |
| **可观测性** | 自动收集服务间通信指标 | 分布式追踪、指标采集、访问日志 |

**为什么需要 Service Mesh：**

在微服务架构中，服务间通信的复杂性（重试、超时、熔断、认证、追踪等）如果由每个服务自行实现，会导致大量重复代码且难以统一管理。Service Mesh 将这些横切关注点从业务代码中剥离出来，下沉到基础设施层。

```
没有 Service Mesh：
  每个服务自行实现：重试逻辑、超时处理、熔断器、TLS 配置、日志记录、追踪注入
  → 大量重复代码、不一致的行为、难以维护

有 Service Mesh：
  业务代码只关注业务逻辑
  Sidecar 代理透明处理：重试、超时、熔断、mTLS、追踪、日志
  → 统一治理、一致性、可观测性
```

### 12.3.2 Sidecar 模式
```
+-------------------+     +-------------------+
|     Pod A          |     |     Pod B          |
| +-------+ +-----+ |     | +-------+ +-----+ |
| | App   | |Side | |     | | App   | |Side | |
| |       | |car  | |     | |       | |car  | |
| +---+---+ +--+--+ |     | +---+---+ +--+--+ |
|     |        |     |     |     |        |     |
+-----+--------+-----+     +-----+--------+-----+
      |        |                 |        |
      +--------+-----------------+--------+
                    Service Mesh
```

### 12.3.3 Istio

Istio 是目前最流行、功能最丰富的 Service Mesh 实现，由 Google、IBM 和 Lyft 联合开发，现为 CNCF 毕业项目。

**Istio 1.22+ 关键特性：**

- **Ambient Mesh**：无 Sidecar 模式（ztunnel + waypoint proxy），大幅降低资源开销
- **Gateway API**：原生支持 Kubernetes Gateway API（替代 Ingress）
- **多集群改进**：简化多集群服务网格的配置和管理
- **eBPF 支持**：使用 eBPF 加速数据平面
- **K8s Gateway API GA**：Gateway API 成为推荐的入口流量管理方式

#### Istio 核心组件

- **Envoy Proxy**：Sidecar 代理（数据平面）
- **Istiod**：控制平面（Pilot + Citadel + Galley 合并为单一组件）
- **Gateway**：入口/出口网关

#### Istio 功能

- 流量管理（路由、重试、超时、熔断）
- 安全（mTLS、认证、授权）
- 可观测性（指标、追踪、日志）

**Istio 完整流量管理配置示例：**

```yaml
# ============================================
# 1. 启用自动 Sidecar 注入
# ============================================
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    istio-injection: enabled  # 自动注入 Envoy Sidecar

# ============================================
# 2. 虚拟服务（VirtualService）- 流量路由
# ============================================
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
  namespace: production
spec:
  hosts:
    - myapp
  gateways:
    - myapp-gateway  # 通过 Gateway 暴露
  http:
    # --- 基于请求头的路由 ---
    - match:
        - headers:
            x-user-type:
              exact: "beta"
      route:
        - destination:
            host: myapp
            subset: v2  # Beta 用户路由到 v2
    # --- 基于权重的路由（金丝雀发布）---
    - route:
        - destination:
            host: myapp
            subset: v1
          weight: 90  # 90% 流量到 v1
        - destination:
            host: myapp
            subset: v2
          weight: 10  # 10% 流量到 v2（金丝雀）
    # --- 重试策略 ---
    retries:
      attempts: 3
      perTryTimeout: 2s
      retryOn: "5xx,reset,connect-failure,refused-stream"
    # --- 超时配置 ---
    timeout: 10s
    # --- 故障注入（测试容错能力）---
    fault:
      abort:
        percentage:
          value: 1  # 1% 的请求返回错误
        httpStatus: 500
    # --- 流量镜像（将流量复制到测试环境）---
    - mirror:
        host: myapp-mirror
      mirrorPercentage:
        value: 100

# ============================================
# 3. 目标规则（DestinationRule）- 负载均衡与熔断
# ============================================
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp
  namespace: production
spec:
  host: myapp
  # --- 流量策略（连接池和熔断）---
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 1000        # 最大连接数
      http:
        h2UpgradePolicy: DEFAULT
        h1MaxPendingRequests: 1024   # HTTP/1.1 最大等待请求数
        h2MaxPendingRequests: 1024   # HTTP/2 最大等待请求数
        maxRequestsPerConnection: 10 # 每连接最大请求数
        maxRetries: 3
        idleTimeout: 60s
    outlierDetection:  # 熔断器配置
      consecutive5xxErrors: 5         # 连续 5 次 5xx 错误触发熔断
      interval: 30s                   # 检测间隔
      baseEjectionTime: 30s           # 基础驱逐时间
      maxEjectionPercent: 50          # 最多驱逐 50% 的实例
      minHealthPercent: 25            # 最少保持 25% 的健康实例
    tls:
      mode: ISTIO_MUTUAL              # mTLS 加密
  # --- 版本子集 ---
  subsets:
    - name: v1
      labels:
        version: v1
      trafficPolicy:
        connectionPool:
          http:
            maxRequestsPerConnection: 5
    - name: v2
      labels:
        version: v2

# ============================================
# 4. Gateway - 入口网关（Gateway API）
```

### 12.3.4 Linkerd

Linkerd 是一个轻量级 Service Mesh，是 CNCF 的第二个毕业项目，专注于简洁、安全和易用。

**Linkerd 2.15+ 关键特性：**

- **超轻量**：Sidecar 代理（linkerd2-proxy）使用 Rust 编写，内存占用约 10-20MB（Istio Envoy 约 100MB+）
- **自动 mTLS**：开箱即用的双向 TLS 加密，无需额外配置
- **零配置注入**：`linkerd inject` 自动注入 Sidecar，无需复杂配置
- **Gateway API**：支持 Kubernetes Gateway API
- **调试工具**：内置 `linkerd viz`、`linkerd jaeger` 可视化插件

**Linkerd vs Istio 对比：**

| 维度 | Linkerd | Istio |
|------|---------|-------|
| **性能** | 轻量（Rust 代理，~10MB 内存） | 较重（Envoy 代理，~100MB 内存） |
| **复杂度** | 简单，开箱即用 | 复杂，功能丰富 |
| **流量管理** | 基础（路由、重试、超时） | 高级（流量镜像、故障注入、流量分割） |
| **可观测性** | 内置（viz 插件） | 需要集成（Kiali、Jaeger、Grafana） |
| **多集群** | 支持（Gateway 模式） | 支持（更完善） |
| **适用场景** | 中小规模、追求简洁 | 大规模、需要高级流量治理 |
| **Sidecar 代理** | linkerd2-proxy（Rust） | Envoy（C++） |
| **CNCF 状态** | 毕业项目 | 毕业项目 |

**Linkerd 快速安装示例：**

## 12.4 Serverless

### 12.4.1 Serverless 概述

Serverless（无服务器计算）是一种云计算执行模型，云厂商动态管理服务器的分配和伸缩，开发者无需关心底层基础设施，只需关注业务逻辑代码。

**核心理念：**

- **按需执行**：代码仅在事件触发时运行，无请求时不产生费用
- **自动伸缩**：从 0 到 N 自动伸缩，无需配置容量
- **零运维**：无需管理服务器、操作系统、运行时
- **按使用付费**：按执行次数和执行时间计费（精确到毫秒）

**Serverless 的两种形态：**

| 形态 | 说明 | 代表 |
|------|------|------|
| **FaaS**（Function as a Service） | 事件触发的无状态函数 | AWS Lambda、Azure Functions |
| **BaaS**（Backend as a Service） | 托管的后端服务 | DynamoDB、Cognito、S3 |

### 12.4.2 FaaS（Function as a Service）
| 平台 | 触发器 | 运行时 |
|------|--------|--------|
| AWS Lambda | API Gateway、S3、SQS | Python、Node.js、Java |
| Azure Functions | HTTP、Timer、Blob | C#、Python、JS |
| Google Functions | HTTP、Pub/Sub | Python、Node.js、Go |
| Cloudflare Workers | HTTP | JS、WASM |

### 12.4.3 Serverless 架构模式
- **API Backend**：API Gateway + Lambda + DynamoDB
- **事件驱动**：S3 → Lambda → SNS → Lambda
- **数据处理**：Kinesis → Lambda → S3 → Athena
- **定时任务**：CloudWatch Events → Lambda

### 12.4.4 Serverless 的优缺点
| 优点 | 缺点 |
|------|------|
| 零运维 | 冷启动延迟 |
| 按需付费 | 执行时间限制 |
| 自动伸缩 | 调试困难 |
| 快速开发 | 厂商锁定 |

### 12.4.5 BaaS（Backend as a Service）

BaaS（Backend as a Service）提供托管的后端服务，开发者无需自行构建和维护后端基础设施。

**主要 BaaS 服务：**

| 服务类别 | AWS | Google Cloud | Azure | 开源替代 |
|---------|-----|-------------|-------|---------|
| **认证** | Cognito | Identity Platform | Active Directory B2C | Keycloak、Auth0 |
| **数据库** | DynamoDB | Firestore | Cosmos DB | Supabase、Appwrite |
| **存储** | S3 | Cloud Storage | Blob Storage | MinIO |
| **实时通信** | API Gateway WebSocket | Firebase Realtime DB | SignalR | Socket.io |
| **搜索** | OpenSearch | Algolia | Cognitive Search | Meilisearch |
| **推送通知** | SNS + Pinpoint | Firebase Cloud Messaging | Notification Hubs | - |

**Firebase 示例（全栈 BaaS）：**

```javascript
// Firebase 实时数据库 + 认证 + 托管
import { initializeApp } from "firebase/app";
import { getAuth, signInWithEmailAndPassword } from "firebase/auth";
import { getDatabase, ref, set, onValue } from "firebase/database";

const app = initializeApp({
  apiKey: "AIza...",
  authDomain: "myapp.firebaseapp.com",
  databaseURL: "https://myapp-default-rtdb.firebaseio.com",
  projectId: "myapp",
});

// 认证
const auth = getAuth(app);
const user = await signInWithEmailAndPassword(auth, email, password);

// 实时数据库
const db = getDatabase(app);
await set(ref(db, `users/${user.uid}`), { name: "Alice", email });

// 实时监听数据变化
onValue(ref(db, "messages"), (snapshot) => {
  const data = snapshot.val();
  console.log("实时数据:", data);
});
```

**Supabase 示例（开源 BaaS 替代）：**

```javascript
import { createClient } from "@supabase/supabase-js";

const supabase = createClient(
  "https://xxx.supabase.co",
  "eyJhbGciOiJIUzI1NiIs..."
);

// 认证
const { data, error } = await supabase.auth.signUp({
  email: "user@example.com",
  password: "secure-password",
});

// 数据库查询（自动生成 REST API）
const { data: posts } = await supabase

  .subscribe();
```

## 12.5 不可变基础设施

### 12.5.1 概念

不可变基础设施（Immutable Infrastructure）是一种运维模式，服务器一旦部署就不再修改。任何配置变更、补丁更新、应用升级都通过构建新的服务器实例（或容器镜像）并替换旧实例来实现，而非在现有实例上修改。

**核心理念：**

- **替换而非修改**：需要变更时，构建新实例替换旧实例
- **基础设施即版本化产物**：每次部署产生一个不可变的版本（如容器镜像、AMI）
- **一致性保证**：同一版本在任何环境运行结果完全一致
- **快速回滚**：回滚只需切换到旧版本，无需"撤销变更"

**可变 vs 不可变对比：**

| 维度 | 可变基础设施 | 不可变基础设施 |
|------|------------|--------------|
| 变更方式 | SSH 登录修改 | 构建新镜像/AMI 替换 |
| 一致性 | 难以保证（配置漂移） | 天然保证（同一镜像） |
| 回滚 | 需要手动撤销变更 | 切换到旧版本 |
| 调试 | 可以登录服务器排查 | 通过日志和监控排查 |
| 审计 | 难以追踪手动变更 | Git 历史记录所有变更 |

### 12.5.2 蓝绿部署

蓝绿部署（Blue-Green Deployment）通过维护两套完全相同的生产环境（蓝和绿），通过切换流量实现零停机部署。

```
    +------+
    |  LB  |
    +--+---+
       |
  +----+----+
  |  Blue   |  ← 当前版本（v1）
  +---------+
  |  Green  |  ← 新版本（v2，待切换）
  +---------+
```

**Kubernetes 蓝绿部署实现：**

```yaml
# ============================================
# 方案 1: 使用 Service selector 切换
# ============================================

# Blue Deployment（当前版本）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      slot: blue
  template:
    metadata:
      labels:
        app: myapp
        slot: blue
        version: "1.0.0"
    spec:
      containers:
        - name: myapp
          image: myapp:1.0.0
          ports:
            - containerPort: 8080

---
# Green Deployment（新版本）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      slot: green
  template:
    metadata:
      labels:
        app: myapp
        slot: green
        version: "2.0.0"
    spec:
      containers:
        - name: myapp
          image: myapp:2.0.0
          ports:
            - containerPort: 8080

---
# Service - 通过修改 selector 切换流量
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
    slot: blue  # 切换到 green 即可完成蓝绿部署
  ports:
    - port: 80
      targetPort: 8080

# 切换命令：
# kubectl patch service myapp -p '{"spec":{"selector":{"slot":"green"}}}'
# 回滚命令：
# kubectl patch service myapp -p '{"spec":{"selector":{"slot":"blue"}}}'
```

```yaml
# ============================================
# 方案 2: 使用 Istio VirtualService 实现蓝绿部署
# ============================================
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
    - myapp
  http:
    - route:
        - destination:
            host: myapp-blue
          weight: 100  # 所有流量到 Blue
        - destination:
            host: myapp-green
          weight: 0    # Green 无流量（预热状态）

# 切换到 Green：
# 将 weight 改为 blue: 0, green: 100
```

### 12.5.3 金丝雀发布

金丝雀发布（Canary Release）是一种渐进式发布策略，先将新版本部署到少量实例，逐步增加新版本的流量比例，观察指标正常后再全量发布。

**Kubernetes 金丝雀发布实现（使用 Istio）：**

```yaml
# ============================================
# 1. 两个版本的 Deployment
# ============================================
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-v1
spec:
  replicas: 4
  selector:
    matchLabels:
      app: myapp
      version: v1
  template:
    metadata:
      labels:
        app: myapp
        version: v1
    spec:
      containers:
        - name: myapp
          image: myapp:1.0.0
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080

---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-v2
spec:
  replicas: 1  # 金丝雀：先部署 1 个副本
  selector:
    matchLabels:
      app: myapp
      version: v2
  template:
    metadata:
      labels:
        app: myapp
        version: v2
    spec:
      containers:
        - name: myapp
          image: myapp:2.0.0
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080

# ============================================
# 2. Istio VirtualService - 流量分配
# ============================================
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
    - myapp
  http:
    - route:
        - destination:
            host: myapp
            subset: v1
          weight: 95  # 95% 流量到 v1
        - destination:
            host: myapp
            subset: v2
          weight: 5   # 5% 流量到 v2（金丝雀）

# ============================================
# 3. DestinationRule - 版本子集定义
# ============================================
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: myapp
spec:
  host: myapp
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2

# ============================================
# 4. 金丝雀发布流程（逐步增加流量）
# ============================================
# Step 1: 5% 流量到 v2，观察 30 分钟
# Step 2: 15% 流量到 v2，观察 30 分钟
# Step 3: 30% 流量到 v2，观察 1 小时
# Step 4: 50% 流量到 v2，观察 2 小时
# Step 5: 100% 流量到 v2，完成发布
```

### 12.5.4 滚动更新

滚动更新（Rolling Update）是 Kubernetes Deployment 的默认更新策略，逐步用新版本 Pod 替换旧版本 Pod，保证在更新过程中始终有足够数量的 Pod 可用。

**Kubernetes RollingUpdate 配置：**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 滚动更新时最多多创建 1 个 Pod（最多 4+1=5 个 Pod）
      maxUnavailable: 0  # 滚动更新时不允许有 Pod 不可用
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
        - name: myapp
          image: myapp:2.0.0
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 3
          lifecycle:
            preStop:
              exec:
                command: ["/bin/sh", "-c", "sleep 10"]  # 优雅终止
```

**滚动更新过程：**

```
初始状态（v1）：
  Pod-1(v1)  Pod-2(v1)  Pod-3(v1)  Pod-4(v1)

Step 1: 创建新 Pod（maxSurge=1）：
  Pod-1(v1)  Pod-2(v1)  Pod-3(v1)  Pod-4(v1)  Pod-5(v2) ← 新建

Step 2: 新 Pod 就绪后，删除旧 Pod（maxUnavailable=0）：
  Pod-1(v1)  Pod-2(v1)  Pod-3(v1)  [Pod-4 删除]  Pod-5(v2)

Step 3: 继续创建新 Pod：
  Pod-1(v1)  Pod-2(v1)  Pod-3(v1)  Pod-6(v2) ← 新建  Pod-5(v2)

Step 4: 继续替换：
  Pod-1(v1)  Pod-2(v1)  [Pod-3 删除]  Pod-6(v2)  Pod-5(v2)

... 重复直到所有 Pod 替换完成 ...

最终状态（v2）：
  Pod-7(v2)  Pod-8(v2)  Pod-9(v2)  Pod-10(v2)
```

**RollingUpdate vs Recreate 对比：**

| 维度 | RollingUpdate | Recreate |
|------|--------------|----------|
| **停机时间** | 零停机（新旧 Pod 并存） | 有停机（先删所有旧 Pod） |
| **资源需求** | 需要额外资源（maxSurge） | 不需要额外资源 |
| **回滚速度** | 快（旧 Pod 还在） | 慢（需要重新创建） |
| **适用场景** | 无状态服务 | 有状态服务、数据库迁移 |

## 12.6 GitOps

### 12.6.1 GitOps 原则

1. 声明式配置
2. 存储在 Git 中
3. 自动拉取（Pull 模式）
4. 持续协调

### 12.6.2 ArgoCD

ArgoCD 是一个 Kubernetes 原生的 GitOps 持续交付工具，由 Intuit 开发，现为 CNCF 毕业项目。它监控 Git 仓库中的 Kubernetes 配置变更，自动将变更同步到集群。

**ArgoCD 2.12+ 关键特性：**

- **ApplicationSet**：支持多集群、多环境的应用批量管理
- **Application in any namespace**：应用资源可以部署到任意命名空间
- **Redis HA**：内置 Redis 高可用支持
- **权限改进**：更细粒度的 RBAC 控制
- **Notifications**：内置通知控制器（Slack、Teams、Webhook 等）

**ArgoCD 架构：**

```
Git 仓库（GitHub/GitLab）
    │
    │ ArgoCD Application Controller（持续监控）
    ▼
ArgoCD Server（API Server + Web UI）
    │
    │ ArgoCD Application Controller
    ▼
Kubernetes Cluster（应用配置同步）
    │
    ├── Namespace A: myapp-dev
    ├── Namespace B: myapp-staging
    └── Namespace C: myapp-prod
```

**ArgoCD Application 配置示例：**

```yaml
# ============================================
# ArgoCD Application - 单环境
# ============================================
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-production
  namespace: argocd
  finalizers:
    - resources-finalizer.argocd.argoproj.io
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/myapp-manifests.git
    targetRevision: main
    path: overlays/production  # Kustomize overlay
    # 或者使用 Helm Chart:
    # chart: myapp
    # repoURL: https://charts.myorg.io
    # targetRevision: "2.5.0"
    # helm:
    #   values: |
    #     replicaCount: 3
    #     image:
    #       tag: "2.5.0"
  destination:
    server: https://kubernetes.default.svc
    namespace: production
  syncPolicy:
```

## 12.7 Dapr（分布式应用运行时）

Dapr（Distributed Application Runtime）是一个可移植的、事件驱动的运行时，使开发者能够轻松构建跨平台、高可用的微服务应用。它提供了一系列标准的 API 构建块，抽象了分布式系统的复杂性。

**Dapr 1.14+ 关键特性：**

- **Workflow API**：内置工作流引擎（基于 Dapr Workflow），替代外部工作流引擎
- **Service Invocation HTTP/1.1**：改进的服务调用性能
- **Pluggable Components**：可插拔组件架构，支持自定义组件
- **Hot Reload**：配置热重载，无需重启 Dapr Sidecar
- **改进的 Observability**：增强的指标和追踪支持

**Dapr 构建块（Building Blocks）：**

| 构建块 | 说明 | 典型用途 |
|--------|------|---------|
| **Service Invocation** | 服务间调用（HTTP/gRPC） | REST API 调用、gRPC 调用 |
| **State Management** | 状态管理 | 分布式状态存储、并发控制 |
| **Publish & Subscribe** | 发布/订阅消息 | 事件驱动架构、异步通信 |
| **Bindings** | 输入/输出绑定 | 连接外部系统（数据库、消息队列） |
| **Secrets** | 密钥管理 | 从密钥存储中获取密钥 |
| **Configuration** | 配置管理 | 从配置存储中获取配置 |
| **Actors** | 虚拟 Actor 模型 | 有状态微服务、并发处理 |
| **Distributed Lock** | 分布式锁 | 分布式互斥、资源竞争 |
| **Workflow** | 工作流编排 | 长时间运行的业务流程 |
| **Resiliency** | 弹性策略 | 重试、超时、熔断 |

**Dapr 架构：**

```
+--------------------------------------------------+
|                    Pod                           |
|  +----------------+     +---------------------+  |
|  |   Application  |     |   Dapr Sidecar      |  |
|  |                |     |                     |  |
|  |  HTTP :8080    |<--->|  Dapr API           |  |
|  |                |     |  :3500 (HTTP)        |  |
|  |                |     |  :50001 (gRPC)       |  |
|  +----------------+     +----------+----------+  |
|                                    |             |
+------------------------------------|-------------+
                                     |
                    +----------------+----------------+
                    |       Dapr Components         |
                    |  State Store: Redis/PostgreSQL |
                    |  Pub/Sub: Kafka/RabbitMQ      |
                    |  Secret Store: Vault/AWS SM    |
                    |  Bindings: S3/SQS/SNS          |
                    +--------------------------------+
```

## 12.8 面试常见问题

**Q1: 什么是云原生？包含哪些核心技术？**

**参考答案：**

云原生是 CNCF 定义的一套构建和运行现代化应用程序的方法论和技术体系。其核心目标是使应用能够在动态的云环境中弹性、可管理、可观测地运行。

**六大核心技术：**

1. **容器化**：使用容器（Docker/containerd）打包应用和依赖，实现环境一致性
2. **微服务**：将应用拆分为独立部署的小型服务，通过 API 通信
3. **Service Mesh**：使用 Istio/Linkerd 等工具管理服务间通信（mTLS、流量治理、可观测性）
4. **不可变基础设施**：通过替换实例（而非修改）实现变更，保证一致性
5. **声明式 API**：通过声明期望状态（如 Kubernetes YAML），由系统自动收敛
6. **DevOps/GitOps**：开发运维一体化，Git 作为配置的唯一事实来源

**CNCF 全景图覆盖的技术分类：**
- 应用定义与开发（Helm、Dapr、Backstage）
- 编排与管理（Kubernetes、Crossplane、Karmada）
- 运行时（containerd、Cilium、Envoy）
- 配置与供应（Terraform、ArgoCD、Flux）
- 可观测性（Prometheus、Grafana、OpenTelemetry）
- Serverless（Knative、KEDA）

---

**Q2: 12-Factor App 的核心原则是什么？**

**参考答案：**

12-Factor App 是一套构建 SaaS 应用的方法论，核心原则是使应用与底层操作系统解耦，便于在云环境中部署和扩展。

**最重要的几个要素：**

1. **配置与环境分离（Factor 3）**：所有配置通过环境变量注入，不硬编码到代码中。这使得同一份代码可以在不同环境（dev/staging/prod）中运行。

2. **无状态进程（Factor 6）**：应用不应在内存或本地文件中存储状态，所有状态应外置到后端服务（Redis、PostgreSQL 等）。这使得应用可以随时启停和水平扩展。

3. **构建、发布、运行分离（Factor 5）**：构建阶段生成不可变产物（容器镜像），发布阶段将产物与配置组合，运行阶段启动应用。每个阶段严格分离。

4. **易处置（Factor 9）**：应用应支持快速启动和优雅终止。通过 SIGTERM 信号处理、preStop hook 等机制实现优雅关闭。

5. **日志作为事件流（Factor 11）**：日志输出到 stdout/stderr，由外部系统（Fluentd/Fluent Bit）收集处理，而非写入本地文件。

---

**Q3: Service Mesh 和 API Gateway 的区别？**

**参考答案：**

| 维度 | Service Mesh | API Gateway |
|------|-------------|-------------|
| **流量方向** | 东西向（服务到服务） | 南北向（外部到内部） |
| **部署模式** | Sidecar（每个 Pod） | 独立网关（集群入口） |
| **核心功能** | mTLS、负载均衡、熔断、重试、追踪 | 认证/授权、限流、请求转换、API 版本管理 |
| **透明性** | 对应用透明 | 需要配置路由规则 |
| **协议支持** | gRPC、HTTP/2、TCP | HTTP/HTTPS、WebSocket |

**关键区别：**
- Service Mesh 解决的是**服务间通信**的治理问题（东西向流量）
- API Gateway 解决的是**外部请求入口**的治理问题（南北向流量）
- 两者是**互补关系**，不是替代关系

**实际架构中通常同时使用：**
```
外部请求 → API Gateway（认证/限流）→ Service Mesh（mTLS/熔断/追踪）→ 后端服务
```

---

**Q4: Istio 的 Sidecar 代理是如何工作的？**

**参考答案：**

Istio 使用 Envoy 作为 Sidecar 代理，通过 iptables（或 eBPF）劫持 Pod 中所有网络流量，透明地拦截所有入站和出站请求。

**工作原理：**

1. **流量劫持**：Istio 的 `istio-init` Init 容器通过配置 iptables 规则，将 Pod 中所有入站和出站的 TCP 流量重定向到 Envoy 代理的端口（15001/15006）。

2. **出站流量**：
   - 应用发出请求 → iptables 劫持 → Envoy Sidecar 拦截
   - Envoy 查找路由规则（VirtualService）确定目标
   - Envoy 执行策略（重试、超时、熔断、mTLS）
   - Envoy 转发请求到目标服务的 Sidecar

3. **入站流量**：
   - 外部请求到达 Pod → iptables 劫持 → Envoy Sidecar 拦截
   - Envoy 执行安全策略（认证、授权）
   - Envoy 转发请求到应用容器

4. **控制平面**：
   - Istiod（控制平面）通过 xDS API（CDS/EDS/LDS/RDS/SDS）将配置推送到所有 Envoy 代理
   - 配置包括：路由规则、服务发现、证书、安全策略等

**性能影响：**
- 增加约 1-2ms 的延迟（Sidecar 代理处理）
- 每个额外约 100MB 内存（Envoy 代理）
- 可通过 Ambient Mesh（无 Sidecar 模式）降低开销

---

**Q5: Serverless 的冷启动是什么？如何优化？**

**参考答案：**

**冷启动（Cold Start）**是指当函数长时间未被执行后，云厂商回收了执行容器。下次触发时需要重新初始化运行环境（下载代码、加载依赖、初始化框架），导致请求延迟显著增加。

**冷启动时间：**
- AWS Lambda：Node.js/Python 约 100-500ms，Java 约 1-5 秒
- 冷启动时间取决于：运行时语言、代码包大小、依赖数量、VPC 配置

**优化策略：**

1. **保持函数温暖**：
   - 使用定时触发器（CloudWatch Events）定期 ping 函数
   - 使用 Warmer 工具（如 serverless-plugin-warmup）

2. **选择轻量运行时**：
   - Python/Node.js 冷启动快于 Java/.NET
   - Go 编译为二进制，冷启动极快

3. **减小部署包大小**：
   - 使用 Lambda Layers 分离依赖
   - 仅打包必要文件
   - 使用 minify 压缩代码

4. **预置并发（Provisioned Concurrency）**：
   ```json
   {
     "ProvisionedConcurrentExecutions": 10
   }
   ```
   预先初始化指定数量的执行环境，消除冷启动

5. **使用 Knative（K8s Serverless）**：
   - Knative 的 Scale-to-Zero 和 Scale-from-Zero 策略可配置
   - 使用 KPA（Knative Pod Autoscaler）智能缩放

6. **架构优化**：
   - 将初始化逻辑放在全局范围（handler 外部）
   - 使用连接池复用数据库连接
   - 避免在 handler 中初始化重资源

---

**Q6: 蓝绿部署和金丝雀发布的区别？**

**参考答案：**

| 维度 | 蓝绿部署 | 金丝雀发布 |
|------|---------|-----------|
| **流量切换** | 一次性全量切换 | 渐进式增加流量 |
| **风险** | 较高（全量切换） | 较低（逐步验证） |
| **资源需求** | 需要 2 倍资源 | 额外资源较少 |
| **回滚速度** | 即时（切换 selector） | 即时（调整权重） |
| **验证方式** | 切换后整体验证 | 切换过程中持续验证 |
| **适用场景** | 大版本升级、紧急修复 | 新功能发布、A/B 测试 |

**蓝绿部署优势：**
- 零停机、即时回滚
- 可以在 Green 环境充分测试后再切换

**金丝雀发布优势：**
- 风险可控，逐步放量
- 可以基于指标（错误率、延迟）自动决策
- 支持 A/B 测试

**Kubernetes 中的实现：**
- 蓝绿部署：两个 Deployment + Service selector 切换
- 金丝雀发布：两个 Deployment + Istio VirtualService 权重调整

**Q7: 什么是 GitOps？有什么优势？**

**参考答案：**

GitOps 是一种将 Git 作为基础设施和应用配置"唯一事实来源"的运维方法论。

**四大核心原则：**
1. **声明式**：所有系统状态用声明式语言描述（Kubernetes YAML）
2. **存储在 Git 中**：所有配置存储在 Git 仓库
3. **自动拉取（Pull 模式）**：Agent（ArgoCD/Flux）从 Git 拉取配置并应用
4. **持续协调**：Agent 持续监控集群状态，自动纠正漂移

**优势：**
1. **变更可审计**：所有变更通过 Git PR/MR，具备完整的变更历史和审批记录
2. **回滚简单**：Git Revert 即可回滚，无需复杂的回滚脚本
3. **安全性高**：CI/CD 无需集群管理员权限，Agent 以只读方式访问 Git
4. **自愈能力**：Agent 持续协调，自动纠正配置漂移（如有人手动修改了 K8s 资源）
5. **多环境一致**：通过 Kustomize/Helm overlay 管理不同环境的差异
6. **团队协作**：通过 Git PR 流程实现团队协作和 Code Review

**代表工具：**
- **ArgoCD**：功能丰富，Web UI 友好，支持多集群
- **Flux**：轻量、模块化、CNCF 毕业项目

**Q8: 不可变基础设施的概念？为什么推荐？**

**参考答案：**

不可变基础设施（Immutable Infrastructure）是一种运维模式，服务器一旦部署就不再修改。任何变更都通过构建新实例（容器镜像/AMI）并替换旧实例来实现。

**为什么推荐不可变基础设施：**

1. **一致性保证**：同一镜像在任何环境运行结果完全一致，消除"在我机器上能跑"问题
2. **消除配置漂移**：不存在"手动修改服务器"的情况，所有变更通过版本化的镜像管理
3. **快速回滚**：回滚只需切换到旧版本镜像，无需"撤销变更"或"恢复配置"
4. **可审计**：每个版本对应一个 Git commit，变更历史完整可追溯
5. **可测试**：可以在部署前对镜像进行完整测试（安全扫描、集成测试）
6. **简化运维**：无需编写复杂的配置管理脚本（如 Ansible Playbook）

**不可变基础设施的实现方式：**
- **容器**：Docker 镜像是最典型的不可变基础设施
- **虚拟机**：使用 Packer 构建 AMI，通过 Auto Scaling Group 替换实例
- **Kubernetes**：Deployment 的滚动更新策略天然支持不可变部署

**与可变基础设施的对比：**
- 可变：SSH 登录服务器修改配置 → 配置漂移、难以追踪、回滚困难
- 不可变：构建新镜像 → 部署新版本 → 替换旧版本 → 一致、可追溯、快速回滚
