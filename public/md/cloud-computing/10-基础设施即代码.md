# 10 基础设施即代码（IaC）

## 10.1 IaC 概述

### 10.1.1 什么是 IaC

基础设施即代码（Infrastructure as Code，IaC）是一种通过编写代码来定义、管理和配置基础设施的方法论，用机器可读的配置文件替代手动操作控制台、点击式界面等传统方式。

**核心理念：**

- **一切皆代码**：服务器、网络、数据库、负载均衡等所有基础设施资源都用代码描述
- **版本控制**：所有基础设施变更纳入 Git 版本管理，具备完整的变更历史和回滚能力
- **自动化执行**：通过 CI/CD 流水线自动执行基础设施的创建、修改和销毁
- **一致性保证**：同一份代码在不同环境（dev/staging/prod）中产生一致的基础设施

**IaC 的两种范式：**

| 范式 | 说明 | 代表工具 |
|------|------|---------|
| 声明式（Declarative） | 描述期望的最终状态，工具自动计算并执行变更 | Terraform、CloudFormation、Pulumi |
| 命令式（Imperative） | 描述具体的执行步骤和操作序列 | Shell 脚本、Chef、Ansible（部分） |

**IaC 的演进历程：**

```
手动操作 → 脚本自动化 → 配置管理（Chef/Puppet）→ 基础设施编排（Terraform/CloudFormation）→ GitOps
```

### 10.1.2 IaC 的优势
- **可重复性**：环境一致性
- **版本控制**：Git 追踪变更历史
- **自动化**：CI/CD 集成
- **可审计**：变更可追溯
- **自文档化**：代码即文档

### 10.1.3 声明式 vs 命令式
| 方式 | 说明 | 代表工具 |
|------|------|---------|
| 声明式 | 描述期望状态，工具自动收敛 | Terraform、CloudFormation |
| 命令式 | 描述执行步骤 | Shell 脚本、Ansible（部分） |

## 10.2 Terraform

### 10.2.1 Terraform 核心概念
```
Terraform
├── Provider（提供者）：管理特定云厂商/服务的 API
├── Resource（资源）：定义具体的基础设施对象
├── Module（模块）：可复用的资源组合
├── State（状态）：记录已创建资源的实际状态
├── Variable（变量）：参数化配置
└── Output（输出）：暴露创建后的信息
```

**Terraform 1.9+ 关键特性：**
- **Removed blocks**：显式声明资源移除，避免意外删除
- **Provider-defined functions**：Provider 可注册自定义函数
- **State encryption enhancement**：增强的状态加密支持
- **Improved testing framework**：内置的 `terraform test` 命令支持模块单元测试和集成测试

### 10.2.2 HCL 语法基础

以下是一个完整的 Terraform 配置示例，涵盖 provider、resource、variable、output、data source 和 locals：

```hcl
# ============================================
# 1. Terraform 配置块与 Provider 声明
# ============================================
terraform {
  required_version = ">= 1.9.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.60"
    }
  }

  # 远程 State 后端配置（S3 + DynamoDB）
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/infrastructure/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}

# Provider 配置
provider "aws" {
  region = var.aws_region

  default_tags {
    tags = {
      Environment = var.environment
      ManagedBy   = "Terraform"
      Project     = var.project_name
    }
  }
}

# ============================================
# 2. Variable 定义（输入变量）
# ============================================
variable "aws_region" {
  description = "AWS 区域"
  type        = string
  default     = "us-east-1"

  validation {
    condition     = can(regex("^ap|us|eu|sa|af|me-[a-z]+-\\d+$", var.aws_region))
    error_message = "AWS 区域格式无效，例如：us-east-1, ap-northeast-1。"
  }
}

variable "environment" {
  description = "部署环境名称"
  type        = string

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "环境必须是 dev、staging 或 prod。"
  }
}

variable "project_name" {
  description = "项目名称"
  type        = string
  default     = "myapp"
}

variable "vpc_cidr" {
  description = "VPC 的 CIDR 块"
  type        = string
  default     = "10.0.0.0/16"
}

variable "public_subnet_cidrs" {
  description = "公有子网 CIDR 列表"
  type        = list(string)
  default     = ["10.0.1.0/24", "10.0.2.0/24", "10.0.3.0/24"]
}

variable "private_subnet_cidrs" {
  description = "私有子网 CIDR 列表"
  type        = list(string)
  default     = ["10.0.10.0/24", "10.0.11.0/24", "10.0.12.0/24"]
}

variable "allowed_ssh_cidrs" {
  description = "允许 SSH 访问的 CIDR 列表"
  type        = list(string)
  default     = []
}

variable "enable_nat_gateway" {
  description = "是否启用 NAT 网关"
  type        = bool
  default     = true
}

# ============================================
# 3. Locals（局部值）
# ============================================
locals {
  common_tags = {
    Environment = var.environment
    Project     = var.project_name
    ManagedBy   = "Terraform"
  }

  vpc_name = "${var.project_name}-${var.environment}-vpc"
}

# ============================================
# 4. Data Source（数据源）
# ============================================
# 查询最新的 Amazon Linux 2023 AMI
data "aws_ami" "amazon_linux_2023" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-2023.*-x86_64"]
  }

  filter {
    name   = "virtualization-type"
    values = ["hvm"]
  }
}

# 查询当前 AWS 账号 ID
data "aws_caller_identity" "current" {}

# 查询可用的 AWS 区域
data "aws_availability_zones" "available" {
  state = "available"
}

# ============================================
# 5. Resource 定义（基础设施资源）
# ============================================

# --- VPC ---
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name = local.vpc_name
  }
}

# --- Internet Gateway ---
resource "aws_internet_gateway" "main" {
  vpc_id = aws_vpc.main.id

  tags = {
    Name = "${var.project_name}-${var.environment}-igw"
  }
}

# --- 公有子网 ---
resource "aws_subnet" "public" {
  count                   = length(var.public_subnet_cidrs)
  vpc_id                  = aws_vpc.main.id
  cidr_block              = var.public_subnet_cidrs[count.index]
  availability_zone       = data.aws_availability_zones.available.names[count.index]
  map_public_ip_on_launch = true

  tags = {
    Name = "${var.project_name}-${var.environment}-public-${count.index + 1}"
    Tier = "Public"
  }
}

# --- 私有子网 ---
resource "aws_subnet" "private" {
  count             = length(var.private_subnet_cidrs)
  vpc_id            = aws_vpc.main.id
  cidr_block        = var.private_subnet_cidrs[count.index]
  availability_zone = data.aws_availability_zones.available.names[count.index]

  tags = {
    Name = "${var.project_name}-${var.environment}-private-${count.index + 1}"
    Tier = "Private"
  }
}

# --- NAT 网关 ---
resource "aws_eip" "nat" {
  count  = var.enable_nat_gateway ? 1 : 0
  domain = "vpc"

  tags = {
    Name = "${var.project_name}-${var.environment}-nat-eip"
  }

  depends_on = [aws_internet_gateway.main]
}

resource "aws_nat_gateway" "main" {
  count         = var.enable_nat_gateway ? 1 : 0
  allocation_id = aws_eip.nat[0].id
  subnet_id     = aws_subnet.public[0].id

  tags = {
    Name = "${var.project_name}-${var.environment}-nat"
  }

  depends_on = [aws_internet_gateway.main]
}

# --- 路由表 ---
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id

  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.main.id
  }

  tags = {
    Name = "${var.project_name}-${var.environment}-public-rt"
  }
}

resource "aws_route_table" "private" {
  count  = var.enable_nat_gateway ? 1 : 0
  vpc_id = aws_vpc.main.id

  route {
    cidr_block     = "0.0.0.0/0"
    nat_gateway_id = aws_nat_gateway.main[0].id
  }

  tags = {
    Name = "${var.project_name}-${var.environment}-private-rt"
  }
}

resource "aws_route_table_association" "public" {
  count          = length(aws_subnet.public)
  subnet_id      = aws_subnet.public[count.index].id
  route_table_id = aws_route_table.public.id
}

resource "aws_route_table_association" "private" {
  count          = var.enable_nat_gateway ? length(aws_subnet.private) : 0
  subnet_id      = aws_subnet.private[count.index].id
  route_table_id = aws_route_table.private[0].id
}

# --- 安全组 ---
resource "aws_security_group" "web" {
  name        = "${var.project_name}-${var.environment}-web-sg"
  description = "Web 服务器安全组"
  vpc_id      = aws_vpc.main.id

  ingress {
    description = "HTTP"
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  ingress {
    description = "HTTPS"
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "${var.project_name}-${var.environment}-web-sg"
  }
}

# --- EC2 实例 ---
resource "aws_instance" "web" {
  count                  = 2
  ami                    = data.aws_ami.amazon_linux_2023.id
  instance_type          = var.environment == "prod" ? "t3.medium" : "t3.micro"
  subnet_id              = aws_subnet.public[count.index % length(aws_subnet.public)].id
  vpc_security_group_ids = [aws_security_group.web.id]

  tags = {
    Name = "${var.project_name}-${var.environment}-web-${count.index + 1}"
  }
}

# --- RDS 子网组 ---
resource "aws_db_subnet_group" "main" {
  name       = "${var.project_name}-${var.environment}-db-subnet-group"
  subnet_ids = aws_subnet.private[*].id

  tags = {
    Name = "${var.project_name}-${var.environment}-db-subnet-group"
  }
}

# --- RDS 实例 ---
resource "aws_db_instance" "main" {
  identifier     = "${var.project_name}-${var.environment}-db"
  engine         = "aurora-postgresql"
  engine_version = "15.4"
  instance_class = var.environment == "prod" ? "db.r6g.large" : "db.t3.medium"

  allocated_storage     = var.environment == "prod" ? 100 : 20
  storage_encrypted     = true
  multi_az              = var.environment == "prod"
  skip_final_snapshot   = var.environment != "prod"
  final_snapshot_identifier = "${var.project_name}-${var.environment}-db-final-snapshot"

  db_subnet_group_name   = aws_db_subnet_group.main.name
  vpc_security_group_ids = [aws_security_group.web.id]

  tags = {
    Name = "${var.project_name}-${var.environment}-db"
  }
}

# ============================================
# 6. Output 定义（输出值）
# ============================================
output "vpc_id" {
  description = "VPC ID"
  value       = aws_vpc.main.id
}

output "public_subnet_ids" {
  description = "公有子网 ID 列表"
  value       = aws_subnet.public[*].id
}

output "private_subnet_ids" {
  description = "私有子网 ID 列表"
  value       = aws_subnet.private[*].id
}

output "web_instance_ids" {
  description = "Web 实例 ID 列表"
  value       = aws_instance.web[*].id
}

output "db_endpoint" {
  description = "RDS 数据库端点"
  value       = aws_db_instance.main.endpoint
  sensitive   = true
}

output "rds_connection_string" {
  description = "RDS 连接字符串"
  value       = "postgresql://${aws_db_instance.main.username}:${aws_db_instance.main.password}@${aws_db_instance.main.endpoint}/${aws_db_instance.main.db_name}"
  sensitive   = true
}
```

### 10.2.3 常用命令

```bash
# ============================================
# 基础工作流命令
# ============================================

# 初始化工作目录（下载 Provider、初始化后端）
terraform init
# 升级 Provider 到版本约束内的最新版本
terraform init -upgrade

# 预览变更（不会实际执行）
terraform plan
# 以结构化 JSON 格式输出 plan 结果（便于 CI/CD 解析）
terraform plan -out=tfplan
# 保存 plan 结果到文件，后续 apply 时使用
terraform plan -out=prod.tfplan

# 执行变更
terraform apply
# 使用之前保存的 plan 文件执行（确保一致性）
terraform apply prod.tfplan
# 跳过交互确认（适用于 CI/CD）
terraform apply -auto-approve

# 销毁所有资源
terraform destroy
terraform destroy -auto-approve

# ============================================
# State 管理命令
# ============================================

# 查看当前 State 中的资源列表
terraform state list

# 查看某个资源的 State 详情
terraform state show aws_vpc.main

# 将已有资源导入到 State 中
terraform import aws_instance.web[0] i-0abcd1234efgh5678

# 从 State 中移除资源（不会删除实际资源）
terraform state rm aws_instance.web[0]

# 将资源在 State 中移动（重命名地址）
terraform state mv aws_instance.old aws_instance.new

# ============================================
# 工作区管理
# ============================================

# 创建新工作区
terraform workspace new dev
terraform workspace new staging
terraform workspace new prod

# 切换工作区
terraform workspace select prod

# 查看当前工作区
terraform workspace show

# 列出所有工作区
terraform workspace list

# ============================================
# 格式化与验证
# ============================================

# 格式化代码（统一风格）
terraform fmt
# 递归格式化（包括子模块）
terraform fmt -recursive

# 验证配置语法
terraform validate

# ============================================
# 测试（Terraform 1.6+）
# ============================================

# 运行测试
terraform test

# ============================================
# 其他实用命令
# ============================================

# 查看所有 Provider 及其版本
terraform providers

# 输出当前配置的变量值
terraform output
# 输出特定变量
terraform output vpc_id
# 以 JSON 格式输出
terraform output -json

# 登录 Terraform Cloud
terraform login

# 强制解锁 State（当 Lock 未正常释放时）
terraform force-unlock <LOCK_ID>
```

### 10.2.4 State 管理

Terraform State 是 Terraform 的核心机制，它记录了已创建基础设施资源的实际状态，是 Terraform 进行 `plan`（计算变更差异）的基础。

**State 的作用：**

1. **映射关系**：维护配置中的资源地址与实际云资源之间的映射
2. **性能优化**：缓存资源属性，避免每次 plan 都查询所有资源
3. **元数据追踪**：记录资源间的依赖关系

**State 文件结构示例：**

```json
{
  "version": 4,
  "terraform_version": "1.9.8",
  "serial": 1,
  "lineage": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "outputs": {
    "vpc_id": { "value": "vpc-0abc123def456789", "type": "string" }
  },
  "resources": [
    {
      "module": "module.vpc",
      "mode": "managed",
      "type": "aws_vpc",
      "name": "main",
      "provider": "provider[\"registry.terraform.io/hashicorp/aws\"]",
      "instances": [
        {
          "schema_version": 1,
          "attributes": {
            "id": "vpc-0abc123def456789",
            "cidr_block": "10.0.0.0/16"
          }
        }
      ]
    }
  ]
}
```

**Remote State 配置（S3 + DynamoDB）：**

```hcl
terraform {
  backend "s3" {
    # State 文件存储桶
    bucket         = "my-terraform-state-prod"
    key            = "infra/terraform.tfstate"
    region         = "us-east-1"

    # 启用 State 文件加密（使用 S3 SSE）
    encrypt        = true

    # DynamoDB 表用于 State Locking
    dynamodb_table = "terraform-locks"

    # 启用版本控制（在 S3 桶上配置）
  }
}
```

**State Locking 原理：**

```
terraform apply
    │
    ▼
尝试在 DynamoDB 中创建 Lock Item
    │
    ├── 成功 → 获取锁 → 执行变更 → 释放锁
    │
    └── 失败（锁已被占用）→ 报错退出
         │
         "Error: Error acquiring the state lock"
```

**State 敏感信息处理：**

- State 文件中会存储数据库密码、密钥等敏感信息（即使标记为 `sensitive`）
- **必须**对 State 文件启用加密（S3 SSE 或 KMS）
- 使用 `sensitive = true` 标记 output，防止 `terraform output` 意外泄露
- 考虑使用 Vault 或 SSM Parameter Store 管理敏感值，而非直接写入 State

**State 最佳实践：**
- 使用 S3 + DynamoDB 存储远程 State
- 启用 State Locking 防止并发修改
- 定期 `terraform plan` 检查漂移
- 使用 `terraform import` 纳管已有资源

## 10.3 Ansible

### 10.3.1 Ansible 概述

Ansible 是一个开源的自动化配置管理工具，由 Red Hat 维护，采用 **Agentless** 架构，通过 SSH（Linux）或 WinRM（Windows）连接目标主机执行任务。

**核心特性：**

- **Agentless**：无需在目标主机安装任何 Agent，通过 SSH/WinRM 连接
- **YAML 编排**：使用人类可读的 YAML 格式编写 Playbook
- **幂等性**：大多数模块天然支持幂等，多次执行结果一致
- **模块丰富**：内置 3000+ 模块，覆盖主流操作系统、云平台、网络设备
- **扩展性强**：支持自定义模块、回调插件、过滤器

**Ansible 2.17+ 关键特性：**
- **ansible-core 2.17**：基于 Python 3.12+，性能优化
- **改进的异步引擎**：更好的大规模并行执行支持
- **增强的模块参数验证**：使用 Ansible Collections 中的 argspec
- **Red Hat Ansible Automation Platform 2.4**：企业级自动化平台

**Ansible 架构：**

```
控制节点（Control Node）
├── Ansible Core
├── Inventories（主机清单）
├── Playbooks（剧本）
├── Roles（角色）
└── Collections（集合）
        │
        │ SSH / WinRM
        ▼
管理节点（Managed Nodes）
├── 无需安装 Agent
└── 仅需 Python（大多数 Linux 已预装）
```

### 10.3.2 核心概念
- **Inventory**：主机清单
- **Playbook**：任务剧本（YAML）
- **Role**：角色（可复用的任务集合）
- **Module**：模块（执行具体操作）

### 10.4.3 Terraform + GitHub Actions 示例

```yaml
# .github/workflows/terraform.yml
name: "Terraform CI/CD"

on:
  push:
    branches: [main]
    paths:
      - "terraform/**"
  pull_request:
    branches: [main]
    paths:
      - "terraform/**"

# 这些权限是 OIDC 认证到 AWS 所需的
permissions:
  id-token: write
  contents: read
  pull-requests: write

env:
  TF_VERSION: "1.9.8"
  AWS_REGION: "us-east-1"
  TF_WORKING_DIR: "terraform/production"

jobs:
  # ============================================
  # Job 1: 格式化和静态分析
  # ============================================
  lint:
    name: "格式化与静态分析"
    runs-on: ubuntu-latest
    steps:
      - name: Checkout 代码
        uses: actions/checkout@v4

      - name: 安装 Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      - name: Terraform 格式检查
        id: fmt
        run: terraform fmt -check -recursive
        continue-on-error: true

      - name: Terraform 初始化
        run: terraform init
        working-directory: ${{ env.TF_WORKING_DIR }}

      - name: Terraform 验证
        run: terraform validate
        working-directory: ${{ env.TF_WORKING_DIR }}

      - name: 运行 tflint
        uses: terraform-linters/tflint@v1
        with:
          working_directory: ${{ env.TF_WORKING_DIR }}

      - name: 运行 Checkov 安全扫描
        uses: bridgecrewio/checkov-action@master
        with:
          directory: ${{ env.TF_WORKING_DIR }}
          framework: terraform
          skip_check: CKV_AWS_999

  # ============================================
  # Job 2: Plan（PR 和 Push 都执行）
  # ============================================
  plan:
    name: "Terraform Plan"
    runs-on: ubuntu-latest
    needs: lint
    outputs:
      plan_exitcode: ${{ steps.plan.outputs.exitcode }}
      plan_output: ${{ steps.plan_output.outputs.stdout }}
    steps:
      - name: Checkout 代码
        uses: actions/checkout@v4

      - name: 安装 Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      - name: 配置 AWS 凭证（OIDC）
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/GitHubActions-Terraform
          role-session-name: github-actions-terraform
          aws-region: ${{ env.AWS_REGION }}

      - name: Terraform 初始化
        run: terraform init
        working-directory: ${{ env.TF_WORKING_DIR }}

      - name: Terraform Plan
        id: plan
        run: terraform plan -no-color -out=tfplan
        working-directory: ${{ env.TF_WORKING_DIR }}

      - name: 获取 Plan 输出
        id: plan_output
        run: |
          PLAN_OUTPUT=$(terraform show -no-color tfplan)
          echo "stdout<<EOF" >> $GITHUB_OUTPUT
          echo "$PLAN_OUTPUT" >> $GITHUB_OUTPUT
          echo "EOF" >> $GITHUB_OUTPUT
        working-directory: ${{ env.TF_WORKING_DIR }}

      - name: 将 Plan 结果发布到 PR 评论
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const output = `${{ steps.plan_output.outputs.stdout }}`;
            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: `## Terraform Plan 结果\n\n\`\`\`hcl\n${output}\n\`\`\``
            });

  # ============================================
  # Job 3: Apply（仅 main 分支合并后执行）
  # ============================================
  apply:
    name: "Terraform Apply"
    runs-on: ubuntu-latest
    needs: plan
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    environment: production  # 需要手动审批
    steps:
      - name: Checkout 代码
        uses: actions/checkout@v4

      - name: 安装 Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      - name: 配置 AWS 凭证（OIDC）
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/GitHubActions-Terraform
          role-session-name: github-actions-terraform
          aws-region: ${{ env.AWS_REGION }}

      - name: Terraform 初始化
        run: terraform init
        working-directory: ${{ env.TF_WORKING_DIR }}

      - name: Terraform Apply
        run: terraform apply -auto-approve tfplan
        working-directory: ${{ env.TF_WORKING_DIR }}

  # ============================================
  # Job 4: 漂移检测（定时任务）
  # ============================================
  drift-detection:
    name: "基础设施漂移检测"
    runs-on: ubuntu-latest
    # 每天凌晨 2 点（UTC）运行
    if: github.event.schedule == '0 2 * * *'
    steps:
      - name: Checkout 代码
        uses: actions/checkout@v4

      - name: 安装 Terraform
        uses: hashicorp/setup-terraform@v3
        with:
          terraform_version: ${{ env.TF_VERSION }}

      - name: 配置 AWS 凭证（OIDC）
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/GitHubActions-Terraform
          role-session-name: github-actions-drift
          aws-region: ${{ env.AWS_REGION }}

      - name: Terraform 初始化
        run: terraform init
        working-directory: ${{ env.TF_WORKING_DIR }}

      - name: 检测基础设施漂移
        id: drift
        run: |
          terraform plan -detailed-exitcode -out=tfplan 2>&1 | tee drift-output.txt
          echo "exitcode=$?" >> $GITHUB_OUTPUT
        working-directory: ${{ env.TF_WORKING_DIR }}

      - name: 漂移检测通知
        if: steps.drift.outputs.exitcode == '2'
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {"text": "基础设施漂移检测发现变更！请查看: ${{ github.server_url }}/${{ github.repository }}/actions/runs/${{ github.run_id }}"}
```

## 10.5 其他 IaC 工具

### 10.5.1 Pulumi

Pulumi 是一个现代化的 IaC 工具，允许开发者使用熟悉的编程语言（Python、TypeScript、Go、C#、Java）来定义和管理基础设施，而非学习专有的 DSL（如 HCL）。

**核心理念：**

- **通用语言**：使用 Python、TypeScript、Go 等主流编程语言
- **真正的编程能力**：支持循环、条件、函数、类、错误处理等编程语言特性
- **共享状态管理**：与 Terraform 类似的 State 管理机制
- **Provider 生态**：支持所有主流云厂商和服务（兼容 Terraform Provider）

**Pulumi vs Terraform 对比：**

| 维度 | Pulumi | Terraform |
|------|--------|-----------|
| 配置语言 | 通用编程语言（Python/TS/Go/C#/Java） | HCL（HashiCorp 配置语言） |
| 学习曲线 | 需要编程基础 | HCL 语法简单易学 |
| 可测试性 | 可使用标准单元测试框架 | 需要 `terraform test` |
| IDE 支持 | 原生支持（语法高亮、自动补全、调试） | 需要 VS Code 扩展 |
| 生态系统 | 兼容 Terraform Provider | 最大的 IaC 生态 |
| 团队协作 | Pulumi Cloud / 自托管 | Terraform Cloud / 自托管 |
| 编排能力 | 编程语言原生能力（循环、条件、函数） | HCL 表达式 + for/sfor/dynamic |

**Pulumi TypeScript 示例：**

```typescript
import * as pulumi from "@pulumi/pulumi";
import * as aws from "@pulumi/aws";

// 变量定义
const projectName = "myapp";
const environment = pulumi.getStack(); // dev / staging / prod

// VPC
const vpc = new aws.ec2.Vpc(`${projectName}-vpc`, {
    cidrBlock: "10.0.0.0/16",
    enableDnsHostnames: true,
    enableDnsSupport: true,
    tags: { Environment: environment },
});

// 子网
const subnets: aws.ec2.Subnet[] = [];
for (let i = 0; i < 3; i++) {
    subnets.push(new aws.ec2.Subnet(`${projectName}-public-${i}`, {
        vpcId: vpc.id,
        cidrBlock: `10.0.${i + 1}.0/24`,
        availabilityZone: `us-east-${String.fromCharCode(97 + i)}`, // a, b, c
        mapPublicIpOnLaunch: true,
        tags: { Tier: "Public" },
    }));
}

// 安全组
const webSg = new aws.ec2.SecurityGroup(`${projectName}-web-sg`, {
    vpcId: vpc.id,
    ingress: [
        { protocol: "tcp", fromPort: 80, toPort: 80, cidrBlocks: ["0.0.0.0/0"] },
        { protocol: "tcp", fromPort: 443, toPort: 443, cidrBlocks: ["0.0.0.0/0"] },
    ],
    egress: [
        { protocol: "-1", fromPort: 0, toPort: 0, cidrBlocks: ["0.0.0.0/0"] },
    ],
});

// EC2 实例
const webServer = new aws.ec2.Instance(`${projectName}-web`, {
    ami: aws.ec2.getAmiOutput({
        owners: ["amazon"],
        mostRecent: true,
        filters: [{ name: "name", values: ["al2023-ami-2023.*-x86_64"] }],
    }).id,
    instanceType: environment === "prod" ? "t3.medium" : "t3.micro",
    subnetId: subnets[0].id,
    vpcSecurityGroupIds: [webSg.id],
    tags: { Name: `${projectName}-web` },
});

// 输出
export const vpcId = vpc.id;
export const publicIp = webServer.publicIp;
```

### 10.5.2 Crossplane

Crossplane 是一个 Kubernetes 原生的基础设施编排工具，通过 CRD（Custom Resource Definition）将云资源映射为 Kubernetes 自定义资源，使用 Kubernetes 声明式 API 来管理基础设施。

**核心理念：**

- **Kubernetes 原生**：基础设施即 Kubernetes 资源，使用 kubectl 管理
- **控制平面统一**：应用和基础设施使用同一个控制平面（K8s）
- **Composition API**：通过组合定义构建平台抽象层
- **Provider 生态**：支持 AWS、GCP、Azure、阿里云等 50+ Provider

**Crossplane vs Terraform vs Pulumi：**

| 维度 | Crossplane | Terraform | Pulumi |
|------|-----------|-----------|--------|
| 控制平面 | Kubernetes | Terraform CLI | Pulumi Engine |
| 配置方式 | Kubernetes YAML（CRD） | HCL | 编程语言 |
| 状态管理 | Kubernetes CR 状态 | State 文件 | Pulumi Cloud |
| 适用场景 | K8s 原生团队 | 通用基础设施 | 开发者友好 |
| 学习曲线 | 需要了解 K8s | HCL 简单 | 需要编程基础 |
| GitOps 集成 | 原生支持（ArgoCD/Flux） | 需要额外工具 | 需要额外工具 |

**Crossplane 示例：**

```yaml
# 1. 安装 Provider

    - endpoint

    - password

        openAPIV3Schema:
          type: object

              type: object
```

### 10.5.3 CDK（Cloud Development Kit）

CDK 是一种允许开发者使用编程语言定义云基础设施的工具框架，主要有两个实现：**AWS CDK** 和 **CDK for Terraform（CDKTF）**。

**AWS CDK：**

- 由 AWS 官方维护，仅支持 AWS 资源
- 支持 TypeScript、Python、Java、C#、Go
- 通过 CloudFormation 进行资源编排
- 提供 Construct 编程模型（Level 1/2/3 抽象）

**CDK for Terraform (CDKTF)：**

- 由 HashiCorp 维护，支持所有 Terraform Provider
- 支持 TypeScript、Python、Java、C#、Go
- 生成 Terraform JSON 配置，由 Terraform 执行
- 兼容 Terraform 生态（Module、Provider、State）

**三大工具对比：**

| 维度 | AWS CDK | CDKTF | Pulumi |
|------|---------|-------|--------|
| 维护方 | AWS | HashiCorp | Pulumi |
| 支持云厂商 | 仅 AWS | 所有 Terraform Provider | 所有 Terraform Provider |
| 底层引擎 | CloudFormation | Terraform | Pulumi Engine |
| 编程语言 | TS/Python/Java/C#/Go | TS/Python/Java/C#/Go | TS/Python/Go/C#/Java |
| 状态管理 | CloudFormation Stack | Terraform State | Pulumi State |
| 生态成熟度 | AWS 生态完善 | Terraform 生态 | 自有 + Terraform 兼容 |

**AWS CDK TypeScript 示例：**

```typescript
import * as cdk from "aws-cdk-lib";
import * as ec2 from "aws-cdk-lib/aws-ec2";
import * as rds from "aws-cdk-lib/aws-rds";
import * as secretsmanager from "aws-cdk-lib/aws-secretsmanager";
import { Construct } from "constructs";

export class MyappStack extends cdk.Stack {
  constructor(scope: Construct, id: string, props?: cdk.StackProps) {
    super(scope, id, props);

    // VPC
    const vpc = new ec2.Vpc(this, "MyAppVpc", {
      vpcName: "myapp-vpc",
      ipAddresses: ec2.IpAddresses.cidr("10.0.0.0/16"),
      maxAzs: 3,
      subnetConfiguration: [
        {
          name: "Public",
          subnetType: ec2.SubnetType.PUBLIC,
          cidrMask: 24,
        },
        {
          name: "Private",
          subnetType: ec2.SubnetType.PRIVATE_WITH_EGRESS,
          cidrMask: 24,
        },
      ],
      natGateways: 1,
    });

    // 数据库凭证
    const dbCredentials = secretsmanager.Secret.fromSecretNameV2(
      this, "DbCredentials", "myapp-db-credentials"
    );

    // RDS PostgreSQL
    const database = new rds.DatabaseInstance(this, "MyAppDb", {
      engine: rds.DatabaseInstanceEngine.postgres({
        version: rds.PostgresEngineVersion.VER_15_4,
      }),
      instanceType: ec2.InstanceType.of(
        ec2.InstanceClass.BURSTABLE3,
        ec2.InstanceSize.MEDIUM
      ),
      vpc,
      vpcSubnets: {
        subnetType: ec2.SubnetType.PRIVATE_WITH_EGRESS,
      },
      multiAz: true,
      storageEncrypted: true,
      credentials: rds.Credentials.fromSecret(dbCredentials),
      deletionProtection: true,
    });

    // 输出
    new cdk.CfnOutput(this, "VpcId", {
      value: vpc.vpcId,
      exportName: "MyAppVpcId",
    });

    new cdk.CfnOutput(this, "DbEndpoint", {
      value: database.dbInstanceEndpointAddress,
      exportName: "MyAppDbEndpoint",
    });
  }
}

// App 入口
const app = new cdk.App();
new MyappStack(app, "MyAppStack", {
  env: {
    region: "us-east-1",
    account: process.env.CDK_DEFAULT_ACCOUNT,
  },
});
```

## 10.6 IaC 最佳实践

- 所有变更通过 Git 提交和 PR 审查
- 使用 Module 复用代码
- State 文件加密存储
- 环境隔离（dev/staging/prod）
- 定期运行 plan 检查漂移
- 敏感信息使用 Vault 管理

**补充最佳实践：**

- **代码结构化**：按环境/组件组织目录结构，使用 Module 封装可复用逻辑
- **变量分离**：将变量定义（`variables.tf`）与配置逻辑（`main.tf`）分离，使用 `*.tfvars` 文件管理不同环境的变量值
- **锁版本**：锁定 Provider 和 Module 的版本号，避免意外升级导致的不兼容
- **最小权限**：CI/CD 使用的云凭证遵循最小权限原则，使用 OIDC 而非长期密钥
- **安全扫描**：集成 Checkov、tfsec、KICS 等安全扫描工具到 CI 流水线
- **成本估算**：使用 Infracost 在 PR 中展示基础设施变更的成本影响
- **测试**：使用 `terraform test` 编写模块的单元测试和集成测试
- **文档自动生成**：使用 terraform-docs 从代码自动生成 README 文档

## 10.7 面试常见问题

**Q1: Terraform 和 Ansible 的区别？分别适用什么场景？**

**参考答案：**

Terraform 和 Ansible 是两个定位不同的自动化工具：

| 维度 | Terraform | Ansible |
|------|-----------|---------|
| **核心定位** | 基础设施编排（Provisioning） | 配置管理（Configuration Management） |
| **工作模式** | 声明式：描述期望状态 | 过程式：描述执行步骤（Playbook） |
| **状态管理** | 有 State 文件，跟踪资源映射 | 无 State，依赖模块幂等性 |
| **适用场景** | 创建 VPC、EC2、RDS 等云资源 | 配置操作系统、安装软件、部署应用 |
| **并发执行** | 原生支持资源依赖图并行创建 | 通过 forks/serial 控制并发 |

**典型分工：**
- Terraform 负责"搭架子"：创建网络、计算、存储等云资源
- Ansible 负责"装软件"：在已创建的资源上安装配置软件和服务

**协作方式：** Terraform 创建资源后，通过 `output` 输出 IP/主机名，Ansible 使用动态 Inventory（`terraform-inventory`）获取这些信息进行配置管理。

---

**Q2: Terraform State 的作用？为什么需要 Remote State？**

**参考答案：**

**State 的三大作用：**
1. **映射关系**：维护代码中的资源地址（如 `aws_instance.web`）与实际云资源 ID（如 `i-0abcd1234`）之间的映射关系
2. **缓存元数据**：缓存资源的属性值，避免每次 `plan` 都调用云 API 查询所有资源
3. **性能优化**：通过依赖图追踪资源间的依赖关系，实现并行创建

**为什么需要 Remote State：**
1. **协作共享**：团队成员需要共享同一个 State 文件，否则各自维护的 State 会导致资源重复创建或冲突
2. **安全性**：本地 State 文件可能包含敏感信息（数据库密码等），Remote State 可以加密存储
3. **锁定机制**：Remote State 后端（如 S3 + DynamoDB）提供 State Locking，防止多人并发修改导致 State 损坏
4. **备份与恢复**：Remote State 支持版本控制（S3 版本控制），可以回滚到历史版本
5. **CI/CD 集成**：CI/CD 流水线需要访问 State 文件来执行 plan 和 apply

---

**Q3: 什么是 GitOps？**

**参考答案：**

GitOps 是一种将 Git 作为基础设施和应用配置"唯一事实来源"（Single Source of Truth）的运维方法论。

**四大核心原则：**
1. **声明式配置**：所有系统状态用声明式语言描述
2. **存储在 Git 中**：所有配置存储在 Git 仓库，享受版本控制、PR 审查、回滚等能力
3. **自动拉取（Pull 模式）**：通过自动化的 Agent（如 ArgoCD/Flux）从 Git 拉取配置并应用，而非推送到目标环境
4. **持续协调（Reconciliation）**：Agent 持续监控集群状态，自动纠正漂移

**GitOps 工作流：**
```
开发者修改配置 → Git Push → PR 审查 → 合并到 main 分支
    → ArgoCD/Flux 检测到变更 → 自动同步到集群 → 配置生效
```

**优势：**
- 变更可审计（Git 历史记录）
- 回滚简单（Git Revert）
- 安全性高（无需给 CI/CD 集群管理员权限）
- 自愈能力（自动纠正配置漂移）

---

**Q4: Terraform 的 Module 是什么？如何设计？**

**参考答案：**

Module 是 Terraform 中实现代码复用和抽象的机制，将多个相关资源封装为一个可复用的单元。

**Module 的类型：**
- **Root Module**：Terraform 工作目录中的配置文件集合
- **Child Module**：被其他 Module 通过 `module` 块调用的 Module
- **Published Module**：发布到 Terraform Registry 的公共/私有 Module

**Module 设计最佳实践：**

1. **单一职责**：每个 Module 负责一个明确的功能（如 VPC、RDS、EKS）
2. **清晰的接口**：通过 `variables.tf` 定义输入，`outputs.tf` 定义输出
3. **版本管理**：使用语义化版本号（SemVer），发布到 Registry
4. **文档完善**：使用 terraform-docs 自动生成 README
5. **可组合性**：Module 可以嵌套调用其他 Module
6. **示例代码**：提供 `examples/` 目录展示使用方法

**Module 目录结构：**
```
modules/vpc/
├── main.tf          # 资源定义
├── variables.tf     # 输入变量
├── outputs.tf       # 输出值
├── versions.tf      # Terraform 和 Provider 版本约束
├── locals.tf        # 局部值
├── README.md        # 文档（自动生成）
└── examples/
    ├── simple/
    │   └── main.tf  # 简单使用示例
    └── complete/
        └── main.tf  # 完整使用示例
```

**Q5: 如何管理 Terraform 中的敏感信息？**

**参考答案：**

管理 Terraform 中的敏感信息需要多层防护：

1. **State 文件加密**：使用 S3 SSE-KMS 或 Terraform Cloud 的内置加密
2. **Output 标记**：使用 `sensitive = true` 标记包含敏感信息的 output
3. **变量加密**：
   - 使用 `terraform.tfvars` 文件（加入 `.gitignore`）
   - 使用 HashiCorp Vault 的 Vault Provider 动态获取密钥
   - 使用 AWS SSM Parameter Store 或 Secrets Manager
4. **CI/CD 凭证**：使用 OIDC（OpenID Connect）而非长期 Access Key
5. **代码审查**：使用 git-secrets、detect-secrets 等工具防止密钥泄露到代码仓库
6. **环境变量**：通过 `TF_VAR_xxx` 环境变量传入，避免硬编码

**示例 - 从 Vault 获取数据库密码：**
```hcl
data "vault_generic_secret" "db_creds" {
  path = "secret/data/myapp/database"
}

resource "aws_db_instance" "main" {
  username = data.vault_generic_secret.db_creds.data["username"]
  password = data.vault_generic_secret.db_creds.data["password"]
}
```

**Q6: 什么是基础设施漂移？如何检测？**

**参考答案：**

**基础设施漂移（Infrastructure Drift）**是指实际基础设施的状态与 Terraform State 中记录的状态（或代码中定义的期望状态）不一致的情况。

**漂移的常见原因：**
- 运维人员通过控制台手动修改了资源配置
- 其他自动化工具或脚本修改了资源
- 云厂商自动更新（如 RDS 小版本升级）
- 外部系统触发了资源变更

**检测方法：**

1. **`terraform plan -detailed-exitcode`**：
   - 退出码 `0`：无变更
   - 退出码 `1`：出错
   - 退出码 `2`：有变更（检测到漂移）

2. **定期 CI 检测**：在 CI 流水线中定时运行 `terraform plan`，检测到漂移时发送告警

3. **Terraform Cloud/Enterprise**：内置漂移检测功能，可配置定期检查

4. **第三方工具**：Driftctl、KICS 等工具可扫描云资源并与 IaC 代码对比

**处理策略：**
- 检测到漂移后，先评估变更是否合理
- 合理变更：通过 `terraform apply` 将漂移纳入管理
- 不合理变更：手动回滚或通过 `terraform apply` 恢复到期望状态
