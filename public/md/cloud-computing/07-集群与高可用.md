# 07 集群与高可用

## 7.1 集群基础概念

### 7.1.1 什么是集群

**集群（Cluster）** 是将多台独立的服务器通过网络连接在一起，协同工作，对外提供统一服务的技术架构。客户端感知不到后端有多台机器，就像访问一台服务器一样。

**集群的核心价值：**

| 价值 | 说明 |
|------|------|
| 高可用 (HA) | 单台机器故障时，其他机器自动接管，服务不中断 |
| 负载均衡 (LB) | 将请求分发到多台机器，提升整体处理能力 |
| 横向扩展 | 通过增加节点提升性能，而非升级单机硬件（Scale Out vs Scale Up） |
| 容错能力 | 部分节点故障不影响整体服务 |

**集群的基本架构：**

```
                    +------------------+
                    |    负载均衡器     |
                    | (LB / VIP / DNS) |
                    +--------+---------+
                             |
              +--------------+--------------+
              |              |              |
        +-----+-----+ +-----+-----+ +-----+-----+
        |  Node 1   | |  Node 2   | |  Node 3   |
        | 10.0.0.1  | | 10.0.0.2  | | 10.0.0.3  |
        +-----------+ +-----------+ +-----------+
              |              |              |
              +--------------+--------------+
                             |
                    +--------+---------+
                    |    共享存储       |
                    | (DB / NFS / SAN) |
                    +------------------+
```

### 7.1.2 集群分类
| 类型 | 说明 | 代表 |
|------|------|------|
| 高可用集群 (HA) | 故障自动转移 | K8s Master HA、Keepalived + HAProxy |
| 负载均衡集群 (LB) | 分散请求 | LVS + Keepalived、云厂商 SLB/ALB |
| 高性能计算集群 (HPC) | 并行计算 | MPI 集群、AWS HPC |

### 7.1.3 集群 vs 分布式
| 维度 | 集群 | 分布式 |
|------|------|--------|
| 目标 | 提升可用性/性能 | 拆分业务逻辑 |
| 耦合度 | 较高 | 较低 |
| 部署 | 同构为主 | 异构常见 |
| 透明性 | 对用户透明 | 业务层面可见 |

## 7.2 负载均衡（基础设施层）

> **注意**：应用层负载均衡（Ribbon、Nginx 反向代理）已在 `java/07-分布式与微服务.md` 中覆盖，本文档聚焦**硬件/网络层** LB（LVS、HAProxy、云厂商 SLB/ALB）。

### 7.2.1 负载均衡层级
```
L4（传输层）：TCP/UDP 转发，基于 IP:Port
  - LVS、HAProxy、云厂商 SLB/NLB

L7（应用层）：HTTP/HTTPS 转发，基于 URL/Header/Cookie
  - HAProxy（mode http）、云厂商 ALB
```

**L4 vs L7 对比：**

| 维度 | L4 负载均衡 | L7 负载均衡 |
|------|------------|------------|
| 工作层 | 传输层（TCP/UDP） | 应用层（HTTP/HTTPS） |
| 决策依据 | IP + 端口 | URL、Header、Cookie |
| 性能 | 极高（内核态） | 较高（用户态解析） |
| 功能 | 简单转发 | 内容路由、改写、SSL 卸载 |
| 代表 | LVS、NLB | HAProxy（http mode）、ALB |

### 7.2.2 LVS（Linux Virtual Server）

**LVS** 是由章文嵩博士发起的 Linux 内核级四层负载均衡项目，工作在内核态，性能极高，是国产开源项目的标杆之一。当前稳定版本为 **IPVS 1.4+**（集成在 Linux 内核 6.x 中）。

**LVS 核心组件：**

```
+--------------------------------------------------+
|                  LVS 架构                         |
|                                                  |
|   Client → Director（负载调度器） → Real Server   |
|                                                  |
|   ipvsadm：用户态管理工具                          |
|   IPVS：内核态调度模块（netfilter/ipvs）           |
+--------------------------------------------------+
```

**LVS 四种工作模式：**

| 模式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **NAT** | Director 修改请求/响应的目标 IP | RS 无需公网 IP | Director 瓶颈，支持 RS 少 |
| **DR**（推荐） | Director 修改 MAC 地址，RS 直接响应客户端 | 性能最高，RS 直接回包 | RS 和 Director 必须在同一网段 |
| **TUN** | IP 封装（IP-in-IP），RS 解封装后直接响应 | RS 可跨网段 | 需要内核支持 IP 隧道 |
| **FULLNAT** | 同时修改源/目标 IP | RS 可跨网段，运维方便 | 需要打内核补丁（阿里云 DRS） |

**LVS-DR 模式工作原理（最常用）：**

```
1. 客户端发送请求到 VIP
2. Director 收到请求，根据调度算法选择 RS
3. Director 将请求的 MAC 地址改为 RS 的 MAC，转发给 RS
4. RS 收到请求后发现目标 IP 是 VIP（配置在 lo 接口上），处理请求
5. RS 直接将响应发送给客户端（不经过 Director）

        Client
           |
           | 请求目标: VIP
           v
    +-------------+
    |  Director   |  修改 MAC → RS 的 MAC
    |  (LVS/DR)   |
    +------+------+
           |
     +-----+-----+
     |           |
     v           v
+---------+ +---------+
|   RS1   | |   RS2   |  lo:0 配置 VIP
| VIP(lo) | | VIP(lo) |  直接响应客户端
+---------+ +---------+
     ^           ^
     |           |
     +-----------+
   ARP 抑制: RS 不响应 VIP 的 ARP 请求
```

**LVS 十种调度算法：**

| 算法 | 说明 |
|------|------|
| rr（轮询） | 依次轮流分配 |
| wrr（加权轮询） | 按权重轮流分配 |
| lc（最少连接） | 分配给连接数最少的服务器 |
| wlc（加权最少连接） | 连接数/权重比最小的优先（默认） |
| lblc（基于地址的最少连接） | 同一目标 IP 分配到同一 RS |
| lblcr（带复制的基于地址最少连接） | lblc + 缓存副本 |
| dh（目标地址哈希） | 基于目标地址哈希 |
| sh（源地址哈希） | 基于源地址哈希 |
| sed（最短期望延迟） | (active+1)/weight 最小的优先 |
| nq（永不排队） | sed 的改进，避免空闲 RS 等待 |

**LVS-DR 实战配置（基于 ipvsadm）：**

```bash
# ======== Director 节点配置 ========

# 1. 开启 IP 转发
echo 1 > /proc/sys/net/ipv4/ip_forward

# 2. 配置 VIP（Director 上）
ip addr add 192.168.1.100/32 dev eth0

# 3. 安装 ipvsadm
yum install -y ipvsadm   # CentOS/RHEL
apt install -y ipvsadm   # Ubuntu/Debian

# 4. 创建虚拟服务（VIP:80，使用 DR 模式，wlc 调度）
ipvsadm -A -t 192.168.1.100:80 -s wlc

# 5. 添加后端真实服务器（DR 模式使用 -g 参数）
ipvsadm -a -t 192.168.1.100:80 -r 192.168.1.11:80 -g -w 3
ipvsadm -a -t 192.168.1.100:80 -r 192.168.1.12:80 -g -w 2
ipvsadm -a -t 192.168.1.100:80 -r 192.168.1.13:80 -g -w 1

# 6. 查看规则
ipvsadm -Ln

# 7. 持久化配置
ipvsadm-save > /etc/sysconfig/ipvsadm

# ======== Real Server 节点配置（每台 RS 都要执行）========

# 1. 在 lo 接口上配置 VIP
ip addr add 192.168.1.100/32 dev lo

# 2. 抑制 ARP 响应（关键！防止 RS 抢答 VIP 的 ARP）
echo 1 > /proc/sys/net/ipv4/conf/lo/arp_ignore
echo 2 > /proc/sys/net/ipv4/conf/lo/arp_announce
echo 1 > /proc/sys/net/ipv4/conf/all/arp_ignore
echo 2 > /proc/sys/net/ipv4/conf/all/arp_announce

# 3. RS 持久化脚本（/etc/rc.local）
cat >> /etc/rc.local << 'EOF'
#!/bin/bash
VIP=192.168.1.100
ip addr add ${VIP}/32 dev lo
echo 1 > /proc/sys/net/ipv4/conf/lo/arp_ignore
echo 2 > /proc/sys/net/ipv4/conf/lo/arp_announce
echo 1 > /proc/sys/net/ipv4/conf/all/arp_ignore
echo 2 > /proc/sys/net/ipv4/conf/all/arp_announce
EOF
chmod +x /etc/rc.local
```

### 7.2.3 HAProxy

**HAProxy** 是一款免费、开源、高性能的四七层负载均衡器和代理服务器，当前稳定版本为 **HAProxy 3.0**（2024 年发布），支持 HTTP/2、HTTP/3、gRPC、QUIC 等协议。

**HAProxy 核心特性：**

- 支持 L4（TCP）和 L7（HTTP/HTTPS）负载均衡
- 支持 SSL/TLS 终止（SSL Offloading）
- 支持健康检查（HTTP、TCP、gRPC）
- 支持连接排空（Graceful Shutdown）
- 支持速率限制和请求过滤
- 内置 Prometheus 指标导出
- 支持 DNS 服务发现
- 单进程事件驱动，内存占用极低

**HAProxy 3.0 新特性：**

| 特性 | 说明 |
|------|------|
| HTTP/3 (QUIC) 支持 | 原生支持 QUIC 协议 |
| 改进的 gRPC 支持 | gRPC 负载均衡、健康检查 |
| 设备标识 (Device ID) | 更好的服务器追踪 |
| 多线程性能优化 | 进一步提升吞吐量 |
| enhanced health checks | 更灵活的健康检查机制 |

**HAProxy 完整配置示例（四七层负载均衡 + SSL + 健康检查 + 监控）：**

```haproxy
# /etc/haproxy/haproxy.cfg
# HAProxy 3.0 配置示例

# ======== 全局配置 ========
global
    # 运行参数
    maxconn 4096                    # 最大并发连接数
    log /dev/log local0 info        # 日志配置
    log /dev/log local1 notice
    daemon                          # 后台运行
    stats socket /run/haproxy/admin.sock mode 660 level admin
    stats timeout 30s

    # SSL 配置
    ssl-default-bind-ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256
    ssl-default-bind-options ssl-min-ver TLSv1.2

    # 性能调优
    nbthread 4                      # 多线程模式（HAProxy 2.0+）
    cpu-map auto:1/1-4 0-3          # CPU 亲和性

# ======== 默认配置 ========
defaults
    log     global
    mode    http                    # 默认 HTTP 模式
    option  httplog
    option  dontlognull
    timeout connect 5s              # 连接超时
    timeout client  30s             # 客户端超时
    timeout server  30s             # 服务端超时
    timeout check   5s              # 健康检查超时
    retries 3                       # 重试次数

# ======== 前端 - HTTP 重定向到 HTTPS ========
frontend http_front
    bind *:80
    mode http
    # 重定向所有 HTTP 请求到 HTTPS
    redirect scheme https code 301 if !{ ssl_fc }

# ======== 前端 - HTTPS 七层负载均衡 ========
frontend https_front
    bind *:443 ssl crt /etc/haproxy/certs/example.com.pem alpn h2,http/1.1
    mode http

    # 访问控制列表
    acl is_api path_beg /api
    acl is_static path_end .css .js .png .jpg .ico
    acl is_websocket hdr(Upgrade) -i websocket

    # 根据路径分发到不同后端
    use_backend api_backend    if is_api
    use_backend static_backend if is_static
    use_backend ws_backend     if is_websocket
    default_backend web_backend

    # 请求头修改
    http-request add-header X-Forwarded-Proto https
    http-request add-header X-Client-IP %[src]

# ======== 后端 - Web 应用 ========
backend web_backend
    mode http
    balance roundrobin             # 轮询算法

    # 健康检查
    option httpchk GET /health
    http-check expect status 200

    # 后端服务器
    server web1 10.0.1.11:8080 check inter 3s fall 3 rise 2 weight 3
    server web2 10.0.1.12:8080 check inter 3s fall 3 rise 2 weight 2
    server web3 10.0.1.13:8080 check inter 3s fall 3 rise 2 weight 1 backup

    # 会话保持（基于 Cookie）
    cookie SERVERID insert indirect nocache httponly

    # 连接排空
    option httpchk GET /health

# ======== 后端 - API 服务 ========
backend api_backend
    mode http
    balance leastconn              # 最少连接算法

    # 限流
    stick-table type ip size 100k expire 30s store http_req_rate(10s)
    http-request track-sc0 src
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 100 }

    server api1 10.0.2.11:8080 check inter 2s fall 3 rise 2
    server api2 10.0.2.12:8080 check inter 2s fall 3 rise 2

# ======== 后端 - 静态资源 ========
backend static_backend
    mode http
    balance roundrobin
    server static1 10.0.3.11:80 check
    # 缓存静态资源
    cache static_cache
    http-request cache-use static_cache
    http-response cache-store static_cache

# ======== 后端 - WebSocket ========
backend ws_backend
    mode http
    balance leastconn
    option forwardfor
    timeout tunnel 1h              # WebSocket 长连接超时
    server ws1 10.0.4.11:8080 check
    server ws2 10.0.4.12:8080 check

# ======== 前端 - TCP 四层负载均衡（MySQL）========
frontend mysql_front
    bind *:3306
    mode tcp
    default_backend mysql_backend

backend mysql_backend
    mode tcp
    balance roundrobin
    option tcp-check               # TCP 健康检查
    server mysql1 10.0.5.11:3306 check inter 3s fall 3 rise 2
    server mysql2 10.0.5.12:3306 check inter 3s fall 3 rise 2

# ======== 统计监控页面 ========
listen stats
    bind *:8404
    mode http
    stats enable
    stats uri /stats
    stats refresh 10s
    stats admin if LOCALHOST       # 本地可管理
    stats auth admin:Admin@123     # 认证
```

**HAProxy 启动与管理：**

```bash
# 安装 HAProxy 3.0（Ubuntu 22.04+）
apt update && apt install -y haproxy

# 验证配置语法
haproxy -c -f /etc/haproxy/haproxy.cfg

# 启动 HAProxy
systemctl enable haproxy
systemctl start haproxy

# 重新加载配置（不中断连接）
systemctl reload haproxy

# 查看状态
haproxy -st /run/haproxy/admin.sock show stat
```

### 7.2.4 云厂商负载均衡
| 云厂商 | 产品 | 类型 | 特点 |
|--------|------|------|------|
| AWS | ALB / NLB / CLB | L7 / L4 / L4（传统） | ALB 支持路径路由、NLB 超低延迟 |
| 阿里云 | SLB / ALB | L4 / L7 | ALB 支持 QUIC、WAF 集成 |
| 腾讯云 | CLB | L4/L7 | 支持按量计费 |
| Azure | LB / App Gateway | L4 / L7 | App Gateway 支持 WAF |
| GCP | Cloud Load Balancing | L4/L7 | 全球负载均衡 |

**AWS 负载均衡选型：**

```
需要 L4 TCP/UDP 超高性能？
  → NLB（Network Load Balancer）
    - 每秒百万级连接
    - 超低延迟（毫秒级）
    - 支持 TLS 终止
    - 静态 IP / Elastic IP

需要 L7 HTTP/HTTPS 路由？
  → ALB（Application Load Balancer）
    - 基于路径、Header、Query 参数路由
    - 支持 WebSocket、gRPC
    - 内置 WAF 集成
    - 支持 Lambda 作为目标
    - 支持固定响应和重定向

需要传统四层负载？
  → CLB（Classic Load Balancer，已不推荐新项目使用）
```

### 7.2.5 负载均衡算法
| 算法 | 说明 | 适用场景 |
|------|------|---------|
| 轮询 (Round Robin) | 依次分配 | 服务器性能相近 |
| 加权轮询 | 按权重分配 | 服务器性能不同 |
| 最少连接 | 分配给连接数最少的 | 长连接场景 |
| IP Hash | 基于 IP 哈希 | 会话保持 |
| 一致性哈希 | 减少节点变更影响 | 缓存集群 |
| 最短期望延迟 | (active+1)/weight 最小 | LVS 默认 |
| 源地址哈希 | 基于源 IP | 会话亲和 |

## 7.3 高可用架构

### 7.3.1 高可用核心原则
- **消除单点故障 (SPOF)**：任何组件都应有冗余
- **故障自动检测与转移**：通过心跳检测、健康检查实现自动切换
- **数据冗余**：多副本存储，保证数据不丢失
- **弹性伸缩**：根据负载自动扩缩容
- **故障隔离**：限制故障影响范围（熔断、降级、限流）
- **可观测性**：监控、告警、日志、链路追踪

### 7.3.2 主备模式（Active-Passive）

```
         +--------+
         |  VIP   |
         +---+----+
             |
      +------+------+
      |             |
 +----+----+   +----+----+
 | Active  |   | Passive |
 | Master  |   | Standby |
 +---------+   +---------+
      |             |
      +------+------+
             |
    +--------+--------+
    |   共享存储/DB    |
    +-----------------+
```

**工作原理：**

1. Active 节点持有 VIP，处理所有请求
2. Passive 节点处于备用状态，实时同步数据
3. Active 节点故障时，Passive 节点接管 VIP，成为新的 Active
4. 切换过程对客户端基本透明（通常在秒级完成）

**Keepalived + HAProxy 实现主备高可用：**

```bash
# ======== Master 节点 Keepalived 配置 ========
# /etc/keepalived/keepalived.conf

global_defs {
    router_id LVS_MASTER           # 路由器标识
    vrrp_skip_check_adv_addr       # 跳过 VRRP 地址检查
    vrrp_strict                    # 严格遵守 VRRP 协议
    vrrp_garp_interval 0           # GARP 报文发送间隔
    vrrp_gna_interval 0
}

# 健康检查脚本
vrrp_script chk_haproxy {
    script "/etc/keepalived/check_haproxy.sh"
    interval 2                     # 每 2 秒检查一次
    weight -20                     # 检查失败时优先级 -20
    fall 3                         # 连续失败 3 次判定为故障
    rise 2                         # 连续成功 2 次判定为恢复
}

vrrp_instance VI_1 {
    state MASTER                   # 状态：MASTER / BACKUP
    interface eth0                 # 绑定的网络接口
    virtual_router_id 51           # 虚拟路由 ID（同一组必须一致）
    priority 100                   # 优先级（MASTER > BACKUP）
    advert_int 1                   # VRRP 广播间隔（秒）

    # 认证
    authentication {
        auth_type PASS
        auth_pass 123456
    }

    # 虚拟 IP
    virtual_ipaddress {
        192.168.1.100/24 dev eth0 label eth0:vip
    }

    # 追踪健康检查脚本
    track_script {
        chk_haproxy
    }

    # 状态变化通知脚本
    notify_master "/etc/keepalived/notify.sh MASTER"
    notify_backup "/etc/keepalived/notify.sh BACKUP"
    notify_fault  "/etc/keepalived/notify.sh FAULT"
}

# ======== Backup 节点 Keepalived 配置 ========
# 仅需修改以下内容，其余与 Master 相同：
#   state BACKUP
#   priority 90

# ======== 健康检查脚本 ========
# /etc/keepalived/check_haproxy.sh
#!/bin/bash
if ! killall -0 haproxy; then
    systemctl restart haproxy
    sleep 2
    if ! killall -0 haproxy; then
        exit 1    # HAProxy 无法启动，降低优先级
    fi
fi
exit 0

# ======== 状态通知脚本 ========
# /etc/keepalived/notify.sh
#!/bin/bash
TYPE=$1
case $TYPE in
    MASTER)
        logger "Keepalived: 切换为 MASTER"
        # 可以在这里添加启动服务的逻辑
        ;;
    BACKUP)
        logger "Keepalived: 切换为 BACKUP"
        ;;
    FAULT)
        logger "Keepalived: 进入 FAULT 状态"
        ;;
esac
```

### 7.3.3 主主模式（Active-Active）

**双活架构**，两个节点同时提供服务，互为备份。适用于读多写少或无状态服务场景。

```
                    +--------+
                    |  VIP1  |  ← 主节点1持有
                    +---+----+
                        |
                 +------+------+
                 |             |
           +-----+-----+ +-----+-----+
           |  Node 1   | |  Node 2   |
           | (Active)  | | (Active)  |
           |  VIP1     | |  VIP2     |
           +-----+-----+ +-----+-----+
                 |             |
                 +------+------+
                        |
                    +--------+
                    |  VIP2  |  ← 主节点2持有
                    +--------+

DNS 轮询或全局负载均衡分发到 VIP1/VIP2
```

**Keepalived 双主配置示例：**

```bash
# ======== 节点1 配置 ========
# /etc/keepalived/keepalived.conf

global_defs {
    router_id NODE1
}

vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass 123456
    }
```

**适用场景：**

| 场景 | 说明 |
|------|------|
| 无状态 Web 服务 | 两个节点都处理请求，资源利用率 100% |
| 数据库双主 | 双向复制，各自处理写操作（需解决冲突） |
| DNS 负载均衡 | 通过 DNS 轮询将请求分发到不同 VIP |

**注意事项：**
- 有状态服务需要解决数据同步问题
- 数据库双主需要处理主键冲突（如奇偶 ID、UUID）
- 需要考虑脑裂问题（使用仲裁机制）

### 7.3.4 多活架构（Multi-Active）

**异地多活** 是最高级别的高可用架构，在多个地理区域部署完整的服务能力，每个区域都能独立处理请求。

```
                    +------------------+
                    |   全局 DNS/GSLB  |
                    |  (Route 53/GeoDNS)|
                    +--------+---------+
                             |
              +--------------+--------------+
              |              |              |
     +--------+-----+ +-----+--------+ +--+---------+
     |  Region A    | |  Region B    | |  Region C  |
     |  (华北)      | |  (华东)      | |  (华南)    |
     |              | |              | |            |
     | +----------+ | | +----------+ | | +--------+ |
     | | LB/ALB   | | | | LB/ALB   | | | | LB/ALB| |
     | +----+-----+ | | +----+-----+ | | +---+----+ |
     |      |       | |      |       | |     |      |
     | +----+-----+ | | +----+-----+ | | +---+----+ |
     | | App 集群  | | | | App 集群  | | | |App集群| |
     | +----+-----+ | | +----+-----+ | | +---+----+ |
     |      |       | |      |       | |     |      |
     | +----+-----+ | | +----+-----+ | | +---+----+ |
     | | DB 主/从  | | | | DB 主/从  | | | |DB主从 | |
     | +----+-----+ | | +----+-----+ | | +---+----+ |
     +------+-------+ +------+-------+ +--+---------+
            |                |               |
            +-------+--------+-------+-------+
                    |
            +-------+--------+
            |  跨区域数据同步   |
            |  (DTS / CDC)    |
            +----------------+
```

**多活架构的关键技术：**

| 技术 | 说明 | 代表方案 |
|------|------|---------|
| 全局负载均衡 | 按地域/延迟/权重分发 | AWS Route 53、GSLB |
| 数据同步 | 跨区域数据复制 | DTS、Canal、Debezium |
| 单元化 | 按用户/租户分片到不同区域 | 京东、阿里双十一方案 |
| 故障切换 | 区域级别自动切换 | Route 53 Health Check |
| 数据一致性 | 跨区域最终一致性 | 分布式事务、消息队列 |

**多活架构的挑战：**

1. **数据一致性**：跨区域同步延迟（通常 50-200ms），需要接受最终一致性
2. **流量调度**：如何将用户路由到最近的/正确的区域
3. **故障切换**：区域级别故障时的流量迁移方案
4. **成本**：多区域部署成本是单区域的 2-3 倍
5. **运维复杂度**：多区域部署、监控、故障排查

## 7.4 Keepalived

### 7.4.1 VRRP 协议

**VRRP（Virtual Router Redundancy Protocol，虚拟路由冗余协议）** 是一种网络层高可用协议，定义在 RFC 3768 中，用于将多个路由器虚拟为一个逻辑路由器，提供默认网关的冗余。

**VRRP 工作原理：**

```
1. 一组路由器组成一个 VRRP 组，拥有一个虚拟 IP（VIP）
2. 通过优先级选举 Master 路由器
3. Master 路由器拥有 VIP，负责转发流量
4. Backup 路由器监听 Master 的 VRRP 广播（默认 1 秒一次）
5. 如果 Backup 在 3*advert_int 内未收到 Master 的广播，则发起选举
6. 优先级最高的 Backup 成为新 Master，接管 VIP

    +------------------------------------------+
    |          VRRP 组 (VRID = 51)              |
    |                                          |
    |  +----------+  +----------+  +----------+|
    |  | Router A |  | Router B |  | Router C ||
    |  |Priority=100| |Priority=90| |Priority=80||
    |  | MASTER   |  | BACKUP   |  | BACKUP   ||
    |  +----+-----+  +----+-----+  +----+-----+|
    |       |               |               |  |
    |       +-------+-------+-------+-------+  |
    |               |                       |  |
    |          +----+----+                  |  |
    |          |  VIP    | 192.168.1.1      |  |
    |          +---------+                  |  |
    +------------------------------------------+

    VRRP 报文类型：
    - ADVERTISEMENT：Master 定期广播（组播 224.0.0.18，协议号 112）
    - 优先级范围：1-254（255 保留，0 表示 Master 放弃）
```

**VRRP 选举流程：**

```
1. 所有路由器启动，初始状态为 INIT
2. 收到其他路由器的 VRRP 报文 → 比较 priority
   - 自身 priority > 报文中的 priority → 转为 MASTER
   - 自身 priority < 报文中的 priority → 转为 BACKUP
3. MASTER 每隔 advert_int（默认 1s）发送 ADVERTISEMENT
4. BACKUP 在 master_down_interval（3 * advert_int）内
   未收到 ADVERTISEMENT → 发起选举
5. 选举规则：priority 最高者成为 MASTER
   （priority 相同时，比较接口 IP 地址）
```

**Keepalived vs VRRP 原生实现：**

| 特性 | Keepalived | VRRP 原生 |
|------|-----------|----------|
| 健康检查 | 内置（HTTP、TCP、脚本） | 无 |
| LVS 集成 | 原生支持 IPVS | 不支持 |
| 脑裂处理 | 支持仲裁脚本 | 无 |
| 通知机制 | 邮件、脚本通知 | 无 |
| 配置灵活度 | 高（支持多种检查方式） | 低 |

## 7.5 数据库高可用

### 7.5.1 主从复制

**主从复制（Master-Slave Replication）** 是数据库高可用的基础方案，将主库的数据变更异步/半同步地复制到从库。

**MySQL 主从复制原理：**

```
                    Master
    +----------------------------------+
    |  Client Write Request            |
    |       |                          |
    |       v                          |
    |  +----------+     +-----------+  |
    |  |  InnoDB  | --> |  Binlog   |  |
    |  | (数据存储)|     | (二进制日志)|  |
    |  +----------+     +-----+-----+  |
    |                        |         |
    |                   Dump Thread    |
    +------------------------+---------+
                             |
                    Binlog Events
                             |
    +------------------------+---------+
    |                        |         |
    |                   IO Thread      |
    |                        |         |
    |                  +-----+-----+   |
    |                  | Relay Log  |   |
    |                  | (中继日志)  |   |
    |                  +-----+-----+   |
    |                        |         |
    |                  SQL Thread      |
    |                        |         |
    |                  +-----+-----+   |
    |                  |  InnoDB   |   |
    |                  | (数据重放) |   |
    |                  +-----------+   |
    |                                  |
    |              Slave               |
    +----------------------------------+
```

**MySQL 主从复制配置：**

```bash
# ======== Master 配置 (/etc/my.cnf) ========
[mysqld]
server-id = 1
log-bin = mysql-bin
binlog_format = ROW              # 推荐使用 ROW 格式
binlog_row_image = FULL
binlog_expire_logs_seconds = 604800  # MySQL 8.0+ 推荐使用（替代已弃用的 expire_logs_days）
gtid_mode = ON                   # 启用 GTID（全局事务 ID）
enforce_gtid_consistency = ON
sync_binlog = 1                  # 每次事务都同步 Binlog 到磁盘
innodb_flush_log_at_trx_commit = 1

# 半同步复制（减少数据丢失）
plugin-load = rpl_semi_sync_master=semisync_master.so
rpl_semi_sync_master_enabled = 1
rpl_semi_sync_master_wait_for_slave_count = 1

# ======== Slave 配置 (/etc/my.cnf) ========
[mysqld]
server-id = 2
relay-log = relay-bin
read_only = ON                   # 从库只读
log-bin = mysql-bin              # 从库也开启 Binlog（级联复制需要）
gtid_mode = ON
enforce_gtid_consistency = ON

# 半同步复制
plugin-load = rpl_semi_sync_slave=semisync_slave.so
rpl_semi_sync_slave_enabled = 1

# ======== 创建复制用户（在 Master 上执行）========
mysql -u root -p

CREATE USER 'repl_user'@'%' IDENTIFIED BY 'Repl@Pass2024';
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'%';
FLUSH PRIVILEGES;

# ======== 配置主从关系（在 Slave 上执行）========
CHANGE MASTER TO
    MASTER_HOST='10.0.1.11',
    MASTER_PORT=3306,
    MASTER_USER='repl_user',
    MASTER_PASSWORD='Repl@Pass2024',
    MASTER_AUTO_POSITION=1;      -- 使用 GTID 自动定位

START SLAVE;

# 查看复制状态
SHOW SLAVE STATUS\G
-- 关键指标：
-- Slave_IO_Running: Yes
-- Slave_SQL_Running: Yes
-- Seconds_Behind_Master: 0（延迟秒数）
```

**复制模式对比：**

| 模式 | 延迟 | 数据安全 | 性能影响 | 适用场景 |
|------|------|---------|---------|---------|
| 异步复制 | 最低 | 可能丢数据 | 最小 | 对数据一致性要求不高 |
| 半同步复制 | 较低 | 基本不丢 | 中等 | 金融、交易场景 |
| 全同步复制 | 最高 | 不丢数据 | 最大 | 极端一致性要求 |

### 7.5.2 读写分离

**读写分离** 是将数据库的读操作和写操作分发到不同的服务器上，主库负责写，从库负责读。

```
                    +------------------+
                    |     应用程序       |
                    +--------+---------+
                             |
                    +--------+---------+
                    |   读写分离中间件    |
                    | (ProxySQL/ShardingSphere) |
                    +--------+---------+
                       |         |
                  写操作 |         | 读操作
                       v         v
                 +-----+---+ +---+-----+
                 | Master  | | Slave 1 |
                 | (Write) | | (Read)  |
                 +---------+ +---------+
                                  |
                            +-----+-----+
                            | Slave 2   |
                            | (Read)    |
                            +-----------+
```

**ProxySQL 读写分离配置：**

```bash
# ======== 安装 ProxySQL ========
# CentOS/RHEL
yum install -y proxysql

# Ubuntu
apt install -y proxysql

# ======== ProxySQL 配置 ========
mysql -u admin -padmin -h 127.0.0.1 -P 6032

# 1. 添加后端 MySQL 服务器
INSERT INTO mysql_servers (hostgroup_id, hostname, port, weight)
VALUES (10, '10.0.1.11', 3306, 1),    -- hostgroup 10 = 写组（Master）
       (20, '10.0.1.12', 3306, 1),    -- hostgroup 20 = 读组（Slave）
       (20, '10.0.1.13', 3306, 1);    -- hostgroup 20 = 读组（Slave）

# 2. 添加监控用户
INSERT INTO mysql_users (username, password, default_hostgroup)
VALUES ('monitor', 'Monitor@2024', 10);

# 3. 配置读写分离规则
INSERT INTO mysql_query_rules (rule_id, active, match_digest, destination_hostgroup, apply)
VALUES (1, 1, '^SELECT.*FOR UPDATE', 10, 1),   -- SELECT FOR UPDATE 走写组
       (2, 1, '^SELECT', 20, 1);                 -- 其他 SELECT 走读组

# 4. 加载配置到运行时
LOAD MYSQL SERVERS TO RUNTIME;
LOAD MYSQL QUERY RULES TO RUNTIME;
LOAD MYSQL USERS TO RUNTIME;

# 5. 保存配置到磁盘
SAVE MYSQL SERVERS TO DISK;
SAVE MYSQL QUERY RULES TO DISK;
SAVE MYSQL USERS TO DISK;
```

**读写分离的注意事项：**

| 问题 | 说明 | 解决方案 |
|------|------|---------|
| 主从延迟 | 从库数据落后于主库 | 半同步复制、关键读走主库 |
| 读写一致性 | 写入后立即读取可能读到旧数据 | 写后读走主库 |
| 事务中读写 | 事务内的读应走主库 | ProxySQL 事务路由 |
| 从库故障 | 从库宕机影响读 | 多从库 + 负载均衡 |

### 7.5.3 分库分表

**分库分表** 是在数据量超过单机承载能力时，将数据分散存储到多个数据库/表中的技术方案。

**分库分表策略：**

```
垂直拆分（按业务拆分）：
+------------------+     +------------------+
|    用户数据库      |     |    订单数据库      |
| user_db          |     | order_db         |
| - user_info      |     | - order_info     |
| - user_profile   |     | - order_item     |
| - user_auth      |     | - payment        |
+------------------+     +------------------+

水平拆分（按规则拆分）：
+------------------+     +------------------+
|  order_db_0      |     |  order_db_1      |
| (user_id % 2=0)  |     | (user_id % 2=1)  |
| - order_0        |     | - order_0        |
| - order_1        |     | - order_1        |
+------------------+     +------------------+

分片键选择原则：
1. 高频查询条件
2. 数据分布均匀
3. 不易变更（如 user_id、order_id）
```

**ShardingSphere-JDBC 分库分表配置：**

```yaml
# application.yml
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://10.0.1.11:3306/order_db_0
        username: app_user
        password: App@Pass2024
      ds1:
        type: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://10.0.1.12:3306/order_db_1
        username: app_user
        password: App@Pass2024

    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds$->{0..1}.t_order_$->{0..3}
            # 分库策略：user_id % 2
            database-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: order_db_mod
            # 分表策略：user_id % 4
            table-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: order_table_mod
            key-generate-strategy:
              column: order_id
              key-generator-name: snowflake

        sharding-algorithms:
          order_db_mod:
            type: MOD
            props:
              sharding-count: 2
          order_table_mod:
            type: MOD
            props:
              sharding-count: 4

        key-generators:
          snowflake:
            type: SNOWFLAKE
            props:
              worker-id: 1

    props:
      sql-show: true
```

**分库分表的挑战：**

| 挑战 | 说明 | 解决方案 |
|------|------|---------|
| 跨库 JOIN | 不同库的表无法直接 JOIN | 冗余字段、应用层组装、数据同步 |
| 分布式事务 | 跨库事务一致性 | Seata、TCC、Saga |
| 全局 ID | 分库后主键冲突 | 雪花算法、Leaf、UUID |
| 数据迁移 | 扩容时需要数据迁移 | 双写、CDC、ShardingSphere |
| 跨库分页 | 不同库的数据合并排序 | 二次查询、流式合并 |

## 7.6 灾难恢复

### 7.6.1 RPO 与 RTO
| 指标 | 说明 | 示例 |
|------|------|------|
| RPO (Recovery Point Objective) | 可容忍的数据丢失量（时间维度） | RPO = 1小时，表示最多丢失 1 小时数据 |
| RTO (Recovery Time Objective) | 可容忍的恢复时间 | RTO = 4小时，表示 4 小时内恢复服务 |

**不同级别的灾备方案：**

| 级别 | RPO | RTO | 方案 | 成本 |
|------|-----|-----|------|------|
| 级别 1 | 24h | 24h | 定期备份 + 冷备 | 低 |
| 级别 2 | 4h | 4h | 增量备份 + 温备 | 中 |
| 级别 3 | 1h | 1h | 实时复制 + 热备 | 高 |
| 级别 4 | 0 | 分钟级 | 异地多活 | 极高 |

### 7.6.2 灾备方案
- **冷备**：数据备份到异地，不运行服务。恢复时间长，成本低
- **温备**：有备用系统和数据，但不实时同步。恢复时间中等
- **热备**：实时同步，自动切换。恢复时间短，成本高
- **异地多活**：多个区域同时提供服务，自动故障切换。成本最高

**灾备演练：**

```bash
# MySQL 灾备恢复演练脚本
#!/bin/bash

# 1. 全量备份（每周）
mysqldump -u root -p --single-transaction \
    --routines --triggers --events \
    --all-databases | gzip > /backup/full_$(date +%Y%m%d).sql.gz

# 2. 增量备份（每天）
mysqldump -u root -p --single-transaction \
    --where="true" db_name > /backup/inc_$(date +%Y%m%d).sql

# 3. 基于 Binlog 的 Point-in-Time Recovery
# 查看当前 Binlog 位置
mysqlbinlog --start-datetime="2024-01-01 00:00:00" \
    --stop-datetime="2024-01-15 12:00:00" \
    /var/lib/mysql/mysql-bin.000001 | mysql -u root -p

# 4. 验证备份
gunzip < /backup/full_20240101.sql.gz | mysql -u root -p --check
```

## 7.7 面试常见问题

### Q1: LVS 和 Nginx 的区别？分别适用什么场景？

**参考答案：**

| 维度 | LVS | Nginx |
|------|-----|-------|
| 工作层 | L4（传输层） | L4/L7（传输层/应用层） |
| 工作位置 | 内核态（IPVS） | 用户态 |
| 性能 | 极高（百万级并发） | 高（万级并发） |
| 功能 | 简单转发 | 路由、重写、缓存、SSL |
| 健康检查 | 基础 | 丰富（HTTP、TCP、gRPC） |
| 配置复杂度 | 较高 | 较低 |

**适用场景：**
- **LVS**：适合作为最前端的大规模流量分发，如运营商、大型互联网公司的入口层负载均衡
- **Nginx**：适合需要七层路由、SSL 终止、静态资源缓存等功能的场景
- **最佳实践**：LVS（四层）+ Nginx（七层）组合使用，LVS 做第一层分发，Nginx 做第二层路由

### Q2: 主备和主主模式的优缺点？

**参考答案：**

| 模式 | 优点 | 缺点 |
|------|------|------|
| 主备 (Active-Passive) | 实现简单、数据一致性好 | 备节点资源浪费（通常 50%）、切换有延迟 |
| 主主 (Active-Active) | 资源利用率高、无切换延迟 | 实现复杂、需解决数据冲突、脑裂风险 |

**选择建议：**
- 对数据一致性要求高、预算有限 → 主备模式
- 无状态服务（如 Web）、需要最大化资源利用 → 主主模式
- 数据库双主需要谨慎，建议使用中间件（如 Orchestrator、MHA）管理

### Q3: 什么是 RPO 和 RTO？

**参考答案：**

- **RPO（Recovery Point Objective，恢复点目标）**：指灾难发生后，系统可能丢失的数据量的时间度量。RPO = 1 小时意味着最多丢失最近 1 小时的数据。
- **RTO（Recovery Time Objective，恢复时间目标）**：指灾难发生后，系统恢复到正常运行状态所需的最大时间。

**举例：**
- 银行核心系统：RPO = 0（零数据丢失），RTO = 分钟级
- 企业官网：RPO = 1 小时，RTO = 4 小时
- 内部工具：RPO = 24 小时，RTO = 24 小时

### Q4: 如何设计一个跨区域的高可用架构？

**参考答案：**

```
1. 基础架构：
   - 至少选择 2 个 AWS Region（如 us-east-1 和 us-west-2）
   - 每个 Region 至少 2 个 AZ
   - 使用 Route 53 做全局 DNS 和健康检查

2. 计算层：
   - EC2/ECS/EKS 多 AZ 部署
   - Auto Scaling 跨 AZ 扩缩容
   - AMI 跨 Region 复制

3. 数据层：
   - RDS Multi-AZ（同 Region 高可用）
   - RDS 跨 Region Read Replica（异地灾备）
   - DynamoDB 开启全局表（Global Table）

4. 网络层：
   - VPC 多 AZ 设计
   - Transit Gateway 跨 Region 互联

5. 故障切换：
   - Route 53 健康检查 + 自动故障转移
   - RPO 接近 0（同步复制），RTO 分钟级
```

### Q5: Keepalived 的 VRRP 是如何工作的？

**参考答案：**

1. **选举**：Keepalived 启动后，根据 priority 选举 Master（优先级最高的节点）
2. **广播**：Master 每隔 advert_int（默认 1 秒）向组播地址 224.0.0.18 发送 VRRP 广播报文
3. **监听**：Backup 节点持续监听 VRRP 报文
4. **故障检测**：如果 Backup 在 master_down_interval（默认 3 秒）内未收到报文，判定 Master 故障
5. **切换**：Backup 发起选举，优先级最高的 Backup 接管 VIP
6. **抢占**：原 Master 恢复后，如果 preempt 开启且 priority 更高，会重新夺回 VIP

**脑裂防护：**
- 使用 vrrp_strict 模式
- 添加仲裁脚本（如检查第三方服务）
- 使用防火墙限制 VRRP 报文的传输范围

### Q6: 集群和分布式有什么区别？

**参考答案：**

| 维度 | 集群 | 分布式 |
|------|------|--------|
| **核心目标** | 提升可用性和性能 | 拆分业务逻辑，解耦系统 |
| **耦合度** | 较高（同构系统） | 较低（异构系统） |
| **部署方式** | 同构为主，部署相同应用 | 异构常见，部署不同服务 |
| **用户感知** | 对用户透明（像一台机器） | 业务层面可见（不同服务） |
| **通信方式** | 内部通信协议 | RPC / HTTP / 消息队列 |
| **典型代表** | LVS 集群、K8s 集群 | 微服务架构、SOA |

**简单理解：**
- 集群是"多台机器做同一件事"（如多个 Web 服务器）
- 分布式是"多台机器做不同的事，组合完成一个目标"（如用户服务、订单服务、支付服务）
- 实际项目中，集群和分布式经常结合使用（分布式系统中的每个服务都可以是一个集群）
