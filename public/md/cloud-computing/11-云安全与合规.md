## 11.1 云安全概述

**关键原则：**

- **"云厂商负责云本身的安全，用户负责云内部的安全"**
- 云厂商提供安全工具和功能，但用户必须正确配置和使用
- 即使使用 PaaS/SaaS，用户仍需对数据安全和访问控制负责
- 责任边界随服务模型（IaaS/PaaS/SaaS）不同而变化

## 11.2 网络安全

### 11.2.1 安全组（Security Group）

安全组（Security Group）是 AWS 中实例级别的虚拟防火墙，用于控制进出 EC2 实例的网络流量。它是云网络安全的第一道防线。

**核心特性：**

- **有状态（Stateful）**：如果允许入站流量，则自动允许对应的出站响应流量，无需单独配置
- **实例级别**：直接绑定到 EC2 实例（ENI），一个实例可以关联多个安全组（最多 5 个）
- **默认拒绝**：默认拒绝所有入站流量，允许所有出站流量
- **引用规则**：可以引用其他安全组作为来源/目标，实现灵活的安全组链
- **即时生效**：规则变更立即生效，无需重启实例

**安全组规则配置示例：**

```json
// Web 层安全组
{
  "SecurityGroupRules": [
    {
      "RuleId": "sg-rule-001",
      "Direction": "ingress",
      "IpProtocol": "tcp",
      "FromPort": 80,
      "ToPort": 80,
      "IpRanges": [{"CidrIp": "0.0.0.0/0", "Description": "HTTP from internet"}]
    },
    {
      "RuleId": "sg-rule-002",
      "Direction": "ingress",
      "IpProtocol": "tcp",
      "FromPort": 443,
      "ToPort": 443,
      "IpRanges": [{"CidrIp": "0.0.0.0/0", "Description": "HTTPS from internet"}]
    },
    {
      "RuleId": "sg-rule-003",
      "Direction": "ingress",
      "IpProtocol": "tcp",
      "FromPort": 22,
      "ToPort": 22,
      "UserIdGroupPairs": [{"GroupId": "sg-bastion", "Description": "SSH from bastion only"}]
    }
  ]
}
```

**Terraform 安全组配置示例：**

```hcl
# Web 层安全组
resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Web 服务器安全组"
  vpc_id      = aws_vpc.main.id

  # 仅允许来自 ALB 的 HTTP/HTTPS 流量
  ingress {
    description     = "HTTP from ALB"
    from_port       = 80
    to_port         = 80
    protocol        = "tcp"
    security_groups = [aws_security_group.alb.id]
  }

  ingress {
    description     = "HTTPS from ALB"
    from_port       = 443
    to_port         = 443
    protocol        = "tcp"
    security_groups = [aws_security_group.alb.id]
  }

  # SSH 仅允许从堡垒机访问
  ingress {
    description     = "SSH from Bastion"
    from_port       = 22
    to_port         = 22
    protocol        = "tcp"
    security_groups = [aws_security_group.bastion.id]
  }

  # 允许所有出站流量
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = { Name = "web-sg" }
}

# 数据库层安全组 - 仅允许 Web 层访问
resource "aws_security_group" "database" {
  name        = "database-sg"
  description = "数据库安全组"
  vpc_id      = aws_vpc.main.id

  # PostgreSQL 仅允许从 Web 安全组访问
  ingress {
    description     = "PostgreSQL from Web tier"
    from_port       = 5432
    to_port         = 5432
    protocol        = "tcp"
    security_groups = [aws_security_group.web.id]
  }

  # MySQL 仅允许从 Web 安全组访问
  ingress {
    description     = "MySQL from Web tier"
    from_port       = 3306
    to_port         = 3306
    protocol        = "tcp"
    security_groups = [aws_security_group.web.id]
  }

  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = { Name = "database-sg" }
}
```

#### 安全组最佳实践
- 遵循最小权限原则
- 分离 Web 层和 DB 层的安全组
- 禁止 0.0.0.0/0 的 SSH/RDP 访问
- 使用堡垒机（Bastion Host）

### 11.2.2 NACL（网络 ACL）

NACL（Network Access Control List）是子网级别的无状态防火墙，作为安全组的补充层提供网络访问控制。

**核心特性：**

- **无状态（Stateless）**：入站和出站规则必须分别配置。如果允许入站的 HTTP 请求，必须同时配置允许出站的 HTTP 响应（端口 1024-65535）
- **子网级别**：关联到整个子网，子网内所有实例都受 NACL 规则约束
- **默认规则**：默认 NACL 允许所有入站和出站流量；自定义 NACL 默认拒绝所有流量
- **规则编号**：规则按编号从小到大评估（编号越小优先级越高），第一个匹配的规则生效
- **支持拒绝规则**：可以显式定义 DENY 规则（安全组只能 ALLOW）

**NACL vs 安全组对比：**

| 维度 | 安全组（SG） | NACL |
|------|-------------|------|
| 级别 | 实例级别（ENI） | 子网级别 |
| 状态 | 有状态（自动允许响应流量） | 无状态（需显式配置双向规则） |
| 规则评估 | 全部规则一起评估 | 按编号顺序评估，首个匹配生效 |
| 拒绝规则 | 不支持（只能 ALLOW） | 支持 ALLOW 和 DENY |
| 默认行为 | 拒绝所有入站，允许所有出站 | 默认 NACL 全部允许；自定义 NACL 全部拒绝 |

**NACL 配置示例（Terraform）：**

```hcl
# 自定义 NACL - 公有子网
resource "aws_network_acl" "public" {
  vpc_id     = aws_vpc.main.id
  subnet_ids = aws_subnet.public[*].id

  # 允许入站 HTTP
  ingress {
    protocol   = "tcp"
    rule_no    = 100
    action     = "allow"
    cidr_block = "0.0.0.0/0"
    from_port  = 80
    to_port    = 80
  }

  # 允许入站 HTTPS
  ingress {
    protocol   = "tcp"
    rule_no    = 110
    action     = "allow"
    cidr_block = "0.0.0.0/0"
    from_port  = 443
    to_port    = 443
  }

  # 允许入站临时端口（用于返回的响应流量）
  ingress {
    protocol   = "tcp"
    rule_no    = 120
    action     = "allow"
    cidr_block = "0.0.0.0/0"
    from_port  = 1024
    to_port    = 65535
  }

  # 允许所有出站流量
  egress {
    protocol   = "-1"
    rule_no    = 100
    action     = "allow"
    cidr_block = "0.0.0.0/0"
    from_port  = 0
    to_port    = 0
  }

  tags = { Name = "public-nacl" }
}
```

### 11.2.3 VPC 流日志

VPC 流日志（VPC Flow Logs）是捕获 VPC 中网络接口（ENI）之间出入流量的功能，是网络安全监控和故障排查的核心工具。

**核心特性：**

- **捕获粒度**：可以捕获 VPC、子网或单个 ENI 级别的流量
- **捕获内容**：源/目标 IP、端口、协议、包/字节数、动作（ACCEPT/REJECT）、流日志记录原因
- **存储目标**：CloudWatch Logs、S3 或 Kinesis Data Firehose
- **不支持记录**：DNS 流量、DHCP 流量、元数据服务流量、AWS License Manager 流量、实例到实例的流量（同一 ENI 内部）

**VPC 流日志字段说明：**

| 字段 | 说明 | 示例 |
|------|------|------|
| version | 流日志版本 | 3 |
| account-id | AWS 账号 ID | 123456789012 |
| interface-id | 网络接口 ID | eni-0abc123def456 |
| srcaddr | 源 IP 地址 | 10.0.1.10 |
| dstaddr | 目标 IP 地址 | 203.0.113.5 |
| srcport | 源端口 | 443 |
| dstport | 目标端口 | 54321 |
| protocol | 协议编号（6=TCP, 17=UDP, 1=ICMP） | 6 |
| packets | 数据包数量 | 15 |
| bytes | 字节数 | 2048 |
| start | 流开始时间（Unix epoch） | 1620000000 |
| end | 流结束时间（Unix epoch） | 1620000060 |
| action | 动作（ACCEPT/REJECT） | ACCEPT |
| log-status | 日志状态（OK/NODATA/SKIPDATA） | OK |

**VPC 流日志配置示例：**

```hcl
# 创建 VPC 流日志 - 发送到 CloudWatch Logs
resource "aws_flow_log" "main" {
  vpc_id          = aws_vpc.main.id
  traffic_type    = "ALL"  # ALL | ACCEPT | REJECT
  iam_role_arn    = aws_iam_role.vpc_flow_log.arn
  log_destination = aws_cloudwatch_log_group.vpc_flow_log.arn

  tags = { Name = "vpc-flow-log" }
}

resource "aws_cloudwatch_log_group" "vpc_flow_log" {
  name              = "/vpc/flow-logs/${var.project_name}"
  retention_in_days = 90
}

resource "aws_iam_role" "vpc_flow_log" {
  name = "vpc-flow-log-role"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Action = "sts:AssumeRole"
      Effect = "Allow"
      Principal = { Service = "vpc-flow-logs.amazonaws.com" }
    }]
  })
}

resource "aws_iam_role_policy" "vpc_flow_log" {
  name = "vpc-flow-log-policy"
  role = aws_iam_role.vpc_flow_log.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Action = [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents",
        "logs:DescribeLogGroups",
        "logs:DescribeLogStreams"
      ]
      Effect   = "Allow"
      Resource = "*"
    }]
  })
}
```

**VPC 流日志分析场景：**

1. **安全审计**：检测异常出站连接（如挖矿、C2 通信）
2. **故障排查**：分析被安全组或 NACL 拒绝的流量
3. **合规检查**：验证网络隔离策略是否生效
4. **带宽分析**：识别流量模式和 Top Talker
5. **入侵检测**：结合 SIEM 系统检测可疑网络活动

### 11.2.4 网络隔离方案
- **VPC 隔离**：不同环境使用不同 VPC
- **子网隔离**：公有/私有/隔离子网
- **安全组隔离**：按功能分层

## 11.3 身份与访问管理

### 11.3.1 IAM 安全最佳实践

IAM（Identity and Access Management）是云安全的基石，几乎所有云安全事件都与 IAM 配置不当有关。以下是从安全视角出发的 IAM 最佳实践：

**1. 启用 MFA（多因素认证）**

为所有 IAM 用户（尤其是管理员）强制启用 MFA，防止密码泄露后被利用：

```hcl
# IAM 策略 - 强制用户启用 MFA
resource "aws_iam_policy" "force_mfa" {
  name        = "ForceMFA"
  description = "强制用户启用 MFA 才能执行操作"

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid    = "AllowViewAccountInfo"
        Effect = "Allow"
        Action = [
          "iam:GetAccountPasswordPolicy",
          "iam:GetAccountSummary",
          "iam:ListVirtualMFADevices"
        ]
        Resource = "*"
      },
      {
        Sid    = "AllowManageOwnMFA"
        Effect = "Allow"
        Action = [
          "iam:CreateVirtualMFADevice",
          "iam:DeleteVirtualMFADevice",
          "iam:EnableMFADevice",
          "iam:ListMFADevices",
          "iam:ResyncMFADevice"
        ]
        Resource = [
          "arn:aws:iam::*:mfa/${aws:username}",
          "arn:aws:iam::*:user/${aws:username}"
        ]
      },
      {
        Sid    = "DenyAllWithoutMFA"
        Effect = "Deny"
        NotAction = [
          "iam:CreateVirtualMFADevice",
          "iam:EnableMFADevice",
          "iam:GetUser",
          "iam:ListMFADevices",
          "iam:ListVirtualMFADevices",
          "iam:ResyncMFADevice"
        ]
        Condition = {
          Bool = {
            "aws:MultiFactorAuthPresent" = "false"
          }
        }
        Resource = "*"
      }
    ]
  })
}
```

**2. 使用临时凭证（Role 而非长期凭证）**

- 应用程序使用 IAM Role（通过 STS AssumeRole）获取临时凭证，而非使用长期 Access Key
- 临时凭证有效期可配置（最短 15 分钟，最长 12 小时）
- 自动轮换，无需手动管理

**3. 定期轮换 Access Key**

- 如果必须使用 Access Key，设置 90 天自动轮换周期
- 使用 AWS Secrets Manager 或 IAM Credential Report 监控未轮换的密钥
- 删除不活跃的 Access Key

**4. 使用 IAM Policy Conditions 限制条件**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:*",
      "Resource": "arn:aws:s3:::myapp-*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": ["10.0.0.0/8", "192.168.0.0/16"]
        },
        "Bool": {
          "aws:SecureTransport": "true"
        },
        "StringEquals": {
          "aws:PrincipalTag/Department": "Engineering"
        }
      }
    }
  ]
}
```

**5. 审计 IAM 权限**

- 使用 **IAM Access Analyzer** 自动识别过度宽松的权限（如 `*:*`、`s3:*`）
- 定期审查 **IAM Credential Report**（每 4 小时生成一次）
- 使用 **AWS Config** 规则监控 IAM 配置合规性（如 `mfa-enabled-for-iam-console-access`）

- 启用 MFA（多因素认证）
- 使用 Role 而非长期凭证
- 定期轮换 Access Key
- 使用 IAM Policy Conditions 限制条件
- 审计 IAM 权限（Access Analyzer）

### 11.3.2 跨账号访问

跨账号访问是多账号 AWS 架构中的常见需求，通过 IAM Role 的 AssumeRole 机制实现安全的跨账号授权。

**工作原理：**

```
源账号（Account A: 111111111111）         目标账号（Account B: 222222222222）
┌─────────────────────────┐              ┌─────────────────────────┐
│  应用程序 / IAM User     │              │  S3 Bucket / RDS 等     │
│                         │              │                         │
│  1. 调用 STS.AssumeRole │──STS API──→ │  2. 验证信任策略         │
│     (提供 Role ARN)      │              │  3. 返回临时凭证         │
│                         │←─ 临时凭证 ──│     (AccessKeyId,        │
│  4. 使用临时凭证访问资源  │─────────────→│      SecretKey, Session) │
└─────────────────────────┘              └─────────────────────────┘
```

**跨账号角色配置示例：**

```hcl
# ============================================
# 目标账号（Account B: 222222222222）中的配置
# ============================================

# 创建被 Assume 的角色
resource "aws_iam_role" "cross_account_role" {
  name = "CrossAccountAccessRole"

  # 信任策略 - 允许源账号的特定角色 Assume
  assume_role_policy = jsonencode({
```

**ExternalId 防御混淆代理人攻击：**

ExternalId 是一个由调用方生成的唯一标识符，嵌入到信任策略中。它防止"混淆代理人"攻击场景：如果第三方（如 SaaS 供应商）同时为多个客户提供服务，ExternalId 确保第三方只能代表正确的客户 AssumeRole。

**IAM Role Chaining（角色链）：**

角色链是指一个角色 Assume 另一个角色，再由被 Assume 的角色 Assume 第三个角色。限制最多 5 层链式调用，每层 AssumeRole 会减少临时凭证的有效期（最少 15 分钟）。

### 11.3.3 服务账户安全（K8s）

Kubernetes 中的服务账户（ServiceAccount）安全是集群安全的关键环节，涉及 Pod 身份、权限控制和云服务集成。

**ServiceAccount 基础：**

- 每个 Pod 默认使用命名空间中的 `default` ServiceAccount
- ServiceAccount 用于 Pod 对 API Server 的身份认证
- 通过 RBAC（Role/ClusterRole + RoleBinding/ClusterRoleBinding）控制权限

**RBAC 最小权限配置示例：**

**IRSA（IAM Roles for Service Accounts）：**

IRSA 是 AWS EKS 提供的功能，允许为 Kubernetes ServiceAccount 关联 IAM Role，使 Pod 可以直接使用 AWS 服务而无需存储 AWS 凭证。

**IRSA 工作原理：**

```
Pod 启动
  │
  ▼
Mutating Webhook 注入 Sidecar 容器
（aws-iam-authenticator / Amazon EKS Pod Identity Webhook）
  │
  ▼
Pod 中的应用调用 AWS SDK
  │
  ▼
Sidecar 拦截请求，检查环境变量：
  - AWS_ROLE_ARN=arn:aws:iam::111111111111:role/my-app-role

  │

  │
  ▼
返回临时 AWS 凭证给应用
```

### 11.4.1 加密
| 类型 | 说明 | 场景 |
|------|------|------|

| 传输加密 (Encryption in Transit) | 网络通信加密 | TLS/SSL |
| 使用中加密 | 处理数据时加密 | 可信执行环境 |

### 11.4.2 KMS（Key Management Service）

AWS KMS（Key Management Service）是一个托管的密钥管理服务，用于创建和管理加密密钥，提供信封加密（Envelope Encryption）机制。

**KMS 核心概念：**

| 概念 | 说明 |
|------|------|
| CMK（Customer Master Key） | 客户主密钥，用于加密数据密钥。包含密钥材料和元数据 |
| 数据密钥（Data Key） | 用于加密实际数据的对称密钥，由 CMK 生成和加密 |
| 信封加密 | 用 CMK 加密数据密钥，用数据密钥加密数据。数据密钥随数据一起存储 |
| Key Policy | 控制 CMK 的访问权限（谁可以使用/管理密钥） |
| Key Rotation | 自动轮换 CMK 的密钥材料（每年一次，不影响已加密数据） |
| Key Alias | 密钥别名，便于引用 CMK |

**信封加密（Envelope Encryption）原理图：**

```
加密流程：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  1. 应用程序请求加密数据                                       │
│         │                                                    │
│         ▼                                                    │
│  2. 调用 KMS GenerateDataKey API                             │
│     请求: CMK ID, KeySpec: AES_256                           │
│         │                                                    │
│         ▼                                                    │
│  3. KMS 返回：                                               │
│     ├── 明文数据密钥（Plaintext Data Key）                     │
│     └── 加密后的数据密钥（Encrypted Data Key，用 CMK 加密）     │
│         │                                                    │
│         ▼                                                    │

│         │                                                    │
│         ▼                                                    │
│  5. 丢弃明文数据密钥（从内存中安全擦除）                         │
│         │                                                    │
│         ▼                                                    │
│  6. 存储加密后的数据和加密后的数据密钥                           │
│                                                              │
│  ┌─────────────────────────────────────────────┐             │
│  │           存储内容                            │             │
│  │  ┌───────────────────────────────────────┐  │             │
│  │  │  加密数据密钥（Encrypted Data Key）     │  │             │
│  │  │  由 CMK 加密                           │  │             │
│  │  └───────────────────────────────────────┘  │             │
│  │  ┌───────────────────────────────────────┐  │             │
│  │  │  加密数据（Encrypted Data）             │  │             │
│  │  │  由数据密钥加密                         │  │             │
│  │  └───────────────────────────────────────┘  │             │
│  └─────────────────────────────────────────────┘             │
│                                                              │
└──────────────────────────────────────────────────────────────┘

解密流程：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  1. 应用程序读取加密数据和加密后的数据密钥                       │
│         │                                                    │
│         ▼                                                    │
│  2. 调用 KMS Decrypt API                                     │
│     请求: 加密后的数据密钥                                     │
│         │                                                    │
│         ▼                                                    │
│  3. KMS 使用 CMK 解密，返回明文数据密钥                         │
│         │                                                    │
│         ▼                                                    │
│  4. 应用程序使用明文数据密钥解密数据                             │
│         │                                                    │
│         ▼                                                    │
│  5. 丢弃明文数据密钥                                          │
│                                                              │
└──────────────────────────────────────────────────────────────┘

信封加密的优势：
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐             │
│  │  数据 A   │     │  数据 B   │     │  数据 C   │             │
│  │ 数据密钥A │     │ 数据密钥B │     │ 数据密钥C │             │
│  └────┬─────┘     └────┬─────┘     └────┬─────┘             │
│       │                │                │                    │
│       └────────────────┼────────────────┘                    │
│                        │                                     │
│                        ▼                                     │
│              ┌──────────────────┐                            │
│              │   CMK（客户主密钥） │                            │
│              │   加密所有数据密钥  │                            │
│              └──────────────────┘                            │
│                                                              │
│  优势：                                                      │
│  1. 性能：数据密钥在本地使用，无需每次都调用 KMS API            │
```

### 11.4.3 密钥轮换

密钥轮换（Key Rotation）是密钥生命周期管理的关键环节，用于限制单个密钥的使用时间和影响范围。

**KMS 自动轮换：**

- AWS KMS 支持对 **客户托管 CMK（Customer Managed CMK）** 启用自动轮换
- 轮换周期：**每年一次**（365 天）
- 轮换机制：KMS 保留所有历史密钥材料版本，使用新密钥材料加密新数据，旧数据仍可使用旧密钥材料解密
- **注意**：自动轮换仅轮换密钥材料（cryptographic material），不会改变 CMK 的 ARN 和策略

```hcl
# 启用自动密钥轮换
resource "aws_kms_key" "app_key" {
  description             = "应用数据加密密钥"
  enable_key_rotation     = true  # 启用自动轮换
  deletion_window_in_days = 30
  policy                  = data.aws_iam_policy_document.key_policy.json

  tags = { Name = "app-data-key" }
}
```

**手动轮换策略：**

对于不支持自动轮换的场景（如自管理的加密密钥），应制定手动轮换策略：

1. **创建新密钥**：生成新的加密密钥
2. **重新加密数据**：使用新密钥重新加密现有数据
3. **更新应用配置**：将应用切换到新密钥
4. **验证**：确认所有数据可使用新密钥解密
5. **废弃旧密钥**：在确认无误后安全销毁旧密钥
6. **轮换周期建议**：对称密钥每年轮换，非对称密钥每 1-2 年轮换

**密钥轮换最佳实践：**

- 启用 KMS 自动轮换（客户托管 CMK）
- 使用密钥别名（Alias）引用密钥，轮换时只需更新别名指向
- 记录密钥轮换事件到审计日志
- 定期测试密钥恢复流程
- 设置密钥删除保护（30 天等待期）

### 11.4.4 数据分类

数据分类是数据安全治理的基础，根据数据的敏感程度和业务影响进行分级管理。

**数据分类框架：**

| 级别 | 名称 | 说明 | 示例 | 加密要求 | 访问控制 |
|------|------|------|------|---------|---------|
| L1 | 公开（Public） | 可公开发布的信息 | 营销材料、公开文档 | 可选 | 低 |
| L2 | 内部（Internal） | 仅限内部使用 | 内部邮件、项目文档 | 传输加密 | 部门级别 |
| L3 | 机密（Confidential） | 敏感业务数据 | 客户信息、财务数据 | 静态+传输加密 | 角色级别 |
| L4 | 受限（Restricted） | 高度敏感/受法规保护 | PII、医疗记录、支付卡数据 | 端到端加密+信封加密 | 个人级别+审计 |

**数据分类实施步骤：**

1. **发现**：使用工具（如 AWS Macie、Microsoft Purview）自动扫描和发现数据资产
2. **标记**：为数据打上分类标签（如 `Classification: Confidential`）
3. **保护**：根据分类级别应用相应的安全控制
4. **监控**：持续监控数据访问和使用情况
5. **审计**：定期审查数据分类的准确性和安全控制的有效性

**AWS Macie 数据分类示例：**

```hcl

    daily_schedule {}
  }
  s3_job_definition {
    bucket_definitions {
      buckets = ["myapp-data-bucket"]
    }
  }
  name            = "s3-sensitive-data-scan"
  status          = "RUNNING"
  custom_data_identifiers {
    name = "custom-identifier-employee-id"

  }
```

## 11.5 容器安全

### 11.5.1 镜像安全
- 使用官方/可信基础镜像
- 定期扫描镜像漏洞（Trivy、Snyk）
- 多阶段构建减少攻击面
- 镜像签名验证

**镜像安全加固 Dockerfile 示例：**

```dockerfile
# 多阶段构建 - 减少最终镜像大小和攻击面
# ============================================
# 阶段 1: 构建阶段
# ============================================
FROM golang:1.22-alpine AS builder

# 安全：不缓存 git 凭证
ENV GOPRIVATE=*
RUN git config --global credential.helper cache --timeout=0

WORKDIR /app

# 先复制依赖文件，利用 Docker 缓存层
COPY go.mod go.sum ./
RUN go mod download && go mod verify

COPY . .

# 安全编译：禁用 CGO，启用 PIE（位置无关可执行文件）
RUN CGO_ENABLED=0 GOOS=linux go build \
    -ldflags="-s -w -extldflags '-static'" \
    -trimpath \
    -o /app/server ./cmd/server

# ============================================
# 阶段 2: 运行阶段
# ============================================
FROM gcr.io/distroless/static-debian12:nonroot

# 安全：使用非 root 用户（distroless:nonroot 默认 UID 65534）
# 无需手动创建用户，镜像已内置

# 安全：复制 CA 证书（用于 HTTPS 请求）
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

# 复制编译好的二进制文件
COPY --from=builder /app/server /app/server

# 安全：设置文件权限
RUN chmod 755 /app/server

# 安全：暴露必要端口
EXPOSE 8080

# 安全：使用非 root 用户运行
USER nonroot:nonroot

ENTRYPOINT ["/app/server"]
```

**镜像扫描（Trivy）：**

```bash
# 扫描本地镜像
trivy image --severity HIGH,CRITICAL myapp:latest

# 扫描并生成 JSON 报告
trivy image --format json --output report.json myapp:latest

# 扫描 Git 仓库中的 Dockerfile 和依赖
trivy fs --severity HIGH,CRITICAL ./myapp

# CI/CD 中使用（发现高危漏洞时退出码非 0）
trivy image --exit-code 1 --severity CRITICAL myapp:latest
```

### 11.5.2 运行时安全
- 非 root 用户运行
- 只读根文件系统
- Seccomp / AppArmor 限制
- 资源限制防止 DoS

**Pod 安全加固示例：**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
  namespace: production
spec:
  serviceAccountName: app-service-account
  automountServiceAccountToken: false  # 不自动挂载 SA Token

  securityContext:
    runAsNonRoot: true          # 强制非 root 运行
    runAsUser: 1000             # 指定运行用户 UID
    runAsGroup: 3000            # 指定运行组 GID
    fsGroup: 2000               # 文件系统组
    seccompProfile:             # Seccomp 配置
      type: RuntimeDefault

  containers:
    - name: app
      image: myapp:latest
      imagePullPolicy: Always

      securityContext:
        allowPrivilegeEscalation: false  # 禁止提权
        readOnlyRootFilesystem: true     # 只读根文件系统
        capabilities:
          drop: ["ALL"]                  # 移除所有 Linux capabilities
        privileged: false                # 禁止特权模式

      # 只读根文件系统需要挂载临时目录
      volumeMounts:
        - name: tmp
          mountPath: /tmp
        - name: cache
          mountPath: /var/cache
        - name: app-logs
          mountPath: /app/logs

      # 资源限制 - 防止 DoS
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "512Mi"

      # 健康检查
      livenessProbe:
        httpGet:
          path: /healthz
          port: 8080
        initialDelaySeconds: 10
        periodSeconds: 15
        failureThreshold: 3

      readinessProbe:
```

### 11.5.3 K8s 安全

**Pod Security Standards（Pod 安全标准）：**

Kubernetes 从 v1.23 开始弃用 PodSecurityPolicy（PSP），推荐使用 Pod Security Standards（PSS）配合 Pod Security Admission（PSA）控制器。

**三个内置安全策略级别：**

| 级别 | 说明 | 适用场景 |
|------|------|---------|
| **Privileged**（特权） | 不限制任何操作，完全开放 | 系统级基础设施（如 CNI、CSI） |
| **Baseline**（基线） | 防止已知的特权提升攻击，最小安全保证 | 通用应用 |
| **Restricted**（受限） | 实施全面的保护策略，遵循最佳实践 | 安全敏感应用 |

**Baseline 策略控制项：**

| 控制 | 说明 |
|------|------|
| HostProcess | 禁止以宿主机进程运行 |
| Host Namespaces | 禁止共享宿主机网络/PID/IPC 命名空间 |
| Host Ports | 禁止使用宿主机端口 |
| Privileged Containers | 禁止特权容器 |
| Capabilities | 限制添加默认 capabilities |
| HostPath Volumes | 限制 HostPath 挂载 |
| SELinux | 限制 SELinux 自定义配置 |
| /proc Mount Type | 限制 /proc 挂载类型 |
| Seccomp | 要求使用 Seccomp 配置 |

**Restricted 策略额外控制项：**

| 控制 | 说明 |
|------|------|
| Run As NonRoot | 强制非 root 运行 |
| Run As User | 限制可运行的用户 |
| Seccomp | 必须使用 RuntimeDefault 或自定义 Seccomp |
| Capabilities | 仅允许默认 capabilities 子集 |
| Volumes | 限制卷类型（禁止 hostPath、gcePersistentDisk 等） |

**命名空间级别的 Pod 安全策略配置：**

```yaml
# 为命名空间应用 Pod 安全标准
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    # 强制执行 Baseline 策略（违规 Pod 被拒绝）
    pod-security.kubernetes.io/enforce: baseline
    # 审计模式：记录违规但不拒绝（用于评估）
    pod-security.kubernetes.io/audit: restricted
    # 告警模式：记录违规但不拒绝
    pod-security.kubernetes.io/warn: restricted
    # 审计版本
    pod-security.kubernetes.io/audit-version: latest
    # 告警版本
    pod-security.kubernetes.io/warn-version: latest
```

**使用 Gatekeeper / Kyverno 实施更细粒度的安全策略：**

```yaml

  name: disallow-latest-tag
spec:

# Kyverno - 要求所有 Pod 设置资源限制

  validationFailureAction: Enforce
  rules:
    - name: check-limits
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: "所有容器必须设置 CPU 和内存的 requests 和 limits"
        pattern:
          spec:
            containers:
              - resources:
                  requests:
                    memory: "?*"
                    cpu: "?*"
                  limits:
                    memory: "?*"
                    cpu: "?*"

# Kyverno - 禁止特权容器

    - name: no-privileged

        any:

      validate:
        message: "禁止特权容器"
        pattern:
          spec:
            containers:
              - =(securityContext):
                  =(privileged): "false"
```

### 11.6.4 Falco（容器运行时安全）

Falco 是一个云原生运行时安全工具，由 Sysdig 开源并捐赠给 CNCF，用于检测容器、Kubernetes 和主机层面的异常行为。

**核心特性：**

- **系统调用监控**：通过 Linux 内核的 eBPF 或内核模块捕获系统调用
- **规则引擎**：使用灵活的规则语法定义检测规则
- **实时告警**：检测到异常行为时实时输出告警
- **Kubernetes 原生**：自动感知 Pod/容器元数据，支持基于 K8s 标签的规则过滤
- **CNCF 毕业**：作为 CNCF 毕业项目，生态成熟

**Falco 规则语法结构：**

```
规则名: 检测条件
  条件: 系统调用 + 参数过滤 + 输出字段
  输出: 告警消息模板
  优先级: CRITICAL / HIGH / MEDIUM / LOW / INFO
```

**Falco 规则示例：**

```yaml
# ============================================
# Falco 规则配置文件: /etc/falco/falco_rules.local.yaml
# ============================================

# 1. 检测容器中启动 Shell（可能被入侵后反弹 Shell）
- rule: Terminal Shell in Container
  desc: 检测在容器中启动交互式 Shell
  condition: >
    spawned_process and container and
    proc.name in (bash, sh, zsh, fish, dash, ksh, tcsh, csh)
    and proc.tty != 0
    and not user.name in (root)
    and container.image.repository != "falco"
  output: >
    在容器中检测到交互式 Shell 启动
    (user=%user.name container=%container.name
     image=%container.image.repository shell=%proc.name
     parent=%proc.pname cmdline=%proc.cmdline)
  priority: WARNING
  tags: [container, shell, mitre_execution]

# 2. 检测容器中读取敏感文件
- rule: Read Sensitive File in Container
  desc: 检测容器中读取 /etc/shadow、/etc/passwd 等敏感文件
  condition: >
    open_read and container and
    fd.name in (/etc/shadow, /etc/sudoers, /etc/pam.d/*,
                /root/.ssh/id_rsa, /root/.aws/credentials,
                /var/run/secrets/kubernetes.io/serviceaccount/token)
    and not proc.name in (sshd, login, sudo, su, passwd, nginx, httpd)
  output: >
    检测到容器中读取敏感文件
    (user=%user.name container=%container.name
     image=%container.image.repository file=%fd.name
     process=%proc.name cmdline=%proc.cmdline)
  priority: CRITICAL
  tags: [container, sensitive_file, mitre_credential_access]

# 3. 检测容器中创建可疑网络连接（反弹 Shell / C2 通信）
- rule: Suspicious Outbound Connection from Container
  desc: 检测容器中建立可疑出站连接
  condition: >
    outbound and container and
    (evt.type in (connect, accept) and
     fd.l4proto = tcp and
     fd.sport in (4444, 5555, 6666, 8888, 9999, 31337, 12345))
  output: >
    检测到容器中建立可疑出站连接
    (user=%user.name container=%container.name
     image=%container.image.repository
     connection=%fd.l4proto/%fd.sport->%fd.rport
     process=%proc.name cmdline=%proc.cmdline)
  priority: HIGH
  tags: [container, network, mitre_command_and_control]

# 4. 检测容器中安装软件包（可能被植入后门）
- rule: Package Management in Container
  desc: 检测容器中运行包管理器
  condition: >
    spawned_process and container and
    proc.name in (apt, apt-get, yum, dnf, pip, pip3, npm, apk, gem)
    and container.image.repository != "build"
  output: >
    检测到容器中运行包管理器
    (user=%user.name container=%container.name
     image=%container.image.repository
     package_manager=%proc.name cmdline=%proc.cmdline)
  priority: INFO
  tags: [container, package_manager]

# 5. 检测 Kubernetes ConfigMap/Secret 挂载后的访问
- rule: Access K8s Secret Volume
  desc: 检测非预期进程访问 K8s Secret 挂载卷
  condition: >
    open_read and container and
    fd.name startswith "/var/run/secrets/" and
    not proc.name in (kubelet, etcd, api-server, coredns)
  output: >
    检测到非预期进程访问 K8s Secret
    (user=%user.name container=%container.name
     file=%fd.name process=%proc.name cmdline=%proc.cmdline)
  priority: WARNING
  tags: [container, k8s, secret]

# 6. 检测容器中提权操作
- rule: Privilege Escalation in Container
  desc: 检测容器中的提权操作
  condition: >
    spawned_process and container and
    proc.name in (su, sudo, chown, chmod, mount, umount, insmod, modprobe)
    and not user.name = root
  output: >
    检测到容器中可能的提权操作
    (user=%user.name container=%container.name
     image=%container.image.repository
     process=%proc.name cmdline=%proc.cmdline
     parent=%proc.pname)
  priority: HIGH
  tags: [container, privilege_escalation, mitre_privilege_escalation]

# 7. 检测异常的容器挂载操作
- rule: Mount Sensitive Host Paths
  desc: 检测容器挂载宿主机敏感路径
  condition: >
    open_write and container and
    fd.name in (/etc, /boot, /lib, /lib64, /usr, /var/run/docker.sock)
    and fd.type = 'directory'
  output: >
    检测到容器挂载宿主机敏感路径
    (user=%user.name container=%container.name
     image=%container.image.repository path=%fd.name)
  priority: CRITICAL
  tags: [container, mount, mitre_persistence]
```

### 11.7.2 AWS Artifact

AWS Artifact 是一项合规报告下载服务，提供 AWS 云基础设施的合规性报告和安全认证文档。

**核心功能：**

- **合规报告下载**：免费下载 AWS 的 SOC 1/2/3、ISO 27001、PCI DSS、HIPAA、FedRAMP 等合规报告
- **协议管理**：在线接受和查看 AWS 的数据处理协议（DPA）、Business Associate Agreement（BAA）等法律协议
- **报告类型**：
  - **SOC Reports**：SOC 1/2/3 Type I 和 Type II 报告
  - **ISO Certificates**：ISO 27001、ISO 27017、ISO 27018 认证
  - **PCI Attestation**：PCI DSS 合规证明（AOC）
  - **HIPAA BAA**：HIPAA 业务伙伴协议
  - **FedRAMP Authorization**：美国政府云授权
  - **CSA STAR**：云安全联盟认证

**使用方式：**

1. 登录 AWS 管理控制台
2. 导航到 AWS Artifact（https://console.aws.amazon.com/artifact）
3. 在 "Reports" 标签页中搜索并下载所需报告
4. 在 "Agreements" 标签页中查看和接受协议

**在审计中的应用：**

- 向客户证明 AWS 基础设施的安全合规性
- 作为企业自身合规审计的证据材料
- 满足行业监管要求（如金融、医疗）
- 支持供应商安全评估（VSA）流程

## 11.8 面试常见问题

**Q1: 云安全的共享责任模型是什么？**

**参考答案：**

共享责任模型明确了云厂商和用户在云安全中的责任划分。核心原则是"云厂商负责云本身的安全，用户负责云内部的安全"。

**云厂商责任（Security OF the Cloud）：**
- 数据中心物理安全（门禁、监控、消防）
- 物理服务器和网络设备
- 虚拟化基础设施（Hypervisor）
- 区域和可用区的高可用性
- 云服务本身的运行安全

**用户责任（Security IN the Cloud）：**
- IAM 身份和访问管理（用户、角色、策略）
- 网络安全配置（安全组、NACL、VPC）
- 操作系统安全（补丁、加固）
- 应用安全（代码审计、漏洞管理）
- 数据安全（加密、分类、备份）
- 合规性（行业法规遵从）

**责任随服务模型变化：**
- IaaS（如 EC2）：用户责任最多，需管理 OS、应用、数据
- PaaS（如 RDS/Lambda）：云厂商管理 OS 和运行时，用户管理应用和数据
- SaaS（如 Office 365）：云厂商管理大部分，用户主要管理数据和访问控制

---

**Q2: 如何保护 AWS 中的数据安全？**

**参考答案：**

数据安全需要从三个维度全面保护：

**1. 静态加密（Encryption at Rest）：**
- S3：启用 SSE-S3（默认）或 SSE-KMS（合规要求高时）
- EBS：创建时启用加密，使用 KMS CMK
- RDS：启用存储加密（创建后不可更改），使用 TLS 传输加密
- 使用 KMS 信封加密管理密钥生命周期

**2. 传输加密（Encryption in Transit）：**
- 强制使用 TLS 1.2+（安全组/NACL 拒绝非加密流量）
- S3：启用"阻止非加密上传"（s3:x-amz-content-sha256 策略条件）
- RDS：强制 SSL 连接（rds.force_ssl 参数组）
- API Gateway：启用 TLS 1.2

**3. 访问控制：**
- 最小权限原则：仅授予必要的权限
- 数据分类：根据敏感级别应用不同的保护措施
- 使用 IAM Policy Conditions 限制访问条件（IP、VPC、MFA）
- S3 Bucket Policy 限制公开访问

**4. 密钥管理：**
- 使用 KMS 管理加密密钥
- 启用自动密钥轮换
- 使用不同的 CMK 隔离不同业务的数据

---

**Q3: KMS 信封加密的原理？**

**参考答案：**

信封加密（Envelope Encryption）是 AWS KMS 的核心机制，解决了"如何用少量主密钥安全加密大量数据"的问题。

**加密流程：**
1. 应用程序调用 KMS `GenerateDataKey` API，传入 CMK ID
2. KMS 使用 CMK 生成一个数据密钥（Data Key），返回两个版本：
   - 明文数据密钥（用于在应用内存中加密数据）
   - 加密后的数据密钥（用 CMK 加密，用于存储）
3. 应用程序使用明文数据密钥在本地加密数据（AES-256-GCM）
4. 加密完成后，从内存中安全擦除明文数据密钥
5. 将加密数据和加密后的数据密钥一起存储

**解密流程：**
1. 应用程序读取加密数据和加密后的数据密钥
2. 调用 KMS `Decrypt` API，传入加密后的数据密钥
3. KMS 使用 CMK 解密，返回明文数据密钥
4. 应用程序使用明文数据密钥解密数据
5. 从内存中安全擦除明文数据密钥

**优势：**
- **性能**：数据密钥在本地使用，避免大量数据通过 KMS API 传输
- **安全**：CMK 永远不离开 KMS 服务，降低密钥泄露风险
- **灵活**：不同数据使用不同的数据密钥，限制单个密钥的影响范围
- **合规**：集中管理密钥策略和审计日志

---

**Q4: 容器安全有哪些关键措施？**

**参考答案：**

容器安全需要覆盖整个生命周期：镜像构建 -> 镜像仓库 -> 部署运行 -> 运行时监控。

**1. 镜像安全：**
- 使用可信基础镜像（如 distroless、Alpine）
- 多阶段构建减少攻击面
- 定期扫描镜像漏洞（Trivy、Snyk）
- 镜像签名验证（Cosign、Notary）
- 禁止使用 `latest` 标签

**2. 部署安全：**
- 使用 Pod Security Standards（Baseline/Restricted）
- 非 root 用户运行容器
- 只读根文件系统
- 禁止特权容器
- 资源限制（CPU/内存 requests 和 limits）
- RBAC 最小权限

**3. 运行时安全：**
- Falco 运行时监控（检测异常系统调用）
- Seccomp/AppArmor 限制系统调用
- NetworkPolicy 网络隔离
- Secret 加密存储（使用 Vault 或 KMS）
- Audit Logging

**4. 供应链安全：**
- 使用 SBOM（Software Bill of Materials）
- 签名验证（SLSA 框架）
- 使用私有可信镜像仓库

**Q5: SOC 2 和 ISO 27001 的区别？**

**参考答案：**

| 维度 | SOC 2 | ISO 27001 |
|------|-------|-----------|
| **发布机构** | AICPA（美国注册会计师协会） | ISO（国际标准化组织） |
| **性质** | 审计报告（不是认证） | 国际标准认证 |
| **适用范围** | 主要面向 SaaS 服务商 | 通用，任何组织 |
| **核心内容** | 5 个信任服务标准（安全、可用性、机密性、完整性、隐私） | 114 个控制措施（ISMS 信息安全管理体系） |
| **审计方式** | Type I（设计评估）/ Type II（运行有效性，至少 6 个月） | 第三方认证机构审核 |
| **有效期** | Type II 报告覆盖审计期 | 3 年（需年审） |
| **地域性** | 主要在北美使用 | 全球通用 |
| **公开性** | 报告通常需要 NDA 才能获取 | 证书可以公开查询 |

**选择建议：**
- 面向北美客户（尤其是企业客户）：优先 SOC 2
- 全球化业务：ISO 27001
- 两者互补，建议同时获取

**Q6: 如何实现 K8s 集群的安全隔离？**

**参考答案：**

K8s 集群安全隔离需要从多个层面实施：

**1. 命名空间隔离：**
- 不同团队/环境使用不同命名空间
- 使用 ResourceQuota 限制资源配额
- 使用 LimitRange 限制默认资源

**2. 网络隔离：**
- NetworkPolicy 控制 Pod 间通信（默认允许所有，需要显式拒绝）
- 按命名空间/标签隔离网络流量
- 使用 Cilium 网络策略（支持 L7 策略）

```yaml
# 示例：拒绝所有入站流量，仅允许特定标签的 Pod 访问
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes: ["Ingress"]
  ingress:
    - from:
        - podSelector:
            matchLabels:
              role: frontend
      ports:
        - port: 8080
          protocol: TCP
```

**3. RBAC 隔离：**
- 每个命名空间创建专用 ServiceAccount
- 使用 Role（非 ClusterRole）限制命名空间级别权限
- 定期审计 RBAC 配置

**4. Pod 安全隔离：**
- Pod Security Standards（Baseline/Restricted）
- Kyverno/Gatekeeper 策略引擎
- Seccomp/AppArmor 限制

**5. 节点隔离：**
- 使用 Taint/Toleration 将特定工作负载调度到特定节点
- 节点池隔离（系统节点池 vs 工作节点池）

**Q7: 什么是零信任架构？在云中如何实现？**

**参考答案：**

零信任架构（Zero Trust Architecture，ZTA）是一种安全模型，核心原则是"永不信任，始终验证"（Never Trust, Always Verify），不基于网络位置隐式信任任何用户或设备。

**核心原则：**
1. **最小权限**：仅授予完成任务所需的最小权限
2. **持续验证**：每次访问请求都需要验证身份和上下文
3. **假设违规**：假设网络已被入侵，设计防御措施
4. **微分段**：将网络划分为小段，限制横向移动

**云中零信任实现方案：**

| 层面 | 实现方式 | AWS 服务/工具 |
|------|---------|-------------|
| **身份验证** | MFA + 条件访问 | IAM MFA、Cognito |
| **网络** | 微分段 + 加密 | Security Group、VPC、PrivateLink |
| **应用** | mTLS + 服务身份 | Istio、App Mesh |
| **数据** | 端到端加密 + 分类 | KMS、Macie |
| **端点** | 设备信任 + 合规检查 | AWS Verified Access |
| **监控** | 持续行为分析 | GuardDuty、CloudTrail |

**AWS 零信任架构示例：**

```
用户 → AWS Verified Access（设备信任验证 + 身份验证）
    → API Gateway（认证 + 授权 + 限流）
        → PrivateLink（私有连接，不经过公网）
            → EKS 集群（Istio mTLS 服务间通信）
                → RDS（VPC 内网访问 + IAM 认证 + TLS 加密）
```
