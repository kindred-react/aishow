## 一、浏览器架构与工作流程

### Q1: 现代浏览器的多进程架构是怎样的？各进程分别负责什么？

**难度**: ⭐⭐

**答案**: 现代浏览器采用多进程架构，主要包括 Browser 进程、Renderer 进程、GPU 进程、Plugin 进程、Network 进程等，各进程职责分离，互不干扰。

**解析**:

```
┌─────────────────────────────────────────────────┐
│                  Browser 进程（主进程）             │
│  - 负责界面显示、用户交互（前进后退等）              │
│  - 子进程管理、文件访问、网络请求调度               │
│  - 唯一与用户直接交互的进程                        │
├─────────────────────────────────────────────────┤
│  ┌───────────┐  ┌───────────┐  ┌───────────┐   │
│  │ Renderer  │  │ Renderer  │  │ Renderer  │   │
│  │  进程 A   │  │  进程 B   │  │  进程 C   │   │
│  │ (Tab页1)  │  │ (Tab页2)  │  │ (Tab页3)  │   │
│  └───────────┘  └───────────┘  └───────────┘   │
│  - 页面渲染、JS 执行、CSS 计算                    │
│  - 每个站点（Site）通常一个进程                    │
│  - 沙箱隔离，崩溃不影响其他标签页                  │
├─────────────────────────────────────────────────┤
│  ┌───────────┐  ┌───────────┐  ┌───────────┐   │
│  │   GPU     │  │  Plugin   │  │  Network  │   │
│  │   进程    │  │   进程    │  │   进程    │   │
│  └───────────┘  └───────────┘  └───────────┘   │
│  - 3D渲染/CSS合成  - 插件运行   - 网络请求       │
└─────────────────────────────────────────────────┘
```

**各进程详解**:

| 进程 | 职责 | 数量 |
|------|------|------|
| Browser 进程 | UI 显示、用户交互、子进程管理、文件/网络调度 | 1 个 |
| Renderer 进程 | HTML 解析、CSS 计算、JS 执行、页面渲染 | 多个（按需） |
| GPU 进程 | 图形渲染、CSS 合成、WebGL/3D | 1 个 |
| Plugin 进程 | Flash 等插件运行（已逐渐淘汰） | 按需 |
| Network 进程 | 网络资源请求 | 1 个 |
| Utility 进程 | 音视频解码等工具任务 | 按需 |

**多进程架构的优势**:
- **稳定性**: 某个标签页崩溃不会影响其他标签页
- **安全性**: Renderer 进程运行在沙箱中，无法直接访问系统资源
- **性能**: 多进程充分利用多核 CPU

**注意事项**:
- Chrome 早期每个 Tab 一个 Renderer 进程，内存开销大
- Chrome 后续优化为按 **Site Isolation**（站点隔离）分配进程，同源页面共享进程
- 可在 `chrome://version` 中查看进程信息，在 `chrome://process-internals` 查看进程详情

### Q2: 从输入 URL 到页面渲染完成，中间经历了哪些过程？

**难度**: ⭐⭐⭐

**答案**: 整个过程涉及网络请求、浏览器解析、渲染合成等多个阶段，是前端面试最高频的综合性问题之一。

**解析**:

```
用户输入URL
    │
    ▼
┌─────────────┐
│ 1. URL解析   │ → 判断是搜索关键词还是URL
└──────┬──────┘
       ▼
┌─────────────┐
│ 2. 缓存检查  │ → 强缓存 → 协商缓存 → 服务器请求
└──────┬──────┘
       ▼
┌─────────────┐
│ 3. DNS解析   │ → 本地DNS缓存 → 系统DNS → 递归查询
└──────┬──────┘
       ▼
┌─────────────┐
│ 4. TCP连接   │ → 三次握手 → TLS握手（HTTPS）
└──────┬──────┘
       ▼
┌─────────────┐
│ 5. HTTP请求  │ → 构建请求报文 → 发送请求
└──────┬──────┘
       ▼
┌─────────────┐
│ 6. 服务器处理 │ → 路由 → 业务逻辑 → 返回响应
└──────┬──────┘
       ▼
┌─────────────┐
│ 7. 响应解析  │ → 状态码判断 → Content-Type判断
└──────┬──────┘
       ▼
┌─────────────┐
│ 8. HTML解析  │ → 字节→字符→Token→Node→DOM树
└──────┬──────┘
       ▼
┌─────────────┐
│ 9. CSS解析   │ → 字节→字符→Token→Node→CSSOM树
└──────┬──────┘
       ▼
┌─────────────┐
│10. JS执行   │ → 解析→编译→执行（可能阻塞DOM解析）
└──────┬──────┘
       ▼
┌─────────────┐
│11. 渲染树构建 │ → DOM + CSSOM = Render Tree
└──────┬──────┘
       ▼
┌─────────────┐
│12. 布局Layout│ → 计算节点位置和大小
└──────┬──────┘
       ▼
┌─────────────┐
│13. 绘制Paint │ → 绘制像素（颜色、文字、图片等）
└──────┬──────┘
       ▼
┌─────────────┐
│14. 合成Composite│ → 图层合成→GPU光栅化→显示
└─────────────┘
```

**各阶段详解**:

**1. URL 解析**
- 浏览器判断输入是搜索关键词还是合法 URL
- 如果是 URL，补全协议（默认 HTTPS）

**2. 缓存检查**
- 先检查强缓存（`Cache-Control`、`Expires`）
- 强缓存未命中则检查协商缓存（`ETag`、`Last-Modified`）
- 缓存命中直接使用，不发送请求

**3. DNS 解析**
- 顺序：浏览器 DNS 缓存 → 操作系统 DNS 缓存 → 本地 hosts 文件 → 路由器 DNS → ISP DNS → 递归查询
- DNS 优化：`dns-prefetch` 预解析、DNS 缓存

**4. TCP 连接**
- 三次握手：SYN → SYN-ACK → ACK
- HTTPS 还需要 TLS 握手（1.2 需要 2-RTT，1.3 只需 1-RTT）
- 支持 HTTP/2 多路复用时，可复用已有连接

**5-7. HTTP 请求与响应**
- 构建请求行、请求头、请求体
- 服务器处理并返回响应（状态码、响应头、响应体）

**8-10. 文档解析**
- HTML 解析构建 DOM 树
- CSS 解析构建 CSSOM 树
- 遇到 `<script>` 标签会阻塞 DOM 解析（除非 `async`/`defer`）

**11-14. 渲染流程**
- DOM + CSSOM → Render Tree（渲染树，不包含不可见元素）
- Layout：计算每个节点的精确位置和大小
- Paint：将渲染树转换为屏幕上的像素
- Composite：将各图层合成最终图像

**面试加分点**:
- 提到 TCP 慢启动、拥塞控制
- 提到 HTTP/2 多路复用、服务器推送
- 提到预加载策略（preload/prefetch/preconnect）
- 提到 `DOMContentLoaded` 和 `load` 事件的触发时机

### Q3: 浏览器的预加载策略有哪些？分别有什么作用？

**难度**: ⭐⭐

**答案**: 浏览器提供了 `preload`、`prefetch`、`preconnect`、`dns-prefetch`、`prerender` 等资源提示（Resource Hints），用于优化页面加载性能。

**解析**:

| 指令 | 作用 | 优先级 | 使用场景 |
|------|------|--------|----------|
| `dns-prefetch` | 仅解析域名 DNS | 最低 | 第三方域名 |
| `preconnect` | DNS + TCP + TLS 握手 | 低 | 即将使用的第三方资源 |
| `prefetch` | 低优先级获取资源 | 最低 | 下一页面可能用到的资源 |
| `preload` | 高优先级获取当前页资源 | 高 | 当前页面关键资源 |
| `prerender` | 预渲染整个页面 | 中 | 下一页面（已废弃） |
| `modulepreload` | 预加载 ES Module | 高 | 关键 JS Module |

**代码示例**:

```html
<!-- DNS 预解析：提前解析第三方域名 -->
<link rel="dns-prefetch" href="//cdn.example.com">

<!-- 预连接：DNS + TCP + TLS 一步到位 -->
<link rel="preconnect" href="https://cdn.example.com" crossorigin>

<!-- 预加载：高优先级加载当前页面关键资源 -->
<link rel="preload" href="/fonts/main.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="/css/critical.css" as="style">
<link rel="preload" href="/js/app.js" as="script">

<!-- 预获取：低优先级加载下一页面可能用到的资源 -->
<link rel="prefetch" href="/next-page-data.json">

<!-- 预加载 ES Module -->
<link rel="modulepreload" href="/src/app-module.js">
```

**注意事项**:
- `preload` 必须配合 `as` 属性使用，否则浏览器无法确定资源类型
- `preload` 加载的字体必须设置 `crossorigin`，否则会被重复加载
- `prefetch` 资源会被存入 HTTP 缓存，下次导航时可直接使用
- `prerender` 在 Chrome 109+ 已被废弃，替代方案是 `Speculation Rules API`
- 不要滥用 `preload`，过多高优先级资源会挤占带宽

**实际应用场景**:

```html
<head>
  <!-- 提前解析 CDN 域名 -->
  <link rel="dns-prefetch" href="//cdn.jsdelivr.net">
  <link rel="preconnect" href="https://cdn.jsdelivr.net" crossorigin>

  <!-- 预加载关键字体 -->
  <link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>

  <!-- 预加载首屏关键 CSS -->
  <link rel="preload" href="/css/critical.css" as="style">

  <!-- 预加载首屏图片 -->
  <link rel="preload" href="/hero-banner.webp" as="image" type="image/webp">

  <!-- 预获取下一页面的数据 -->
  <link rel="prefetch" href="/api/page2-data">
</head>
```

### Q4: Service Worker 的生命周期是怎样的？有哪些缓存策略？

**难度**: ⭐⭐⭐

**答案**: Service Worker 生命周期包括安装（install）、激活（activate）、等待（waiting）、终止（terminated）等状态，配合 Cache API 可实现多种缓存策略。

**解析**:

```
注册(register)
    │
    ▼
┌──────────┐
│  安装中   │ ← install 事件
│ installing│   可预缓存静态资源
└────┬─────┘
     │ 安装成功
     ▼
┌──────────┐
│  等待中   │ ← 旧 SW 仍在控制页面
│ waiting  │   skipWaiting() 可跳过
└────┬─────┘
     │ 旧 SW 释放控制权
     ▼
┌──────────┐
│  激活中   │ ← activate 事件
│ activating│   可清理旧缓存
└────┬─────┘
     │ 激活成功
     ▼
┌──────────┐
│  已激活   │ ← fetch/message 事件
│  active  │   拦截网络请求
└────┬─────┘
     │ 页面关闭
     ▼
┌──────────┐
│  已终止   │ ← 空闲时自动终止
│terminated│   需要时重新唤醒
└──────────┘
```

**完整代码示例**:

```javascript
// 注册 Service Worker
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js', { scope: '/' })
    .then(registration => {
      console.log('SW 注册成功，作用域：', registration.scope);
    })
    .catch(err => console.error('SW 注册失败：', err));
}

// sw.js - Service Worker 脚本

// 1. 安装阶段：预缓存关键资源
const CACHE_NAME = 'v1.0.0';
const PRECACHE_URLS = [
  '/',
  '/index.html',
  '/css/style.css',
  '/js/app.js',
  '/images/logo.png'
];

self.addEventListener('install', event => {
  console.log('[SW] 安装中...');
  event.waitUntil(
    caches.open(CACHE_NAME)
      .then(cache => cache.addAll(PRECACHE_URLS))
      .then(() => self.skipWaiting()) // 跳过等待，立即激活
  );
});

// 2. 激活阶段：清理旧版本缓存
self.addEventListener('activate', event => {
  console.log('[SW] 激活中...');
  event.waitUntil(
    caches.keys().then(keys =>
      Promise.all(
        keys.filter(key => key !== CACHE_NAME)
            .map(key => caches.delete(key))
      )
    ).then(() => self.clients.claim()) // 立即控制所有页面
  );
});

// 3. 拦截请求：缓存策略
self.addEventListener('fetch', event => {
  // 策略一：缓存优先（Cache First）
  if (event.request.url.match(/\.(js|css|png|jpg|jpeg|gif|svg|woff2)$/)) {
    event.respondWith(
      caches.match(event.request).then(cached => {
        if (cached) return cached;
        return fetch(event.request).then(response => {
          const clone = response.clone();
          caches.open(CACHE_NAME).then(cache => cache.put(event.request, clone));
          return response;
        });
      })
    );
    return;
  }

  // 策略二：网络优先（Network First）
  if (event.request.url.includes('/api/')) {
    event.respondWith(
      fetch(event.request)
        .then(response => {
          const clone = response.clone();
          caches.open(CACHE_NAME).then(cache => cache.put(event.request, clone));
          return response;
        })
        .catch(() => caches.match(event.request))
    );
    return;
  }

  // 策略三：仅网络（Network Only）
  event.respondWith(fetch(event.request));
});
```

**常见缓存策略对比**:

| 策略 | 优先级 | 适用场景 | 离线可用 |
|------|--------|----------|----------|
| Cache First | 缓存 > 网络 | 静态资源（字体、图片） | 是 |
| Network First | 网络 > 缓存 | API 请求（数据易变） | 是（降级） |
| Stale While Revalidate | 缓存 + 后台更新 | 频繁更新但不紧急的资源 | 是 |
| Cache Only | 仅缓存 | 预缓存的静态资源 | 是 |
| Network Only | 仅网络 | 实时数据（支付、登录） | 否 |

**Stale While Revalidate 示例**:

```javascript
self.addEventListener('fetch', event => {
  event.respondWith(
    caches.open(CACHE_NAME).then(cache =>
      cache.match(event.request).then(cached => {
        const fetchPromise = fetch(event.request).then(response => {
          cache.put(event.request, response.clone());
          return response;
        });
        // 立即返回缓存，后台更新
        return cached || fetchPromise;
      })
    )
  );
});
```

## 二、浏览器渲染原理

### Q5: 浏览器的渲染流程是怎样的？从 HTML 到页面显示经历了哪些步骤？

**难度**: ⭐⭐

**答案**: 浏览器渲染流程包括构建 DOM 树、构建 CSSOM 树、生成渲染树（Render Tree）、布局（Layout）、绘制（Paint）、合成（Composite）六个主要阶段。

**解析**:

```
HTML 文件                    CSS 文件
    │                           │
    ▼                           ▼
┌──────────┐             ┌──────────┐
│ 字节流    │             │ 字节流    │
│ → 字符    │             │ → 字符    │
│ → Token   │             │ → Token   │
│ → Node    │             │ → Node    │
└────┬─────┘             └────┬─────┘
     ▼                        ▼
┌──────────┐           ┌──────────┐
│  DOM 树  │           │ CSSOM 树 │
└────┬─────┘           └────┬─────┘
     │                       │
     └───────────┬───────────┘
                 ▼
         ┌──────────────┐
         │  渲染树       │
         │ Render Tree   │
         │ (可见节点+样式)│
         └──────┬───────┘
                ▼
         ┌──────────────┐
         │   布局Layout  │
         │ 计算位置和大小 │
         └──────┬───────┘
                ▼
         ┌──────────────┐
         │  绘制Paint    │
         │ 绘制像素信息  │
         └──────┬───────┘
                ▼
         ┌──────────────┐
         │  合成Composite │
         │ 图层合成输出  │
         └──────────────┘
```

**各阶段详解**:

**1. DOM 树构建**
- HTML 解析器将字节流转换为字符，再通过词法分析器生成 Token
- Token 转换为 Node 节点，构建 DOM 树
- 遇到 `<link>` 会异步下载 CSS（不阻塞 DOM 构建）
- 遇到 `<script>` 会阻塞 DOM 构建（除非 `async`/`defer`）

**2. CSSOM 树构建**
- CSS 解析器将 CSS 文本转换为 CSSOM 树
- CSSOM 树包含所有 CSS 规则的层级关系
- CSSOM 构建是**渲染阻塞**的，必须等 CSSOM 构建完成才能进入下一阶段

**3. 渲染树（Render Tree）**
- DOM 树和 CSSOM 树合并生成渲染树
- 渲染树只包含**需要显示的节点**（`display: none` 的节点不包含）
- `visibility: hidden` 的节点**包含**在渲染树中（占位但不可见）
- `<head>`、`<meta>`、`<script>` 等不可见节点不包含在渲染树中

**4. 布局（Layout / Reflow）**
- 计算渲染树中每个节点的精确位置和大小
- 从根节点开始递归计算，称为"布局"或"重排"
- 布局结果是一个盒模型，包含位置（x, y）和尺寸（width, height）

**5. 绘制（Paint）**
- 将布局计算的结果转换为屏幕上的实际像素
- 包括绘制颜色、文字、图片、边框、阴影等
- 绘制顺序通常是：背景 → 边框 → 内容 → 轮廓

**6. 合成（Composite）**
- 将页面分成多个图层（Layer）
- 每个图层独立绘制，然后由合成线程合并
- GPU 加速的属性（transform、opacity）只触发合成，不触发布局和绘制
- 合成操作在 GPU 上完成，性能最优

**触发渲染更新的操作**:

```javascript
// 修改几何属性 → 触发 Layout + Paint + Composite
element.style.width = '200px';
element.style.padding = '10px';

// 修改外观属性 → 触发 Paint + Composite
element.style.backgroundColor = 'red';
element.style.color = 'blue';

// 仅修改合成属性 → 只触发 Composite（性能最优）
element.style.transform = 'translateX(100px)';
element.style.opacity = '0.5';
```

### Q6: 重绘（Repaint）和回流/重排（Reflow）有什么区别？如何优化？

**难度**: ⭐⭐

**答案**: 回流（Reflow）会引起几何属性的变化，需要重新计算布局；重绘（Repaint）只影响外观，不影响布局。回流一定会触发重绘，但重绘不一定触发回流。

**解析**:

| 操作 | 影响范围 | 性能消耗 | 示例 |
|------|----------|----------|------|
| 回流 (Reflow) | 几何属性变化 | 高 | width、height、margin、padding、display |
| 重绘 (Repaint) | 外观变化 | 中 | color、background、border-color、visibility |
| 合成 (Composite) | 仅图层变化 | 低 | transform、opacity、will-change |

**触发回流的常见操作**:

```javascript
// 几何属性修改
element.style.width = '100px';
element.style.height = '200px';
element.style.margin = '10px';
element.style.padding = '5px';
element.style.border = '1px solid red';
element.style.fontSize = '16px';
element.style.display = 'block';    // display 切换
element.style.position = 'absolute'; // position 切换

// 读取布局属性（强制同步布局！）
const width = element.offsetWidth;   // 读取 offsetWidth
const height = element.clientHeight; // 读取 clientHeight
const top = element.getBoundingClientRect().top; // 读取位置
const style = getComputedStyle(element); // 读取计算样式

// DOM 操作
document.body.appendChild(node);     // 添加节点
element.removeChild(child);          // 删除节点
parent.insertBefore(new, ref);       // 插入节点
```

**强制同步布局（Layout Thrashing）问题**:

```javascript
// ❌ 错误：读写交替，触发多次回流
for (let i = 0; i < items.length; i++) {
  items[i].style.width = container.offsetWidth + 'px'; // 每次循环都强制回流！
}

// ✅ 正确：批量读取，批量写入
const containerWidth = container.offsetWidth; // 一次性读取
for (let i = 0; i < items.length; i++) {
  items[i].style.width = containerWidth + 'px'; // 批量写入
}
```

**优化策略**:

```javascript
// 1. 使用 transform 代替 top/left
// ❌ 触发回流
element.style.left = '100px';
element.style.top = '200px';
// ✅ 只触发合成
element.style.transform = 'translate(100px, 200px)';

// 2. 使用 opacity 代替 visibility
// ❌ 触发重绘
element.style.visibility = 'hidden';
// ✅ 只触发合成
element.style.opacity = '0';

// 3. 使用 DocumentFragment 批量 DOM 操作
const fragment = document.createDocumentFragment();
for (let i = 0; i < 1000; i++) {
  const li = document.createElement('li');
  li.textContent = `Item ${i}`;
  fragment.appendChild(li);
}
ul.appendChild(fragment); // 一次性插入，只触发一次回流

// 4. 使用 CSS 类名批量修改样式
// ❌ 多次修改内联样式
element.style.width = '100px';
element.style.height = '200px';
element.style.margin = '10px';
// ✅ 一次性修改类名
element.className = 'optimized-layout';

// 5. 使用 requestAnimationFrame 合并更新
function update() {
  element.style.transform = 'translateX(' + x + 'px)';
  requestAnimationFrame(update);
}
requestAnimationFrame(update);

// 6. 离线 DOM 操作
const clone = element.cloneNode(true); // 克隆节点
// 在克隆节点上做大量修改...
clone.style.width = '200px';
clone.style.height = '300px';
element.parentNode.replaceChild(clone, element); // 一次性替换

// 7. 使用 will-change 提示浏览器
.animated-element {
  will-change: transform, opacity;
}
```

### Q7: 虚拟列表的原理是什么？如何实现一个虚拟列表？

**难度**: ⭐⭐⭐

**答案**: 虚拟列表（Virtual List）的核心思想是只渲染可视区域内的列表项，通过计算滚动位置动态替换渲染内容，从而解决大量 DOM 节点导致的性能问题。

**解析**:

```
┌─────────────────────────┐
│     可视区域 Viewport    │
│ ┌─────────────────────┐ │
│ │  Item 5  (可见)      │ │
│ │  Item 6  (可见)      │ │
│ │  Item 7  (可见)      │ │ ← 实际渲染的 DOM
│ │  Item 8  (可见)      │ │
│ │  Item 9  (可见)      │ │
│ └─────────────────────┘ │
│                         │
│  ↑ 滚动条               │
│  Item 0-4 (不可见，不渲染)│
│  Item 10-9999 (不可见)   │
└─────────────────────────┘
```

**完整实现代码**:

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <title>虚拟列表</title>
  <style>
    .virtual-list-container {
      width: 400px;
      height: 600px;
      overflow-y: auto;
      border: 1px solid #ddd;
      position: relative;
    }
    .virtual-list-phantom {
      position: absolute;
      left: 0;
      top: 0;
      right: 0;
      z-index: -1;
    }
    .virtual-list-content {
      position: absolute;
      left: 0;
      right: 0;
      top: 0;
    }
    .virtual-list-item {
      padding: 12px 16px;
      border-bottom: 1px solid #eee;
      box-sizing: border-box;
    }
  </style>
</head>
<body>
  <div id="app"></div>

  <script>
    class VirtualList {
      constructor({ container, items, itemHeight, renderItem }) {
        this.container = typeof container === 'string'
          ? document.querySelector(container) : container;
        this.items = items;
        this.itemHeight = itemHeight;
        this.renderItem = renderItem;
        this.visibleCount = Math.ceil(
          this.container.clientHeight / this.itemHeight
        );
        this.buffer = 5; // 上下缓冲区

        this.init();
      }

      init() {
        // 创建占位元素（撑开滚动条高度）
        this.phantom = document.createElement('div');
        this.phantom.className = 'virtual-list-phantom';
        this.phantom.style.height = `${this.items.length * this.itemHeight}px`;
        this.container.appendChild(this.phantom);

        // 创建内容容器
        this.content = document.createElement('div');
        this.content.className = 'virtual-list-content';
        this.container.appendChild(this.content);

        // 监听滚动事件（使用 requestAnimationFrame 节流）
        this.container.addEventListener('scroll', () => {
          requestAnimationFrame(() => this.render());
        });

        this.render();
      }

      render() {
        const scrollTop = this.container.scrollTop;
        const start = Math.max(0,
          Math.floor(scrollTop / this.itemHeight) - this.buffer
        );
        const end = Math.min(
          this.items.length,
          start + this.visibleCount + this.buffer * 2
        );

        // 偏移量，保持内容在正确位置
        const offsetY = start * this.itemHeight;
        this.content.style.transform = `translateY(${offsetY}px)`;

        // 只渲染可见区域的项
        const fragment = document.createDocumentFragment();
        for (let i = start; i < end; i++) {
          const item = this.renderItem(this.items[i], i);
          item.style.height = `${this.itemHeight}px`;
          fragment.appendChild(item);
        }

        this.content.innerHTML = '';
        this.content.appendChild(fragment);
      }
    }

    // 使用示例
    const items = Array.from({ length: 100000 }, (_, i) => ({
      id: i,
      text: `列表项 ${i} - 这是第 ${i} 条数据`,
    }));

    new VirtualList({
      container: '#app',
      items,
      itemHeight: 44,
      renderItem(item, index) {
        const div = document.createElement('div');
        div.className = 'virtual-list-item';
        div.textContent = item.text;
        return div;
      },
    });
  </script>
</body>
</html>
```

**虚拟列表的关键点**:
- **占位元素（Phantom）**: 设置与完整列表相同的高度，使滚动条正确显示
- **内容偏移（Offset）**: 使用 `transform: translateY()` 将可见内容定位到正确位置
- **缓冲区（Buffer）**: 上下各多渲染几项，避免快速滚动时出现白屏
- **动态高度**: 对于高度不固定的列表项，需要维护一个位置缓存表，记录每项的累积高度

**动态高度虚拟列表的思路**:

```javascript
// 维护位置缓存
const positions = [];
// 每次渲染后记录实际高度
function updatePositions() {
  const items = content.children;
  let totalHeight = 0;
  for (let i = 0; i < items.length; i++) {
    positions[startIndex + i] = {
      top: totalHeight,
      bottom: totalHeight + items[i].offsetHeight,
      height: items[i].offsetHeight
    };
    totalHeight += items[i].offsetHeight;
  }
}
// 通过二分查找定位 startIndex
function findStartIndex(scrollTop) {
  let left = 0, right = positions.length - 1;
  while (left <= right) {
    const mid = Math.floor((left + right) / 2);
    if (positions[mid].bottom > scrollTop) {
      right = mid - 1;
    } else {
      left = mid + 1;
    }
  }
  return left;
}
```

### Q8: MutationObserver 的作用是什么？使用时有哪些性能注意事项？

**难度**: ⭐⭐

**答案**: MutationObserver 是一个可以监听 DOM 变动的 API，能够异步观察 DOM 树的变化（属性修改、子节点增删、文本变化等），比传统的 Mutation Events 性能好得多。

**解析**:

```javascript
// 创建 MutationObserver 实例
const observer = new MutationObserver((mutationsList, observer) => {
  for (const mutation of mutationsList) {
    switch (mutation.type) {
      case 'childList':
        console.log('子节点变化:', mutation.addedNodes, mutation.removedNodes);
        break;
      case 'attributes':
        console.log(`属性 ${mutation.attributeName} 变化:`,
          mutation.target.getAttribute(mutation.attributeName));
        break;
      case 'characterData':
        console.log('文本内容变化:', mutation.target.textContent);
        break;
    }
  }
});

// 配置观察选项
observer.observe(targetNode, {
  childList: true,      // 观察子节点的增删
  attributes: true,     // 观察属性变化
  characterData: true,  // 观察文本内容变化
  subtree: true,        // 观察所有后代节点
  attributeFilter: ['class', 'style'], // 只观察特定属性
  attributeOldValue: true, // 记录属性旧值
  characterDataOldValue: true, // 记录文本旧值
});

// 停止观察
observer.disconnect();

// 清空记录队列（立即处理所有待处理的 MutationRecord）
const records = observer.takeRecords();
```

**性能注意事项**:

```javascript
// 1. 避免在回调中触发新的 DOM 变更（防止无限循环）
const observer = new MutationObserver((mutations) => {
  observer.disconnect(); // 先断开连接
  // 处理变更...
  processMutations(mutations);
  observer.observe(target, config); // 处理完后重新连接
});

// 2. 使用 takeRecords() 批量处理
function batchProcess() {
  const records = observer.takeRecords();
  if (records.length > 0) {
    processBatch(records);
  }
}

// 3. 精确配置观察选项，减少不必要的通知
observer.observe(target, {
  childList: true,       // 只在需要时开启
  // attributes: false,   // 不需要属性变化时关闭
  // subtree: false,      // 不需要观察子树时关闭
  attributeFilter: ['data-id'], // 只观察特定属性
});

// 4. 实际应用：监听元素进入 DOM
function observeElement(selector, callback) {
  const observer = new MutationObserver((mutations) => {
    for (const mutation of mutations) {
      for (const node of mutation.addedNodes) {
        if (node.matches?.(selector)) {
          callback(node);
        }
        // 检查子节点
        if (node.querySelectorAll) {
          const matches = node.querySelectorAll(selector);
          matches.forEach(callback);
        }
      }
    }
  });

  observer.observe(document.body, { childList: true, subtree: true });
  return observer;
}

// 使用：监听动态添加的 .card 元素
const observer = observeElement('.card', (card) => {
  console.log('新卡片已添加:', card);
  card.classList.add('animate-in');
});
```

**MutationObserver vs Mutation Events**:

| 特性 | MutationObserver | Mutation Events |
|------|-----------------|-----------------|
| 触发方式 | 异步（微任务） | 同步 |
| 性能 | 高（批量处理） | 低（每次变更都触发） |
| 批量处理 | 支持（一次回调多个变更） | 不支持 |
| 状态 | 推荐 | 已废弃 |

## 三、V8 引擎与垃圾回收

### Q10: V8 引擎的执行流程是怎样的？Ignition 和 TurboFan 分别起什么作用？

**难度**: ⭐⭐⭐

**答案**: V8 引擎采用解释器（Ignition）+ 编译器（TurboFan）的双引擎架构，先通过 Ignition 解释执行字节码，再对热点代码通过 TurboFan 编译为优化机器码。

**解析**:

```
JavaScript 源代码
       │
       ▼
┌──────────────┐
│  Parser 解析器 │ → 生成 AST（抽象语法树）
└──────┬───────┘
       ▼
┌──────────────┐
│ Ignition 解释器│ → 生成字节码（Bytecode）并解释执行
│              │ → 同时收集类型反馈信息（Type Feedback）
└──────┬───────┘
       │
       ▼ (热点代码检测)
┌──────────────┐
│ TurboFan 编译器│ → 基于类型反馈进行推测优化
│              │ → 编译为高效的机器码
└──────┬───────┘
       │
       ▼ (类型变化导致推测失败)
┌──────────────┐
│ 去优化 Deopt  │ → 回退到 Ignition 解释执行
│              │ → 重新收集类型反馈
└──────────────┘
```

**各阶段详解**:

**1. Parser（解析器）**
- 将 JavaScript 源代码解析为 AST
- 分为预解析（Pre-parser）和全量解析
- 预解析只解析顶层结构，跳过函数体内部（懒解析）
- 被立即执行的代码进行全量解析

```javascript
// 预解析：function 声明会被预解析，但函数体不会立即解析
function foo() {
  // 函数体在 foo() 被调用时才解析
  const a = 1;
  const b = 2;
  return a + b;
}

// 立即执行：IIFE 会被全量解析
(function() {
  console.log('立即执行');
})();
```

**2. Ignition（解释器）**
- 将 AST 转换为字节码（Bytecode）
- 解释执行字节码
- 收集类型反馈信息（Type Feedback），记录每次操作的操作数类型

```javascript
function add(a, b) {
  return a + b;
}

// 前几次调用收集类型反馈
add(1, 2);       // 反馈：a=Number, b=Number
add(3, 4);       // 反馈：a=Number, b=Number
// TurboFan 推测 a 和 b 都是 Number，编译优化版本

add('hello', 'world'); // 类型变化！去优化（Deoptimize）
// 回退到 Ignition 解释执行
```

**3. TurboFan（优化编译器）**
- 基于类型反馈进行推测性优化（Speculative Optimization）
- 编译为高效的机器码
- 如果推测失败（类型变化），执行去优化（Deoptimization）

**4. 隐藏类（Hidden Class）**

```javascript
// V8 使用隐藏类来优化对象属性访问
const p1 = { x: 1, y: 2 }; // HiddenClass: C0 → C1(x) → C2(x,y)
const p2 = { x: 3, y: 4 }; // 复用 HiddenClass C2，属性访问极快

// ❌ 避免动态添加属性（会创建新的隐藏类）
function Point(x, y) {
  this.x = x;  // C0 → C1
  this.y = y;  // C1 → C2
}
const p = new Point(1, 2);

// ✅ 在构造函数中声明所有属性
function Point(x, y, z) {
  this.x = x;
  this.y = y;
  this.z = z;  // 即使暂时不用，也预先声明
}

// ❌ 避免不同顺序添加属性（创建不同的隐藏类转换路径）
const a = { a: 1, b: 2 }; // C0 → C1(a) → C2(a,b)
const b = { b: 3, a: 4 }; // C0 → C3(b) → C4(b,a)  ← 不同的隐藏类！
```

**5. 内联缓存（Inline Cache, IC）**

```javascript
function getX(obj) {
  return obj.x;
}

// 第一次调用：未命中缓存，记录 obj 的隐藏类和属性偏移
getX({ x: 1 }); // IC miss → 记录 HiddenClass H1, offset 0

// 第二次调用：相同隐藏类，直接使用缓存的偏移量
getX({ x: 2 }); // IC hit → 直接从 offset 0 读取（极快）

// 不同隐藏类：缓存未命中
getX({ y: 0, x: 3 }); // IC miss → 新的隐藏类 H2
```

### Q11: V8 的垃圾回收机制是怎样的？新生代和老生代有什么区别？

**难度**: ⭐⭐⭐

**答案**: V8 采用分代垃圾回收策略，将堆内存分为新生代（Young Generation）和老生代（Old Generation），不同代使用不同的回收算法。

**解析**:

```
┌─────────────────────────────────────────┐
│                V8 堆内存                 │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │         新生代 (Young)           │    │
│  │  ┌───────────┐ ┌───────────┐   │    │
│  │  │ From 空间  │ │  To 空间   │   │    │
│  │  │ (使用中)   │ │ (空闲)     │   │    │
│  │  │           │ │           │   │    │
│  │  │  Scavenge │ │           │   │    │
│  │  │  算法     │ │           │   │    │
│  │  └───────────┘ └───────────┘   │    │
│  │  容量小（1-8MB），存活时间短的对象│    │
│  └──────────────┬──────────────────┘    │
│                 │ 经历过 2 次 Scavenge   │
│                 │ 仍然存活的对象晋升      │
│                 ▼                        │
│  ┌─────────────────────────────────┐    │
│  │         老生代 (Old)             │    │
│  │                                 │    │
│  │  Mark-Sweep（标记清除）          │    │
│  │  Mark-Compact（标记整理）        │    │
│  │                                 │    │
│  │  容量大，存活时间长的对象        │    │
│  └─────────────────────────────────┘    │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │     大对象区 (Large Object)      │    │
│  │  超过一定大小的对象直接分配到     │    │
│  │  老生代                         │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

**新生代回收 - Scavenge 算法**:

```
Scavenge 前:                Scavenge 后:
┌───────────┐ ┌──────────┐ ┌───────────┐ ┌──────────┐
│ From 空间  │ │ To 空间   │ │ From 空间  │ │ To 空间   │
│           │ │          │ │           │ │          │
│ [A] 存活  │ │ (空闲)   │ │ (清空)    │ │ [A] 存活  │
│ [B] 死亡  │ │          │ │           │ │ [C] 存活  │
│ [C] 存活  │ │          │ │           │ │          │
│ [D] 死亡  │ │          │ │           │ │          │
└───────────┘ └──────────┘ └───────────┘ └──────────┘
                              ↓ 角色互换 ↓
                         From ↔ To 交换
```

- 新生代分为 From 空间和 To 空间
- 检查 From 空间中的存活对象，复制到 To 空间
- 清空 From 空间，From 和 To 角色互换
- 对象经历过 **2 次** Scavenge 仍然存活，晋升到老生代
- To 空间使用超过 25% 时，对象直接晋升到老生代

**老生代回收 - Mark-Sweep / Mark-Compact**:

```
Mark-Sweep（标记清除）:
标记阶段: [A]✓ [B]✗ [C]✓ [D]✗ [E]✓
清除阶段: [A]   [ ]   [C]   [ ]   [E]   ← 产生内存碎片

Mark-Compact（标记整理）:
标记阶段: [A]✓ [B]✗ [C]✓ [D]✗ [E]✓
整理阶段: [A] [C] [E] [  ] [  ]   ← 紧凑排列，无碎片
```

- **Mark-Sweep**: 标记所有可达对象，清除不可达对象。缺点是会产生内存碎片
- **Mark-Compact**: 在 Mark-Sweep 基础上，将存活对象向一端移动，解决内存碎片问题
- 老生代通常使用 Mark-Sweep，当碎片过多时使用 Mark-Compact

**增量标记与并发回收**:

```
传统 GC:  ─────────────────────────────
          |<--- 全程暂停 (Stop-The-World) --->|
          JS执行    GC暂停    JS执行

增量 GC:  ──┬──┬──┬──┬──┬──┬──┬──┬──
          JS|GC|JS|GC|JS|GC|JS|GC|JS
          小步执行，减少暂停时间

并发 GC:  ─────────────────────────────
          JS执行 ─────────────────────
          GC线程 ─────────────────────
          (在后台线程执行，几乎不暂停JS)
```

- **增量标记（Incremental Marking）**: 将标记阶段拆分为多个小步骤，与 JS 执行交替进行
- **并发回收（Concurrent）**: GC 在后台线程执行，不阻塞主线程
- **并行回收（Parallel）**: 多个辅助线程同时执行 GC

**优化建议**:

```javascript
// 1. 避免产生大量短生命周期对象
// ❌ 在循环中创建大量临时对象
for (let i = 0; i < 1000000; i++) {
  const temp = { x: i, y: i * 2 }; // 大量对象进入新生代
}
// ✅ 复用对象

  temp.x = i;
  temp.y = i * 2;
  process(temp);
}

// 2. 及时解除引用

// data 不再使用后解除引用
// data = null; // 如果 data 是局部变量，函数结束后自动回收

// 3. 避免闭包中持有大对象引用
function setup() {
  const hugeArray = new Array(1000000).fill(0);
  return {
    getLength() { return hugeArray.length; }
    // hugeArray 被闭包引用，无法回收
  };
  // ✅ 只保留需要的数据
  // const length = hugeArray.length;
  // return { getLength() { return length; } };
```

### Q12: 什么是内存泄漏？常见的内存泄漏场景有哪些？如何检测？

**难度**: ⭐⭐

**答案**: 内存泄漏是指不再需要的对象仍然被引用，无法被垃圾回收器回收，导致可用内存持续减少。

**解析**:

**常见内存泄漏场景**:

```javascript
// 1. 意外的全局变量
function leak() {
  // ❌ 未使用 var/let/const 声明，成为全局变量
  leakedVar = 'this is a leak';
  // ❌ this 指向 window
  this.leakedToo = 'also a leak';
}

// 2. 被遗忘的定时器
const hugeData = getData();
setInterval(() => {
  const node = document.getElementById('node');
  if (node) {
    node.innerHTML = hugeData.join(','); // hugeData 永远不会被回收
  }
}, 1000);
// ✅ 清除定时器
// const id = setInterval(...);
// clearInterval(id);

// 3. 闭包持有引用
function outer() {
  const hugeArray = new Array(1000000);
  return function inner() {
    // 即使只用到 hugeArray 的 length
    // 整个 hugeArray 都不会被回收
    return hugeArray.length;
  };
}
// ✅ 只保留需要的值
function outer() {
  const hugeArray = new Array(1000000);
  const length = hugeArray.length;
  return function inner() {
    return length;
  };
}

// 4. DOM 引用
const elements = [];
document.addEventListener('click', (e) => {
  elements.push(e.target); // 持有 DOM 引用
  // 即使元素从 DOM 中移除，引用仍然存在
});
// ✅ 使用 WeakSet
const elements = new WeakSet();

// 5. 未移除的事件监听器
const handler = () => { /* ... */ };
element.addEventListener('click', handler);
// 元素移除后，如果 handler 引用了大对象，会泄漏
// ✅ 移除监听器
// element.removeEventListener('click', handler);

// 6. Map/Set 中的对象引用
const map = new Map();
const obj = { data: new Array(1000000) };
map.set(obj, 'value');
obj = null; // 但 map 中仍然引用该对象
// ✅ 使用 WeakMap
const weakMap = new WeakMap();
weakMap.set(obj, 'value');
obj = null; // 对象可以被回收
```

**内存泄漏检测方法**:

```javascript
// 1. Chrome DevTools Memory 面板
// - Take Heap Snapshot：拍摄堆快照，对比多次快照找出泄漏
// - Record Allocation Timeline：记录内存分配时间线
// - Record Allocation Profile：记录函数级别的内存分配

// 2. 使用 performance.memory（Chrome 专有）
console.log(`已用堆: ${performance.memory.usedJSHeapSize / 1024 / 1024} MB`);
console.log(`总堆: ${performance.memory.totalJSHeapSize / 1024 / 1024} MB`);
console.log(`堆上限: ${performance.memory.jsHeapSizeLimit / 1024 / 1024} MB`);

// 3. 使用 WeakRef 检测对象是否被回收
const obj = { data: 'test' };
const weakRef = new WeakRef(obj);
// ... 一段时间后
const ref = weakRef.deref();
if (ref) {
  console.log('对象仍然存活');
} else {
  console.log('对象已被回收');
}

// 4. FinalizationRegistry 监听对象回收
const registry = new FinalizationRegistry((heldValue) => {
  console.log(`${heldValue} 已被回收`);
});
registry.register(obj, 'myObject');
```

### Q13: Web Worker 和 SharedArrayBuffer 的作用是什么？有哪些使用限制？

**难度**: ⭐⭐⭐

**答案**: Web Worker 允许在后台线程中运行 JavaScript，避免阻塞主线程。SharedArrayBuffer 允许多个 Worker 共享同一块内存，实现高效的数据共享。

**解析**:

**Web Worker 使用示例**:

```javascript
// main.js - 主线程
const worker = new Worker('worker.js');

// 发送消息给 Worker
worker.postMessage({ type: 'compute', data: [1, 2, 3, 4, 5] });

// 接收 Worker 的消息
worker.onmessage = (event) => {
  console.log('Worker 返回结果:', event.data);
};

// 错误处理
worker.onerror = (error) => {
  console.error('Worker 错误:', error.message);
};

// 终止 Worker
// worker.terminate();
```

```javascript
// worker.js - Worker 线程
self.onmessage = (event) => {
  const { type, data } = event.data;

  if (type === 'compute') {
    // 执行耗时计算
    const result = data.reduce((sum, n) => sum + n * n, 0);
    self.postMessage({ type: 'result', value: result });
  }
};
```

**Worker 的限制**:
- 无法访问 DOM（`document`、`window`、`parent`）
- 无法使用 `alert`、`confirm`、`prompt`
- 无法读取本地文件（但可以使用 `fetch` 和 `XMLHttpRequest`）
- 与主线程通过 `postMessage` 通信（结构化克隆，有性能开销）

**SharedArrayBuffer + Atomics**:

```javascript
// main.js
const sab = new SharedArrayBuffer(1024); // 共享 1KB 内存
const int32Array = new Int32Array(sab);

const worker = new Worker('shared-worker.js');
worker.postMessage({ sab });

// 主线程写入数据
int32Array[0] = 42;

// 使用 Atomics 等待 Worker 处理完成
Atomics.wait(int32Array, 1, 0); // 等待 int32Array[1] 不再为 0
console.log('Worker 处理结果:', int32Array[1]);
```

```javascript
// shared-worker.js
self.onmessage = (event) => {
  const sab = event.data.sab;
  const int32Array = new Int32Array(sab);

  // 读取共享内存中的数据
  const value = int32Array[0]; // 42

  // 执行计算
  const result = value * 2; // 84

  // 写入结果
  int32Array[1] = result;

  // 通知主线程
  Atomics.notify(int32Array, 1, 1);
};
```

**SharedArrayBuffer 的安全限制**:
- 由于 Spectre 漏洞，SharedArrayBuffer 需要页面设置安全头才能使用
- 必须满足以下条件之一：
  - 页面是安全的上下文（HTTPS）
  - 设置 `Cross-Origin-Opener-Policy: same-origin`
  - 设置 `Cross-Origin-Embedder-Policy: require-corp`

```
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
```

**Worker 类型对比**:

| 类型 | 作用域 | 生命周期 | 使用场景 |
|------|--------|----------|----------|
| Dedicated Worker | 单个页面 | 随页面关闭 | 耗时计算 |
| Shared Worker | 多个同源页面 | 所有页面关闭后终止 | 跨标签页共享状态 |
| Service Worker | 所有同源页面 | 独立于页面 | 离线缓存、推送通知 |

## 四、浏览器存储与缓存

### Q14: Cookie、LocalStorage、SessionStorage、IndexedDB 有什么区别？

**难度**: ⭐

**答案**: 这四种存储方式在容量、生命周期、作用域、API 风格等方面各有不同，适用于不同的场景。

**解析**:

| 特性 | Cookie | LocalStorage | SessionStorage | IndexedDB |
|------|--------|-------------|----------------|-----------|
| 容量 | 4KB | 5-10MB | 5-10MB | 无限制（通常数百MB+） |
| 生命周期 | 可设置过期时间 | 永久存储 | 页面关闭后清除 | 永久存储 |
| 作用域 | 同源 + 路径 | 同源 | 同源 + 同标签页 | 同源 |
| 随请求发送 | 是（HTTP 请求头） | 否 | 否 | 否 |
| API | `document.cookie` | `localStorage` | `sessionStorage` | 异步 API |
| 数据类型 | 字符串 | 字符串 | 字符串 | 结构化克隆（任意类型） |
| 安全性 | 可设置 HttpOnly | 无特殊保护 | 无特殊保护 | 无特殊保护 |

**Cookie 详解**:

```javascript
// 设置 Cookie
document.cookie = 'name=value; max-age=3600; path=/; domain=.example.com; secure; samesite=strict';

// 读取 Cookie（只能读取非 HttpOnly 的）
console.log(document.cookie); // 'name=value; other=cookie'

// 删除 Cookie（设置 max-age=0 或过期时间为过去）
document.cookie = 'name=; max-age=0; path=/';

// Cookie 属性说明：
// - max-age: 过期秒数（优先于 expires）
// - expires: 过期时间（GMT 格式）
// - path: 作用路径
// - domain: 作用域名（.example.com 可跨子域）
// - secure: 仅 HTTPS 发送
// - httpOnly: 仅服务器可读，JS 无法访问（防 XSS）
// - sameSite: Strict/Lax/None（防 CSRF）
```

**LocalStorage / SessionStorage**:

```javascript
// LocalStorage - 持久存储
localStorage.setItem('key', JSON.stringify({ name: '张三', age: 25 }));
const data = JSON.parse(localStorage.getItem('key'));
localStorage.removeItem('key');
localStorage.clear(); // 清空所有

// SessionStorage - 会话存储（标签页关闭即清除）
sessionStorage.setItem('tabData', '仅当前标签页可用');

// 监听 storage 事件（仅 LocalStorage 跨标签页触发）
window.addEventListener('storage', (e) => {
  console.log('Key:', e.key);
  console.log('Old Value:', e.oldValue);
  console.log('New Value:', e.newValue);
  console.log('Source:', e.url);
});
```

**IndexedDB 详解**:

```javascript
// 打开/创建数据库
const request = indexedDB.open('MyDatabase', 1);

request.onupgradeneeded = (event) => {
  const db = event.target.result;
  // 创建对象存储（类似表）
  if (!db.objectStoreNames.contains('users')) {
    const store = db.createObjectStore('users', { keyPath: 'id' });
    store.createIndex('name', 'name', { unique: false });
    store.createIndex('email', 'email', { unique: true });
  }
};

request.onsuccess = (event) => {
  const db = event.target.result;

  // 添加数据
  const tx = db.transaction('users', 'readwrite');
  const store = tx.objectStore('users');
  store.add({ id: 1, name: '张三', email: 'zhangsan@example.com' });
  store.add({ id: 2, name: '李四', email: 'lisi@example.com' });

  // 查询数据
  const getRequest = store.get(1);
  getRequest.onsuccess = () => {
    console.log(getRequest.result); // { id: 1, name: '张三', ... }
  };

  // 通过索引查询
  const index = store.index('name');
  const nameRequest = index.get('张三');
  nameRequest.onsuccess = () => {
    console.log(nameRequest.result);
  };

  // 遍历所有数据
  const cursorRequest = store.openCursor();
  cursorRequest.onsuccess = (e) => {
    const cursor = e.target.result;
    if (cursor) {
      console.log(cursor.value);
      cursor.continue();
    }
  };
};
```

**使用场景建议**:
- **Cookie**: 用户认证 Token、用户偏好设置（需要随请求发送的场景）
- **LocalStorage**: 主题设置、语言偏好、不敏感的用户配置
- **SessionStorage**: 表单临时数据、单页应用的临时状态
- **IndexedDB**: 离线应用数据、大量结构化数据、文件存储

### Q15: 浏览器缓存机制是怎样的？强缓存和协商缓存有什么区别？

**难度**: ⭐⭐

**答案**: 浏览器缓存分为强缓存和协商缓存。强缓存直接使用本地缓存不发送请求，协商缓存会向服务器验证资源是否更新。

**解析**:

```
浏览器请求资源
    │
    ▼
┌──────────────────┐
│  检查强缓存        │
│ Cache-Control     │
│ Expires           │
└────┬─────────────┘
     │
     ├── 命中 → 直接使用缓存（200 from cache / disk cache）
     │         状态码：200（from cache）
     │
     └── 未命中
          │
          ▼
     ┌──────────────────┐
     │  检查协商缓存      │
     │ ETag / If-None-Match│
     │ Last-Modified /   │
     │ If-Modified-Since │
     └────┬─────────────┘
          │
          ├── 未修改 → 使用缓存（304 Not Modified）
          │
          └── 已修改 → 返回新资源（200 OK）
```

**强缓存**:

```http
# Cache-Control（HTTP/1.1，优先级更高）
Cache-Control: max-age=31536000       # 缓存 1 年
Cache-Control: no-cache               # 不使用强缓存，走协商缓存
Cache-Control: no-store               # 完全不缓存
Cache-Control: public                 # 可被任何中间缓存代理缓存
Cache-Control: private                # 仅浏览器可缓存
Cache-Control: s-maxage=3600          # 共享缓存（CDN）的缓存时间
Cache-Control: must-revalidate        # 过期后必须验证
Cache-Control: immutable              # 资源永不变化，不需要重新验证

# Expires（HTTP/1.0，已过时）
Expires: Wed, 21 Oct 2025 07:28:00 GMT  # 绝对过期时间
```

**协商缓存**:

```http
# 方式一：ETag / If-None-Match（优先级更高）
# 响应头
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
# 请求头
If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"

# 方式二：Last-Modified / If-Modified-Since
# 响应头
Last-Modified: Mon, 01 Jan 2024 00:00:00 GMT
# 请求头
If-Modified-Since: Mon, 01 Jan 2024 00:00:00 GMT
```

**实际应用中的缓存策略**:

```nginx
# Nginx 配置示例
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2)$ {
    # 带 hash 的静态资源：强缓存 1 年
    add_header Cache-Control "public, max-age=31536000, immutable";
}

location /api/ {
    # API 接口：不缓存
    add_header Cache-Control "no-store";
}

location / {
    # HTML 文件：协商缓存
    add_header Cache-Control "no-cache";
    etag on;
}
```

```html
<!-- HTML 中引用带 hash 的静态资源 -->
<link rel="stylesheet" href="/css/app.a1b2c3d4.css">
<script src="/js/app.e5f6g7h8.js"></script>
<img src="/images/logo.i9j0k1l2.png">
```

**缓存决策流程**:

| 资源类型 | 缓存策略 | 原因 |
|----------|----------|------|
| HTML | 协商缓存（no-cache） | 确保每次获取最新入口文件 |
| JS/CSS（带 hash） | 强缓存 1 年（immutable） | 文件名包含 hash，内容变化则 URL 变化 |
| 图片/字体 | 强缓存 + 协商缓存 | 资源较大，减少请求 |
| API 数据 | 不缓存（no-store） | 数据实时性要求高 |

**DevTools 中的缓存状态**:
- `from memory cache`: 从内存中读取（最快，关闭标签页后失效）
- `from disk cache`: 从磁盘中读取（较快，持久化存储）
- `from ServiceWorker`: 从 Service Worker 缓存中读取
- 大文件通常存 disk cache，小文件存 memory cache

### Q16: IndexedDB 的事务机制是怎样的？如何封装一个 IndexedDB 工具库？

**难度**: ⭐⭐⭐

**答案**: IndexedDB 的所有读写操作必须在事务（Transaction）中执行，事务具有隔离性和原子性。

**解析**:

```javascript
/**
 * IndexedDB 工具库封装
 */
class IDBHelper {
  constructor(dbName, version = 1) {
    this.dbName = dbName;
    this.version = version;
    this.db = null;
  }

  // 打开数据库
  open(stores) {
    return new Promise((resolve, reject) => {
      const request = indexedDB.open(this.dbName, this.version);

      request.onupgradeneeded = (event) => {
        const db = event.target.result;
        for (const store of stores) {
          if (!db.objectStoreNames.contains(store.name)) {
            const objectStore = db.createObjectStore(store.name, {
              keyPath: store.keyPath || 'id',
              autoIncrement: store.autoIncrement || false,
            });
            // 创建索引
            if (store.indexes) {
              for (const index of store.indexes) {
                objectStore.createIndex(
                  index.name,
                  index.keyPath,
                  index.options || {}
                );
              }
            }
          }
        }
      };

      request.onsuccess = (event) => {
        this.db = event.target.result;
        resolve(this.db);
      };

      request.onerror = (event) => {
        reject(event.target.error);
      };
    });
  }

  // 事务模式
  _transaction(storeName, mode = 'readonly') {
    const tx = this.db.transaction(storeName, mode);
    const store = tx.objectStore(storeName);
    return { tx, store };
  }

  // 添加数据
  add(storeName, data) {
    return new Promise((resolve, reject) => {
      const { tx, store } = this._transaction(storeName, 'readwrite');
      const request = store.add(data);
      request.onsuccess = () => resolve(request.result);
      request.onerror = () => reject(request.error);
    });
  }

  // 更新数据
  put(storeName, data) {
    return new Promise((resolve, reject) => {
      const { tx, store } = this._transaction(storeName, 'readwrite');
      const request = store.put(data);
      request.onsuccess = () => resolve(request.result);
      request.onerror = () => reject(request.error);
    });
  }

  // 获取单条数据
  get(storeName, key) {
    return new Promise((resolve, reject) => {
      const { store } = this._transaction(storeName, 'readonly');
      const request = store.get(key);
      request.onsuccess = () => resolve(request.result);
      request.onerror = () => reject(request.error);
    });
  }

  // 获取所有数据
  getAll(storeName) {
    return new Promise((resolve, reject) => {
      const { store } = this._transaction(storeName, 'readonly');
      const request = store.getAll();
      request.onsuccess = () => resolve(request.result);
      request.onerror = () => reject(request.error);
    });
  }

  // 删除数据
  delete(storeName, key) {
    return new Promise((resolve, reject) => {
      const { tx, store } = this._transaction(storeName, 'readwrite');
      const request = store.delete(key);
      request.onsuccess = () => resolve();
      request.onerror = () => reject(request.error);
    });
  }

  // 通过索引查询
  getByIndex(storeName, indexName, value) {
    return new Promise((resolve, reject) => {
      const { store } = this._transaction(storeName, 'readonly');
      const index = store.index(indexName);
      const request = index.get(value);
      request.onsuccess = () => resolve(request.result);
      request.onerror = () => reject(request.error);
    });
  }

  // 清空对象存储
  clear(storeName) {
    return new Promise((resolve, reject) => {
      const { tx, store } = this._transaction(storeName, 'readwrite');
      const request = store.clear();
      request.onsuccess = () => resolve();
      request.onerror = () => reject(request.error);
    });
  }
}

// 使用示例
const db = new IDBHelper('MyApp', 1);
await db.open([
  {
    name: 'users',
    keyPath: 'id',
    indexes: [
      { name: 'name', keyPath: 'name' },
      { name: 'email', keyPath: 'email', options: { unique: true } },
    ],
  },
  {
    name: 'posts',
    keyPath: 'id',
    autoIncrement: true,
    indexes: [
      { name: 'authorId', keyPath: 'authorId' },
    ],
  },
]);

// 添加用户
await db.add('users', { id: 1, name: '张三', email: 'zhangsan@test.com' });

// 查询用户
const user = await db.get('users', 1);

// 通过索引查询
const userByName = await db.getByIndex('users', 'name', '张三');

// 获取所有用户
const allUsers = await db.getAll('users');
```

**事务模式**:
- `readonly`: 只读事务，可与其他 readonly 事务并发
- `readwrite`: 读写事务，独占访问
- `versionchange`: 版本变更事务（仅在 onupgradeneeded 中使用）

**事务注意事项**:
- 事务在创建时确定作用域和模式
- 事务在微任务结束后自动提交
- 如果事务中没有待处理的请求，事务会提前提交
- 事务中的操作要么全部成功，要么全部回滚

### Q17: Cookie 的 SameSite 属性有什么作用？各值之间有什么区别？

**难度**: ⭐⭐

**答案**: SameSite 属性用于控制 Cookie 在跨站请求时是否发送，是防止 CSRF 攻击的重要机制。

**解析**:

| SameSite 值 | 跨站链接跳转 | 跨站 POST 表单 | 跨站 AJAX/Fetch | 顶级导航 GET |
|-------------|-------------|---------------|----------------|-------------|
| `Strict` | 不发送 | 不发送 | 不发送 | 不发送 |
| `Lax` | 不发送 | 不发送 | 不发送 | **发送** |
| `None` | 发送 | 发送 | 发送 | 发送 |

```http
# Strict - 最严格，完全禁止跨站发送
Set-Cookie: session=abc123; SameSite=Strict

# Lax - 默认值，允许顶级导航 GET 请求携带 Cookie
Set-Cookie: session=abc123; SameSite=Lax

# None - 允许跨站发送（必须配合 Secure）
Set-Cookie: session=abc123; SameSite=None; Secure
```

**实际场景分析**:

```
场景1: 从 https://blog.example.com 点击链接到 https://shop.example.com
- SameSite=Strict: 不发送 shop.example.com 的 Cookie → 用户需要重新登录
- SameSite=Lax: 发送 Cookie（顶级导航 GET）→ 用户保持登录状态
- SameSite=None: 发送 Cookie

场景2: 从 https://evil.com 发起 POST 表单到 https://bank.com/transfer
- SameSite=Strict: 不发送 Cookie → CSRF 攻击失败
- SameSite=Lax: 不发送 Cookie → CSRF 攻击失败
- SameSite=None: 发送 Cookie → 可能被 CSRF 攻击

场景3: 从 https://evil.com 使用 fetch 请求 https://api.example.com/data
- SameSite=Strict: 不发送 Cookie
- SameSite=Lax: 不发送 Cookie
- SameSite=None: 发送 Cookie
```

**Chrome 80+ 的变化**:
- 默认 SameSite 值从 `None` 改为 `Lax`
- 如果不显式设置 SameSite，Cookie 默认为 `Lax`
- 跨域请求携带 Cookie 必须显式设置 `SameSite=None; Secure`

**前后端配合示例**:

```javascript
// 前端：跨域请求携带 Cookie
fetch('https://api.example.com/data', {
  credentials: 'include', // 跨域请求携带 Cookie
  method: 'GET',
  mode: 'cors',
});

// 后端（Node.js Express）：
// 设置支持跨域携带 Cookie
app.use(cors({
  origin: 'https://frontend.example.com',
  credentials: true, // 允许携带 Cookie
}));

// 设置 Cookie
res.cookie('token', 'jwt-token', {
  httpOnly: true,
  secure: true,
  sameSite: 'none', // 跨域场景需要 none
  maxAge: 3600000,
});
```

### Q18: 如何设计一个前端缓存层，结合 LocalStorage 和 IndexedDB？

**难度**: ⭐⭐⭐

**答案**: 可以设计一个多层缓存策略，小数据用 LocalStorage（同步读取快），大数据用 IndexedDB（异步但容量大），配合内存缓存实现最快读取。

**解析**:

```javascript
/**
 * 多层缓存管理器
 * L1: 内存缓存（最快，容量最小）
 * L2: LocalStorage（同步，容量 5MB）
 * L3: IndexedDB（异步，容量大）
 */
class MultiLayerCache {
  constructor() {
    this.memoryCache = new Map(); // L1
    this.memoryCacheMaxSize = 100;
    this.dbName = 'CacheDB';
    this.storeName = 'cache';
    this.db = null;
  }

  // 初始化 IndexedDB
  async init() {
    this.db = await new Promise((resolve, reject) => {
      const request = indexedDB.open(this.dbName, 1);
      request.onupgradeneeded = (e) => {
        e.target.result.createObjectStore(this.storeName, { keyPath: 'key' });
      };
      request.onsuccess = (e) => resolve(e.target.result);
      request.onerror = (e) => reject(e.target.error);
    });
  }

  // 设置缓存
  async set(key, value, options = {}) {
    const { ttl = 3600000, useLocalStorage = false } = options; // 默认 1 小时
    const entry = {
      value,
      expireAt: Date.now() + ttl,
    };

    // L1: 内存缓存
    this._setMemory(key, entry);

    // L2: LocalStorage（仅小数据）
    if (useLocalStorage) {
      try {
        localStorage.setItem(`cache_${key}`, JSON.stringify(entry));
      } catch (e) {
        console.warn('LocalStorage 写入失败:', e);
      }
    }

    // L3: IndexedDB
    if (this.db) {
      const tx = this.db.transaction(this.storeName, 'readwrite');
      tx.objectStore(this.storeName).put({ key, ...entry });
    }
  }

  // 获取缓存
  async get(key) {
    // L1: 内存缓存
    const memoryEntry = this.memoryCache.get(key);
    if (memoryEntry && !this._isExpired(memoryEntry)) {
      return memoryEntry.value;
    }

    // L2: LocalStorage
    const localEntry = this._getLocalStorage(key);
    if (localEntry && !this._isExpired(localEntry)) {
      this._setMemory(key, localEntry); // 回填内存缓存
      return localEntry.value;
    }

    // L3: IndexedDB
    if (this.db) {
      const dbEntry = await this._getIDB(key);
      if (dbEntry && !this._isExpired(dbEntry)) {
        this._setMemory(key, dbEntry); // 回填内存缓存
        return dbEntry.value;
      }
    }

    return null; // 缓存未命中
  }

  // 删除缓存
  async remove(key) {
    this.memoryCache.delete(key);
    localStorage.removeItem(`cache_${key}`);
    if (this.db) {
      const tx = this.db.transaction(this.storeName, 'readwrite');
      tx.objectStore(this.storeName).delete(key);
    }
  }

  // 清空所有缓存
  async clear() {
    this.memoryCache.clear();
    const keys = Object.keys(localStorage).filter(k => k.startsWith('cache_'));
    keys.forEach(k => localStorage.removeItem(k));
    if (this.db) {
      const tx = this.db.transaction(this.storeName, 'readwrite');
      tx.objectStore(this.storeName).clear();
    }
  }

  // 内部方法
  _setMemory(key, entry) {
    if (this.memoryCache.size >= this.memoryCacheMaxSize) {
      const firstKey = this.memoryCache.keys().next().value;
      this.memoryCache.delete(firstKey);
    }
    this.memoryCache.set(key, entry);
  }

  _getLocalStorage(key) {
    try {
      const raw = localStorage.getItem(`cache_${key}`);
      return raw ? JSON.parse(raw) : null;
    } catch {
      return null;
    }
  }

  _getIDB(key) {
    return new Promise((resolve) => {
      const tx = this.db.transaction(this.storeName, 'readonly');
      const request = tx.objectStore(this.storeName).get(key);
      request.onsuccess = () => resolve(request.result);
      request.onerror = () => resolve(null);
    });
  }

  _isExpired(entry) {
    if (!entry.expireAt) return false;
    return Date.now() > entry.expireAt;
  }
}

// 使用示例
const cache = new MultiLayerCache();
await cache.init();

// 设置缓存
await cache.set('user_info', { name: '张三', age: 25 }, { ttl: 1800000 });
await cache.set('config', { theme: 'dark' }, { ttl: 86400000, useLocalStorage: true });

// 获取缓存
const userInfo = await cache.get('user_info');
const config = await cache.get('config');

// 删除缓存
await cache.remove('user_info');
```

### Q19: 什么是同源策略？有哪些跨域解决方案？

**难度**: ⭐⭐

**答案**: 同源策略（Same-Origin Policy）是浏览器的基本安全机制，限制一个源的文档或脚本如何与另一个源的资源进行交互。同源是指协议、域名、端口完全相同。

**解析**:

```
URL: https://www.example.com:443/page

协议: https
域名: www.example.com
端口: 443

以下 URL 与上述 URL 不同源：
http://www.example.com    ← 协议不同
https://api.example.com   ← 域名不同
https://www.example.com:8080 ← 端口不同
```

**跨域解决方案**:

**1. CORS（跨域资源共享）- 最常用**

```javascript
// 前端
fetch('https://api.example.com/data', {
  method: 'GET',
  headers: { 'Content-Type': 'application/json' },
  mode: 'cors',
  credentials: 'include', // 携带 Cookie
});

// 后端（Node.js Express）
const cors = require('cors');
app.use(cors({
  origin: ['https://www.example.com'],
  credentials: true,
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
}));

// 或手动设置响应头
app.use((req, res, next) => {
  res.header('Access-Control-Allow-Origin', 'https://www.example.com');
  res.header('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE');
  res.header('Access-Control-Allow-Headers', 'Content-Type, Authorization');
  res.header('Access-Control-Allow-Credentials', 'true');
  res.header('Access-Control-Max-Age', '86400'); // 预检请求缓存时间

  // 处理预检请求
  if (req.method === 'OPTIONS') {
    return res.sendStatus(204);
  }
  next();
});
```

**CORS 请求分类**:

| 类型 | 触发条件 | 流程 |
|------|----------|------|
| 简单请求 | GET/HEAD/POST，Content-Type 为 text/plain 等 | 直接发送请求 |
| 预检请求 | PUT/DELETE，自定义 Header，Content-Type 为 application/json | 先发 OPTIONS 预检，再发实际请求 |

### Q20: 跨标签页通信有哪些方式？

**难度**: ⭐⭐

**答案**: 浏览器提供了多种跨标签页通信方式，包括 BroadcastChannel、SharedWorker、localStorage 事件、postMessage 等。

**解析**:

**1. BroadcastChannel - 最推荐**

```javascript
// 标签页 A
const channel = new BroadcastChannel('my-channel');
channel.postMessage({ type: 'update', data: { count: 1 } });
channel.onmessage = (event) => {
  console.log('收到消息:', event.data);
};
channel.close(); // 关闭频道

// 标签页 B（同源）
const channel = new BroadcastChannel('my-channel');
channel.onmessage = (event) => {
  console.log('收到消息:', event.data);
  // { type: 'update', data: { count: 1 } }
};
```

**2. localStorage 事件**

```javascript
// 标签页 A：写入 localStorage
localStorage.setItem('shared-data', JSON.stringify({ count: 1 }));

// 标签页 B：监听 storage 事件
window.addEventListener('storage', (event) => {
  console.log('Key:', event.key);           // 'shared-data'
  console.log('New Value:', event.newValue); // '{"count":1}'
  console.log('Old Value:', event.oldValue); // null
  console.log('Source:', event.url);         // 来源页面 URL
  console.log('Storage Area:', event.storageArea); // localStorage
});

// 注意：storage 事件只在其他标签页触发，当前页面不触发
```

**3. SharedWorker**

```javascript
// shared-worker.js
const connections = new Set();

self.onconnect = (event) => {
  const port = event.ports[0];
  connections.add(port);

  port.onmessage = (e) => {
    // 广播给所有连接的标签页
    connections.forEach(conn => {
      if (conn !== port) {
        conn.postMessage(e.data);
      }
    });
  };

  port.start();
};

// 标签页 A 和 B 都可以这样连接
const worker = new SharedWorker('shared-worker.js');
worker.port.start();
worker.port.postMessage('Hello from Tab A');
worker.port.onmessage = (e) => console.log('收到:', e.data);
```

**4. postMessage + window.open**

```javascript
// 标签页 A：打开新标签页
const newWindow = window.open('https://example.com/page-b');
newWindow.postMessage({ type: 'greeting', data: 'hello' }, '*');

// 标签页 B
window.addEventListener('message', (event) => {
  if (event.data.type === 'greeting') {
    console.log(event.data.data);
  }
});
```

**5. Service Worker**

```javascript
// 主线程发送消息给 Service Worker
navigator.serviceWorker.controller.postMessage({
  type: 'broadcast',
  data: 'hello all tabs'
});

// Service Worker 接收并转发
self.addEventListener('message', (event) => {
  if (event.data.type === 'broadcast') {
    self.clients.matchAll().then(clients => {
      clients.forEach(client => {
        client.postMessage(event.data.data);
      });
    });
  }
});

// 各标签页接收
navigator.serviceWorker.addEventListener('message', (event) => {
  console.log('SW 消息:', event.data);
});
```

**方式对比**:

| 方式 | 复杂度 | 兼容性 | 适用场景 |
|------|--------|--------|----------|
| BroadcastChannel | 低 | 现代浏览器 | 通用跨标签页通信 |
| localStorage 事件 | 低 | 所有浏览器 | 简单数据同步 |
| SharedWorker | 中 | 现代浏览器 | 需要共享状态的场景 |
| postMessage | 中 | 所有浏览器 | 已知窗口引用 |
| Service Worker | 高 | 现代浏览器 | 离线应用、推送通知 |

### Q21: postMessage 的使用方式和安全注意事项有哪些？

**难度**: ⭐⭐

**答案**: `window.postMessage` 是一种安全的跨源通信机制，允许不同窗口之间传递数据。使用时必须验证消息来源。

**解析**:

```javascript
// ===== 父页面 =====
const iframe = document.getElementById('cross-origin-iframe');

// 发送消息
iframe.contentWindow.postMessage(
  { type: 'REQUEST_DATA', payload: { id: 123 } },
  'https://child.example.com' // 目标源（不要用 '*'）
);

// 接收消息
window.addEventListener('message', (event) => {
  // 安全校验：必须验证来源
  if (event.origin !== 'https://child.example.com') {
    return; // 拒绝来自未知源的消息
  }

  // 安全校验：验证数据格式
  if (!event.data || event.data.type !== 'RESPONSE_DATA') {
    return;
  }

  console.log('收到数据:', event.data.payload);
});

// ===== 子页面（iframe） =====
// 接收消息
window.addEventListener('message', (event) => {
  if (event.origin !== 'https://parent.example.com') return;

  if (event.data.type === 'REQUEST_DATA') {
    const data = fetchData(event.data.payload.id);
    // 回复消息
    event.source.postMessage(
      { type: 'RESPONSE_DATA', payload: data },
      event.origin // 回复给来源
    );
  }
});
```

**安全注意事项**:

```javascript
// 1. 始终验证 event.origin
window.addEventListener('message', (event) => {
  // ❌ 危险：不验证来源
  // console.log(event.data);

  // ✅ 安全：白名单验证
  const ALLOWED_ORIGINS = [
    'https://trusted.example.com',
    'https://partner.example.com',
  ];
  if (!ALLOWED_ORIGINS.includes(event.origin)) {
    console.warn('拒绝来自未知源的消息:', event.origin);
    return;
  }
});

// 2. 验证消息数据结构
function isValidMessage(data) {
  return data
    && typeof data === 'object'
    && typeof data.type === 'string'
    && ['REQUEST', 'RESPONSE', 'NOTIFY'].includes(data.type);
}

// 3. 不要使用 '*' 作为 targetOrigin
// ❌ 危险
window.postMessage(sensitiveData, '*');

// ✅ 安全：指定确切的目标源
window.postMessage(sensitiveData, 'https://specific.example.com');

// 4. 传输数据时避免包含敏感信息
// ❌ 不要传输 token、密码等
// ✅ 如果必须传输，使用一次性 token
```

### Q22: CORS 预检请求（Preflight）的原理是什么？如何减少预检请求？

**难度**: ⭐⭐⭐

**答案**: 预检请求是浏览器在发送非简单请求前，先发送一个 OPTIONS 请求来确认服务器是否允许该跨域请求。

**解析**:

**触发预检请求的条件（满足任一即触发）**:

```javascript
// 1. 使用了非简单方法
fetch(url, { method: 'PUT' });       // PUT
fetch(url, { method: 'DELETE' });    // DELETE
fetch(url, { method: 'PATCH' });     // PATCH

// 2. 使用了自定义请求头
fetch(url, {
  headers: {
    'Content-Type': 'application/json',  // 非 text/plain
    'Authorization': 'Bearer token',      // 自定义头
    'X-Custom-Header': 'value',           // 自定义头
  }
});

// 3. 请求体为非简单类型
fetch(url, {
  method: 'POST',
  body: new FormData(),    // FormData 不是简单类型
});

// 简单请求的条件：
// - 方法：GET、HEAD、POST
// - Content-Type：text/plain、multipart/form-data、application/x-www-form-urlencoded
// - 只使用浏览器自动设置的请求头（Accept、Accept-Language 等）
```

**预检请求流程**:

```
浏览器                                    服务器
  │                                         │
  │  ① OPTIONS /data                        │
  │  Origin: https://example.com            │
  │  Access-Control-Request-Method: PUT     │
  │  Access-Control-Request-Headers:         │
  │    Content-Type, Authorization           │
  │ ──────────────────────────────────────→ │
  │                                         │
  │  ② 200 OK                               │
  │  Access-Control-Allow-Origin: *         │
  │  Access-Control-Allow-Methods: PUT      │
  │  Access-Control-Allow-Headers:           │
  │    Content-Type, Authorization           │
  │  Access-Control-Max-Age: 86400          │
  │ ←────────────────────────────────────── │
  │                                         │
  │  ③ PUT /data                            │
  │  Origin: https://example.com            │
  │  Content-Type: application/json         │
  │  Authorization: Bearer token            │
  │ ──────────────────────────────────────→ │
  │                                         │
  │  ④ 200 OK                               │
  │  Access-Control-Allow-Origin: *         │
  │  { "data": "response" }                 │
  │ ←────────────────────────────────────── │
```

**减少预检请求的策略**:

```javascript
// 1. 设置 Access-Control-Max-Age 缓存预检结果
```

## 六、事件机制与任务调度

### Q23: 浏览器事件模型是怎样的？捕获、目标和冒泡三个阶段分别是什么？

**难度**: ⭐

**答案**: DOM 事件传播分为三个阶段：捕获阶段（从 window 到目标元素）、目标阶段（在目标元素上）、冒泡阶段（从目标元素到 window）。

**解析**:

```
点击按钮事件传播路径：

Window
  ↓ 捕获阶段（从外到内）
Document
  ↓
html
  ↓
body
  ↓
div
  ↓
button ← 目标阶段
  ↑ 冒泡阶段（从内到外）
div
  ↑
body
  ↑
html
  ↑
Document
  ↑
Window
```

**代码示例**:

```html
<div id="outer" style="padding: 40px; background: #eee;">
  <div id="middle" style="padding: 40px; background: #ddd;">
    <button id="inner">点击我</button>
  </div>
</div>

<script>
  const outer = document.getElementById('outer');
  const middle = document.getElementById('middle');
  const inner = document.getElementById('inner');

  // 第三个参数：true = 捕获阶段，false = 冒泡阶段（默认）
  outer.addEventListener('click', () => console.log('1. outer 捕获'), true);
  middle.addEventListener('click', () => console.log('2. middle 捕获'), true);
  inner.addEventListener('click', () => console.log('3. inner 捕获'), true);
  inner.addEventListener('click', () => console.log('4. inner 冒泡'), false);
  middle.addEventListener('click', () => console.log('5. middle 冒泡'), false);
  outer.addEventListener('click', () => console.log('6. outer 冒泡'), false);

  // 点击 button 输出顺序：
  // 1. outer 捕获
  // 2. middle 捕获
  // 3. inner 捕获
  // 4. inner 冒泡
  // 5. middle 冒泡
  // 6. outer 冒泡
</script>
```

**事件对象的重要属性和方法**:

```javascript
element.addEventListener('click', (event) => {
  // 属性
  event.type;           // 事件类型：'click'
  event.target;         // 触发事件的元素（实际被点击的元素）
  event.currentTarget;  // 绑定事件监听器的元素
  event.bubbles;        // 是否冒泡
  event.eventPhase;     // 事件阶段：0=无, 1=捕获, 2=目标, 3=冒泡

  // 方法
  event.stopPropagation();    // 阻止事件继续传播（捕获/冒泡）
  event.stopImmediatePropagation(); // 阻止传播 + 阻止同元素其他监听器
  event.preventDefault();     // 阻止默认行为（如链接跳转、表单提交）
});
```

**stopPropagation vs stopImmediatePropagation**:

```javascript
element.addEventListener('click', (e) => {
  console.log('监听器 1');
  e.stopPropagation(); // 阻止事件传播到父/子元素
  // 但同元素上的其他监听器仍会执行
});

element.addEventListener('click', (e) => {
  console.log('监听器 2'); // 仍然执行
  e.stopImmediatePropagation(); // 阻止传播 + 阻止后续监听器
});

element.addEventListener('click', (e) => {
  console.log('监听器 3'); // 不会执行
});
```

### Q24: 什么是事件委托？它有什么优势？

**难度**: ⭐

**答案**: 事件委托利用事件冒泡机制，将子元素的事件监听器绑定到父元素上，通过 `event.target` 判断实际触发事件的元素。

**解析**:

```javascript
// ❌ 不使用事件委托：为每个按钮绑定事件（性能差）
const buttons = document.querySelectorAll('.btn');
buttons.forEach(btn => {
  btn.addEventListener('click', (e) => {
    console.log(`按钮 ${e.target.textContent} 被点击`);
  });
});

// ✅ 使用事件委托：只在父元素上绑定一个事件
document.querySelector('.btn-container').addEventListener('click', (e) => {
  // 判断点击的是否是按钮
  const btn = e.target.closest('.btn'); // 向上查找最近的 .btn
  if (!btn) return;

  console.log(`按钮 ${btn.textContent} 被点击`);
  console.log(`按钮 data-id: ${btn.dataset.id}`);
});

// 动态添加的元素也能自动响应事件
const newBtn = document.createElement('button');
newBtn.className = 'btn';
newBtn.textContent = '新按钮';
newBtn.dataset.id = 'new';
document.querySelector('.btn-container').appendChild(newBtn);
```

**事件委托的优势**:
1. **减少内存消耗**: 只需一个事件监听器，而不是 N 个
2. **动态元素自动绑定**: 后续添加的子元素自动拥有事件处理
3. **简化代码**: 不需要为每个元素单独绑定/解绑事件

**事件委托的注意事项**:

```javascript
// 1. 使用 closest() 处理子元素嵌套
container.addEventListener('click', (e) => {
  const item = e.target.closest('.list-item');
  if (!item) return; // 点击的不是列表项

  // 即使点击的是列表项内部的子元素，也能正确匹配
  handleItemClick(item);
});

// 2. 事件委托不适用于不冒泡的事件
// focus、blur、mouseenter、mouseleave 不冒泡
// 可以使用 focusin/focusout 代替
container.addEventListener('focusin', (e) => {
  if (e.target.matches('.input-field')) {
    console.log('输入框获得焦点');
  }
});

// 3. 注意 stopPropagation 的影响
// 如果子元素调用了 stopPropagation()，事件不会冒泡到父元素
child.addEventListener('click', (e) => {
  e.stopPropagation(); // 父元素的事件委托将不会触发
});
```

### Q25: requestAnimationFrame 和 requestIdleCallback 分别是什么？有什么区别？

**难度**: ⭐⭐

**答案**: `requestAnimationFrame` 在浏览器下一次重绘前执行回调，适合动画和视觉更新。`requestIdleCallback` 在浏览器空闲时执行回调，适合低优先级的后台任务。

**解析**:

**requestAnimationFrame**:

```javascript
// 基本用法
function animate() {
  // 更新动画状态
  element.style.transform = `translateX(${x}px)`;
  x += 2;

  // 在下一次重绘前继续
  requestAnimationFrame(animate);
}
requestAnimationFrame(animate);

// 与 setTimeout 的区别
// setTimeout(fn, 16) ≈ 60fps，但不精确
// requestAnimationFrame 与浏览器刷新率同步（通常 60Hz 或 120Hz）
// requestAnimationFrame 在后台标签页中会自动暂停（节省性能）
// setTimeout 在后台标签页中仍会执行（浪费性能）

// 取消动画
let animationId = requestAnimationFrame(animate);
cancelAnimationFrame(animationId);
```

**requestIdleCallback**:

```javascript
// 基本用法
function processIdleTasks(deadline) {
  // deadline.timeRemaining() 返回当前空闲期剩余时间（毫秒）
  while (deadline.timeRemaining() > 0 && tasks.length > 0) {
    const task = tasks.shift();
    task(); // 执行一个任务
  }

  // 如果还有任务，安排在下一个空闲期
  if (tasks.length > 0) {
    requestIdleCallback(processIdleTasks);
  }
}
requestIdleCallback(processIdleTasks);

// 设置超时时间（确保任务最终执行）
requestIdleCallback(processIdleTasks, { timeout: 2000 });
// 即使浏览器一直不空闲，2秒后也会强制执行

// 取消
const id = requestIdleCallback(processIdleTasks);
cancelIdleCallback(id);
```

**对比**:

| 特性 | requestAnimationFrame | requestIdleCallback |
|------|----------------------|---------------------|
| 执行时机 | 下一次重绘前 | 浏览器空闲时 |
| 执行频率 | 与屏幕刷新率同步（60/120Hz） | 不确定，取决于浏览器空闲程度 |
| 适用场景 | 动画、视觉更新 | 数据分析、日志上报、预加载 |
| 后台标签页 | 暂停 | 可能不执行 |
| 精确性 | 高（与渲染同步） | 低（空闲时间不确定） |
| 回调参数 | timestamp（时间戳） | deadline（空闲截止时间） |

**实际应用：任务调度器**:

```javascript
class TaskScheduler {
  constructor() {
    this.highPriorityTasks = [];
    this.lowPriorityTasks = [];
    this.isRunning = false;
  }

  // 高优先级任务（使用 rAF，在下一帧执行）
  addHighPriority(task) {
    this.highPriorityTasks.push(task);
    if (!this.isRunning) this.run();
  }

  // 低优先级任务（使用 rIC，在空闲时执行）
  addLowPriority(task) {
    this.lowPriorityTasks.push(task);
    if (!this.isRunning) this.run();
  }

  run() {
    this.isRunning = true;

    // 先处理高优先级任务
    if (this.highPriorityTasks.length > 0) {
      requestAnimationFrame(() => {
        const task = this.highPriorityTasks.shift();
        task();
        if (this.highPriorityTasks.length > 0 || this.lowPriorityTasks.length > 0) {
          this.run();
        } else {
          this.isRunning = false;
        }
      });
      return;
    }

    // 再处理低优先级任务
    if (this.lowPriorityTasks.length > 0) {
      requestIdleCallback((deadline) => {
        while (deadline.timeRemaining() > 0 && this.lowPriorityTasks.length > 0) {
          const task = this.lowPriorityTasks.shift();
          task();
        }
        if (this.lowPriorityTasks.length > 0) {
          this.run();
        } else {
          this.isRunning = false;
        }
      }, { timeout: 1000 });
      return;
    }

    this.isRunning = false;
  }
}

// 使用
const scheduler = new TaskScheduler();
scheduler.addHighPriority(() => updateUI());
scheduler.addLowPriority(() => reportAnalytics());
scheduler.addLowPriority(() => prefetchData());
```

### Q26: requestIdleCallback 的使用场景和限制有哪些？

**难度**: ⭐⭐

**答案**: requestIdleCallback 适合执行不紧急的后台任务，但有执行时间限制和兼容性问题。

**解析**:

**典型使用场景**:

```javascript
// 1. 分批处理大数据
function processLargeData(data, chunkSize = 100) {
  let index = 0;

  function processChunk(deadline) {
    while (index < data.length && deadline.timeRemaining() > 0) {
      // 处理一批数据
      const end = Math.min(index + chunkSize, data.length);
      for (let i = index; i < end; i++) {
        heavyComputation(data[i]);
      }
      index = end;
    }

    if (index < data.length) {
      requestIdleCallback(processChunk);
    } else {
      console.log('所有数据处理完成');
    }
  }

  requestIdleCallback(processChunk);
}

// 2. 日志上报（不阻塞主线程）
function sendAnalytics(events) {
  requestIdleCallback(() => {
    if (navigator.sendBeacon) {
      navigator.sendBeacon('/analytics', JSON.stringify(events));
    } else {
      fetch('/analytics', {
        method: 'POST',
        body: JSON.stringify(events),
        keepalive: true,
      });
    }
  }, { timeout: 5000 });
}

// 3. 预加载下一页数据
function prefetchNextPage() {
  requestIdleCallback(() => {
    fetch('/api/page2')
      .then(res => res.json())
      .then(data => {
        sessionStorage.setItem('page2-cache', JSON.stringify(data));
      });
  }, { timeout: 3000 });
}

// 4. 预渲染/预计算
function precomputeLayout() {
  requestIdleCallback(() => {
    const nextLayout = calculateLayout(nextItems);
    layoutCache.set('next', nextLayout);
  });
}
```

**限制和注意事项**:

```javascript
// 1. 空闲时间不保证
// requestIdleCallback 的回调可能在很长时间内不被调用
// （如果用户持续交互、动画持续运行）
// 使用 timeout 参数确保最终执行
requestIdleCallback(task, { timeout: 2000 });

// 2. 单次空闲时间很短
// deadline.timeRemaining() 通常只有几毫秒
// 不要在回调中执行耗时操作
requestIdleCallback((deadline) => {
  console.log(deadline.timeRemaining()); // 可能只有 5-50ms
  // ❌ 不要执行耗时操作
  // sortHugeArray();
  // ✅ 每次只处理一小部分
  processOneItem();
  if (hasMoreItems) {
    requestIdleCallback(processMore);
  }
});

// 3. 兼容性问题
// requestIdleCallback 在 Safari 中不支持
// Polyfill：
if (!window.requestIdleCallback) {
  window.requestIdleCallback = (callback, options) => {
    const start = Date.now();
    return setTimeout(() => {
      callback({
        didTimeout: false,
        timeRemaining: () => Math.max(0, 50 - (Date.now() - start)),
      });
    }, 1);
  };
  window.cancelIdleCallback = (id) => clearTimeout(id);
}

// 4. 不要在空闲回调中修改 DOM
// 修改 DOM 可能触发样式计算和布局，影响用户体验
requestIdleCallback(() => {
  // ❌ 不要修改 DOM
  // element.style.color = 'red';
  // ✅ 准备数据，在 rAF 中修改 DOM
  prepareData();
  requestAnimationFrame(() => {
    element.style.color = 'red';
  });
});
```

### Q27: 浏览器的事件循环（Event Loop）在浏览器和 Node.js 中有什么区别？

**难度**: ⭐⭐⭐

**答案**: 浏览器和 Node.js 都基于事件循环，但宏任务和微任务的处理方式有所不同。

**解析**:

**浏览器事件循环**:

```
┌─────────────────────────────────────────┐
│               调用栈（Call Stack）        │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│              微任务队列                   │
│  - Promise.then/catch/finally           │
│  - MutationObserver                      │
│  - queueMicrotask                       │
│  - Object.observe（已废弃）              │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│              宏任务队列                   │
│  - setTimeout / setInterval              │
│  - requestAnimationFrame                │
│  - I/O 操作                              │
│  - UI 渲染                               │
│  - MessageChannel                        │
└─────────────────────────────────────────┘

执行顺序：
1. 执行同步代码（调用栈）
2. 调用栈清空后，执行所有微任务
3. 微任务清空后，执行一个宏任务
4. 执行完宏任务后，再次清空微任务
5. 重复步骤 3-4
```

```javascript
console.log('1. 同步代码');

setTimeout(() => {
  console.log('2. 宏任务 setTimeout');
  Promise.resolve().then(() => {
    console.log('3. 微任务（在 setTimeout 内）');
  });
}, 0);

Promise.resolve().then(() => {
  console.log('4. 微任务 Promise');
  setTimeout(() => {
    console.log('5. 宏任务（在 Promise 内）');
  }, 0);
});

console.log('6. 同步代码');

// 输出顺序：
// 1. 同步代码
// 6. 同步代码
// 4. 微任务 Promise
// 2. 宏任务 setTimeout
// 3. 微任务（在 setTimeout 内）
// 5. 宏任务（在 Promise 内）
```

**Node.js 事件循环（与浏览器的区别）**:

```
Node.js 事件循环阶段：
   ┌───────────────────────────┐
┌─>│         timers            │  setTimeout / setInterval
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │     pending callbacks     │  I/O 回调
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │       idle, prepare       │  内部使用
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │         poll              │  获取新的 I/O 事件
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │        check              │  setImmediate 回调
│  └───────────┬───────────────┘
│  ┌───────────▼───────────────┐
│  │    close callbacks        │  关闭事件
│  └───────────┬───────────────┘
│              │
└──────────────┘
```

**关键区别**:

| 特性 | 浏览器 | Node.js |
|------|--------|---------|
| 微任务执行时机 | 每个宏任务后 | 每个阶段后 |
| 宏任务队列 | 单一队列 | 多阶段（timers、poll、check 等） |
| requestAnimationFrame | 有 | 无 |
| setImmediate | 无 | 有 |
| process.nextTick | 无 | 有（优先级高于微任务） |

```javascript
// Node.js 特有
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
// 输出不确定！取决于事件循环当前阶段

// process.nextTick 优先级最高
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));
// 输出：nextTick → promise
```

### Q28: Performance API 有哪些？如何使用 Navigation Timing 和 Resource Timing？

**难度**: ⭐⭐

**答案**: Performance API 提供了一组高精度时间度量接口，包括 Navigation Timing（页面导航计时）、Resource Timing（资源加载计时）、User Timing（自定义计时）等。

**解析**:

**Navigation Timing 2.0**:

```
redirect → appCache → DNS → TCP → request → response → processing → onload
|<------------------------ Navigation Timing ------------------------>|

详细时间线：
┌─────────────────────────────────────────────────────────────┐
│                    navigationStart                           │
│  ┌──────────────┐                                            │
│  │  redirectStart│→ redirectEnd                              │
│  └──────────────┘                                            │
│  ┌──────────────┐                                            │
│  │  fetchStart   │（开始获取文档）                             │
│  └──────┬───────┘                                            │
│         ┌▼──────────────┐                                    │
│         │ domainLookupStart│→ domainLookupEnd（DNS 查询）      │
│         └──────────────┘                                     │
│         ┌──────────────┐                                     │
│         │ connectStart  │→ connectEnd（TCP 连接）              │
│         └──────────────┘                                     │
│         ┌──────────────┐                                     │
│         │ requestStart  │（发送请求）                          │
│         └──────┬───────┘                                     │
│         ┌──────▼───────┐                                     │
│         │ responseStart │→ responseEnd（接收响应）             │
│         └──────────────┘                                     │
│         ┌──────────────┐                                     │
│         │ domLoading   │（开始解析 DOM）                      │

│         │ domContentLoaded│（DOMContentLoaded 事件触发）       │

│         ┌──────▼───────┐                                     │
│         │ domComplete  │（页面完全加载）                       │

│         └──────────────┘                                     │
└─────────────────────────────────────────────────────────────┘
```

**代码示例**:

```javascript
// 1. Navigation Timing - 页面加载性能
const [navEntry] = performance.getEntriesByType('navigation');

if (navEntry) {
  // DNS 查询时间
  const dnsTime = navEntry.domainLookupEnd - navEntry.domainLookupStart;

  // TCP 连接时间
  const tcpTime = navEntry.connectEnd - navEntry.connectStart;

  // SSL 握手时间（HTTPS）
  const sslTime = navEntry.secureConnectionStart > 0
    ? navEntry.connectEnd - navEntry.secureConnectionStart
    : 0;

  // 请求到响应时间（TTFB）
  const ttfb = navEntry.responseStart - navEntry.requestStart;

  // 响应内容下载时间
  const downloadTime = navEntry.responseEnd - navEntry.responseStart;

  // DOM 解析时间
  const domParseTime = navEntry.domInteractive - navEntry.responseEnd;

  // DOMContentLoaded 时间
  const dclTime = navEntry.domContentLoadedEventEnd - navEntry.startTime;

  // 页面完全加载时间
  const loadTime = navEntry.loadEventEnd - navEntry.startTime;

  console.table({
    'DNS 查询': `${dnsTime}ms`,
    'TCP 连接': `${tcpTime}ms`,
    'SSL 握手': `${sslTime}ms`,
```

```javascript
function smoothScroll(element, target) {
  const start = element.scrollTop;
  const distance = target - start;
  const duration = 500;
  let startTime = null;

  function step(timestamp) {
    if (!startTime) startTime = timestamp;
    const progress = Math.min((timestamp - startTime) / duration, 1);
    const eased = 1 - Math.pow(1 - progress, 3); // easeOutCubic
    element.scrollTop = start + distance * eased;
    if (progress < 1) requestAnimationFrame(step);
  }

  requestAnimationFrame(step);
}

// 5. 防抖和节流
function debounce(fn, delay) {
  let timer;
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

function throttle(fn, interval) {
  let lastTime = 0;
  return function (...args) {
    const now = Date.now();
    if (now - lastTime >= interval) {
      lastTime = now;
      fn.apply(this, args);
    }
  };
}
```
