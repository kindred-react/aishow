## 一、TCP/UDP

### Q1: TCP 三次握手的过程和目的是什么？

**难度**: ⭐⭐

**答案**: 三次握手：客户端发送 SYN，服务端回复 SYN+ACK，客户端发送 ACK。目的是确认双方的发送和接收能力正常，建立可靠连接。

**解析**:
```
客户端                    服务器
  |---SYN (seq=x)-------->|
  |                        | 服务器进入 SYN_RCVD
  |<--SYN+ACK (seq=y,ack=x+1)---|
  |  客户端进入 ESTABLISHED |
  |---ACK (ack=y+1)------>|
  |                        | 服务器进入 ESTABLISHED

为什么是三次不是两次？
- 两次无法确认客户端的接收能力
- 防止已失效的连接请求到达服务器，造成资源浪费
```

---

### Q2: TCP 四次挥手的过程和目的是什么？

**难度**: ⭐⭐

**答案**: 四次挥手：主动方发送 FIN，被动方回复 ACK，被动方发送 FIN，主动方回复 ACK。目的是双方都能通知对方数据发送完毕，安全关闭连接。

**解析**:
```
客户端（主动关闭）          服务器（被动关闭）
  |---FIN (seq=u)-------->|
  |  进入 FIN_WAIT_1      |  进入 CLOSE_WAIT
  |<--ACK (ack=u+1)------|
  |  进入 FIN_WAIT_2      |  （继续发送未完成的数据）
  |<--FIN (seq=w)--------|
  |  进入 TIME_WAIT       |  进入 LAST_ACK
  |---ACK (ack=w+1)------>|
  |                        |  进入 CLOSED
  |  等待 2MSL 后进入 CLOSED

为什么是四次不是三次？
- TCP 是全双工的，双方都需要单独关闭
- 被动方收到 FIN 后可能还有数据未发送完
```

### Q3: 为什么需要 TIME_WAIT 状态？等待 2MSL 的原因？

**难度**: ⭐⭐

**答案**: TIME_WAIT 持续 2MSL（最大报文段生存时间），原因：(1) 确保最后一个 ACK 能到达被动方（如果丢失，被动方会重发 FIN）；(2) 让本连接的所有报文在网络中消失，防止影响新连接。

**解析**:
```
2MSL 的原因：
1. 确保被动方重发的 FIN 能被收到和响应
2. 确保网络中残留的报文过期

TIME_WAIT 过多的问题：
- 占用连接资源（端口和内存）
- 解决方案：
  - net.ipv4.tcp_tw_reuse = 1（允许复用 TIME_WAIT 连接）
  - net.ipv4.tcp_tw_recycle = 1（不推荐，可能导致 NAT 问题）
  - 长连接代替短连接
  - 客户端主动关闭连接
```

### Q4: TCP 拥塞控制有哪些算法？

**难度**: ⭐⭐

**答案**: 四种算法：慢启动（指数增长）、拥塞避免（线性增长）、快重传（收到 3 个重复 ACK 立即重传）、快恢复（将拥塞窗口减半而非回到慢启动）。

**解析**:
```
慢启动：
  cwnd 从 1 开始，每收到一个 ACK，cwnd 翻倍（指数增长）
  直到达到慢启动门限 ssthresh

拥塞避免：
  cwnd 超过 ssthresh 后，每个 RTT cwnd 增加 1（线性增长）

快重传：
  收到 3 个重复 ACK，不等待超时，立即重传丢失的报文

快恢复：
  快重传后将 ssthresh = cwnd / 2，cwnd = ssthresh（而非 1）
  直接进入拥塞避免阶段

拥塞窗口变化：
  超时 → ssthresh = cwnd/2, cwnd = 1（慢启动）
  3个重复ACK → ssthresh = cwnd/2, cwnd = ssthresh（快恢复）
```

### Q5: TCP 如何保证可靠传输？

**难度**: ⭐⭐

**答案**: (1) 校验和（检测数据损坏）；(2) 序列号（按序到达）；(3) 确认应答 + 重传（超时重传、快速重传）；(4) 流量控制（滑动窗口）；(5) 拥塞控制。

**解析**:
```
可靠传输机制：
1. 校验和 — 检测传输中的比特错误
2. 序列号 — 标识数据顺序，接收方按序重组
3. 确认应答 — 接收方确认已收到的数据
4. 超时重传 — 发送方未收到 ACK，超时后重传
5. 快速重传 — 收到 3 个重复 ACK 立即重传
6. 滑动窗口 — 控制发送速率，实现流量控制
7. 拥塞控制 — 避免网络过载
```

### Q6: TCP 和 UDP 的区别？

**难度**: ⭐

**答案**: TCP 面向连接、可靠传输、有序、有流量/拥塞控制、开销大；UDP 无连接、不可靠、无序、无控制、开销小、速度快。

**解析**:
| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠 | 不可靠 |
| 有序性 | 有序 | 无序 |
| 传输速度 | 较慢 | 快 |
| 开销 | 大（20字节头部） | 小（8字节头部） |
| 应用场景 | HTTP、FTP、SMTP | DNS、视频流、游戏 |

### Q7: TCP 滑动窗口的原理是什么？

**难度**: ⭐⭐

**答案**: 滑动窗口是发送方和接收方维护的缓冲区大小，控制数据发送速率，实现流量控制。接收方通过 ACK 中的窗口大小通知发送方自己的接收能力。

**解析**:
```
发送方窗口：
[已发送已确认 | 已发送未确认 | 未发送可发送 | 不可发送]
              ←—— 窗口大小 ——→

接收方通过 ACK 告知窗口大小：
- 窗口为 0 时，发送方停止发送
- 窗口更新后，发送方恢复发送

零窗口探测：
- 接收方窗口为 0 后，发送方定期发送探测报文
- 防止接收方发送的窗口更新报文丢失导致死锁
```

### Q8: 什么是 CLOSE_WAIT 状态？如何避免过多？

**难度**: ⭐⭐

**答案**: CLOSE_WAIT 是被动关闭方收到 FIN 后、发送 FIN 前的状态。过多说明被动方没有正确关闭连接（代码中未调用 close()）。

**解析**:
```
CLOSE_WAIT 产生原因：
1. 服务端收到 FIN 后回复了 ACK，但没有调用 close()
2. 通常是代码 bug：没有正确关闭数据库连接、文件句柄等

解决方案：
1. 确保所有连接在不再使用时调用 close()
2. 使用连接池管理连接
3. 设置合理的超时时间
4. 代码审查确保资源释放

排查命令：
netstat -anp | grep CLOSE_WAIT
```

### Q9: HTTP 请求在 TCP 传输过程中的完整流程？

**难度**: ⭐⭐

**答案**: DNS 解析 → TCP 三次握手 → 发送 HTTP 请求 → 服务器处理 → 返回 HTTP 响应 → TCP 四次挥手（或 keep-alive 复用连接）。

**解析**:
```
1. DNS 解析：域名 → IP 地址（递归查询 + 迭代查询）
2. TCP 连接：三次握手建立连接
3. TLS 握手（HTTPS）：协商加密参数
4. 发送请求：HTTP 请求报文
5. 服务器处理：路由、中间件、业务逻辑
6. 返回响应：HTTP 响应报文
7. 浏览器解析：HTML → CSSOM → DOM → Render Tree → Layout → Paint
8. TCP 断开：四次挥手（keep-alive 则复用连接）

DNS 优化：
- DNS 缓存（浏览器、OS、路由器）
- DNS 预解析：<link rel="dns-prefetch">
- HTTPDNS：绕过运营商 LocalDNS
```

### Q10: TCP 粘包问题是什么？如何解决？

**难度**: ⭐⭐

**答案**: TCP 粘包是发送方多个小包被合并成一个大包，或一个大包被拆分成多个小包。解决方法：消息定长、消息头包含长度、特殊分隔符。

**解析**:
```
粘包原因：
1. Nagle 算法 — 小包合并发送
2. TCP 优化 — 合并多个小段
3. 接收方读取不及时 — 缓冲区积压

解决方案：
1. 消息定长 — 每条消息固定长度，不足补齐
2. 长度前缀 — 消息头包含消息体长度
3. 特殊分隔符 — 用 \r\n 等分隔消息
4. 应用层协议 — HTTP/1.1 使用 Content-Length 或 chunked

// Node.js 示例：长度前缀
function encode(data) {
  const body = Buffer.from(JSON.stringify(data));
  const header = Buffer.alloc(4);
  header.writeUInt32BE(body.length);
  return Buffer.concat([header, body]);
}
```

## 二、HTTP/HTTPS

### Q11: HTTP/1.1、HTTP/2、HTTP/3 的核心区别？

**难度**: ⭐⭐

**答案**: HTTP/1.1 持久连接但有队头阻塞；HTTP/2 二进制分帧、多路复用、头部压缩；HTTP/3 基于 QUIC（UDP），解决 TCP 队头阻塞，支持 0-RTT。

**解析**:
| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|----------|--------|--------|
| 传输层 | TCP | TCP | QUIC (UDP) |
| 多路复用 | 否（管道化有 HOL） | 是 | 是 |
| 头部压缩 | 否 | HPACK | QPACK |
| 队头阻塞 | TCP 层面 | TCP 层面 | 无 |
| 连接迁移 | 不支持 | 不支持 | 支持 |
| 握手延迟 | 1-RTT + TCP | 1-RTT + TCP | 0-RTT（后续） |

### Q12: HTTPS 的握手过程是怎样的？

**难度**: ⭐⭐

**答案**: TLS 1.2：Client Hello → Server Hello + 证书 → 客户端验证证书 → 生成预主密钥（用公钥加密）→ 双方生成会话密钥 → 加密通信。TLS 1.3 简化为 1-RTT。

**解析**:
```
TLS 1.2 握手（2-RTT）：
1. 客户端 → 服务器：Client Hello（TLS版本、加密套件、随机数1）
2. 服务器 → 客户端：Server Hello + 证书 + 服务器随机数2
3. 客户端 → 服务器：预主密钥（用服务器公钥加密）
4. 双方生成主密钥：PRF(预主密钥, "master secret", 随机数1+随机数2)
5. 客户端 → 服务器：Change Cipher Spec + Finished
6. 服务器 → 客户端：Change Cipher Spec + Finished

TLS 1.3 握手（1-RTT）：
1. 客户端 → 服务器：Client Hello + Key Share
2. 服务器 → 客户端：Server Hello + Key Share + 证书 + Finished
3. 客户端 → 服务器：Finished
```

### Q13: HTTP 常见状态码及其含义？

**难度**: ⭐

**答案**: 200 OK、201 Created、204 No Content、301 永久重定向、302 临时重定向、304 未修改、400 请求错误、401 未认证、403 无权限、404 未找到、429 请求过多、500 服务器错误、502 网关错误、504 网关超时。

**解析**:
```
重点区分：
- 301 vs 302：301 永久重定向（SEO 更新索引），302 临时重定向
- 401 vs 403：401 未认证（需登录），403 已认证但无权限
- 502 vs 504：502 上游服务器错误，504 上游服务器超时
```

### Q14: HTTP 缓存机制有哪些？

**难度**: ⭐⭐

**答案**: 强缓存（Cache-Control/Expires）和协商缓存（ETag/If-None-Match、Last-Modified/If-Modified-Since）。强缓存不发送请求，协商缓存发送请求验证。

**解析**:
```
强缓存：
  Cache-Control: max-age=31536000（优先级高）
  Expires: Wed, 09 Apr 2027 00:00:00 GMT
  → 不发送请求，直接使用缓存

协商缓存：
  ETag + If-None-Match（优先级高）
  Last-Modified + If-Modified-Since
  → 发送请求验证，304 使用缓存，200 返回新资源

缓存策略：
  HTML → 协商缓存（每次验证）
  JS/CSS → 强缓存（内容hash，文件名变化时更新）
  图片 → 强缓存 + 协商缓存
  API → no-cache/no-store
```

### Q15: GET 和 POST 的区别？

**难度**: ⭐

**答案**: 语义上 GET 获取资源（安全、幂等），POST 创建资源（不安全、不幂等）。技术上 GET 参数在 URL，POST 在请求体；GET 有长度限制，POST 没有；GET 可缓存，POST 不可缓存。

**解析**:
```
本质区别是语义，不是参数位置：
- GET 用于获取数据，是安全且幂等的
- POST 用于提交数据，不是安全也不是幂等的

其他区别：
- GET 请求可被缓存、收藏、记录历史
- POST 请求不能
- GET 参数在 URL 中，POST 在请求体中
- GET 的 URL 长度有限制（浏览器/服务器配置）
- CORS 预检：简单请求（GET）不预检，复杂请求（Content-Type: application/json 的 POST）需要预检
```

### Q16: Cookie、Session、JWT 的区别？

**难度**: ⭐⭐

**答案**: Cookie 存储在客户端，Session 存储在服务端（通过 Cookie 中的 Session ID 关联），JWT 是自包含的令牌（包含用户信息，无需服务端存储）。

**解析**:
```
Cookie：
  - 存储在浏览器，每次请求自动携带
  - 大小限制 4KB
  - 可设置 HttpOnly、Secure、SameSite、Expires/Max-Age

Session：
  - 数据存储在服务端
  - 客户端只存 Session ID（Cookie）
  - 服务端需要存储 Session 数据（内存/Redis）
  - 分布式环境需要 Session 共享（Redis）

JWT：
  - Header.Payload.Signature 三段式
  - 自包含用户信息，无需服务端存储
  - 适合分布式系统
  - 无法主动失效（除非维护黑名单）
  - 注意：不要在 Payload 中放敏感信息

选择建议：
- 单体应用 → Session
- 分布式/微服务 → JWT
- 需要"记住我" → Cookie + JWT
```

### Q17: 什么是跨域？如何解决？

**难度**: ⭐⭐

**答案**: 跨域是浏览器的同源策略（协议+域名+端口）限制。解决方法：CORS（Access-Control-Allow-Origin）、代理服务器、JSONP（仅 GET）、WebSocket、postMessage。

**解析**:
```javascript
// CORS — 服务端设置响应头
// 简单请求：直接发送，浏览器检查响应头
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Allow-Credentials: true

// 预检请求（OPTIONS）
// 非简单请求（PUT、DELETE、自定义Header）先发 OPTIONS 请求

// 代理服务器（开发环境）
// vite.config.js
export default {
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:3000',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, '')
      }
    }
  }
};

// JSONP（仅 GET，已过时）
function jsonp(url, callback) {
  const script = document.createElement('script');
  script.src = `${url}?callback=${callback}`;
  document.body.appendChild(script);
}
```

### Q18: HTTP/2 多路复用解决了什么问题？

**难度**: ⭐⭐

**答案**: HTTP/1.1 的管道化虽然支持并发请求，但响应必须按顺序返回（队头阻塞）。HTTP/2 通过二进制分帧和 Stream 实现真正的多路复用，一个 TCP 连接上可以并行交错发送多个请求和响应。

**解析**:
```
HTTP/1.1 管道化问题：
  请求1 → 请求2 → 请求3
  响应1 → 响应2 → 响应3（必须按顺序）
  如果响应1慢，响应2和3都被阻塞

HTTP/2 多路复用：
  Stream1: 请求1 → 响应1
  Stream2: 请求2 → 响应2（可以乱序返回）
  Stream3: 请求3 → 响应3

注意：HTTP/2 仍有 TCP 层面的队头阻塞
（TCP 丢包时，所有 Stream 都要等待重传）
HTTP/3 使用 QUIC（基于 UDP）解决了这个问题
```

### Q19: 什么是 Content-Security-Policy（CSP）？

**难度**: ⭐⭐

**答案**: CSP 是 HTTP 响应头，限制页面可以加载哪些资源（脚本、样式、图片等），防止 XSS 攻击。

**解析**:
```
// 常用 CSP 指令
Content-Security-Policy:
  default-src 'self';           // 默认只允许同源
  script-src 'self' 'nonce-xxx'; // 脚本只允许同源或指定 nonce
  style-src 'self' 'unsafe-inline'; // 样式允许内联
  img-src 'self' data: https:;   // 图片允许 data URI 和 HTTPS
  connect-src 'self' https://api.example.com; // AJAX 限制
  frame-ancestors 'none';        // 禁止被 iframe 嵌入

// CSP 与内联脚本冲突
// 解决方案：使用 nonce 或 hash
<script nonce="abc123">
  // 只有 nonce 匹配的脚本才能执行
</script>
```

### Q20: HTTP/2 Server Push 的作用和限制？

**难度**: ⭐⭐

**答案**: Server Push 允许服务器在客户端请求之前主动推送资源。但由于命中率低、缓存管理复杂、与浏览器预加载机制重叠，实际效果有限，HTTP/3 中已移除。

**解析**:
```
Server Push 流程：
1. 客户端请求 index.html
2. 服务器返回 index.html，同时推送 style.css 和 app.js
3. 客户端收到后放入缓存

限制：
- 浏览器可能拒绝推送的资源（已在缓存中）
- 无法准确预测客户端需要什么
- 消耗服务器带宽
- 缓存管理复杂

替代方案：
- <link rel="preload" href="style.css" as="style">
- HTTP/103 Early Hints
```

### Q21: WebSocket 和 HTTP 的区别？

**难度**: ⭐⭐

**答案**: WebSocket 是全双工通信协议，建立连接后双方可随时发送数据；HTTP 是请求-响应模式，单向通信。WebSocket 基于 HTTP 升级建立连接。

**解析**:
```
WebSocket 建立过程：
1. 客户端发送 HTTP 升级请求
   GET /chat HTTP/1.1
   Upgrade: websocket
   Connection: Upgrade
   Sec-WebSocket-Key: xxx

2. 服务器返回 101 Switching Protocols
   HTTP/1.1 101 Switching Protocols
   Upgrade: websocket
   Connection: Upgrade

3. 之后使用 WebSocket 协议通信（帧格式）

应用场景：即时通讯、在线协作、股票行情、游戏
```

**难度**: ⭐⭐

**答案**: HPACK 使用静态表（预定义的 61 个常见头部）、动态表（之前传输过的头部）和哈夫曼编码压缩 HTTP 头部，减少传输数据量。

## 三、浏览器原理

### Q23: 浏览器渲染流水线的步骤？

**难度**: ⭐⭐

**答案**: HTML → DOM 树、CSS → CSSOM 树 → Render Tree → Layout（计算位置大小）→ Paint（绘制像素）→ Composite（合成层）。

**解析**:
```
1. 解析 HTML → DOM 树
2. 解析 CSS → CSSOM 树
3. DOM + CSSOM → Render Tree（只包含可见节点）
4. Layout — 计算每个节点的确切位置和大小
5. Paint — 将节点绘制为像素
6. Composite — 将多个图层合成为最终图像

优化关键：
- 减少重排（Layout）：批量修改 DOM、使用 transform
- 减少重绘（Paint）：避免频繁修改样式
- 使用 GPU 加速：transform、opacity、will-change
- 合成层优化：减少不必要的层提升
```

### Q24: 重排（Reflow）和重绘（Repaint）的区别？

**难度**: ⭐⭐

**答案**: 重排是元素几何属性变化导致重新计算布局（影响子树）；重绘是元素外观变化（颜色、背景）不影响布局。重排一定触发重绘，重绘不一定触发重排。

**解析**:
```
触发重排的操作：
- 修改 width/height/padding/margin/border
- 修改 display/position/float
- 添加/删除 DOM 节点
- 读取 offsetWidth/offsetHeight/clientWidth 等
- 窗口 resize

触发重绘的操作：
- 修改 color/background/border-color
- 修改 visibility/box-shadow/outline
- 修改 text-decoration

优化策略：
1. 批量修改 DOM（DocumentFragment）
2. 使用 transform 代替 top/left（只触发 Composite）
3. 使用 opacity 代替 visibility（只触发 Composite）
4. 避免频繁读取布局属性（读写分离）
5. 使用 requestAnimationFrame 批量处理
```

### Q25: 浏览器的垃圾回收机制？

**难度**: ⭐⭐

**答案**: 主要有两种 GC 算法：标记清除（Mark-Sweep）和引用计数（Reference Counting）。现代浏览器使用标记清除 + 分代回收（新生代 Scavenge、老生代 Mark-Sweep/Mark-Compact）。

**解析**:
```
标记清除（Mark-Sweep）：
1. 从根对象（window/global）开始遍历
2. 可达的对象标记为"活跃"
3. 未标记的对象被回收

分代回收：
- 新生代（存活时间短）：Scavenge 算法
  - 分为 From 和 To 两个空间
  - 每次将 From 中存活对象复制到 To
  - From 和 To 交换
  - 空间小、回收快
- 老生代（存活时间长）：Mark-Sweep / Mark-Compact
  - 空间大、回收慢
  - 对象经过多次 Scavenge 仍然存活，晋升到老生代

内存泄漏常见原因：
1. 意外的全局变量
2. 被遗忘的定时器
3. 闭包引用大对象
4. DOM 引用未移除
5. 控制台日志（开发环境）
```

### Q26: 浏览器的事件循环机制？

**难度**: ⭐⭐

**答案**: 执行同步代码 → 清空微任务队列 → 取一个宏任务执行 → 清空微任务队列 → 循环。微任务优先级高于宏任务。

**解析**:
```
宏任务：setTimeout、setInterval、requestAnimationFrame、I/O、UI 渲染
微任务：Promise.then/catch/finally、MutationObserver、queueMicrotask

执行顺序：
1. 执行同步代码（属于第一个宏任务）
2. 清空所有微任务
3. 取下一个宏任务执行
4. 清空所有微任务
5. 如有必要，进行 UI 渲染
6. 回到步骤 3

requestAnimationFrame 在 UI 渲染前执行
requestIdleCallback 在浏览器空闲时执行
```

### Q27: 浏览器存储有哪些？区别是什么？

**难度**: ⭐

**答案**: Cookie（4KB、每次请求携带）、localStorage（5MB、持久化）、sessionStorage（5MB、会话级）、IndexedDB（大容量、结构化数据）。

**解析**:
| 特性 | Cookie | localStorage | sessionStorage | IndexedDB |
|------|--------|-------------|---------------|-----------|
| 容量 | 4KB | 5MB | 5MB | 数百MB+ |
| 生命周期 | 可设置过期 | 永久 | 会话 | 永久 |
| 请求携带 | 自动 | 不 | 不 | 不 |
| API | document.cookie | localStorage | sessionStorage | IndexedDB API |
| 数据格式 | 字符串 | 字符串 | 字符串 | 结构化 |

### Q28: 什么是同源策略？有哪些限制？

**难度**: ⭐⭐

**答案**: 同源策略要求协议、域名、端口完全相同。限制：不能发送 AJAX 请求（CORS 除外）、不能操作其他窗口的 DOM、不能读取其他域的 Cookie/Storage。

**解析**:
```
不受同源策略限制的：
- <script src="..."> — JSONP
- <link rel="stylesheet"> — CSS
- <img> — 图片
- <video>/<audio> — 媒体
- <form> — 表单提交
- <a href="..."> — 跳转

受同源策略限制的：
- AJAX/Fetch 请求
- DOM 操作（iframe）
- Cookie/Storage 访问
- IndexedDB 访问
```

### Q29: 浏览器的预加载和预获取有哪些？

**难度**: ⭐⭐

**答案**: dns-prefetch（DNS 预解析）、preconnect（TCP+TLS 预连接）、preload（预加载关键资源）、prefetch（预获取下个页面资源）、prerender（预渲染下个页面）。

**解析**:
```html
<!-- DNS 预解析 -->
<link rel="dns-prefetch" href="//cdn.example.com">

<!-- 预连接（DNS + TCP + TLS） -->
<link rel="preconnect" href="//api.example.com" crossorigin>

<!-- 预加载关键资源 -->
<link rel="preload" href="critical.css" as="style">
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>

<!-- 预获取（低优先级） -->
<link rel="prefetch" href="/next-page.js">

<!-- 预渲染（高开销，谨慎使用） -->
<link rel="prerender" href="/next-page">
```

### Q30: 什么是浏览器的合成层（Composite Layer）？如何优化？

**难度**: ⭐⭐⭐

**答案**: 合成层是浏览器为某些元素创建的独立图层，GPU 单独渲染后与其他层合成。过多合成层会消耗内存。

**解析**:
```
创建合成层的条件：
1. 3D 变换（transform: translateZ(0)）
2. will-change: transform/opacity
3. <video>、<canvas>、<iframe>
4. opacity 动画
5. position: fixed（部分浏览器）

优化建议：
1. 不要滥用 will-change（只在需要时使用）
2. 使用 transform 代替 top/left
3. 使用 opacity 代替 visibility
4. 减少不必要的合成层
5. 使用 Chrome DevTools Layers 面板分析

// good — 只触发 Composite
.element {
  transform: translateX(100px);
  opacity: 0.5;
}

// bad — 触发 Layout + Paint
.element {
  left: 100px;
  background: red;
}
```

### Q31: 什么是 requestAnimationFrame？与 setTimeout 的区别？

**难度**: ⭐⭐

**答案**: rAF 在下一次重绘前执行，与浏览器刷新率同步（通常 60fps）；setTimeout 最小延迟 4ms，不与渲染周期同步。

**解析**:
```
rAF 优势：
1. 与刷新率同步，避免掉帧
2. 页面不可见时自动暂停
3. 浏览器优化多个 rAF 回调

setTimeout 问题：
1. 不与渲染周期同步
2. 页面不可见时仍执行
3. 嵌套调用延迟至少 4ms

动画应使用 rAF：
function animate() {
  element.style.transform = `translateX(${x}px)`;
  requestAnimationFrame(animate);
}
```

### Q32: 浏览器如何处理 JavaScript 阻塞？

**难度**: ⭐⭐

**答案**: JavaScript 会阻塞 DOM 解析。解决方案：script 标签放在底部、使用 async/defer 属性、使用动态创建 script。

**解析**:
```html
<!-- 阻塞解析 -->
<script src="app.js"></script>

<!-- async — 异步下载，下载完立即执行（不保证顺序） -->
<script async src="app.js"></script>

<!-- defer — 异步下载，DOM 解析完后按顺序执行 -->
<script defer src="app.js"></script>

<!-- 推荐：defer 保证执行顺序，不阻塞解析 -->
<script defer src="vendor.js"></script>
<script defer src="app.js"></script>
```

## 四、网络安全

### Q33: XSS 攻击是什么？如何防御？

**难度**: ⭐⭐

**答案**: XSS（跨站脚本攻击）是向页面注入恶意脚本。分为存储型（恶意代码存入数据库）、反射型（URL 参数中包含恶意代码）、DOM 型（前端 JS 不当操作 DOM）。

**解析**:
```
防御措施：
1. 输入转义 — 对用户输入进行 HTML 实体编码
2. CSP（Content-Security-Policy）— 限制脚本来源
3. HttpOnly Cookie — 防止 JS 读取 Cookie
4. 使用 textContent 代替 innerHTML
5. 框架自动转义 — React/Vue 默认转义

// bad
element.innerHTML = userInput;

// good
element.textContent = userInput;
// 或使用 DOMPurify 库
element.innerHTML = DOMPurify.sanitize(userInput);
```

### Q34: CSRF 攻击是什么？如何防御？

**难度**: ⭐⭐

**答案**: CSRF（跨站请求伪造）是利用用户已登录的身份，在用户不知情的情况下发送恶意请求。

**解析**:
```
防御措施：
1. CSRF Token — 每次请求携带随机 Token
2. SameSite Cookie — 限制跨站请求携带 Cookie
   Set-Cookie: sessionid=abc; SameSite=Strict/Lax/None
3. 验证 Referer/Origin 头
4. 关键操作二次确认（短信验证码）
5. 使用自定义请求头（X-Requested-With）

SameSite 三种模式：
- Strict — 完全禁止跨站 Cookie（最安全）
- Lax — 允许顶级导航的 GET 请求携带 Cookie（默认）
- None — 允许跨站（需要 Secure 属性）
```

---

### Q35: 什么是中间人攻击（MITM）？HTTPS 如何防御？

**难度**: ⭐⭐

**答案**: 中间人攻击是攻击者拦截客户端和服务器之间的通信。HTTPS 通过 TLS 加密 + 证书验证防御：客户端验证服务器证书的合法性，确保通信双方身份正确。

**解析**:
```
HTTPS 防御 MITM 的原理：
1. 服务器出示数字证书（CA 签发）
2. 客户端验证证书链（根证书 → 中间证书 → 服务器证书）
3. 客户端验证证书域名与访问域名一致
4. 使用证书中的公钥加密预主密钥
5. 只有持有私钥的服务器能解密

攻击者无法伪造证书（CA 不会为攻击者签发合法证书）
自签名证书不被浏览器信任
```

### Q36: 什么是点击劫持（Clickjacking）？如何防御？

**难度**: ⭐⭐

**答案**: 点击劫持是将恶意页面通过透明 iframe 覆盖在正常页面上，诱导用户点击。

**解析**:
```
防御措施：
1. X-Frame-Options 响应头
   X-Frame-Options: DENY（禁止被 iframe 嵌入）
   X-Frame-Options: SAMEORIGIN（只允许同源嵌入）

2. CSP frame-ancestors
   Content-Security-Policy: frame-ancestors 'self'

3. 前端防御 — 检查是否被 iframe 嵌入
   if (window.top !== window.self) {
     window.top.location = window.self.location;
   }
```

### Q37: Cookie 的 SameSite 属性有哪些值？

**难度**: ⭐⭐

**答案**: Strict（完全禁止跨站 Cookie）、Lax（允许顶级导航 GET 请求携带 Cookie，默认值）、None（允许跨站，需要 Secure 属性）。

**解析**:
```
SameSite=Strict：
- 所有跨站请求都不携带 Cookie
- 最安全，但用户体验差（从外部链接进入需要重新登录）

SameSite=Lax（默认）：
- 顶级导航 GET 请求携带 Cookie（如从搜索引擎点击进入）
- POST、iframe、AJAX 等不携带

SameSite=None：
- 允许所有跨站请求携带 Cookie
- 必须配合 Secure（HTTPS）
- 适用于第三方嵌入（如支付页面）
```

### Q38: 什么是 CORS？预检请求的流程？

**难度**: ⭐⭐

**答案**: CORS（跨源资源共享）通过 HTTP 头控制跨域访问。简单请求直接发送；非简单请求先发送 OPTIONS 预检请求。

**解析**:
```
简单请求条件（同时满足）：
- 方法：GET、HEAD、POST
- Content-Type：text/plain、multipart/form-data、application/x-www-form-urlencoded
- 只使用浏览器自动设置的请求头

非简单请求（触发预检）：
- 方法：PUT、DELETE、PATCH
- Content-Type：application/json
- 自定义请求头：Authorization、X-Custom-Header

预检请求流程：
1. 客户端发送 OPTIONS 请求
   OPTIONS /api/data HTTP/1.1
   Origin: https://example.com
   Access-Control-Request-Method: PUT
   Access-Control-Request-Headers: Authorization

2. 服务器返回允许信息
   Access-Control-Allow-Origin: https://example.com
   Access-Control-Allow-Methods: PUT
   Access-Control-Allow-Headers: Authorization
   Access-Control-Max-Age: 86400

3. 客户端发送实际请求
```

### Q39: 什么是 SQL 注入？如何防御？

**难度**: ⭐⭐

**答案**: SQL 注入是通过在用户输入中嵌入 SQL 代码来攻击数据库。防御：参数化查询（PreparedStatement）、输入验证、ORM、最小权限原则。

**解析**:
```javascript
// bad — 字符串拼接
const query = `SELECT * FROM users WHERE name = '${userInput}'`;
// 输入：' OR '1'='1' → SELECT * FROM users WHERE name = '' OR '1'='1'

// good — 参数化查询
const query = 'SELECT * FROM users WHERE name = ?';
db.query(query, [userInput]);

// good — ORM
const users = await User.findAll({ where: { name: userInput } });
```

### Q40: 前端安全最佳实践有哪些？

**难度**: ⭐⭐

**答案**: (1) HTTPS 全站加密；(2) CSP 策略；(3) HttpOnly + Secure + SameSite Cookie；(4) 输入验证和输出转义；(5) CSRF Token；(6) 敏感信息不存前端；(7) 定期依赖审计。

**解析**:
```
安全检查清单：
- [ ] 全站 HTTPS
- [ ] CSP 策略配置
- [ ] Cookie: HttpOnly + Secure + SameSite=Lax
- [ ] XSS 防护（转义 + CSP）
- [ ] CSRF 防护（Token + SameSite）
- [ ] 点击劫持防护（X-Frame-Options）
- [ ] 敏感数据不存 localStorage
- [ ] API 接口认证和限流
- [ ] 依赖安全审计（npm audit）
- [ ] X-Content-Type-Options: nosniff
```
