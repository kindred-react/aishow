## 一、首屏优化

### Q1: SSR、SSG、CSR、ISR 的区别和适用场景？

**难度**: ⭐⭐

**答案**: CSR 客户端渲染（首屏慢，交互好）；SSR 服务端渲染（首屏快，SEO 好）；SSG 静态生成（最快，CDN 分发）；ISR 增量静态生成（兼顾性能和动态更新）。

**解析**:
```
CSR（Client Side Rendering）：
  浏览器加载 HTML → 加载 JS → 渲染页面
  优点：交互好、前后端分离
  缺点：首屏慢、SEO 差
  适用：后台管理系统、SPA

SSR（Server Side Rendering）：
  服务器渲染 HTML → 浏览器显示 → 加载 JS → hydrate
  优点：首屏快、SEO 好
  缺点：服务器压力大
  适用：内容型网站、SEO 要求高

SSG（Static Site Generation）：
  构建时生成静态 HTML → CDN 分发
  优点：最快、最稳定
  缺点：不适合动态内容
  适用：博客、文档、营销页

ISR（Incremental Static Regeneration）：
  SSG + 增量更新（设置 revalidate 时间）
  优点：兼顾静态性能和动态更新
  适用：内容更新不频繁的页面
```

### Q4: 预加载（preload）和预获取（prefetch）的区别？

**难度**: ⭐⭐

**答案**: preload 高优先级预加载当前页面需要的资源；prefetch 低优先级预获取下一个页面可能需要的资源。

**解析**:
```html
<!-- preload — 高优先级，当前页面需要 -->
<link rel="preload" href="critical-font.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="hero-image.webp" as="image">

<!-- prefetch — 低优先级，下一个页面可能需要 -->
<link rel="prefetch" href="/next-page.js">

<!-- dns-prefetch — DNS 预解析 -->
<link rel="dns-prefetch" href="//cdn.example.com">

<!-- preconnect — TCP + TLS 预连接 -->
<link rel="preconnect" href="//api.example.com" crossorigin>
```

## 二、Core Web Vitals

### Q9: Core Web Vitals 包含哪些指标？

**难度**: ⭐⭐

**答案**: LCP（Largest Contentful Paint，最大内容元素渲染时间，<= 2.5s）、INP（Interaction to Next Paint，交互到下一次绘制的延迟，<= 200ms）、CLS（Cumulative Layout Shift，累积布局偏移，<= 0.1）。

**解析**:
```
LCP（Largest Contentful Paint）：
  - 定义：视口中最大的内容元素渲染完成的时间
  - 目标：<= 2.5s
  - 常见 LCP 元素：大图、标题文本、视频

INP（Interaction to Next Paint）：
  - 定义：用户交互到下一次绘制的时间
  - 目标：<= 200ms
  - 替代了 FID（First Input Delay）

CLS（Cumulative Layout Shift）：
  - 定义：页面生命周期内累积的布局偏移分数
  - 目标：<= 0.1
  - 计算：影响分数 = 影响距离 * 视口比例
```

### Q11: 如何优化 INP？

**答案**: (1) 减少主线程阻塞（长任务拆分）；(2) 使用 Web Worker；(3) 减少 JavaScript 体积；(4) 使用 debounce/throttle；(5) 优化事件处理器。

**解析**:

// 长任务拆分
function processLargeArray(items, callback) {
  const CHUNK_SIZE = 50;
  let index = 0;
  function processChunk() {
    const end = Math.min(index + CHUNK_SIZE, items.length);
    for (let i = index; i < end; i++) callback(items[i]);
    index = end;
    if (index < items.length) requestAnimationFrame(processChunk);
  }
  processChunk();
}

// 使用 requestIdleCallback 执行低优先级任务

  // 在浏览器空闲时执行
});

## 三、运行时优化

### Q15: 虚拟列表的实现原理？

**难度**: ⭐⭐⭐

**答案**: 只渲染可视区域内的元素，通过 scrollTop 计算起始索引，用占位元素撑开容器高度模拟完整列表。支持固定高度和动态高度。

**解析**:
```jsx
function VirtualList({ items, itemHeight = 50, containerHeight = 500 }) {
  const [scrollTop, setScrollTop] = useState(0);
  const totalHeight = items.length * itemHeight;
  const startIndex = Math.floor(scrollTop / itemHeight);
  const endIndex = Math.min(
    startIndex + Math.ceil(containerHeight / itemHeight) + 1,
    items.length
  );
  const visibleItems = items.slice(startIndex, endIndex);
  const offsetY = startIndex * itemHeight;

  return (
    <div style={{ height: containerHeight, overflow: 'auto' }}
         onScroll={e => setScrollTop(e.target.scrollTop)}>
      <div style={{ height: totalHeight, position: 'relative' }}>
        <div style={{ position: 'absolute', top: offsetY, width: '100%' }}>
          {visibleItems.map((item, i) => (
            <div key={startIndex + i} style={{ height: itemHeight }}>{item}</div>
          ))}
        </div>
      </div>
    </div>
  );
}
```

### Q16: React.memo / useMemo / useCallback 的使用原则？

**难度**: ⭐⭐

**答案**: React.memo 避免子组件不必要的重渲染；useMemo 缓存计算结果；useCallback 缓存函数引用。不要过度优化，只在确认有性能问题时使用。

**解析**:
```jsx
// 误区 1：所有组件都用 React.memo
// React.memo 本身有浅比较开销，简单组件不需要

// 误区 2：依赖数组不正确
const memoized = useMemo(() => compute(a, b), [a]); // 忘记 b

// 误区 3：在 useCallback 中创建闭包
const handleClick = useCallback(() => {
  console.log(count); // count 是旧值
}, []); // 应该依赖 [count]

// 正确原则：
// 1. 传递给 React.memo 子组件的 props 使用 useMemo/useCallback
// 2. 计算开销大的操作使用 useMemo
// 3. 优先考虑组件拆分和数据结构优化
```

### Q22: requestAnimationFrame 和 requestIdleCallback 的使用场景？

**难度**: ⭐⭐

**答案**: rAF 用于动画和视觉更新（下一帧前执行）；requestIdleCallback 用于低优先级后台任务（浏览器空闲时执行）。

**解析**:
```javascript
// rAF — 动画

  element.style.transform = `translateX(${x}px)`;
  requestAnimationFrame(animate);
}

// requestIdleCallback — 后台任务
requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0 && tasks.length) {
    performTask(tasks.pop());
  }
  if (tasks.length) requestIdleCallback(processTasks);
});
```

### Q26: 文件指纹（contenthash）的作用？

**难度**: ⭐⭐

**答案**: 文件指纹根据文件内容生成唯一的 hash 值作为文件名。内容变化时 hash 变化，文件名变化，浏览器重新请求；内容不变时文件名不变，命中长期缓存。

**解析**:
```javascript
// Webpack 配置
output: {
  filename: '[name].[contenthash:8].js',
  chunkFilename: '[name].[contenthash:8].js'
}

// hash vs chunkhash vs contenthash
// hash — 整个项目构建相关
// chunkhash — 每个 chunk 相关
// contenthash — 每个文件内容相关（推荐）
```

### Q30: 如何使用 Chrome DevTools 进行性能分析？

**难度**: ⭐⭐

**答案**: (1) Performance 面板 — 录制页面加载过程，分析长任务、布局偏移、网络请求；(2) Network 面板 — 分析请求瀑布图、资源大小；(3) Coverage 面板 — 分析 JS/CSS 使用率；(4) Memory 面板 — 分析内存泄漏。

**解析**:
```
Performance 面板分析步骤：
1. 打开 DevTools → Performance
2. 点击 Record，操作页面，停止录制
3. 分析 Main 线程：
   - 红色长条 = 长任务（>50ms）
   - 紫色 = 布局（Layout）
   - 绿色 = 绘制（Paint）
   - 蓝色 = 合成（Composite）
4. 查看 Bottom-Up / Call Tree / Event Log

Coverage 面板：
- 查看未使用的 JS/CSS 代码比例
- 帮助 Tree Shaking 和代码分割决策
```
