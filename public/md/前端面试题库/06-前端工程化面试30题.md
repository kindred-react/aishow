## 一、Webpack

### Q1: loader 和 plugin 的区别？

**难度**: ⭐⭐

**答案**: loader 是函数，用于转换文件内容（如 babel-loader 转译 JS）；plugin 是类，通过 Tapable 事件钩子扩展 Webpack 功能（如压缩、生成 HTML）。

**解析**:
```
loader：
- 本质是函数
- 在 module.rules 中配置
- 预处理文件（CSS → JS 对象，TS → JS）
- 执行顺序：从右到左，从下到上
- 遵循单一职责原则

plugin：
- 本质是类，原型上有 apply 方法
- 在 plugins 中配置
- 在打包过程的不同阶段执行
- 可以访问 compiler 和 compilation 对象
- 基于 Tapable 事件系统

常见 loader：babel-loader、css-loader、style-loader、file-loader（Webpack 5 已内置 Asset Modules 替代）、ts-loader
常见 plugin：HtmlWebpackPlugin、MiniCssExtractPlugin、TerserPlugin、DefinePlugin
```

### Q2: Webpack 的打包过程是怎样的？

**难度**: ⭐⭐

**答案**: 初始化参数 → 编译模块（从入口递归分析依赖）→ 确定模块图 → 打包（loader 处理 + plugin 增强）→ 输出文件。

**解析**:
```
1. 初始化：读取 webpack.config.js，合并默认配置
2. 编译：从 entry 开始，递归解析所有依赖模块
3. 每个模块被 loader 链处理（如 babel-loader 转译）
4. 生成模块图（Module Graph）
5. 根据依赖关系组装 Chunk
6. 每个 Chunk 生成对应的 bundle 文件
7. 输出到 output 目录

事件钩子：
before-run → run → before-compile → compile →
make → after-compile → emit → after-emit → done
```

### Q3: Tree Shaking 的原理是什么？

**难度**: ⭐⭐

**答案**: Tree Shaking 依赖 ES Module 的静态结构，在编译时分析模块的导入导出关系，移除未被使用的代码。需要 `sideEffects: false` 配合。

**解析**:
```javascript
// 可以被 Tree Shaking
export function used() { return 'used'; }
export function unused() { return 'unused'; } // 未被引用，会被删除

// 不能被 Tree Shaking 的情况
// 1. CommonJS（动态 require）
const module = require('./module');

// 2. 副作用代码
window.globalVar = 'hello'; // 副作用
Array.prototype.myMethod = function() {}; // 副作用

// 3. 默认导出对象
export default { method1() {}, method2() {} }; // 即使只用 method1，method2 也会被打包

// package.json
{ "sideEffects": false } // 告诉 Webpack 包没有副作用
```

### Q5: Webpack 热更新（HMR）的原理？

**难度**: ⭐⭐⭐

**答案**: Webpack Dev Server 通过 WebSocket 与浏览器保持连接。文件变化时，Webpack 重新编译变更的模块，生成新的 hash，通过 WebSocket 推送给浏览器，浏览器对比 hash，通过 JSONP 请求获取变更的模块代码，替换旧的模块。

**解析**:
```
HMR 流程：
1. Webpack 监听文件变化
2. 重新编译变更的模块
3. 生成新的 hash（manifest）
4. 通过 WebSocket 推送 hash 给浏览器
5. 浏览器对比 hash，发现变化
6. 通过 JSONP 请求获取变更的模块代码
7. 替换旧的模块（module.hot.accept）
8. 如果没有 accept 处理，则整页刷新

HMR 与 Live Reload 的区别：
- HMR：只替换变更的模块，保留应用状态
- Live Reload：整页刷新，丢失应用状态
```

### Q6: Webpack 5 模块联邦（Module Federation）的原理？

**难度**: ⭐⭐⭐

**答案**: 模块联邦允许不同应用之间共享模块，实现微前端架构。通过 `exposes` 暴露模块，`remotes` 引用远程模块，`shared` 共享依赖。

**解析**:
```javascript
// app1（远程应用）
new ModuleFederationPlugin({
  name: 'app1',
  filename: 'remoteEntry.js',
  exposes: {
    './Button': './src/components/Button',
  },
  shared: ['react', 'react-dom']
});

// app2（宿主应用）
new ModuleFederationPlugin({
  name: 'app2',
  remotes: {
    app1: 'app1@http://localhost:3001/remoteEntry.js',
  },
  shared: ['react', 'react-dom']
});

// app2 中使用 app1 的组件
import Button from 'app1/Button';
```

### Q7: Webpack 的 source map 有哪些类型？

**难度**: ⭐⭐

**答案**: 开发环境用 `eval-cheap-module-source-map`（快速），生产环境用 `hidden-source-map`（不暴露源码）或 `nosources-source-map`。

**解析**:
```
开发环境推荐：
eval-cheap-module-source-map
- 速度快（eval + cheap = 不生成列映射）
- 能看到源码

生产环境推荐：
hidden-source-map
- 生成 source map 但不在 bundle 中引用
- 适合上传到错误监控平台（如 Sentry）

nosources-source-map
- 有映射关系但不包含源码内容
- 安全性更好

source-map
- 完整的 source map（最慢但最详细）
```

### Q10: 如何编写自定义 Webpack plugin？

**难度**: ⭐⭐⭐

**答案**: Plugin 是一个类，原型上有 `apply` 方法，接收 `compiler` 对象，通过 `compiler.hooks` 监听事件。

**解析**:
```javascript
class MyPlugin {
  constructor(options) {
    this.options = options;
  }

  apply(compiler) {
    compiler.hooks.emit.tapAsync('MyPlugin', (compilation, callback) => {
      // 在输出资源到 output 目录前执行
      const assetName = 'my-asset.txt';
      const content = 'Hello from MyPlugin';
      compilation.assets[assetName] = {
        source: () => content,
        size: () => content.length
      };
      callback();
    });
  }
}

// 使用
plugins: [new MyPlugin({ option: 'value' })]
```

## 二、Vite

### Q11: Vite 的开发服务器为什么这么快？

**难度**: ⭐⭐

**答案**: Vite 利用浏览器原生 ESM 支持，开发模式下无需打包，按需编译请求的模块。启动时只需启动服务器，不需要打包所有模块。

**解析**:
```
Vite Dev Server 工作原理：
1. 浏览器请求 HTML
2. Vite 返回 HTML（含 <script type="module">）
3. 浏览器解析 import，发起 HTTP 请求
4. Vite 拦截请求，按需编译（TS/Vue/JSX → ESM）
5. 浏览器继续解析依赖，逐个请求

esbuild 预构建：
- 将 node_modules 中的依赖预打包为 ESM
- 合并多个小模块，减少 HTTP 请求数
- esbuild（Go 编写）比 Babel 快 10-100 倍

Webpack Dev Server：
- 启动时打包所有模块（即使未使用）
- 项目越大，启动越慢
```

**难度**: ⭐⭐

**答案**: Rollup 的 Tree Shaking 效果更好，对 ESM 的支持更完善。Vite 在 Rollup 基础上增加了代码分割和 CSS 处理。

```javascript
          'react-vendor': ['react', 'react-dom'],
          'utils': ['lodash-es', 'axios']
        }
      }
    },
    minify: 'esbuild', // 或 'terser'
    chunkSizeWarningLimit: 500
  }
};
```

### Q13: Vite 和 Webpack 的对比？

**难度**: ⭐⭐

**答案**: Vite 开发快（无需打包），生产用 Rollup；Webpack 成熟稳定，生态丰富。大型项目两者都适合。

**解析**:
| 维度 | Vite | Webpack |
|------|------|---------|
| 开发启动 | 毫秒级 | 秒级 |
| HMR | 快（单模块） | 较慢（可能重建 bundle） |
| 生产构建 | Rollup | 自身 |
| 生态 | 较新，增长快 | 成熟，插件丰富 |
| 配置 | 简单 | 复杂 |
| 兼容性 | 现代浏览器 | 兼容旧浏览器 |

## 三、ESM vs CJS

### Q16: ESM 和 CJS 的值传递机制有什么区别？

**难度**: ⭐⭐

**答案**: ESM 是值的实时引用（修改原模块会影响已导入的值），CJS 是值的拷贝（修改原模块不影响已导入的值）。

**解析**:
```javascript
// ESM — 值引用
// moduleA.mjs
export let count = 1;
export function increment() { count++; }
// moduleB.mjs
import { count, increment } from './moduleA.mjs';
console.log(count); // 1
increment();
console.log(count); // 2（实时反映变化）

// CJS — 值拷贝
// moduleA.js
let count = 1;
module.exports = { count };
// moduleB.js
const { count } = require('./moduleA');
count = 2; // 不影响 moduleA
```

---

### Q17: ESM 和 CJS 如何互相导入？

**难度**: ⭐⭐

**答案**: ESM 可以 `import` CJS（自动默认导入），CJS 不能 `require` ESM（需要用 `import()` 动态导入）。

**解析**:
```javascript
// ESM 导入 CJS（OK）
import fs from 'fs'; // 默认导入
import { readFileSync } from 'fs'; // 命名导入

// CJS 导入 ESM（需要动态导入）
// const { foo } = require('./module.mjs'); // 错误
const { foo } = await import('./module.mjs'); // OK（动态导入）

// package.json "type": "module" → .js 文件默认为 ESM
// .cjs 强制 CJS，.mjs 强制 ESM
```

---

### Q18: 什么是 Tree Shaking？为什么 CJS 不支持？

**难度**: ⭐⭐

**答案**: Tree Shaking 需要在编译时静态分析模块的导入导出关系。ESM 的 `import/export` 是静态的，编译时就能确定；CJS 的 `require/module.exports` 是动态的，运行时才能确定。

### Q19: package.json 中 type: module 的作用？

**难度**: ⭐

**答案**: 设置 `"type": "module"` 后，`.js` 文件默认按 ESM 解析。`.cjs` 强制 CJS，`.mjs` 强制 ESM。

**解析**:
```json
{
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.mjs",
  "exports": {
    ".": {
      "require": "./dist/index.cjs",
      "import": "./dist/index.mjs"
    }
  }
}
```

### Q21: qiankun 的 JS 沙箱是如何实现的？

**难度**: ⭐⭐⭐

**答案**: qiankun 使用 Proxy 沙箱（推荐）和快照沙箱（降级方案）。Proxy 沙箱为每个子应用创建 fakeWindow，拦截全局变量的读写。

**解析**:
```javascript
class ProxySandbox {
  constructor() {
    const fakeWindow = {};
    this.proxy = new Proxy(fakeWindow, {
      get(target, key) {
        if (key in target) return target[key];
        return window[key]; // 优先从 fakeWindow 获取
      },
      set(target, key, value) {
        target[key] = value; // 写入 fakeWindow
        return true;
      }
    });
  }
}
```

## 五、Babel/AST

### Q23: Babel 的编译过程是什么？

**难度**: ⭐⭐

**答案**: Parse（解析源码为 AST）→ Traverse（遍历 AST，应用插件转换）→ Generate（将 AST 生成目标代码）。

**解析**:
```
源代码 → Parser → AST → Transformer → 新 AST → Generator → 目标代码

AST 节点类型：
- VariableDeclaration（变量声明）
- FunctionDeclaration（函数声明）
- BinaryExpression（二元表达式）
- CallExpression（函数调用）
- MemberExpression（成员访问）

// 示例：const a = 1; 的 AST
{
  type: 'VariableDeclaration',
  kind: 'const',
  declarations: [{
    type: 'VariableDeclarator',
    id: { type: 'Identifier', name: 'a' },
    init: { type: 'NumericLiteral', value: 1 }
  }]
}
```

### Q24: 如何编写一个 Babel 插件？

**难度**: ⭐⭐⭐

**答案**: Babel 插件是一个函数，返回 `visitor` 对象，定义对特定 AST 节点的访问和处理逻辑。

**解析**:
```javascript
// babel-plugin-remove-console.js
module.exports = function({ types: t }) {
  return {
    name: 'remove-console',
    visitor: {
      CallExpression(path) {
        if (
          t.isMemberExpression(path.node.callee) &&
          path.node.callee.object.name === 'console'
        ) {
          path.parentPath.remove();
        }
      }
    }
  };
};

// 使用
// babel.config.js
module.exports = { plugins: ['./babel-plugin-remove-console'] };
```

### Q28: npm/pnpm/yarn 的区别？

**难度**: ⭐⭐

**答案**: npm/yarn v1 扁平化结构（幽灵依赖），pnpm 非扁平结构（严格隔离，符号链接），安装速度 pnpm 最快，磁盘占用最小。

**解析**:
```
幽灵依赖问题：
扁平化结构下，可以直接 require 未声明的依赖
pnpm 通过非扁平结构解决：
node_modules/
  .pnpm/lodash@4.17.21/node_modules/lodash/  → 真实文件
  lodash/  → 符号链接 → .pnpm/lodash@4.17.21/node_modules/lodash
  deep-equal/  → 不存在（未声明，无法访问）
```

### Q29: CI/CD 的基本流程是什么？

**难度**: ⭐⭐

**答案**: CI（持续集成）：代码提交 → 自动构建 → 自动测试 → 代码质量检查。CD（持续部署）：测试通过 → 自动部署到测试/生产环境。

**解析**:
```yaml
# GitHub Actions 示例
name: CI/CD
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with: { node-version: '18' }
      - run: pnpm install
      - run: pnpm build
      - run: pnpm test
      - run: pnpm lint
  deploy:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - run: pnpm deploy
```

### Q30: 如何实现前端项目的自动化测试？

**难度**: ⭐⭐

**答案**: 单元测试（Jest + Testing Library）、集成测试（Cypress/Playwright）、E2E 测试、视觉回归测试（Percy/Chromatic）。

**解析**:
```javascript
// Jest 单元测试
describe('Button', () => {
  it('should render with text', () => {
    render(<Button>Click me</Button>);
    expect(screen.getByText('Click me')).toBeInTheDocument();
  });
});

// Playwright E2E 测试
test('login flow', async ({ page }) => {
  await page.goto('/login');
  await page.fill('[name="email"]', 'test@example.com');
  await page.fill('[name="password"]', 'password');
  await page.click('button[type="submit"]');
  await expect(page).toHaveURL('/dashboard');
});
```
