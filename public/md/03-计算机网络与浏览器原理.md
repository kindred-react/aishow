## HTTP/HTTPS

### HTTP版本演进

#### HTTP/1.0 (1996)

**核心特点：**
- 每个请求都需要建立新的TCP连接
- 无状态协议
- 支持GET、POST、HEAD方法

**工作流程：**
```
客户端                    服务器
  |                         |
  |---建立TCP连接---------->|
  |---HTTP请求------------->|
  |<--HTTP响应-------------|
  |---关闭TCP连接---------->|
  |                         |
  |---建立新TCP连接-------->|
  |---HTTP请求------------->|
  |<--HTTP响应-------------|
  |---关闭TCP连接---------->|
```

**面试要点：**
- 每次请求都建立连接，性能开销大
- 无法复用连接，导致延迟高
- 无连接复用机制

#### HTTP/1.1 (1997)

**核心改进：**
- **持久连接**：Connection: keep-alive，默认开启
- **管道化**：可以同时发送多个请求
- **分块传输编码**：Transfer-Encoding: chunked
- **缓存控制**：Cache-Control、ETag
- **新增方法**：PUT、DELETE、OPTIONS、TRACE、CONNECT

**持久连接工作流程：**
```
客户端                    服务器
  |                         |
  |---建立TCP连接---------->|
  |---请求1---------------->|
  |<--响应1-----------------|
  |---请求2(同一连接)------>|
  |<--响应2-----------------|
  |---请求3(同一连接)------>|
  |<--响应3-----------------|
  |---关闭TCP连接---------->|
```

**管道化工作流程：**
```
客户端                    服务器
  |                         |
  |---建立TCP连接---------->|
  |---请求1---------------->|
  |---请求2(不等待响应)---->|
  |---请求3(不等待响应)---->|
  |<--响应1-----------------|
  |<--响应2-----------------|
  |<--响应3-----------------|
```

**面试要点：**
- 持久连接减少了TCP建立/关闭的开销
- 管道化有队头阻塞问题（Head-of-Line Blocking）
- 请求必须按顺序返回响应

#### HTTP/2 (2015)

**核心特性：**

1. **二进制分帧层**
   - HTTP/1.x是文本协议，HTTP/2是二进制协议
   - 所有传输的数据被分割为更小的消息和帧

2. **多路复用**
   - 单个TCP连接上可以同时发送多个请求
   - 解决了HTTP/1.1的队头阻塞问题

3. **头部压缩**
   - 使用HPACK算法压缩HTTP头部
   - 减少传输数据量

4. **服务器推送**
   - 服务器可以主动推送资源给客户端
   - 减少客户端请求次数

5. **请求优先级**
   - 可以为不同的请求设置优先级
   - 重要资源优先加载

**多路复用工作流程：**
```
客户端                    服务器
  |                         |
  |---建立TCP连接---------->|
  |                         |
  |---Stream1: 请求1------->|
  |---Stream2: 请求2------->| (同时发送)
  |---Stream3: 请求3------->|
  |                         |
  |<--Stream2: 响应2-------| (乱序返回)
  |<--Stream1: 响应1-------|
  |<--Stream3: 响应3-------|
```

**二进制帧结构：**
```
+-----------------------------------------------+
|                 Length (24 bits)             |
+-----------+-----+-----------------------------+
|   Type    | F  |        Stream Identifier     |
| (8 bits)  | L  |          (31 bits)          |
+-----------+-----+-----------------------------+
|                   Frame Payload               |
+-----------------------------------------------+
```

**面试要点：**
- HTTP/2解决了HTTP/1.1的队头阻塞问题
- 多路复用提高了连接利用率
- 但仍有TCP层面的队头阻塞

#### HTTP/3 (2022)

**核心特性：**

1. **基于QUIC协议**
   - QUIC基于UDP，而不是TCP
   - 解决了TCP层面的队头阻塞问题

2. **连接迁移**
   - 网络切换时（如WiFi到4G）连接不断开
   - 基于连接ID而非IP地址

3. **0-RTT连接建立**
   - 首次连接需要1-RTT
   - 后续连接可以实现0-RTT

4. **内置拥塞控制**
   - 不依赖TCP的拥塞控制
   - 更灵活的拥塞控制算法

**HTTP/3连接建立：**
```
客户端                    服务器
  |                         |
  |---Client Hello---------->| (包含之前连接信息)
  |<--Server Hello----------|
  |<--加密握手--------------|
  |---0-RTT数据------------->| (可以立即发送数据)
```

**HTTP/2 vs HTTP/3对比：**

| 特性 | HTTP/2 | HTTP/3 |
|------|--------|--------|
| 传输层 | TCP | QUIC (UDP) |
| 队头阻塞 | TCP层面存在 | 无 |
| 连接迁移 | 不支持 | 支持 |
| 握手延迟 | 1-RTT | 0-RTT (后续连接) |
| 拥塞控制 | TCP内置 | QUIC内置 |

**面试要点：**
- HTTP/3解决了HTTP/2的TCP队头阻塞问题
- QUIC协议结合了TCP的可靠性和UDP的速度
- 连接迁移对移动端体验提升明显

### HTTP方法

| 方法 | 描述 | 幂等性 | 安全性 | 请求体 |
|------|------|--------|--------|--------|
| GET | 获取资源 | 是 | 是 | 否 |
| POST | 创建资源 | 否 | 否 | 是 |
| PUT | 更新资源（整体） | 是 | 否 | 是 |
| PATCH | 更新资源（部分） | 否 | 否 | 是 |
| DELETE | 删除资源 | 是 | 否 | 否 |
| HEAD | 获取响应头 | 是 | 是 | 否 |
| OPTIONS | 获取支持的方法 | 是 | 是 | 否 |
| CONNECT | 建立隧道 | 否 | 否 | 是 |
| TRACE | 回显请求 | 是 | 是 | 否 |

**幂等性定义：**
- 相同的请求执行多次，结果与执行一次相同
- GET、PUT、DELETE、HEAD、OPTIONS、TRACE是幂等的
- POST、PATCH不是幂等的

**安全性定义：**
- 不修改服务器状态
- GET、HEAD、OPTIONS、TRACE是安全的

**面试要点：**
- GET和POST的区别不仅是参数位置，还有语义和缓存策略
- 幂等性是RESTful API设计的重要概念
- PUT和PATCH的区别在于更新方式

### HTTP状态码

#### 2xx 成功

| 状态码 | 描述 | 使用场景 |
|--------|------|----------|
| 200 | OK | 请求成功 |
| 201 | Created | 资源创建成功 |
| 202 | Accepted | 请求已接受，异步处理中 |
| 204 | No Content | 请求成功，无返回内容 |

#### 3xx 重定向

| 状态码 | 描述 | 使用场景 |
|--------|------|----------|
| 301 | Moved Permanently | 永久重定向 |
| 302 | Found | 临时重定向 |
| 303 | See Other | 重定向到GET请求 |
| 304 | Not Modified | 资源未修改，使用缓存 |
| 307 | Temporary Redirect | 临时重定向，保持方法 |
| 308 | Permanent Redirect | 永久重定向，保持方法 |

**301 vs 302：**
```
301 永久重定向：
  - SEO：搜索引擎会更新索引
  - 缓存：浏览器会缓存重定向
  - 示例：http → https

302 临时重定向：
  - SEO：搜索引擎保留原索引
  - 缓存：浏览器不缓存重定向
  - 示例：登录后跳转
```

#### 4xx 客户端错误

| 状态码 | 描述 | 使用场景 |
|--------|------|----------|
| 400 | Bad Request | 请求参数错误 |
| 401 | Unauthorized | 未认证 |
| 403 | Forbidden | 无权限 |
| 404 | Not Found | 资源不存在 |
| 405 | Method Not Allowed | 方法不支持 |
| 409 | Conflict | 资源冲突 |
| 413 | Payload Too Large | 请求体过大 |
| 429 | Too Many Requests | 请求过多（限流） |

**401 vs 403：**
```
401 Unauthorized：
  - 身份未验证
  - 需要登录或提供认证信息
  - 响应头：WWW-Authenticate

403 Forbidden：
  - 身份已验证，但无权限
  - 已登录，但资源访问被拒绝
```

#### 5xx 服务器错误

| 状态码 | 描述 | 使用场景 |
|--------|------|----------|
| 500 | Internal Server Error | 服务器内部错误 |
| 502 | Bad Gateway | 网关错误 |
| 503 | Service Unavailable | 服务不可用 |
| 504 | Gateway Timeout | 网关超时 |

**面试要点：**
- 301和302的区别影响SEO和缓存
- 401和403的区别在于认证状态
- 502和504的区别在于错误类型

### HTTPS握手过程

#### HTTPS概述

HTTPS = HTTP + SSL/TLS

**SSL/TLS版本：**
- SSL 1.0、2.0、3.0（已废弃）
- TLS 1.0、1.1（已废弃）
- TLS 1.2（主流）
- TLS 1.3（最新，推荐）

#### TLS 1.2握手过程

**完整握手流程：**
```
客户端                                        服务器
  |                                            |
  |---Client Hello---------------------------->|
  |   - 支持的TLS版本                           |
  |   - 支持的加密套件                          |
  |   - 随机数1 (Client Random)                |
  |                                            |
  |<--Server Hello-----------------------------|
  |   - 选择的TLS版本                           |
  |   - 选择的加密套件                          |
  |   - 随机数2 (Server Random)                |
  |                                            |
  |<--Server Certificate-----------------------|
  |   - 服务器证书（公钥）                      |
  |                                            |
  |<--Server Key Exchange (可选)---------------|
  |   - 服务器密钥交换参数                      |
  |                                            |
  |<--Certificate Request (可选)---------------|
  |   - 请求客户端证书                          |
  |                                            |
  |<--Server Hello Done------------------------|
  |                                            |
  |---Certificate Verify (可选)--------------->|
  |   - 客户端证书                              |
  |                                            |
  |---Client Key Exchange--------------------->|
  |   - 预主密钥（用服务器公钥加密）            |
  |                                            |
  |---Change Cipher Spec---------------------->|
  |   - 通知后续使用加密通信                    |
  |                                            |
  |---Finished-------------------------------->|
  |   - 握手完成消息（加密）                    |
  |                                            |
  |<--Change Cipher Spec-----------------------|
  |   - 通知后续使用加密通信                    |
  |                                            |
  |<--Finished---------------------------------|
  |   - 握手完成消息（加密）                    |
  |                                            |
  |========= 加密通信开始 ==========|
```

**密钥生成过程：**
```
1. 客户端生成随机数1 (Client Random)
2. 服务器生成随机数2 (Server Random)
3. 客户端生成预主密钥 (Pre-Master Secret)
4. 用服务器公钥加密预主密钥，发送给服务器
5. 服务器用私钥解密得到预主密钥
6. 双方使用相同的算法生成主密钥：
   Master Secret = PRF(Pre-Master Secret,
                        "master secret",
                        Client Random + Server Random)
7. 使用主密钥生成会话密钥：
   - 对称加密密钥
   - MAC密钥
   - IV (初始化向量)
```

#### TLS 1.3握手过程

**TLS 1.3优化：**
- 减少握手RTT（从2-RTT到1-RTT）
- 移除不安全的加密算法
- 支持0-RTT恢复

**1-RTT握手流程：**
```
客户端                                        服务器
  |                                            |
  |---Client Hello---------------------------->|
  |   - 支持的TLS版本                           |
  |   - 支持的加密套件                          |
  |   - 随机数1 (Client Random)                |
  |   - 密钥共享参数 (Key Share)                |
  |                                            |
  |<--Server Hello-----------------------------|
  |   - 选择的TLS版本                           |
  |   - 选择的加密套件                          |
  |   - 随机数2 (Server Random)                |
  |   - 密钥共享参数 (Key Share)                |
  |                                            |
  |<--Encrypted Extensions---------------------|
  |   - 加密的扩展信息                          |
  |                                            |
  |<--Certificate-----------------------------|
  |   - 服务器证书                              |
  |                                            |
  |<--Certificate Verify----------------------|
  |   - 证书验证                                |
  |                                            |
  |<--Finished---------------------------------|
  |   - 握手完成消息（加密）                    |
  |                                            |
  |---Finished-------------------------------->|
  |   - 握手完成消息（加密）                    |
  |                                            |
  |========= 加密通信开始 ==========|
```

**0-RTT恢复流程：**
```
客户端                                        服务器
  |                                            |
  |---Client Hello + Early Data--------------->|
  |   - 包含之前的会话信息                      |
  |   - 可以立即发送应用数据                    |
  |                                            |
  |<--Server Hello + Finished-----------------|
  |                                            |
  |========= 加密通信继续 ==========|
```

#### HTTPS面试要点

**Q1: HTTPS和HTTP的区别？**

A:
1. **安全性**：HTTPS加密传输，HTTP明文传输
2. **协议**：HTTPS = HTTP + SSL/TLS
3. **端口**：HTTPS 443，HTTP 80
4. **性能**：HTTPS有握手开销，HTTP更快
5. **证书**：HTTPS需要SSL证书，HTTP不需要
6. **SEO**：HTTPS搜索引擎权重更高

**Q2: TLS握手过程是怎样的？**

A: 见上面TLS 1.2和TLS 1.3握手流程图。

**Q3: 中间人攻击如何防御？**

A:
1. **证书验证**：客户端验证服务器证书
   - 证书是否由可信CA签发
   - 证书是否过期
   - 证书域名是否匹配
2. **证书固定**：客户端内置服务器证书
3. **HSTS**：强制使用HTTPS
4. **公钥固定**：HPKP（已废弃）

**Q4: HTTPS性能如何优化？**

A:
1. **会话复用**：Session ID / Session Ticket
2. **TLS 1.3**：减少握手RTT
3. **OCSP Stapling**：服务器提供证书状态
4. **False Start**：提前发送应用数据
5. **连接复用**：HTTP/2多路复用
6. **证书优化**：使用ECDSA证书，证书链优化

### HTTP缓存机制

#### 缓存类型

**强缓存**
- 不向服务器发送请求
- 直接从缓存读取
- 由Cache-Control和Expires控制

**协商缓存**
- 向服务器发送请求
- 服务器验证资源是否修改
- 由ETag和Last-Modified控制

#### 强缓存

**Cache-Control**

| 指令 | 描述 | 示例 |
|------|------|------|
| max-age | 缓存最大时间（秒） | Cache-Control: max-age=3600 |
| no-cache | 不使用强缓存，使用协商缓存 | Cache-Control: no-cache |
| no-store | 不缓存 | Cache-Control: no-store |
| public | 可以被任何缓存存储 | Cache-Control: public |
| private | 只能被浏览器缓存 | Cache-Control: private |
| must-revalidate | 过期后必须验证 | Cache-Control: must-revalidate |
| immutable | 资源不会改变 | Cache-Control: immutable |

**Expires**
```
Expires: Wed, 09 Apr 2027 00:00:00 GMT
```
- HTTP/1.0字段
- 使用绝对时间
- 优先级低于Cache-Control

**强缓存流程：**
```
客户端                    服务器
  |                         |
  |---请求资源------------->|
  |<--响应(带Cache-Control)-|
  |                         |
  |---再次请求资源--------->|
  | (检查max-age未过期)      |
  | (直接使用缓存)           |
  |                         |
```

#### 协商缓存

**ETag**
```
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
```
- 资源的唯一标识
- 优先级高于Last-Modified

**Last-Modified**
```
Last-Modified: Wed, 09 Apr 2026 00:00:00 GMT
```
- 资源最后修改时间
- 精度到秒

**协商缓存流程：**
```
客户端                    服务器
  |                         |
  |---请求资源------------->|
  |<--响应(带ETag)----------|
  |                         |
  |---再次请求资源--------->|
  | (带If-None-Match)       |
  |                         |
  |---验证ETag------------->|
  | (资源未修改)             |
  |<--304 Not Modified------|
  | (使用缓存)               |
```

**ETag vs Last-Modified：**

| 特性 | ETag | Last-Modified |
|------|------|---------------|
| 精度 | 高（文件哈希） | 低（秒级） |
| 优先级 | 高 | 低 |
| 适用场景 | 精确验证 | 简单验证 |

#### 缓存策略

**HTML文件**
```
Cache-Control: no-cache
ETag: "xxx"
```
- 使用协商缓存
- 确保获取最新版本

**CSS/JS文件（带hash）**
```
Cache-Control: max-age=31536000, immutable
```
- 使用强缓存
- 文件名变化自动失效

**图片/字体**
```
Cache-Control: max-age=86400
```
- 使用强缓存
- 较长的缓存时间

**API响应**
```
Cache-Control: no-cache, no-store
```
- 不缓存
- 确保数据实时性

#### 缓存面试要点

**Q1: 强缓存和协商缓存的区别？**

A:
1. **强缓存**：不发送请求，直接使用缓存
2. **协商缓存**：发送请求，验证后使用缓存
3. **优先级**：先检查强缓存，再检查协商缓存

**Q2: Cache-Control的max-age=0和no-cache的区别？**

A:
- **max-age=0**：资源立即过期，但可以使用协商缓存
- **no-cache**：不使用强缓存，必须使用协商缓存
- 实际效果相同，但语义不同

**Q3: 如何实现前端资源的长期缓存？**

A:
1. 文件名加hash：`app.abc123.js`
2. 设置长期缓存：`Cache-Control: max-age=31536000, immutable`
3. HTML文件使用协商缓存
4. 修改文件内容后，hash变化，自动失效

**Q4: ETag和Last-Modified如何选择？**

A:
1. **ETag优先**：精度高，优先使用
2. **Last-Modified备用**：兼容性好
3. **同时使用**：服务器优先验证ETag

## TCP/UDP

### 三次握手

#### 握手过程

```
客户端                    服务器
  |                         |
  |---SYN, seq=x----------->| (x为随机数)
  |   SYN_SENT              |
  |                         |
  |<--SYN+ACK, seq=y, ack=x+1|
  |   SYN_RCVD              |
  |                         |
  |---ACK, seq=x+1, ack=y+1>|
  |   ESTABLISHED          |
  |   ESTABLISHED          |
  |                         |
  |========= 数据传输 ==========|
```

#### 状态转换

```
客户端                          服务器
  |                               |
CLOSED                          CLOSED
  |                               |
  |---SYN----------------------->|
  |                               |
SYN_SENT                        LISTEN
  |                               |
  |<--SYN+ACK---------------------|
  |                               |
  |                               SYN_RCVD
  |                               |
  |---ACK------------------------>|
  |                               |
ESTABLISHED                     ESTABLISHED
```

#### 为什么需要三次握手？

1. **防止已失效的连接请求**
   - 客户端发送的SYN在网络中滞留
   - 客户端重发SYN，建立连接
   - 滞留的SYN到达服务器
   - 三次握手可以识别这种情况

2. **同步初始序列号**
   - 双方都需要确认对方的初始序列号
   - 确保数据传输的可靠性

3. **确认双方的收发能力**
   - 第一次：客户端发送能力
   - 第二次：服务器接收和发送能力
   - 第三次：客户端接收能力

#### 面试题

**Q1: 为什么不是两次握手？**

A:
1. 无法确认客户端的接收能力
2. 无法防止已失效的连接请求
3. 无法同步双方的初始序列号

**Q2: 三次握手可以携带数据吗？**

A:
1. **第一次握手**：不能携带数据
2. **第二次握手**：可以携带数据
3. **第三次握手**：可以携带数据
4. 原因：防止SYN Flood攻击

**Q3: SYN Flood攻击如何防御？**

A:
1. **SYN Cookie**：不分配资源，计算cookie
2. **缩短SYN Timeout**：快速释放资源
3. **SYN Cache**：缓存SYN请求
4. **防火墙**：过滤异常SYN

### 四次挥手

#### 挥手过程

```
客户端                    服务器
  |                         |
  |---FIN, seq=u----------->| (主动关闭)
  |   FIN-WAIT-1           |
  |                         |
  |<--ACK, seq=v, ack=u+1   |
  |   FIN-WAIT-2           | CLOSE-WAIT
  |                         |
  |<--FIN, seq=w------------| (被动关闭)
  |   TIME-WAIT            | LAST-ACK
  |                         |
  |---ACK, seq=u+1, ack=w+1>|
  |   CLOSED               | CLOSED
  |                         |
  |========= 连接关闭 ==========|
```

#### 状态转换

```
客户端                          服务器
  |                               |
ESTABLISHED                     ESTABLISHED
  |                               |
  |---FIN------------------------>|
  |                               |
FIN-WAIT-1                      CLOSE-WAIT
  |                               |
  |<--ACK-------------------------|
  |                               |
FIN-WAIT-2                      LAST-ACK
  |                               |
  |<--FIN-------------------------|
  |                               |
TIME-WAIT                       CLOSED
  |                               |
  |---ACK------------------------>|
  |                               |
CLOSED
```

#### 为什么需要四次挥手？

1. **TCP是全双工的**
   - 每个方向都需要单独关闭
   - 主动关闭方发送FIN
   - 被动关闭方发送ACK
   - 被动关闭方发送FIN
   - 主动关闭方发送ACK

2. **确保数据传输完成**
   - 被动关闭方可能还有数据要发送
   - 需要等待数据传输完成

#### TIME_WAIT状态

**作用：**
1. **确保最后一个ACK到达**
   - 如果ACK丢失，服务器会重发FIN
   - 客户端需要重发ACK

2. **等待滞留的报文消失**
   - 确保旧连接的报文不会影响新连接

**持续时间：**
- 2MSL（Maximum Segment Lifetime）
- 通常为1-4分钟

**问题：**
- 大量TIME_WAIT会占用资源
- 可能导致端口耗尽

**解决方案：**
1. **调整内核参数**：减少TIME_WAIT时间
2. **SO_REUSEADDR**：端口复用
3. **负载均衡**：分散连接

#### 面试题

**Q1: 为什么需要TIME_WAIT状态？**

A:
1. 确保最后一个ACK到达服务器
2. 等待滞留的报文消失
3. 防止旧连接的报文影响新连接

**Q2: 大量TIME_WAIT如何解决？**

A:
1. 调整内核参数：`net.ipv4.tcp_tw_reuse`
2. 使用SO_REUSEADDR
3. 优化连接池
4. 负载均衡

**Q3: CLOSE_WAIT状态过多是什么原因？**

A:
1. 服务器没有关闭连接
2. 代码逻辑问题：没有调用close()
3. 资源泄漏：连接未正确释放

### 拥塞控制

#### 拥塞控制算法

**1. 慢启动（Slow Start）**
- 初始窗口：1 MSS
- 每收到一个ACK，窗口加倍
- 指数增长：1, 2, 4, 8, 16...
- 达到ssthresh后，进入拥塞避免

```
窗口大小
  ^
  |      /\
  |     /  \
  |    /    \__________
  |   /                \
  |  /                  \
  | /                    \
  |/                      \
  +------------------------> 时间
    慢启动  拥塞避免
```

**2. 拥塞避免（Congestion Avoidance）**
- 每个RTT，窗口增加1 MSS
- 线性增长
- 检测到拥塞后，窗口减半

**3. 快重传（Fast Retransmit）**
- 收到3个重复ACK
- 立即重传丢失的报文
- 不等待超时

**4. 快恢复（Fast Recovery）**
- 收到3个重复ACK
- 窗口减半
- 进入快恢复状态
- 收到新的ACK后，退出快恢复

#### 拥塞控制流程

```
窗口大小
  ^
  |      /\        /\
  |     /  \      /  \
  |    /    \    /    \__________
  |   /      \  /              \
  |  /        \/                \
  | /                           /
  |/                           /
  +----------------------------> 时间
    慢启动  拥塞避免  快恢复
```

#### 面试题

**Q1: 拥塞控制和流量控制的区别？**

A:
1. **拥塞控制**：防止网络拥塞
   - 考虑整个网络的负载
   - 算法：慢启动、拥塞避免、快重传、快恢复

2. **流量控制**：防止接收方溢出
   - 考虑接收方的处理能力
   - 机制：滑动窗口

**Q2: 快重传和快恢复的作用？**

A:
1. **快重传**：快速检测和重传丢失的报文
2. **快恢复**：快速恢复拥塞窗口
3. 避免等待超时，提高性能

### 可靠传输机制

#### 1. 序列号和确认号

**序列号（Sequence Number）**
- 标识报文的顺序
- 确保数据有序到达

**确认号（Acknowledgment Number）**
- 确认已收到的数据
- 期望收到的下一个序列号

```
客户端                    服务器
  |                         |
  |---seq=1, data="A"------>|
  |                         |
  |<--ack=2-----------------|
  |   (确认收到seq=1)        |
  |                         |
  |---seq=2, data="B"------>|
  |                         |
  |<--ack=3-----------------|
  |   (确认收到seq=2)        |
```

#### 2. 重传机制

**超时重传**
- 设置RTO（Retransmission Timeout）
- 超时后重传
- RTO动态调整

**快速重传**
- 收到3个重复ACK
- 立即重传
- 不等待超时

```
客户端                    服务器
  |                         |
  |---seq=1---------------->|
  |                         |
  |---seq=2---------------->| (丢失)
  |                         |
  |---seq=3---------------->|
  |<--ack=2-----------------|
  |   (重传seq=2)           |
  |                         |
  |---seq=2---------------->| (重传)
  |                         |
  |<--ack=4-----------------|
```

#### 3. 滑动窗口

**作用：**
- 流量控制
- 提高传输效率

**工作原理：**
```
发送窗口：
  [已发送已确认] [已发送未确认] [未发送可发送] [不可发送]
  |<-----------窗口大小---------->|

接收窗口：
  [已接收] [未接收期望] [未接收不期望]
  |<---窗口大小--->|
```

**Q1: TCP如何保证可靠性？**

A:
1. **序列号**：确保数据有序
2. **确认号**：确认数据到达
3. **重传机制**：超时重传、快速重传
4. **滑动窗口**：流量控制
5. **校验和**：检测数据错误

**Q2: 滑动窗口的作用？**

A:
1. **流量控制**：防止接收方溢出

## DNS解析

### DNS概述

**DNS（Domain Name System）**
- 域名系统
- 将域名解析为IP地址
- 分布式数据库

**域名结构：**
```
www.example.com.
|    |       |
主机 三级域名 顶级域名
```

### DNS查询类型

#### 递归查询

**定义：**
- 客户端向DNS服务器发起查询
- DNS服务器负责完成整个查询过程
- 返回最终结果

**流程：**
```
客户端                    DNS服务器
  |                         |
  |---查询www.example.com-->|
  |                         |
  |<--返回IP地址------------|
  |   (DNS服务器完成所有查询) |
```

#### 迭代查询

**定义：**
- 客户端向DNS服务器发起查询
- DNS服务器返回下一级DNS服务器地址
- 客户端继续向下一级DNS服务器查询

**流程：**
```
客户端                    DNS服务器
  |                         |
  |---查询www.example.com-->|
  |                         |
  |<--返回根DNS服务器--------|
  |                         |
  |---查询www.example.com-->| (根DNS)
  |                         |
  |<--返回.com DNS服务器-----|
  |                         |
  |---查询www.example.com-->| (.com DNS)
  |                         |
  |<--返回example.com DNS---|
  |                         |
  |---查询www.example.com-->| (example.com DNS)
  |                         |
  |<--返回IP地址------------|
```

#### 递归查询 vs 迭代查询

| 特性 | 递归查询 | 迭代查询 |
|------|----------|----------|
| 客户端负担 | 小 | 大 |
| 服务器负担 | 大 | 小 |
| 查询次数 | 1次 | 多次 |
| 实际使用 | 常用 | 较少 |

### DNS解析流程

**完整解析流程：**
```
客户端                    本地DNS
  |                         |
  |---查询www.example.com-->|
  |                         |
  |<--返回IP地址------------|
  |   (缓存命中)             |
  |                         |
  |                         |
  |---查询www.example.com-->|
  |                         |
  |---查询根DNS------------->|
  |<--返回.com DNS----------|
  |                         |
  |---查询.com DNS---------->|
  |<--返回example.com DNS----|
  |                         |
  |---查询example.com DNS--->|
  |<--返回IP地址------------|
  |                         |
  |<--返回IP地址------------|
```

### DNS缓存

**缓存层次：**
1. **浏览器缓存**
   - Chrome、Firefox等
   - 缓存时间：1-5分钟

2. **操作系统缓存**
   - /etc/hosts
   - DNS缓存服务

3. **本地DNS缓存**
   - ISP提供的DNS服务器
   - 缓存时间：根据TTL

4. **权威DNS缓存**
   - 域名的权威DNS服务器
   - 缓存时间：根据TTL

**TTL（Time To Live）**
- 缓存生存时间
- 单位：秒
- 典型值：300-86400秒

### CDN原理

**CDN（Content Delivery Network）**
- 内容分发网络
- 将内容缓存到边缘节点
- 就近访问，提高速度

**CDN工作流程：**
```
用户                    CDN边缘节点                  源站
  |                         |                         |
  |---请求资源-------------->|                         |
  |                         |---请求资源-------------->|
  |                         |<--返回资源---------------|
  |<--返回资源(缓存)---------|                         |
  |                         |                         |
  |---请求资源-------------->| (缓存命中)              |
  |<--返回资源(缓存)---------|                         |
```

**CDN关键技术：**
1. **DNS解析**
   - 根据用户IP返回最近的CDN节点

2. **缓存策略**
   - 静态资源缓存
   - 动态资源回源

3. **负载均衡**
   - 分发请求到不同节点

### DNS面试要点

**Q1: DNS解析的完整流程？**

A: 见上面DNS解析流程图。

**Q2: 递归查询和迭代查询的区别？**

A:
1. **递归查询**：服务器完成所有查询，返回最终结果
2. **迭代查询**：服务器返回下一级服务器，客户端继续查询

**Q3: DNS缓存如何工作？**

A:
1. 浏览器缓存 → 操作系统缓存 → 本地DNS缓存
2. 根据TTL决定缓存时间
3. 缓存命中则直接返回

**Q4: CDN的原理？**

A:
1. 将内容缓存到边缘节点
2. DNS解析返回最近的CDN节点
3. 就近访问，提高速度
4. 减轻源站压力

## 跨域与安全

### 同源策略

**同源定义：**
- 协议相同
- 域名相同
- 端口相同

**示例：**
```
同源：
- http://www.example.com/page1.html
- http://www.example.com/page2.html

不同源：
- http://www.example.com
- https://www.example.com (协议不同)
- http://sub.example.com (子域名不同)
- http://www.example.com:8080 (端口不同)
```

**同源策略限制：**
1. Cookie、LocalStorage、IndexedDB
2. DOM访问
3. AJAX请求

**不受同源策略限制：**
1. 跨域资源嵌入
   - `<script src="...">`
   - `<img src="...">`
   - `<link href="...">`
   - `<iframe src="...">`

2. 表单提交
   - `<form action="...">`

3. 链接跳转
   - `<a href="...">`

### CORS详解

**CORS（Cross-Origin Resource Sharing）**
- 跨域资源共享
- W3C标准
- 通过HTTP头部控制

#### 简单请求

**条件：**
1. 方法：GET、POST、HEAD
2. 头部：Accept、Accept-Language、Content-Language、Content-Type
3. Content-Type：application/x-www-form-urlencoded、multipart/form-data、text/plain

**流程：**
```
客户端                    服务器
  |                         |
  |---GET请求-------------->|
  |   Origin: http://a.com   |
  |                         |
  |<--响应------------------|
  |   Access-Control-Allow-Origin: http://a.com
```

#### 预检请求

**条件：**
1. 方法：PUT、DELETE、PATCH
2. 头部：自定义头部
3. Content-Type：application/json

**流程：**
```
客户端                    服务器
  |                         |
  |---OPTIONS请求---------->|
  |   Origin: http://a.com   |
  |   Access-Control-Request-Method: PUT
  |   Access-Control-Request-Headers: Content-Type
  |                         |
  |<--响应------------------|
  |   Access-Control-Allow-Origin: http://a.com
  |   Access-Control-Allow-Methods: PUT, GET, POST
  |   Access-Control-Allow-Headers: Content-Type
  |   Access-Control-Max-Age: 86400
  |                         |
  |---PUT请求-------------->|
  |   Origin: http://a.com   |
  |                         |
  |<--响应------------------|
  |   Access-Control-Allow-Origin: http://a.com
```

#### CORS头部

| 头部 | 描述 | 示例 |
|------|------|------|
| Access-Control-Allow-Origin | 允许的源 | Access-Control-Allow-Origin: * |
| Access-Control-Allow-Methods | 允许的方法 | Access-Control-Allow-Methods: GET, POST |
| Access-Control-Allow-Headers | 允许的头部 | Access-Control-Allow-Headers: Content-Type |
| Access-Control-Allow-Credentials | 允许携带Cookie | Access-Control-Allow-Credentials: true |
| Access-Control-Max-Age | 预检缓存时间 | Access-Control-Max-Age: 86400 |
| Access-Control-Expose-Headers | 允许客户端访问的头部 | Access-Control-Expose-Headers: X-Custom-Header |

#### CORS面试要点

**Q1: 什么是同源策略？**

A:
1. 同源：协议、域名、端口相同
2. 限制：Cookie、DOM、AJAX
3. 目的：安全性

**Q2: CORS的预检请求？**

A:
1. 触发条件：PUT、DELETE、自定义头部等
2. 流程：OPTIONS请求 → 服务器响应 → 实际请求
3. 目的：验证服务器是否允许跨域

**Q3: CORS如何携带Cookie？**

A:
1. 前端：`withCredentials: true`
2. 后端：`Access-Control-Allow-Credentials: true`
3. 后端：`Access-Control-Allow-Origin`不能为`*`

### JSONP

**JSONP（JSON with Padding）**
- 老旧浏览器支持
- 只支持GET请求
- 利用`<script>`标签的跨域特性

**实现原理：**
```javascript
// 前端
function handleResponse(data) {
  console.log(data);
}

const script = document.createElement('script');
script.src = 'http://api.example.com/data?callback=handleResponse';
document.body.appendChild(script);
```

```javascript
// 后端
app.get('/data', (req, res) => {
  const callback = req.query.callback;
  const data = { name: 'John' };
  res.send(`${callback}(${JSON.stringify(data)})`);
});
```

**JSONP vs CORS：**

| 特性 | JSONP | CORS |
|------|-------|------|
| 方法 | 只支持GET | 支持所有方法 |
| 浏览器支持 | 所有浏览器 | 现代浏览器 |
| 安全性 | 较低 | 较高 |
| 错误处理 | 困难 | 容易 |
| 标准化 | 非标准 | W3C标准 |

### CSRF防御

**CSRF（Cross-Site Request Forgery）**
- 跨站请求伪造
- 利用用户的登录状态
- 在用户不知情的情况下发送请求

**攻击流程：**
```
用户                    恶意网站                    目标网站
  |                         |                         |
  |---登录目标网站---------->|                         |
  |<--获取Cookie------------|                         |
  |                         |                         |
  |---访问恶意网站---------->|                         |
  |                         |---发送请求-------------->|
  |                         | (携带用户的Cookie)       |
  |                         |<--响应-------------------|
```

**防御方法：**

1. **CSRF Token**
   - 服务器生成随机token
   - 客户端在请求中携带token
   - 服务器验证token

```javascript
// 前端
fetch('/api/data', {
  method: 'POST',
  headers: {
    'X-CSRF-Token': getCsrfToken()
  },
  body: JSON.stringify(data)
});
```

2. **SameSite Cookie**
   - 限制Cookie的跨站发送
   - 值：Strict、Lax、None

```javascript
// 后端
res.cookie('sessionid', 'xxx', {
  sameSite: 'strict'
});
```

3. **验证Referer**
   - 检查请求来源
   - 防止跨站请求

```javascript
// 后端
const referer = req.headers.referer;
if (!referer || !referer.startsWith('https://example.com')) {
  return res.status(403).send('Forbidden');
}
```

4. **双重Cookie**
   - 在Cookie中存储token
   - 在请求中携带token
   - 服务器验证两者是否一致

### XSS防御

**XSS（Cross-Site Scripting）**
- 跨站脚本攻击
- 在网页中注入恶意脚本
- 窃取用户信息

**XSS类型：**

1. **存储型XSS**
   - 恶意脚本存储在服务器
   - 所有访问该页面的用户都会执行

2. **反射型XSS**
   - 恶意脚本通过URL参数传递
   - 用户点击链接后执行

3. **DOM型XSS**
   - 恶意脚本修改DOM
   - 在客户端执行

**防御方法：**

1. **输入过滤**
   - 过滤特殊字符
   - 限制输入长度

```javascript
function escapeHtml(unsafe) {
  return unsafe
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;")
    .replace(/'/g, "&#039;");
}
```

2. **输出编码**
   - HTML编码
   - JavaScript编码
   - URL编码

3. **Content Security Policy（CSP）**
   - 限制资源来源
   - 防止脚本注入

```javascript
// 设置CSP
res.setHeader('Content-Security-Policy', "default-src 'self'");
```

4. **HttpOnly Cookie**
   - 防止JavaScript访问Cookie
   - 减少XSS危害

```javascript
// 后端
res.cookie('sessionid', 'xxx', {
  httpOnly: true
});
```

### CSP

**CSP（Content Security Policy）**
- 内容安全策略
- HTTP头部
- 限制资源来源

**CSP指令：**

| 指令 | 描述 | 示例 |
|------|------|------|
| default-src | 默认策略 | default-src 'self' |
| script-src | 脚本来源 | script-src 'self' cdn.example.com |
| style-src | 样式来源 | style-src 'self' 'unsafe-inline' |
| img-src | 图片来源 | img-src 'self' data: |
| connect-src | 连接来源 | connect-src 'self' api.example.com |
| font-src | 字体来源 | font-src 'self' fonts.example.com |
| frame-src | 框架来源 | frame-src 'self' |
| object-src | 对象来源 | object-src 'none' |

**CSP示例：**
```http
Content-Security-Policy: default-src 'self'; script-src 'self' cdn.example.com; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:;
```

**CSP报告：**
```javascript
// 设置报告URI
Content-Security-Policy-Report-Only: default-src 'self'; report-uri /csp-report

// 接收报告
app.post('/csp-report', (req, res) => {
  console.log(req.body);
  res.status(204).end();
});
```

### Cookie/Session/JWT/Token

#### Cookie

**特点：**
- 存储在浏览器
- 每次请求自动携带
- 大小限制：4KB
- 可以设置过期时间

**属性：**
```javascript
res.cookie('name', 'value', {
  maxAge: 3600000,      // 过期时间（毫秒）
  expires: new Date(),  // 过期日期
  path: '/',            // 路径
  domain: '.example.com', // 域名
  secure: true,         // 仅HTTPS
  httpOnly: true,       // 仅HTTP
  sameSite: 'strict'    // SameSite策略
});
```

#### Session

**特点：**
- 存储在服务器
- 通过Cookie传递Session ID
- 可以存储敏感信息

**流程：**
```
客户端                    服务器
  |                         |
  |---登录请求-------------->|
  |                         |
  |<--Set-Cookie: sessionid=xxx|
  |                         |
  |---请求(携带sessionid)---->|
  |                         |
  |<--响应------------------|
```

#### JWT（JSON Web Token）

**结构：**
```
Header.Payload.Signature
```

**示例：**
```javascript
// Header
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload
{
  "sub": "1234567890",
  "name": "John Doe",
  "iat": 1516239022
}

// Signature
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)
```

**JWT流程：**
```
客户端                    服务器
  |                         |
  |---登录请求-------------->|
  |                         |
  |<--JWT Token-------------|
  |                         |
  |---请求(携带JWT)--------->|
  |   Authorization: Bearer xxx|
  |                         |
  |<--响应------------------|
```

**JWT vs Session：**

| 特性 | JWT | Session |
|------|-----|---------|
| 存储位置 | 客户端 | 服务器 |
| 无状态 | 是 | 否 |
| 扩展性 | 好 | 差 |
| 安全性 | 较低 | 较高 |
| 大小 | 较大 | 较小 |

#### Token

**特点：**
- 无状态
- 可以存储在Cookie、LocalStorage
- 需要手动携带

**示例：**
```javascript
// 前端
fetch('/api/data', {
  headers: {
    'Authorization': `Bearer ${token}`
  }
});

// 后端
const token = req.headers.authorization?.split(' ')[1];
const decoded = jwt.verify(token, secret);
```

## 浏览器渲染原理

### 渲染流程

**完整渲染流程：**
```
1. 解析HTML → DOM树
2. 解析CSS → CSSOM树
3. 合并DOM和CSSOM → 渲染树
4. 布局（Layout/Reflow）
5. 绘制（Paint/Repaint）
6. 合成（Composite）
```

**流程图：**
```
HTML
  ↓
解析HTML
  ↓
DOM树 ←→ CSSOM ← CSS
  ↓
渲染树
  ↓
布局（计算位置和大小）
  ↓
绘制（绘制像素）
  ↓
合成（合成图层）
  ↓
显示
```

### DOM树构建

**DOM（Document Object Model）**
- 文档对象模型
- 树形结构
- 表示HTML文档的结构

**HTML示例：**
```html
<!DOCTYPE html>
<html>
<head>
  <title>Page Title</title>
</head>
<body>
  <div>
    <h1>Heading</h1>
    <p>Paragraph</p>
  </div>
</body>
</html>
```

**DOM树：**
```
Document
  ├── html
  │   ├── head
  │   │   └── title
  │   │       └── "Page Title"
  │   └── body
  │       └── div
  │           ├── h1
  │           │   └── "Heading"
  │           └── p
  │               └── "Paragraph"
```

### CSSOM构建

**CSSOM（CSS Object Model）**
- CSS对象模型
- 树形结构
- 表示CSS的样式规则

**CSS示例：**
```css
body {
  font-size: 16px;
  color: #333;
}

div {
  margin: 10px;
  padding: 5px;
}

h1 {
  font-size: 24px;
  color: #000;
}
```

**CSSOM树：**
```
CSSOM
  ├── body
  │   ├── font-size: 16px
  │   └── color: #333
  └── div
      ├── margin: 10px
      ├── padding: 5px
      └── h1
          ├── font-size: 24px
          └── color: #000
```

### 渲染树

**渲染树（Render Tree）**
- 合并DOM和CSSOM
- 只包含可见节点
- 用于布局和绘制

**渲染树构建：**
```
DOM树 + CSSOM树 → 渲染树
```

**不可见节点：**
- `display: none`：不在渲染树中
- `visibility: hidden`：在渲染树中，但不绘制
- `<script>`、`<link>`：不在渲染树中

### 布局（Layout/Reflow）

**布局（Layout）**
- 计算元素的位置和大小
- 也称为Reflow

**布局过程：**
```
1. 计算每个元素的尺寸
2. 计算每个元素的位置
3. 处理浮动、定位等
4. 生成布局树
```

**触发布局的操作：**
- 添加/删除元素
- 修改元素样式（width、height、margin等）
- 修改元素内容
- 修改浏览器窗口大小

**布局优化：**
1. 避免频繁修改样式
2. 使用class批量修改样式
3. 避免逐条修改样式

### 绘制（Paint/Repaint）

**绘制（Paint）**
- 将渲染树绘制到屏幕
- 也称为Repaint

**绘制过程：**
```
1. 遍历渲染树
2. 为每个节点绘制像素
3. 处理透明度、阴影等
4. 生成位图
```

**触发布绘制的操作：**
- 修改颜色、背景
- 修改边框样式
- 修改文本内容
- 修改可见性

**绘制优化：**
1. 避免频繁修改样式
2. 使用transform和opacity代替top/left和display
3. 使用will-change提示浏览器

### 合成（Composite）

**合成（Composite）**
- 将多个图层合成到一起
- 使用GPU加速

**合成层：**
```
1. 主层
2. 定位层（position: absolute/fixed）
3. 3D变换层（transform: translate3d）
4. 透明度层（opacity < 1）
5. 滤镜层（filter）
6. will-change层
```

**合成优化：**
1. 使用transform和opacity
2. 使用will-change
3. 避免过多的合成层

### 关键渲染路径

**关键渲染路径（Critical Rendering Path）**
- 影响首屏渲染的路径
- 优化关键渲染路径可以提升首屏速度

**关键渲染路径步骤：**
```
1. 构建DOM树
2. 构建CSSOM树
3. 合并DOM和CSSOM → 渲染树
4. 布局
5. 绘制
6. 合成
```

**优化关键渲染路径：**
1. 减少DOM节点
2. 优化CSS选择器
3. 内联关键CSS
4. 异步加载非关键CSS
5. 使用媒体查询

### 渲染面试要点

**Q1: 浏览器渲染流程？**

A: 见上面完整渲染流程图。

**Q2: DOM树和渲染树的区别？**

A:
1. **DOM树**：包含所有节点
2. **渲染树**：只包含可见节点
3. 渲染树用于布局和绘制

**Q3: 回流和重绘的区别？**

A:
1. **回流（Reflow）**：计算元素的位置和大小
2. **重绘（Repaint）**：绘制元素的像素
3. 回流一定会触发重绘，重绘不一定触发回流

**Q4: 如何优化渲染性能？**

A:
1. 避免频繁修改样式
2. 使用class批量修改样式
3. 使用transform和opacity
4. 使用will-change
5. 避免强制同步布局

**Q5: 什么是关键渲染路径？**

A:
1. 影响首屏渲染的路径
2. 包括DOM构建、CSSOM构建、布局、绘制、合成
3. 优化关键渲染路径可以提升首屏速度

## 浏览器事件循环

### Event Loop机制

**Event Loop**
- 事件循环
- JavaScript的执行机制
- 处理异步任务

**Event Loop组成：**
```
1. 执行栈（Call Stack）
2. 任务队列（Task Queue）
   - 宏任务队列（Macrotask Queue）
   - 微任务队列（Microtask Queue）
```

**Event Loop流程：**
```
1. 执行同步代码（执行栈）
2. 执行所有微任务
3. 执行一个宏任务
4. 重复步骤2-3
```

**流程图：**
```
执行栈
  ↓
执行同步代码
  ↓
检查微任务队列
  ↓
执行所有微任务
  ↓
检查宏任务队列
  ↓
执行一个宏任务
  ↓
重复
```

### 宏任务和微任务

**宏任务（Macrotask）**
- setTimeout
- setInterval
- setImmediate（Node.js）
- I/O
- UI渲染

**微任务（Microtask）**
- Promise.then/catch/finally
- MutationObserver
- queueMicrotask
- process.nextTick（Node.js）

**执行顺序：**
```
1. 同步代码
2. 所有微任务
3. 一个宏任务
4. 所有微任务
5. 一个宏任务
6. ...
```

**示例：**
```javascript
console.log('1');

setTimeout(() => {
  console.log('2');
}, 0);

Promise.resolve().then(() => {
  console.log('3');
});

console.log('4');

// 输出：1 4 3 2
```

### requestAnimationFrame

**requestAnimationFrame**
- 在下一次重绘之前执行回调
- 适合动画
- 自动暂停（页面不可见时）

**示例：**
```javascript
function animate() {
  // 更新动画
  requestAnimationFrame(animate);
}

requestAnimationFrame(animate);
```

**vs setTimeout：**
```
requestAnimationFrame:
  - 在重绘之前执行
  - 自动暂停
  - 适合动画

setTimeout:
  - 在指定时间后执行
  - 不自动暂停
  - 适合定时任务
```

### requestIdleCallback

**requestIdleCallback**
- 在浏览器空闲时执行回调
- 适合低优先级任务
- 可以设置超时时间

**示例：**
```javascript
requestIdleCallback((deadline) => {
  while (deadline.timeRemaining() > 0) {
    // 执行任务
  }
}, {
  timeout: 2000
});
```

**vs requestAnimationFrame：**
```
requestIdleCallback:
  - 在浏览器空闲时执行
  - 适合低优先级任务

requestAnimationFrame:
  - 在重绘之前执行
  - 适合动画
```

### 虚拟列表

**虚拟列表（Virtual List）**

**示例：**
```javascript
function VirtualList({ items, itemHeight, containerHeight }) {
  const [scrollTop, setScrollTop] = useState(0);

  const startIndex = Math.floor(scrollTop / itemHeight);
  const endIndex = Math.min(
    startIndex + Math.ceil(containerHeight / itemHeight),
    items.length - 1
  );

  const visibleItems = items.slice(startIndex, endIndex + 1);

  return (
    <div
      style={{ height: containerHeight, overflow: 'auto' }}
      onScroll={(e) => setScrollTop(e.target.scrollTop)}
    >
      <div style={{ height: items.length * itemHeight }}>
        {visibleItems.map((item, index) => (
          <div
            key={startIndex + index}
            style={{
              height: itemHeight,
              position: 'absolute',
              top: (startIndex + index) * itemHeight,
            }}
          >
            {item}
          </div>
        ))}
      </div>
    </div>
  );
}
```

### OAuth2.0流程

**OAuth2.0**
- 开放授权协议
- 允许第三方应用访问用户资源
- 不需要用户名密码

**授权码模式流程：**
```
用户                    第三方应用                    授权服务器                    资源服务器
  |                         |                             |                             |
  |---访问第三方应用-------->|                             |                             |
  |                         |---重定向到授权服务器-------->|                             |
  |                         | (client_id, redirect_uri,   |                             |
  |                         |  scope, state)              |                             |
  |                         |                             |                             |
  |---授权----------------->|                             |                             |
  |                         |<--重定向回第三方应用--------|                             |
  |                         | (授权码code)                |                             |
  |                         |                             |                             |
  |                         |---用授权码换取access_token-->|                             |
  |                         | (client_id, client_secret,  |                             |
  |                         |  code, redirect_uri)        |                             |
  |                         |                             |                             |
  |                         |<--access_token, refresh_token|                           |
  |                         |                             |                             |
  |                         |---用access_token访问资源---->|--------------------------->|
  |                         |                             |                             |
```

   - 较不安全
   - 适合纯前端应用

3. **密码模式（Resource Owner Password Credentials）**
   - 较不安全
   - 适合可信应用

4. **客户端模式（Client Credentials）**
   - 不需要用户授权
   - 适合服务间通信

## 11. V8 垃圾回收机制

### 11.1 分代回收策略

V8 将堆内存分为新生代和老生代两个区域，采用不同的回收算法。

```
堆内存布局：
┌─────────────────────────────────┐
│         新生代 (Young)           │
│  ┌──────────┬──────────┐        │
│  │ From 空间 │ To 空间   │        │
│  │ (1-8MB)  │ (1-8MB)  │        │
│  └──────────┴──────────┘        │
├─────────────────────────────────┤
│         老生代 (Old)             │
│  ┌─────────────────────────┐    │
│  │ Mark-Sweep / Mark-Compact│    │
│  │ (大部分堆空间)            │    │
│  └─────────────────────────┘    │
├─────────────────────────────────┤
│         大对象区 (Large)         │
│  直接分配在老生代（> 256KB）      │
└─────────────────────────────────┘
```

### 11.2 新生代 Scavenge 算法

```
Scavenge 采用半空间复制（Cheney 算法）：

1. 新对象分配在 From 空间
2. 当 From 空间快满时，触发 Scavenge
3. 检查 From 空间中的存活对象：
   - 存活对象复制到 To 空间
   - 如果对象已经经历过一次 Scavenge，晋升到老生代
4. From 和 To 角色互换
5. 清空原来的 From 空间

优势：速度快，只检查存活对象
劣势：堆空间利用率只有 50%（From + To）

晋升条件：
- 对象已经经历过一次 Scavenge
- To 空间使用超过 25%（避免复制开销过大）
```

### 11.3 老生代 Mark-Sweep / Mark-Compact

```
Mark-Sweep（标记清除）：
  1. 标记阶段：从根对象（全局变量、栈变量）出发，标记所有可达对象
  2. 清除阶段：清除所有未标记的对象

问题：清除后产生内存碎片

Mark-Compact（标记整理）：
  1. 标记阶段：同 Mark-Sweep
  2. 整理阶段：将存活对象移动到一端，消除内存碎片

V8 的策略：
  - 大部分时间使用 Mark-Sweep（速度快）
  - 当内存碎片过多时，使用 Mark-Compact（消除碎片）
```

### 11.4 增量标记与并发回收

```
问题：全堆 Stop-The-World（STW）会导致应用卡顿

解决方案：
1. 增量标记（Incremental Marking）
   - 将标记工作拆分为多个小步骤
   - 每个步骤之间执行 JavaScript 代码
   - 标记过程中新产生的对象标记为黑色（确保不被遗漏）

2. 并发回收（Concurrent）
   - 在辅助线程中执行标记和清除
   - 主线程继续执行 JavaScript
   - 需要处理并发修改问题（三色标记法）

3. 并行回收（Parallel）
   - 多个辅助线程同时执行回收
   - 仍然需要 STW，但时间更短

三色标记法：
  - 白色：未被访问（可能是垃圾）
  - 灰色：已被访问，但其引用的对象未完全访问
  - 黑色：已被访问，且其引用的对象都已访问

写屏障（Write Barrier）：
  - 当黑色对象引用白色对象时，将白色对象标记为灰色
  - 确保增量标记的正确性
```

### 11.5 面试问答

**Q: 如何避免触发频繁的 GC？**

```javascript
// 1. 避免在循环中创建大量临时对象
// bad
for (let i = 0; i < 10000; i++) {
  const obj = { x: i, y: i * 2 }; // 每次循环创建新对象
}
// good: 复用对象
const obj = { x: 0, y: 0 };
for (let i = 0; i < 10000; i++) {
  obj.x = i;
  obj.y = i * 2;
}

// 2. 避免大字符串拼接
// bad
let str = '';
for (let i = 0; i < 10000; i++) {
  str += 'x'; // 每次创建新字符串
}
// good
const arr = [];
for (let i = 0; i < 10000; i++) {
  arr.push('x');
}
const str = arr.join('');

// 3. 使用对象池复用对象
class ObjectPool {
  constructor(createFn) {
    this.createFn = createFn;
    this.pool = [];
  }
  acquire() {
    return this.pool.pop() || this.createFn();
  }
  release(obj) {
    this.pool.push(obj);
  }
}
```

## 12. Web Worker

### 12.1 基本概念

Web Worker 允许在后台线程中运行 JavaScript，不阻塞主线程。适用于 CPU 密集型任务。

```
主线程 ←→ Web Worker（独立的全局上下文）
  - 通过 postMessage 通信
  - 不能直接操作 DOM
  - 有自己的事件循环
  - 可以使用 XMLHttpRequest、fetch、IndexedDB、WebSocket
```

### 12.2 使用方式

```javascript
// 主线程
const worker = new Worker('worker.js');

// 发送消息
worker.postMessage({ type: 'calculate', data: [1, 2, 3, 4, 5] });

// 接收消息
worker.onmessage = (e) => {
  console.log('Worker 结果:', e.data);
};

// 错误处理
worker.onerror = (e) => {
  console.error('Worker 错误:', e.message);
};

// 终止 Worker
worker.terminate();

// worker.js
self.onmessage = (e) => {
  const { type, data } = e.data;
  if (type === 'calculate') {
    const result = data.reduce((sum, n) => sum + n, 0);
    self.postMessage(result);
  }
};
```

### 12.3 内联 Worker

```javascript
// 使用 Blob 创建内联 Worker（无需单独文件）
```

## 13. Service Worker

### 13.1 生命周期

```
Service Worker 生命周期：
  1. 注册（register）
  2. 安装（install）— 缓存静态资源
  3. 等待激活（waiting）— 等待旧 SW 控制的页面关闭
  4. 激活（activate）— 清理旧缓存
  5. 控制页面（fetch/message 事件）
  6. 终止（idle 超时后自动终止，需要时重新唤醒）
```

```javascript
// 注册 Service Worker
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js', { scope: '/' })
    .then(registration => {
      console.log('SW 注册成功，作用域:', registration.scope);
    })
    .catch(err => {
      console.error('SW 注册失败:', err);
    });
}

// sw.js
const CACHE_NAME = 'my-app-v1';
const ASSETS_TO_CACHE = [
  '/',
  '/index.html',
  '/styles.css',
  '/app.js',
  '/logo.png'
];

// 1. 安装事件 — 预缓存静态资源
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME)
      .then(cache => cache.addAll(ASSETS_TO_CACHE))
      .then(() => self.skipWaiting()) // 跳过等待，立即激活
  );
});

// 2. 激活事件 — 清理旧缓存
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then(keys => {
      return Promise.all(
        keys.filter(key => key !== CACHE_NAME)
            .map(key => caches.delete(key))
      );
    }).then(() => self.clients.claim()) // 立即控制所有页面
  );
});

// 3. 拦截网络请求
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request)
      .then(cachedResponse => {
        if (cachedResponse) {
          return cachedResponse;
        }
        return fetch(event.request);
      })
  );
});
```

### 13.2 缓存策略

```javascript
// 策略 1: Cache First（缓存优先）
// 适合：不常变化的静态资源
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then(cached => {
      return cached || fetch(event.request).then(response => {
        const clone = response.clone();
        caches.open(CACHE_NAME).then(cache => cache.put(event.request, clone));
        return response;
      });
    })
  );
});

// 策略 2: Network First（网络优先）
// 适合：频繁变化的 API 数据
self.addEventListener('fetch', (event) => {
  event.respondWith(
    fetch(event.request)
      .then(response => {
        const clone = response.clone();
        caches.open(CACHE_NAME).then(cache => cache.put(event.request, clone));
        return response;
      })
      .catch(() => caches.match(event.request))
  );
});

// 策略 3: Stale While Revalidate（缓存同时更新）
// 适合：可以容忍短暂过期的资源
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then(cached => {
      const fetchPromise = fetch(event.request).then(response => {
        const clone = response.clone();
        caches.open(CACHE_NAME).then(cache => cache.put(event.request, clone));
        return response;
      });
      return cached || fetchPromise;
    })
  );
});

// 策略 4: Cache Only（仅缓存）
// 适合：预缓存的静态资源

// 策略 5: Network Only（仅网络）
// 适合：实时性要求高的 API
```

## 14. HTTP/2 帧结构深入

### 14.1 帧格式

```
HTTP/2 通信的基本单位是帧（Frame），所有通信都在一个 TCP 连接上完成。

帧结构（9 字节头部 + 可变长度载荷）：
┌──────────────────┬──────────────┬──────────────┐
│  Length (3字节)   │ Type (1字节)  │ Flags (1字节) │
├──────────────────┼──────────────┴──────────────┤
│                R (1位) │ Stream Identifier (31位) │
├──────────────────┴─────────────────────────────┤
│                  Frame Payload (0 ~ 2^24-1)     │
└────────────────────────────────────────────────┘

Length: 帧载荷的长度（不包括 9 字节头部）
Type: 帧类型
Flags: 帧标志位
R: 保留位，必须为 0
Stream Identifier: 流标识符
```

### 14.2 帧类型

```
DATA (0x0) — 传输请求/响应体
  Flags: END_STREAM（流结束）、PADDED（填充）

HEADERS (0x1) — 传输请求/响应头
  Flags: END_STREAM、END_HEADERS、PADDED、PRIORITY

PRIORITY (0x2) — 指定流的优先级
  载荷：依赖流 ID + 权重

RST_STREAM (0x3) — 终止流
  载荷：错误码

SETTINGS (0x4) — 连接配置
  Flags: ACK
  参数：
    - HEADER_TABLE_SIZE（HPACK 动态表大小）
    - MAX_CONCURRENT_STREAMS（最大并发流数）
    - INITIAL_WINDOW_SIZE（初始窗口大小）
    - MAX_FRAME_SIZE（最大帧大小）

PING (0x6) — 心跳检测
GOAWAY (0x7) — 关闭连接
WINDOW_UPDATE (0x8) — 流量控制（更新窗口大小）
PUSH_PROMISE (0x5) — 服务器推送
```

### 14.3 流控制（Flow Control）

```javascript
// HTTP/2 的流控制机制：
// 1. 连接级别流控制 — 整个连接的窗口
// 2. 流级别流控制 — 每个流的窗口

// 窗口大小：
// - 初始值由 SETTINGS 帧指定（默认 65535 字节）
// - 发送数据时减少窗口
// - 接收 WINDOW_UPDATE 帧时增加窗口
// - 窗口为 0 时停止发送

// 流控制解决的问题：
// 1. 快速发送方不会压垮慢速接收方
// 2. 防止代理缓冲区溢出
// 3. 允许接收方按自己的节奏处理数据
```

### 14.4 HPACK 头部压缩

```
HTTP/2 使用 HPACK 算法压缩头部，解决 HTTP/1.1 头部冗余问题。

HPACK 三大技术：

1. 静态表（Static Table）
   - 61 个预定义的常用头部
   - 例：:method: GET, :path: /, :scheme: https
   - 客户端和服务端都有相同的静态表

2. 动态表（Dynamic Table）
   - 缓存之前发送过的头部
   - FIFO 策略，新条目替换旧条目
   - 大小由 SETTINGS_HEADER_TABLE_SIZE 控制

3. 哈夫曼编码（Huffman Coding）
   - 对字符串值进行压缩
   - 使用预定义的哈夫曼编码表

压缩示例：
  第一次请求：
    :method: GET → 索引 2（静态表）
    :path: /index.html → 哈夫曼编码
    host: example.com → 哈夫曼编码

  第二次请求（相同路径）：
    :method: GET → 索引 2
    :path: /index.html → 索引（动态表）
    host: example.com → 索引（动态表）
    → 只需发送几个索引号，极大减少传输量
```

### 14.5 HTTP/2 vs HTTP/1.1

```
HTTP/1.1 的问题：
  1. 队头阻塞（Head-of-Line Blocking）— 同一连接上的请求必须排队
  2. 头部冗余 — 每个请求都重复发送相同的头部
  3. 明文传输 — 不安全

HTTP/2 的改进：
  1. 多路复用 — 一个 TCP 连接上并行多个请求/响应
  2. 头部压缩 — HPACK 算法
  3. 二进制分帧 — 更高效、更安全
  4. 服务器推送 — 服务器可以主动推送资源
  5. 流优先级 — 可以设置请求的优先级

HTTP/2 的局限：
  1. TCP 层的队头阻塞 — 一个 TCP 包丢失，所有流都受影响
  2. 握手开销大 — TCP + TLS 握手
  → HTTP/3 (QUIC) 解决了这些问题
```
