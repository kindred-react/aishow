1,有哪些常见的loader?,用过哪些?

1. raw-loader 加载原始文件内容，不进行任何转换
2. file-loader 加载文件并返回文件路径
3. url-loader 与 file-loader 类似，但是可以将文件转换为 base64 编码的字符串,
可以设置 limit 参数，当文件大小小于 limit 时，返回 base64 编码的字符串，
否则使用 file-loader 加载文件
（Webpack 5 已内置 Asset Modules 替代：asset/resource、asset/inline）
4. source-map-loader 加载 source map 文件,方便调试
5. image-loader 加载图片文件,可以设置压缩参数,减少图片大小
6. json-loader 加载 json 文件,可以将 json 文件转换为 javascript 对象
7. babel-loader 加载 javascript 文件,可以使用最新的 javascript 语法
8. ts-loader 加载 typescript 文件,可以将 typescript 文件转换为 javascript 文件
9. css-loader 加载 css 文件,可以将 css 文件转换为 javascript 对象
10. style-loader 可以将 css-loader 加载的 css 文件插入到 html 文件的 head 标签中
11. sass-loader 加载 sass 文件,可以将 sass 文件转换为 css 文件
12. less-loader 加载 less 文件,可以将 less 文件转换为 css 文件
13. postcss-loader 加载 postcss 文件,可以使用 postcss 插件处理 css 文件,扩展了 css-loader 的功能
14. eslint-loader 加载 javascript/typescript 文件，配合 typescript-eslint 解析器（TSLint 已于2019年弃用）
15. vue-loader 加载 vue 文件,可以将 vue 文件转换为 javascript 对象

2,webpack 有哪些常见的 plugin?,用过哪些?插件

1. DefinePlugin 定义全局变量,可以在代码中使用
2. html-webpack-plugin 生成 html 文件,可以自动引入打包后的 js 文件,简化了 html 文件的编写
3. IgnorePlugin 忽略指定的模块,可以减少打包后的文件大小
4. TerserPlugin 压缩 javascript 文件，支持 ES6+ 语法（Webpack 5 生产模式默认使用，uglifyjs-webpack-plugin 已不推荐）
5. mini-css-extract-plugin 提取 css 文件,可以将 css 文件单独打包,减少 html 文件的大小,支持按需加载
6. clean-webpack-plugin 清理打包目录,可以在每次打包前清理旧的打包文件,避免打包目录混乱
7. webpack-bundle-analyzer 分析打包后的文件,可以可视化查看打包后的文件大小,方便优化
8. speed-measure-webpack-plugin 分析打包后的文件,可以可视化查看每个插件和 loader 的耗时,方便优化

3. loader 和 plugin 的区别?
loader本质是函数,用来加载文件,可以将文件转换为 javascript 对象
plugin本质是类,用来扩展 webpack 功能,可以在 webpack 打包过程中执行一些额外的操作,基于事件流框架,Tapable,webpack 广播了一些事件,插件可以监听这些事件,在事件触发时执行一些操作,改变webpack的打包行为
1. loader 是用来加载文件的,可以将文件转换为 javascript 对象,预处理文件,例如将 css 文件转换为 javascript 对象,将 typescript 文件转换为 javascript 文件,通过module.rules配置
2. plugin 是用来扩展 webpack 功能的,可以在 webpack 打包过程中执行一些额外的操作,
例如压缩 javascript 文件,生成 html 文件等,通过plugins配置,实例化插件,添加到webpack配置中

4. webpack 打包过程? 初始化,递归编译,确定入口文件,分析依赖关系,打包模块,输出文件,输出完成
运行流程是一个串行的过程,从入口文件开始,根据依赖关系,递归地加载所有依赖的模块,
每个模块都被 loader 处理,转换为 javascript 对象,最后打包成一个或多个 bundle 文件
1. 初始化参数,读取 webpack 配置文件,初始化命令行参数,例如入口文件,输出目录,插件等
2. 编译模块,根据依赖关系,递归地加载所有依赖的模块,每个模块都被 loader 处理,转换为 javascript 对象
3. 确定入口文件,根据配置的入口文件,确定打包的起点
4. 分析依赖关系,根据入口文件,递归地分析所有依赖的模块,确定模块之间的依赖关系
5. 打包模块,根据依赖关系,将所有模块打包成一个或多个 bundle 文件
6. 输出文件,将打包后的文件输出到指定目录,可以配置输出文件名,输出路径等,包含多个 bundle 文件
7. 输出完成,打包过程结束,输出完成
webpack会在特定的时间点广播事件,例如编译完成,可以监听这些事件,在事件触发时执行一些操作,例如生成 html 文件,压缩 javascript 文件等

5. 使用webpack过程中,使用过提效插件
1. webpack-bundle-analyzer 分析打包后的文件,可以可视化查看打包后的文件大小,方便优化
2. speed-measure-webpack-plugin 分析打包后的文件,可以可视化查看每个插件和 loader 的耗时,方便优化
3. webpack-merge 合并 webpack 配置文件,可以将多个配置文件合并为一个,方便管理
4. size-plugin 分析打包后的文件,可以可视化查看每个模块的大小,方便优化
5. hot-module-replacement-plugin 热模块替换插件,可以在开发过程中,实时更新修改的模块,而不需要刷新整个页面,提高开发效率

6. 文件指纹是什么?怎么用?
1. 文件指纹是指打包后的文件的文件名,可以根据文件内容,生成一个唯一的字符串,作为文件名,例如根据文件内容,生成一个 md5 字符串,作为文件名,可以根据文件内容,生成一个唯一的文件名,避免文件缓存问题
2. 如何使用文件指纹? 可以在 webpack 配置文件中,设置 output.filename 和 output.chunkFilename 为 [name].[contenthash:8].js,可以根据文件内容,生成一个 8 位的 md5 字符串,作为文件名,可以根据文件内容,生成一个唯一的文件名,避免文件缓存问题
hash 和整个项目构建有关,只有项目文件改变,hash 才会改变,如果项目文件没有改变,hash 不会改变,
chunkhash 和每个 chunk 有关,只有 chunk 内容改变,chunkhash 才会改变,
contenthash 和每个模块有关,只有模块内容改变,contenthash 才会改变,

7. js 文件指纹? 可以在 webpack 配置文件中,设置 output.filename 为 [name].[contenthash:8].js,可以根据文件内容,生成一个 8 位的 md5 字符串,作为文件名,可以根据文件内容,生成一个唯一的文件名,避免文件缓存问题

8. css 文件指纹? 可以在 webpack 配置文件中,设置 mini-css-extract-plugin 插件的 filename 为 [name].[contenthash:8].css,可以根据文件内容,生成一个 8 位的 md5 字符串,作为文件名,可以根据文件内容,生成一个唯一的文件名,避免文件缓存问题

9. 图片文件指纹? 可以在 webpack 配置文件中,设置 file-loader 或 url-loader 的 name 为 [name].[contenthash:8].[ext],可以根据文件内容,生成一个 8 位的 md5 字符串,作为文件名,可以根据文件内容,生成一个唯一的文件名,避免文件缓存问题

10. 如何优化webpack打包效率?
高版本,多线程,多实例,压缩代码,缩小打包范围,开启缓存,并行打包,提取公共代码,使用 tree shaking 等(没有用到的模块自动去除)
1. 减少打包体积,可以通过压缩 javascript 文件,压缩 css 文件,压缩图片文件等,来减少打包体积,提高加载速度
2. 合理配置 loader,可以根据文件类型,选择合适的 loader,避免加载不必要的文件,提高打包效率
3. 合理配置 plugin,可以根据需求,选择合适的 plugin,避免加载不必要的插件,提高打包效率
4. 开启缓存,可以开启 webpack 的缓存功能,避免重复打包,提高打包效率
5. 并行打包,可以开启 webpack 的并行打包功能,利用多核 cpu,并行打包,提高打包效率

11. 是否写过自定义 loader 或 plugin? 有,例如自定义一个 loader,用来加载 markdown 文件,将 markdown 文件转换为 html 文件,自定义一个 plugin,用来在打包过程中,生成 html 文件
loader是支持链式调用的,遵循单一职责原则,每个 loader 只负责一种文件类型的转换,例如将 css 文件转换为 javascript 对象,将 typescript 文件转换为 javascript 文件,通过module.rules配置
传给loader的内容是utf-8编码的字符串,loader需要将其转换为javascript对象,进行处理,最后返回处理后的结果,尽可能异步化loader,避免阻塞打包过程,loader是纯函数,没有状态

12. 是否写过自定义 plugin? 有,例如自定义一个 plugin,用来在打包过程中,生成 html 文件,自定义一个 plugin,用来在打包过程中,压缩 javascript 文件等
plugin是用来扩展 webpack 功能的,可以在 webpack 打包过程中执行一些额外的操作,例如压缩 javascript 文件,生成 html 文件等,通过plugins配置,实例化插件,添加到webpack配置中
compiler 是 webpack 的编译器,负责整个打包过程,可以监听事件,在事件触发时执行一些操作,例如生成 html 文件,压缩 javascript 文件等
compilation 暴露与模块和依赖有关的颗粒度更小的事件钩子
plugin 需要在原型上绑定apply方法,才能访问compiler实例

13. babel的原理
babel是一个 javascript 编译器,可以将 ES6 代码转换为 ES5 代码,也可以将 JSX 代码转换为 javascript 代码
1. 解析: 解析器将源代码转换为抽象语法树(AST)(词法分析,语法分析)
2. 转换: 遍历AST,对其中的节点进行转换,例如将ES6代码转换为ES5代码
3. 生成: 生成器将转换后的AST转换为目标代码

14. source map 是什么? 怎么用?
source map 是一种映射关系,可以将编译后的代码映射回原始代码,方便调试,可以在 webpack 配置文件中,设置 devtool 为 source-map,可以生成 source map 文件,可以在浏览器中,开启 source map 功能,查看原始代码,方便调试
线上的话
1. 可以在 webpack 配置文件中,设置 devtool 为 nosources-source-map,可以生成 source map 文件,但是不会包含原始代码,可以在浏览器中,开启 source map 功能,查看原始代码,方便调试
2. hidden-source-map: sentry 等错误监控平台,可以将 source map 文件上传到平台,方便定位错误,查看原始代码,方便调试
3. source-map: nginx 等服务器,只对白名单那开放

15. 文件监听的原理?
--watch 文件监听的原理是,轮询文件系统,定时检查文件是否发生改变,如果文件发生改变,就重新打包,如果文件没有改变,就使用缓存的文件,避免重复打包,提高打包效率

16. webpack 的热更新原理?
热更新是指在不刷新浏览器的情况下,更新页面的某些部分,实现快速开发,热更新的原理是,在 webpack 打包过程中,开启 devServer 热更新功能,当文件发生改变,就会触发 webpack 重新打包,打包完成后,会将打包后的文件发送到浏览器,浏览器会根据文件的 hash 值,判断是否需要更新,如果需要更新,就会更新页面的某些部分,实现热更新
维护chunk diff: 浏览器会根据文件的 hash 值,判断是否需要更新,websocket 推送更习惯的hash,对比WDS,发起请求,获取差异,局部刷新页面

17. webpack的事件机制?
基于tapable, 封装了事件订阅发布机制,可以在打包过程中,监听这些事件,在事件触发时执行一些操作,例如生成 html 文件,压缩 javascript 文件等
webpack 暴露了一些事件钩子,可以在打包过程中,监听这些事件,在事件触发时执行一些操作,例如生成 html 文件,压缩 javascript 文件等
before-run: 开始运行编译前触发
run: 开始运行编译时触发
before-compile: 开始编译前触发
compile: 开始编译时触发
this-compilation: 开始编译时触发,与compile不同的是,只针对当前编译实例
after-compile: 编译完成时触发
invalid: 文件被修改时触发
emit: 生成资源到 output 目录前触发
after-emit: 生成资源到 output 目录后触发
done: 编译完成时触发

19. 模块联邦(Module Federation)
模块联邦是 webpack5 新增的功能,可以实现微应用架构,多个应用之间可以共享代码,提高开发效率,
模块联邦的原理是,在应用之间,定义一个共享的模块,多个应用可以引用这个模块,实现代码共享,模块联邦的配置比较复杂,需要注意的是,多个应用之间,需要保持相同的 webpack 版本,否则会出现兼容性问题

20. 什么是webpack?
webpack 是一个 javascript 模块打包工具,可以将多个模块打包成一个或多个 bundle,可以实现代码分割,按需加载,减少首页加载时间,提高加载速度,webpack 可以与其他工具结合使用,例如 babel,typescript,react 等,可以实现更加强大的功能

21. webpack的核心概念
entry 入口文件,webpack 打包的起点,可以是一个文件,也可以是多个文件,可以是相对路径,也可以是绝对路径
output 输出文件,webpack 打包的终点,可以是一个文件,也可以是多个文件,可以是相对路径,也可以是绝对路径
loader 加载器,可以将不同类型的文件,转换为 javascript 模块,例如将 css 文件转换为 javascript 模块,将图片文件转换为 javascript 模块
plugin 插件,可以在 webpack 打包过程中,执行一些操作,例如生成 html 文件,压缩 javascript 文件等
mode 模式,可以设置为 development 或 production,分别对应开发模式和生产模式,开发模式下,打包速度快,但是打包后的文件体积大,生产模式下,打包速度慢,但是打包后的文件体积小
code splitting 代码分割,可以将代码分割成多个 chunk,按需加载,减少首页加载时间,提高加载速度
resolve 解析,可以配置 webpack 如何解析模块,例如如何查找模块,如何解析模块的路径等
devServer 开发服务器,可以配置 webpack 开发服务器的行为,例如如何启动服务器,如何监听文件变化等

22. mode 模式
可以设置为 development 或 production,分别对应开发模式和生产模式,开发模式下,打包速度快,但是打包后的文件体积大,生产模式下,打包速度慢,但是打包后的文件体积小
区别:
1. 开发模式下,打包速度快,但是打包后的文件体积大,生产模式下,打包速度慢,但是打包后的文件体积小
2. 开发模式下,开启了热更新功能,可以在不刷新浏览器的情况下,更新页面的某些部分,实现快速开发,生产模式下,关闭了热更新功能,不能在不刷新浏览器的情况下,更新页面的某些部分,只能通过刷新浏览器,更新页面的所有部分
3. 开发模式下,开启了 source map 功能,可以在浏览器中,开启 source map 功能,查看原始代码,方便调试,生产模式下,关闭了 source map 功能,不能在浏览器中,查看原始代码,不方便调试

23. 代码分割(code splitting)
代码分割是 webpack 新增的功能,可以将代码分割成多个 chunk,按需加载,减少首页加载时间,提高加载速度,代码分割的配置比较复杂,需要注意的是,多个 chunk 之间,需要保持相同的 webpack 版本,否则会出现兼容性问题
1. 通过配置 entry 入口文件,可以将代码分割成多个 chunk,每个 chunk 对应一个入口文件,例如将 app.js 分割成 app.js 和 vendor.js,其中 app.js 包含应用的主要代码, vendor.js 包含应用的第三方库代码
2. 通过配置 optimization.splitChunks 选项,可以将代码分割成多个 chunk,每个 chunk 对应一个入口文件,例如将 app.js 分割成 app.js 和 vendor.js,其中 app.js 包含应用的主要代码, vendor.js 包含应用的第三方库代码
3. 动态导入(dynamic import),可以在应用运行时,根据需要,动态加载代码,例如在点击按钮时,加载某个模块,实现按需加载,减少首页加载时间,提高加载速度

## 24. Vite 原理

### 24.1 ESM-based 开发服务器

Vite 在开发模式下利用浏览器原生 ES Module 支持，无需打包即可直接提供源代码。

```javascript
// Vite 开发服务器工作原理
// 1. 浏览器请求 HTML
// 2. Vite 返回 HTML，其中 <script type="module"> 引用入口文件
// 3. 浏览器解析到 import 语句，发起 HTTP 请求获取模块
// 4. Vite 拦截请求，按需编译 TypeScript/Vue/JSX 并返回 ES Module 代码
// 5. 浏览器继续解析依赖，逐个请求

// 传统 Webpack Dev Server：
// 1. 启动时打包所有模块（即使未使用）
// 2. 将所有代码打包成 bundle.js
// 3. 使用 eval 或 WebSocket 实现 HMR

// Vite Dev Server：
// 1. 启动时无需打包（毫秒级启动）
// 2. 按需编译（只编译浏览器请求的模块）
// 3. 利用浏览器缓存（未修改的模块使用 304 缓存）
// 4. HMR 通过 ESM 动态 import 实现
```

### 24.2 esbuild 预构建

```javascript
// Vite 使用 esbuild 进行预构建（依赖预打包）
// 为什么需要预构建？
// 1. CommonJS / UMD 转换为 ESM
// 2. 将多个小模块合并为大模块（减少 HTTP 请求数）
// 3. 性能：esbuild 比 Babel 快 10-100 倍（Go 编写）

// vite.config.js
export default {
  optimizeDeps: {
    // 强制预构建
    include: ['lodash', 'react', 'vue'],
    // 排除预构建
    exclude: ['your-local-package'],
    // esbuild 配置
    esbuildOptions: {
      target: 'es2020'
    }
  }
};

// 预构建产物存放在 node_modules/.vite 目录
// .vite/deps/react.js — 合并后的 ESM 模块
// .vite/deps/react.js.map — Source Map
```

### 24.3 Rollup 生产构建

```javascript
// 生产环境使用 Rollup 打包
// 原因：Rollup 的 Tree Shaking 效果更好
// Vite 在 Rollup 基础上增加了代码分割和 CSS 处理

// vite.config.js
export default {
  build: {
    // Rollup 配置
    rollupOptions: {
      output: {
        manualChunks: {
          'react-vendor': ['react', 'react-dom'],
          'utils': ['lodash-es', 'axios']
        }
      }
    },
    // 代码分割策略
    chunkSizeWarningLimit: 500, // KB
    // CSS 代码分割
    cssCodeSplit: true,
    // 构建目标
    target: 'es2020',
    // 压缩
    minify: 'esbuild' // 或 'terser'
  }
};
```

### 24.4 Vite vs Webpack 对比

| 维度 | Vite | Webpack |
|---|---|---|
| **开发启动** | 毫秒级（无需打包） | 秒级（需要打包） |
| **HMR 速度** | 快（只更新单个模块） | 较慢（可能重建整个 bundle） |
| **生产构建** | Rollup | 自身 |
| **生态成熟度** | 较新，但增长快 | 成熟，插件丰富 |
| **配置复杂度** | 简单（开箱即用） | 复杂（需要大量配置） |
| **大型项目** | 适合（按需编译） | 适合（成熟稳定） |
| **兼容性** | 现代浏览器 | 兼容旧浏览器 |

### 24.5 面试问答

**Q: Vite 为什么开发快、生产用 Rollup？**

开发模式利用浏览器原生 ESM，无需打包，按需编译。生产模式需要 Tree Shaking、代码分割、压缩等优化，Rollup 在这些方面更成熟。

## 25. ESM vs CJS 深入对比

### 25.1 加载机制差异

```javascript
// CommonJS (CJS)
// 1. 运行时加载（动态）
// 2. 值的拷贝（修改原模块不影响已加载的值）
// 3. 同步加载
// 4. this 指向 module.exports

// moduleA.js (CJS)
const count = 1;
module.exports = { count };

// moduleB.js (CJS)
const { count } = require('./moduleA');
count = 2; // 不会影响 moduleA 中的 count

// ES Module (ESM)
// 1. 编译时加载（静态）
// 2. 值的引用（只读，修改原模块会影响）
// 3. 异步加载
// 4. this 指向 undefined

// moduleA.mjs (ESM)
export let count = 1;
export function increment() { count++; }

// moduleB.mjs (ESM)
import { count, increment } from './moduleA.js';
console.log(count); // 1
increment();
console.log(count); // 2（实时引用）
```

### 25.2 循环依赖处理

```javascript
// CJS 循环依赖 — 可能得到不完整的值
// a.js
const b = require('./b');
console.log('a got b:', b.bValue); // undefined（b.js 还没执行完）
exports.aValue = 1;

// b.js
const a = require('./a');
console.log('b got a:', a.aValue); // 1（a.js 已导出 aValue）
exports.bValue = 2;

// ESM 循环依赖 — 引用实时绑定，但可能遇到 TDZ
// a.mjs
import { bValue } from './b.mjs';
console.log(bValue); // 2（实时引用，b.mjs 已执行完）
export const aValue = 1;

// b.mjs
import { aValue } from './a.mjs';
console.log(aValue); // 1
export const bValue = 2;
```

## 26. Tree Shaking 原理深入

### 26.1 ES Module 静态分析

```javascript
// Tree Shaking 依赖 ES Module 的静态结构
// 编译时就能确定模块的导入导出关系

// 可以被 Tree Shaking 的代码
export function used() { return 'used'; }
export function unused() { return 'unused'; } // 未被引用，会被删除

// 不能被 Tree Shaking 的情况
// 1. CommonJS 模块（动态 require）
const module = require('./module'); // 无法静态分析

// 2. 副作用代码
// 副作用：模块执行时会影响外部状态
window.globalVar = 'hello'; // 副作用，不会被删除
Array.prototype.myMethod = function() {}; // 副作用

// 3. 动态导入
const module = import('./module'); // 运行时确定
```

### 26.2 sideEffects 字段

```json
// package.json
{
  "name": "my-lib",
  "sideEffects": false,
  // 或指定有副作用的文件
  "sideEffects": [
    "*.css",
    "*.scss",
    "./src/polyfill.js"
  ]
}
```

```javascript
// sideEffects: false
// 告诉打包工具：这个包的所有文件都没有副作用
// 可以安全地删除未使用的导出

// sideEffects: ["*.css"]
// CSS 文件有副作用（影响全局样式），不应被删除

// 注意：如果设置了 sideEffects: false，但代码确实有副作用
// 可能导致运行时错误（如 polyfill 未执行）
```

### 26.3 失效场景

```javascript
// 1. 导入整个模块
import _ from 'lodash'; // 导入整个 lodash，无法 Tree Shake
import { debounce } from 'lodash-es'; // 可以 Tree Shake

// 2. 类的副作用
class Animal {
  static type = 'animal'; // 类属性可能有副作用
}
export { Animal };

// 3. 重新导出
export * from './module'; // 重新导出所有，无法精确分析

// 4. 默认导出对象
export default {
  method1() {},
  method2() {} // 即使只使用 method1，method2 也会被打包
};

// 5. 动态访问属性
const methods = {
  foo() {},
  bar() {}
};
const fn = methods[name]; // 动态访问，无法 Tree Shake
```

### 27.3 JS 沙箱

```javascript
// qiankun 的 JS 沙箱方案：

// 1. Proxy 沙箱（推荐）
// 为每个子应用创建一个 fakeWindow，通过 Proxy 拦截全局变量的读写
class ProxySandbox {
  constructor() {
    const fakeWindow = {};
    this.proxy = new Proxy(fakeWindow, {
      get(target, key) {
        // 优先从 fakeWindow 获取
        if (key in target) return target[key];
        // 否则从真实 window 获取
        return window[key];
      },
      set(target, key, value) {
        target[key] = value; // 写入 fakeWindow
        return true;
      }
    });
  }
}

// 2. 快照沙箱（降级方案）
// 在激活时记录 window 状态，失活时恢复
class SnapshotSandbox {
  constructor() {
    this.windowSnapshot = {};
    this.modifyPropsMap = {};
  }
  activate() {
    this.windowSnapshot = { ...window };
    Object.keys(this.modifyPropsMap).forEach(key => {
      window[key] = this.modifyPropsMap[key];
    });
  }
  deactivate() {
    Object.keys(window).forEach(key => {
      if (window[key] !== this.windowSnapshot[key]) {
        this.modifyPropsMap[key] = window[key];
        window[key] = this.windowSnapshot[key];
      }
    });
  }
}
```

### 28.2 编写 Babel 插件

```javascript
// babel-plugin-const-to-var.js
module.exports = function({ types: t }) {
  return {
    name: 'const-to-var',
    visitor: {
      VariableDeclaration(path) {
        if (path.node.kind === 'const') {
          // 将 const 替换为 var
          path.node.kind = 'var';
        }
      },
      // 自动国际化示例
      JSXText(path) {
        const text = path.node.value.trim();
        if (text && /^[\u4e00-\u9fa5]/.test(text)) {
          path.replaceWith(
            t.jsxExpressionContainer(
              t.callExpression(
                t.identifier('t'),
                [t.stringLiteral(text)]
              )
            )
          );
        }
      }
    }
  };
};

// 使用
// babel.config.js
module.exports = {
  plugins: ['./babel-plugin-const-to-var']
};
```

## 30. npm/pnpm/yarn 对比

| 维度 | npm | yarn (v1) | yarn (v3+) | pnpm |
|---|---|---|---|---|
| **安装速度** | 慢 | 快 | 快 | 最快 |
| **磁盘占用** | 大 | 大 | 大 | 最小 |
| **依赖结构** | 扁平 | 扁平 | PnP / 零安装 | 非扁平（软链接） |
| **锁文件** | package-lock.json | yarn.lock | yarn.lock | pnpm-lock.yaml |
| **Workspace** | 支持 | 支持 | 支持 | 原生支持 |
| **安全性** | 一般 | 一般 | 好 | 最好（严格隔离） |
| **幽灵依赖** | 存在 | 存在 | 可避免 | 避免 |

```bash
# 幽灵依赖问题：
# 扁平化结构下，可以直接 require 未声明的依赖
# 例如：package.json 声明了 lodash@4，
# 但 lodash@4 依赖了 deep-equal@1，
# 在扁平结构中可以直接 require('deep-equal')

# pnpm 的解决方案：
# .pnpm 目录结构
# node_modules/
#   .pnpm/
#     lodash@4.17.21/
#       node_modules/
#         lodash/          → 真实文件
#         deep-equal/      → 真实文件（lodash 的依赖）
#   lodash/                → 符号链接 → .pnpm/lodash@4.17.21/node_modules/lodash
#   deep-equal/            → 不存在（未声明，无法直接访问）

# pnpm 的严格模式确保只能访问 package.json 中声明的依赖
```
