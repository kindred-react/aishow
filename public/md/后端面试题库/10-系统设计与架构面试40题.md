### Q1: 系统设计的完整流程是什么？⭐

**答案：**

系统设计面试通常遵循以下四个阶段：

**1. 需求澄清（Understand the Problem）**
- 功能需求：系统需要做什么
- 非功能需求：性能、可用性、可扩展性、一致性要求
- 明确范围和边界

**2. 估算（Estimation）**
- QPS/TPS估算
- 存储量估算
- 带宽估算
- 常用经验：2^10=1024，1MB=10^6字节

**3. 高层设计（High-Level Design）**
- 画出系统架构图
- 确定核心组件（客户端、负载均衡、API网关、微服务、缓存、数据库、消息队列）
- 确定数据流

**4. 详细设计（Deep Dive）**
- 数据库Schema设计
- API设计
- 核心算法设计
- 解决特定技术挑战

**示例——短链系统估算：**
- 假设日活用户1000万，每人每天生成10条短链
- 每天新增：10^8条
- 每条记录约100字节，每天存储：10GB
- 5年存储：约18TB
- 读QPS：10^8 / 86400 ≈ 1200，峰值约6000 QPS

### Q2: 如何设计一个高并发架构？⭐⭐

**答案：**

高并发架构的核心思路是**分层防御、逐级削峰**：

**1. 接入层**
- DNS轮询 + CDN静态加速
- 负载均衡（Nginx/LVS）
- 限流（令牌桶/漏桶）
- WAF防火墙

**2. 应用层**
- 微服务拆分，服务独立扩缩容
- 异步化（消息队列削峰填谷）
- 缓存（Redis多级缓存）
- 连接池复用（HTTP、数据库、Redis）
- 线程池隔离

**3. 数据层**
- 读写分离
- 分库分表
- 数据库连接池
- 热点数据缓存

**4. 保障机制**
- 熔断降级（Sentinel/Hystrix）
- 限流（单机限流 + 分布式限流）
- 超时控制
- 重试与退避
- 灰度发布、蓝绿部署

```
客户端 → CDN → WAF → 负载均衡 → API网关
                                    ↓
                              微服务集群
                              ↙    ↓    ↘
                           缓存   DB    MQ
```

### Q3: 常见的分布式ID方案有哪些？各有什么优缺点？⭐⭐

**答案：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| UUID | 简单、无中心化 | 无序、36字符长、索引性能差 | 内部系统、非主键 |
| 数据库自增 | 简单、有序 | 单点瓶颈、暴露数据量 | 小型系统 |
| 号段模式 | 性能好、减少DB压力 | 仍有DB依赖 | 中大型系统 |
| 雪花算法 | 有序、高性能、无DB依赖 | 时钟回拨问题 | 大型分布式系统 |
| Redis自增 | 性能好 | 依赖Redis可用性 | 中型系统 |

**雪花算法（Snowflake）详解：**

```
0 | 41位时间戳 | 10位机器ID | 12位序列号
```

- 41位时间戳：约69年
- 10位机器ID：1024个节点
- 12位序列号：每毫秒4096个ID
- 单机每秒可生成约400万ID

```java
public class SnowflakeIdGenerator {
    private final long epoch = 1609459200000L; // 2021-01-01
    private final long machineIdBits = 10L;
    private final long sequenceBits = 12L;
    private final long maxMachineId = ~(-1L << machineIdBits);
    private final long maxSequence = ~(-1L << sequenceBits);

    private final long machineId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;

    public SnowflakeIdGenerator(long machineId) {
        if (machineId < 0 || machineId > maxMachineId) {
            throw new IllegalArgumentException("Machine ID out of range");
        }
        this.machineId = machineId;
    }

    public synchronized long nextId() {
        long currentTimestamp = System.currentTimeMillis();
        if (currentTimestamp < lastTimestamp) {
            throw new RuntimeException("Clock moved backwards");
        }
        if (currentTimestamp == lastTimestamp) {
            sequence = (sequence + 1) & maxSequence;
            if (sequence == 0) {
                currentTimestamp = waitNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        lastTimestamp = currentTimestamp;
        return ((currentTimestamp - epoch) << (machineIdBits + sequenceBits))
                | (machineId << sequenceBits)
                | sequence;
    }

    private long waitNextMillis(long lastTimestamp) {
        long timestamp = System.currentTimeMillis();
        while (timestamp <= lastTimestamp) {
            timestamp = System.currentTimeMillis();
        }
        return timestamp;
    }
}
```

**时钟回拨解决方案：**
- 回拨时间短（<5ms）：等待
- 回拨时间长：抛异常 + 告警
- 多Worker：备用机器ID
- NTP时钟同步 + 时钟漂移检测

### Q4: 什么是CAP定理？在实际中如何取舍？⭐⭐

**答案：**

**CAP定理：** 分布式系统无法同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance），最多只能同时满足两个。

**三大属性：**
- **C（一致性）**：所有节点在同一时刻看到的数据一致
- **A（可用性）**：每个请求都能在合理时间内收到非错误响应
- **P（分区容错性）**：网络分区发生时，系统仍能继续工作

**实际取舍：**

| 选择 | 特点 | 典型系统 |
|------|------|----------|
| CP | 牺牲可用性，保证一致性 | ZooKeeper、Etcd、HBase |
| AP | 牺牲一致性，保证可用性 | Cassandra、Eureka、DynamoDB |
| CA | 理论存在，实际P不可避免 | 单机MySQL（非分布式） |

**重要理解：**
- 网络分区（P）在分布式系统中是必然存在的，所以实际选择是 **CP** 或 **AP**
- CP系统在分区发生时可能拒绝服务（返回错误）
- AP系统在分区发生时可能返回旧数据（最终一致性）
- 大多数互联网应用选择 **AP + 最终一致性**（BASE理论）

### Q5: 什么是BASE理论？⭐⭐

**答案：**

BASE理论是对CAP中AP方案的延伸，是大规模分布式系统的实践指导：

- **BA（Basically Available）**：基本可用
  - 响应时间上的损失：正常0.5ms → 降级后1-2ms
  - 功能上的损失：大促时关闭推荐、评价等非核心功能

- **S（Soft State）**：软状态
  - 允许系统中的数据存在中间状态
  - 中间状态不影响系统整体可用性

- **E（Eventually Consistent）**：最终一致性
  - 系统中所有的数据副本经过一定时间后，最终能达到一致状态
  - 不是强一致性，但保证在没有新更新操作后，数据最终一致

**BASE vs ACID：**

| 维度 | ACID | BASE |
|------|------|------|
| 一致性 | 强一致性 | 最终一致性 |
| 可用性 | 较低 | 高 |
| 隔离性 | 严格 | 松弛 |
| 适用场景 | 传统RDBMS | 分布式系统 |

### Q6: 常见的分布式事务方案有哪些？⭐⭐⭐

**答案：**

**1. 2PC（两阶段提交）**
- 阶段一（Prepare）：协调者询问所有参与者是否可以提交
- 阶段二（Commit/Rollback）：所有参与者同意则提交，否则回滚
- 缺点：同步阻塞、单点故障、数据不一致风险

**2. TCC（Try-Confirm-Cancel）**
- Try：预留资源（冻结库存、冻结金额）
- Confirm：确认操作（扣减库存、扣减金额）
- Cancel：取消操作（解冻库存、解冻金额）
- 缺点：业务侵入性强、需要实现三个接口、补偿逻辑复杂

```java
// TCC接口示例
public interface OrderTccService {
    @TwoPhaseBusinessAction(
        name = "createOrder",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    void try(@BusinessActionContextParameter(paramName = "orderId") String orderId,
             @BusinessActionContextParameter(paramName = "amount") BigDecimal amount);

    boolean confirm(@BusinessActionContext context);

    boolean cancel(@BusinessActionContext context);
}
```

**3. Saga模式**
- 将长事务拆分为多个本地短事务
- 每个本地事务有对应的补偿操作
- 正常流程：T1 → T2 → T3 → ... → Tn
- 补偿流程：C1 ← C2 ← ... ← Cn（某个事务失败时反向补偿）
- 两种实现：编排式（Orchestration）、协调式（Choreography）

**4. 本地消息表**
- 业务数据和消息数据存入同一个数据库（本地事务保证一致性）
- 消息表记录待发送的消息
- 后台定时任务扫描消息表，发送到MQ
- 消费者消费成功后回调确认

```sql
CREATE TABLE local_message (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    biz_id VARCHAR(64) NOT NULL COMMENT '业务ID',
    biz_type VARCHAR(32) NOT NULL COMMENT '业务类型',
    topic VARCHAR(64) NOT NULL COMMENT '消息主题',
    content TEXT NOT NULL COMMENT '消息内容',
    status TINYINT DEFAULT 0 COMMENT '0-待发送 1-已发送 2-消费成功',
    retry_count INT DEFAULT 0 COMMENT '重试次数',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status (status, retry_count)
);
```

**5. 事务消息（RocketMQ）**
- 半消息发送：消息先发到Broker，对消费者不可见
- 本地事务执行：执行本地业务逻辑
- 提交/回滚：根据本地事务结果提交或回滚消息
- 回查机制：Broker定期回查生产者本地事务状态

**方案对比：**

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| 2PC | 强一致 | 低 | 中 | 数据库层面 |
| TCC | 最终一致 | 中 | 高 | 资金类业务 |
| Saga | 最终一致 | 高 | 中 | 长流程业务 |
| 本地消息表 | 最终一致 | 中 | 低 | 异步场景 |
| 事务消息 | 最终一致 | 高 | 中 | 异步场景 |

### Q7: 分布式锁有哪些实现方式？如何对比？⭐⭐

**答案：**

**1. Redis分布式锁**

```java
// Redisson实现
RLock lock = redissonClient.getLock("order:lock:" + orderId);
try {
    // 尝试加锁，等待5秒，锁自动释放时间30秒
    if (lock.tryLock(5, 30, TimeUnit.SECONDS)) {
        // 执行业务逻辑
    }
} finally {
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

Redis锁要点：
- 使用SETNX + 过期时间（原子操作）
- 加锁和设置过期时间必须是原子操作
- 使用Lua脚本保证解锁的原子性
- Redisson的Watch Dog机制自动续期
- RedLock算法解决单点故障（多节点）

**2. ZooKeeper分布式锁**

```java
// Curator实现
InterProcessMutex lock = new InterProcessMutex(
    curatorClient, "/locks/order/" + orderId
);
try {
    if (lock.acquire(5, TimeUnit.SECONDS)) {
        // 执行业务逻辑
    }
} finally {
    lock.release();
}
```

ZooKeeper锁原理：
- 创建临时顺序节点
- 判断自己是否为最小序号节点
- 监听前一个节点的删除事件（避免羊群效应）
- 临时节点：客户端断开连接自动释放

**3. 数据库分布式锁**

```sql
-- 基于唯一索引
INSERT INTO distributed_lock (lock_key, lock_value, expire_time)
VALUES ('order:123', 'uuid', DATE_ADD(NOW(), INTERVAL 30 SECOND))
ON DUPLICATE KEY UPDATE lock_value = IF(lock_value = 'uuid', 'uuid', lock_value);

-- 基于乐观锁（版本号）
UPDATE stock SET count = count - 1, version = version + 1
WHERE id = 1 AND version = #{version};
```

**方案对比：**

| 维度 | Redis | ZooKeeper | 数据库 |
|------|-------|-----------|--------|
| 性能 | 高（10万+QPS） | 中（数万QPS） | 低 |
| 可靠性 | 中（异步复制） | 高（ZAB协议） | 高 |
| 实现复杂度 | 低 | 中 | 低 |
| 锁释放 | 过期时间/Watch Dog | 临时节点 | 超时/手动 |
| 公平性 | 非公平 | 公平（顺序节点） | 非公平 |
| 适用场景 | 高性能场景 | 强一致性场景 | 低频场景 |

### Q8: 常见的负载均衡算法有哪些？⭐⭐

**答案：**

**1. 轮询（Round Robin）**
- 按顺序依次分配请求
- 优点：简单、无状态
- 缺点：不考虑服务器性能差异

**2. 加权轮询（Weighted Round Robin）**
- 根据服务器权重分配
- Nginx配置：`weight=3` 表示每3次分配1次

**3. 随机（Random）**
- 随机选择服务器
- 加权随机：按权重概率分配

**4. 最少连接数（Least Connections）**
- 将请求分配给当前连接数最少的服务器
- 加权最少连接：`least_conn`（Nginx）

**5. 一致性哈希（Consistent Hash）**
- 将服务器和请求的Key都映射到哈希环上
- 顺时针找最近的服务器节点
- 虚拟节点解决数据倾斜问题

```java
public class ConsistentHash<T> {
    private final TreeMap<Long, T> ring = new TreeMap<>();
    private final int virtualNodes;

    public ConsistentHash(int virtualNodes, List<T> nodes) {
        this.virtualNodes = virtualNodes;
        for (T node : nodes) {
            add(node);
        }
    }

    public void add(T node) {
        for (int i = 0; i < virtualNodes; i++) {
            String virtualNodeName = node.toString() + "-VN" + i;
            long hash = hash(virtualNodeName);
            ring.put(hash, node);
        }
    }

    public T get(String key) {
        if (ring.isEmpty()) return null;
        long hash = hash(key);
        Map.Entry<Long, T> entry = ring.ceilingEntry(hash);
        if (entry == null) {
            entry = ring.firstEntry();
        }
        return entry.getValue();
    }

    // FNV-1a哈希
    private long hash(String str) {
        final long p = 1099511628211L;
        long hash = 2166136261L;
        for (int i = 0; i < str.length(); i++) {
            hash ^= str.charAt(i);
            hash *= p;
        }
        return hash;
    }
}
```

**6. IP Hash**
- 根据客户端IP的哈希值分配
- 保证同一客户端的请求始终到同一服务器
- 适用于需要会话保持的场景

**7. P2C（Power of Two Choices）**
- 随机选择两个服务器，取负载较低的那个
- gRPC默认负载均衡策略
- 接近最优分配，性能好

### Q9: 一致性哈希的原理是什么？如何解决数据倾斜？⭐⭐

**答案：**

**原理：**
1. 将整个哈希值空间组织成一个虚拟的圆环（0 ~ 2^32-1）
2. 将服务器节点（IP/主机名）哈希后映射到环上
3. 将数据Key哈希后映射到环上
4. 从Key的位置顺时针查找，找到的第一个服务器节点就是该Key的存储节点

**节点增减影响：**
- 增加节点：只有顺时针方向相邻节点的一部分数据需要迁移
- 删除节点：该节点的数据迁移到顺时针方向的下一个节点
- 影响范围：仅影响环上相邻节点之间的数据

**数据倾斜问题：**
- 当节点数较少时，节点在环上的分布可能不均匀
- 导致某些节点承载大量数据

**解决方案——虚拟节点（Virtual Node）：**
- 为每个物理节点创建多个虚拟节点
- 虚拟节点在环上均匀分布
- 虚拟节点数越多，分布越均匀
- 通常每个物理节点对应150-200个虚拟节点

```
物理节点: A, B, C
虚拟节点: A-1, A-2, A-3, B-1, B-2, B-3, C-1, C-2, C-3
```

**应用场景：**
- Memcached客户端
- Cassandra数据分片
- DynamoDB分区
- 负载均衡

### Q10: 什么是分库分表？垂直拆分和水平拆分的区别？⭐⭐

**答案：**

**分库分表原因：**
- 单库数据量过大（>1000万行），查询性能下降
- 单库并发连接数达到瓶颈
- 单表数据文件过大，影响IO性能

**垂直拆分（Vertical Sharding）：**

垂直分库：按业务功能拆分到不同数据库
```
user_db    → 用户相关表
order_db   → 订单相关表
product_db → 商品相关表
```

垂直分表：将大表按列拆分
```
user_base    → id, name, phone（常用字段）
user_detail  → id, address, bio（不常用字段）
```

**水平拆分（Horizontal Sharding）：**

水平分库：相同结构的表分布到不同数据库
```
order_db_0 → orders（user_id % 4 == 0）
order_db_1 → orders（user_id % 4 == 1）
order_db_2 → orders（user_id % 4 == 2）
order_db_3 → orders（user_id % 4 == 3）
```

水平分表：同一数据库内按规则拆分
```
orders_0, orders_1, orders_2, orders_3
```

**分片策略：**
- 哈希取模：`hash(user_id) % N`
- 范围分片：按ID范围或时间范围
- 一致性哈希：减少扩容数据迁移

**分库分表带来的问题：**
1. 分布式事务（TCC/Saga/本地消息表）
2. 跨库JOIN（数据冗余/应用层组装）
3. 全局ID（雪花算法/号段模式）
4. 数据迁移（扩容时重新分片）
5. 排序分页（需要归并排序）

**常用中间件：**
- ShardingSphere（Apache，支持读写分离、分库分表、分布式事务）
- MyCat
- Vitess（Google，基于MySQL）

### Q11: 读写分离的原理是什么？如何实现？⭐⭐

**答案：**

**原理：**
- 主库（Master）：负责写操作
- 从库（Slave）：负责读操作
- 主库将变更同步到从库（binlog复制）

**MySQL主从复制流程：**
1. Master将变更写入binlog
2. Slave的IO Thread请求Master的binlog
3. Master的Dump Thread发送binlog事件给Slave
4. Slave的IO Thread将binlog写入relay log
5. Slave的SQL Thread重放relay log

**实现方式：**

**1. 代码层面**
```java
// 使用AbstractRoutingDataSource动态切换数据源
public class DynamicDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return DataSourceContextHolder.get();
    }
}

// AOP切面自动切换
@Aspect
@Component
public class DataSourceAspect {
    @Before("@annotation(readOnly)")
    public void switchToSlave(JoinPoint joinPoint, ReadOnly readOnly) {
        DataSourceContextHolder.setSlave();
    }

    @Before("@annotation(readWrite)")
    public void switchToMaster(JoinPoint joinPoint, ReadWrite readWrite) {
        DataSourceContextHolder.setMaster();
    }
}
```

**2. 中间件层面**
- ShardingSphere读写分离
- MyCat读写分离
- ProxySQL

**延迟问题：**
- 主从复制存在延迟（通常几十毫秒到几秒）
- 解决方案：
  - 强制读主：关键业务写后立即读走主库
  - 半同步复制：至少一个从库确认接收
  - 并行复制：MySQL 5.7+ 多线程回放

### Q12: 如何设计一个短链系统？⭐⭐⭐

**答案：**

**核心功能：**

```java
public class ShortUrlGenerator {
    private static final String BASE62 = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";

    public static String encode(long num) {
        StringBuilder sb = new StringBuilder();
        while (num > 0) {
            sb.append(BASE62.charAt((int) (num % 62)));
            num /= 62;
        }
        // 补齐长度
        while (sb.length() < 7) {
            sb.append('0');
        }
        return sb.reverse().toString();
    }
}
```

**方案二：哈希 + 冲突检测**
```java
public String generateShortUrl(String longUrl) {
    // MD5取前7位
    String md5 = DigestUtils.md5Hex(longUrl);
    String shortCode = md5.substring(0, 7);
    // 冲突检测
    if (exists(shortCode)) {
        // 加盐重试
        md5 = DigestUtils.md5Hex(longUrl + System.currentTimeMillis());
        shortCode = md5.substring(0, 7);
    }
    return shortCode;
}
```

### Q16: Feed流系统有哪些模式？如何设计？⭐⭐⭐

**答案：**

**三种模式：**

**1. 推模式（Fan-out on Write）**
- 用户发帖时，推送到所有粉丝的收件箱
- 读操作：直接从收件箱读取，O(1)
- 写操作：O(粉丝数)，大V发帖成本高
- 适用：粉丝数不多的场景

**2. 拉模式（Fan-out on Read）**
- 用户发帖时，只写入自己的发件箱
- 读操作：遍历关注列表，拉取每个关注者的最新帖子
- 写操作：O(1)
- 读操作：O(关注数)，关注多时延迟高
- 适用：读少写多、大V多的场景

**3. 推拉结合（Hybrid）**
- 普通用户发帖：推送到粉丝收件箱（推模式）
- 大V发帖：只写入发件箱，粉丝读时实时拉取（拉模式）
- 阈值：粉丝数 > 10万使用拉模式
- 适用：微博、Twitter等大V+普通用户混合场景

**数据库设计：**

```sql
-- 发件箱（用户发布的帖子）
CREATE TABLE outbox (

    user_id BIGINT NOT NULL,
    content TEXT NOT NULL,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_time (user_id, create_time DESC)
);

-- 收件箱（粉丝的时间线）
CREATE TABLE inbox (
    id BIGINT PRIMARY KEY,

    post_id BIGINT NOT NULL COMMENT '帖子ID',
    author_id BIGINT NOT NULL COMMENT '作者ID',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_time (user_id, create_time DESC)
);
```

**缓存设计：**
- 热点用户的Feed列表缓存到Redis（ZSET，score为时间戳）
- 分页使用游标翻页（避免深分页问题）


### Q17: 常见的限流算法有哪些？⭐⭐

**答案：**

**1. 固定窗口计数器**
- 将时间划分为固定窗口（如1秒）
- 每个窗口内计数，超过阈值则拒绝
- 缺点：临界点突发流量（窗口边界两倍流量）

**2. 滑动窗口计数器**
- 将窗口细分为小格子（如1秒窗口分为10个100ms格子）
- 滑动计算最近1秒的请求总数
- 解决了固定窗口的临界问题

**3. 漏桶算法（Leaky Bucket）**
- 请求进入桶中，桶以固定速率漏出
- 桶满则拒绝请求
- 特点：匀速处理，无法应对突发流量
- 适用：流量整形（如TCP流量控制）

**4. 令牌桶算法（Token Bucket）**
- 以固定速率向桶中放入令牌
- 请求需要获取令牌，桶中有令牌则通过
- 桶满则丢弃令牌
- 特点：允许突发流量（上限为桶的容量/burst capacity）
- 适用：API限流（Google Guava RateLimiter）

```java
// Guava RateLimiter示例
RateLimiter rateLimiter = RateLimiter.create(100); // 每秒100个令牌

if (rateLimiter.tryAcquire()) {
    // 获取到令牌，处理请求
    handleRequest();
} else {
    // 被限流
    throw new RateLimitException("请求过于频繁");
}
```

**5. 滑动日志算法**
- 记录每个请求的时间戳
- 统计最近时间窗口内的请求数
- 最精确但内存消耗大

**Redis + Lua实现滑动窗口限流：**
```lua
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

-- 移除窗口外的记录
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
-- 获取当前窗口内的请求数
local count = redis.call('ZCARD', key)
if count < limit then
    redis.call('ZADD', key, now, now .. math.random())
    redis.call('EXPIRE', key, window / 1000)
    return 1
else
    return 0
end
```

**对比：**

| 算法 | 突发流量 | 精确度 | 内存消耗 | 适用场景 |
|------|----------|--------|----------|----------|
| 固定窗口 | 不支持 | 低 | 低 | 简单限流 |
| 滑动窗口 | 部分 | 中 | 中 | 通用限流 |
| 漏桶 | 不支持 | 中 | 低 | 流量整形 |
| 令牌桶 | 支持 | 中 | 低 | API限流 |
| 滑动日志 | 支持 | 高 | 高 | 精确限流 |

### Q20: 微服务拆分的原则是什么？⭐⭐

**答案：**

**拆分原则：**

**1. 单一职责原则**
- 每个微服务只负责一个业务领域
- 高内聚、低耦合

**2. 业务边界原则（DDD驱动）**
- 按限界上下文（Bounded Context）拆分
- 每个微服务对应一个领域模型

**3. 数据自治原则**
- 每个微服务拥有独立数据库
- 服务间通过API通信，不直接访问对方数据库

**4. 独立部署原则**
- 每个服务可以独立部署、独立升级
- 不影响其他服务

**5. 演进式拆分原则**
- 先拆分变化最频繁的模块
- 不要一次性拆得太细

**常见拆分方式：**
```
单体应用
  ├── 用户服务（用户注册、登录、权限）
  ├── 商品服务（商品管理、分类、搜索）
  ├── 订单服务（下单、支付、退款）
  ├── 库存服务（库存管理、预扣）
  ├── 支付服务（支付、退款、对账）
  └── 通知服务（短信、邮件、推送）
```

**拆分粒度判断：**
- 太粗：失去了微服务的意义
- 太细：服务间通信成本高、运维复杂
- 建议：3-5人团队维护一个微服务

**服务间通信方式：**

| 方式 | 协议 | 特点 | 适用场景 |
|------|------|------|----------|
| 同步HTTP | HTTP/REST | 简单、通用 | 外部API、简单调用 |
| 同步RPC | gRPC/Protobuf | 高性能、强类型 | 内部高频调用 |
| 异步消息 | MQ | 解耦、削峰 | 异步处理、事件驱动 |

### Q21: 服务间通信gRPC和HTTP如何选择？⭐⭐

**答案：**

**gRPC特点：**

- 二进制序列化，性能高
- 强类型接口定义（.proto文件）
- 支持双向流式通信
- 代码自动生成

**HTTP/REST特点：**
- 基于HTTP/1.1（或HTTP/2）
- JSON格式，可读性好
- 生态成熟，工具丰富
- 跨语言兼容性好
- 浏览器原生支持

**对比：**

| 维度 | gRPC | HTTP/REST |
|------|------|-----------|
| 序列化 | Protobuf（二进制） | JSON（文本） |

### Q22: 如何设计一个搜索引擎？⭐⭐⭐

**答案：**

**核心概念——倒排索引（Inverted Index）：**

```
正排索引：文档ID → 文档内容
倒排索引：单词 → 包含该单词的文档ID列表

示例：
Doc1: "Hello World"
Doc2: "Hello Java"

倒排索引：
"hello" → [Doc1, Doc2]
"world" → [Doc1]
"java"  → [Doc2]
```

**搜索引擎架构：**
```
用户查询 → 查询解析 → 分词 → 检索 → 排序 → 返回结果
                                  ↑
                           倒排索引（内存+磁盘）
                                  ↑
                           文档写入 → 分词 → 建索引
```

**核心流程：**

**1. 分词（Tokenization）**
- 中文分词：IK Analyzer、结巴分词
- 英文分词：空格、词干提取
- 同义词扩展、停用词过滤

**2. 索引构建**
- 正排索引：存储文档原始内容
- 倒排索引：单词 → 文档列表
- 词频（TF）、文档频率（IDF）

**3. 相关性排序（TF-IDF / BM25）**
```
TF-IDF = TF × IDF
TF = 词在文档中出现次数 / 文档总词数
IDF = log(总文档数 / 包含该词的文档数)

BM25（改进版）：
score = Σ IDF(qi) × (f(qi,D) × (k1+1)) / (f(qi,D) + k1 × (1-b+b×|D|/avgdl))
```

**4. 分布式架构（Elasticsearch）**
- 索引分片（Primary Shard + Replica Shard）
- 写入流程：Coordinator → Hash路由 → Primary Shard → Replica Shard
- 查询流程：Coordinator → Scatter到所有Shard → Gather合并 → 返回

**ES集群设计：**
```
Node1: Master + Data
Node2: Data
Node3: Data

Index: product_index
├── Primary Shard 0 (Node1)
│   └── Replica Shard 0 (Node2)
├── Primary Shard 1 (Node2)
│   └── Replica Shard 1 (Node3)
└── Primary Shard 2 (Node3)
    └── Replica Shard 2 (Node1)
```

### Q23: CDN的工作原理是什么？⭐

**答案：**

**CDN（Content Delivery Network，内容分发网络）**

**工作原理：**
1. 用户请求域名DNS解析
2. DNS返回CDN节点的IP（通过CNAME指向CDN）
3. CDN的GSLB（全局负载均衡）选择最近的边缘节点
4. 边缘节点检查缓存
   - 命中：直接返回缓存内容
   - 未命中：回源到源站获取内容，缓存后返回

**缓存策略：**
- 强缓存：Cache-Control / Expires
- 协商缓存：ETag / Last-Modified
- CDN节点配置缓存过期时间

**适用内容：**
- 静态资源：图片、CSS、JS、字体
- 视频流媒体
- 软件下载
- HTML页面（需注意动态内容）

**CDN配置示例：**
```
回源Host: www.example.com
缓存规则:
  *.jpg, *.png, *.css, *.js → 缓存30天
  *.html → 缓存1小时
  /api/* → 不缓存（动态请求）
```

**优势：**
- 加速访问（就近节点）
- 减轻源站压力
- 高可用（多节点冗余）
- 防DDoS（CDN节点分散流量）

### Q25: 熔断降级的设计原理是什么？⭐⭐

**答案：**

**熔断器模式（Circuit Breaker Pattern）：**

三种状态：
```
      失败率超过阈值
关闭 ──────────────→ 打开
  ↑                    │
  │    等待超时时间后    │
  └────────────────────┘
         半开
```

- **关闭（Closed）**：正常状态，请求通过
- **打开（Open）**：熔断状态，请求直接被拒绝，快速失败
- **半开（Half-Open）**：试探状态，放行少量请求，成功则关闭，失败则继续打开

**Sentinel熔断策略：**

**1. 慢调用比例**
- 最大响应时间 > 设定阈值 → 慢调用
- 慢调用比例 > 设定比例 → 熔断

**2. 异常比例**
- 异常比例 > 设定比例 → 熔断

**3. 异常数**
- 异常数 > 设定阈值 → 熔断

**降级策略：**
- 返回默认值（兜底数据）
- 返回缓存数据
- 返回友好提示
- 降级到备用服务

```java
// Sentinel降级示例
@SentinelResource(
    value = "getProductInfo",
    fallback = "getProductInfoFallback",
    blockHandler = "getProductInfoBlock"
)
public ProductInfo getProductInfo(Long productId) {
    return productClient.getProduct(productId);
}

// 业务异常降级
public ProductInfo getProductInfoFallback(Long productId, Throwable ex) {
    return ProductInfo.defaultProduct(); // 返回默认商品信息
}

// 限流/熔断降级
public ProductInfo getProductInfoBlock(Long productId, BlockException ex) {
    return ProductInfo.degradedProduct(); // 返回降级商品信息
}
```

### Q26: 如何处理分布式系统中的数据一致性问题？⭐⭐⭐

**答案：**

**一致性级别：**
1. **强一致性**：写入后立即对所有后续读取可见（如银行转账）
2. **弱一致性**：写入后不保证立即可见
3. **最终一致性**：写入后，在没有新更新的情况下，最终所有副本一致

**常见一致性方案：**

**1. 同步双写**
- 写入主库后同步写入从库
- 强一致性但性能差

**2. 异步复制**
- 主库写入后异步复制到从库
- 性能好但有延迟

**3. 读写Quorum（NWR模型）**
- N：副本总数
- W：写操作需要的最小确认数
- R：读操作需要读取的最小副本数
- 强一致条件：W + R > N
- 例：N=3, W=2, R=2 → 每次读至少读到2个副本，写至少确认2个副本

**4. 版本向量（Vector Clock）**
- 每个数据项维护一个版本向量
- 解决多副本并发更新的冲突
- 客户端合并或服务端合并

**5. CRDT（无冲突复制数据类型）**
- 数学上保证合并操作的结果与执行顺序无关
- 类型：G-Counter（计数器）、G-Set（集合）、OR-Set（有序集合）
- 适用：协作编辑、分布式计数

**实践建议：**
- 核心业务（支付、库存）→ 强一致性（分布式事务）
- 非核心业务（点赞、评论数）→ 最终一致性
- 缓存一致性 → 延迟双删 / Canal监听binlog
- 搜索索引一致性 → Canal监听binlog → 更新ES

### Q27: 什么是服务雪崩？如何预防？⭐⭐

**答案：**

**服务雪崩：** 一个服务的不可用导致级联故障，最终整个系统不可用。

**雪崩过程：**
```
服务A故障 → 服务B等待超时 → 服务B线程池耗尽
→ 服务B不可用 → 服务C调用B失败 → 服务C也不可用
→ 级联故障 → 整个系统崩溃
```

**预防方案：**

**1. 熔断（Circuit Breaker）**
- 失败率达到阈值时快速熔断
- 避免级联故障扩散

**2. 降级（Degradation）**
- 非核心功能降级（如关闭推荐、评论）
- 返回兜底数据

**3. 限流（Rate Limiting）**
- 限制请求量，保护系统不被压垮

**4. 超时控制**
- 设置合理的超时时间
- 避免长时间等待

**5. 线程池隔离**
- 不同服务使用独立线程池
- 一个服务的故障不影响其他服务

```java
// Hystrix线程池隔离
@HystrixCommand(
    threadPoolKey = "orderServicePool",
    threadPoolProperties = {
        @HystrixProperty(name = "coreSize", value = "20"),
        @HystrixProperty(name = "maxQueueSize", value = "100"),
        @HystrixProperty(name = "queueSizeRejectionThreshold", value = "80")
    },
    fallbackMethod = "fallback"
)
public Order getOrder(Long orderId) {
    return orderClient.getOrder(orderId);
}
```

**6. 重试与退避**
- 指数退避重试
- 限制重试次数

**7. 服务健康检查**
- 心跳检测
- 自动摘除不健康节点

### Q31: 什么是幂等性？如何保证接口幂等？⭐⭐

**答案：**

**幂等性定义：** 同一个操作执行一次或多次，产生的效果相同，不会因重复执行而产生副作用。

**需要幂等的场景：**
- 支付接口（防止重复扣款）
- 下单接口（防止重复下单）
- 消息消费（防止重复消费）
- 接口重试（网络超时导致的重试）

**幂等方案：**

**1. 唯一索引**
```sql
-- 订单表唯一索引
CREATE UNIQUE INDEX uk_user_product ON orders(user_id, product_id, status);
-- 重复插入会报错，捕获异常返回
```

**2. 乐观锁（版本号）**
```sql
UPDATE account SET balance = balance - 100, version = version + 1
WHERE id = 1 AND version = #{version};
-- 影响行数为0说明已被修改，拒绝操作
```

**3. 状态机**
```java
// 只有待支付状态才能变为已支付
if (order.getStatus() != OrderStatus.UNPAID) {
    throw new BizException("订单状态不正确");
}
order.setStatus(OrderStatus.PAID);
```

**4. Token机制**
```
1. 客户端请求获取Token
2. 服务端生成Token存入Redis（设置过期时间）
3. 客户端携带Token提交请求
4. 服务端删除Token（原子操作）
   - 删除成功：首次请求，处理业务
   - 删除失败：重复请求，拒绝
```

```java
        "    return 0 " +

        new DefaultRedisScript<>(script, Long.class),
        Collections.singletonList("idempotent:" + token)
    );
    return result != null && result == 1L;
}
```

**5. 去重表**

### Q32: 如何设计一个延迟任务系统？⭐⭐⭐

**答案：**

**常见实现方案：**

**1. 定时任务轮询数据库**
```sql
-- 查询到期的延迟任务
SELECT * FROM delayed_task
WHERE status = 0 AND execute_time <= NOW()
LIMIT 100;
```
- 优点：简单
- 缺点：轮询有延迟、数据库压力大

**2. Java DelayQueue**
```java
public class DelayTask implements Delayed {
    private String taskId;
    private long executeTime;

    public DelayTask(String taskId, long delayMs) {
        this.taskId = taskId;
        this.executeTime = System.currentTimeMillis() + delayMs;
    }

    @Override
    public long getDelay(TimeUnit unit) {
        return unit.convert(executeTime - System.currentTimeMillis(), TimeUnit.MILLISECONDS);
    }

    @Override
    public int compareTo(Delayed o) {
        return Long.compare(this.executeTime, ((DelayTask) o).executeTime);
    }
}

// 使用
DelayQueue<DelayTask> queue = new DelayQueue<>();
queue.put(new DelayTask("task1", 5000)); // 5秒后执行
DelayTask task = queue.take(); // 阻塞直到有任务到期
```
- 缺点：单机、重启丢失

**3. Redis ZSet（推荐）**
```java
// 添加延迟任务
public void addDelayTask(String taskId, long delaySeconds) {
    double score = System.currentTimeMillis() / 1000.0 + delaySeconds;
    redisTemplate.opsForZSet().add("delay:tasks", taskId, score);
}

// 消费延迟任务
public void consumeDelayTasks() {
    long now = System.currentTimeMillis() / 1000;
    // 获取到期的任务
    Set<String> tasks = redisTemplate.opsForZSet()
        .rangeByScore("delay:tasks", 0, now, 0, 100);
    for (String taskId : tasks) {
        // 移除任务（原子操作）
        Boolean removed = redisTemplate.opsForZSet()
            .remove("delay:tasks", taskId);
        if (Boolean.TRUE.equals(removed)) {
            // 执行任务
            executeTask(taskId);
        }
    }
}
```

**4. 时间轮（Netty HashedWheelTimer）**
- 将时间划分为多个槽位（tick）
- 每个tick推进一格
- 适用于大量短延迟任务

**5. 消息队列延迟消息**
- RocketMQ：支持18个固定延迟级别（1s/5s/10s/.../2h）
- RabbitMQ：死信队列 + TTL实现
- Kafka：需要自己实现

**方案对比：**

| 方案 | 精度 | 可靠性 | 分布式 | 适用场景 |
|------|------|--------|--------|----------|
| DB轮询 | 秒级 | 高 | 是 | 小规模 |
| DelayQueue | 毫秒级 | 低 | 否 | 单机 |
| Redis ZSet | 秒级 | 中 | 是 | 中大规模 |
| 时间轮 | 毫秒级 | 低 | 否 | 高频短延迟 |
| MQ延迟消息 | 秒级 | 高 | 是 | 大规模 |

### Q36: 如何设计一个分布式缓存系统？⭐⭐⭐

**答案：**

**缓存架构：**
```
客户端 → 本地缓存（Caffeine） → Redis集群 → 数据库
```

**缓存使用模式：**

**1. Cache-Aside（旁路缓存）**
- 读：先读缓存，miss则读DB，回写缓存
- 写：先更新DB，再删除缓存
- 最常用的模式

**2. Read-Through / Write-Through**
- 缓存组件负责同步读写DB
- 应用层只与缓存交互

**3. Write-Behind（异步写）**
- 先更新缓存，异步批量写入DB
- 性能好但可能丢数据

**常见问题及解决方案：**

**缓存穿透（查询不存在的数据）**
- 布隆过滤器
- 缓存空值（TTL短）

**缓存击穿（热点Key过期）**
- 互斥锁（SETNX）
- 热点数据永不过期（逻辑过期）
- 提前续期

**缓存雪崩（大量Key同时过期）**
- 过期时间加随机值
- 多级缓存
- 熔断降级

**缓存与数据库一致性：**
```
方案一：更新DB + 删除缓存（推荐）
  - 可能出现短暂不一致（删除缓存失败）
  - 延迟双删：先删缓存 → 更新DB → 延迟再删缓存

方案二：更新DB + 更新缓存
  - 并发写时容易出现数据不一致

方案三：订阅binlog（最终一致性）
  - Canal监听MySQL binlog
  - 异步更新/删除Redis缓存
```

**Redis集群方案：**
- 主从复制 + 哨兵（Sentinel）
- Redis Cluster（官方集群，16384个槽位）
- Codis（代理模式）
- 客户端分片

**评论排序方案：**
- Wilson Score：考虑点赞数和总投票数，避免少量投票导致评分极端
```
Wilson Score = (up + 1.96^2 / (up+down) - 1.96 * sqrt((up*down)/(up+down) + 1.96^2/(4*(up+down)^2))) / (1 + 1.96^2/(up+down))
```

- Reddit Hot算法：考虑时间衰减
```
Hot Score = log10(|up - down|) + sign(up-down) * t / 45000
```

**性能优化：**
- 评论数缓存到Redis
- 热门评论缓存
- 分页使用游标（避免深分页）
- 敏感词过滤使用DFA算法或Trie树

**敏感词过滤（DFA算法）：**
```java
public class SensitiveWordFilter {
    private TrieNode root = new TrieNode();

    public void addWord(String word) {
        TrieNode node = root;
        for (char c : word.toCharArray()) {
            node.children.putIfAbsent(c, new TrieNode());
            node = node.children.get(c);
        }
        node.isEnd = true;
    }

    public String filter(String text) {
        StringBuilder sb = new StringBuilder();
        TrieNode node = root;
        int start = 0;
        for (int i = 0; i < text.length(); i++) {
            char c = text.charAt(i);
            if (node.children.containsKey(c)) {
                node = node.children.get(c);
                if (node.isEnd) {
                    // 发现敏感词，替换为***
                    for (int j = start; j <= i; j++) {
                        sb.append('*');
                    }
                    start = i + 1;
                    node = root;
                }
            } else {
                sb.append(text.charAt(start));
                start++;
                i = start - 1;
                node = root;
            }
        }
        sb.append(text.substring(start));
        return sb.toString();
    }
}
```

```java
// 点赞
public void like(Long userId, Long targetId) {
    String key = "like:" + targetId;
    // 1. 判断是否已点赞
    Boolean isLiked = redisTemplate.opsForSet().isMember(key, userId);
    if (Boolean.TRUE.equals(isLiked)) {

        redisTemplate.opsForSet().remove(key, userId);

    mqProducer.send(new LikeEvent(userId, targetId));
}

public boolean isLiked(Long userId, Long targetId) {

}

public long getLikeCount(Long targetId) {

}

// 获取点赞用户列表（分页）
public Set<Long> getLikeUsers(Long targetId, long offset, long count) {
```

### Q40: 如何进行系统容量评估和压测？⭐⭐

**答案：**

**容量评估步骤：**

**1. 业务指标估算**
```
日活用户（DAU）：1000万
人均请求：20次/天
日请求量：2亿次
峰值QPS：2亿 / (24 * 3600) * 3（峰值系数）≈ 7000 QPS
读写比：约 10:1
读QPS：约 6300
写QPS：约 700
```

**2. 存储估算**
```
每条记录大小：1KB
日新增记录：700（写QPS）× 86400 = 6000万
日存储量：6000万 × 1KB = 60GB
年存储量：60GB × 365 = 22TB
保留3年：66TB
```

**3. 带宽估算**
```
平均响应大小：10KB
峰值带宽：7000 × 10KB = 70MB/s = 560Mbps
```

**压测方案：**

**1. 压测工具**
- JMeter：功能全面，支持多种协议
- wrk/ab：HTTP压测
- Gatling：基于Scala，支持DSL脚本
- Locust：基于Python，分布式压测

**2. 压测类型**
- 基准测试：单接口性能
- 负载测试：逐步增加并发
- 压力测试：超过设计容量
- 稳定性测试：长时间运行
- 浪涌测试：瞬时大量请求

**3. 压测指标**
- QPS/TPS
- 响应时间（P50/P90/P95/P99）
- 错误率
- CPU/内存/网络/磁盘使用率
- JVM GC情况

**4. JMeter压测脚本示例：**

**5. 压测注意事项**
