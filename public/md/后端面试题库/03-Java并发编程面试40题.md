## 一、线程基础（6题）

### Q1: 创建线程有哪几种方式？

**难度**: ⭐

**答案**: Java 创建线程有四种方式：继承 Thread 类、实现 Runnable 接口、实现 Callable 接口、使用线程池。

**解析**:

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 继承 Thread | 简单直接 | Java 单继承限制 | 简单场景 |
| 实现 Runnable | 灵活，可继承其他类 | 无返回值，不能抛异常 | 通用场景 |
| 实现 Callable | 有返回值，可抛异常 | 使用稍复杂 | 需要结果的场景 |
| 线程池 | 线程复用，控制并发 | 配置复杂 | **生产环境首选** |

```java
// Callable + FutureTask
FutureTask<Integer> futureTask = new FutureTask<>(() -> {
    int sum = 0;
    for (int i = 1; i <= 100; i++) sum += i;
    return sum;
});
new Thread(futureTask).start();
Integer result = futureTask.get(); // 阻塞获取结果
```

**面试技巧**: 强调生产环境必须使用线程池，禁止直接 new Thread。

### Q2: 线程的生命周期有哪些状态？

**难度**: ⭐⭐

**答案**: Java 线程有 6 种状态：NEW（新建）、RUNNABLE（可运行）、BLOCKED（阻塞）、WAITING（等待）、TIMED_WAITING（超时等待）、TERMINATED（终止）。

**解析**:
```
                    ┌──────────┐
          start()   │          │  run() 执行完毕
    NEW ──────────> │ RUNNABLE │ ──────────> TERMINATED
                    │          │
                    └────┬─────┘
                         │
           ┌─────────────┼──────────────────┐
           │             │                  │
    获取锁失败     wait()/join()      sleep(time)/wait(time)
           │             │                  │
           v             v                  v
     ┌──────────┐  ┌──────────┐    ┌───────────────┐
     │ BLOCKED  │  │ WAITING  │    │ TIMED_WAITING │
     └──────────┘  └──────────┘    └───────────────┘
           │             │                  │
      获取锁成功    notify()/notifyAll()   超时或interrupt
           │        /interrupt             │
           └─────────────┼──────────────────┘
                         v
                    ┌──────────┐
                    │ RUNNABLE │
                    └──────────┘
```

- `RUNNABLE` 包含了操作系统层面的 Ready 和 Running 状态
- `BLOCKED`：等待获取 synchronized 锁
- `WAITING`：调用 `wait()`、`join()`、`LockSupport.park()` 后无限期等待
- `TIMED_WAITING`：调用 `sleep(ms)`、`wait(ms)`、`join(ms)` 后有限期等待

### Q3: 守护线程和用户线程的区别？

**难度**: ⭐⭐

**答案**: 守护线程（Daemon Thread）为其他线程提供服务（如 GC 线程），当所有用户线程结束后 JVM 退出，守护线程会被强制终止。用户线程是程序的业务线程，JVM 会等待所有用户线程结束。

**解析**:
- `setDaemon(true)` 必须在 `start()` 之前调用
- 守护线程不能持有需要关闭的资源（文件、数据库连接等），因为可能被强制终止
- 典型守护线程：GC 线程、引用处理线程

```java
Thread daemon = new Thread(() -> {
    while (true) {
        try { Thread.sleep(500); } catch (InterruptedException e) {}
        System.out.println("守护线程运行中...");
    }
});
daemon.setDaemon(true);
daemon.start();
// main 线程结束后，守护线程自动终止
```

### Q4: start() 和 run() 的区别？

**难度**: ⭐

**答案**: `start()` 会启动一个新线程，JVM 回调 `run()` 方法。直接调用 `run()` 不会创建新线程，而是在当前线程中同步执行。

**解析**:
```java
Thread t = new Thread(() -> System.out.println(Thread.currentThread().getName()));
t.run();   // 输出 "main"，在当前线程执行
t.start(); // 输出 "Thread-0"，在新线程执行
```

### Q5: sleep() 和 wait() 的区别？

**难度**: ⭐⭐

**答案**:

| 对比项 | sleep() | wait() |
|--------|---------|--------|
| 所属类 | Thread | Object |
| 释放锁 | 不释放 | 释放 |
| 使用位置 | 任何地方 | 必须在 synchronized 块中 |
| 唤醒方式 | 超时自动唤醒 | notify()/notifyAll() 或超时 |
| 用途 | 暂停当前线程 | 线程间通信 |

**解析**: `sleep()` 不释放锁是常见考点。如果线程在 synchronized 块中调用 sleep()，其他线程无法进入该同步块。

### Q6: 如何优雅地停止一个线程？

**难度**: ⭐⭐

**答案**: 推荐使用 `interrupt()` 机制：设置中断标志位，线程自行检查并退出。不推荐使用 `stop()`（已废弃，不安全）。

**解析**:
```java
Thread t = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        try {
            // 工作逻辑
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            // sleep 期间被中断，清除中断标志并抛出异常
            Thread.currentThread().interrupt(); // 恢复中断标志
            break; // 退出循环
        }
    }
    // 清理资源
});
t.start();
t.interrupt(); // 设置中断标志
```

- `interrupt()` 只是设置标志，不会强制停止线程
- 如果线程在 `sleep()`/`wait()`/`join()` 中被中断，会抛出 `InterruptedException` 并清除中断标志
- `Thread.interrupted()` 检查并清除中断标志，`isInterrupted()` 只检查不清除

## 二、synchronized（6题）

### Q7: synchronized 的使用方式有哪些？

**难度**: ⭐⭐

**答案**: synchronized 可以修饰方法或代码块，锁的对象不同：

| 使用方式 | 锁对象 | 范围 |
|---------|--------|------|
| `synchronized` 修饰实例方法 | this（当前对象） | 该实例的所有同步方法互斥 |
| `synchronized` 修饰静态方法 | Class 对象 | 该类的所有同步静态方法互斥 |
| `synchronized(obj)` 代码块 | 指定对象 | 所有使用同一对象锁的代码块互斥 |
| `synchronized(ClassName.class)` | Class 对象 | 类级别的锁 |

```java
// 实例方法锁
public synchronized void method1() { /* 锁 this */ }

// 静态方法锁
public static synchronized void method2() { /* 锁 ClassName.class */ }

// 代码块锁
public void method3() {
    synchronized (lock) { /* 锁 lock 对象 */ }
}
```

### Q8: synchronized 的锁升级过程是怎样的？

**难度**: ⭐⭐⭐

**答案**: JDK 6 引入偏向锁、轻量级锁、重量级锁的锁升级机制（不可降级）。

**解析**:
```
无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁
```

1. **偏向锁**：第一个访问同步块的线程，在对象头中记录线程 ID。后续该线程进入同步块时，只需检查对象头的线程 ID，无需 CAS
   - 适用于只有一个线程访问同步块的场景
   - JDK 15 默认禁用偏向锁（JEP 374）
   - JDK 18（JEP 450）彻底移除偏向锁，锁升级路径变为：无锁 → 轻量级锁 → 重量级锁

2. **轻量级锁**：当有第二个线程竞争时，偏向锁升级为轻量级锁。线程在自己的栈帧中创建 Lock Record，通过 CAS 将对象头指向 Lock Record
   - 适用于少量线程短时间竞争的场景
   - 竞争线程通过自旋等待

3. **重量级锁**：自旋超过一定次数（自适应自旋）仍未获取锁，升级为重量级锁。线程进入阻塞状态，由操作系统调度
   - 适用于高竞争场景
   - 涉及用户态和内核态切换，开销大

### Q9: synchronized 是可重入锁吗？原理是什么？

**难度**: ⭐⭐

**答案**: synchronized 是可重入锁。同一个线程可以重复获取同一把锁，不会产生死锁。

**解析**:
- 可重入的原理：对象头中记录了持有锁的线程 ID 和获取次数（重入次数）
- 每次进入同步块，重入次数 +1；每次退出，重入次数 -1；当次数为 0 时释放锁

```java
public synchronized void methodA() {
    methodB(); // 同一线程重入，不会死锁
}

public synchronized void methodB() {
    // 可以正常执行
}
```

---

### Q10: synchronized 锁的是什么？锁对象和同步代码块的关系？

**难度**: ⭐⭐

**答案**: synchronized 锁的是**对象**（对象头中的 Mark Word），不是代码块。多个线程只有竞争**同一个对象锁**时才会互斥。

**解析**:
```java
class Counter {
    private final Object lock = new Object();
    private int count;

    public void increment() {
        synchronized (lock) { // 锁 lock 对象
            count++;
        }
    }

    public void decrement() {
        synchronized (lock) { // 同一个锁对象，互斥
            count--;
        }
    }

    public void other() {
        synchronized (this) { // 锁 this 对象，与 lock 不是同一把锁，不互斥
            // ...
        }
    }
}
```

### Q11: synchronized 和 Lock 的区别？

**难度**: ⭐⭐

**答案**:

| 对比项 | synchronized | Lock（ReentrantLock） |
|--------|-------------|----------------------|
| 实现方式 | JVM 层面（monitorenter/monitorexit） | API 层面（java.util.concurrent） |
| 锁释放 | 自动释放（退出同步块） | 必须手动 unlock()（finally 中释放） |
| 可中断 | 不可中断 | lockInterruptibly() 可中断 |
| 超时获取 | 不支持 | tryLock(timeout) 支持 |
| 公平锁 | 非公平 | 支持公平/非公平 |
| 条件绑定 | 单一条件（wait/notify） | 多个 Condition |
| 性能 | JDK 6 后大幅优化 | 高竞争下略优 |

**解析**: 实际开发中优先使用 synchronized（简洁、不易出错），只在需要高级功能（可中断、超时、公平锁、多条件）时使用 ReentrantLock。

### Q12: synchronized 锁住字符串常量会有什么问题？

**难度**: ⭐⭐⭐

**答案**: 字符串常量会被放入字符串常量池，不同代码位置使用相同的字符串字面量会指向同一个对象，导致不相关的同步块互相阻塞。

**解析**:
```java
// 两个不相关的同步块使用了相同的字符串常量
public void methodA() {
    synchronized ("lock") { // 常量池中的 "lock"
        // ...
    }
}

public void methodB() {
    synchronized ("lock") { // 同一个 "lock" 对象！
        // methodA 和 methodB 会互相阻塞
    }
}

// 解决方案：使用 new Object() 作为锁对象
private final Object lockA = new Object();
private final Object lockB = new Object();
```

## 三、volatile（4题）

### Q13: volatile 关键字的作用是什么？

**难度**: ⭐⭐

**答案**: volatile 保证变量的**可见性**和**有序性**（禁止指令重排序），但不保证**原子性**。

**解析**:
- **可见性**：一个线程修改 volatile 变量后，其他线程能立即看到最新值
  - 原理：写入 volatile 变量时会刷新到主内存，读取 volatile 变量时会从主内存重新加载
  - 底层通过 `lock` 前缀指令实现（JMM 层面插入内存屏障）

- **有序性**：禁止编译器和 CPU 对 volatile 变量前后的指令进行重排序
  - 写操作之前的指令不会被重排到写之后
  - 读操作之后的指令不会被重排到读之前

```java
// 可见性示例
volatile boolean running = true;

// 线程A
while (running) {
    // 没有 volatile 时，可能永远读不到 running = false
}

// 线程B
running = false; // volatile 保证线程A立即看到变化
```

---

### Q14: volatile 能保证原子性吗？举例说明。

**难度**: ⭐⭐

**答案**: volatile 不能保证原子性。例如 `volatile int count; count++` 不是原子操作。

**解析**:
- `count++` 实际上是三个操作：读取 count、+1、写回 count
- 即使 volatile 保证了每次读到最新值，但两个线程可能同时读到同一个值，各自 +1 后写回，导致只加了 1

```java
volatile int count = 0;

// 线程A: 读 count(0) -> +1 -> 写 count(1)
// 线程B: 读 count(0) -> +1 -> 写 count(1)
// 结果 count = 1，期望 count = 2

// 解决方案：使用 AtomicInteger
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet(); // CAS 保证原子性
```

---

### Q15: volatile 的底层实现原理？

**难度**: ⭐⭐⭐

**答案**: volatile 通过**内存屏障（Memory Barrier）**实现可见性和有序性。

**解析**:
- 底层使用 `lock` 前缀指令（x86 架构）
- 写 volatile 变量时：
  1. 将当前处理器缓存行的数据写回主内存
  2. 使其他处理器中缓存了该内存地址的数据无效（MESI 缓存一致性协议）
- 内存屏障类型：
  - StoreStore 屏障：保证前面的普通写操作在 volatile 写之前完成
  - StoreLoad 屏障：保证 volatile 写对后续的读可见（最关键的屏障）
  - LoadLoad 屏障：保证 volatile 读之后的读不会被重排到 volatile 读之前
  - LoadStore 屏障：保证 volatile 读之后的写不会被重排到 volatile 读之前

---

### Q16: volatile 在单例模式双重检查锁中的应用？

**难度**: ⭐⭐⭐

**答案**: 在 DCL（Double-Checked Locking）单例模式中，`volatile` 修饰实例变量是为了防止指令重排序导致其他线程获取到未初始化完成的对象。

**解析**:
```java
public class Singleton {
    private static volatile Singleton instance; // 必须加 volatile

    public static Singleton getInstance() {
        if (instance == null) {                    // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {            // 第二次检查
                    instance = new Singleton();    // 问题出在这里
                }
            }
        }
        return instance;
    }
}
```

- `new Singleton()` 不是原子操作，分为三步：
  1. 分配内存空间
  2. 初始化对象
  3. 将引用指向内存地址
- 没有 volatile 时，步骤 2 和 3 可能被重排序为 1->3->2
- 线程 A 执行了 1->3，此时 instance 不为 null 但未初始化
- 线程 B 第一次检查 instance != null，直接返回未初始化的对象

## 四、Lock 与 AQS（6题）

### Q17: ReentrantLock 的核心特性？

**难度**: ⭐⭐

**答案**: ReentrantLock 是可重入的独占锁，支持公平锁/非公平锁、可中断、超时获取、多个条件变量。

**解析**:
```java
ReentrantLock lock = new ReentrantLock(true); // true = 公平锁

lock.lock(); // 获取锁
try {
    // 临界区
} finally {
    lock.unlock(); // 必须在 finally 中释放
}

// 可中断获取锁
lock.lockInterruptibly();

// 超时获取锁
if (lock.tryLock(5, TimeUnit.SECONDS)) {
    try {
        // 获取锁成功
    } finally {
        lock.unlock();
    }
}

// 多条件变量
Condition notEmpty = lock.newCondition();
Condition notFull = lock.newCondition();
notEmpty.await();       // 等待
notEmpty.signal();      // 唤醒一个
notEmpty.signalAll();   // 唤醒所有
```

- 公平锁：按等待时间排队获取锁，吞吐量低但避免饥饿
- 非公平锁（默认）：新来的线程直接尝试获取锁，失败再排队，吞吐量高

### Q18: AQS（AbstractQueuedSynchronizer）的原理？

**难度**: ⭐⭐⭐

**答案**: AQS 是 Java 并发包中锁和同步器的基础框架。核心思想是：如果共享资源被占用，其他线程进入**CLH 双向队列**等待，资源释放后唤醒队列中的线程。

**解析**:
- AQS 核心属性：
  - `volatile int state`：同步状态（0 表示空闲，>0 表示被占用）
  - `CLH 双向队列`：存储等待的线程
- 两种模式：
  - **独占模式**（Exclusive）：如 ReentrantLock
  - **共享模式**（Shared）：如 Semaphore、CountDownLatch

```
AQS 工作流程：
tryAcquire(state) 成功 -> 获取锁
tryAcquire(state) 失败 -> 加入 CLH 队列 -> 自旋/CAS -> park 等待
释放锁 -> unpark 唤醒队列头节点 -> 头节点 tryAcquire
```

- ReentrantLock、ReentrantReadWriteLock、Semaphore、CountDownLatch 都基于 AQS 实现

---

### Q19: ReentrantReadWriteLock 的原理？读写锁会产生饥饿吗？

**难度**: ⭐⭐⭐

**答案**: 读写锁将锁分为读锁（共享锁）和写锁（独占锁）。读读不互斥、读写互斥、写写互斥。

**解析**:
- 读锁：多个线程可以同时持有读锁
- 写锁：只有一个线程可以持有写锁，且获取写锁时不能有读锁
- 饥饿问题：如果读操作很多，写线程可能一直获取不到锁（读饥饿）
- 解决方案：使用公平锁，或使用 `StampedLock`

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();

// 读锁
rwLock.readLock().lock();
try {
    // 多个线程可以同时读
} finally {
    rwLock.readLock().unlock();
}

// 写锁
rwLock.writeLock().lock();
try {
    // 独占写
} finally {
    rwLock.writeLock().unlock();
}
```

### Q20: StampedLock 是什么？解决了什么问题？

**难度**: ⭐⭐⭐

**答案**: StampedLock 是 JDK 8 引入的高性能读写锁，支持**乐观读**，解决了读写锁的读饥饿问题。

**解析**:
- 三种模式：
  1. **写锁**（writeLock）：独占锁
  2. **读锁**（readLock）：悲观读锁，与写锁互斥
  3. **乐观读**（tryOptimisticRead）：不加锁，读取后通过 validate(stamp) 检查期间是否有写操作

```java
StampedLock sl = new StampedLock();

// 乐观读
long stamp = sl.tryOptimisticRead();
int current = data; // 读取数据
if (!sl.validate(stamp)) { // 检查期间是否有写操作
    stamp = sl.readLock(); // 升级为悲观读锁
    try {
        current = data;
    } finally {
        sl.unlockRead(stamp);
    }
}

// 写锁
long writeStamp = sl.writeLock();
try {
    data = newValue;
} finally {
    sl.unlockWrite(writeStamp);
}
```

- StampedLock 不可重入，不支持 Condition
- 适用于读多写少且读操作很快的场景

### Q21: LockSupport 的作用？

**难度**: ⭐⭐

**答案**: LockSupport 是 AQS 的基础工具类，提供 `park()` 和 `unpark()` 方法来阻塞和唤醒线程。

**解析**:
- `LockSupport.park()`：阻塞当前线程
- `LockSupport.unpark(thread)`：唤醒指定线程
- 与 wait/notify 的区别：
  - unpark 可以在 park 之前调用（"许可证"机制）
  - 不需要在同步块中使用
  - 可以精准唤醒指定线程

```java
// 先 unpark 再 park，不会阻塞
LockSupport.unpark(Thread.currentThread());
LockSupport.park(); // 不会阻塞，因为"许可证"已存在
```

### Q22: Condition 的作用？与 wait/notify 的区别？

**难度**: ⭐⭐

**答案**: Condition 是 Lock 配合使用的条件变量，支持创建多个等待队列，比 Object 的 wait/notify 更灵活。

**解析**:

| 对比项 | wait/notify | Condition |
|--------|------------|-----------|
| 使用前提 | 必须在 synchronized 中 | 必须在 Lock 中 |
| 等待队列数 | 1 个 | 可以创建多个 |
| 唤醒方式 | notify/notifyAll | signal/signalAll |
| 超时等待 | wait(ms) | await(ms) |
| 不响应中断 | 不支持 | awaitUninterruptibly() |

```java
Lock lock = new ReentrantLock();
Condition notEmpty = lock.newCondition();
Condition notFull = lock.newCondition();

// 生产者
lock.lock();
try {
    while (queue.isEmpty()) notEmpty.await();
    queue.add(item);
    notFull.signal();
} finally {
    lock.unlock();
}
```

## 五、CAS 与原子类（4题）

### Q23: CAS 的原理是什么？有什么问题？

**难度**: ⭐⭐⭐

**答案**: CAS（Compare-And-Swap）是一种无锁算法，通过 CPU 指令（如 `cmpxchg`）原子地比较并交换值。流程：读取当前值 V，与期望值 E 比较，如果相等则更新为新值 N，否则重试。

**解析**:
```
CAS(V, E, N): V 是内存中的值，E 是期望值，N 是新值
if (V == E) {
    V = N; // 更新成功
    return true;
} else {
    return false; // 更新失败，重试
}
```

CAS 的三个问题：
1. **ABA 问题**：值从 A 变为 B 又变回 A，CAS 认为没有变化
   - 解决：使用版本号（AtomicStampedReference）
2. **自旋开销**：长时间 CAS 失败会一直重试，浪费 CPU
   - 解决：限制自旋次数、使用 LongAdder
3. **只能保证单个变量的原子操作**：多个变量需要加锁
   - 解决：使用 AtomicReference 包装多个变量

---

### Q24: ABA 问题如何解决？

**难度**: ⭐⭐⭐

**答案**: 使用 `AtomicStampedReference`，在 CAS 的基础上增加版本号（stamp），每次更新版本号递增。

**解析**:
```java
AtomicStampedReference<String> ref = new AtomicStampedReference<>("A", 1);

int stamp = ref.getStamp(); // 获取版本号
String expected = ref.getReference();

// 线程B：A -> B -> A
ref.compareAndSet("A", "B", 1, 2); // stamp: 1 -> 2
ref.compareAndSet("B", "A", 2, 3); // stamp: 2 -> 3

// 线程A：CAS 失败，因为版本号已变化
boolean success = ref.compareAndSet(expected, "C", stamp, stamp + 1);
// expected="A" 匹配，但 stamp=1 不匹配当前 stamp=3，CAS 失败
```

### Q25: AtomicInteger 的原理？

**难度**: ⭐⭐

**答案**: AtomicInteger 基于 CAS（Unsafe.compareAndSwapInt）实现原子操作，内部维护一个 volatile int value。

**解析**:
```java
// incrementAndGet 源码
public final int incrementAndGet() {
    return U.getAndAddInt(this, VALUE, 1) + 1;
}

// Unsafe.getAndAddInt
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset); // 读取最新值
    } while (!compareAndSwapInt(o, offset, v, v + delta)); // CAS 自旋
    return v;
}
```

- 常用原子类：AtomicInteger、AtomicLong、AtomicBoolean、AtomicReference
- 数组原子类：AtomicIntegerArray、AtomicLongArray
- 引用原子类：AtomicReference、AtomicStampedReference、AtomicMarkableReference
- JDK 9+ 引入了 java.lang.invoke.VarHandle 作为 Unsafe 的更安全替代方案，提供标准化的原子操作

### Q26: LongAdder 比 AtomicLong 快在哪里？

**难度**: ⭐⭐⭐

**答案**: LongAdder 采用**分段 CAS**（Cell 数组）策略，将竞争分散到多个 Cell 上，最后求和。高并发下性能远优于 AtomicLong。

**解析**:
```
AtomicLong: 所有线程 CAS 竞争同一个 value -> 高竞争

LongAdder:
  base（无竞争时直接 CAS）
  Cell[0], Cell[1], Cell[2], ...（高竞争时线程分散到不同 Cell）
  sum() = base + sum(Cell[])  // 最终求和
```

- LongAdder 适合高并发计数场景（如统计 QPS）
- 缺点：sum() 不是强一致性的（调用时可能有其他线程在更新）

## 六、线程池（6题）

### Q27: ThreadPoolExecutor 的 7 大核心参数？

**难度**: ⭐⭐⭐

**答案**:

```java
public ThreadPoolExecutor(
    int corePoolSize,        // 核心线程数
    int maximumPoolSize,     // 最大线程数
    long keepAliveTime,      // 非核心线程空闲存活时间
    TimeUnit unit,           // 时间单位
    BlockingQueue<Runnable> workQueue,  // 任务队列
    ThreadFactory threadFactory,        // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)
```

**解析**:
- **corePoolSize**：即使空闲也不会被回收的线程数量
- **maximumPoolSize**：线程池允许的最大线程数
- **workQueue**：核心线程都在忙时，新任务放入队列
- **keepAliveTime**：非核心线程空闲超过此时间后被回收（JDK 8 后核心线程也可设置回收）
- **threadFactory**：自定义线程创建（设置线程名、优先级、守护线程等）
- **handler**：队列满且线程数达到最大时的拒绝策略

### Q28: 线程池的任务提交流程？

**难度**: ⭐⭐⭐

**答案**:
```
提交任务
  1. 如果当前线程数 < corePoolSize -> 创建核心线程执行
  2. 如果当前线程数 >= corePoolSize -> 加入 workQueue
  3. 如果 workQueue 已满 且 线程数 < maximumPoolSize -> 创建非核心线程执行
  4. 如果 workQueue 已满 且 线程数 >= maximumPoolSize -> 执行拒绝策略
```

**解析**:
- 核心线程创建后即使空闲也不会回收（除非设置 `allowCoreThreadTimeOut(true)`）
- 非核心线程在 `keepAliveTime` 后被回收
- 任务队列满的判断取决于队列类型：`ArrayBlockingQueue` 有界队列会满，`LinkedBlockingQueue` 默认无界不会满

### Q29: 4 种拒绝策略分别是什么？

**难度**: ⭐⭐

**答案**:

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| AbortPolicy（默认） | 抛出 RejectedExecutionException | 关键任务，需要感知异常 |
| CallerRunsPolicy | 由提交任务的线程执行 | 不丢弃任务，适合非关键任务 |
| DiscardPolicy | 静默丢弃任务 | 可容忍丢失的场景 |
| DiscardOldestPolicy | 丢弃队列中最老的任务 | 优先处理新任务 |

```java
// 自定义拒绝策略
new RejectedExecutionHandler() {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        // 记录日志、持久化到数据库、发送告警等
        log.warn("Task rejected, queue size: {}", executor.getQueue().size());
    }
}
```

### Q30: 为什么禁止使用 Executors 创建线程池？

**难度**: ⭐⭐

**答案**: `Executors` 提供的工厂方法使用无界队列（`LinkedBlockingQueue`）或 Integer.MAX_VALUE 的最大线程数，可能导致 OOM。

**解析**:
- `Executors.newFixedThreadPool(n)`：核心线程数 = 最大线程数 = n，队列是 `LinkedBlockingQueue`（默认容量 Integer.MAX_VALUE），任务堆积可能导致 OOM
- `Executors.newCachedThreadPool()`：最大线程数 = Integer.MAX_VALUE，高并发时可能创建大量线程导致 OOM
- `Executors.newSingleThreadExecutor()`：队列是 `LinkedBlockingQueue`（Integer.MAX_VALUE），同样有 OOM 风险

```java
// 推荐手动创建线程池
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10, 20, 60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),  // 有界队列
    new ThreadFactoryBuilder().setNameFormat("my-pool-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

### Q31: 如何合理设置线程池参数？

**难度**: ⭐⭐⭐

**答案**: 根据任务类型（CPU 密集型 / IO 密集型）设置不同的参数。

**解析**:
- **CPU 密集型**（加密、计算、压缩）：
  - 核心线程数：`CPU 核数 + 1`（Nacos 推荐公式）
  - 原因：CPU 密集型任务切换线程反而降低效率

- **IO 密集型**（网络请求、数据库查询、文件读写）：
  - 核心线程数：`CPU 核数 * 2` 或 `CPU 核数 / (1 - 阻塞系数)`
  - 原因：线程在等待 IO 时不占用 CPU，可以创建更多线程

- **混合型**：拆分为 CPU 密集型和 IO 密集型任务，分别使用不同线程池

```java
int cpuCores = Runtime.getRuntime().availableProcessors();

// CPU 密集型
int coreSize = cpuCores + 1;

// IO 密集型
int coreSize = cpuCores * 2;
// 或更精确：int coreSize = (int)(cpuCores / (1 - 0.8)); // 阻塞系数 0.8
```

### Q32: 线程池如何关闭？

**难度**: ⭐⭐

**答案**: 两种方式：`shutdown()`（优雅关闭）和 `shutdownNow()`（强制关闭）。

**解析**:
- `shutdown()`：
  - 停止接受新任务
  - 等待已提交的任务执行完毕
  - 不会中断正在执行的任务
- `shutdownNow()`：
  - 停止接受新任务
  - 尝试中断所有正在执行的任务
  - 返回未执行的任务列表

```java
executor.shutdown(); // 优雅关闭
try {
    if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
        executor.shutdownNow(); // 超时后强制关闭
    }
} catch (InterruptedException e) {
    executor.shutdownNow();
}
```

## 七、CompletableFuture（4题）

### Q33: CompletableFuture 的 thenApply 和 thenCompose 的区别？

**难度**: ⭐⭐⭐

**答案**: `thenApply` 接收同步函数，返回 `CompletableFuture<U>`；`thenCompose` 接收返回 `CompletableFuture` 的函数，用于**扁平化**异步链。

**解析**:
```java
// thenApply：同步转换（类似 Stream.map）
CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> 1)
    .thenApply(i -> i + 1)          // 同步：1 -> 2
    .thenApply(i -> "Result: " + i); // 同步：2 -> "Result: 2"

// thenCompose：异步扁平化（类似 Stream.flatMap）
CompletableFuture<String> future2 = CompletableFuture.supplyAsync(() -> 1)
    .thenCompose(i -> CompletableFuture.supplyAsync(() -> "Result: " + i));
// 适用于：转换函数本身返回 CompletableFuture 的场景
```

- 如果 `thenApply` 的转换函数本身返回 `CompletableFuture`，则会产生 `CompletableFuture<CompletableFuture<U>>` 嵌套，这时需要用 `thenCompose`（flatMap）来扁平化
- `thenCompose` 将嵌套的 Future 扁平化为 `CompletableFuture<U>`

### Q34: CompletableFuture 如何处理异常？

**难度**: ⭐⭐⭐

**答案**: 通过 `exceptionally()`、`handle()`、`whenComplete()` 处理异常。

**解析**:
```java
CompletableFuture.supplyAsync(() -> {
    if (Math.random() > 0.5) throw new RuntimeException("error");
    return "success";
})
// 方式1：exceptionally - 只处理异常，返回默认值
.exceptionally(ex -> {
    log.error("异常", ex);
    return "default value";
})
// 方式2：handle - 同时处理正常值和异常
.handle((result, ex) -> {
    if (ex != null) {
        return "default value";
    }
    return result;
})
// 方式3：whenComplete - 处理但不改变结果
.whenComplete((result, ex) -> {
    if (ex != null) log.error("异常", ex);
    else log.info("结果: {}", result);
});
```

### Q35: CompletableFuture 的 allOf 和 anyOf 的区别？

**难度**: ⭐⭐

**答案**: `allOf` 等待所有 Future 完成，`anyOf` 等待任意一个 Future 完成。

**解析**:
```java
CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> "A");
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> "B");
CompletableFuture<String> f3 = CompletableFuture.supplyAsync(() -> "C");

// allOf：等待所有完成
CompletableFuture<Void> all = CompletableFuture.allOf(f1, f2, f3);
all.join(); // 阻塞等待
// 获取所有结果
List<String> results = Stream.of(f1, f2, f3)
    .map(CompletableFuture::join)
    .collect(Collectors.toList());

// anyOf：等待任意一个完成
CompletableFuture<Object> any = CompletableFuture.anyOf(f1, f2, f3);
Object firstResult = any.join(); // 返回最先完成的结果
```

### Q36: CompletableFuture 的异步回调是在哪个线程执行的？

**难度**: ⭐⭐

**答案**: 如果不指定线程池，`thenApply` 等回调默认在**完成上一个任务的线程**中执行。如果指定了线程池，则在指定线程池中执行。

**解析**:
```java
// 不指定线程池：回调在完成上一个任务的线程中执行
CompletableFuture.supplyAsync(() -> {
    // 在 ForkJoinPool.commonPool() 线程中执行
    return fetchData();
}).thenApply(data -> {
    // 也在 ForkJoinPool 线程中执行（与上一个任务同线程）
    return process(data);
});

// 指定线程池：回调在指定线程池中执行
// 实际生产禁止使用 Executors 创建线程池，此处仅为演示
ExecutorService myPool = new ThreadPoolExecutor(4, 4, 0L, TimeUnit.MILLISECONDS, new LinkedBlockingQueue<>());
CompletableFuture.supplyAsync(() -> fetchData(), myPool)
    .thenApplyAsync(data -> process(data), myPool); // 在 myPool 中执行
```

## 八、并发工具类（4题）

### Q37: CountDownLatch 和 CyclicBarrier 的区别？

**难度**: ⭐⭐⭐

**答案**:

| 对比项 | CountDownLatch | CyclicBarrier |
|--------|---------------|---------------|
| 作用 | 让一个/多个线程等待其他线程完成 | 让一组线程互相等待，全部到达后一起继续 |
| 计数器 | 递减到 0 后不可重置 | 到达 0 后自动重置（可循环使用） |
| 线程角色 | 等待线程 vs 工作线程 | 所有线程都是参与者 |
| 回调 | 无 | 可指定 barrierAction |

**解析**:
```java
// CountDownLatch：主线程等待 3 个子线程完成
CountDownLatch latch = new CountDownLatch(3);
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doWork();
        latch.countDown(); // 计数 -1
    }).start();
}
latch.await(); // 主线程等待，直到计数为 0

// CyclicBarrier：3 个线程互相等待，全部到达后一起执行
CyclicBarrier barrier = new CyclicBarrier(3, () -> {
    System.out.println("所有线程到达屏障点");
});
for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        doWork();
        barrier.await(); // 等待其他线程
        // 所有线程到达后继续执行
    }).start();
}
```

### Q38: Semaphore 的使用场景？

**难度**: ⭐⭐

**答案**: Semaphore（信号量）用于控制同时访问共享资源的线程数量，常用于限流。

**解析**:
```java
// 限制最多 3 个线程同时访问
Semaphore semaphore = new Semaphore(3);

for (int i = 0; i < 10; i++) {
    new Thread(() -> {
        try {
            semaphore.acquire(); // 获取许可
            accessResource();   // 最多 3 个线程同时执行
        } finally {
            semaphore.release(); // 释放许可
        }
    }).start();
}
```

- `acquire()`/`release()`：获取/释放一个许可
- `tryAcquire()`：尝试获取，不阻塞
- 应用场景：数据库连接池限流、接口限流、停车场车位控制

### Q39: ThreadLocal 的原理？内存泄漏问题？

**难度**: ⭐⭐⭐

**答案**: ThreadLocal 为每个线程维护独立的变量副本。内部通过 `ThreadLocalMap`（key 是 ThreadLocal 的弱引用，value 是强引用）存储。

**解析**:
```java
// ThreadLocal 工作原理
Thread
  └── ThreadLocalMap
        ├── Entry(ThreadLocal<?> key, Object value)  // key 是弱引用
        ├── Entry(ThreadLocal<?> key, Object value)
        └── ...
```

- **内存泄漏原因**：ThreadLocalMap 的 key 是弱引用，GC 后 key 变为 null，但 value 是强引用不会被回收
- **解决方案**：使用完 ThreadLocal 后调用 `remove()` 方法

```java
ThreadLocal<User> userThreadLocal = new ThreadLocal<>();

try {
    userThreadLocal.set(currentUser);
    // 使用 userThreadLocal.get()
} finally {
    userThreadLocal.remove(); // 必须清理！
}
```

- ThreadLocal 的应用场景：数据库连接、Session 管理、日期格式化（SimpleDateFormat 非线程安全）、链路追踪 TraceId

### Q40: ForkJoinPool 的原理？

**难度**: ⭐⭐⭐

**答案**: ForkJoinPool 是 JDK 7 引入的并行计算框架，采用**工作窃取（Work-Stealing）**算法。大任务被拆分为小任务（Fork），小任务结果合并（Join）。

**解析**:
- 工作窃取：空闲线程从其他线程的任务队列尾部"窃取"任务执行
- 每个线程有自己的双端队列（Deque），支持 LIFO（自己执行）和 FIFO（被窃取）
- 适合**分治**场景：归并排序、大数据量计算、并行流

```java
// ForkJoinTask 使用示例
class SumTask extends RecursiveTask<Long> {
    private final long[] array;
    private final int start, end;

    protected Long compute() {
        if (end - start <= 1000) {
            // 小任务直接计算
            long sum = 0;
            for (int i = start; i < end; i++) sum += array[i];
            return sum;
        }
        // 大任务拆分
        int mid = (start + end) / 2;
        SumTask left = new SumTask(array, start, mid);
        SumTask right = new SumTask(array, mid, end);
        left.fork(); // 异步执行
        return right.compute() + left.join(); // 合并结果
    }
}
```
