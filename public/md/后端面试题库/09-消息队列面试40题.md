### Q1: 什么是消息队列？有哪些应用场景？ ⭐

**答案：**

消息队列（Message Queue，MQ）是一种应用程序之间的通信方法，应用程序通过读写队列中的消息来进行通信，而不需要直接调用对方。

**核心应用场景：**

1. **异步处理**：将耗时操作异步化，提升响应速度。如用户注册后发送邮件、短信通知。
2. **流量削峰**：在高并发场景下，将请求先放入队列，按消费能力逐步处理。如秒杀系统。
3. **应用解耦**：生产者和消费者不需要知道对方的存在，通过消息队列进行通信。如订单系统与库存系统解耦。
4. **日志处理**：大规模日志的异步收集与处理。
5. **分布式事务**：通过消息队列实现最终一致性。

**消息队列的缺点：**
- 系统可用性降低（MQ宕机影响业务）
- 系统复杂度增加（消息重复、丢失、顺序等问题）
- 一致性问题（分布式环境下难以保证强一致性）

### Q2: Kafka的基本架构是什么？ ⭐

**答案：**

Kafka 的核心架构由以下组件构成：

```
Producer → Broker(Topic/Partition) ← Consumer Group
                    ↑
               ZooKeeper / KRaft
```

**核心组件：**

1. **Broker**：Kafka服务节点，一个或多个Broker组成Kafka集群。
2. **Topic**：消息的逻辑分类，生产者将消息发送到指定Topic，消费者从Topic订阅消息。
3. **Partition**：Topic的物理分片，每个Topic可以有多个Partition，分布在不同Broker上，实现水平扩展。
4. **Producer**：消息生产者，负责将消息发布到Kafka Topic。
5. **Consumer**：消息消费者，从Kafka Topic拉取消息进行处理。
6. **Consumer Group**：消费者组，一个组内的消费者共同消费一个Topic的所有分区，每条消息只会被组内一个消费者消费。
7. **ZooKeeper / KRaft**：用于集群管理、Leader选举、元数据存储（Kafka 2.8+ 开始支持 KRaft 模式替代 ZooKeeper）。

**消息流转过程：**
```
Producer → 选择Partition → 写入Leader副本 → ISR同步 → Consumer拉取
```

### Q3: Kafka中Topic和Partition的关系是什么？为什么要分区？ ⭐

**答案：**

**关系：**
- Topic 是逻辑概念，Partition 是物理概念。
- 一个 Topic 包含一个或多个 Partition，每个 Partition 是一个有序的、不可变的日志序列。
- 每个 Partition 有一个 Leader 和多个 Follower，Leader 处理读写请求，Follower 从 Leader 同步数据。

**分区的原因：**

1. **水平扩展**：一个 Topic 的数据量可能非常大，单机无法存储，分区可以将数据分布到多个 Broker 上。
2. **并发处理**：不同分区可以并行读写，提高吞吐量。分区数决定了 Topic 的最大并行消费能力。
3. **负载均衡**：分区分布在不同 Broker 上，实现负载均衡。

**分区数选择建议：**
- 分区数一般设置为消费者数量的整数倍（通常3-10倍）。
- 分区数过多会增加延迟和管理开销（文件句柄、内存占用）。
- Kafka 3.x 推荐单集群分区数不超过 20 万。

### Q4: Kafka消息的存储机制是怎样的？ ⭐⭐

**答案：**

Kafka 消息以**追加写（Append-Only）**的方式存储在磁盘上的日志文件中。

**存储结构：**

```
/kafka-logs/
  └── topic_name-partition_id/
        ├── 00000000000000000000.log      # 消息日志文件
        ├── 00000000000000000000.index    # 偏移量索引
        ├── 00000000000000000000.timeindex # 时间戳索引
        └── leader-epoch-checkpoint
```

**日志段（Log Segment）：**
- 一个 Partition 的日志被划分为多个日志段（Segment），每个 Segment 包含 `.log`、`.index`、`.timeindex` 三个文件。
- 当 `.log` 文件大小达到 `log.segment.bytes`（默认1GB）或时间达到 `log.roll.hours`（默认168小时）时，会创建新的 Segment。
- 文件名是当前 Segment 第一条消息的 offset。

**消息查找过程：**
1. 通过二分查找 `.index` 文件，找到目标 offset 对应的物理位置。
2. 从 `.log` 文件的物理位置开始顺序扫描，找到目标消息。

**稀疏索引：**
- Kafka 的索引是稀疏索引，不是每条消息都有索引项。
- 默认每隔 `log.index.interval.bytes`（默认4KB）创建一个索引项，在内存占用和查找效率之间取得平衡。

### Q5: 什么是零拷贝（Zero-Copy）？Kafka如何利用零拷贝？ ⭐⭐

**答案：**

**零拷贝**是指避免 CPU 在内存之间进行数据拷贝的技术，减少上下文切换和 CPU 拷贝次数。

**传统数据传输（4次拷贝 + 4次上下文切换）：**
```
磁盘 → 内核缓冲区 → 用户缓冲区 → Socket缓冲区 → 网卡
```

**零拷贝方式（2次拷贝 + 2次上下文切换）：**
```
磁盘 → 内核缓冲区 → 网卡（通过DMA直接传输）
```

**Kafka 使用的零拷贝技术：**

1. **sendfile 系统调用**（Linux 2.1+）：
   - Kafka 消费者拉取消息时，Broker 通过 `sendfile()` 将磁盘文件直接发送到网络 Socket。
   - 数据从文件系统页缓存（Page Cache）直接传输到网卡缓冲区，完全在内核空间完成。

```java
// Java中对应 FileChannel.transferTo()
FileChannel fileChannel = new RandomAccessFile(file, "r").getChannel();
fileChannel.transferTo(position, count, socketChannel);
```

2. **mmap 内存映射**：
   - Kafka 将索引文件通过 `mmap()` 映射到内存，避免将文件内容拷贝到用户空间。
   - 适用于索引文件的读取。

**零拷贝带来的性能提升：**
- 减少 CPU 拷贝次数，降低 CPU 占用。
- 减少用户态和内核态之间的上下文切换。
- 充分利用 DMA（Direct Memory Access）技术。

### Q6: Kafka的副本机制是怎样的？ISR是什么？ ⭐⭐

**答案：**

**副本机制：**
- 每个 Partition 有多个副本，分为 Leader 副本和 Follower 副本。
- Leader 处理所有的读写请求，Follower 不处理客户端请求，只从 Leader 同步数据。
- 当 Leader 宕机时，从 Follower 中选举新的 Leader。

**ISR（In-Sync Replicas，同步副本集合）：**
- ISR 是与 Leader 保持同步的副本集合。
- Follower 通过向 Leader 发送 Fetch Request 来拉取数据，如果 Follower 在 `replica.lag.time.max.ms`（默认30秒）内完成同步，就保持在 ISR 中。
- 如果 Follower 落后太多（落后时间超过阈值或落后消息数超过阈值），就会被踢出 ISR，进入 OSR（Out-of-Sync Replicas）。

```
All Replicas = ISR + OSR
```

**ISR 的作用：**
- ISR 中的副本都是数据最新的副本，Leader 选举优先从 ISR 中选择。
- `acks=all` 时，消息需要写入所有 ISR 副本才算发送成功。

**相关配置：**
- `min.insync.replicas`：最小同步副本数，配合 `acks=all` 使用，保证数据可靠性。
- `unclean.leader.election.enable`：是否允许非 ISR 副本成为 Leader（默认false，建议保持false）。

### Q7: Kafka的Leader选举流程是怎样的？ ⭐⭐⭐

**答案：**

Kafka 的 Leader 选举分为两种场景：

**1. 分区重新分配时的 Controller 选举：**
- Kafka 集群中有一个 Broker 作为 Controller（通过 ZooKeeper 临时节点选举）。
- Controller 负责管理分区和副本的状态，包括 Leader 选举。

**2. 分区 Leader 选举（Broker 宕机时）：**

```
步骤：
1. Controller 监听到 Broker 宕机（ZooKeeper Session 超时）
2. Controller 找出该 Broker 上所有 Leader 分区
3. 对每个分区执行 Leader 选举：
   a. 从 ISR 列表中按顺序选择第一个存活的副本作为新 Leader
   b. 如果 ISR 为空且 unclean.leader.election.enable=true，从 OSR 中选择
   c. 如果 ISR 为空且 unclean.leader.election.enable=false，该分区不可用
4. Controller 将新的 Leader 信息更新到 ZooKeeper
5. 通知所有相关 Broker 更新元数据
```

**OfflinePartitionLeaderElectionStrategy（Kafka 2.4+）：**
- `ELECTED_FROM_ISR`（默认）：只从 ISR 中选举，保证数据不丢失。
- `OLDEST_FIRST`：从 OSR 中选择最老的副本，减少数据丢失但可能不一致。

**KRaft 模式（Kafka 3.x+）：**
- 使用 KRaft 协议替代 ZooKeeper，Controller 通过 Raft 协议选举。
- 元数据存储在内部 Topic `__cluster_metadata` 中。

### Q8: Kafka生产者的acks参数有哪些取值？分别代表什么含义？ ⭐⭐

**答案：**

`acks` 参数控制生产者发送消息时的可靠性级别：

| 取值 | 含义 | 可靠性 | 吞吐量 |
|------|------|--------|--------|
| `acks=0` | 生产者发送消息后不等Broker响应 | 最低 | 最高 |
| `acks=1` | 等待Leader写入成功后返回响应 | 中等 | 中等 |
| `acks=all`（或`acks=-1`） | 等待ISR中所有副本写入成功后返回响应 | 最高 | 最低 |

**详细说明：**

```java
props.put("acks", "all");
// 配合以下参数使用
props.put("min.insync.replicas", "2");  // ISR中最少同步副本数
props.put("retries", Integer.MAX_VALUE); // 重试次数
props.put("enable.idempotence", "true"); // 开启幂等
```

**acks=0 的风险：** 消息可能丢失，适用于日志收集等允许丢失的场景。

**acks=1 的风险：** Leader 写入成功但 Follower 尚未同步时 Leader 宕机，消息会丢失。

**acks=all 的注意事项：**
- 必须配合 `min.insync.replicas` 使用（建议设为2或更大）。
- 如果 ISR 中副本数少于 `min.insync.replicas`，会抛出 `NotEnoughReplicasException`。
- 建议同时开启幂等性（`enable.idempotence=true`）。

### Q9: Kafka如何保证消息不丢失？ ⭐⭐⭐

**答案：**

消息可能在三个阶段丢失：生产者发送阶段、Broker 存储阶段、消费者消费阶段。

**1. 生产者端防丢失：**

```java
Properties props = new Properties();
props.put("acks", "all");                    // 所有ISR副本确认
props.put("retries", Integer.MAX_VALUE);     // 无限重试
props.put("max.in.flight.requests.per.connection", 5); // 开启幂等时最大5
props.put("enable.idempotence", "true");     // 开启幂等性
props.put("delivery.timeout.ms", 120000);    // 投递超时时间
props.put("batch.size", 16384);              // 批量发送
props.put("linger.ms", 10);                  // 等待时间
```

**2. Broker 端防丢失：**

```properties
# server.properties
min.insync.replicas=2           # ISR最小同步副本数
unclean.leader.election.enable=false  # 禁止非ISR副本成为Leader
default.replication.factor=3    # 默认副本数
log.flush.interval.messages=10000  # 消息刷盘间隔（建议依赖OS刷盘）
```

**3. 消费者端防丢失：**

```java
// 手动提交offset，确保消息处理完成后再提交
props.put("enable.auto.commit", "false");

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));
    for (ConsumerRecord<String, String> record : records) {
        // 1. 处理消息
        processMessage(record);
        // 2. 处理成功后再提交offset
    }
    consumer.commitSync(); // 同步提交
}
```

**最佳实践总结：**
- 生产者：`acks=all` + 幂等性 + 重试
- Broker：`replication.factor >= 3` + `min.insync.replicas >= 2`
- 消费者：关闭自动提交 + 手动提交 + 幂等消费

### Q10: Kafka的幂等性和事务机制是什么？ ⭐⭐⭐

**答案：**

**幂等性（Idempotence）：**

Kafka 0.11+ 引入了幂等生产者，解决生产者重试导致的消息重复问题。

```java
props.put("enable.idempotence", "true");  // 开启幂等性（默认true，Kafka 3.0+）
```

**原理：**
- 每个 Producer 初始化时分配一个 `PID（Producer ID）`。
- 每条消息携带 `Sequence Number`，Broker 为每个 `<PID, Partition>` 维护序列号。
- 如果 Broker 收到序列号小于等于当前序列号的消息，直接丢弃（重复消息）。
- 如果序列号不连续（中间有消息丢失），抛出异常。

**限制：** 幂等性只能保证**单分区单会话**内的 Exactly-Once，跨分区或跨会话需要事务。

**事务（Transaction）：**

Kafka 0.11+ 引入了事务机制，保证跨分区跨会话的 Exactly-Once 语义。

```java
// 初始化事务
props.put("transactional.id", "my-transactional-id");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();

try {
    producer.beginTransaction();
    // 向多个分区发送消息
    producer.send(new ProducerRecord<>("topic1", "key1", "value1"));
    producer.send(new ProducerRecord<>("topic2", "key2", "value2"));
    // 提交事务
    producer.commitTransaction();
} catch (ProducerFencedException | OutOfOrderSequenceException e) {
    producer.close();
} catch (KafkaException e) {
    // 中止事务
    producer.abortTransaction();
}
```

**事务原理：**
- 使用 `__transaction_state` 内部 Topic 存储事务状态。
- 事务协调器（Transaction Coordinator）管理事务生命周期。
- 消费者需要设置 `isolation.level=read_committed`，只读取已提交的消息。

**事务状态流转：**
```
Empty → Ongoing → PrepareCommit → CompleteCommit
Empty → Ongoing → PrepareAbort → CompleteAbort
```

### Q11: 什么是Exactly-Once语义？Kafka如何实现？ ⭐⭐⭐

**答案：**

**消息传递语义三种级别：**

| 语义 | 含义 | 场景 |
|------|------|------|
| At-Most-Once | 最多一次，消息可能丢失但不会重复 | 日志收集 |
| At-Least-Once | 至少一次，消息不会丢失但可能重复 | 大多数默认配置 |
| Exactly-Once | 恰好一次，消息不丢失也不重复 | 金融交易 |

**Kafka 实现 Exactly-Once 的方式：**

1. **幂等生产者**：单分区单会话内的 Exactly-Once。
2. **事务机制**：跨分区跨会话的 Exactly-Once（仅限 Kafka 到 Kafka）。
3. **Kafka Streams + 事务**：消费-处理-生产一体化事务。

**Kafka 事务的局限性：**
- Kafka 事务只能保证 Kafka 内部的 Exactly-Once。
- 如果消费 Kafka 消息后写入外部系统（如数据库），需要外部系统配合实现端到端的 Exactly-Once。

**端到端 Exactly-Once 方案：**

```
方案1：Kafka事务 + 外部系统事务（两阶段提交）
方案2：幂等消费 + 唯一约束（推荐）
  - 消费者处理消息时，使用消息的唯一ID作为数据库唯一约束
  - 重复消费时数据库唯一约束会拒绝插入，实现幂等
方案3：Kafka Connect + Exactly-Once Support
```

### Q12: Kafka如何提高发送性能？ ⭐⭐

**答案：**

**1. 批量发送（Batching）：**

```java
props.put("batch.size", 32768);   // 批次大小（字节），默认16KB
props.put("linger.ms", 20);       // 等待时间（毫秒），默认0
// 两个条件满足其一即发送：批次满 或 等待时间到
```

**2. 压缩（Compression）：**

```java
props.put("compression.type", "lz4");  // none/gzip/snappy/lz4/zstd
// 推荐：lz4（压缩速度快）或 zstd（压缩比高，Kafka 2.1+）
```

| 压缩算法 | 压缩比 | 压缩速度 | 解压速度 |
|----------|--------|----------|----------|
| none | 1:1 | - | - |
| gzip | 高 | 慢 | 快 |
| snappy | 中 | 快 | 快 |
| lz4 | 中高 | 很快 | 很快 |
| zstd | 最高 | 中 | 快 |

**3. 缓冲区调优：**

```java
props.put("buffer.memory", 67108864);  // 发送缓冲区大小，默认64MB
props.put("max.block.ms", 60000);      // 缓冲区满时的最大阻塞时间
```

**4. 异步发送：**

```java
// 异步发送 + 回调
producer.send(record, (metadata, exception) -> {
    if (exception != null) {
        // 处理发送失败
    }
});
```

**5. 分区策略优化：**
- 合理设置分区数，避免热点分区。
- 自定义分区器，根据业务 Key 均匀分布。

### Q13: Kafka的分区策略有哪些？ ⭐⭐

**答案：**

**Kafka 生产者发送消息时的分区选择逻辑：**

```
1. 指定了分区 → 直接使用指定分区
2. 没有指定分区但有Key → 对Key进行hash，对分区数取模
3. 没有指定分区也没有Key → 使用黏性分区（Sticky Partitioner）
```

**1. 指定分区：**
```java
new ProducerRecord<>("topic", 0, "key", "value");  // 指定分区0
```

**2. 按 Key 哈希分区（保证相同Key的消息进入同一分区）：**
```java
new ProducerRecord<>("topic", "key", "value");  // 按key hash分区
```

**3. 黏性分区（Kafka 2.4+）：**
- 无 Key 时，随机选择一个分区，尽量将批次填满后再切换分区。
- 相比轮询策略，减少了频繁切换分区带来的请求次数。

**4. 自定义分区器：**

```java
public class CustomPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
        int numPartitions = partitions.size();
        // 自定义分区逻辑：例如按用户ID分区
        if (key instanceof String) {
            return Math.abs(key.hashCode()) % numPartitions;
        }
        return ThreadLocalRandom.current().nextInt(numPartitions);
    }

    @Override
    public void close() {}

    @Override
    public void configure(Map<String, ?> configs) {}
}

// 使用自定义分区器
props.put("partitioner.class", "com.example.CustomPartitioner");
```

### Q14: Kafka消费者的Offset管理机制是怎样的？ ⭐⭐

**答案：**

**Offset 概念：**
- Offset 是消息在 Partition 中的唯一标识（偏移量），是一个递增的64位整数。
- 消费者通过 Offset 来跟踪消费进度。

**Offset 存储位置（版本差异）：**

| Kafka 版本 | Offset 存储位置 |
|------------|----------------|
| 0.9 之前 | ZooKeeper（`/consumers/<group>/offsets/<topic>/<partition>`）|
| 0.9+ | Kafka 内部 Topic `__consumer_offsets` |

**Offset 提交方式：**

```java
// 1. 自动提交（默认）
props.put("enable.auto.commit", "true");
props.put("auto.commit.interval.ms", "5000");  // 每5秒自动提交

// 2. 同步手动提交
consumer.commitSync();  // 处理完消息后同步提交

// 3. 异步手动提交
consumer.commitAsync((offsets, exception) -> {
    if (exception != null) {
        // 处理提交失败
    }
});

// 4. 指定分区和offset提交
Map<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();
offsets.put(new TopicPartition("topic", 0), new OffsetAndMetadata(lastOffset + 1));
consumer.commitSync(offsets);
```

**Offset 提交最佳实践：**
- 关闭自动提交，使用手动提交。
- 先处理消息，再提交 Offset。
- 结合业务幂等性，防止重复消费。

**指定 Offset 消费：**
```java
// 从最早的消息开始消费
consumer.seekToBeginning(Collections.singleton(new TopicPartition("topic", 0)));

// 从指定offset开始消费
consumer.seek(new TopicPartition("topic", 0), 1000);

// 从最新消息开始消费
consumer.seekToEnd(Collections.singleton(new TopicPartition("topic", 0)));
```

### Q15: 什么是Consumer Rebalance？如何避免不必要的Rebalance？ ⭐⭐⭐

**答案：**

**Rebalance 概念：**
- 当消费者组成员发生变化（新消费者加入、消费者离开、消费者宕机）时，Kafka 会重新分配分区与消费者的对应关系。
- Rebalance 期间消费者无法消费消息，会导致短暂的消费暂停。

**触发 Rebalance 的条件：**
1. 消费者组成员变化（加入/离开/崩溃）。
2. 订阅的 Topic 数量变化。
3. 订阅的 Topic 分区数变化。

**Rebalance 协议（Kafka 2.4+）：**

| 协议 | 版本 | 特点 |
|------|------|------|
| Eager | 0.11+ | 所有消费者先放弃所有分区，再重新分配（Stop-The-World） |
| Cooperative Sticky | 2.4+ | 增量Rebalance，只重新分配需要移动的分区 |

```java
// 使用 Cooperative Sticky 协议
props.put("partition.assignment.strategy",
    "org.apache.kafka.clients.consumer.CooperativeStickyAssignor");
```

**避免不必要 Rebalance 的方法：**

```properties
# 1. 调大心跳间隔和会话超时
session.timeout.ms=30000          # 会话超时（默认10s），建议25-30s
heartbeat.interval.ms=10000       # 心跳间隔（默认3s），建议 session.timeout.ms 的 1/3
max.poll.interval.ms=600000       # 最大处理时间（默认5min），根据业务调整

# 2. 合理设置 max.poll.records
max.poll.records=500              # 每次拉取的最大消息数

# 3. 避免消费者处理时间过长
#    如果处理时间可能超过 max.poll.interval.ms，考虑：
#    - 使用异步处理
#    - 增大 max.poll.interval.ms
#    - 减小 max.poll.records
```

**Rebalance 监听：**
```java
consumer.subscribe(Arrays.asList("topic"), new ConsumerRebalanceListener() {
    @Override
    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
        // 分区被回收前提交offset
        consumer.commitSync();
    }

    @Override
    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
        // 分区分配后可以做一些初始化工作
    }
});
```

### Q16: RabbitMQ的基本架构是什么？Exchange有哪些类型？ ⭐

**答案：**

**RabbitMQ 核心组件：**

```
Producer → Exchange → Binding → Queue → Consumer
                        ↑
                    Binding Key
```

1. **Producer**：消息生产者，将消息发送到 Exchange。
2. **Exchange**：交换机，接收生产者消息并根据路由规则将消息路由到一个或多个 Queue。
3. **Queue**：消息队列，存储消息，消费者从 Queue 获取消息。
4. **Binding**：绑定关系，Exchange 和 Queue 之间的关联，包含路由键。
5. **Consumer**：消息消费者，从 Queue 获取消息进行处理。
6. **Connection**：TCP 连接。
7. **Channel**：信道，复用 TCP 连接，减少连接开销。

**Exchange 类型：**

| 类型 | 路由规则 | 使用场景 |
|------|----------|----------|
| Direct | 精确匹配 routing key | 点对点消息 |
| Fanout | 广播到所有绑定的 Queue | 广播通知 |
| Topic | 通配符匹配 routing key（`*`匹配一个词，`#`匹配零或多个词） | 多条件路由 |
| Headers | 匹配消息头属性 | 复杂条件路由 |

**Exchange 类型示例：**

```java
// Direct Exchange
channel.exchangeDeclare("direct.ex", "direct");
channel.basicPublish("direct.ex", "order.create", null, message.getBytes());

// Fanout Exchange
channel.exchangeDeclare("fanout.ex", "fanout");
channel.basicPublish("fanout.ex", "", null, message.getBytes());

// Topic Exchange
channel.exchangeDeclare("topic.ex", "topic");
channel.basicPublish("topic.ex", "order.create.success", null, message.getBytes());
// 绑定：order.*.success 匹配 order.create.success
```

### Q17: RabbitMQ如何保证消息不丢失？ ⭐⭐

**答案：**

RabbitMQ 消息可能在三个阶段丢失：生产者发送、Broker 存储、消费者消费。

**1. 生产者端 - Confirm 模式：**

```java
// 开启 Confirm 模式
channel.confirmSelect();

// 异步确认
channel.addConfirmListener(new ConfirmListener() {
    @Override
    public void handleAck(long deliveryTag, boolean multiple) {
        // 消息成功到达Broker
    }

    @Override
    public void handleNack(long deliveryTag, boolean multiple) {
        // 消息未到达Broker，需要重发
    }
});

// 发送消息
channel.basicPublish("exchange", "routingKey", null, message.getBytes());
```

**2. 生产者端 - Return 模式（消息不可路由时）：**

```java
channel.addReturnListener((replyCode, replyText, exchange, routingKey, properties, body) -> {
    // 消息无法路由到队列时的回调
    log.warn("Message returned: {}", new String(body));
});

// 发送时设置 mandatory=true
channel.basicPublish("exchange", "routingKey", true, null, message.getBytes());
```

**3. Broker 端 - 持久化：**

```java
// Queue持久化
channel.queueDeclare("queue_name", true, false, false, null);

// Exchange持久化
channel.exchangeDeclare("exchange_name", "direct", true);

// Message持久化
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .deliveryMode(2)  // 持久化
    .build();
channel.basicPublish("exchange", "routingKey", props, message.getBytes());
```

**4. 消费者端 - 手动ACK：**

```java
// 关闭自动ACK
boolean autoAck = false;
channel.basicConsume("queue_name", autoAck, new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag, Envelope envelope,
                               AMQP.BasicProperties properties, byte[] body) {
        try {
            processMessage(body);
            // 处理成功后手动ACK
            channel.basicAck(envelope.getDeliveryTag(), false);
        } catch (Exception e) {
            // 处理失败，拒绝消息并重新入队
            channel.basicNack(envelope.getDeliveryTag(), false, true);
        }
    }
});
```

### Q18: RabbitMQ的死信队列是什么？如何使用？ ⭐⭐

**答案：**

**死信（Dead Letter）：** 消息在以下情况会变成死信：
1. 消息被拒绝（`basic.reject` / `basic.nack`），且 `requeue=false`。
2. 消息 TTL 过期。
3. 队列达到最大长度。

**死信队列（DLX，Dead Letter Exchange）：** 当消息变成死信后，会被路由到死信交换机，再由死信交换机路由到死信队列。

**配置死信队列：**

```java
// 1. 声明死信交换机和死信队列
channel.exchangeDeclare("dlx.exchange", "direct", true);
channel.queueDeclare("dlx.queue", true, false, false, null);
channel.queueBind("dlx.queue", "dlx.exchange", "dlx.routingKey");

// 2. 声明业务队列，并指定死信交换机
Map<String, Object> args = new HashMap<>();
args.put("x-dead-letter-exchange", "dlx.exchange");           // 死信交换机
args.put("x-dead-letter-routing-key", "dlx.routingKey");      // 死信路由键
args.put("x-message-ttl", 60000);                              // 消息TTL 60秒
args.put("x-max-length", 10000);                               // 队列最大长度

channel.queueDeclare("business.queue", true, false, false, args);
channel.queueBind("business.queue", "business.exchange", "business.routingKey");
```

**死信队列的应用场景：**
- 消息消费失败后的重试机制。
- 延迟队列实现（通过 TTL + DLX）。
- 异常消息的集中处理和告警。

### Q19: RabbitMQ如何实现延迟队列？ ⭐⭐

**答案：**

**方案1：TTL + 死信队列（常用方案）：**

```java
// 延迟队列实现步骤：
// 1. 创建死信交换机和死信队列
channel.exchangeDeclare("delay.dlx.exchange", "direct", true);
channel.queueDeclare("delay.dlx.queue", true, false, false, null);
channel.queueBind("delay.dlx.queue", "delay.dlx.exchange", "delay.process");

// 2. 创建延迟队列（不消费，只等待TTL过期后转到死信队列）
Map<String, Object> args = new HashMap<>();
args.put("x-dead-letter-exchange", "delay.dlx.exchange");
args.put("x-dead-letter-routing-key", "delay.process");
args.put("x-message-ttl", 30000);  // 30秒延迟

channel.queueDeclare("delay.queue.30s", true, false, false, args);

// 3. 发送消息到延迟队列
channel.basicPublish("", "delay.queue.30s", null, message.getBytes());

// 4. 消费者从死信队列消费（30秒后才能收到消息）
channel.basicConsume("delay.dlx.queue", false, new DefaultConsumer(channel) {
    @Override
    public void handleDelivery(String consumerTag, Envelope envelope,
                               AMQP.BasicProperties properties, byte[] body) {
        // 处理延迟消息
        channel.basicAck(envelope.getDeliveryTag(), false);
    }
});
```

**方案2：RabbitMQ 延迟消息插件（rabbitmq_delayed_message_exchange）：**

```java
// 安装插件后，使用 x-delayed-message 类型的 Exchange
Map<String, Object> args = new HashMap<>();
args.put("x-delayed-type", "direct");
channel.exchangeDeclare("delayed.exchange", "x-delayed-message", true, false, args);

// 发送延迟消息
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .headers(Map.of("x-delay", 30000))  // 延迟30秒
    .build();
channel.basicPublish("delayed.exchange", "routingKey", props, message.getBytes());
```

**方案对比：**
- TTL + DLX：不需要插件，但需要为每种延迟时间创建一个队列。
- 延迟消息插件：灵活，每条消息可以设置不同的延迟时间，但需要安装插件。

### Q20: RabbitMQ如何保证消息的顺序性？ ⭐⭐

**答案：**

RabbitMQ 本身不保证消息的全局顺序性，但可以通过以下方式保证**局部顺序性**：

**方案1：单消费者 + 单队列**

```
Producer → Queue（只有一个消费者） → Consumer
```

- 最简单的方式，但吞吐量低。

**方案2：多个队列 + 相同Key路由到同一队列**

```java
// 根据业务Key（如订单ID）hash到固定队列
int queueIndex = Math.abs(orderId.hashCode()) % QUEUE_COUNT;
String queueName = "order.queue." + queueIndex;

// 每个队列只有一个消费者，保证同一订单的消息顺序消费
channel.basicPublish("", queueName, null, message.getBytes());
```

**方案3：使用 Consistent Hashing Exchange 插件**

```java
// 使用一致性哈希交换机，相同routing key的消息路由到同一队列
channel.exchangeDeclare("consistent.hash.ex", "x-consistent-hash", true);
channel.basicPublish("consistent.hash.ex", orderId, null, message.getBytes());
```

**注意事项：**
- RabbitMQ 的顺序性保证是**队列级别**的，不是全局的。
- 如果需要全局顺序，只能使用单队列单消费者。
- 消费者处理失败时，如果使用 `requeue=true`，消息会回到队列头部，可能影响顺序。

### Q21: RocketMQ的基本架构是什么？ ⭐⭐

**答案：**

**RocketMQ 核心架构：**

```
                    NameServer（集群）
                   /        |        \
          Broker1(Master)  Broker2(Master)  Broker3(Master)
          Broker1(Slave)   Broker2(Slave)   Broker3(Slave)
               ↑               ↑               ↑
            Producer        Consumer         Consumer
```

**核心组件：**

1. **NameServer**：
   - 注册中心，管理 Broker 的路由信息。
   - 每个 NameServer 是独立的，不互相通信（与 ZooKeeper 不同）。
   - Broker 定时向所有 NameServer 注册路由信息。
   - Producer/Consumer 从 NameServer 获取路由信息。

2. **Broker**：
   - 消息存储和转发，Master-Slave 架构。
   - Master 负责读写，Slave 负责备份（可配置读写分离）。
   - 支持同步/异步刷盘、同步/异步复制。

3. **Producer**：
   - 消息生产者，支持多种发送方式（同步/异步/单向）。
   - 支持事务消息。

4. **Consumer**：
   - 消息消费者，支持 Push 和 Pull 两种消费模式。
   - 支持集群消费和广播消费。

**与 Kafka 架构对比：**

| 特性 | RocketMQ | Kafka |
|------|----------|-------|
| 注册中心 | NameServer（无状态） | ZooKeeper / KRaft |
| 消息模型 | 主题+队列（1主多从） | 主题+分区（多副本） |
| 消费模式 | Push/Pull | Pull |
| 事务消息 | 支持 | 支持（Kafka 0.11+） |
| 延迟消息 | 原生支持（18个延迟级别） | 不支持 |
| 消息回溯 | 支持 | 支持 |
| 适用场景 | 电商、金融 | 日志、大数据 |

### Q22: RocketMQ事务消息的实现原理是什么？ ⭐⭐⭐

**答案：**

**事务消息流程：**

```
步骤1: Producer 发送半消息（Half Message）到 Broker
步骤2: Broker 存储半消息（对Consumer不可见）
步骤3: Broker 返回确认
步骤4: Producer 执行本地事务
步骤5a: 本地事务成功 → 发送 Commit → Broker 投递消息给 Consumer
步骤5b: 本地事务失败 → 发送 Rollback → Broker 删除半消息
步骤5c: 网络异常 → Broker 发起回查 → Producer 返回事务状态
```

**代码示例：**

```java
// 1. 创建事务监听器
TransactionListener transactionListener = new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 执行本地事务
        try {
            // 业务逻辑：如创建订单
            orderService.createOrder(msg.getBody());
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 事务回查：检查本地事务状态
        Order order = orderService.queryByTransactionId(msg.getTransactionId());
        if (order != null && order.getStatus() == 1) {
            return LocalTransactionState.COMMIT_MESSAGE;
        }
        return LocalTransactionState.ROLLBACK_MESSAGE;
    }
};

// 2. 创建事务生产者
TransactionMQProducer producer = new TransactionMQProducer("transaction-producer-group");
producer.setTransactionListener(transactionListener);
producer.start();

// 3. 发送事务消息
Message msg = new Message("order-topic", "order-tag", "orderKey", orderJson.getBytes());
TransactionSendResult result = producer.sendMessageInTransaction(msg, null);
```

**半消息存储机制：**
- 半消息存储在 `RMQ_SYS_TRANS_HALF_TOPIC` 中。
- Commit 后的消息会存储到真实 Topic，并从半消息 Topic 中删除。
- Rollback 后的消息直接从半消息 Topic 中删除。
- 如果 Broker 未收到 Commit/Rollback，会定期回查（默认回查15次，间隔60秒）。

### Q23: RocketMQ的顺序消息如何实现？ ⭐⭐

**答案：**

**RocketMQ 顺序消息原理：**
- 利用 MessageQueue 的 FIFO 特性，相同 Key 的消息发送到同一个 Queue。
- 消费者对同一个 Queue 只使用一个线程消费。

**代码示例：**

```java
// === 生产者：顺序发送 ===
DefaultMQProducer producer = new DefaultMQProducer("order-producer-group");
producer.start();

// 相同订单ID的消息发送到同一个Queue
for (OrderStep step : orderSteps) {
    Message msg = new Message("order-topic", "order-tag",
        step.getOrderId(), step.getJson().getBytes());
    // 使用 hashKey 选择 Queue，保证相同 orderId 的消息进入同一 Queue
    SendResult result = producer.send(msg, new MessageQueueSelector() {
        @Override
        public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
            String orderId = (String) arg;
            int index = Math.abs(orderId.hashCode()) % mqs.size();
            return mqs.get(index);
        }
    }, step.getOrderId());
}

// === 消费者：顺序消费 ===
DefaultMQPushConsumer consumer = new DefaultMQPushConsumer("order-consumer-group");
consumer.subscribe("order-topic", "order-tag");

// 注册顺序消息监听器（单线程消费）
consumer.registerMessageListener(new MessageListenerOrderly() {
    @Override
    public ConsumeOrderlyStatus consumeMessage(List<MessageExt> msgs,
                                               ConsumeOrderlyContext context) {
        for (MessageExt msg : msgs) {
            // 顺序处理消息
            processOrderStep(msg);
        }
        return ConsumeOrderlyStatus.SUCCESS;
    }
});

consumer.start();
```

**注意事项：**
- `MessageListenerOrderly` 保证单 Queue 内的顺序消费。
- 如果消费失败返回 `SUSPEND_CURRENT_QUEUE_A_MOMENT`，会暂停当前 Queue 一段时间后重试。
- 顺序消息的吞吐量低于并发消费，需要权衡。

### Q25: 如何处理消息积压问题？ ⭐⭐⭐

**答案：**

**消息积压的原因：**
1. 消费者处理速度慢于生产速度。
2. 消费者出现异常或重启。
3. 突发流量高峰。
4. 消费者 Rebalance 导致消费暂停。

**处理方案：**

**方案1：紧急扩容消费者**

```
步骤：
1. 临时增加 Partition 数量（Kafka）
2. 部署多个消费者实例
3. 消费完成后恢复原有配置
```

```java
// Kafka 增加分区
adminClient.createPartitions(Collections.singletonMap("topic",
    NewPartitions.increaseTo(20)));
```

**方案2：临时消费者转发**

```
步骤：
1. 创建一个临时 Topic，分区数设为积压量的合理值
2. 编写一个临时消费者程序，从积压队列快速拉取消息
3. 将消息转发到临时 Topic（不做业务处理，只转发）
4. 启动多个消费者从临时 Topic 消费处理
5. 处理完成后删除临时 Topic
```

**方案3：消费者性能优化**

```java
// Kafka 消费者优化
props.put("max.poll.records", 1000);          // 增加每次拉取的消息数
props.put("fetch.min.bytes", 1024 * 1024);    // 最小拉取字节数
props.put("fetch.max.wait.ms", 500);          // 最大等待时间

// 异步处理：消费者只做消息接收，业务逻辑异步处理
consumer.subscribe(Arrays.asList("topic"), new ConsumerRebalanceListener() {
    @Override
    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {}
    @Override
    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {}
});

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        // 提交到线程池异步处理
        executorService.submit(() -> processMessage(record));
    }
}
```

**方案4：消息丢弃（极端情况）**
- 如果积压的消息已经过时（如实时数据），可以考虑跳过。
- 使用 `consumer.seekToEnd()` 跳过积压的消息。

**预防措施：**
- 监控消息积压量（Lag），设置告警阈值。
- 消费者做好性能测试，预留扩容能力。
- 生产者做好限流，避免突发流量。

### Q27: 如何解决消息重复消费问题？ ⭐⭐

**答案：**

**消息重复的原因：**
1. 生产者发送成功但未收到 ACK，触发重试。
2. 消费者处理完成但提交 Offset 前宕机，Rebalance 后重新消费。
3. 网络抖动导致重复投递。

**解决方案：幂等消费**

**方案1：数据库唯一约束**

```java
// 利用消息的唯一ID作为唯一约束
public void processMessage(Message message) {
    String messageId = message.getMessageId();
    try {
        // INSERT 时利用唯一约束防重
        messageLogMapper.insert(new MessageLog(messageId, "PROCESSING"));
        // 处理业务逻辑
        doBusinessLogic(message);
        messageLogMapper.updateStatus(messageId, "SUCCESS");
    } catch (DuplicateKeyException e) {
        // 重复消息，直接返回
        log.info("Duplicate message: {}", messageId);
    }
}
```

**方案2：Redis 去重**

```java
public boolean isDuplicate(String messageId) {
    Boolean result = redisTemplate.opsForValue()
        .setIfAbsent("msg:" + messageId, "1", 24, TimeUnit.HOURS);
    return Boolean.FALSE.equals(result);
}
```

**方案3：状态判断（业务幂等）**

```java
// 以订单状态为例
public void processOrder(OrderMessage message) {
    Order order = orderMapper.selectById(message.getOrderId());
    if (order.getStatus() == OrderStatus.PROCESSED) {
        return; // 已处理，直接返回
    }
    // 处理订单
    order.setStatus(OrderStatus.PROCESSED);
    orderMapper.updateById(order);
}
```

**方案4：Kafka 幂等生产者**

```java
props.put("enable.idempotence", "true");
```

**最佳实践：**
- 优先使用业务幂等（状态判断），最可靠。
- 数据库唯一约束作为兜底。
- Redis 去重适合高并发场景，但要注意 Redis 宕机的情况。

### Q30: Kafka和RabbitMQ的消息模型有什么区别？ ⭐⭐

**答案：**

| 特性 | Kafka | RabbitMQ |
|------|-------|----------|
| 消息模型 | 发布-订阅（基于日志） | 发布-订阅（基于交换机） |
| 消费方式 | Pull（拉取） | Push/Pull（推/拉） |
| 消息存储 | 持久化到磁盘（基于日志段） | 内存+磁盘（可配置） |
| 消息保留 | 基于时间或大小，可回溯 | 消费后删除（默认） |
| 消费进度 | 消费者维护 Offset | Broker 维护（ACK 机制） |
| 消费者模型 | Consumer Group（组内竞争） | 独立消费者（共享队列） |
| 消息路由 | 分区（Partition） | 交换机（Exchange）+ 路由键 |
| 消息顺序 | 分区内有序 | 队列内有序 |
| 消息回溯 | 支持（重置 Offset） | 不支持（消费后删除） |

**核心区别总结：**

1. **Kafka 是日志流模型**：消息持久化存储，可以重复消费，适合大数据场景。
2. **RabbitMQ 是消息传递模型**：消息消费后删除，适合业务消息传递场景。
3. **Kafka 是 Pull 模型**：消费者主动拉取，可以控制消费速率。
4. **RabbitMQ 是 Push 模型**：Broker 推送消息给消费者，低延迟。

### Q31: Kafka的消费者组（Consumer Group）机制是怎样的？ ⭐

**答案：**

**消费者组概念：**
- 一个消费者组包含多个消费者实例。
- 一个 Topic 的每个分区只会被组内的一个消费者消费。
- 如果消费者数量多于分区数量，多余的消费者会空闲。

**分区分配策略：**

| 策略 | 说明 |
|------|------|
| Range（默认） | 按分区范围分配，可能导致不均衡 |
| RoundRobin | 轮询分配，尽量均匀 |
| Sticky | 尽量保持上次分配不变，减少 Rebalance 影响 |
| CooperativeSticky | Kafka 2.4+，增量 Rebalance |

```java
// 配置分区分配策略
props.put("partition.assignment.strategy",
    "org.apache.kafka.clients.consumer.RoundRobinAssignor");
```

**分配示例（Topic有4个分区，消费者组有2个消费者）：**

```
Range策略：
  Consumer1: Partition-0, Partition-1
  Consumer2: Partition-2, Partition-3

RoundRobin策略：
  Consumer1: Partition-0, Partition-2
  Consumer2: Partition-1, Partition-3
```

**消费者组的应用场景：**
- **点对点模式**：一个消费者组内只有一个消费者，每条消息只被消费一次。
- **发布-订阅模式**：多个消费者组订阅同一个 Topic，每条消息被每个组消费一次。

### Q32: Kafka的日志清理策略有哪些？ ⭐⭐

**答案：**

Kafka 提供两种日志清理策略：

**1. 基于时间的删除（默认）：**

```properties

log.retention.minutes=           # 或按分钟配置
log.retention.ms=                # 或按毫秒配置

```

**2. 基于大小的删除：**

```properties
log.retention.bytes=1073741824   # 每个分区最大大小（默认1GB，-1表示无限制）
```

**3. 基于日志段大小的滚动：**

```properties
log.segment.bytes=1073741824     # 每个日志段最大大小（默认1GB）
```

**配置示例：**

### Q33: RabbitMQ的集群模式有哪些？ ⭐⭐

**答案：**

**RabbitMQ 集群模式：**

**1. 普通集群（Classic Queue Cluster）：**
- 多个 RabbitMQ 节点组成集群。
- Exchange 和 Queue 的元数据在所有节点间同步。
- 消息只存储在 Queue 所在的节点，其他节点通过转发访问。
- **问题**：Queue 所在节点宕机，消息不可用（除非做了镜像）。

**2. 镜像队列集群（Mirrored Queue Cluster）：**
- 将 Queue 的消息镜像到多个节点。
- 主节点（Master）处理读写，从节点（Slave）同步数据。
- Master 宕机时自动选举新的 Master。

```java
// 配置镜像策略
rabbitmqctl set_policy ha-all "^(?!amq\.).*" '{"ha-mode":"all","ha-sync-mode":"automatic"}'
```

**3. 仲裁队列（Quorum Queue，RabbitMQ 3.8+）：**
- 基于 Raft 共识算法实现。
- 数据一致性更强，推荐使用。

```java
// 声明仲裁队列
channel.queueDeclare("quorum.queue", true, false, false,
    Map.of("x-queue-type", "quorum"));
```

**模式对比：**

| 特性 | 普通集群 | 镜像队列 | 仲裁队列 |
|------|----------|----------|----------|
| 数据冗余 | 无 | 同步/异步 | Raft 同步 |
| 可用性 | 低 | 高 | 高 |
| 一致性 | 无 | 最终一致 | 强一致 |
| 性能 | 高 | 中 | 中 |
| 推荐度 | 不推荐 | 逐步淘汰 | 推荐 |

### Q35: 什么是消息的TTL（Time To Live）？各MQ如何实现？ ⭐⭐

**Kafka：**
- Kafka 没有单条消息的 TTL，只有基于日志段的保留策略。

### Q36: Kafka的Controller是什么？有什么作用？ ⭐⭐⭐

**答案：**

**Controller 概念：**
- Kafka 集群中有一个 Broker 被选举为 Controller。
- Controller 是集群的"大脑"，负责管理和协调。

**Controller 的职责：**

1. **分区 Leader 选举**：当 Broker 宕机时，Controller 负责为受影响的分区选举新的 Leader。
2. **分区分配**：当创建 Topic 或增加分区时，Controller 负责分配分区到 Broker。
3. **Broker 上下线管理**：监听 Broker 的注册和下线。
4. **副本管理**：管理副本的分配和重新分配。
5. **元数据管理**：维护集群的元数据信息。

**Controller 选举过程：**

```
1. 每个 Broker 启动时尝试在 ZooKeeper 创建临时节点 /controller
2. 创建成功的 Broker 成为 Controller
3. 其他 Broker 监听 /controller 节点的变化
4. Controller 宕机时，临时节点被删除
5. 其他 Broker 重新竞争创建 /controller 节点
```

**KRaft 模式（Kafka 3.3+）：**
- Controller 不再通过 ZooKeeper 选举，而是使用 KRaft 协议。
- Controller 的元数据存储在内部 Topic `__cluster_metadata` 中。
- 活跃 Controller 通过 Raft 协议选举。

### Q38: Kafka的Page Cache是如何工作的？ ⭐⭐⭐

**答案：**

**Page Cache 概念：**
- Kafka 大量使用操作系统的页缓存（Page Cache），而不是自己管理缓存。
- 消息写入时先写入 Page Cache，由操作系统异步刷盘。
- 消息读取时优先从 Page Cache 读取，命中则无需磁盘 IO。

**工作原理：**

```
写入流程：
Producer → Broker → Page Cache → 异步刷盘到磁盘

读取流程：
Consumer ← Broker ← Page Cache（命中）← 磁盘（未命中）
```

**为什么 Kafka 使用 Page Cache：**

1. **避免 JVM GC**：不使用堆内存缓存，避免 GC 停顿。
2. **利用操作系统优化**：操作系统的页缓存管理非常成熟，包括预读、写合并等。
3. **进程重启不丢失**：Page Cache 在进程重启后仍然有效（只要操作系统不重启）。
4. **共享缓存**：多个进程/线程可以共享同一份 Page Cache。

**相关配置：**

```properties
# Kafka Broker 的 JVM 堆内存不需要太大
# 大部分内存留给 Page Cache
# 建议：JVM 堆 6-8GB，剩余内存给 Page Cache

# 刷盘策略
log.flush.interval.messages=10000   # 每隔多少条消息刷盘（不建议配置，依赖OS）
log.flush.interval.ms=1000          # 每隔多少毫秒刷盘（不建议配置，依赖OS）
# 推荐依赖操作系统的后台刷盘机制
```

**优化建议：**
- Broker 机器的内存尽量大，给 Page Cache 留足空间。
- 不要在 Broker 上运行其他消耗内存的进程。
- 使用 `vm.dirty_ratio` 和 `vm.dirty_background_ratio` 调优操作系统刷盘策略。

### Q39: 消息队列如何实现消息追踪？ ⭐⭐

**答案：**

**方案1：消息ID + Trace日志**

```java
// 生产者生成唯一消息ID
String traceId = UUID.randomUUID().toString();
String messageId = MDC.get("traceId") + "-" + System.currentTimeMillis();

// 在消息Header中携带追踪信息
ProducerRecord<String, String> record = new ProducerRecord<>("topic", "value");
record.headers().add("traceId", traceId.getBytes());
record.headers().add("messageId", messageId.getBytes());
record.headers().add("sendTime", String.valueOf(System.currentTimeMillis()).getBytes());
record.headers().add("source", "order-service".getBytes());

// 消费者记录追踪日志
@KafkaListener(topics = "topic")
public void consume(ConsumerRecord<String, String> record) {
    String traceId = new String(record.headers().lastHeader("traceId").value());
    MDC.put("traceId", traceId);
    log.info("Consumed message from topic: {}", record.value());
    MDC.remove("traceId");
}
```

**方案2：分布式链路追踪（Spring Cloud Sleuth / OpenTelemetry）**

```java
// Spring Cloud Sleuth 自动在消息Header中添加 trace 信息
// Kafka 消息Header中会自动携带：
// - traceId
// - spanId
// - parentId
// - sample

// 配置
spring.sleuth.messaging.kafka.enabled=true
```

**方案3：Kafka 消息追踪工具**

```

```

**方案4：RabbitMQ Firehose Tracing**


### Q40: 消息队列在生产环境中有哪些最佳实践？ ⭐⭐⭐

**答案：**

**1. 命名规范：**
```
Topic/Queue 命名：{业务}.{操作}.{环境}
示例：order.created.prod、payment.callback.dev
Consumer Group 命名：{应用名}-{功能}-{环境}
示例：order-service-created-consumer-prod
```

**2. 消息格式规范：**
```json
{
  "messageId": "uuid",
  "traceId": "trace-id",
  "timestamp": 1710000000000,
  "source": "order-service",
  "type": "ORDER_CREATED",
  "version": "1.0",
  "data": {
    "orderId": "123456",
    "amount": 100.00
  }
}
```

**3. 生产者最佳实践：**
```java
// Kafka
props.put("acks", "all");
props.put("enable.idempotence", "true");
props.put("retries", 3);
props.put("max.in.flight.requests.per.connection", 5);
props.put("compression.type", "lz4");
props.put("batch.size", 16384);
props.put("linger.ms", 5);
```

**4. 消费者最佳实践：**
```java
// 关闭自动提交
props.put("enable.auto.commit", "false");
// 合理设置拉取参数
props.put("max.poll.records", 500);
props.put("max.poll.interval.ms", 300000);
// 异常处理
try {
    processMessage(record);
    consumer.commitSync();
} catch (Exception e) {
    // 记录异常，发送到死信队列
    sendToDLQ(record, e);
    consumer.commitSync();
}
```

**5. 监控告警：**
- Consumer Lag > 阈值 → 告警
- 消息发送失败率 > 0.1% → 告警
- Broker 宕机 → 紧急告警
- 磁盘使用率 > 80% → 告警

**6. 容灾方案：**
- 多机房部署，跨区域复制（MirrorMaker / Replicator）。
- 定期演练故障切换。
- 消息备份策略。

**7. 安全配置：**
```properties
# Kafka 安全配置
ssl.truststore.location=/path/to/truststore.jks
ssl.keystore.location=/path/to/keystore.jks
sasl.mechanism=PLAIN
security.protocol=SASL_SSL
```
