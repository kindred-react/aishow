## 一、面向对象（8题）

### Q1: 什么是封装？封装的好处是什么？

**难度**: ⭐

**答案**: 封装是将数据（属性）和操作数据的方法绑定在一起，通过访问修饰符控制外部访问权限，隐藏内部实现细节，只暴露必要的接口。

**解析**:
- 封装的核心是"隐藏实现细节，暴露必要接口"
- 通过 `private` 修饰属性，提供 `public` 的 getter/setter 方法
- 可以在 setter 中加入参数校验逻辑，保证数据合法性
- 好处：提高安全性、降低耦合度、便于维护和扩展

```java
public class User {
    private String name;
    private int age;

    public void setAge(int age) {
        if (age >= 0 && age <= 150) { // 封装内部校验逻辑
            this.age = age;
        }
    }
}
```

**面试技巧**: 封装不仅仅是 private + getter/setter，更是一种设计思想。可以结合实际项目说明如何通过封装保护业务规则。

### Q2: 继承的本质是什么？Java 为什么只支持单继承？

**难度**: ⭐

**答案**: 继承是子类复用父类属性和行为的机制，本质是"is-a"关系。Java 只支持单继承是为了避免**菱形继承问题**（Diamond Problem），简化语言设计。但可以通过实现多个接口来弥补。

**解析**:
- 菱形继承问题：如果类 B 和 C 都继承了 A，D 同时继承 B 和 C，那么 A 的方法在 D 中会有两份拷贝，调用时产生歧义
- C++ 支持多继承，需要通过虚继承来解决菱形问题，增加了复杂性
- Java 通过"单继承 + 多接口"的设计在灵活性和简洁性之间取得平衡

```
    A
   / \
  B   C    ← 菱形继承问题
   \ /
    D
```

### Q3: 什么是多态？实现多态的三个必要条件是什么？

**难度**: ⭐⭐

**答案**: 多态是同一个方法调用，不同对象有不同行为。三个必要条件：**继承、重写、向上转型**（父类引用指向子类对象）。

**解析**:
- 编译时多态：方法重载（Overload），编译器根据参数类型决定调用哪个方法
- 运行时多态：方法重写（Override），JVM 在运行时根据实际对象类型调用方法
- 底层实现：JVM 通过**虚方法表（Virtual Method Table）**实现动态绑定

```java
Animal animal = new Dog(); // 向上转型
animal.eat(); // 运行时调用 Dog 的 eat()，而非 Animal 的 eat()
```

**面试技巧**: 多态提高了代码的扩展性，新增子类不需要修改已有代码，符合开闭原则。可以结合策略模式举例。

### Q4: 方法重载（Overload）和方法重写（Override）的区别？

**难度**: ⭐⭐

**答案**:

| 对比项 | 重载（Overload） | 重写（Override） |
|--------|-----------------|-----------------|
| 发生位置 | 同一个类中 | 子类与父类之间 |
| 方法签名 | 方法名相同，参数列表不同 | 方法名、参数列表完全相同 |
| 返回类型 | 无要求 | 必须相同或协变返回类型 |
| 访问修饰符 | 无要求 | 子类不能比父类更严格 |
| 异常 | 无要求 | 子类不能抛出比父类更宽泛的异常 |
| 绑定时机 | 编译时（静态绑定） | 运行时（动态绑定） |

**解析**:
- 重载不能仅通过返回类型区分，否则编译错误
- 静态方法不能被重写，子类定义同名静态方法只是"隐藏"
- `@Override` 注解用于显式标记重写，编译器会检查方法签名是否正确

### Q5: 抽象类和接口的区别是什么？什么时候用哪个？

**难度**: ⭐⭐

**答案**:

| 对比项 | 抽象类 | 接口 |
|--------|--------|------|
| 构造方法 | 可以有 | 不能有 |
| 成员变量 | 可以有实例变量 | 默认 `public static final` 常量 |
| 方法实现 | 可以有具体方法 | Java 8 前全抽象，8+ 支持 default/static 方法 |
| 继承/实现 | 单继承 | 多实现 |
| 设计语义 | "is-a" 关系 | "can-do" 关系 |
| 使用场景 | 代码复用、共享状态 | 定义契约、多重能力 |

**解析**:
- Java 8 接口新增 `default` 方法解决了接口演进问题
- Java 9 接口新增 `private` 方法用于 default 方法之间的代码复用
- 当两个接口有相同 default 方法时，实现类必须重写并指定调用哪个

```java
// 两个接口有相同 default 方法
class Duck implements Flyable, Swimmable {
    @Override
    public void move() {
        Flyable.super.move(); // 指定调用 Flyable 的 default 方法
    }
}
```

**面试技巧**: 回答时先说本质区别（抽象类用于代码复用，接口用于定义契约），再补充 Java 8+ 的变化，最后给出实际选择建议。

### Q6: this 和 super 关键字的作用？

**难度**: ⭐

**答案**: `this` 指向当前对象实例，`super` 指向父类引用。两者都可用于调用构造方法、访问成员变量和方法。

**解析**:
- `this()` 调用本类其他构造方法，必须放在构造方法第一行
- `super()` 调用父类构造方法，必须放在子类构造方法第一行
- `this` 和 `super` 不能同时出现在构造方法中（因为都必须在第一行）
- 访问成员时：`this.field` 区分局部变量和实例变量，`super.field` 访问父类被隐藏的字段

```java
public class Dog extends Animal {
    private String breed;

    public Dog(String name, String breed) {
        super(name);  // 调用父类构造方法
        this.breed = breed;  // 区分同名的局部变量和实例变量
    }
}
```

### Q7: final 关键字的用途有哪些？

**难度**: ⭐⭐

**答案**: `final` 可以修饰类、方法、变量，分别表示"不可继承"、"不可重写"、"不可变"。

**解析**:
- **final 类**：不能被继承，如 `String`、`Integer`、`StringBuilder`
- **final 方法**：不能被子类重写，但可以被重载
- **final 变量**：
  - 基本类型：值不可变
  - 引用类型：引用不可变（不能指向其他对象），但对象内部状态可以改变
- **final 参数**：方法参数加 final 表示方法内不能修改该参数
- **final 与 immutable**：final 不等于不可变，`final List<String>` 仍然可以 add/remove 元素

```java
final List<String> list = new ArrayList<>();
list.add("hello");  // 合法，修改的是对象内部状态
// list = new ArrayList<>();  // 编译错误，引用不可变
```

### Q8: equals() 和 == 的区别？为什么重写 equals() 必须重写 hashCode()？

**难度**: ⭐⭐⭐

**答案**: `==` 比较引用地址（内存地址），`equals()` 比较内容值（可被重写）。重写 `equals()` 必须重写 `hashCode()` 是因为 Java 规范要求：**两个 equals 为 true 的对象，hashCode 必须相同**，否则在 HashMap/HashSet 中会产生逻辑错误。

**解析**:
- `Object.equals()` 默认实现就是 `==`
- 如果只重写 `equals()` 不重写 `hashCode()`，两个"相等"的对象会有不同的哈希值，在 HashMap 中会被放到不同桶中，导致 `contains()`、`get()` 等操作失败

```java
// 正确重写 equals 和 hashCode（使用 Objects 工具类）
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (o == null || getClass() != o.getClass()) return false;
    User user = (User) o;
    return Objects.equals(name, user.name) && age == user.age;
}

@Override
public int hashCode() {
    return Objects.hash(name, age);
}
```

**面试技巧**: 这是高频考点，务必能完整说出 equals 的自反性、对称性、传递性、一致性、非空性五个特性。

## 二、集合框架概览（4题）

### Q9: Java 集合框架的体系结构是怎样的？

**难度**: ⭐⭐

**答案**: Java 集合框架主要分为两大体系：`Collection` 和 `Map`。

**解析**:
```
Iterable
  └── Collection
        ├── List（有序、可重复）
        │     ├── ArrayList      → 动态数组
        │     ├── LinkedList     → 双向链表
        │     └── CopyOnWriteArrayList → 读写分离（线程安全）
        ├── Set（无序、不可重复）
        │     ├── HashSet        → HashMap
        │     ├── LinkedHashSet  → HashMap + 双向链表
        │     └── TreeSet        → 红黑树
        └── Queue（队列）
              ├── PriorityQueue   → 堆
              └── BlockingQueue

Map（键值对）
  ├── HashMap           → 数组 + 链表 + 红黑树
  ├── LinkedHashMap     → HashMap + 双向链表
  ├── TreeMap           → 红黑树
  ├── ConcurrentHashMap → CAS + synchronized
  └── WeakHashMap       → 弱引用键
```

**面试技巧**: 回答时先说顶层接口，再按 List/Set/Queue/Map 分别展开，最后总结选择策略。

### Q10: ArrayList 和 LinkedList 的区别？各适用什么场景？

**难度**: ⭐⭐

**答案**:

| 对比项 | ArrayList | LinkedList |
|--------|-----------|------------|
| 底层结构 | 动态数组（Object[]） | 双向链表 |
| 随机访问 | O(1) | O(n) |
| 头部插入/删除 | O(n)（需移动元素） | O(1) |
| 尾部插入 | 均摊 O(1) | O(1) |
| 中间插入/删除 | O(n) | O(n)（查找 O(n) + 操作 O(1)） |
| 内存占用 | 紧凑 | 每个节点额外存储前后指针 |
| 实现 RandomAccess | 是 | 否 |

**解析**:
- ArrayList 在大多数场景下性能优于 LinkedList（CPU 缓存友好、内存连续）
- LinkedList 的优势在于频繁的头尾操作（如用作队列/双端队列）
- 实际开发中 90% 以上场景使用 ArrayList

### Q11: HashMap 和 TreeMap 的区别？

**难度**: ⭐⭐

**答案**: HashMap 基于哈希表，O(1) 平均查询；TreeMap 基于红黑树，O(log n) 查询但保持键有序。

**解析**:
- HashMap 允许一个 null 键和多个 null 值，TreeMap 不允许 null 键
- TreeMap 实现 `SortedMap` 接口，支持 `firstKey()`、`lastKey()`、`subMap()` 等范围操作
- 如果需要自定义排序，TreeMap 可以传入 `Comparator`

### Q12: HashSet 是如何保证元素不重复的？

**难度**: ⭐⭐

**答案**: HashSet 底层基于 `HashMap` 实现，添加元素时将元素作为 HashMap 的 key，value 为一个固定的 `PRESENT` 对象。HashMap 的 key 不允许重复，从而保证 HashSet 元素唯一。

**解析**:
- 判断重复的逻辑：先计算 hashCode，如果相同再调用 `equals()` 比较
- 因此存入 HashSet 的对象必须正确重写 `hashCode()` 和 `equals()`

```java
// HashSet 核心源码
private static final Object PRESENT = new Object();
public boolean add(E e) {
    return map.put(e, PRESENT) == null; // put 返回 null 表示之前不存在
}
```

## 三、泛型与反射（6题）

### Q13: 什么是泛型？泛型的好处是什么？

**难度**: ⭐⭐

**答案**: 泛型是 Java 5 引入的参数化类型机制，允许在定义类、接口、方法时使用类型参数，在使用时指定具体类型。

**解析**:
- 好处：编译时类型检查、消除强制类型转换、提高代码复用性
- 泛型只在编译时生效，编译后会被**类型擦除**（Type Erasure）替换为 Object 或上限类型

```java
// 泛型类
public class Box<T> {
    private T value;
    public T getValue() { return value; }
    public void setValue(T value) { this.value = value; }
}

// 泛型方法
public <T> T getFirst(List<T> list) {
    return list.get(0);
}
```

---

### Q14: 什么是类型擦除？类型擦除会带来什么问题？

**难度**: ⭐⭐⭐

**答案**: 类型擦除是 Java 泛型的实现方式。编译器在编译时将泛型类型信息擦除，替换为 Object（无界）或上限类型（有界）。泛型在运行时不存在。

**解析**:
- `List<String>` 和 `List<Integer>` 在运行时都是 `List`
- 带来的问题：
  - 不能创建泛型数组：`new List<String>[10]` 编译错误
  - 不能使用 `instanceof` 判断泛型类型：`if (obj instanceof List<String>)` 编译错误
  - 泛型类型参数不能是基本类型：`List<int>` 需要写 `List<Integer>`
- 解决方案：使用类型令牌（Type Token）如 `Class<T>` 传递类型信息

```java
// 类型擦除示例
List<String> list1 = new ArrayList<>();
List<Integer> list2 = new ArrayList<>();
System.out.println(list1.getClass() == list2.getClass()); // true，都是 ArrayList
```

### Q15: 什么是通配符？`? extends T` 和 `? super T` 的区别？

**难度**: ⭐⭐⭐

**答案**: 通配符 `?` 表示未知类型。`? extends T` 是上界通配符（协变），`? super T` 是下界通配符（逆变）。

**解析**:
- **PECS 原则（Producer Extends, Consumer Super）**：
  - 如果要从集合中读取（生产者），使用 `? extends T`
  - 如果要向集合中写入（消费者），使用 `? super T`

```java
// 上界通配符：只能读，不能写
public double sum(List<? extends Number> list) {
    double total = 0;
    for (Number n : list) {
        total += n.doubleValue(); // 可以读取为 Number
    }
    // list.add(1);  // 编译错误，不能写入
    return total;
}

// 下界通配符：可以写，读取只能为 Object
public void addNumbers(List<? super Integer> list) {
    list.add(1);     // 可以写入 Integer 及其子类
    list.add(2);
    // Integer n = list.get(0);  // 编译错误，读取只能为 Object
}
```

### Q16: 什么是反射？反射的原理是什么？

**难度**: ⭐⭐⭐

**答案**: 反射是在运行时动态获取类的信息（字段、方法、构造器）并调用对象方法的机制。原理是 JVM 在加载类时会创建对应的 `Class` 对象，包含该类的完整结构信息。

**解析**:
- 获取 Class 对象的三种方式：
  1. `Class.forName("com.example.MyClass")` —— 最常用，动态加载
  2. `MyClass.class` —— 编译时确定
  3. `obj.getClass()` —— 运行时获取

```java
Class<?> clazz = Class.forName("com.example.User");
// 创建实例
Object obj = clazz.getDeclaredConstructor().newInstance();
// 调用方法
Method method = clazz.getMethod("setName", String.class);
method.invoke(obj, "Alice");
// 访问私有字段
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);
field.set(obj, "Bob");
```

- 反射的应用场景：Spring IoC 容器、JDBC 驱动加载、注解处理器、序列化/反序列化框架
- 反射的缺点：性能开销大（比直接调用慢 10-100 倍）、破坏封装性、代码可读性差

### Q17: 什么是动态代理？JDK 动态代理和 CGLIB 代理的区别？

**难度**: ⭐⭐⭐

**答案**: 动态代理是在运行时动态生成代理类，在不修改源码的前提下对方法进行增强。JDK 动态代理基于接口，CGLIB 基于继承。

**解析**:

| 对比项 | JDK 动态代理 | CGLIB 代理 |
|--------|-------------|-----------|
| 实现方式 | 基于接口（`Proxy` + `InvocationHandler`） | 基于继承（生成子类） |
| 要求 | 目标类必须实现接口 | 目标类不能是 final 类 |
| 性能 | JDK 8 后优于 CGLIB | JDK 8 前优于 JDK 代理 |
| 依赖 | JDK 内置 | 需要第三方库 |

```java
// JDK 动态代理
public class LogInvocationHandler implements InvocationHandler {
    private final Object target;
    public LogInvocationHandler(Object target) { this.target = target; }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("Before: " + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("After: " + method.getName());
        return result;
    }
}

// 使用
UserService userService = new UserServiceImpl();
UserService proxy = (UserService) Proxy.newProxyInstance(
    userService.getClass().getClassLoader(),
    userService.getClass().getInterfaces(),
    new LogInvocationHandler(userService)
);
```

### Q18: 反射和泛型结合使用时需要注意什么？

**难度**: ⭐⭐⭐

**答案**: 由于类型擦除，反射获取到的泛型类型信息有限。但可以通过 `ParameterizedType` 等接口获取声明时的泛型类型。

**解析**:
- `Field.getGenericType()` 返回 `Type`，如果是泛型字段可以强转为 `ParameterizedType` 获取实际类型参数
- `Method.getGenericReturnType()` 可以获取方法的泛型返回类型

```java
// 获取字段的泛型类型
class GenericClass {
    private List<String> names;
}

Field field = GenericClass.class.getDeclaredField("names");
Type genericType = field.getGenericType();
if (genericType instanceof ParameterizedType) {
    ParameterizedType pt = (ParameterizedType) genericType;
    Type[] typeArgs = pt.getActualTypeArguments();
    System.out.println(typeArgs[0]); // class java.lang.String
}
```

## 四、String 专题（6题）

### Q19: String 为什么是不可变的？不可变的好处是什么？

**难度**: ⭐⭐

**答案**: String 不可变是因为其内部使用 `private final char[] value`（JDK 8）或 `private final byte[] value`（JDK 9+）存储数据，且没有提供修改方法。String 类被 `final` 修饰，不能被继承。

**解析**:
不可变的好处：
1. **线程安全**：不可变对象天然线程安全，可以在多线程中共享
2. **字符串常量池**：相同的字符串字面量可以共享同一个对象，节省内存
3. **安全性**：作为 HashMap 的 key 时，hashCode 不会改变
4. **缓存 hashCode**：String 缓存了 hashCode 值，避免重复计算

```java
// JDK 8
public final class String implements Serializable, Comparable<String>, CharSequence {
    private final char[] value;
    private int hash; // 缓存 hashCode，默认 0
}
```

### Q20: String、StringBuilder、StringBuffer 的区别？

**难度**: ⭐⭐

**答案**:

| 对比项 | String | StringBuilder | StringBuffer |
|--------|--------|--------------|-------------|
| 可变性 | 不可变 | 可变 | 可变 |
| 线程安全 | 安全（不可变） | 不安全 | 安全（synchronized） |
| 性能 | 拼接时创建大量对象 | 最快 | 较慢（锁开销） |
| 使用场景 | 少量拼接 | 单线程大量拼接 | 多线程大量拼接 |

**解析**:
- 字符串拼接在循环中应使用 StringBuilder，避免创建大量临时 String 对象
- JDK 9 引入了 `StringConcatFactory`，使用 `invokeDynamic` 优化字符串拼接
- 实际开发中 StringBuffer 很少使用，多线程场景通常用其他方式处理

### Q21: 什么是字符串常量池？`intern()` 方法的作用？

**难度**: ⭐⭐⭐

**答案**: 字符串常量池是 JVM 中的一块特殊内存区域，用于存储字符串字面量，避免重复创建。`intern()` 方法返回字符串在常量池中的引用，如果常量池中不存在则加入。

**解析**:
- JDK 6：常量池在永久代（方法区），`intern()` 会将字符串复制到永久代
- JDK 7+：常量池移到堆中，`intern()` 只是将堆中字符串的引用放入常量池

```java
String s1 = new String("hello"); // 堆中创建一个对象
String s2 = s1.intern();         // 返回常量池中的 "hello"
String s3 = "hello";             // 指向常量池中的 "hello"

System.out.println(s2 == s3);  // true
System.out.println(s1 == s3);  // false
```

**面试技巧**: 常量池面试常考字符串比较题，需要理解字面量、new、intern 的区别。

### Q22: 字符串拼接有哪些方式？哪种最高效？

**难度**: ⭐⭐

**答案**: 字符串拼接方式有 `+` 运算符、`concat()`、`StringBuilder.append()`、`String.format()`、`String.join()`。

**解析**:
- `+` 运算符：编译器会自动优化为 StringBuilder.append()（JDK 5+），但在循环中每次迭代会创建新的 StringBuilder
- 循环拼接必须手动使用 StringBuilder
- `String.join()` 适合用分隔符连接多个字符串

```java
// 循环中字符串拼接（错误示范）
String result = "";
for (String s : list) {
    result += s; // 每次循环创建新的 StringBuilder，性能差
}

// 正确做法
StringBuilder sb = new StringBuilder();
for (String s : list) {
    sb.append(s);
}
String result = sb.toString();
```

### Q23: `new String("hello")` 创建了几个对象？

**难度**: ⭐⭐

**答案**: 如果常量池中已有 "hello"，创建 1 个对象（堆中的 String 对象）；如果常量池中没有，创建 2 个对象（常量池中 1 个 + 堆中 1 个）。

**解析**:
- `"hello"` 字面量首先在常量池中查找，不存在则创建
- `new String("hello")` 会在堆中再创建一个 String 对象
- 堆中的 String 对象内部引用常量池中的 char[]

```java
String s1 = "hello";               // 常量池
String s2 = new String("hello");   // 堆 + 引用常量池
String s3 = "hello";               // 复用常量池

System.out.println(s1 == s3);  // true
System.out.println(s1 == s2);  // false
```

### Q24: String 的 hashCode() 是怎么计算的？

**难度**: ⭐⭐⭐

**答案**: String 的 hashCode 使用多项式哈希算法：`s[0]*31^(n-1) + s[1]*31^(n-2) + ... + s[n-1]`，其中 n 是字符串长度。

**解析**:
- 选择 31 的原因：31 是奇素数，乘法可以用移位和减法代替（`31 * i == (i << 5) - i`），性能好
- hashCode 在第一次调用时计算并缓存，后续直接返回缓存值
- 两个字符串内容相同则 hashCode 相同，但 hashCode 相同不代表内容相同（哈希碰撞）

```java
// String.hashCode() 源码（简化）
public int hashCode() {
    int h = hash; // 缓存值，默认 0
    if (h == 0 && value.length > 0) {
        char val[] = value;
        for (int i = 0; i < value.length; i++) {
            h = 31 * h + val[i];
        }
        hash = h;
    }
    return h;
}
```

## 五、异常处理（4题）

### Q25: Java 异常体系结构是怎样的？

**难度**: ⭐⭐

**答案**: Java 异常的顶层父类是 `Throwable`，分为 `Error` 和 `Exception` 两大分支。`Exception` 又分为 `Checked Exception`（编译时异常）和 `Unchecked Exception`（运行时异常）。

**解析**:
```
Throwable
├── Error（不可恢复，不应捕获）
│     ├── OutOfMemoryError
│     ├── StackOverflowError
│     └── NoClassDefFoundError
└── Exception
      ├── Checked Exception（编译时检查，必须处理）
      │     ├── IOException
      │     ├── SQLException
      │     └── FileNotFoundException
      └── RuntimeException（运行时异常，不强制处理）
            ├── NullPointerException
            ├── IllegalArgumentException
            ├── ClassCastException
            └── IndexOutOfBoundsException
```

### Q26: try-catch-finally 的执行顺序？finally 一定会执行吗？

**难度**: ⭐⭐

**答案**: 执行顺序为 try -> catch（如有异常）-> finally。finally 在正常情况下一定会执行，但以下情况不会：
1. 在 try 或 catch 中调用了 `System.exit(0)`
2. JVM 崩溃或被强制终止
3. 守护线程被终止

**解析**:
- finally 中不建议使用 return，会覆盖 try/catch 中的 return 值
- 如果 finally 和 try 都有 return，finally 的 return 会生效（编译器会警告）

```java
public int test() {
    try {
        return 1;
    } finally {
        return 2; // 警告：finally 中的 return 会覆盖 try 的 return
    }
}
// 结果返回 2
```

### Q27: checked exception 和 unchecked exception 的区别？

**难度**: ⭐⭐

**答案**:

| 对比项 | Checked Exception | Unchecked Exception |
|--------|------------------|---------------------|
| 继承 | Exception（非 RuntimeException） | RuntimeException |
| 编译检查 | 必须处理（try-catch 或 throws） | 不强制处理 |
| 常见类型 | IOException, SQLException | NullPointerException, ClassCastException |
| 使用场景 | 可预期的外部异常 | 编程错误 |

**解析**:
- Checked Exception 的设计初衷是强制程序员处理可预期的异常，但实际使用中经常被滥用（`catch (Exception e) {}`），被广泛批评
- 现代 Java 框架（Spring）倾向于使用 Unchecked Exception，减少样板代码

### Q28: throw 和 throws 的区别？

**难度**: ⭐

**答案**: `throw` 用于在方法内部抛出异常对象，`throws` 用于在方法签名上声明可能抛出的异常类型。

**解析**:
- `throw` 是一个语句，后面跟异常对象：`throw new RuntimeException("error")`
- `throws` 是方法签名的一部分，声明该方法可能抛出的异常：`void method() throws IOException`
- `throw` 可以抛出 Checked 或 Unchecked 异常，但 Checked 异常必须在方法签名中用 `throws` 声明

## 六、Java 新特性（6题）

### Q29: Java 10 的 var 关键字是什么？有什么限制？

**难度**: ⭐⭐

**答案**: `var` 是 Java 10 引入的局部变量类型推断（Local-Variable Type Inference），编译器根据右侧表达式自动推断类型。

**解析**:
- `var` 仅适用于局部变量，不能用于成员变量、方法参数、返回类型
- 必须有初始化值，`var x;` 编译错误
- 初始化值为 null 时无法推断类型：`var x = null;` 编译错误
- 本质是语法糖，编译后字节码中仍然是具体类型

```java
var list = new ArrayList<String>(); // 推断为 ArrayList<String>
var stream = list.stream();          // 推断为 Stream<String>
for (var item : list) {             // 增强 for 循环中也支持
    System.out.println(item);
}
```

### Q30: Java 14 的 Record 类是什么？解决了什么问题？

**难度**: ⭐⭐

**答案**: Record 是 Java 14 预览、16 正式引入的不可变数据载体，自动生成构造方法、getter、equals()、hashCode()、toString()。

**解析**:
- Record 是 `final` 的，不能被继承
- 可以自定义紧凑构造方法（Compact Constructor）进行参数校验
- 可以实现接口，但不能继承其他类

```java
// 传统写法需要 50+ 行
public record Point(int x, int y) {}

// 等价于
public final class Point {
    private final int x;
    private final int y;
    public Point(int x, int y) { this.x = x; this.y = y; }
    public int x() { return x; }
    public int y() { return y; }
    // 自动生成 equals, hashCode, toString
}

// 带参数校验
public record Age(int value) {
    public Age {
        if (value < 0 || value > 150) {
            throw new IllegalArgumentException("Invalid age");
        }
    }
}
```

### Q31: Java 的 Sealed Class（密封类）是什么？

**难度**: ⭐⭐⭐

**答案**: Sealed Class 用于限制类的继承层次，通过 `permits` 关键字明确指定哪些类可以继承。

**解析**:
- 三种修饰符配合使用：`sealed`（密封）、`non-sealed`（不密封）、`final`（最终）
- 子类必须与密封类在同一个模块中（Java 9+ 模块化）或同一个包中

```java
public sealed class Shape permits Circle, Rectangle, Triangle {}

public final class Circle extends Shape {}      // final：不能再被继承
public non-sealed class Rectangle extends Shape {} // non-sealed：开放继承
public sealed class Triangle extends Shape permits EquilateralTriangle {}
```

**面试技巧**: Sealed Class（Java 15 预览特性，Java 17 正式转正）与模式匹配（Pattern Matching for instanceof）配合使用，可以实现穷举匹配，编译器可以检查是否处理了所有子类型。

### Q32: Java 21 的虚拟线程（Virtual Threads）是什么？

**难度**: ⭐⭐⭐

**答案**: 虚拟线程是 Java 21 正式引入的轻量级线程，由 JVM 而非操作系统管理，可以创建数百万个而不会耗尽系统资源。

**解析**:
- 平台线程（传统线程）与操作系统线程 1:1 映射，创建成本高
- 虚拟线程是 M:N 映射，少量平台线程承载大量虚拟线程
- 适合 I/O 密集型任务，不适合 CPU 密集型任务
- 使用方式：`Thread.startVirtualThread()` 或 `Executors.newVirtualThreadPerTaskExecutor()`

```java
// 创建虚拟线程
Thread vt = Thread.startVirtualThread(() -> {
    // I/O 操作
});

// 虚拟线程执行器
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 100_000).forEach(i -> {
        executor.submit(() -> {
            // 百万级并发
        });
    });
}
```

### Q33: Java 14 的 Switch 表达式有什么改进？

**难度**: ⭐⭐

**答案**: Java 14 将 switch 从语句升级为表达式，支持使用 `->` 箭头语法、`yield` 返回值，并支持模式匹配。

**解析**:
- 传统 switch 使用 `:` + `break`，容易遗漏 break 导致 fall-through
- 新语法使用 `->` 箭头，不需要 break，更简洁安全
- 可以作为表达式赋值给变量

```java
// 传统写法
String result;
switch (day) {
    case MONDAY: result = "工作日"; break;
    case SUNDAY: result = "休息日"; break;
    default: result = "其他";
}

// 新写法（Java 14+）
String result = switch (day) {
    case MONDAY, FRIDAY -> "工作日";
    case SUNDAY -> "休息日";
    default -> "其他";
};

// 使用 yield 返回值
String result = switch (day) {
    case MONDAY -> {
        System.out.println("周一");
        yield "工作日"; // yield 代替 return
    }
    default -> "其他";
};
```

### Q34: instanceof 模式匹配是什么？

**难度**: ⭐⭐

**答案**: instanceof 模式匹配（Java 16 正式引入）将类型检查和类型转换合并为一步，简化代码。

**解析**:
- 传统写法需要先 instanceof 判断，再强制类型转换
- 模式匹配一步完成，变量作用域自动限定在匹配的分支内

```java
// 传统写法
if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length());
}

// 模式匹配（Java 16+）
if (obj instanceof String s) {
    System.out.println(s.length()); // s 已自动转换，无需强转
}
```

## 七、注解（3题）

### Q35: 什么是注解？Java 内置注解有哪些？

**难度**: ⭐⭐

**答案**: 注解（Annotation）是 Java 5 引入的元数据机制，用于为代码提供附加信息，不影响代码逻辑。注解可以被编译器、框架或运行时读取处理。

**解析**:
Java 内置注解：
- `@Override`：标记方法重写，编译器检查
- `@Deprecated`：标记已过时的元素，编译器警告
- `@SuppressWarnings`：抑制编译器警告
- `@FunctionalInterface`：标记函数式接口（只有一个抽象方法）

```java
@Override
@Deprecated
@SuppressWarnings("unchecked")
public void oldMethod() {}
```

---

### Q36: 元注解有哪些？自定义注解如何实现？

**难度**: ⭐⭐⭐

**答案**: 元注解是修饰注解的注解，有 `@Retention`、`@Target`、`@Documented`、`@Inherited`、`@Repeatable`（Java 8+）。

**解析**:
- `@Retention`：注解的保留策略
  - `SOURCE`：仅源码，编译后丢弃（如 `@Override`）
  - `CLASS`：保留到字节码，运行时不可获取（默认）
  - `RUNTIME`：运行时可通过反射获取（如 `@Autowired`）
- `@Target`：注解的使用位置（TYPE、FIELD、METHOD、PARAMETER 等）

```java
@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.METHOD, ElementType.FIELD})
public @interface MyLog {
    String value() default "";
    int level() default 0;
}

// 使用
@MyLog(value = "login", level = 1)
public void login() {}
```

### Q37: 注解在 Spring 框架中是如何被处理的？

**难度**: ⭐⭐⭐

**答案**: Spring 通过反射在运行时读取类/方法/字段上的注解信息，然后执行相应的处理逻辑。核心处理流程是 `BeanPostProcessor` + 反射。

**解析**:
- `@Autowired`：`AutowiredAnnotationBeanPostProcessor` 通过反射查找匹配的 Bean 并注入
- `@Transactional`：`TransactionInterceptor` 通过 AOP 代理拦截方法调用，开启/提交事务
- `@EventListener`：`EventListenerMethodProcessor` 扫描带有该注解的方法并注册为事件监听器
- Spring Boot 的 `spring.factories` / `@Conditional` 系列注解实现条件化配置

## 八、综合题（3题）

### Q38: Java 中值传递还是引用传递？

**难度**: ⭐⭐⭐

**答案**: Java 中**只有值传递**。基本类型传递的是值的拷贝，引用类型传递的是引用地址的拷贝。

**解析**:
- 基本类型：方法内修改参数不影响原变量
- 引用类型：方法内修改参数指向的对象状态会影响原对象，但修改参数引用本身不影响原引用

```java
void changeRef(User user) {
    user = new User("new"); // 修改的是引用的拷贝，不影响外部
}

void changeState(User user) {
    user.setName("new"); // 修改对象状态，影响外部
}
```

### Q39: Java 中的四种引用类型？

**难度**: ⭐⭐⭐

**答案**: Java 有四种引用类型，按引用强度从强到弱：强引用、软引用、弱引用、虚引用。

**解析**:

| 引用类型 | 回收时机 | 用途 |
|---------|---------|------|
| 强引用 | 永不回收（只要可达） | 普通变量 |
| 软引用（SoftReference） | 内存不足时回收 | 缓存 |
| 弱引用（WeakReference） | 下次 GC 时回收 | WeakHashMap、ThreadLocal |
| 虚引用（PhantomReference） | 随时回收，无法获取对象 | 跟踪 GC 回收、管理堆外内存 |

```java
// 软引用缓存
SoftReference<byte[]> cache = new SoftReference<>(new byte[1024 * 1024 * 10]);

// WeakHashMap：key 使用弱引用，GC 后自动清理
WeakHashMap<String, Object> map = new WeakHashMap<>();
```

### Q40: Java 中 `1.0 == 1.0f` 的结果是什么？为什么浮点运算不精确？

**难度**: ⭐⭐

**答案**: `1.0 == 1.0f` 结果为 `true`。但浮点数运算存在精度问题，如 `0.1 + 0.2 != 0.3`。

**解析**:
- 浮点数使用 IEEE 754 标准存储，部分十进制小数无法精确表示为二进制
- 比较浮点数应使用误差范围：`Math.abs(a - b) < 1e-6`
- 精确计算使用 `BigDecimal`

```java
System.out.println(0.1 + 0.2);          // 0.30000000000000004
System.out.println(0.1 + 0.2 == 0.3);  // false

// 使用 BigDecimal
BigDecimal a = new BigDecimal("0.1");
BigDecimal b = new BigDecimal("0.2");
System.out.println(a.add(b));  // 0.3
```
