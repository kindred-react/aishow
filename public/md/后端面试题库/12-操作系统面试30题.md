### Q1: 进程和线程有什么区别？ ⭐

**答案：**

进程（Process）是操作系统资源分配的基本单位，线程（Thread）是CPU调度的基本单位。两者的核心区别如下：

| 维度 | 进程 | 线程 |
|------|------|------|
| 定义 | 资源分配的基本单位 | CPU调度的基本单位 |
| 资源 | 拥有独立的地址空间、文件描述符等 | 共享所属进程的资源 |
| 创建开销 | 大（需要分配独立资源） | 小（共享进程资源） |
| 切换开销 | 大（需要切换页表、寄存器等） | 小（只需切换寄存器和栈） |
| 通信方式 | 管道、消息队列、共享内存、Socket等 | 直接读写共享变量（需同步） |
| 健壮性 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能导致整个进程崩溃 |
| 包含关系 | 一个进程至少包含一个线程 | 线程是进程的一部分 |

**Java中的体现：**
- Java中通过`java.lang.Process`表示操作系统进程
- 通过`java.lang.Thread`表示线程
- Java的线程在HotSpot JVM中默认是1:1映射到操作系统内核线程的（Linux上的pthread）

### Q2: 进程有哪些状态？状态之间如何转换？ ⭐

**答案：**

经典的三态模型中，进程有以下三种基本状态：

```
        ┌──────────┐
  ┌─────│  就绪态   │─────┐
  │     └──────────┘     │
  │  调度/时间片到      I/O完成/事件到达
  │                       │
  ▼                       │
┌──────────┐         ┌──────────┐
│  运行态   │────────▶│  阻塞态   │
└──────────┘  I/O请求  └──────────┘
  │                       ▲
  │    时间片到            │
  └───────────────────────┘
```

**五态模型**增加了两种状态：
- **创建态（New）**：进程正在被创建，分配资源
- **终止态（Terminated）**：进程执行完毕或被终止，回收资源

**状态转换说明：**
1. **就绪 → 运行**：调度器选择该进程分配CPU
2. **运行 → 就绪**：时间片用完 / 被高优先级进程抢占
3. **运行 → 阻塞**：请求I/O操作 / 等待信号量 / 调用sleep
4. **阻塞 → 就绪**：I/O完成 / 信号量可用 / 被唤醒
5. **运行 → 终止**：进程正常结束 / 调用exit / 被kill

**Linux中的进程状态：**
- `R`（Running/Runnable）：运行或就绪
- `S`（Interruptible Sleep）：可中断睡眠
- `D`（Uninterruptible Sleep）：不可中断睡眠（通常在等待磁盘I/O）
- `T`（Stopped）：停止
- `Z`（Zombie）：僵尸进程

### Q3: 进程间通信有哪些方式？各自有什么特点？ ⭐⭐

**答案：**

**1. 管道（Pipe）**
- 匿名管道：半双工，只能用于具有亲缘关系（父子进程）的进程间通信
- 命名管道（FIFO）：半双工，可用于无亲缘关系的进程间通信
- 数据遵循先进先出原则
- 内核提供同步机制

```c
// 匿名管道示例
int pipefd[2];
pipe(pipefd);  // pipefd[0]读端, pipefd[1]写端
pid_t pid = fork();
if (pid == 0) { // 子进程
    close(pipefd[1]); // 关闭写端
    read(pipefd[0], buf, sizeof(buf));
} else { // 父进程
    close(pipefd[0]); // 关闭读端
    write(pipefd[1], "hello", 5);
}
```

**2. 消息队列（Message Queue）**
- 消息链表，存放在内核中
- 有格式的消息（消息类型+消息体）
- 可以按类型读取消息
- 生命周期随内核

**3. 共享内存（Shared Memory）**
- 最快的IPC方式，两个进程映射同一块物理内存
- 需要配合信号量进行同步
- 减少了数据的拷贝次数

**4. 信号量（Semaphore）**
- 本质是一个计数器，用于控制多个进程对共享资源的访问
- 常与共享内存配合使用，实现同步和互斥

**5. 信号（Signal）**
- 异步通信机制，用于通知接收进程某个事件已经发生
- 只能携带少量信息（信号编号）

**6. Socket（套接字）**
- 可用于同一台主机或不同主机的进程间通信
- 全双工通信
- 支持TCP/UDP协议

| 方式 | 特点 | 适用场景 |
|------|------|----------|
| 管道 | 简单、半双工、数据无格式 | 父子进程、简单数据流 |
| 消息队列 | 有格式、可按类型读取 | 小量数据、结构化消息 |
| 共享内存 | 最快、需同步机制 | 大量数据传输 |
| 信号量 | 计数器、用于同步 | 资源互斥访问 |
| Socket | 全双工、跨网络 | 分布式系统、网络通信 |

### Q4: 常见的进程调度算法有哪些？ ⭐⭐

**答案：**

**1. 先来先服务（FCFS, First Come First Served）**
- 按到达顺序调度，非抢占式
- 优点：简单公平
- 缺点：短作业等待时间长（护航效应）

**2. 短作业优先（SJF, Shortest Job First）**
- 选择执行时间最短的进程，非抢占式
- 优点：平均等待时间最短
- 缺点：长作业可能饥饿；难以预知执行时间

**3. 最短剩余时间优先（SRTF）**
- SJF的抢占式版本
- 每当有新进程到达时，比较剩余执行时间

**4. 优先级调度（Priority Scheduling）**
- 按优先级选择进程，可抢占或非抢占
- 缺点：低优先级进程可能饥饿
- 解决方案：老化（Aging）——逐渐提高等待进程的优先级

**5. 时间片轮转（RR, Round Robin）**
- 每个进程分配固定时间片，用完就轮到下一个
- 时间片太大退化为FCFS，太小则切换频繁
- 适合交互式系统

**6. 多级反馈队列（MLFQ, Multilevel Feedback Queue）**
- 设置多个就绪队列，优先级不同，时间片不同
- 新进程先进入高优先级队列
- 如果时间片用完未完成，降入低优先级队列
- 低优先级队列时间片更大
- 综合了FCFS、SJF、RR的优点

**Linux调度器演进：**
- Linux 2.4：O(n)调度器
- Linux 2.6：O(1)调度器
- Linux 2.6.23+：CFS（Completely Fair Scheduler，完全公平调度器）
- CFS使用红黑树管理进程，按虚拟运行时间（vruntime）排序

### Q5: 什么是死锁？产生死锁的四个必要条件是什么？ ⭐⭐

**答案：**

**死锁**是指两个或多个进程因互相等待对方释放资源而永久阻塞的状态。

**产生死锁的四个必要条件（Coffman条件）：**

1. **互斥条件（Mutual Exclusion）**：资源一次只能被一个进程使用
2. **占有并等待（Hold and Wait）**：进程持有至少一个资源，同时等待获取其他进程持有的资源
3. **不可抢占（No Preemption）**：已分配给进程的资源不能被强制剥夺
4. **循环等待（Circular Wait）**：存在进程的循环等待链 P0→P1→P2→...→P0

**死锁处理策略：**

| 策略 | 方法 | 说明 |
|------|------|------|
| 死锁预防 | 破坏四个条件之一 | 资源排序法、一次性申请所有资源 |
| 死锁避免 | 银行家算法 | 运行时动态检查资源分配是否安全 |
| 死锁检测 | 资源分配图 | 定期检测是否存在环路 |
| 死锁恢复 | 终止进程/资源抢占 | 检测到死锁后进行恢复 |
| 鸵鸟策略 | 忽略死锁 | 不处理，适用于死锁发生概率低的系统 |

### Q6: 详细说明银行家算法的原理和步骤。 ⭐⭐⭐

**答案：**

银行家算法是Dijkstra提出的死锁避免算法，通过模拟资源分配来判断系统是否处于安全状态。

**核心概念：**
- **安全状态**：存在一个安全序列，使所有进程都能执行完毕
- **不安全状态**：不存在安全序列，可能导致死锁

**数据结构：**
- `Available`：可用的各类资源数量
- `Max`：每个进程对各类资源的最大需求
- `Allocation`：每个进程已分配的各类资源数量
- `Need`：每个进程还需要的各类资源数量，`Need = Max - Allocation`

**算法步骤：**

```
1. 初始化 Work = Available, Finish[i] = false 对所有 i

2. 寻找满足以下条件的进程 i：
   - Finish[i] == false
   - Need[i] <= Work

3. 如果找到：
   Work = Work + Allocation[i]
   Finish[i] = true
   回到步骤2

4. 如果所有 Finish[i] == true，则系统处于安全状态
   否则处于不安全状态
```

**资源请求算法：**
当进程Pi请求资源Request[i]时：
1. 如果 `Request[i] <= Need[i]`，继续；否则报错（请求超出最大需求）
2. 如果 `Request[i] <= Available`，继续；否则进程等待
3. 假定分配资源，修改状态：
   - `Available = Available - Request[i]`
   - `Allocation[i] = Allocation[i] + Request[i]`
   - `Need[i] = Need[i] - Request[i]`
4. 执行安全性算法检查是否安全
5. 如果安全则分配，否则恢复原状态并让进程等待

**示例：**
假设系统有3类资源A、B、C，数量分别为10、5、7，有5个进程：

| 进程 | Max(A,B,C) | Allocation(A,B,C) | Need(A,B,C) |
|------|-----------|-------------------|-------------|
| P0   | 7,5,3     | 0,1,0             | 7,4,3       |
| P1   | 3,2,2     | 2,0,0             | 1,2,2       |
| P2   | 9,0,2     | 3,0,2             | 6,0,0       |
| P3   | 2,2,2     | 2,1,1             | 0,1,1       |
| P4   | 4,3,3     | 0,0,2             | 4,3,1       |

Available = (3,3,2)

安全序列之一：P1→P3→P4→P2→P0

### Q7: 什么是虚拟内存？它的工作原理是什么？ ⭐⭐

**答案：**

**虚拟内存**是操作系统提供的一种内存管理技术，它使每个进程拥有独立的、连续的虚拟地址空间，而实际上物理内存是离散分配的。

**核心思想：**
- 将程序使用到的数据按需加载到物理内存
- 不常用的数据暂存到磁盘（Swap分区）
- 通过页表将虚拟地址映射到物理地址

**工作原理：**

```
CPU发出虚拟地址
    │
    ▼
┌─────────────┐
│  分页单元    │  拆分为页号 + 页内偏移
└─────────────┘
    │
    ▼
┌─────────────┐
│  TLB查找    │  快速查找页表缓存
└─────────────┘
    │
    ├─ TLB命中 → 直接得到物理地址
    │
    └─ TLB未命中 → 查页表
         │
         ├─ 页在内存 → 加载到TLB，得到物理地址
         │
         └─ 页不在内存 → 缺页中断(Page Fault)
              │
              ▼
         从磁盘加载页面到物理内存
         更新页表
         重新执行指令
```

**优点：**
1. 每个进程有独立的地址空间，互不干扰
2. 内存使用效率高，按需加载
3. 程序可以使用比物理内存更大的地址空间
4. 提供内存保护机制

**Linux虚拟内存相关参数：**
```bash
# 查看虚拟内存信息
cat /proc/meminfo
# 查看进程虚拟内存映射
cat /proc/<pid>/maps
# 查看缺页中断统计
ps -o min_flt,maj_flt -p <pid>
```

### Q8: 分页和分段有什么区别？ ⭐⭐

**答案：**

| 维度 | 分页（Paging） | 分段（Segmentation） |
|------|---------------|---------------------|
| 划分依据 | 物理划分，按固定大小 | 逻辑划分，按程序模块 |
| 大小 | 固定（如4KB） | 不固定，由程序决定 |
| 地址空间 | 一维（页号+偏移） | 二维（段号+段内偏移） |
| 碎片问题 | 内部碎片（页内未用完） | 外部碎片（段间空隙） |
| 共享 | 不方便（页无逻辑意义） | 方便（段是逻辑单元） |
| 用户可见性 | 对用户透明 | 用户可见 |
| 目的 | 管理物理内存 | 方便程序组织和共享 |

**段页式存储管理：**
结合了分段和分页的优点：
- 先将程序按逻辑分段
- 再将每段分页
- 地址结构：段号 + 段内页号 + 页内偏移

**Linux的实现：**
- Linux主要使用分页机制
- 段机制基本不用（平坦内存模型）
- x86_64架构使用4级页表（PGD→PUD→PMD→PTE）
- Linux 4.14+ 支持5级页表（增加 P4D 层），支持最大 128TB 虚拟地址空间
- 页大小默认4KB，支持大页（Huge Pages，2MB/1GB）

### Q9: 什么是TLB？它有什么作用？ ⭐⭐

**答案：**

**TLB（Translation Lookaside Buffer，旁路转换缓冲/页表缓存）** 是一种高速缓存，用于加速虚拟地址到物理地址的转换。

**工作原理：**
1. CPU生成虚拟地址后，首先查找TLB
2. 如果TLB命中（找到对应的页表项），直接获得物理地址
3. 如果TLB未命中，需要访问内存中的页表
4. 从页表获取物理地址后，将该映射关系缓存到TLB中

**性能影响：**
- TLB命中通常只需1-2个时钟周期
- TLB未命中需要访问内存中的页表（多级页表可能需要4次内存访问）
- TLB命中率对程序性能影响很大

**TLB的分类：**
- **统一TLB**：指令和数据共用一个TLB
- **分离TLB**：指令TLB（ITLB）和数据TLB（DTLB）分开

**TLB的替换策略：**
- LRU（最近最少使用）
- Random（随机替换）

**大页（Huge Pages）的意义：**
- 使用大页可以减少页表项数量
- 同样的TLB容量可以覆盖更大的地址空间
- 减少TLB未命中次数
- 适用于数据库等大内存应用

```bash
# 查看TLB信息
cat /proc/cpuinfo | grep -i tlb
# 查看大页配置
cat /proc/meminfo | grep -i huge
```

### Q10: 常见的页面置换算法有哪些？ ⭐⭐

**答案：**

当发生缺页中断且物理内存已满时，需要选择一个页面换出，这就是页面置换算法。

**1. 最佳置换算法（OPT, Optimal）**
- 淘汰将来最长时间不会被访问的页面
- 理论最优，实际不可实现（无法预知未来访问）
- 用作其他算法的性能评估基准

**2. 先进先出（FIFO）**
- 淘汰最早进入内存的页面
- 实现简单，但性能差
- 可能出现Belady异常（增加物理页框数反而增加缺页次数）

**3. 最近最久未使用（LRU, Least Recently Used）**
- 淘汰最近最长时间未被访问的页面
- 性能接近OPT，但实现开销大
- 需要维护访问时间戳或链表

```java
// LRU简单实现（基于LinkedHashMap）
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;

    public LRUCache(int capacity) {
        super(capacity, 0.75f, true); // accessOrder=true
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;
    }
}
```

**4. 最不经常使用（LFU, Least Frequently Used）**
- 淘汰访问频率最低的页面
- 需要维护每个页面的访问计数
- 可能导致新页面被快速淘汰

**5. 时钟算法（Clock / Second Chance）**
- LRU的近似实现
- 使用循环链表和访问位
- 扫描时跳过访问位为1的页面（将其置0），淘汰第一个访问位为0的页面

**6. 改进型时钟算法**
- 同时考虑访问位和修改位
- 优先淘汰未访问且未修改的页面

**算法比较：**

| 算法 | 性能 | 实现复杂度 | Belady异常 |
|------|------|-----------|-----------|
| OPT | 最优 | 不可实现 | 无 |
| LRU | 接近最优 | 较高 | 无 |
| LFU | 较好 | 高 | 无 |
| Clock | 较好 | 低 | 无 |
| FIFO | 差 | 最低 | 有 |

### Q11: Linux文件系统中inode是什么？ ⭐⭐

**答案：**

**inode（索引节点）** 是Linux文件系统中存储文件元数据的数据结构。每个文件（或目录）对应一个inode。

**inode存储的信息：**
- 文件类型（普通文件、目录、链接等）
- 文件权限（rwx）
- 文件所有者（UID）和组（GID）
- 文件大小
- 时间戳（atime/mtime/ctime）
- 链接数（硬链接计数）
- 数据块的位置指针（直接指针 + 间接指针 + 双重间接指针 + 三重间接指针）

**inode的结构（以ext4为例）：**
```
┌──────────────────┐
│ 文件类型和权限    │
│ 所有者信息        │
│ 文件大小          │
│ 时间戳            │
│ 链接数            │
│ 数据块指针        │
│  ├─ 12个直接指针  │  每个指向一个4KB数据块（共48KB）
│  ├─ 1个间接指针    │  指向一个块指针表（4KB/4B=1024个指针，共4MB）
│  ├─ 1个双重间接指针│  共4GB
│  └─ 1个三重间接指针│  共4TB
└──────────────────┘
```

**inode的特点：**
- inode不存储文件名，文件名存储在目录文件中
- 目录文件的内容是：文件名 → inode编号的映射表
- inode数量在格式化时确定，用完后即使磁盘空间未满也无法创建新文件

**相关命令：**
```bash
# 查看文件inode信息
stat filename
# 查看文件inode编号
ls -i filename
# 查看文件系统inode使用情况
df -i
# 根据inode编号查找文件
find / -inum <inode_number>
```

### Q12: 软链接和硬链接有什么区别？ ⭐

**答案：**

| 维度 | 硬链接（Hard Link） | 软链接（Symbolic Link） |
|------|-------------------|------------------------|
| 本质 | 指向同一个inode | 指向文件路径的快捷方式 |
| 创建命令 | `ln source target` | `ln -s source target` |
| 跨文件系统 | 不可以 | 可以 |
| 删除原文件 | 不影响（链接数-1） | 链接失效（dangling link） |
| 目录 | 不支持（防止循环） | 支持 |
| inode | 与原文件相同 | 不同 |
| 权限 | 与原文件相同 | 777（链接本身权限） |

**硬链接原理：**
- 创建硬链接就是在目录中新增一个文件名→inode的映射
- inode中的链接计数+1
- 删除文件时链接计数-1，减到0时才真正释放数据块

**软链接原理：**
- 创建一个新文件，内容是目标文件的路径字符串
- 访问时通过路径解析找到目标文件
- 类似于Windows的快捷方式

```bash
# 创建硬链接
ln file.txt hardlink.txt
# 创建软链接
ln -s file.txt softlink.txt
# 查看链接信息
ls -li  # 显示inode编号
```

### Q13: Linux的五种IO模型分别是什么？ ⭐⭐

**答案：**

**1. 阻塞IO（Blocking IO）**
- 最传统的IO模型
- 用户线程发起IO操作后一直阻塞，直到数据准备好
- 期间CPU不执行其他任务

**2. 非阻塞IO（Non-blocking IO）**
- 发起IO操作后立即返回
- 如果数据未准备好，返回EWOULDBLOCK错误
- 需要用户线程不断轮询（busy waiting）

**3. IO多路复用（IO Multiplexing）**
- 使用select/poll/epoll等系统调用
- 一个线程可以同时监听多个文件描述符
- 当某个FD就绪时返回，再进行实际的读写操作
- 适用于高并发连接场景（如Nginx、Redis）

**4. 信号驱动IO（Signal-driven IO）**
- 注册SIGIO信号处理函数
- 数据准备好时内核发送信号通知用户线程
- 用户线程在信号处理函数中进行IO操作

**5. 异步IO（Asynchronous IO, AIO）**
- 发起IO操作后立即返回
- 内核完成数据拷贝（从内核缓冲区到用户缓冲区）后通知用户线程
- 真正的异步，全程不需要用户线程参与

**对比总结：**

| 模型 | 等待数据就绪 | 等待数据拷贝 | 是否真正异步 |
|------|------------|------------|------------|
| 阻塞IO | 阻塞 | 阻塞 | 否 |
| 非阻塞IO | 轮询 | 阻塞 | 否 |
| IO多路复用 | 阻塞（监听多个FD） | 阻塞 | 否 |
| 信号驱动IO | 不阻塞 | 阻塞 | 否 |
| 异步IO | 不阻塞 | 不阻塞 | 是 |

**同步vs异步的关键区别：** 数据从内核空间拷贝到用户空间的过程是否阻塞。

### Q14: select、poll、epoll的区别与原理是什么？ ⭐⭐⭐

**答案：**

**1. select**

**原理：**
- 将需要监听的FD集合从用户空间拷贝到内核空间
- 内核遍历所有FD，检查是否有就绪事件
- 将就绪的FD集合拷贝回用户空间
- 用户程序遍历FD集合找到就绪的FD

**缺点：**
- FD数量限制：默认1024（`FD_SETSIZE`）
- 每次调用都需要将FD集合在用户空间和内核空间之间拷贝
- 内核需要线性遍历所有FD，O(n)复杂度
- 用户程序也需要遍历整个FD集合

```c
int select(int nfds, fd_set *readfds, fd_set *writefds,
           fd_set *exceptfds, struct timeval *timeout);
```

**2. poll**

**原理：**
- 与select类似，但使用pollfd结构体数组代替位图
- 没有FD数量限制

**缺点：**
- 仍然需要将pollfd数组在用户空间和内核空间之间拷贝
- 仍然是O(n)的遍历复杂度

```c
int poll(struct pollfd *fds, nfds_t nfds, int timeout);

struct pollfd {
    int fd;         // 文件描述符
    short events;   // 关注的事件
    short revents;  // 实际发生的事件
};
```

**3. epoll**

**原理：**
- epoll由三个核心组件组成：
  - `epoll_create`：创建epoll实例（红黑树 + 就绪链表）
  - `epoll_ctl`：添加/修改/删除监听的FD（O(log n)）
  - `epoll_wait`：等待就绪事件（O(就绪FD数量)）

**工作模式：**
- **LT（Level Triggered，水平触发）**：默认模式，只要FD就绪就会通知，直到数据被读完
- **ET（Edge Triggered，边沿触发）**：只在FD从未就绪变为就绪时通知一次，需要一次性读完数据（配合非阻塞IO）

**优点：**
- 没有FD数量限制（仅受系统内存限制）
- 只返回就绪的FD，不需要遍历
- 使用mmap共享内存，减少拷贝开销
- 基于事件通知机制，不需要每次都传递所有FD

```c
int epoll_create(int size);
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
```

**三者对比：**

| 维度 | select | poll | epoll |
|------|--------|------|-------|
| FD数量限制 | 1024 | 无 | 无 |
| 时间复杂度 | O(n) | O(n) | O(活跃FD数) |
| 数据传递 | 每次全量拷贝 | 每次全量拷贝 | 内核共享内存 |
| 内核实现 | 线性遍历 | 线性遍历 | 回调+红黑树 |
| 触发模式 | LT | LT | LT/ET |
| 适用场景 | 少量连接 | 少量连接 | 大量连接（C10K） |

### Q15: 什么是零拷贝技术？有哪些实现方式？ ⭐⭐⭐

**答案：**

**零拷贝**是指避免CPU在内存之间进行数据拷贝的技术，减少上下文切换和CPU拷贝次数，提高IO性能。

**传统IO的数据拷贝过程（4次拷贝 + 4次上下文切换）：**
```
磁盘 → 内核缓冲区 → 用户缓冲区 → Socket缓冲区 → 网卡
         ①              ②              ③            ④
```

**1. mmap + write（3次拷贝 + 4次上下文切换）**
- 将文件映射到用户空间内存
- 省去了内核缓冲区到用户缓冲区的拷贝
- 但仍然需要CPU参与拷贝到Socket缓冲区

```java
// Java中使用mmap
FileChannel channel = FileChannel.open(Paths.get("file.txt"));
MappedByteBuffer buffer = channel.map(FileChannel.MapMode.READ_ONLY, 0, channel.size());
```

**2. sendfile（2次拷贝 + 2次上下文切换，Linux 2.1）**
- 在内核空间直接将数据从文件缓冲区拷贝到Socket缓冲区
- 不需要经过用户空间
- 适用于文件传输场景

```c
ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);
```

**3. sendfile + DMA Scatter/Gather（1次拷贝 + 2次上下文切换，Linux 2.4）**
- 结合硬件的Scatter/Gather功能
- 只拷贝描述信息（含偏移量和长度）到Socket缓冲区
- DMA直接从内核缓冲区读取数据发送到网卡
- 真正的零CPU拷贝

**4. splice（0次用户空间与内核空间之间的数据拷贝）**
- 在两个文件描述符之间移动数据
- 至少一个必须是管道
- 完全在内核空间操作

```c
ssize_t splice(int fd_in, loff_t *off_in, int fd_out,
               loff_t *off_out, size_t len, unsigned int flags);
```

**Java中的零拷贝：**
- `FileChannel.transferTo()`：底层调用sendfile
- `FileChannel.transferFrom()`：底层调用sendfile
- `MappedByteBuffer`：底层调用mmap
- Netty中使用`FileRegion`实现零拷贝文件传输

**应用场景：**
- Kafka使用零拷贝提高消息传输效率
- Nginx使用sendfile提供静态文件服务
- RocketMQ使用mmap读写CommitLog

### Q16: 什么是CPU上下文切换？有哪些类型？ ⭐⭐

**答案：**

**CPU上下文切换**是指CPU从一个任务切换到另一个任务时，需要保存当前任务的状态并恢复下一个任务的状态的过程。

**CPU上下文包括：**
- 程序计数器（PC）
- 寄存器状态
- 栈指针（SP）
- 程序状态字（PSW）

**上下文切换的类型：**

**1. 进程上下文切换**
- 切换不同进程之间的执行环境
- 需要切换虚拟地址空间（切换页表）
- 涉及TLB刷新
- 开销最大

**2. 线程上下文切换**
- 同一进程内的线程切换
- 不需要切换虚拟地址空间
- 只需切换寄存器、栈指针等
- 开销比进程切换小

**3. 中断上下文切换**
- 硬件中断或软件中断触发
- 不涉及进程状态保存
- 优先级高于进程上下文

**上下文切换的开销：**
- 直接开销：保存和恢复寄存器、切换页表、刷新TLB
- 间接开销：CPU缓存失效（L1/L2/L3 Cache中的数据可能不再有效）
- 频繁的上下文切换会严重影响系统性能

**查看上下文切换统计：**
```bash
# 查看系统级上下文切换
vmstat 1
# cs列表示每秒上下文切换次数

# 查看进程级上下文切换
pidstat -w 1
# cswch: 自愿上下文切换（等待资源）
# nvcswch: 非自愿上下文切换（被抢占）

# 使用perf分析
perf stat -e context-switches -a sleep 10
```

**Java中减少上下文切换的方法：**
- 使用线程池控制线程数量
- 使用协程/虚拟线程（Java 21 Virtual Threads）
- 减少锁竞争
- 使用CAS代替synchronized

### Q17: 用户态和内核态有什么区别？如何切换？ ⭐

**答案：**

**用户态和内核态**是CPU的两种运行特权级别，用于保护操作系统内核不被用户程序破坏。

| 维度 | 用户态 | 内核态 |
|------|--------|--------|
| 特权级 | Ring 3（低特权） | Ring 0（高特权） |
| 可执行指令 | 非特权指令 | 所有指令 |
| 可访问资源 | 用户空间内存 | 所有内存和硬件 |
| 崩溃影响 | 仅影响当前进程 | 可能导致系统崩溃 |

**用户态切换到内核态的场景（系统调用）：**
1. **系统调用（System Call）**：如read、write、open、fork等
2. **异常（Exception）**：如缺页中断、除零错误
3. **硬件中断（Interrupt）**：如网卡收到数据、时钟中断

**切换过程：**
```
用户程序
    │
    ▼ 调用系统调用（如read）
    │
    │ 保存用户态寄存器到内核栈
    │ 设置CPU特权级为Ring 0
    ▼
内核态执行系统调用处理函数
    │
    │ 检查参数、执行操作
    │ 将结果拷贝到用户空间
    │ 恢复用户态寄存器
    │ 设置CPU特权级为Ring 3
    ▼
返回用户程序
```

**系统调用的开销：**
- 上下文切换开销（保存/恢复寄存器）
- 权限检查开销
- 内核与用户空间的数据拷贝

**优化建议：**
- 减少系统调用次数（批量操作、缓冲）
- 使用更高效的系统调用（如`epoll`替代`select`）
- 使用零拷贝技术减少数据拷贝

### Q18: Linux信号机制是怎样的？常见信号有哪些？ ⭐⭐

**答案：**

**信号**是Linux中一种异步通信机制，用于通知进程某个事件已经发生。

**信号处理方式：**
1. **默认处理**：系统定义的默认行为（终止、忽略、核心转储等）
2. **忽略处理**：忽略该信号（SIGKILL和SIGSTOP不能被忽略）
3. **捕获处理**：注册信号处理函数（SIGKILL和SIGSTOP不能被捕获）

**常见信号：**

| 信号 | 编号 | 默认行为 | 说明 |
|------|------|---------|------|
| SIGHUP | 1 | 终止 | 终端挂起 |
| SIGINT | 2 | 终止 | Ctrl+C中断 |
| SIGQUIT | 3 | 核心转储 | Ctrl+\退出 |
| SIGKILL | 9 | 终止 | 强制终止（不可捕获/忽略） |
| SIGSEGV | 11 | 核心转储 | 段错误（非法内存访问） |
| SIGPIPE | 13 | 终止 | 管道破裂（写入无人读取的管道） |
| SIGALRM | 14 | 终止 | 定时器信号 |
| SIGTERM | 15 | 终止 | 终止信号（可被捕获，kill默认信号） |
| SIGCHLD | 17 | 忽略 | 子进程状态改变 |
| SIGSTOP | 19 | 停止 | 暂停进程（不可捕获/忽略） |
| SIGCONT | 18 | 继续 | 继续已暂停的进程 |

**信号相关命令和系统调用：**
```bash
# 发送信号
kill -9 <pid>        # 发送SIGKILL
kill -15 <pid>       # 发送SIGTERM
kill -SIGTERM <pid>

# 查看所有信号
kill -l

# 查看进程收到的信号
cat /proc/<pid>/status | grep Sig
```

```c
// C语言信号处理示例
#include <signal.h>
#include <stdio.h>

void sigint_handler(int sig) {
    printf("Received SIGINT\n");
}

int main() {
    signal(SIGINT, sigint_handler);
    while(1) {
        sleep(1);
    }
    return 0;
}
```

**SIGCHLD与僵尸进程：**
- 子进程退出时会向父进程发送SIGCHLD信号
- 父进程应在信号处理函数中调用`wait()/waitpid()`回收子进程
- 如果父进程不处理SIGCHLD，子进程会变成僵尸进程

### Q19: Shell管道的原理是什么？ ⭐⭐

**答案：**

**Shell管道（Pipe）** 使用`|`符号连接两个命令，将前一个命令的标准输出作为后一个命令的标准输入。

**底层原理：**
1. Shell解析管道命令后，创建匿名管道（`pipe()`系统调用）
2. 对管道中的每个命令`fork()`子进程
3. 第一个命令的标准输出重定向到管道写端
4. 最后一个命令的标准输入重定向到管道读端
5. 中间命令的标准输入和标准输出分别重定向到管道的读端和写端

**示例分析：**
```bash
cat file.txt | grep "hello" | wc -l
```

执行过程：
```
Shell
  │
  ├─ pipe(fd1)     // 创建管道1（cat → grep）
  ├─ pipe(fd2)     // 创建管道2（grep → wc）
  │
  ├─ fork() → cat进程
  │    stdout → fd1[1]（管道1写端）
  │    close(fd1[0], fd2[0], fd2[1])
  │
  ├─ fork() → grep进程
  │    stdin  → fd1[0]（管道1读端）
  │    stdout → fd2[1]（管道2写端）
  │    close(fd1[1], fd2[0])
  │
  └─ fork() → wc进程
       stdin  → fd2[0]（管道2读端）
       close(fd1[0], fd1[1], fd2[1])
```

**管道的特点：**
- 半双工通信（单向数据流）
- 数据在内核缓冲区中传递，不需要磁盘
- 管道有容量限制（Linux默认64KB）
- 写端关闭后，读端返回EOF
- 读端关闭后，写端收到SIGPIPE信号

**重定向原理：**
```bash
# 输出重定向（>）
command > file.txt    # 等价于 open(file.txt) → dup2(fd, STDOUT_FILENO)

# 输入重定向（<）
command < file.txt    # 等价于 open(file.txt) → dup2(fd, STDIN_FILENO)

# 追加重定向（>>）
command >> file.txt   # 等价于 open(file.txt, O_APPEND)

# Here Document
command << EOF
hello
world
EOF
```

```bash
echo "vm.swappiness = 10" >> /etc/sysctl.conf

# 脏页写回
echo "vm.dirty_ratio = 10" >> /etc/sysctl.conf
echo "vm.dirty_background_ratio = 5" >> /etc/sysctl.conf

# 内存过载保护
echo "vm.overcommit_memory = 0" >> /etc/sysctl.conf
```

**4. 内核参数查看**

```bash
# 查看所有内核参数
sysctl -a

# 查看指定参数
sysctl net.core.somaxconn
cat /proc/sys/net/core/somaxconn

# 查看网络连接状态统计
netstat -s | grep -i "timewait"
```

### Q22: 什么是孤儿进程和僵尸进程？如何处理？ ⭐⭐

**答案：**

**孤儿进程（Orphan Process）**
- 父进程先于子进程退出，子进程成为孤儿
- init进程（PID=1，现代Linux中是systemd）会收养孤儿进程
- 孤儿进程不会造成危害，init会负责回收其资源

**僵尸进程（Zombie Process）**
- 子进程退出，但父进程没有调用`wait()/waitpid()`回收
- 子进程的PCB（进程控制块）仍然保留在内核中
- 僵尸进程无法被`kill -9`杀死（因为它已经"死"了）
- 大量僵尸进程会消耗PID资源

**处理僵尸进程的方法：**

```c
// 方法1：父进程调用wait/waitpid
pid_t pid = wait(NULL);  // 阻塞等待任意子进程
pid_t pid = waitpid(child_pid, &status, WNOHANG);  // 非阻塞

// 方法2：捕获SIGCHLD信号
signal(SIGCHLD, sigchld_handler);
void sigchld_handler(int sig) {
    while (waitpid(-1, NULL, WNOHANG) > 0);
}

// 方法3：fork两次（将子进程托付给init）
if (fork() == 0) {
    if (fork() == 0) {
        // 孙子进程，父进程（子进程）立即退出
        // 孙子进程成为孤儿，被init收养
    } else {
        exit(0);  // 子进程退出
    }
}
```

**Java中的处理：**
- Java进程通过`Process.waitFor()`等待子进程结束
- 建议使用`ProcessBuilder`并正确处理输入输出流，避免子进程阻塞

### Q23: 什么是协程？与线程有什么区别？ ⭐⭐⭐

**答案：**

**协程（Coroutine）** 是一种用户态的轻量级线程，由程序自身控制调度，不需要操作系统内核参与。

**协程 vs 线程：**

| 维度 | 线程 | 协程 |
|------|------|------|
| 调度者 | 操作系统内核 | 用户程序（运行时） |
| 切换开销 | 大（需要内核介入） | 小（用户态切换） |
| 栈大小 | 默认1MB（Linux） | 可以很小（几KB） |
| 创建数量 | 通常几千 | 可以几十万甚至百万 |
| 并发模型 | 抢占式 | 协作式（也可抢占） |
| 阻塞操作 | 阻塞整个线程 | 可以挂起当前协程，切换到其他协程 |

**协程的实现原理：**
- 保存当前协程的上下文（寄存器、栈指针、程序计数器）
- 切换到另一个协程执行
- 所有操作在用户空间完成，不需要内核参与

**各语言的协程实现：**
- **Go**：goroutine（M:N调度模型，M个goroutine映射到N个OS线程）
- **Kotlin**：协程（suspend/resume机制）
- **Python**：asyncio（async/await语法）
- **Java**：Project Loom / Virtual Threads（Java 21正式引入）

**Java虚拟线程（Virtual Threads）：**
```java
// Java 21 虚拟线程
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    IntStream.range(0, 100_000).forEach(i -> {
        executor.submit(() -> {
            Thread.sleep(Duration.ofSeconds(1));
            return i;
        });
    });
}
// 可以轻松创建百万级虚拟线程
```

### Q24: Linux中如何查看和修改进程的优先级？ ⭐

**答案：**

**进程优先级相关概念：**

- **Nice值**：范围-20到19，值越小优先级越高，普通用户只能调高（降低优先级）
- **PR值**：内核实际使用的优先级，PR = 20 + Nice
- **RT优先级**：实时进程的优先级，范围0-99

**查看优先级：**
```bash
# top命令查看
top
# NI列显示Nice值，PR列显示优先级

# ps命令查看
ps -eo pid,ni,pri,cmd

# 查看指定进程
ps -p <pid> -o pid,ni,pri,cmd
```

**修改优先级：**
```bash
# 启动时指定优先级
nice -n 10 command        # Nice值为10

# 修改已运行进程的优先级
renice -n 5 -p <pid>     # 将进程Nice值改为5

# 实时优先级
chrt -f 50 command       # FIFO实时调度，优先级50
chrt -r 50 command       # RR实时调度，优先级50
```

**注意：**
- 只有root用户可以将Nice值设为负数（提高优先级）
- 实时优先级进程可能导致系统卡顿，使用需谨慎

### Q25: 什么是Buffer和Cache？有什么区别？ ⭐⭐

**答案：**

**Buffer（缓冲区）**
- 用于缓冲写入操作
- 数据从用户空间写入Buffer，再由操作系统异步写入磁盘
- 目的：减少磁盘IO次数，合并随机写为顺序写

**Cache（缓存）**
- 用于缓存读取操作
- 从磁盘读取的数据缓存在Cache中，下次读取时直接从内存获取
- 目的：加快读取速度

**区别总结：**

| 维度 | Buffer | Cache |
|------|--------|-------|
| 作用 | 加速写入 | 加速读取 |
| 方向 | 应用→磁盘 | 磁盘→应用 |
| 数据来源 | 应用程序写入 | 磁盘读取 |
| 释放影响 | 可能丢失未写入数据 | 仅影响读取性能 |

**查看Buffer和Cache：**
```bash
free -h
# buff/cache列就是Buffer和Cache的总和

# 详细查看
cat /proc/meminfo | grep -E "Buffers|Cached"
# Buffers: 对应块设备的缓冲
# Cached: 对应文件页缓存

# 清理Cache（需要root权限，一般不建议）
echo 1 > /proc/sys/vm/drop_caches  # 清除页缓存
echo 2 > /proc/sys/vm/drop_caches  # 清除目录项和inode缓存
echo 3 > /proc/sys/vm/drop_caches  # 清除所有
```

**实际应用：**
- 数据库通常使用直接IO（`O_DIRECT`）绕过Page Cache，自己管理缓存
- Redis使用`O_NOATIME`减少元数据更新
- Kafka依赖Page Cache提高消息读写性能

### Q26: 什么是共享内存？如何实现进程间通信？ ⭐⭐

**答案：**

**共享内存**是允许多个进程访问同一块物理内存的IPC机制，是最快的IPC方式。

**工作原理：**
1. 在物理内存中分配一块区域
2. 将这块物理内存映射到各个进程的虚拟地址空间
3. 进程直接读写这块内存即可通信
4. 需要配合同步机制（信号量、互斥锁）避免竞态条件

**System V共享内存：**
```c
#include <sys/shm.h>
#include <string.h>

// 创建/获取共享内存
int shmid = shmget(key_t key, size_t size, IPC_CREAT | 0666);

// 映射到进程地址空间
char *shmaddr = (char *)shmat(shmid, NULL, 0);

// 写入数据
strcpy(shmaddr, "Hello from process 1");

// 读取数据
printf("Read: %s\n", shmaddr);

// 解除映射
shmdt(shmaddr);

// 删除共享内存
shmctl(shmid, IPC_RMID, NULL);
```

**POSIX共享内存：**
```c
#include <sys/mman.h>
#include <fcntl.h>

// 创建共享内存对象
int fd = shm_open("/my_shm", O_CREAT | O_RDWR, 0666);
ftruncate(fd, size);

// 映射
void *ptr = mmap(NULL, size, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);

// 使用完毕
munmap(ptr, size);
shm_unlink("/my_shm");
```

**查看共享内存：**
```bash
ipcs -m              # 查看System V共享内存
ipcrm -m <shmid>     # 删除共享内存
```

**Java中使用共享内存：**
- 通过MappedByteBuffer映射文件实现
- 使用Unsafe类直接操作内存
- 第三方库如Chronicle Map

### Q27: Linux中进程的内存布局是怎样的？ ⭐⭐

**答案：**

Linux进程中虚拟地址空间的典型布局（从低地址到高地址）：

```
高地址
┌──────────────────┐
│    内核空间       │  用户代码不可直接访问
├──────────────────┤
│    栈（Stack）    │  向下增长，局部变量、函数调用栈帧
│        ↓         │
├──────────────────┤
│                  │
│    空闲区域       │  用于栈和堆的扩展
│                  │
│        ↑         │
├──────────────────┤
│    堆（Heap）     │  向上增长，malloc/new分配
├──────────────────┤
│    BSS段         │  未初始化的全局变量和静态变量
├──────────────────┤
│    数据段（Data） │  已初始化的全局变量和静态变量
├──────────────────┤
│    代码段（Text） │  程序代码（只读）
低地址
```

**各段详细说明：**

| 段 | 内容 | 权限 | 是否共享 |
|----|------|------|---------|
| 代码段（Text） | 机器指令 | 只读+可执行 | 多进程共享 |
| 数据段（Data） | 已初始化全局/静态变量 | 可读写 | 不共享 |
| BSS段 | 未初始化全局/静态变量 | 可读写 | 不共享 |
| 堆（Heap） | 动态分配的内存 | 可读写 | 不共享 |
| 栈（Stack） | 局部变量、函数参数、返回地址 | 可读写 | 不共享 |
| 内存映射区（mmap） | 共享库、文件映射 | 可配置 | 可共享 |

**查看进程内存布局：**
```bash
# 查看进程内存映射
cat /proc/<pid>/maps

# 查看进程内存使用
pmap -x <pid>

# 使用smaps查看详细信息
cat /proc/<pid>/smaps
```

### Q28: 什么是DMA？它在IO过程中有什么作用？ ⭐⭐

**答案：**

**DMA（Direct Memory Access，直接内存访问）** 是一种允许硬件设备直接与内存进行数据传输的技术，不需要CPU参与。

**没有DMA的IO过程：**
```
磁盘 → CPU → 内存
```
- CPU需要全程参与数据搬运
- CPU在搬运数据时无法执行其他任务
- 大量数据传输时CPU利用率极低

**有DMA的IO过程：**
```
磁盘 → DMA控制器 → 内存
```
- CPU只需发送IO指令给DMA控制器
- DMA控制器负责数据搬运
- 搬运完成后DMA发送中断通知CPU
- CPU可以同时执行其他任务

**DMA的工作流程：**
1. CPU设置DMA控制器（源地址、目标地址、传输大小、操作模式）
2. DMA控制器向磁盘控制器发送读请求
3. 磁盘控制器将数据发送到DMA控制器
4. DMA控制器将数据写入内存
5. 传输完成后DMA向CPU发送中断
6. CPU处理中断

**DMA的类型：**
- **飞越DMA（Flyby DMA）**：数据直接在设备和内存之间传输
- **分散收集DMA（Scatter-Gather DMA）**：支持将数据传输到不连续的内存区域

**现代IO过程（DMA + 中断）：**
```
磁盘 → DMA → 内核缓冲区（Page Cache） → CPU拷贝 → 用户缓冲区
```

### Q29: 什么是NUMA架构？对性能有什么影响？ ⭐⭐⭐

**答案：**

**NUMA（Non-Uniform Memory Access，非统一内存访问）** 是一种多处理器架构，每个CPU（或CPU组）有自己的本地内存，访问本地内存比访问远程内存快。

**UMA vs NUMA：**
- **UMA（SMP）**：所有CPU共享同一块内存，访问速度一致
- **NUMA**：每个CPU有自己的本地内存，访问远程内存需要经过互联总线，延迟更高

**NUMA架构示意：**
```
┌──────────────┐        ┌──────────────┐
│  CPU 0       │        │  CPU 1       │
│  本地内存     │◄──────►│  本地内存     │
│  (Node 0)    │ 互联总线 │  (Node 1)    │
└──────────────┘        └──────────────┘
```

**性能影响：**
- 访问本地内存：~80ns
- 访问远程内存：~120ns（增加约50%延迟）
- 如果进程频繁跨NUMA节点访问内存，性能会显著下降

**查看NUMA信息：**
```bash
# 查看NUMA节点
numactl --hardware

# 查看进程的NUMA策略
numactl -p <pid>

# 绑定进程到指定NUMA节点
numactl --cpunodebind=0 --membind=0 command

# 查看NUMA统计
cat /proc/<pid>/numa_maps
```

**Java中的NUMA优化：**
- JVM参数`-XX:+UseNUMA`（默认开启）
- G1垃圾收集器可以感知NUMA
- 大页（Huge Pages）与NUMA配合使用效果更好

### Q30: 如何排查Linux系统CPU负载高的问题？ ⭐⭐⭐

**答案：**

**排查步骤：**

**第一步：确认CPU负载情况**
```bash
# 查看系统负载
uptime
# load average: 1分钟、5分钟、15分钟平均负载
# 负载 > CPU核数 说明系统过载

# 查看CPU使用率
top
# 按1显示各CPU核心使用率
# 注意：us(用户态)、sy(内核态)、wa(IO等待)、hi(硬中断)、si(软中断)
```

**第二步：定位CPU使用率高的进程**
```bash
# 按CPU排序
top -o %CPU

# 查看进程的线程
top -Hp <pid>

# 使用pidstat
pidstat -u 1 -p <pid>
```

**第三步：分析线程在做什么**
```bash
# 查看线程调用栈
jstack <pid> > thread_dump.txt  # Java应用

# 使用perf
perf top -p <pid>
perf record -g -p <pid> sleep 30
perf report

# 使用strace跟踪系统调用
strace -p <pid> -c            # 统计系统调用
strace -p <pid> -e trace=write  # 跟踪指定系统调用
```

**第四步：根据CPU使用类型判断**

| CPU使用类型 | 可能原因 | 排查方向 |
|------------|---------|---------|
| us高 | 用户态计算密集 | 代码逻辑、算法优化、GC |
| sy高 | 系统调用频繁 | 锁竞争、线程上下文切换、系统调用 |
| wa高 | IO等待 | 磁盘性能、SQL慢查询、日志量 |
| hi/si高 | 中断频繁 | 网卡中断、硬件问题 |

**第五步：常见场景分析**

**场景1：Java应用CPU高**
```bash
# 找到CPU使用率最高的Java进程
top -c | grep java

# 找到CPU使用率最高的线程
top -Hp <pid>

# 将线程ID转为16进制
printf "%x\n" <tid>

# 在jstack输出中搜索该线程
jstack <pid> | grep -A 30 "0x<hex_tid>"
```

**场景2：上下文切换过多**
```bash
vmstat 1
# cs列：上下文切换次数
# 如果cs > 几万，说明上下文切换过于频繁

# 查看具体进程的上下文切换
pidstat -w 1
```

**场景3：IO等待高**
```bash
iostat -x 1
# %util接近100%说明磁盘繁忙
# await高说明IO响应慢

# 查看进程的IO
pidstat -d 1
iotop
```
