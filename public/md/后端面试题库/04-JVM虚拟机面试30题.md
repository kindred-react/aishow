## 一、JVM 架构（4题）

### Q1: JVM 的整体架构是怎样的？

**难度**: ⭐⭐

**答案**: JVM 由三个核心子系统组成：类加载子系统、运行时数据区、执行引擎。

**解析**:
```
┌─────────────────────────────────────────────────────────┐
│                    类加载子系统                           │
│  (加载 -> 验证 -> 准备 -> 解析 -> 初始化)                 │
└────────────────────┬────────────────────────────────────┘
                     v
┌─────────────────────────────────────────────────────────┐
│                    运行时数据区                           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │  方法区   │  │    堆    │  │ 虚拟机栈  │              │
│  │ (元空间)  │  │(新生代+   │  │  (栈帧)   │              │
│  │          │  │ 老年代)   │  │          │              │
│  └──────────┘  └──────────┘  └──────────┘              │
│  ┌──────────┐  ┌──────────┐                             │
│  │ 本地方法栈│  │ 程序计数器│                             │
│  └──────────┘  └──────────┘                             │
└────────────────────┬────────────────────────────────────┘
                     v
┌─────────────────────────────────────────────────────────┐
│                    执行引擎                               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │  解释器   │  │ JIT编译器 │  │  GC回收器 │              │
│  └──────────┘  └──────────┘  └──────────┘              │
└─────────────────────────────────────────────────────────┘
```

### Q2: JRE、JDK、JVM 的关系？

**难度**: ⭐

**答案**: JDK 包含 JRE，JRE 包含 JVM。

**解析**:
```
JDK (Java Development Kit)
├── JRE (Java Runtime Environment)
│   ├── JVM (Java Virtual Machine)
│   │   ├── 类加载器
│   │   ├── 运行时数据区
│   │   ├── 执行引擎
│   │   └── 本地库接口 (JNI)
│   └── Java 核心类库 (rt.jar)
└── 开发工具 (javac, jdb, javadoc 等)
```

### Q3: Java 代码的执行流程？

**难度**: ⭐

**答案**: `.java` 源文件 -> `javac` 编译 -> `.class` 字节码 -> 类加载器加载 -> 运行时数据区 -> 执行引擎（解释执行或 JIT 编译执行）。

**解析**:
- 字节码是一种中间表示，与平台无关
- 执行引擎有两种执行方式：
  1. 解释执行：逐条解释字节码，启动快但执行慢
  2. JIT 编译：将热点代码编译为本地机器码，启动慢但执行快

```bash
# 查看字节码
javac HelloWorld.java
javap -c HelloWorld.class   # 简洁字节码
javap -v HelloWorld.class   # 详细字节码
```

### Q4: 什么是字节码？为什么要使用字节码？

**难度**: ⭐⭐

**答案**: 字节码是 Java 源代码编译后的中间表示（`.class` 文件），由 JVM 执行。

**解析**:
- 字节码的优势：
  1. **跨平台**：一次编译，到处运行（Write Once, Run Anywhere）
  2. **安全性**：字节码可以被验证（Verifier），防止恶意代码
  3. **紧凑性**：比机器码更紧凑，便于网络传输
  4. **可优化**：JVM 可以在运行时对字节码进行优化（JIT 编译）

## 二、类加载（6题）

### Q5: 类加载的过程分为哪几个阶段？

**难度**: ⭐⭐

**答案**: 类加载分为 5 个阶段：加载（Loading）-> 链接（Linking：验证、准备、解析）-> 初始化（Initialization）。

**解析**:
```
加载 (Loading)
    ↓
链接 (Linking)
    ├─ 验证 (Verification) - 字节码合法性检查
    ├─ 准备 (Preparation) - 为静态变量分配内存并设置默认值
    └─ 解析 (Resolution) - 符号引用转换为直接引用
    ↓
初始化 (Initialization) - 执行 <clinit> 方法，赋初始值
```

- **加载**：通过类的全限定名获取二进制字节流，生成 Class 对象
- **验证**：文件格式验证、元数据验证、字节码验证、符号引用验证
- **准备**：为静态变量分配内存并设置**默认值**（非初始值）
  - `static int a = 10` -> 准备阶段 a = 0，初始化阶段 a = 10
  - `static final int b = 10` -> 准备阶段 b = 10（常量在编译期放入常量池）
- **解析**：将常量池中的符号引用替换为直接引用
- **初始化**：执行 `<clinit>` 方法（静态变量赋值 + 静态代码块）

### Q6: 什么是双亲委派模型？

**难度**: ⭐⭐⭐

**答案**: 双亲委派模型（Parents Delegation Model）是指类加载器收到类加载请求时，先委托给父类加载器尝试加载，父类加载器无法加载时，子类加载器才尝试加载。

**解析**:
```
Bootstrap ClassLoader (启动类加载器)
    ↓ 加载 rt.jar、tools.jar
Extension/Platform ClassLoader (扩展/平台类加载器)
    ↓ JDK 9+ 加载模块路径（JDK 8 及以下加载 JAVA_HOME/lib/ext 下的 jar，JDK 9+ 已移除 ext 目录）
Application/System ClassLoader (应用/系统类加载器)
    ↓ 加载 classpath 下的类
Custom ClassLoader (自定义类加载器)
```

- 双亲委派的好处：
  1. **安全性**：防止核心类被篡改（如自定义 `java.lang.String`）
  2. **避免重复加载**：父加载器已加载的类，子加载器不需要重复加载

---

### Q7: 有哪些打破双亲委派的场景？

**难度**: ⭐⭐⭐

**答案**:

1. **SPI 机制**（Service Provider Interface）：如 JDBC、JNDI
   - 问题：核心类（如 `java.sql.Driver`）在 rt.jar 中，由 Bootstrap 加载，但实现类在 classpath 中，Bootstrap 无法加载
   - 解决：使用**线程上下文类加载器**（Thread Context ClassLoader）打破委派

2. **Tomcat 类加载**：每个 Web 应用有独立的类加载器，优先加载 Web 应用自己的类
   - 原因：不同 Web 应用可能依赖同一个库的不同版本

3. **OSGi/模块化**：每个模块有自己的类加载器，形成网状结构而非树状

4. **热部署/热加载**：自定义类加载器，丢弃旧的类加载器并创建新的

```java
// JDBC 打破双亲委派
// DriverManager 在 rt.jar 中，由 Bootstrap 加载
// MySQL 驱动在 classpath 中，Bootstrap 无法加载
// 使用线程上下文类加载器加载驱动
ServiceLoader<Driver> drivers = ServiceLoader.load(Driver.class);
```

### Q8: 类加载器有哪些？各自加载什么？

**难度**: ⭐⭐

**答案**:

| 类加载器 | 加载路径 | 说明 |
|---------|---------|------|
| Bootstrap ClassLoader | `JAVA_HOME/lib` | C++ 实现，加载核心类库 |
| Platform ClassLoader | JDK 9+ 加载模块路径（替代已移除的 JAVA_HOME/lib/ext） | 加载扩展/平台类库 |
| Application ClassLoader | classpath | 加载应用类 |
| Custom ClassLoader | 自定义路径 | 用户自定义 |

```java
ClassLoader systemCL = ClassLoader.getSystemClassLoader();
// jdk.internal.loader.ClassLoaders$AppClassLoader
ClassLoader platformCL = systemCL.getParent();
// jdk.internal.loader.ClassLoaders$PlatformClassLoader
ClassLoader bootstrapCL = platformCL.getParent();
// null（Bootstrap 是 C++ 实现，Java 中表示为 null）
```

### Q9: 类什么时候被初始化？（主动引用 vs 被动引用）

**难度**: ⭐⭐

**答案**: 类的初始化时机由 JVM 规范严格规定，只有**主动引用**才会触发初始化。

**解析**:
主动引用（触发初始化）：
1. `new`、`getstatic`、`putstatic`、`invokestatic` 指令
2. 反射调用（`Class.forName()`）
3. 初始化子类时，父类未初始化则先初始化父类
4. JVM 启动时的主类（含 `main()` 方法的类）

被动引用（不触发初始化）：
1. 通过子类引用父类的静态字段，只初始化父类
2. 定义数组引用：`MyClass[] arr = new MyClass[10]` 不会触发 MyClass 初始化
3. 引用常量：`static final` 常量在编译期放入调用类的常量池

```java
class Parent {
    static int a = 1;
    static { System.out.println("Parent init"); }
}
class Child extends Parent {
    static int b = 2;
    static { System.out.println("Child init"); }
}

// 通过子类引用父类静态字段
System.out.println(Child.a); // 只输出 "Parent init"，Child 不初始化
```

### Q10: 如何自定义类加载器？

**难度**: ⭐⭐⭐

**答案**: 继承 `ClassLoader` 类，重写 `findClass()` 方法。如果要打破双亲委派，还需重写 `loadClass()` 方法。

**解析**:
```java
public class CustomClassLoader extends ClassLoader {
    private String classPath;

    public CustomClassLoader(String classPath) {
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            byte[] data = loadByte(name);
            return defineClass(name, data, 0, data.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name);
        }
    }

    private byte[] loadByte(String name) throws IOException {
        name = name.replaceAll("\\.", "/");
        FileInputStream fis = new FileInputStream(classPath + "/" + name + ".class");
        byte[] data = new byte[fis.available()];
        fis.read(data);
        fis.close();
        return data;
    }

    // 打破双亲委派
    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        synchronized (getClassLoadingLock(name)) {
            Class<?> c = findLoadedClass(name);
            if (c == null) {
                if (name.startsWith("com.example")) {
                    c = findClass(name); // 自定义类不委托给父加载器
                } else {
                    c = super.loadClass(name, resolve); // 其他类走双亲委派
                }
            }
            return c;
        }
    }
}
```

## 三、内存模型（4题）

### Q11: JVM 运行时数据区包含哪些部分？

**难度**: ⭐⭐

**答案**: 运行时数据区分为**线程共享**和**线程私有**两部分。

**解析**:
- **线程共享**：
  - **堆（Heap）**：存放对象实例和数组，GC 主要管理区域
  - **方法区（Method Area）/ 元空间（Metaspace）**：存放类信息、常量、静态变量

- **线程私有**：
  - **虚拟机栈（VM Stack）**：每个方法执行时创建栈帧，包含局部变量表、操作数栈、动态链接、方法返回地址
  - **本地方法栈（Native Method Stack）**：为 Native 方法服务
  - **程序计数器（PC Register）**：记录当前线程执行的字节码行号

```
┌─────────────────────────────────────────┐
│              线程共享                     │
│  ┌──────────────┐  ┌──────────────┐     │
│  │     堆       │  │   方法区      │     │
│  │  (新生代+     │  │  (元空间)     │     │
│  │   老年代)    │  │              │     │
│  └──────────────┘  └──────────────┘     │
├─────────────────────────────────────────┤
│              线程私有（每个线程一份）       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │ 虚拟机栈  │ │ 本地方法栈│ │ 程序计数器│ │
│  └──────────┘ └──────────┘ └──────────┘ │
└─────────────────────────────────────────┘
```

---

### Q12: 堆内存是如何划分的？

**难度**: ⭐⭐

**答案**: JDK 8+ 堆内存分为**新生代**（Eden + S0 + S1）和**老年代**，默认比例 1:2。

**解析**:
```
堆 (100%)
├── 新生代 (33.3%)
│   ├── Eden (80%)         新对象分配在这里
│   ├── Survivor 0 (10%)   存活对象复制到这里
│   └── Survivor 1 (10%)   S0 和 S1 交替使用
└── 老年代 (66.7%)         长期存活的对象
```

- 新对象优先在 Eden 区分配
- Minor GC 后存活对象复制到 Survivor 区（年龄 +1）
- 年龄达到阈值（默认 15）的对象晋升到老年代
- 大对象直接进入老年代（`-XX:PretenureSizeThreshold`）

```bash
# 相关 JVM 参数
-Xms512m              # 初始堆大小
-Xmx1024m             # 最大堆大小
-Xmn256m              # 新生代大小
-XX:SurvivorRatio=8   # Eden:S0:S1 = 8:1:1
-XX:NewRatio=2        # 新生代:老年代 = 1:2
-XX:MaxTenuringThreshold=15  # 晋升老年代年龄阈值
```

### Q13: 方法区、永久代、元空间的区别？

**难度**: ⭐⭐⭐

**答案**:

| 版本 | 实现方式 | 存储位置 | 大小限制 |
|------|---------|---------|---------|
| JDK 7 及之前 | 永久代（PermGen） | JVM 内存 | 固定大小，容易 OOM |
| JDK 8+ | 元空间（Metaspace） | 本地内存（Native Memory） | 受物理内存限制 |

**解析**:
- JDK 7：字符串常量池从永久代移到堆中
- JDK 8：永久代完全移除，替换为元空间
- 元空间存储：类信息、字段信息、方法信息、常量池
- 元空间的优势：不受 JVM 内存限制，不容易 OOM；GC 效率更高

```bash
# 元空间相关参数
-XX:MetaspaceSize=256m         # 初始元空间大小
-XX:MaxMetaspaceSize=512m      # 最大元空间大小
-XX:+PrintGCDetails            # 打印 GC 详细日志
```

### Q14: 虚拟机栈中栈帧包含什么？StackOverflowError 是怎么产生的？

**难度**: ⭐⭐

**答案**: 栈帧（Stack Frame）包含：局部变量表、操作数栈、动态链接、方法返回地址。StackOverflowError 是栈深度超过限制时抛出的错误。

**解析**:
- **局部变量表**：存放方法参数和局部变量（基本类型和引用）
- **操作数栈**：方法执行的工作区，用于算术运算和方法调用
- **动态链接**：指向运行时常量池的方法引用
- **方法返回地址**：方法正常退出或异常退出后返回到调用者的地址

```java
// StackOverflowError：递归过深
public void recursion() {
    recursion(); // 没有终止条件，栈帧不断压栈
}
// Exception in thread "main" java.lang.StackOverflowError

// 解决方案：设置更大的栈大小
// -Xss1m（默认 512k~1m）
```

## 四、垃圾回收（8题）

### Q15: 如何判断对象可以被回收？

**难度**: ⭐⭐

**答案**: 两种方式：**引用计数法**和**可达性分析**。JVM 使用可达性分析。

**解析**:
- **引用计数法**（已淘汰）：对象被引用一次计数 +1，引用断开计数 -1，计数为 0 时回收
  - 缺点：无法解决循环引用问题

- **可达性分析**（JVM 使用）：从 GC Roots 开始向下搜索，不可达的对象可以被回收
  - GC Roots 包括：
    1. 虚拟机栈中的局部变量
    2. 方法区中的静态变量和常量
    3. 本地方法栈中的 JNI 引用
    4. JVM 内部引用（基本类型对应的 Class 对象、常驻异常等）

---

### Q16: 常见的 GC 算法有哪些？

**难度**: ⭐⭐

**答案**:

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 标记-清除 | 标记可达对象，清除未标记的 | 简单 | 产生内存碎片 |
| 标记-复制 | 将存活对象复制到另一半空间 | 无碎片 | 空间利用率低（50%） |
| 标记-整理 | 标记后将存活对象向一端移动 | 无碎片 | 移动对象有开销 |

**解析**:
- **新生代**使用标记-复制算法（Eden + S0/S1 交替使用）
- **老年代**使用标记-整理算法（或标记-清除）
- 分代收集的理由：大部分对象朝生夕灭，少数对象长期存活

---

### Q17: 常见的垃圾回收器有哪些？

**难度**: ⭐⭐⭐

**答案**:

| 回收器 | 分代 | 算法 | 特点 | 适用场景 |
|--------|------|------|------|---------|
| Serial | 新生代 | 标记-复制 | 单线程，STW | 客户端模式 |
| ParNew | 新生代 | 标记-复制 | 多线程，STW | 配合 CMS |
| Parallel Scavenge | 新生代 | 标记-复制 | 多线程，吞吐量优先 | 后台计算 |
| Serial Old | 老年代 | 标记-整理 | 单线程，STW | 客户端模式 |
| CMS | 老年代 | 标记-清除 | 低延迟，多阶段并发 | Web 应用 |
| G1 | 全堆 | 标记-整理+复制 | 分区收集，可预测停顿 | 服务器应用（JDK 9 默认） |
| ZGC | 全堆 | 着色指针+读屏障 | 超低延迟（<1ms） | 大内存、低延迟 |

### Q18: CMS 回收器的工作流程？

**难度**: ⭐⭐⭐

**答案**: CMS（Concurrent Mark Sweep）分为 4 个阶段：

```
1. 初始标记 (Initial Mark)     —— STW，标记 GC Roots 直接关联的对象
2. 并发标记 (Concurrent Mark)   —— 与用户线程并发，遍历对象图
3. 重新标记 (Remark)            —— STW，修正并发标记期间发生变化的对象
4. 并发清除 (Concurrent Sweep)  —— 与用户线程并发，清除垃圾对象
```

**解析**:
- CMS 的优点：大部分阶段与用户线程并发，低延迟
- CMS 的缺点：
  1. 使用标记-清除算法，产生内存碎片
  2. 对 CPU 资源敏感（并发阶段占用 CPU）
  3. 无法处理"浮动垃圾"（并发标记期间新产生的垃圾）
  4. 产生 Concurrent Mode Failure 时退化为 Serial Old
- JDK 14 中 CMS 被完全移除

### Q19: G1 回收器的工作原理？

**难度**: ⭐⭐⭐

**答案**: G1（Garbage-First）将堆划分为多个大小相等的 Region（1~32MB），每个 Region 可以是 Eden、Survivor、Old 或 Humongous。

**解析**:
```
G1 堆布局：
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ S  │ O  │ E  │ H  │ O  │ E  │ S  │
└────┴────┴────┴────┴────┴────┴────┴────┘
  E=Eden  S=Survivor  O=Old  H=Humongous
```

- G1 的核心特点：
  1. **优先回收价值最大的 Region**（Garbage-First 名称由来）
  2. **可预测的停顿时间**：通过 `-XX:MaxGCPauseMillis` 设置目标
  3. **无内存碎片**：使用标记-整理+复制算法
  4. **分区收集**：不需要全堆扫描

- G1 回收流程：
  1. 初始标记（STW）
  2. 并发标记
  3. 最终标记（STW）
  4. 筛选回收（STW，选择回收价值高的 Region）

```bash
# G1 相关参数
-XX:+UseG1GC                    # 使用 G1
-XX:MaxGCPauseMillis=200        # 目标停顿时间
-XX:G1HeapRegionSize=8m         # Region 大小
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发标记的堆占用率
```

### Q20: ZGC 的原理？为什么能做到亚毫秒级停顿？

**难度**: ⭐⭐⭐

**答案**: ZGC（Z Garbage Collector）通过**着色指针（Colored Pointers）**和**读屏障（Load Barrier）**实现几乎全程并发，停顿时间 < 1ms。

**解析**:
- ZGC 的核心技术：
  1. **着色指针**：在指针的空闲位（高 4 位）存储 GC 状态信息（Marked0、Marked1、Remapped、Finalizable）
  2. **读屏障**：每次读取引用时检查指针状态，如果对象正在移动则帮助完成移动
  3. **并发整理**：对象移动与用户线程并发执行，通过读屏障保证正确性

- ZGC 回收流程：
  1. 并发标记
  2. 并发预备重定位
  3. 并发重定位（移动对象）
  4. 并发重映射（更新引用）

- ZGC 支持 TB 级别堆内存，停顿时间与堆大小无关
- JDK 16 引入 ZGC 的并发线程栈扫描，进一步减少 STW

---

### Q21: 什么是 Minor GC、Major GC 和 Full GC？

**难度**: ⭐⭐

**答案**:

| GC 类型 | 范围 | 触发条件 | 停顿时间 |
|---------|------|---------|---------|
| Minor GC | 新生代 | Eden 区满 | 短（几十毫秒） |
| Major GC | 老年代 | 老年代空间不足 | 较长 |
| Full GC | 全堆 + 元空间 | 老年代空间不足、System.gc()、元空间不足 | 最长 |

**解析**:
- Minor GC 频率高但速度快（新生代空间小）
- Full GC 尽量避免，会导致较长的 STW
- 触发 Full GC 的常见原因：
  1. 老年代空间不足
  2. 元空间不足
  3. System.gc()（建议加 `-XX:+DisableExplicitGC`）
  4. promotion failure（ Survivor 区放不下存活对象）

---

### Q22: 对象什么时候进入老年代？

**难度**: ⭐⭐

**答案**:
1. **年龄达到阈值**：对象在 Survivor 区每经过一次 Minor GC 年龄 +1，达到阈值（默认 15）晋升
2. **大对象直接进入**：超过 `-XX:PretenureSizeThreshold` 的对象直接在老年代分配
3. **动态年龄判断**：Survivor 区中相同年龄的对象大小总和超过 Survivor 空间一半，大于等于该年龄的对象直接晋升
4. **空间分配担保失败**：Minor GC 后 Survivor 放不下存活对象

## 五、JIT 编译（3题）

### Q23: 什么是逃逸分析？

**难度**: ⭐⭐⭐

**答案**: 逃逸分析是 JIT 编译器的优化技术，分析对象的作用域是否"逃逸"出方法或线程。如果对象不逃逸，可以进行**栈上分配**、**标量替换**、**锁消除**等优化。

**解析**:
- 对象逃逸的三种级别：
  1. **不逃逸**：对象只在方法内部使用
  2. **方法逃逸**：对象作为方法参数传递或返回
  3. **线程逃逸**：对象被赋值给类变量或其他线程可访问的变量

- 优化手段：
  - **栈上分配**：不逃逸的对象在栈上分配，方法结束自动回收，减轻 GC 压力
  - **标量替换**：将对象拆解为基本类型，在栈上分配
  - **锁消除**：不逃逸的对象不可能被其他线程访问，消除不必要的 synchronized

```java
// 锁消除示例
public String concat(String s1, String s2) {
    // sb 不逃出方法，synchronized 会被消除
    StringBuffer sb = new StringBuffer();
    sb.append(s1);
    sb.append(s2);
    return sb.toString();
}
```

---

### Q24: 什么是 JIT 编译的分层编译？

**难度**: ⭐⭐

**答案**: JIT 编译器将热点代码编译为本地机器码，分层编译分为三个层次。

**解析**:
```
解释执行 (Interpreter)
    ↓ 执行次数达到阈值
C1 编译 (Client Compiler) —— 快速编译，简单优化
    ↓ 执行次数进一步增加
C2 编译 (Server Compiler) —— 深度优化，编译慢但执行快
```

- **C1 编译器**：编译快，优化少（内联、去虚拟化等）
- **C2 编译器**：编译慢，深度优化（逃逸分析、循环展开、分支预测等）
- 热点代码判断：方法调用计数器 + 回边计数器（循环），超过阈值触发编译

```bash
# JIT 相关参数
-XX:+PrintCompilation    # 打印 JIT 编译信息
-XX:CompileThreshold=10000  # 方法调用计数器阈值
```

### Q25: 什么是方法内联？

**难度**: ⭐⭐

**答案**: 方法内联是将目标方法的代码直接替换到调用处，消除方法调用的开销（压栈、跳转、出栈）。

**解析**:
```java
// 内联前
public int add(int a, int b) { return a + b; }
public int compute() { return add(1, 2); }

// 内联后
public int compute() { return 1 + 2; }
```

- 内联是 JIT 最重要的优化之一，也是其他优化（如逃逸分析）的基础
- 热点方法（调用频繁的小方法）更容易被内联
- `-XX:MaxInlineSize` 控制内联方法的最大字节码大小

## 六、内存管理（3题）

### Q26: 如何排查 OOM 问题？

**难度**: ⭐⭐⭐

**答案**: OOM 排查步骤：

```bash
# 1. 获取 Heap Dump
-XX:+HeapDumpOnOutOfMemoryError           # OOM 时自动 dump
-XX:HeapDumpPath=/tmp/heapdump.hprof      # dump 文件路径
# 或手动 dump
jmap -dump:format=b,file=heap.hprof <pid>

# 2. 分析 Heap Dump
# 使用 MAT (Memory Analyzer Tool) 或 VisualVM
# 查看对象占用、GC Roots、Dominator Tree

# 3. 常见 OOM 场景
# - 堆 OOM：对象太多或内存泄漏
# - 元空间 OOM：动态生成类过多（如 CGLIB 代理）
# - 直接内存 OOM：NIO ByteBuffer.allocateDirect()
```

**解析**:
- 内存泄漏的常见原因：
  1. 静态集合类不断添加元素
  2. ThreadLocal 未 remove
  3. 资源未关闭（连接、流）
  4. 内部类持有外部类引用
  5. 监听器和回调未注销

### Q27: 什么是直接内存？和堆内存有什么区别？

**难度**: ⭐⭐

**答案**: 直接内存（Direct Memory）是通过 `Unsafe.allocateMemory()` 分配的堆外内存，不受 JVM 堆大小限制，由操作系统直接管理。

**解析**:
- 使用场景：NIO 的 `ByteBuffer.allocateDirect()`、Netty 的堆外内存池
- 优点：减少一次堆内到堆外的数据拷贝（零拷贝），提高 IO 性能
- 缺点：不受 GC 管理，需要手动释放；分配和回收开销大
- OOM 时也会抛出 `OutOfMemoryError: Direct buffer memory`

```java
// 直接内存分配
ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024); // 1MB 直接内存

// 直接内存不受 -Xmx 限制，受 -XX:MaxDirectMemorySize 限制
```

## 七、JMM（2题）

### Q29: Java 内存模型（JMM）是什么？

**难度**: ⭐⭐⭐

**答案**: JMM（Java Memory Model）是 Java 虚拟机规范中定义的一套规则，描述多线程环境下变量的访问规则，保证并发编程的正确性。

**解析**:
- JMM 规定了所有变量都存储在**主内存**中，每个线程有自己的**工作内存**（CPU 缓存/寄存器）
- 线程对变量的操作必须在工作内存中进行，不能直接读写主内存
- JMM 的三大特性：
  1. **原子性**：基本数据类型的读写是原子的（long/double 除外），使用 synchronized/Lock/AtomicXxx 保证复合操作原子性
  2. **可见性**：volatile、synchronized、final 保证
  3. **有序性**：volatile、synchronized 保证

```
线程A的工作内存          主内存          线程B的工作内存
    │                    │                    │
    │── load ──────────> │                    │
    │                    │ <───────── load ──│
    │<── read ────────── │                    │
    │                    │ ───────── read ──>│
```

### Q30: happens-before 原则有哪些？

**难度**: ⭐⭐⭐

**答案**: happens-before 是 JMM 中定义的操作之间的偏序关系，如果操作 A happens-before 操作 B，则 A 的结果对 B 可见。

**解析**:
8 条 happens-before 规则：
1. **程序顺序规则**：同一线程中，前面的操作 happens-before 后面的操作
2. **volatile 变量规则**：volatile 写 happens-before 后续对该变量的读
3. **锁规则**：解锁 happens-before 后续对同一把锁的加锁
4. **线程启动规则**：Thread.start() happens-before 该线程的所有操作
5. **线程终止规则**：线程的所有操作 happens-before Thread.join() 返回
6. **线程中断规则**：interrupt() happens-before 被中断线程检测到中断
7. **对象终结规则**：构造函数执行完毕 happens-before finalize()
8. **传递性**：如果 A happens-before B，B happens-before C，则 A happens-before C

```java
// 示例：锁规则
// 线程A
synchronized(lock) {
    x = 1;  // 写入 x
}
// 线程B
synchronized(lock) {
    System.out.println(x); // 一定能看到 x = 1
}
// 解锁 happens-before 加锁
```
