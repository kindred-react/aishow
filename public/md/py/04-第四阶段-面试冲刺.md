## 4.1 Python高级面试题 TOP 20（含标准答案）
Q1: GIL是什么？对多线程有什么影响？
GIL(Global Interpreter Lock)是CPython的全局互斥锁，同一时刻只允许一个线程执行Python字节码。

影响：CPU密集型任务多线程无法利用多核（甚至更慢，因为线程切换开销），IO密集型影响不大（IO等待时释放GIL）。

绕过方式：
1. 多进程(multiprocessing) - CPU密集型首选
2. 协程(asyncio) - IO密集型首选
3. C扩展/Cython - 将计算密集部分用C实现
4. subprocess - 调用外部进程

Q2: 装饰器的原理？手写一个带参数的装饰器
装饰器本质是高阶函数+闭包，接收一个函数作为参数，返回一个增强版的新函数。

关键点：
- functools.wraps保留原函数元信息
- *args/**kwargs透传参数
- 嵌套函数形成闭包

手写带参数装饰器见第一阶段文档1.2.1节。

Q3: 深拷贝和浅拷贝的区别？
浅拷贝(shallow copy)：创建新对象，但内部元素的引用不变。list.copy()、copy.copy()
深拷贝(deep copy)：递归复制所有内部元素。copy.deepcopy()

示例：
a = [[1,2], [3,4]]
b = a.copy()  # 浅拷贝
b[0][0] = 99  # a[0][0]也变成99！

c = copy.deepcopy(a)  # 深拷贝
c[0][0] = 88  # a不受影响

Q4: 可变对象和不可变对象有哪些？
不可变(immutable)：int, float, str, tuple, bool, frozenset
可变(mutable)：list, dict, set, 自定义类实例

关键区别：
- 不可变对象可以作为字典的key
- 函数默认参数使用可变对象有陷阱
- 不可变对象线程安全

Q5: 生成器和迭代器的区别？
迭代器：实现了__iter__和__next__协议的对象
生成器：使用yield关键字创建的迭代器（更简洁的写法）

区别：
- 生成器是迭代器的子集
- 生成器自动实现迭代器协议
- 生成器支持惰性求值，节省内存
- yield from可以委托给子生成器

Q6: asyncio的原理？协程和线程的区别？
asyncio基于事件循环(Event Loop) + 协程实现异步IO。

协程 vs 线程：
- 协程在用户态切换，线程在内核态切换
- 协程切换开销极小（纳秒级），线程切换较慢（微秒级）
- 协程单线程运行，无线程安全问题
- 协程适合IO密集型，不适合CPU密集型
- 一个线程可以运行数千个协程

Q7: Python的垃圾回收机制？
三种机制：
1. 引用计数（主）：每个对象维护引用计数，为0立即回收
2. 标记-清除：处理循环引用的容器对象
3. 分代回收：分0/1/2三代，新对象在0代，存活越久代越高，扫描频率越低

gc模块：gc.get_threshold() 默认返回 (700, 10, 10)，含义是：0代回收 700 次后触发 1 代回收，1代回收 10 次后触发 2 代回收（具体值因 CPython 版本而异）

Q8: 闭包是什么？循环中的闭包陷阱？
闭包：函数嵌套中，内层函数引用外层函数的变量，外层函数返回内层函数。

循环陷阱：
funcs = [lambda: i for i in range(3)]
# 所有函数都返回2（因为i最终是2）

解决：
funcs = [lambda i=i: i for i in range(3)]  # 默认参数绑定

Q9: 元类是什么？有什么应用场景？
元类是"类的类"，控制类的创建过程。

type()是默认元类，可以通过继承type创建自定义元类。

应用场景：
1. Django ORM：Model元类自动创建数据库表
2. 单例模式：元类控制只创建一个实例
3. 接口检查：元类验证子类是否实现了所有抽象方法
4. 属性自动注册：元类自动收集带有特定装饰器的方法

Q10: @staticmethod和@classmethod的区别？
@staticmethod：静态方法，不接收self或cls参数，与类无关的工具函数
@classmethod：类方法，接收cls参数，可以访问类属性和调用类方法

使用场景：
- staticmethod：纯工具函数，如数学计算
- classmethod：工厂方法（替代构造函数）、访问类变量

## 4.2 数据库面试题 TOP 10
Q1: 如何优化一个慢SQL？
1. EXPLAIN查看执行计划，关注type/key/rows/Extra
2. 确认是否走了索引，避免ALL全表扫描
3. 检查索引是否失效（函数/左模糊/运算/隐式转换）
4. 优化SQL写法（避免SELECT *、子查询改JOIN）
5. 添加合适的索引（联合索引注意最左前缀）
6. 考虑分库分表（数据量太大时）

Q2: 事务的ACID特性？隔离级别？
ACID：原子性(undo log)、一致性(约束+事务)、隔离性(MVCC+锁)、持久性(redo log)

四种隔离级别：读未提交、读已提交、可重复读(MySQL默认)、串行化

InnoDB在RR级别通过MVCC+Next-Key Lock基本解决了幻读

Q3: MVCC原理？
多版本并发控制。每行数据有两个隐藏列：trx_id(最后修改的事务ID)和roll_pointer(undo log指针)。

ReadView包含：活跃事务ID列表、最小活跃事务ID、下一个要分配的事务ID。

可见性判断：trx_id < min_trx_id(可见)、trx_id > max_trx_id(不可见)、在活跃列表中(不可见)

Q4: Redis缓存穿透/击穿/雪崩？
穿透：查询不存在的数据。解决：布隆过滤器/缓存空值
击穿：热点Key过期。解决：互斥锁/永不过期+异步更新
雪崩：大量Key同时过期。解决：随机过期时间/多级缓存/Redis高可用

详见第二阶段文档2.2.2节

Q5: Redis分布式锁怎么实现？
SET key value NX EX（原子操作）
释放锁用Lua脚本（先GET判断再DEL）

问题：
- 锁过期但业务未完成 -> 看门狗自动续期(Redisson)
- 主从切换导致锁丢失 -> RedLock算法(多节点)

Q6: 消息队列如何保证不丢失？
生产者：confirm模式/事务模式
Broker：持久化(durable)+集群
消费者：手动ACK（处理完再确认）

全链路保障：生产者confirm -> Broker持久化 -> 消费者手动ACK

Q7: 如何处理消息重复消费？
消费者端实现幂等性：
1. 唯一消息ID + Redis SETNX去重
2. 数据库唯一索引（如订单号）
3. 乐观锁（版本号机制）
4. 状态机（已处理的状态不再处理）

Q8: 分库分表方案？
垂直拆分：按业务模块拆到不同库
水平拆分：同一张表按分片键拆到多表/多库

分片键选择：高基数、查询频率高
跨库JOIN：数据冗余/应用层聚合
分布式ID：雪花算法/Leaf

Q9: 数据库连接池原理？
预先创建一批连接放在池中，需要时从池中获取，用完归还。

关键参数：
- pool_size：初始连接数
- max_overflow：最大溢出连接数
- pool_timeout：获取连接超时时间
- pool_recycle：连接回收时间

SQLAlchemy默认使用QueuePool

Q10: SQL和NoSQL的区别？选型？
SQL(MySQL/PostgreSQL)：结构化数据、ACID事务、复杂查询、关系型
NoSQL(Redis/MongoDB)：灵活schema、高并发、水平扩展、特定场景

选型：
- 需要事务和复杂查询 -> SQL
- 需要高并发和灵活schema -> NoSQL
- 实际项目通常SQL + Redis组合使用

LRU缓存实现（面试高频手写题）：
from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity: int):
        self.cache = OrderedDict()
        self.capacity = capacity

    def get(self, key: int) -> int:
        if key not in self.cache:
            return -1
        # 移到末尾（最近使用）
        self.cache.move_to_end(key)
        return self.cache[key]

    def put(self, key: int, value: int) -> None:
        if key in self.cache:
            self.cache.move_to_end(key)
        self.cache[key] = value
        if len(self.cache) > self.capacity:
            # 移除最久未使用的（第一个）
            self.cache.popitem(last=False)

# 时间复杂度：get O(1), put O(1)
# 空间复杂度：O(capacity)

### 1. 网络协议面试题 TOP 10（⭐高频）

#### Q1: TCP 三次握手/四次挥手

```
三次握手（建立连接）：
客户端                    服务端
  |---- SYN (seq=x) ---->|
  |<-- SYN+ACK (seq=y, ack=x+1) --|
  |---- ACK (ack=y+1) -->|
  |                       |
  |===== 连接建立 ========|

为什么需要三次？
- 客户端确认服务端的接收和发送能力
- 服务端确认客户端的接收和发送能力
- 防止已失效的连接请求到达服务端（两次握手无法防止）

四次挥手（断开连接）：
客户端                    服务端
  |---- FIN (seq=u) ---->|  (客户端不再发送数据)
  |<-- ACK (ack=u+1) ----|  (服务端确认)
  |                       |  (服务端继续发送未完成的数据)
  |<-- FIN (seq=w) ----|  (服务端也不再发送数据)
  |---- ACK (ack=w+1) -->|
  |                       |
  |===== 连接断开 ========|  (客户端等待 2MSL 后关闭)

为什么需要四次？
- TCP 是全双工的，每个方向都需要单独关闭
- 服务端收到 FIN 后可能还有数据要发送，所以先回 ACK，等数据发完再发 FIN

TIME_WAIT 状态：
- 持续 2MSL（最大报文段生存时间，约 60 秒）
- 目的：确保最后一个 ACK 到达服务端；让网络中残留的报文过期
- 如果 TIME_WAIT 过多：开启 SO_REUSEADDR 或减少 MSL
```

#### Q2: HTTP 状态码

```
2xx 成功：
  200 OK              - 请求成功
  201 Created         - 资源创建成功
  204 No Content      - 成功但无返回内容

3xx 重定向：

#### Q3: HTTPS 握手过程

```
1. 客户端发送 Client Hello：
   - 支持的 TLS 版本
   - 支持的加密套件列表
   - 随机数（Random_C）

2. 服务端发送 Server Hello：
   - 选择的 TLS 版本
   - 选择的加密套件
   - 随机数（Random_S）
   - 服务端证书（包含公钥）

3. 客户端验证证书：
   - 检查证书是否可信（CA 链）
   - 检查域名是否匹配
   - 检查证书是否过期

4. 客户端生成预主密钥（Pre-Master Secret）：
   - 用服务端公钥加密 Pre-Master Secret
   - 发送给服务端

5. 双方生成会话密钥：
   - Master Secret = PRF(Pre-Master Secret, Random_C, Random_S)
   - 从 Master Secret 派生出对称加密密钥

6. 客户端发送 Finished（加密）：
   - 用会话密钥加密的完成消息

7. 服务端发送 Finished（加密）：
   - 用会话密钥加密的完成消息

8. 后续通信使用对称加密（AES-GCM 等）

为什么用对称加密传输数据？
- 非对称加密（RSA）性能差，只用于密钥交换
- 对称加密（AES）性能好，用于数据传输
```

#### Q2: 进程与线程

```
进程 vs 线程：
- 进程是资源分配的基本单位，线程是 CPU 调度的基本单位
- 进程有独立的地址空间，线程共享进程的地址空间
- 进程创建开销大，线程创建开销小
- 进程间通信需要 IPC（管道、共享内存、信号），线程间直接共享数据

僵尸进程：子进程退出，父进程未调用 wait()，子进程的 PCB 仍存在
孤儿进程：父进程退出，子进程被 init 进程收养

查看僵尸进程：ps aux | grep 'Z'
```

#### Q3: IO 模型（select/poll/epoll）

```
五种 IO 模型：
1. 阻塞 IO：等待数据准备好（默认）
2. 非阻塞 IO：轮询检查，不阻塞
3. IO 多路复用：一个线程监控多个 FD
4. 信号驱动 IO：内核准备好后发信号通知
5. 异步 IO：内核完成所有操作后通知

select：
- FD 数量限制（1024）
- 每次调用需要传入全部 FD 数组（O(n) 复制）
- 内核需要遍历全部 FD（O(n)）
- 适合少量连接

poll：
- 无 FD 数量限制
- 仍然是 O(n) 遍历
- 适合中等数量连接

epoll（Linux 推荐）：
- 红黑树管理 FD（O(log n) 增删）
- 事件驱动，只返回活跃的 FD（O(1) 获取）
- LT（水平触发）vs ET（边缘触发）
- 适合大量连接（Nginx、Redis 都用 epoll）

面试回答：Redis 为什么快？单线程 + epoll IO 多路复用 + 纯内存操作
```

#### Q4: 内存管理

```
虚拟内存：
- 每个进程有独立的虚拟地址空间
- 通过页表映射到物理内存
- 页缺失（Page Fault）：访问的页面不在物理内存中

Swap：
- 物理内存不足时，将不常用的页面交换到磁盘
- Swap 过多会导致性能严重下降

OOM Killer：
- 内存耗尽时，内核选择进程杀死
- oom_score：分数越高越容易被杀
- oom_score_adj：手动调整优先级

查看内存：

cat /proc/meminfo    # 详细信息
vmstat 1             # 实时监控
```

### 5. Redis 面试题补充（⭐高频）

#### Q1: Redis 为什么快（单线程 IO 多路复用）

```
1. 纯内存操作：所有数据存在内存中，读写速度极快
2. 单线程模型：无线程切换和锁竞争开销
3. IO 多路复用：epoll 监听多个连接，单线程处理并发请求
4. 高效的数据结构：
   - SDS（Simple Dynamic String）：O(1) 获取长度，二进制安全
   - 压缩列表（ziplist）/ 紧凑列表（listpack）：节省内存
   - 跳表（skiplist）：O(log N) 的有序集合操作
   - 整数集合（intset）：小集合使用连续内存

注意：Redis 6.0 引入多线程，但仅用于网络 IO，命令执行仍然是单线程。
```

#### Q2: Redis 数据结构底层实现

| Redis 类型 | 底层实现 | 适用场景 |
|-----------|---------|---------|
| String | SDS | 缓存、计数器、分布式锁 |
| List | quicklist（双向链表 + 压缩列表） | 消息队列、最新列表 |
| Hash | ziplist / hashtable | 对象存储 |
| Set | intset / hashtable | 标签、去重 |
| ZSet | ziplist / skiplist + hashtable | 排行榜、延迟队列 |

#### Q3: Redis 过期策略（惰性删除 + 定期删除）

```
Redis 使用两种策略配合：

1. 惰性删除（Lazy Delete）：
   - 访问 key 时检查是否过期，过期则删除
   - 优点：不浪费 CPU
   - 缺点：过期 key 可能长期占用内存

2. 定期删除（Periodic Delete）：
   - 每隔 100ms 随机检查一批 key（默认 20 个）
   - 删除其中过期的 key
   - 如果过期 key 比例 > 25%，继续检查下一批
   - 优点：平衡 CPU 和内存

3. 内存淘汰策略（内存满时）：
   noeviction       - 不淘汰，写入报错（默认）
   allkeys-lru      - 淘汰最近最少使用的 key
   allkeys-lfu      - 淘汰使用频率最低的 key
   allkeys-random   - 随机淘汰
   volatile-lru     - 在设置了过期时间的 key 中淘汰 LRU
   volatile-lfu     - 在设置了过期时间的 key 中淘汰 LFU
   volatile-ttl     - 淘汰 TTL 最短的 key
```

#### Q4: Redis Cluster 原理

```
数据分片：
- 16384 个 Hash Slot
- 每个 Master 负责一部分 Slot
- key 的 Slot = CRC16(key) % 16384

高可用：
- 每个 Master 至少 1 个 Slave
- Master 故障时，Slave 自动提升为 Master
- 节点间通过 Gossip 协议通信

客户端请求：
- 如果 key 在当前节点，直接处理
- 如果 key 不在当前节点，返回 MOVED 重定向
- 客户端缓存 Slot -> Node 映射，减少重定向

集群限制：
- 不支持多 key 跨 slot 操作（可用 Hash Tag 解决：{user:1}:name）
- 不支持 SELECT 数据库（只能用 db0）
- 事务只能在同一个 slot 的 key 上执行
```

### 6. 常见场景题补充（⭐⭐中频）

#### Q1: 如何实现单点登录（SSO）

```
方案1：基于 Cookie + 中央认证服务器
1. 用户访问系统A，未登录，重定向到 SSO 认证中心
2. 用户在 SSO 登录成功，SSO 创建全局 Session
3. SSO 重定向回系统A，URL 带 Token
4. 系统A 用 Token 换取用户信息，创建本地 Session
5. 用户访问系统B，重定向到 SSO，SSO 发现已登录
6. SSO 重定向回系统B，带 Token，系统B 创建本地 Session

方案2：基于 JWT
1. SSO 登录成功后签发 JWT
2. 各系统验证 JWT 签名获取用户信息
3. 无需维护 Session，天然支持分布式

方案3：CAS 协议
- 标准化的 SSO 协议
- TGT（Ticket Granting Ticket）：全局票据
- ST（Service Ticket）：服务票据
```

#### Q2: 如何设计一个全局唯一的发号器

```
方案对比：
1. 数据库自增：简单但性能差，单点
2. UUID：无序，索引性能差
3. 雪花算法：推荐，趋势有序，高性能
4. Leaf-segment：美团方案，数据库号段 + 双 buffer

推荐方案：雪花算法
- 64位 ID：1位符号 + 41位时间戳 + 10位机器ID + 12位序列号
- 每毫秒可生成 4096 个 ID
- 趋势递增，对索引友好
- 分布式安全（机器ID不同）

时钟回拨问题：
- 方案1：等待时钟追上（简单但可能阻塞）
- 方案2：使用历史最大时间戳（可能重复）
- 方案3：多 worker 备份（切换到其他 worker）
```

#### Q3: 如何实现接口幂等性

```
幂等性：同一个请求执行一次和多次的效果相同。

常见方案：

1. 唯一索引（数据库层）
   - 利用数据库唯一索引约束
   - 重复请求会报 DuplicateKeyException
   - 适合：创建订单、注册

2. Token 机制（Redis）
   - 请求前先获取 Token（Redis SETNX）
   - 提交请求时携带 Token
   - 服务端删除 Token，重复请求因 Token 不存在而拒绝
   - 适合：表单提交、支付

3. 乐观锁（版本号）
   - UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?
   - 重复请求因版本号不匹配而失败
   - 适合：更新操作

4. 状态机
   - 订单状态：待支付 -> 已支付 -> 已发货 -> 已完成
   - 重复支付请求因状态不是"待支付"而拒绝
   - 适合：状态流转类操作

5. 去重表
   - 创建唯一请求ID的记录
   - 重复请求因记录已存在而拒绝

# Token 机制实现
import redis
import uuid

class IdempotentService:
    def __init__(self):
        self.redis = redis.Redis()

    def get_token(self):
        """获取幂等 Token"""
        token = str(uuid.uuid4())
        self.redis.setex(f"idempotent:{token}", 3600, "1")
        return token

    def process_with_idempotent(self, token, business_func):
        """带幂等检查的业务处理"""
        key = f"idempotent:{token}"
        # Lua 脚本保证原子性：检查并删除
        lua_script = """
        if redis.call("get", KEYS[1]) then
            return redis.call("del", KEYS[1])
        else
            return 0
        end
        """
        result = self.redis.eval(lua_script, 1, key)
        if result == 1:
            return business_func()  # 首次请求，执行业务
        else:
            return {"error": "重复请求"}  # 重复请求
