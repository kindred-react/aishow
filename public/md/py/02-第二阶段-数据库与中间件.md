索引原理（面试必考）：
- B+树结构：所有数据存在叶子节点，叶子节点通过链表连接，支持范围查询
- 聚簇索引 vs 非聚簇索引：聚簇索引叶子节点存完整数据行，非聚簇索引存主键值
- 覆盖索引：查询的列全部在索引中，不需要回表（Using index）
- 最左前缀原则：联合索引(a,b,c)可以覆盖 a / a,b / a,b,c 的查询
- 索引下推：MySQL 5.6+，在索引遍历过程中对索引中包含的字段执行过滤
-- 索引失效的常见场景

SELECT * FROM users WHERE name != 'admin';           -- 不等于可能失效
SELECT * FROM users WHERE name IS NULL;              -- IS NULL可以用索引

SELECT * FROM users WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
SELECT * FROM users WHERE name LIKE 'john%';

### 2.2.2 缓存三大问题（面试必考）
问题1：缓存穿透
定义：查询一个不存在的数据，缓存中没有，每次都打到数据库。
- 解决方案1：布隆过滤器 - 在缓存前加一层布隆过滤器，不存在的直接返回
- 解决方案2：缓存空值 - 查询不到的也缓存一个空值，设置较短过期时间
- 解决方案3：参数校验 - 在入口层对请求参数做合法性校验
# 布隆过滤器实现缓存穿透防护
from pybloom_live import ScalableBloomFilter

bf = ScalableBloomFilter(initial_capacity=100000)

# 写入时同步更新布隆过滤器
def create_article(article_id, content):
    db.save(article_id, content)
    redis.set(f"article:{article_id}", content)
    bf.add(article_id)  # 加入布隆过滤器

# 查询时先过布隆过滤器
def get_article(article_id):
    if article_id not in bf:  # 一定不存在
        return None
    cached = redis.get(f"article:{article_id}")
    if cached:
        return cached
    article = db.get(article_id)
    if article:
        redis.set(f"article:{article_id}", article)
    else:
        redis.set(f"article:{article_id}", "", ex=60)  # 缓存空值60秒
    return article

问题2：缓存击穿
定义：某个热点Key过期瞬间，大量请求同时打到数据库。
- 解决方案1：互斥锁 - 只允许一个线程查数据库并重建缓存
- 解决方案2：永不过期 + 异步更新 - 逻辑上不设置过期，后台线程定时刷新
- 解决方案3：热点数据预加载 - 在过期前主动刷新
# 互斥锁方案
def get_hot_data(key):
    data = redis.get(key)
    if data:
        return data

    # 获取分布式锁
    lock_key = f"lock:{key}"
    if redis.set(lock_key, "1", nx=True, ex=10):  # nx=不存在才设置
        try:
            data = db.query(key)  # 查数据库
            redis.set(key, data, ex=3600)  # 重建缓存
            return data
        finally:
            redis.delete(lock_key)
    else:
        time.sleep(0.1)  # 等待后重试
        return get_hot_data(key)

问题3：缓存雪崩
定义：大量Key在同一时间过期，或Redis宕机，大量请求打到数据库。
- 解决方案1：随机过期时间 - 给缓存过期时间加随机值（如3600 ± 随机300秒）
- 解决方案2：多级缓存 - 本地缓存(Caffeine) + Redis缓存 + 数据库
- 解决方案3：Redis高可用 - 主从复制 + 哨兵模式 + Cluster集群
- 解决方案4：熔断降级 - 数据库压力过大时触发降级，返回默认值

# Redis分布式锁（基础版）
def acquire_lock(lock_key, request_id, expire=10):
    """获取锁：SET key value NX EX"""
    return redis.set(lock_key, request_id, nx=True, ex=expire)

def release_lock(lock_key, request_id):
    """释放锁：Lua脚本保证原子性"""
    lua_script = """
    if redis.call("get", KEYS[1]) == ARGV[1] then
        return redis.call("del", KEYS[1])
    else
        return 0
    end
    """
    return redis.eval(lua_script, 1, lock_key, request_id)

# ===== 秒杀系统核心代码 =====
import redis
import json
from fastapi import FastAPI, HTTPException, Request
from contextlib import contextmanager
from slowapi import Limiter
from slowapi.util import get_remote_address

app = FastAPI()
r = redis.Redis(host='localhost', port=6379, db=0)
limiter = Limiter(key_func=get_remote_address)  # 基于 IP 限流

# ===== 1. Redis预扣库存（Lua脚本保证原子性）=====
DEDUCT_STOCK_LUA = """
local key = KEYS[1]
local quantity = tonumber(ARGV[1])
local current = tonumber(redis.call('GET', key) or '0')
if current >= quantity then
    redis.call('DECRBY', key, quantity)
    return 1
else
    return 0
end
"""

@app.post("/seckill/{product_id}")
@limiter.limit("100/second")
async def seckill(request: Request, product_id: int, user_id: int, request_id: str = None):
    """秒杀接口：集成限流、幂等性、Redis预扣库存"""
    # 0. 幂等性检查（防止重复下单）
    if request_id:
        idempotent_key = f"idempotent:{request_id}"
        if not r.set(idempotent_key, "1", nx=True, ex=300):
            return {"status": "duplicate", "message": "请勿重复提交"}

    # 1. Redis预扣库存
    stock_key = f"stock:{product_id}"
    success = r.eval(DEDUCT_STOCK_LUA, 1, stock_key, 1)
    if not success:
        raise HTTPException(status_code=400, detail="库存不足")

    # 2. 发送MQ消息异步下单
    order_data = {
        "user_id": user_id,
        "product_id": product_id,
        "timestamp": time.time()
    }
    mq_client.send("seckill_orders", json.dumps(order_data))

    return {"status": "success", "message": "下单中..."}

# ===== 4. MQ消费者（异步下单）=====
def consume_orders():
    for message in mq_client.consume("seckill_orders"):
        order_data = json.loads(message)
        try:
            # 数据库下单（乐观锁）
            with get_db() as db:
                order = create_order(db, order_data)
                # 扣减数据库库存
                update_db_stock(db, order_data["product_id"])
                db.commit()
        except Exception as e:
            # 重试或进入死信队列
            mq_client.retry(message, max_retries=3)

### 1. MySQL 慢查询日志分析实战（⭐高频）

#### 概念讲解

慢查询日志（Slow Query Log）记录了执行时间超过 `long_query_time` 阈值的 SQL 语句，是排查数据库性能问题的第一手资料。

#### slow\_query\_log 配置

```sql
-- 查看当前慢查询配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';
SHOW VARIABLES LIKE 'slow_query_log_file';

-- 开启慢查询日志
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;          -- 超过1秒记录
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
SET GLOBAL log_queries_not_using_indexes = 'ON';  -- 记录未使用索引的查询

-- 永久配置（my.cnf）
[mysqld]
slow_query_log = 1
long_query_time = 1
slow_query_log_file = /var/log/mysql/slow.log
log_queries_not_using_indexes = 1
```

#### long\_query\_time 阈值

- 建议值：0.5~2 秒（根据业务 SLA 调整）
- 生产环境建议先设为 0，收集所有慢查询，再逐步调高
- 注意：`long_query_time` 精度到秒，设为 0.5 实际可能记录 >= 1 秒的查询（取决于版本）

#### mysqldumpslow 分析

```bash
# 基本用法
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# -s 排序方式：t=总时间, at=平均时间, l=锁时间, c=次数
# -t 显示前N条
# -g 正则过滤

# 常用组合
mysqldumpslow -s at -t 5 /var/log/mysql/slow.log     # 平均执行时间最慢的5条
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log     # 执行次数最多的10条
mysqldumpslow -s t -t 5 -g "SELECT.*FROM orders" /var/log/mysql/slow.log  # 过滤orders表
```

#### pt-query-digest 报告解读

```bash

apt-get install percona-toolkit

# 分析慢查询日志
pt-query-digest /var/log/mysql/slow.log > slow_report.txt

# 分析实时查询
pt-query-digest --processlist h=localhost,u=root,p=password

# 报告关键指标解读：

# 4. Rows examined: 扫描行数（越大越需要优化）

```

**pt-query-digest 输出示例：**
```

# ==== ================== ============== ======= ======= ===== =============
#    1 0xABC123...        320.5s 45.2%    1500   0.213s  0.12 SELECT orders
#    2 0xDEF456...        180.2s 25.4%     800   0.225s  0.08 SELECT users

# Query 1: 320.5s total, 0.213s avg, 1500 calls

# Rows sent: 1,500 avg (1 per call)

```

#### 面试问答

**Q: 发现慢查询后如何优化？**
1. `EXPLAIN` 分析执行计划，看是否走索引
2. 检查索引是否失效（函数、左模糊、隐式转换等）
3. 优化 SQL 写法（避免 SELECT *、子查询改 JOIN）
4. 加索引或优化现有索引
5. 考虑分库分表、读写分离

### 2. 读写分离方案（⭐高频）

#### 概念讲解

读写分离将数据库的读操作和写操作分发到不同的数据库实例，主库负责写，从库负责读，通过主从复制同步数据。

#### 主从延迟问题及解决方案

```
[写入] --> [主库 Master]
                |
                | binlog 复制（异步）
                v
[读取] --> [从库 Slave1, Slave2, ...]
```

**主从延迟原因：**
1. 主库并发写入量大，binlog 生成速度快
2. 从库单线程回放（MySQL 5.6 之前）
3. 大事务（一次更新大量数据）
4. 从库硬件性能低于主库
5. 网络延迟

**解决方案：**

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| 强制走主库 | 写后读请求强制路由到主库 | 写后立即读的场景 |
| 半同步复制 | 主库等待至少一个从库确认 | 对一致性要求高的场景 |
| 并行复制 | 从库多线程回放 binlog | MySQL 5.7+ |
| GTID 复制 | 全局事务ID，便于故障恢复 | 主从切换频繁的场景 |
| 延迟监控 | 监控 Seconds\_Behind\_Master | 运维告警 |

```sql
-- 查看主从延迟
SHOW SLAVE STATUS\G
-- 关键字段：Seconds_Behind_Master（0表示无延迟）
```

#### 应用层读写分离实现

```python
# ===== Python 应用层读写分离 =====
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

class ReadWriteSplitter:
    """简单的读写分离路由"""
    def __init__(self, master_url, slave_urls):
        self.master_engine = create_engine(master_url)
        self.slave_engines = [create_engine(url) for url in slave_urls]
        self._slave_index = 0

    def get_master_session(self):
        """获取写会话（连接主库）"""
        Session = sessionmaker(bind=self.master_engine)
        return Session()

    def get_slave_session(self):
        """获取读会话（连接从库，轮询）"""
        engine = self.slave_engines[self._slave_index % len(self.slave_engines)]
        self._slave_index += 1
        Session = sessionmaker(bind=engine)
        return Session()

# 使用
db = ReadWriteSplitter(
    master_url="mysql://root:xxx@master-db:3306/mydb",
    slave_urls=[
        "mysql://root:xxx@slave-db0:3306/mydb",
        "mysql://root:xxx@slave-db1:3306/mydb",
    ]
)

# 写操作走主库
with db.get_master_session() as session:
    session.add(User(name="Alice"))
    session.commit()

# 读操作走从库
with db.get_slave_session() as session:
    users = session.query(User).all()
```

### 3. MySQL InnoDB vs MyISAM 对比（⭐高频）

#### 对比表

| 特性 | InnoDB | MyISAM |
|------|--------|--------|
| 事务支持 | 支持（ACID） | 不支持 |
| 锁粒度 | 行级锁 | 表级锁 |
| 外键约束 | 支持 | 不支持 |
| 崩溃恢复 | 支持（redo log） | 不支持 |
| 全文索引 | 5.6+ 支持 | 原生支持 |
| 索引结构 | 聚簇索引 | 非聚簇索引 |
| COUNT(*) | 需要遍历索引 | 直接存储行数 |
| 存储空间 | 较大（数据和索引在一起） | 较小（数据和索引分开） |

| 热备份 | 支持（mysqldump/ibbackup） | 不支持 |
| 适用场景 | 高并发读写、事务 | 读多写少、不需要事务 |

#### 面试问答

**Q: 为什么互联网项目几乎都用 InnoDB？**
1. 支持事务，保证数据一致性
2. 行级锁，高并发性能好
3. 支持外键，保证数据完整性
4. 崩溃恢复能力强（redo log + checkpoint）
5. MVCC 支持高并发读

**Q: MyISAM 还有什么使用场景？**
- 全文索引（InnoDB 5.6+ 也支持了）
- 只读或读多写少的场景（如日志表、配置表）
- COUNT(*) 不需要扫表

### 4. MySQL 三大日志（⭐高频）

#### redo log（WAL、checkpoint）

**Write-Ahead Logging（WAL）机制：** 先写日志，再写磁盘。

```
[修改操作] --> [写入 redo log buffer] --> [写入 redo log file] --> [写入数据页（异步）]
```

```sql
-- redo log 配置
SHOW VARIABLES LIKE 'innodb_log_buffer_size';     -- 默认 16MB
SHOW VARIABLES LIKE 'innodb_log_file_size';        -- 默认 48MB（5.7）/ 2GB（8.0）
SHOW VARIABLES LIKE 'innodb_flush_log_at_trx_commit';
-- 0: 每秒写入并刷盘（可能丢失1秒数据）
-- 1: 每次事务提交都刷盘（最安全，默认值）
-- 2: 每次事务提交写入OS缓存，每秒刷盘（可能丢失1秒数据）
```

**Checkpoint 机制：** 定期将脏页（内存中修改但未刷盘的数据页）刷回磁盘，缩短崩溃恢复时间。checkpoint 之后的数据页已持久化，对应的 redo log 可以被覆盖。

#### undo log（MVCC、回滚）

```
[修改操作] --> [将旧数据写入 undo log] --> [修改数据页]
```

- **回滚**：事务回滚时，根据 undo log 恢复旧数据
- **MVCC**：通过 undo log 链实现多版本并发控制
  - 每行数据有两个隐藏列：`trx_id`（最后修改的事务ID）和 `roll_pointer`（指向 undo log 的指针）
  - 不同版本通过 `roll_pointer` 形成链表

#### binlog（三种格式、主从复制）

```sql
-- binlog 配置
SHOW VARIABLES LIKE 'log_bin';                    -- 是否开启
SHOW VARIABLES LIKE 'binlog_format';              -- 格式
SHOW VARIABLES LIKE 'sync_binlog';                -- 刷盘策略
SHOW VARIABLES LIKE 'expire_logs_days';           -- 保留天数
```

**三种格式：**

| 格式 | 内容 | 优点 | 缺点 |
|------|------|------|------|
| STATEMENT | 记录 SQL 语句 | 日志量小 | 不安全（NOW()、UUID() 等函数主从不一致） |
| ROW | 记录行变更 | 最安全，主从一致 | 日志量大（大事务时） |
| MIXED | 自动选择 | 兼顾大小和安全 | 行为不可预测 |

**主从复制流程：**
```
1. Master 将变更写入 binlog
2. Dump 线程读取 binlog 发送给 Slave
3. Slave 的 IO 线程接收并写入 relay log
4. Slave 的 SQL 线程回放 relay log
```

#### 两阶段提交

保证 redo log 和 binlog 的一致性：

```
阶段1（Prepare）：
  InnoDB 写入 redo log（标记为 prepare 状态）
  --> 告诉 Binlog：可以写了

阶段2（Commit）：
  Binlog 写入成功 --> InnoDB 将 redo log 标记为 commit
  Binlog 写入失败 --> InnoDB 回滚 redo log
```

**为什么需要两阶段提交？**
- 如果先写 redo log 再写 binlog：redo log 写入后崩溃，binlog 没写入，主库恢复后数据存在，从库没有 --> 主从不一致
- 如果先写 binlog 再写 redo log：binlog 写入后崩溃，redo log 没写入，主库恢复后数据丢失，从库有 --> 主从不一致
- 两阶段提交保证两者原子性：要么都成功，要么都失败

### 5. B+树 vs B树 vs 红黑树 vs 哈希（⭐高频）

#### 各数据结构对比

| 特性 | B+树 | B树 | 红黑树 | 哈希 |
|------|------|------|--------|------|
| 查找时间 | O(log\_d N) | O(log\_d N) | O(log N) | O(1) |
| 范围查询 | 优秀（叶子链表） | 一般 | 一般 | 不支持 |
| 磁盘IO | 最少（树矮胖） | 较多 | 很多（树高） | 1次 |
| 数据存储 | 仅叶子节点 | 所有节点 | 所有节点 | 桶中 |
| 有序性 | 有序 | 有序 | 有序 | 无序 |
| MySQL 使用 | InnoDB 索引 | - | 内存索引 | Memory 引擎索引 |

#### 为什么 MySQL 选 B+ 树

1. **磁盘 IO 次数少**：B+ 树的非叶子节点不存数据，每个节点可以存更多 key，树更矮。3 层 B+ 树可以存约 2000 万条数据（假设每页 16KB，key 8 字节，指针 6 字节）。

2. **范围查询高效**：叶子节点通过双向链表连接，范围查询只需找到起点后遍历链表。

3. **查询性能稳定**：每次查询都从根到叶子，路径长度相同，都是 O(log N)。

4. **缓存友好**：非叶子节点不存数据，一页可以放更多 key，缓存在 Buffer Pool 中的索引页覆盖更多范围。

```
B+ 树结构示意：
              [10 | 20 | 30]              <-- 非叶子节点（仅存key）
             /    |     |     \
        [1-10]  [11-20] [21-30] [31-40]   <-- 叶子节点（存数据 + 链表连接）
          |       |       |       |
         链表连接 <--------->--------->     <-- 范围查询直接遍历链表
```

### 6. Kafka 消息可靠性深入（⭐高频）

#### acks=0/1/all 区别

| 配置 | 含义 | 可靠性 | 吞吐量 |
|------|------|--------|--------|
| acks=0 | 生产者不等待任何确认 | 最低（可能丢失） | 最高 |
| acks=1 | 等待 Leader 写入成功 | 中等（Leader 挂了可能丢失） | 中等 |
| acks=all（或 -1） | 等待所有 ISR 副本写入成功 | 最高 | 最低 |

```python
# Kafka Python 生产者配置
from kafka import KafkaProducer

producer = KafkaProducer(
    bootstrap_servers=['kafka1:9092', 'kafka2:9092'],
    acks='all',              # 等待所有ISR副本确认
    retries=3,               # 重试次数
    max_in_flight_requests_per_connection=1,  # 顺序保证
    enable_idempotence=True, # 开启幂等性
    compression_type='gzip',  # 压缩
)
```

#### ISR 机制

```
ISR（In-Sync Replicas）：与 Leader 保持同步的副本集合

[Leader] <--同步--> [Follower1]  (在ISR中)
[Leader] <--同步--> [Follower2]  (在ISR中)
[Leader] <--落后--> [Follower3]  (不在ISR中，OSR)

min.insync.replicas = 2  # ISR中最少需要2个副本确认
```

- **ISR 动态调整**：Follower 落后太多（`replica.lag.time.max.ms`，默认 10 秒）会被踢出 ISR
- **unclean.leader.election.enable**：是否允许非 ISR 副本成为 Leader（数据丢失风险）

#### min.insync.replicas

```properties
# broker 配置
min.insync.replicas=2  # ISR 中最少需要 2 个副本

# 配合 acks=all 使用：
# 如果 ISR 中副本数 < min.insync.replicas，写入会返回 NotEnoughReplicasException
# 这是"宁可不可用，也不丢数据"的策略
```

#### 幂等生产者（enable.idempotence）

```properties
# 开启幂等生产者
enable.idempotence=true
# 等价于设置：
# acks=all
# retries=Integer.MAX_VALUE
# max.in.flight.requests.per.connection=1
```

原理：每个 Producer 分配一个 PID（Producer ID），每条消息带有一个 Sequence Number。Broker 收到消息后检查 `<PID, Partition, SeqNum>`，重复的直接丢弃。

### 7. 延迟队列实现方案（⭐⭐中频）

#### Redis ZSet 实现

```python
import redis
import time
import json
import threading

r = redis.Redis(host='localhost', port=6379)

class RedisDelayQueue:
    """基于 Redis ZSet 的延迟队列"""
    def __init__(self, queue_name):
        self.queue_name = queue_name
        self.running = False

    def push(self, task_data, delay_seconds):
        """添加延迟任务"""
        score = time.time() + delay_seconds
        r.zadd(self.queue_name, {json.dumps(task_data): score})

    def pop(self):
        """取出到期的任务"""
        now = time.time()
        # ZRANGEBYSCORE 取出所有 score <= now 的任务
        results = r.zrangebyscore(self.queue_name, 0, now, start=0, num=1)
        if not results:
            return None

        task_json = results[0]
        # 使用 ZREM 原子删除（防止多消费者重复消费）
        if r.zrem(self.queue_name, task_json):
            return json.loads(task_json)
        return None

    def start_consumer(self, callback, poll_interval=1):
        """启动消费者"""
        self.running = True
        while self.running:
            task = self.pop()
            if task:
                callback(task)
            else:
                time.sleep(poll_interval)

# 使用
dq = RedisDelayQueue("delay_queue")
dq.push({"order_id": 1001, "action": "timeout_cancel"}, delay_seconds=1800)  # 30分钟后执行

def handle_task(task):
    print(f"处理延迟任务: {task}")

dq.start_consumer(handle_task)
```

#### RabbitMQ 死信队列 + TTL

```python
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()

# 1. 声明死信队列（最终消费的队列）
channel.queue_declare(queue='delayed_orders', durable=True)

# 2. 声明延迟队列（设置 TTL 和死信路由）
args = {
    'x-message-ttl': 30000,                    # 30秒后过期
    'x-dead-letter-exchange': '',              # 死信路由到默认 exchange
    'x-dead-letter-routing-key': 'delayed_orders',  # 死信路由到目标队列
}
channel.queue_declare(queue='pending_orders', arguments=args, durable=True)

# 3. 发送消息到延迟队列
channel.basic_publish(
    exchange='',
    routing_key='pending_orders',
    body='{"order_id": 1001}',
    properties=pika.BasicProperties(delivery_mode=2)  # 持久化
)

# 4. 消费死信队列（30秒后消息会自动路由到这里）
def callback(ch, method, properties, body):
    print(f"处理超时订单: {body}")
    ch.basic_ack(delivery_tag=method.delivery_tag)

channel.basic_consume(queue='delayed_orders', on_message_callback=callback)
channel.start_consuming()
```

### 8. 数据库连接池深入（⭐⭐中频）

#### QueuePool 原理

SQLAlchemy 默认使用 `QueuePool`，工作原理：

```
[应用请求连接]
      |
      v
[QueuePool]
  |-- pool_size=5        # 常驻连接数
  |-- max_overflow=10    # 最大溢出连接数
  |-- pool_timeout=30    # 获取连接超时时间
  |
  |-- [conn1] [conn2] [conn3] [conn4] [conn5]  <-- 常驻连接
  |-- [conn6] ... [conn15]                     <-- 溢出连接（用完回收）
```

- 请求连接时，优先从池中取空闲连接
- 池满时，创建溢出连接（不超过 max_overflow）
- 溢出连接空闲后自动回收
- 常驻连接永远保持

#### pool\_size/max\_overflow 调优

```python
from sqlalchemy import create_engine

engine = create_engine(
    "mysql+pymysql://root:xxx@localhost/mydb",
    pool_size=10,           # 常驻连接数（建议 = CPU核数 * 2 ~ 4）
    max_overflow=20,        # 溢出连接数（应对突发流量）
    pool_timeout=30,        # 获取连接超时（秒）
    pool_recycle=3600,      # 连接回收时间（秒），防止 MySQL 8小时断开
    pool_pre_ping=True,     # 使用前先 ping，防止连接已断开
    echo=False,             # 不打印 SQL
)

# 调优经验：
# 1. pool_size：通常 5~20，取决于并发量和数据库承受能力
# 2. max_overflow：pool_size 的 1~3 倍
# 3. pool_recycle：小于 MySQL 的 wait_timeout（默认 8 小时）
# 4. pool_pre_ping=True：防止 "MySQL has gone away" 错误
```

#### 连接泄漏排查

